# Skill Completion Standard v1 -- baseline raised post LT+CEPS

**Sealed**: 2026-05-25. **Predecessor**: ad-hoc skill creation. **Replaces**: implicit "build until it feels done".

A new PP skill is *complete* only when all seven clauses below are satisfied with empirical evidence. Missing evidence == not complete. Reality Contract applies: the evidence is the test output, not the description of the test.

---

## The seven clauses

### C1 -- Empirical pass-gate declared *before* the skill is built

The skill's success metric is written down before the first byte of skill content is. The pass-gate states:

- the metric (what is measured, on what fixture)
- the threshold (the numeric value above which the skill ships)
- the kill condition (the threshold below which the skill is abandoned or pivoted)

Without a pass-gate declared up front, every output retroactively becomes "good enough" -- the skill silently regresses to a placebo. The pass-gate is the only force that can answer *"is this actually better than the baseline?"*.

**Evidence required**: a `vault/plans/<skill>-plan.md` artifact containing the pass-gate clause, dated before the first commit that introduces skill content.

### C2 -- Side-by-side comparison with the no-skill baseline

The skill is run against the same prompts twice: once with the skill present, once without. The two outputs are scored by the same rubric. The skill ships only if with-skill aggregate score >= without-skill aggregate score + a declared delta (see C1).

Without the comparison, a skill that just adds verbose preamble can score higher on "depth" without producing better answers. The comparison forces *delta-vs-baseline*, not absolute quality.

**Evidence required**: `tools/test_<skill>_empirical.py` (or equivalent) producing a JSON fixture readable by the Owner for blind-eval. The fixture must include both arms (with / without) and a scoring template.

### C3 -- No-collision verified against existing modules

The skill must not double-fire or contradict any of:

- JIT skill loader trigger families (`tools/jit_skill_loader.py` -- currently 10 families)
- Intent-Lock (concurrent-execution protection)
- Arch-Check piggyback (>=2 design verbs)
- Vague-prompt lint signal
- Active Stop / PreToolUse / PostToolUse / UserPromptSubmit hooks
- The existing 29 baseline tests in `tests/`

The collision test runs prompts that trigger overlapping subsystems and verifies the new skill defers, coexists, or wins per its declared mutex policy. Coexistence is acceptable when the advisories are orthogonal; deference is required when the semantic overlap is real.

**Evidence required**: a no-collision section in `tools/test_<skill>_*.py` with at least 4 prompts covering: (a) trigger-only-new-skill, (b) trigger-only-existing-module, (c) both-trigger expected coexistence, (d) neither-trigger expected silence.

### C4 -- Distribution integrated with CEPS

When the skill records a behavior, learning, or pattern that should compound across sessions, it MUST route through CEPS:

- `tools/ceps.py::record_error` for failures the skill caught
- `vault/ceps/events.jsonl` for telemetry the skill emits
- The lateral-thinking audit-trail block format is the reference shape

Skills that fire-and-forget without writing to CEPS lose the compounding effect that the system promises. Memory of past failures, ratings, and patterns disappears with the session.

**Evidence required**: a CEPS write-path in the skill's runtime tooling, or an explicit declaration in the skill's frontmatter that the skill is purely advisory and emits no telemetry.

### C5 -- Auto-test stub generated by `tools/ceps_test_gen.py`

If the skill produces a behavior that future regressions could silently undo, a CEPS event of category `regression` / `security` / `drift` is recorded so the stub generator produces a SKIP-by-default test stub in `tests/ceps_generated/`. The Owner fills the assertion when convenient; the stub itself is the placeholder that ensures the regression cannot vanish from the test surface unnoticed.

**Evidence required**: a CEPS event for the skill's primary correctness invariant + a stub visible under `tests/ceps_generated/test_<sig[:12]>.py`. Stubs older than 7 days are flagged WARN by `tools/ceps_test_gen.py audit` -- the gate amendment that keeps stale stubs from rotting silently.

### C6 -- Atomic write for all markdown appends

After the 2026-05-23 apex-completion-standard corruption (shell `cat >>` heredoc lost an entire axis section while a post-write hook ran), every markdown append in this codebase MUST use the atomic pattern:

```python
fd, tmp = tempfile.mkstemp(prefix=f".{path.name}.", suffix=".tmp", dir=path.parent)
with os.fdopen(fd, "w", encoding="utf-8", newline="\n") as fh:
    fh.write(new_content)
os.replace(tmp, path)
```

`tools/_apex_axis_restore.py`, `tools/_append_lessons_override.py`, and `tools/ceps.py::_atomic_append` are the reference implementations. Shell `cat >>` to a tracked markdown is prohibited.

**Evidence required**: no `cat >>` invocation against tracked markdown in the skill's tooling. Atomic helpers are used for every append.

### C7 -- RTK proxy compatibility

The skill's outputs must remain legible after the RTK token-compression proxy rewrites the command stream. RTK is per-command (not session-wide) and ships in PreToolUse for `Bash`; skill tooling that hard-depends on raw output may silently break under compression.

The contract:

- Skill tools accept compressed stdin (RTK is fail-open; uncompressed input remains the dominant case)
- Skill output is interpretable by downstream consumers without raw byte assumptions
- If the skill emits structured data (JSON, JSONL), it round-trips through RTK without semantic loss

**Evidence required**: skill tooling does not parse arbitrary slices of raw command output (it uses subprocess capture, structured parsers, or schema-validated JSON). Free-text scraping of tool stderr is prohibited.

## Enforcement

A skill that fails any clause cannot be merged to `main`. The plan document for the new skill must answer each clause explicitly with a pointer to the evidence. The Owner-facing PR description must include a Skill Completion Standard table:

| Clause | Evidence path | Status |
|---|---|---|
| C1 pass-gate declared | `vault/plans/<skill>-plan.md` section X | [x] |
| C2 side-by-side test  | `tools/test_<skill>_empirical.py` | [x] |
| C3 no-collision       | `tools/test_<skill>_collision.py` | [x] |
| C4 CEPS distribution  | record_error call at `<file>:<line>` | [x] |
| C5 auto-test stub     | `tests/ceps_generated/test_<sig>.py` | [x] |
| C6 atomic appends     | grep proof: no `cat >>` against tracked .md | [x] |
| C7 RTK compatible     | no raw-stdout slicing in skill tooling | [x] |

Missing rows or `[ ]` checkboxes block the merge.
