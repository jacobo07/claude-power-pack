# Knowledge Capture Governance

CGF Workstream C (2026-07-22). Three systems capture or move "knowledge" in
this estate: `fable_distillation` (FD), `akos_knowledge` (AKOS), and the
global `compound-learnings` skill. The CGF audit flagged them as an apparent
triple-overlap ("CONSOLIDATE -- 3 overlapping surfaces"). Direct inspection
of all three shows they are **not duplicates** — each owns a genuinely
different object, at a different cadence, producing a different artifact
type. The real defect was not duplication; it was a missing bridge between
two of them. This document is the consolidation: it fixes the bridge and
draws the ownership lines so the next session never re-diagnoses this triple
as an overlap again.

## The one real gap: a write-without-read

`fd_07_flywheel.py`'s `_writeback()` (line ~226-234) already flags high-value
findings as Owner-promotable: when a deposit's destination is `hard_rule`, it
separately appends to `ukdl_candidates_<repo>.jsonl` with
`status: "candidate -- Owner promotes to ukdl-universal.md"`. Grepping the
whole estate for `ukdl_candidates` turns up exactly one writer (FD itself)
and zero readers. This is the pattern named in
[[feedback_write_without_read_incomplete_system]]: a WRITER with no READER is
documentation, not recovery — these candidates have been accumulating with no
path back to the Owner's attention.

`modules/fable_distillation/ukdl_queue.py` (separately, PLANNED per
`vault/liveness/reachability_registry.json`) is NOT the reader either —
`fd_07_flywheel.py`'s `_writeback` writes the candidate file directly, inline,
never calling into `ukdl_queue.py` at all. The two modules are unconnected
siblings, not a producer/consumer pair.

**Fix applied:** `compound-learnings/SKILL.md` Step 1 (Gather Learnings) now
names `~/.claude/state/fable_distillation/ukdl_candidates_*.jsonl` as an
additional gather source, read alongside the existing session-log sources.
This is not a fourth system — it is compound-learnings' own existing
gather step, extended to see FD's own promotion queue, closing the loop the
CGF audit predicted without adding a new moving part.

## Known gap, not fixed here (Owner decision, not urgent)

`compound-learnings/SKILL.md`'s documented PRIMARY input path,
`<cwd>/.claude/cache/learnings/*.md`, has no producer anywhere in this
estate — every real run has always used the `memory/sessions/session_*.md`
fallback. This is not blocking (the fallback works and is exercised), so it
is named here rather than force-fixed: either (a) something should write
per-session distilled learning files to that path, or (b) the primary-path
reference should be removed from the skill doc as aspirational-but-unbuilt.
Left for the Owner; not in scope for this consolidation pass.

## Ruling: no fourth system

None of purpose, authority, input, output, schema, dedup, or retention
overlap enough between any two of the three to justify a merge, and AKOS's
read-only distribution direction is architecturally the *inverse* of the
other two's capture direction — merging it in would conflate a consumer with
two producers. The fix was a one-line routing addition to an existing
gather step, not a new coordinating system. Per CLAUDE.md's own standing
rule: "No crear un cuarto sistema... salvo que la arquitectura lo exija con
evidencia clara y no exista owner adecuado ya" — no such evidence exists
here; each system already has a clear owner.
