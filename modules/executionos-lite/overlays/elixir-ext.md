# Overlay: Elixir Extended (MC-OVO-30-EXT — 8 of 8 language overlays)

Loads when CWD contains `mix.exs` AND (Nerves signatures OR Ash signatures OR Livebook signatures OR `rel/` release config OR Gleam `gleam.toml`). ~290 tokens.

**Cross-link:** base OTP/GenServer/Phoenix discipline lives in `elixir.md`. This file covers the adjacent-but-distinct Elixir surfaces: Nerves (embedded), Ash (resource framework), Livebook (notebooks), mix releases, Gleam interop, and Elixir 1.18+ / OTP 27+ specifics. Do not duplicate base rules here — load both overlays together.

- **Stack split:** (a) Nerves firmware = `:nerves_bootstrap` + `target` config + burn-to-SD workflow, cross-compiles to `MIX_TARGET=rpi4` / `rpi5` / `bbb` / `x86_64`. (b) Ash app = `use Ash.Resource` + `use Ash.Api` + code-interfaces, data layer via `AshPostgres.DataLayer`. (c) Livebook notebook = `.livemd` files with smart cells, often sidecar to an umbrella project. (d) Gleam alongside Elixir = `gleam.toml` + `rebar.config` producing BEAM modules callable from Elixir via `:module_name.fn(arg)`. These stacks compose; a single repo may run all four.
- **Nerves discipline:** `MIX_TARGET` controls the entire build graph — always set it explicitly (`MIX_TARGET=rpi4 mix firmware`), never rely on the shell default. `:nerves_system_<target>` dep pinned to an exact tag (no `~>`) — firmware reproducibility depends on it. `VintageNet` for network config, `NervesHub` for OTA updates, `Nerves.Runtime` for partition info. Never call `System.halt/1` in firmware code — use `Nerves.Runtime.reboot()` so the watchdog + A/B partition logic fires correctly. Host-build (dev-on-laptop) uses `MIX_TARGET=host` and stubs hardware calls via `:nerves_system_host` shims — keep hardware-facing modules behind a behaviour so the host can inject a mock.
- **Ash discipline:** actions are the API surface — every resource declares `actions do read :list; create :register ...` explicitly; no implicit CRUD. Authorization via `policies do ... end` runs at the data-layer boundary, not in the controller. `code_interface do define :register, args: [:email] end` exposes actions as plain functions — prefer these over direct `Ash.Api.create/2` calls so callers get typed errors. NEVER bypass Ash to write raw Ecto queries against an Ash-managed table — you lose authorization, notifications, and calculations. If an escape hatch is genuinely needed, add a `manual` action with a documented reason; don't sneak a `Repo.insert/1` in a sibling module.
- **Livebook discipline:** `.livemd` notebooks are source-controlled Markdown with executable cells — commit them. Smart cells (DB query, chart, map) render deterministic Markdown + Elixir; edits produce clean diffs. For production use Kino + Explorer + VegaLite for dataframes + charts. NEVER store secrets in a `.livemd` — use Livebook secrets (stored in `~/.livebook/`) referenced via `System.fetch_env!("LB_SECRET_NAME")`. For shared notebooks, the "attached runtime" mode connects Livebook to a running IEx node — treat it as a production console and audit.
- **Mix releases (not Distillery):** `mix release` is the one true path since Elixir 1.9. `releases:` block in `mix.exs` declares name, applications, config_providers. Runtime config in `config/runtime.exs` (executed on boot, reads env vars). For container deploys: `MIX_ENV=prod mix release --overwrite` in a builder stage, copy `_build/prod/rel/<name>/` into a minimal runtime image. `bin/<name> start` foreground, `bin/<name> daemon` background, `bin/<name> remote` attach IEx to running node. Distillery is EOL — if you see it, migrate.
- **Observability:** `:telemetry` is the standard event bus (no custom logging DSLs). Instrument with `:telemetry.execute([:my_app, :req, :stop], %{duration: t}, meta)`. `opentelemetry_*` libs export to OTLP-compatible collectors. `:recon` + `:observer_cli` for prod node introspection (pair with `Node.connect/1` from an ops bastion). Log structured JSON via `:logger_formatter_json` in prod — plaintext logger output is for dev only.
- **Gleam interop:** Gleam compiles to BEAM modules with snake_case names. From Elixir: `:my_gleam_module.function(arg)` — note the atom form, no `GleamModule` alias. Type errors at the Gleam boundary are compile-time on Gleam's side, runtime on Elixir's — wrap Gleam calls in an Elixir boundary module that pattern-matches `{:ok, _}` / `{:error, _}` instead of passing raw returns upward. For a Gleam-first repo using Elixir as a library, the inverse applies: Elixir modules appear as `@external_type` in Gleam and are `unsafe` — keep a thin typed shim.
- **Elixir 1.18 / OTP 27 specifics:** `Kernel.tap/2` and `Kernel.then/2` idiomatic in pipelines since 1.12 (don't rewrite to `with` chains unnecessarily). `mix format --check-formatted` required in CI. `@doc since:` + `@deprecated` markers drive `mix xref` deprecation reports. OTP 27 brings `:persistent_term` enhancements + `:maps.iterator/1` ordering guarantees — don't code around old OTP 25 quirks if the project pins 27+.
- **Verification gate:** `mix compile --warnings-as-errors` + `mix test --cover` (target >=80% on lib, >=60% overall) + `mix format --check-formatted` + `mix credo --strict` + `mix dialyzer` (PLT cached in CI) + Nerves: `MIX_TARGET=<target> mix firmware` produces `.fw` artifact + Ash: `mix ash.codegen --check` (no pending migrations) + Release: `MIX_ENV=prod mix release --overwrite && _build/prod/rel/<name>/bin/<name> eval ':ok'`. All must exit 0. For Nerves on CI without real hardware, `MIX_TARGET=host` for unit tests + a cross-compile smoke on one target; full burn-and-boot is out of scope for pre-merge.

### Fragile-language note (Ley 25 context)

Nerves on production firmware is the strongest BEAM-wins-everything case: supervision trees + hot code reload + `:heart` watchdog + A/B partitions give you fault containment down to the hardware. If a fragility score lands >=4 AND the target is embedded/IoT, Nerves is the default — there is no comparable stack (Rust-for-Linux is close but lacks OTP's supervisor ergonomics). Ash is orthogonal to fragility — it's a framework choice for domain-modeling velocity, not a runtime property; an Ash app and a plain Phoenix app have identical fragility characteristics modulo team familiarity. Livebook is dev-tooling; never serve production traffic from it.

### Elixir 1.18 / Gleam 1.x migration note

Elixir 1.18 added set-theoretic type inference (opt-in, non-breaking) — expect noisier `@spec` mismatch output. Don't bundle "fix all new type warnings" with feature work; dedicated PR. Gleam hit 1.0 in early 2024 and is stable; 1.x minor versions are additive, but `rebar3_gleam` tooling changed in 1.5 — pin `rebar.config` explicitly and commit `rebar.lock`. Do NOT migrate Elixir modules to Gleam for type-safety reasons unless the team has Gleam on-call; the type win is real but the Hex ecosystem + IDE tooling gap is real too.
