===============================================================================================
PART I — THE OBSERVABILITY OBJECT: WHY CROSS-SYSTEM VISIBILITY IS UNOWNED
===============================================================================================

1.1 The founding observation. The Claude Power Pack estate, as OBSERVED across the eighteen sealed
dataset domains and seventy-plus executable modules the Non-Duplication Ledger inventories, contains
extensive PER-SYSTEM telemetry: the Cognitive OS (CO) tracks session cost and adoption of its own
recommendations (CO-12); the Graphify Kernel (GK) tracks knowledge-node freshness, traversal, and
navigation health (GK-09); the recall_roi mechanism tracks how often one distilled asset is reused;
done_gate verifies that a single deliverable's completion criteria were actually met; liveness
tracks whether a pane or session is still active. Each of these is a real, working telemetry
surface. None of them, individually or summed, answers the question this dataset exists to answer:
of the seventy-plus systems the estate has built, which are actually being invoked in real missions,
which have gone quietly unused since their sealing, and which real institutional work is happening
OUTSIDE every registry the estate currently maintains? (OBSERVED: GK-09's scope is the knowledge
substrate only — coordinates, nodes, freshness; CO-12's scope is one adoption axis per session — did
the Owner accept this session's recommendation; DERIVED: neither substrate the two systems observe
recall_roi/done_gate/liveness — the SYSTEM as the unit of observation, only KNOWLEDGE, ADOPTION,
ASSET-REUSE, COMPLETION, and ALIVENESS respectively.) The gap is not a missing feature of any one
telemetry surface; it is the absence of a surface whose unit of observation is the SYSTEM itself,
read across the whole estate at once.

1.2 Why the gap cannot be closed by extending an existing telemetry surface. The natural reflex, when
an institution notices it cannot see its own systems, is to extend the telemetry surface that already
exists and is closest to the need — in this case, GK-09, because "knowledge navigation" sounds
adjacent to "system visibility." That reflex fails on inspection for the same structural reason
IAS-F1's own founding argument (Part I §1.2 of that dataset) rejects building a "seventy-first system"
without a shared ontology first: GK-09's object is the knowledge substrate's own health, and every
field, index, and traversal record it maintains is keyed to a coordinate, not to a system. Extending
GK-09 to also track system usage would either (a) bolt an unrelated schema onto a substrate that was
never built to carry it, producing the exact kind of awkward, over-general telemetry IAS-F1's Anti-
Pattern 1 warns against, or (b) require GK-09 to become, in effect, a second and competing
observability system under a knowledge-navigation name — which the Non-Duplication Ledger's own
Family F row (candidates 51-60) explicitly forecloses by routing the observability candidates to a
dedicated BUILD verdict rather than a MERGE into GK-09. (NORMATIVE: the correct move, as with F1, is
a dedicated system whose object is the thing no existing system owns, not a graft onto the nearest
neighbor.)

1.3 The precise boundary against GK-09, restated with a worked contrast. Consider two questions an
Owner might ask on the same afternoon. Question A: "is the pricing-strategy knowledge node stale, and
has anyone traversed it this month?" That is a GK-09 question — its object is a coordinate, its answer
comes from GK-09's freshness and traversal fields, and E1 does not answer it, does not duplicate the
answer, and does not maintain a parallel index of knowledge-node freshness. Question B: "of the
seventy-plus systems we have built, which ones has any mission actually invoked in the last quarter,
and which have zero recorded invocations despite being fully built and documented?" That is an E1
question — its object is the SYSTEM, not the knowledge node, and no existing telemetry surface,
including GK-09, can answer it, because GK-09's traversal counter is scoped to knowledge nodes and
has no field for "was this SYSTEM invoked." (DERIVED: the two questions are not the same question
asked two ways; they have different objects under IAS-F1's formal level test, Part VIII §8.2 — Question
A's object is a resource GK owns, Question B's object is the set of systems, an ensemble-level object
no resource-level system owns.) This contrast is the operational form of the distinct-object criterion
IAS-F1 Part II §2.2 supplies, applied specifically and by name to the E1-versus-GK-09 boundary the
ledger requires E1 to prove.

1.4 The precise boundary against CO-12, restated with a worked contrast. CO-12 is the estate's gap
radar: given a specific recommendation the Cognitive OS or a sibling system has surfaced to the
Owner, CO-12 tracks whether that ONE recommendation was adopted, on ONE session, along ONE axis
(adopted / not adopted / partially adopted). It is, in IAS-F1's vocabulary, a resource-level
capability whose object is a single decision's adoption status — a session-scoped fact, not a
system-scoped one. E1's usage observatory (Part III of this dataset) asks a structurally different
question: not "was this recommendation adopted," but "was this SYSTEM invoked at all, by any
mission, in the observation window" — a question with no adoption semantics, no single-session scope,
and no dependency on a recommendation having been surfaced in the first place. A system can be
heavily used with zero CO-12 adoption events (a background gate that runs automatically, never
surfaced as a recommendation) and a recommendation can be perfectly adopted (CO-12 = 100%) while the
system that produced it is invoked exactly once, in the session where the adoption was recorded, and
never again — the two measurements are ORTHOGONAL, not overlapping, which is the precise sense in
which E1 does not duplicate CO-12's object even though both are, loosely, "usage" telemetry. (DERIVED:
orthogonality of the measured axis, not mere topical proximity, is the test the Non-Duplication Ledger
applies; two systems that both use the word "usage" are not thereby duplicates.)

1.5 What the observability object actually is, defined precisely. The OBSERVABILITY OBJECT, DEFINED
in this corpus, is the time-indexed, evidenced record of: (a) which systems in the ensemble were
invoked, by which consuming system or mission, in a given observation window (the usage facet, Part
III); (b) which systems show zero invocation across a sustained window despite being live and
addressable (the dead-capability facet, Part IV); (c) which real institutional work occurred with no
corresponding entry in any system's registry at all (the dark-matter facet, Part V); (d) how
knowledge, decisions, evidence, and resources actually MOVE along the typed relation graph IAS-F1
Part VI defines, as opposed to how the estate's documentation claims they move (the flow facets,
Parts VI-IX); (e) how long each such movement takes end-to-end (the latency facet, Part X); (f) where
the graph's actual throughput concentrates or stalls (the bottleneck facet, Part XI); (g) the causal
chain behind a specific decision, traced back through the systems that fed it (the provenance facet,
Part XII); (h) how much of the estate's distilled knowledge is actually consumed downstream versus
merely produced (the consumption facet, Part XIII); and (i) a composite read of the ensemble's
operating health synthesizing (a) through (h) (Part XIV). (DEFINED: this nine-facet object is the
complete scope of IAS-E1; NORMATIVE: any candidate facet not reducible to one of these nine, on
inspection, either belongs to a sibling system or is out of scope for this dataset.)

1.6 Why observability must come before allocation, and why E1 sits where the Compounding Graph places
it. The COMPOUNDING_GRAPH.md primary loop begins at IAS-E1 and flows into IAS-C1 (Capability
Portfolio), because an allocation decision made without knowing which systems are actually earning
their keep is not allocation — it is guessing dressed in portfolio language. (DERIVED from the
Compounding Graph's own stated loop: "better observation → better allocation → better routing → better
outcomes → better fitness → better architecture → better allocation.") This ordering is not
incidental; it mirrors IAS-F1's own foundational placement (an ontology before a mechanism) at one
level down the stack — E1 is not a foundation in F1's sense, but it is a PRECONDITION for every
economics and ecology dataset in the family, because IAS-C1's portfolio math, IAS-D1's fitness
function, and IAS-G1's complexity budget all require a real usage record as an input, and none of
the three builds that record itself (each explicitly reads it from E1, per the Strengthening Matrix
and Compounding Graph edges). A federation that tried to allocate capital before it could observe
usage would be repeating, at the economics layer, the exact error IAS-F1 diagnosed at the
vocabulary layer: building a coordinator before the substrate the coordinator needs exists.

1.7 Why this is not merely "more logging." A skeptical reading might reduce E1 to "the estate should
log more," which understates the object by conflating raw event capture with the evidenced, typed,
cross-system RECORD this dataset defines. Any individual system can, and many already do, emit logs
of their own invocations — that is first-order telemetry, scoped to one system's own resource (its
own invocation count), and it is not what E1 is. E1's object is the SYNTHESIS of those per-system
logs (where they exist) plus the DETECTION of activity that occurred with no log at all (dark
matter, Part V) into one cross-system record with typed relations, evidence levels, and staleness
markers inherited from IAS-F1's property model (Part IV of that dataset). Logging is a first-order
capability whose object is one system's own events; observability, in this dataset's sense, is a
second-order capability under IAS-F1's Part VII §7.2 test, because its direct object is the
invocation behavior of OTHER systems and capabilities, synthesized and evaluated, not merely
recorded. (DERIVED, applying IAS-F1's own mechanical second-order test to E1 as a worked instance.)

FINAL LAW — PART I. No resource-level telemetry surface — not knowledge navigation, not adoption
tracking, not asset reuse, not completion verification, not liveness — can answer "which of our
seventy-plus systems are actually earning their keep," because each is scoped to its own resource;
the observability fabric exists because that question has an ensemble-level object, and an
ensemble-level object, per the family's own admission test, cannot be answered by summing
resource-level answers, only by building the fabric that reads the ensemble directly.

PART II — ONTOLOGY RESOLUTION: SYSTEMS, CAPABILITIES, AND FLOWS RESOLVED AGAINST IAS-F1
===============================================================================================

2.1 Why E1 resolves against F1 before defining a single engine. Per IAS-F1's own closing discipline
(Part XVIII of that dataset, "a foundation dataset is proven... by whether independently-authored
siblings can resolve their objects against it without inventing a single word of their own"), this
Part discharges that obligation explicitly for E1, naming every F1 term this dataset consumes and
confirming zero undefined terms remain before Part III begins building mechanism. (NORMATIVE: this
is not throat-clearing; a sibling that skips this resolution and invents its own vocabulary for
"system," "capability," or "relation" is the precise failure IAS-F1 Anti-Pattern 1 names.)

2.2 SYSTEM, SUBSYSTEM, ENGINE, CAPABILITY — consumed verbatim. E1 uses IAS-F1's Part II definitions
without modification: a system is the four-conjunctive-test object (distinct object, authority,
interface, addressability); a capability is a nameable, provider-independent unit of what a system
can do; a subsystem is a facet of a parent's object with delegated authority; an engine is the
executable core (introduced in F1 Part III, consumed here for E1's own architecture in Part XVI).
E1's usage observatory (Part III) counts invocations at the SYSTEM granularity by default, because
that is the granularity IAS-F1's addressing model (Part IX of that dataset) guarantees is stable
across refactors — counting at the file-path or component granularity would break every time a
system is reorganized internally, which is precisely the failure IAS-F1 Part IX §9.1 warns the
address triple exists to prevent.

2.3 The typed relation graph — the substrate E1 actually reads. E1's single most load-bearing
dependency on IAS-F1 is the six-type relation graph of that dataset's Part VI: FEEDS, DEPENDS-ON,
COMPETES-WITH, DUPLICATES, STRENGTHENS, CONSUMES. Every one of E1's nine facets (Part I §1.5) is,
mechanically, an operation over this graph's edges, not over the bare system list. The usage
observatory (Part III) counts CONSUMES edges (the volatile, mission-time relation type F1 Part VI
§6.7 defines) within an observation window. The dead-capability detector (Part IV) looks for
systems with zero inbound CONSUMES edges across a sustained window despite having outbound FEEDS or
STRENGTHENS edges declared in their own registry entry — a mismatch between DECLARED capability and
OBSERVED invocation. The flow observatory (Parts VI-IX) traces FEEDS and DEPENDS-ON edges by
payload type (knowledge, decision, evidence, resource). The bottleneck map (Part XI) reads edge
DENSITY and edge LATENCY together across the whole graph. The provenance tracker (Part XII) walks
DEPENDS-ON and FEEDS edges backward from a decision to its causal inputs. (DERIVED: E1 introduces no
new relation type; it introduces measurement OVER the six types F1 already defines, which is the
precise, checkable sense in which E1 strengthens rather than replaces F1's graph.)

2.4 The property model — consumed for evidence levels and staleness, extended for one field. E1
consumes IAS-F1's Part IV property model (cost, evidence level, reuse horizon, model-dependence,
consumers, authority) as the record every system in the ensemble already carries. E1 reads the
CONSUMERS field (F1 Part IV §4.2: "the set, and the count, of other systems or mission types that
depend on this system's output... computed from the typed graph") and is, in fact, the specific
mechanism F1 names but does not build for computing that field honestly — F1 Part IV §4.6 names the
DOUBLE-BOOKED CONSUMER failure and resolves it by defining CONSUMERS as "an observed dependency... not
a claimed audience, falsifiable against the Control Bus's event log (IAS-A2)." E1 is the second half
of that falsifiability claim: where IAS-A2 carries the event log, E1 is the system that reads it,
synthesizes it into the CONSUMERS count, and detects when a system's self-reported consumer list
diverges from the observed one. (DERIVED, naming the specific F1 forward-reference E1 discharges —
this is the concrete evidence, per IAS-F1 Part XVIII's own closing test, that F1's anticipation of
E1's needs was accurate.) E1 extends the property model by exactly one new field, LAST-OBSERVED-
INVOCATION (a timestamp, DEFINED here, of the most recent CONSUMES edge recorded against a system),
which is additive under F1's own Part XVII versioning discipline (an added field, not a redefinition
of an existing one) and is the raw material Part IV's dead-capability detector reduces to a verdict.

2.5 Resolving "evidence" and "claimed-until-observed" — inherited, not reinvented. E1 inherits IAS-F1's
Part III §3.3 discipline wholesale: an observability record is claimed state until it is reconciled
against the disk (or, for E1 specifically, against the Control Bus's event log and the estate's own
file/registry system). This matters concretely for E1 because the fabric's entire value proposition —
telling the truth about usage — collapses if the fabric itself reports unreconciled claims as fact.
Every count Part III through Part XIV of this dataset produces carries the SAME evidence-level
vocabulary (OBSERVED / DERIVED / HYPOTHESIZED / UNRESOLVED) IAS-F1 Article V mandates estate-wide, and
Part XV of this dataset (the Telemetry-Before-Claims Contract) is the operational rule that makes this
non-optional rather than aspirational.

2.6 What E1 does NOT resolve against F1, stated as a boundary. E1 does not consume F1's authority
model (Part X of that dataset) as a decision-making mechanism — E1 has no final authority over any
decision class; every count and flag it produces is advisory input into IAS-C1, IAS-D1, and IAS-G1's
own decisions (Part XXII of this dataset specifies these feeds precisely). E1 does not consume F1's
lifecycle model (Part XI of that dataset) to manage its own system's lifecycle beyond noting its own
addressability; lifecycle management of the OBSERVED systems belongs to whichever system holds
authority over admitting, retiring, or merging them (D2A, IAS-D1), not to the fabric that merely
reports their state. (NORMATIVE: E1 reads the ensemble; it does not govern it, an application of the
same restraint IAS-F1 Part I §1.2 models for itself relative to the seventy systems it defines
vocabulary for.)

2.7 Resolving the "second-order" question for E1 itself, using F1's own mechanical test. Applying
IAS-F1 Part VII §7.2's test directly: what does E1's usage-observatory capability act upon? Not a
resource-level object (a session, a pane, a coordinate) but the invocation behavior of OTHER
systems and capabilities — the CONSUMES, FEEDS, and DEPENDS-ON edges those other systems generate in
the course of their own operation. The answer is unambiguous where F1's own self-classification (F1
Part VII §7.6) was CONTESTED: E1's object recurses onto other capabilities cleanly, satisfying the
second-order test without the edge-case ambiguity F1 itself carries, because E1, unlike F1, is not a
vocabulary but an operational reader of other systems' behavior. (DERIVED, applying F1's mechanical
test as a worked, non-foundation-dataset instance — the retrodiction protocol F1 Part XVI §16.3
anticipates being run against exactly this kind of case.)

2.8 The Aggregator Near-Miss check, applied to E1 honestly. IAS-F1 Part VII §7.7 names the AGGREGATOR
NEAR-MISS: a capability that collects output from many systems and presents it as a report is
first-order, because its produced object is a display, not a decision over other capabilities. E1
must clear this bar honestly rather than assert it does. The usage observatory and flow observatory
facets (Parts III, VI-IX), read in isolation, risk landing on the aggregator side of the line — a
usage COUNT, by itself, is a report. E1 clears the bar specifically because its dead-capability
detector (Part IV) and dark-matter detector (Part V) do not merely report; they FLAG systems as
candidates for a decision (retirement review, registry correction) that a consuming system (IAS-D1,
D2A) then acts on, and the health observatory (Part XIV) synthesizes the facets into a composite
verdict, not a bare display. (HONEST ADMISSION, per Article V: the pure count-producing sub-facets of
E1, taken alone, would be first-order aggregation; E1's second-order status rests on the flagging and
synthesis layer, not on the raw counting layer, and this dataset states that dependency explicitly
rather than claiming second-order status for every sub-facet uniformly.)

FINAL LAW — PART II. A dataset that measures the ensemble must first prove it is speaking the
ensemble's own language; E1 resolves every term it uses against IAS-F1's ontology before building a
single engine, and where its own second-order status depends on the flagging layer rather than the
counting layer beneath it, it says so plainly rather than borrowing second-order status by
association with its own siblings.

PART III — THE CAPABILITY USAGE OBSERVATORY
===============================================================================================

3.1 The core question and why it has been unanswerable until now. "Of the seventy-plus systems the
estate has built, which have actually been invoked, by what, and how often, in a given observation
window?" is, as established in Part I, a question no existing resource-level telemetry surface can
answer, because each is scoped to its own resource rather than to the SYSTEM as the unit of count.
The USAGE OBSERVATORY, DEFINED here as E1's first and most foundational engine, closes this gap by
maintaining one record per addressable system (per IAS-F1's Part IX addressing model) carrying an
invocation count, a first-observed-invocation timestamp, a last-observed-invocation timestamp (Part
II §2.4's new property field), and the identity of each consuming system or mission type, sourced
from CONSUMES edges recorded on the Control Bus (IAS-A2) plus, where the Control Bus has no record,
from a secondary reconciliation pass against file-modification and execution-log evidence the estate
already produces incidentally (specified fully in Part V's dark-matter detection, since a CONSUMES
edge with no Control Bus record and no secondary evidence either is, definitionally, invisible to the
observatory and cannot be claimed as observed).

3.2 The two data sources and why neither alone suffices. The observatory's primary source is the
Control Bus's event log (IAS-A2), which records CONSUMES edges at mission time as a structural
byproduct of routing (IAS-A1) and cross-system event delivery. This source is HIGH-PRECISION where
present (an actual routing decision was made and logged) but INCOMPLETE by construction, because not
every invocation of every system necessarily passes through IAS-A2's bus — a system invoked directly
by a human operator, by a hook, or by a script outside any routed mission leaves no bus event. The
observatory's secondary source is INCIDENTAL EVIDENCE: file-modification timestamps on a system's own
output artifacts, log lines the system itself emits, or a downstream artifact whose provenance chain
(Part XII) names the system as a contributor. This source is LOWER-PRECISION (it infers invocation
rather than observing a routing decision directly) but catches invocations the bus misses. (DERIVED:
neither source alone satisfies the evidence-honesty discipline of IAS-F1 Part IV §4.6 — a
bus-only observatory would systematically undercount systems invoked outside routed missions, exactly
the population the dark-matter detector of Part V exists to surface, while an incidental-evidence-only
observatory would be too noisy to trust for allocation decisions. The observatory therefore reports
BOTH counts side by side, per-system, each carrying its own evidence level — OBSERVED for bus-sourced
counts, DERIVED for incidental-evidence-sourced counts — rather than collapsing them into one number
that hides which kind of evidence backs it.)

3.3 The usage record schema. Each system's usage record, DEFINED here, carries: STABLE ID (from
IAS-F1 Part IX, the addressing anchor); OBSERVATION WINDOW (the time range the record covers, never
"all time" by default, because an all-time count silently favors systems built early and penalizes
recently-sealed ones that have not yet had a chance to accumulate invocations — a specific
countermetric named formally in Part XX); BUS-SOURCED INVOCATION COUNT with evidence level OBSERVED;
INCIDENTAL-EVIDENCE INVOCATION COUNT with evidence level DERIVED; CONSUMER LIST (the distinct set of
systems or mission types recorded as invokers, cross-checked against the system's own self-reported
CONSUMERS property per Part II §2.4, flagging any divergence); LAST-OBSERVED-INVOCATION timestamp; and
a STALENESS flag inherited from IAS-F1 Part IV §4.3 (a record not reconciled within its declared
cadence is flagged stale rather than silently trusted).

3.4 Worked example on synthetic estate data. Consider four synthetic systems, illustrative only and
not asserting real measurements of any actual PP module, observed over a ninety-day synthetic window.
System "Sigma-11" shows a bus-sourced count of 340 invocations from six distinct consuming mission
types, a last-observed-invocation of two days before window close, and zero divergence between its
self-reported CONSUMERS list and the observed consumer list — a clean, high-confidence, actively-used
system. System "Sigma-14" shows a bus-sourced count of zero but an incidental-evidence count of
twelve, all inferred from log lines it emits directly when a human operator invokes it via a hook
outside the routed mission path — a real but bus-invisible usage pattern, correctly reported as
DERIVED rather than OBSERVED, and correctly NOT flagged dead despite its zero bus count, because the
observatory checks both sources before any downstream verdict (Part IV) is rendered. System "Sigma-19"
shows a bus-sourced count of 4, all from a single consuming mission type, while its own registry entry
self-reports eleven declared consumers — a nine-consumer divergence the observatory flags for review
rather than silently trusting the self-report, per Part II §2.4's double-booked-consumer resolution.
System "Sigma-22" shows zero on both sources across the full ninety-day window despite being fully
built, documented, and addressable — the observatory's output for Sigma-22 is not itself a verdict
("Sigma-22 is dead") but the raw evidence Part IV's dead-capability detector consumes to render that
verdict formally, after an additional check this Part's own edge-case discussion (§3.6) requires.

3.5 Aggregation to the ensemble level. Beyond the per-system record, the observatory produces one
ensemble-wide summary per observation window: the USAGE DISTRIBUTION, a ranked list of all addressable
systems by combined invocation count, with the explicit caveat (NORMATIVE) that rank alone is not a
value judgment — a rarely-invoked system that is DEPENDS-ON (not merely FEEDS) for a critical mission
type may carry more institutional weight per invocation than a frequently-invoked but easily-
substitutable one, a distinction IAS-D1's fitness function, not E1, is responsible for weighing. The
observatory also produces the CONCENTRATION metric: what share of total invocations the top decile of
systems accounts for, which is the raw signal IAS-C1's portfolio math and IAS-D1's biodiversity
judgment (per IAS-F1 Part XV) both consume as an input, without E1 itself rendering a verdict on
whether high concentration is healthy specialization or dangerous monoculture.

3.6 The window-boundary edge case. A system sealed three days before an observation window's close
will show near-zero invocations regardless of its eventual real usage, simply because it has not had
time to accumulate consumers — the NEW-SYSTEM EDGE CASE. The observatory's resolution, DERIVED from
the same principle IAS-F1 Part IV §4.6 applies to unmeasured properties, is to mark such a system's
usage record UNRESOLVED rather than folding it into the zero-usage population the dead-capability
detector screens, with the resolution threshold set at the system's own declared minimum-adoption-
horizon (a field this dataset borrows from IAS-F1's reuse-horizon property, Part IV §4.2, read at
system-admission time) rather than a single estate-wide constant, because different system types have
genuinely different natural ramp-up periods and a single constant would either falsely flag fast-
ramping systems or falsely protect slow-ramping ones past their honest grace period.

3.7 Integration point, stated for forward reference. The usage observatory's output — per-system
records plus the ensemble-wide distribution and concentration metrics — is the primary input Part IV's
dead-capability detector and Part V's dark-matter detector consume; it is also the first of the
Compounding Graph's stated E1-to-C1 edges ("usage / dead-capability / dark-matter" feeding "allocation,
extinction, topology"), specified fully in Part XXII.

FINAL LAW — PART III. A system's usage is not what its own documentation claims nor what its
registry entry self-reports; it is the count of CONSUMES edges actually observed on the bus plus the
incidental evidence the bus misses, each honestly labeled by the evidence level that backs it — and
an observatory that collapses two different kinds of evidence into one confident number has already
begun the drift toward observability theater Part XIX exists to name.

PART IV — THE DEAD-CAPABILITY DETECTOR
===============================================================================================

4.1 From raw usage to a verdict, and why the gap between them is deliberate. Part III's observatory
produces evidence; it renders no verdict, by design (§3.4's closing line). The DEAD-CAPABILITY
DETECTOR, DEFINED here as E1's second engine, is the mechanism that converts a usage record into one
of three verdicts — LIVE, DORMANT, or DEAD — and it is built as a separate engine from the observatory
specifically because the conversion from evidence to verdict is where an observability fabric is most
tempted to cut corners, and separating the two makes the verdict logic auditable independent of the
counting logic beneath it. (NORMATIVE: this separation mirrors IAS-F1's own discipline of keeping
definitions, Parts II-IX, structurally separate from invariants, Part V — a fact and a judgment about
the fact are different objects, and collapsing them loses the ability to audit either independently.)

4.2 The three-verdict taxonomy, defined precisely. LIVE: a system whose usage record (Part III) shows
at least one invocation, from either source, within the CURRENT reconciliation window (the shortest
window IAS-F1 Part IV §4.3 staleness discipline recognizes as fresh for that system's declared
reconciliation cadence), with no unresolved consumer divergence exceeding the threshold Part III §3.4
names. DORMANT: a system with zero invocations in the current window but at least one invocation in
the PRIOR window, or a system correctly marked UNRESOLVED under the new-system edge case (§3.6) whose
grace period has not yet elapsed — dormancy is a holding state, not a verdict of failure, and exists
specifically so the detector does not conflate "unused this quarter" with "unused ever." DEAD: a
system with zero invocations, from either source, across TWO OR MORE consecutive full reconciliation
windows, past its new-system grace period, with no incidental evidence of any kind (Part III §3.2)
supporting even a low-confidence inference of use. (DEFINED: the two-window requirement is a
deliberate NORMATIVE choice, not a measured constant — a single-window zero-count is common for
seasonal or infrequently-needed systems, discussed as an edge case in §4.5, and a detector that
declared DEAD on one window's silence would generate false positives at a rate this dataset judges
unacceptable for a verdict feeding retirement decisions.)

4.3 Why DEAD is an input to retirement, never a retirement decision itself. The detector's output is
explicitly advisory, per Part II §2.6's authority boundary: E1 holds no final authority over any
system's continued existence. A DEAD verdict is a flag consumed by IAS-D1's ecology (which judges
whether the dead system should be retired, per D2A's RETIRE disposition, or is dormant for a
structural reason IAS-D1's fitness-landscape analysis — Part XV of that dataset's own domain —
recognizes, such as a system built for a mission type that has not recurred but is expected to) and
by D2A's own decision process for the specific act of retirement. (DERIVED from Part II §2.6 and the
Compounding Graph's stated edge: "IAS-E1 usage / dead-capability / dark-matter → IAS-C1, IAS-D1,
IAS-G1 → allocation, extinction, topology" — E1 produces the flag; IAS-D1 and D2A act on it.) A
detector that auto-retired systems on its own DEAD verdict would violate IAS-F1's authority invariant
(Invariant Three, singular authority per decision class) by claiming final authority over the
retirement decision class, which the ledger and matrix both assign elsewhere.

4.4 Worked example: rendering a verdict on the four synthetic systems from Part III. Continuing the
worked example of §3.4: Sigma-11 (340 bus-sourced invocations, current window) renders LIVE, cleanly.
Sigma-14 (zero bus-sourced, twelve incidental-evidence invocations, current window) also renders LIVE
— the detector reads BOTH sources per §4.2's "either source" clause, which is the specific mechanism
that prevents the detector from mistaking a bus-invisible-but-real usage pattern for death, a failure
mode this dataset names explicitly in Part XIX as a near-miss of observability theater working in the
opposite, false-negative direction. Sigma-19 (4 bus-sourced invocations, nine-consumer registry
divergence) renders LIVE but CARRIES A FLAG — the detector does not downgrade a system's liveness
verdict because of a consumer-count divergence; it renders the liveness verdict on invocation
evidence alone and surfaces the divergence as a SEPARATE finding for the property-honesty invariant
(IAS-F1 Invariant Five) to consume, keeping the two judgments (is it used / is its self-report
accurate) independent, which is the same separation-of-fact-from-judgment discipline named in §4.1
applied one layer deeper. Sigma-22 (zero on both sources, full window, past grace period) renders
DEAD, and only Sigma-22, of the four synthetic systems, is passed to IAS-D1 as a retirement-review
candidate.

4.5 Edge case: the seasonal or event-triggered system. A system built specifically to handle an
infrequent but recurring institutional event — a quarterly audit, an annual compliance sweep, a
once-per-model-generation migration helper — will show a genuine, correct pattern of long DEAD-looking
silences punctuated by bursts of heavy use, and a naive two-window rule would flag it DEAD in the
silent windows despite it being exactly as designed. The detector's resolution, DERIVED from the
property model's REUSE HORIZON field (IAS-F1 Part IV §4.2), is to read a system's declared EXPECTED
INVOCATION CADENCE (a sub-field of reuse horizon this dataset specifies as an additive extension,
consistent with Part II §2.4's precedent) before applying the two-window rule, and to scale the
window count to the declared cadence rather than to a fixed calendar period — a system with a
declared quarterly cadence is evaluated over quarters, not over the estate's default reconciliation
window, and is not flagged DEAD until it has missed TWO declared cadence cycles, not two arbitrary
observation windows. Where a system has no declared cadence at all (the common case), the default
two-reconciliation-window rule of §4.2 applies, and the absence of a declared cadence is itself an
UNRESOLVED property the detector surfaces rather than silently assuming a default cadence.

4.6 Adversarial case: gaming the detector with a heartbeat invocation. A system's own maintainer,
aware that a DEAD verdict triggers a retirement-review flag, could arrange for the system to invoke
itself, or to be invoked by a trivial, content-free mission, purely to keep its usage record non-zero
— a HEARTBEAT-GAMING pattern. The detector's defense is structural rather than purely statistical: the
usage record (Part III §3.3) carries the CONSUMER LIST, not merely a count, and a system whose entire
observed consumer list, across the full observation history, consists of one recurring, structurally
identical, content-free invocation pattern (same consuming mission type, same payload shape, no
downstream FEEDS edge produced from the invocation) is flagged not LIVE-clean but LIVE-SUSPECT, a
fourth de facto category surfaced to IAS-D1 alongside the three formal verdicts, because a system kept
alive by a heartbeat that produces no real downstream effect is functionally dead even though it is
technically invoked — DERIVED directly from the second-order capability test's own spirit (IAS-F1 Part
VII): an invocation that produces no output any other system consumes has not exercised the system's
actual capability, only its address.

4.7 The countermetric this Part exists to protect against, named forward. A dead-capability detector
that is trusted uncritically creates a perverse incentive for institutional actors to invoke systems
purely to avoid a DEAD flag, which is precisely the heartbeat-gaming pattern of §4.6 generalized from
an adversarial case to a systemic risk if the detector's output is treated as a hard performance
score rather than an evidenced flag. Part XX formalizes the countermetric (LIVE-SUSPECT rate as a
share of total LIVE verdicts) that makes this gaming pattern visible at the ensemble level rather than
only catchable case-by-case.

FINAL LAW — PART IV. Dead is not zero usage; it is zero usage past a cadence-aware grace period with
no incidental evidence and no declared seasonal cause, and even then the detector's output is a flag
for another system's decision, never a retirement — because an observability fabric that could kill
what it merely failed to see using would have converted itself, silently, from a witness into a judge.

===============================================================================================
PART V — THE INSTITUTIONAL DARK-MATTER DETECTOR
===============================================================================================

5.1 The inverse problem, and why it is harder than dead-capability detection. Part IV solves: "this
system is registered; is it used?" This Part solves the structurally harder inverse: "real
institutional work is happening; is there a registered system that accounts for it at all?" DARK
MATTER, DEFINED in this corpus by direct analogy to the astrophysical term the estate's own prior
documentation has used informally, is real, recurring institutional work — decisions made, knowledge
produced, missions completed — for which no system in any registry claims authorship, meaning the
work is invisible to every allocation, ecology, and architecture decision the family's other datasets
make, not because the work doesn't matter but because it has never been named. This is structurally
harder than dead-capability detection because Part IV starts from a known, addressable system and asks
whether it was used, while this Part must start from OBSERVED WORK PRODUCT with no known system to
check against — a search problem, not a lookup problem.

5.2 The three dark-matter signatures. Dark matter manifests in three recurring shapes, each requiring
a different detection method. UNREGISTERED RECURRING PATTERN: the same sequence of actions,
recognizable by structural similarity (same input shape, same output shape, same triggering
condition) recurs across multiple missions or sessions with no named system producing it — typically a
manual workaround, a copy-pasted prompt fragment, or an ad hoc script an operator has quietly re-run
enough times that it has become, in effect, an unregistered system. ORPHANED ARTIFACT TRAIL: a class
of output artifact (a file shape, a document type, a decision record) appears repeatedly in the
estate's history with no FEEDS or CONSUMES edge in the Control Bus's log connecting it to any
registered system's declared outputs — the artifacts exist, are consumed downstream by real
missions, but their producing mechanism has never been formally registered. SHADOW CAPABILITY: a
registered system's OWN documentation or self-report understates what it actually does — it performs
a genuinely distinct additional capability, invoked regularly, that was never separately named,
meaning the system is, in effect, two capabilities wearing one registry entry, and the second
capability is dark matter hiding inside a lit system.

5.3 Detection method one: pattern-recurrence mining over the incidental-evidence stream. The primary
detection mechanism reuses Part III §3.2's incidental-evidence source (file-modification patterns,
log lines, artifact provenance) but inverts its purpose: instead of matching evidence to a KNOWN
system's stable ID, the detector clusters unmatched evidence by structural similarity (same input
shape, same output shape, similar timing pattern relative to mission boundaries) and flags any
cluster exceeding a recurrence threshold as an UNREGISTERED RECURRING PATTERN candidate. (DERIVED:
this reuses the observatory's own secondary source rather than building a new evidence pipeline, per
Article III's strengthen-never-replace discipline applied to E1's own internal engines, not only to
its sibling systems.) The recurrence threshold is deliberately conservative — a pattern must recur at
least a stated minimum count within a stated minimum span before being flagged, because a single
one-off improvisation is not dark matter, it is normal ad hoc problem-solving, and a detector that
flagged every non-repeating action as dark matter would drown its own genuine findings in noise.

5.4 Detection method two: artifact-trail reconciliation against the relation graph. For the ORPHANED
ARTIFACT TRAIL signature, the detector walks the estate's actual output artifacts (documents,
decisions, distilled deposits — the estate's own produced record, read as ground truth) and attempts
to match each artifact class to a producing system via the Control Bus's FEEDS edges (IAS-F1 Part VI
§6.2). An artifact class with a nonzero, recurring population and zero matching FEEDS edges across
the full observation history is flagged as an orphaned trail. This method is DERIVED as the structural
inverse of Part III's usage observatory: where the observatory asks "does this registered system have
observed consumers," the orphan detector asks "does this observed artifact have a registered
producer" — the same CONSUMES/FEEDS graph read from opposite directions.

5.5 Detection method three: capability-drift comparison for shadow capabilities. For the SHADOW
CAPABILITY signature, the detector compares a system's DECLARED capability signature (IAS-F1 Part IX
§9.2's addressing-triple component) against the OBSERVED shape of its actual invocations over the
observation window — specifically, whether the input/output pairs recorded for the system cluster
into one coherent group matching the declared capability, or into two or more distinguishable
clusters, one matching the declaration and one (or more) not. A second, statistically distinguishable
invocation cluster with no corresponding declared capability is flagged as a shadow-capability
candidate. This method carries the lowest confidence of the three (evidence level DERIVED, trending
toward HYPOTHESIZED for small sample sizes) because distinguishing a genuine second capability from
mere variance in how one capability is invoked requires enough observed volume to cluster reliably,
and the detector explicitly declines to flag shadow capabilities from thin samples rather than risk a
false positive that sends IAS-D1 chasing a capability that does not really exist.

5.6 Worked example on synthetic estate data. Consider a synthetic pattern-recurrence finding: across
eleven missions in a ninety-day window, the incidental-evidence stream shows a recurring three-step
sequence — a manual export of a specific document type, a manual reformat matching one consistent
template, and a manual re-import into a downstream system — with no registered system's declared
outputs matching the reformat step. The pattern recurs eleven times with a consistent structural
signature, exceeding the conservative recurrence threshold, and is flagged UNREGISTERED RECURRING
PATTERN with evidence level DERIVED. The flag is not itself a recommendation to build a new system —
that judgment belongs to D2A, applying the same BUILD/MERGE/SUBENGINE/EXTEND/COMPOSITION/REFERENCE
disposition vocabulary IAS-F1 imports (Part II §2.2) to this newly surfaced candidate exactly as it
would to any of the family's original 150 candidates — but it is the first time this eleven-times-
repeated manual workaround has been named at all, which is the dark-matter detector's entire
contribution: making invisible, recurring work visible enough to be judged.

5.7 Why dark matter is expected to be large relative to registered systems, and why that is not itself
alarming. HYPOTHESIZED, carried honestly as a hypothesis rather than a measured fact until Part XXI's
evaluation protocol is run: in any institution that has grown seventy-plus systems organically, the
volume of real, recurring, unregistered work is expected to exceed the volume any single dataset
audit would predict, because registration has a cost (the very Non-Duplication Ledger and Strengthening
Matrix discipline this corpus enforces) that recurring ad hoc work never had to pay. A large dark-
matter finding is not, by itself, evidence of institutional dysfunction; it is evidence that the
detector is working, and the correct institutional response — again, not E1's own decision — is
triage by D2A, not alarm.

5.8 Prioritization: not every dark-matter finding warrants equal urgency. Because §5.7 anticipates a
potentially large volume of findings relative to the estate's registered system count, this Part
specifies a simple, DERIVED prioritization signal rather than leaving every finding to compete for
D2A's attention equally: a finding's RECURRENCE RATE (how often the pattern appears per unit time,
already computed as part of the detection method itself, §5.3) combined with its RESOURCE-FLOW WEIGHT
(the aggregate cost, per Part IX, this dataset's own resource-flow observatory attributes to the
manual or workaround steps involved, where that cost is itself observable) together approximate the
finding's real institutional stakes without requiring the detector to make any qualitative judgment
about the work's importance. A high-recurrence, high-cost finding is surfaced to D2A ahead of a
low-recurrence, low-cost one, not because the latter is unimportant, but because triage capacity is
itself a scarce resource this dataset's own discipline (Part IX §9.5's cost-versus-value boundary)
requires be spent where the evidence indicates the highest expected institutional return.

5.9 The dark-matter detector's own relationship to the estate's registration cost, stated as a closing
structural observation. Named here because it clarifies why dark matter recurs at all rather than
being a rare anomaly: Part V §5.7 already observes that registration carries a real cost the
Non-Duplication Ledger and Strengthening Matrix discipline impose, while ad hoc recurring work never
had to pay that cost to exist. This asymmetry means the dark-matter detector is not merely a bug
finder; it is, in a precise sense, a continuous MEASUREMENT of how much real institutional value the
estate's own registration overhead is currently leaving unregistered — a number that, tracked across
successive windows (per the trend-reading discipline Part XIV §14.7 establishes for the Health
Observatory generally), tells D2A and IAS-D1 something genuinely new about the estate's own
registration friction, independent of any single finding's own content.

FINAL LAW — PART V. Every registered system was, at some point, unregistered work that recurred often
enough to be noticed; the dark-matter detector's only claim is that noticing should not be left to
chance, because institutional work that no registry can see is institutional work no allocation,
ecology, or architecture decision can ever account for, however real its contribution.

PART VI — THE FLOW OBSERVATORY I: KNOWLEDGE FLOW
===============================================================================================

6.1 Why flow is a distinct facet from usage. Part III's usage observatory answers "was this system
invoked;" it does not answer "what moved through the edge when it was." A FLOW, DEFINED here, is the
typed PAYLOAD carried by a FEEDS, DEPENDS-ON, or CONSUMES edge (IAS-F1 Part VI) between two systems
in the course of one invocation — and this dataset's flow observatory, split across four Parts (VI
through IX) by payload type, is what lets IAS-D1's ecology and IAS-G1's architecture see not just
that systems interact but WHAT actually crosses the boundary between them, at what volume, and
whether the declared payload type matches the observed one. This Part covers KNOWLEDGE FLOW: the
movement of distilled facts, patterns, principles, or context across system boundaries.

6.2 Why knowledge flow is E1's object and not GK's, restated precisely. The boundary drawn in Part I
§1.3 bears restating in flow-specific terms because this is the facet most likely to be mistaken for
GK-09's own object. GK-09 observes the health of the knowledge SUBSTRATE — is a coordinate fresh, has
it been traversed, is its index current. This Part observes the MOVEMENT of knowledge ACROSS SYSTEM
BOUNDARIES — when System P feeds a knowledge payload to System Q as part of a FEEDS edge, did the
payload arrive, what was its size and shape, and did Q's own behavior change as a result (a question
GK-09's coordinate-level freshness metric cannot answer, because GK-09 does not track cross-system
consumption, only node-level traversal within its own substrate). (DERIVED, restating Part I §1.3's
distinct-object argument at the flow granularity: the object here is the EDGE's payload, not the
NODE's health.)

6.3 The knowledge-flow record schema. Each observed knowledge-flow event, DEFINED here, carries:
SOURCE SYSTEM and TARGET SYSTEM (both resolved to IAS-F1 stable IDs); PAYLOAD CLASS (a coarse
typology — a distilled principle, a raw context pack, a hard-rule reference, a session-lesson excerpt
— sufficient for aggregation without requiring content-level inspection, which this dataset
deliberately avoids per the reality-contract discipline against building content classifiers this
mission is not scoped to specify); PAYLOAD SIZE (a proxy measure — token count or record count,
whichever the source system already reports, never a new measurement this dataset invents); TIMESTAMP;
and a CONSUMPTION-CONFIRMED flag, DEFINED as true only when the target system's own subsequent
behavior (a decision, an output, a further FEEDS edge) can be traced back to the payload via the
Control Bus's provenance chain — a payload that arrived but left no observable trace in the target's
subsequent behavior is recorded as DELIVERED-UNCONFIRMED, a distinction this Part treats as load-
bearing because delivery without confirmed consumption is exactly the population Part XIII's
consumption tracker exists to separate from genuine reuse.

6.4 Worked example: tracing a knowledge payload from production to confirmed use. Consider a
synthetic trace: a distillation system produces a new principle deposit, which FEEDS into a
hard-rules registry system as a candidate rule (payload class: distilled principle, size: one deposit
record). The hard-rules system's own subsequent behavior — a new rule entry citing the deposit as its
origin, appearing within the observation window — provides the provenance link the CONSUMPTION-
CONFIRMED flag requires, so this flow event is recorded CONFIRMED. A second synthetic trace: the same
distillation system produces a second deposit, which is recorded as DELIVERED to a second candidate
consumer (an evaluation system) via a FEEDS edge, but no subsequent behavior of the evaluation system
in the observation window can be traced back to it — recorded DELIVERED-UNCONFIRMED. Read together,
these two traces demonstrate the observatory's core value: a naive "knowledge produced" count would
show two units of output; the flow observatory shows one confirmed, institutionally active knowledge
flow and one delivered-but-inert one, a distinction invisible to any system that counts production
without tracing consumption.

6.5 Aggregation: the knowledge-flow density map. At the ensemble level, the observatory produces a
DENSITY MAP: for every ordered pair of systems with at least one observed knowledge-flow edge, the
confirmed-flow count, the delivered-unconfirmed count, and the ratio between them (the CONFIRMATION
RATE). A low confirmation rate on an otherwise high-volume edge is a specific, actionable finding —
it suggests the two systems are structurally connected (a FEEDS edge exists) but the knowledge moving
along it is not actually changing the target's behavior, which is a different and more subtle failure
than a dead edge (Part IV) or a missing edge (Part V): the edge is alive, but the flow along it is,
functionally, closer to noise than to institutional learning.

6.6 Edge case: knowledge flow through an intermediary with no declared FEEDS edge. Knowledge
frequently reaches its eventual consumer indirectly — System P feeds System Q, which incorporates the
knowledge into its own output, which System R then consumes, without any direct FEEDS edge ever
existing between P and R. The observatory does not synthesize a phantom P-to-R edge; per Part II
§2.3, it reads the graph IAS-F1 defines, and a transitive path through Q is recorded as two separate,
correctly-attributed flow events (P-to-Q, Q-to-R), never collapsed into a single misleading direct
edge. Where a downstream analysis (IAS-B2's insight synthesis, per the Compounding Graph) needs the
transitive path, it is the CONSUMING dataset's responsibility to walk the two-hop chain the
observatory's records make walkable, not the observatory's responsibility to pre-compute every
possible transitive closure, a scope discipline consistent with Part II §2.6's authority boundary.

6.7 Adversarial case: volume inflation through payload fragmentation. A system could inflate its
apparent knowledge-flow contribution by splitting one substantive payload into many small, separately-
logged fragments, each generating its own flow event and its own CONSUMPTION-CONFIRMED opportunity,
gaming any metric that counts flow EVENTS rather than flow SUBSTANCE. The defense, DERIVED from the
same principle as the heartbeat-gaming defense of Part IV §4.6, is that the density map's headline
metric is the CONFIRMATION RATE (a ratio), not the raw event count, and a fragmentation strategy that
produces many small events with the same underlying confirmation behavior does not change the ratio —
it only makes the denominator and numerator larger in lockstep, which Part XX's countermetric design
treats explicitly as a case where a ratio-based metric is deliberately chosen over a count-based one
specifically to resist this gaming shape.

6.8 Failure mode: mistaking volume for value. The single most likely misreading of the knowledge-flow
density map, named here and formalized as an anti-pattern in this dataset's closing sections, is
treating high flow VOLUME between two systems as evidence the relationship is healthy or important
without checking the CONFIRMATION RATE alongside it — a high-volume, low-confirmation edge is arguably
a worse finding than a low-volume, high-confirmation one, because it represents institutional effort
(producing and transmitting the payload) that is not converting into institutional learning, and a
reader who stops at volume alone will miss exactly the finding the observatory was built to surface.

FINAL LAW — PART VI. Knowledge that was produced and knowledge that was consumed are different facts,
and an estate that measures only the first has built a production ledger, not an observatory; the
flow observatory exists specifically to keep the two facts separate until confirmation evidence
earns the right to merge them.

PART VII — THE FLOW OBSERVATORY II: DECISION FLOW
===============================================================================================

7.1 What a decision flow is, and how it differs from a knowledge flow. Where Part VI traces the
movement of FACTS across system boundaries, this Part traces the movement of DECISIONS — a
system's output that constrains or determines a subsequent action, as opposed to a fact that merely
informs one. A DECISION FLOW, DEFINED here, is an edge along which System P's output is recorded as a
binding or strongly-weighted input into System Q's own decision process, distinguished from a
knowledge flow by the presence of a downstream ACTION whose shape can be traced, at least in part, to
P's decision rather than merely to information P supplied. (DERIVED: the distinction is one of
FUNCTION, not of payload format — the same artifact could, in different contexts, be consumed as
background knowledge in one flow and as a binding decision input in another, and the observatory
classifies each observed flow event by its OBSERVED consequence, not by the artifact's own type.)

7.2 Why decision flow matters independently of knowledge flow. A system can have excellent knowledge-
flow confirmation rates (Part VI) while its actual DECISIONS are ignored or overridden downstream —
its outputs are read, understood, even cited, but never actually change what any consuming system
does. Conversely, a system's decisions can propagate with high fidelity through the ensemble while its
broader knowledge output goes largely unconfirmed, if its consumers treat it strictly as a decision
authority and never draw on its supporting rationale. Because IAS-F1's authority model (Part X of
that dataset) already distinguishes advisory from final authority per decision class, this Part's
decision-flow tracking is the empirical check on whether that DECLARED authority structure matches
OBSERVED behavior — a system with declared advisory authority whose recommendations are, in practice,
never overridden functions, empirically, closer to final authority than its own registry entry
states, and the reverse divergence (declared final authority routinely worked around) is an even more
serious finding, named as an adversarial case in §7.6.

7.3 The decision-flow record schema. Each observed decision-flow event, DEFINED here, carries: SOURCE
SYSTEM, TARGET SYSTEM, DECISION CLASS (resolved against IAS-F1 Part X's own decision-class
vocabulary, never a new taxonomy this dataset invents); DECLARED AUTHORITY TYPE at the time of the
event (advisory or final, read from the source system's property record); OBSERVED OUTCOME (the
target system's actual action — adopted-as-stated, adopted-with-modification, or overridden); and a
TRACE REFERENCE linking to the provenance chain (Part XII of this dataset) that substantiates the
causal claim, because a decision-flow record without a traceable causal link is, per the same
discipline as Part VI's consumption-confirmation requirement, not distinguishable from mere temporal
coincidence.

7.4 Worked example: authority drift made visible. Consider a synthetic trace across a ninety-day
window: a system with DECLARED advisory authority over a specific decision class produces
recommendations consumed by four distinct downstream systems, and in thirty-one of thirty-four
observed decision-flow events, the outcome is recorded ADOPTED-AS-STATED, with the remaining three
ADOPTED-WITH-MODIFICATION and zero OVERRIDDEN. The decision-flow observatory does not itself reclassify
the system's authority — that remains IAS-F1's own authority-model concern, governed by whichever
process grants and revises authority grants (Part X of that dataset) — but it surfaces the ninety-one-
percent as-stated-adoption rate as a finding: an advisory system behaving, empirically, almost
identically to a final-authority one, which is precisely the kind of observation IAS-A2's conflict-
arbitration mechanism and IAS-D1's fitness analysis are positioned to act on, per the Compounding
Graph's stated edges into those two consumers.

7.5 The inverse worked example: declared-final authority routinely reversed. A second synthetic trace:
a system with DECLARED final authority over a narrow decision class shows, across the same window,
eleven decision-flow events, of which six are recorded OVERRIDDEN — a downstream system or a human
operator routinely acting contrary to the final-authority system's own decision. This is a materially
more serious finding than the advisory-drift case of §7.4, because IAS-F1's Invariant Three (Part V
§5.4 of that dataset: exactly one final authority per decision class) is being violated in PRACTICE
even where it holds on PAPER — the registry says one system decides, the observed behavior shows a
different, unregistered actor deciding six times out of eleven. The decision-flow observatory flags
this as an INVARIANT-THREE PRACTICE VIOLATION, a specific, named finding type distinct from the
ordinary authority-drift finding of §7.4, and routes it to IAS-A2 (whose conflict-arbitration
mechanism is the ledger-designated owner of multi-system authority disputes) rather than attempting to
resolve the conflict itself.

7.6 Adversarial case: authority laundering through informal override. A system whose final-authority
decisions are inconvenient could be routinely bypassed not through a formal override mechanism (which
would generate a clean, traceable decision-flow event) but through informal, undocumented workarounds
— a human operator or another system simply not invoking the authoritative system at all for a class
of decisions it nominally owns, redirecting the decision through an unregistered channel. This is a
close cousin of Part V's dark-matter detection, and the decision-flow observatory's defense is the
same structural move: an authoritative system whose declared decision class shows abnormally LOW
decision-flow volume relative to the estimated real-world frequency of that decision class (a
frequency inferred from the artifact-trail method of Part V §5.4, applied here to decisions rather than
artifacts) is flagged not as a healthy quiet system but as a candidate for AUTHORITY LAUNDERING —
decisions of that class are still being made, just not through the registered authority, which the
dark-matter detector's cross-reference makes visible precisely because it was built to find exactly
this shape of gap.

7.7 Integration and the boundary this Part respects. The decision-flow observatory produces findings,
never rulings — it does not adjudicate whether an override was justified, only that it occurred and
at what rate, leaving the adjudication to IAS-A2's arbitration mechanism (multi-system conflict
resolution, per the Non-Duplication Ledger's Family A row) and to D2A where an authority-grant itself
needs revision. This is the same authority-boundary discipline Part II §2.6 states for E1 as a whole,
applied here at its sharpest point, because decision-flow tracking sits closer to a genuine governance
function than any other facet of the fabric, and the temptation to let the observatory quietly start
adjudicating rather than merely reporting is correspondingly higher.

7.8 Edge case: the partial-adoption ambiguity. Part VII §7.3's schema classifies each decision-flow
outcome as ADOPTED-AS-STATED, ADOPTED-WITH-MODIFICATION, or OVERRIDDEN, and the middle category
carries a genuine ambiguity this Part must resolve rather than leave implicit: a modification can be
trivial (a formatting change, a scope narrowing that preserves the recommendation's substance) or
substantial (a change that reverses the recommendation's practical effect while nominally citing it as
the basis). Treating both as the same ADOPTED-WITH-MODIFICATION bucket would blur a distinction that
matters for the same authority-drift analysis Part VII §7.4-7.5 develops. The resolution, DERIVED from
the same consumption-confirmation discipline Part VI establishes for knowledge flow, is that a
modification's magnitude is assessed by comparing the DOWNSTREAM ARTIFACT's own subsequent flow
behavior (Parts VI-IX) against what the ORIGINAL recommendation would have produced, where that
comparison is traceable — a modification that changes which systems the downstream artifact goes on
to FEED, relative to the original recommendation's own stated targets, is classified SUBSTANTIAL-
MODIFICATION, a fourth, more granular category this Part adds specifically to keep the authority-drift
finding of §7.4 from being diluted by an ambiguous middle bucket; a modification with no such
downstream divergence remains ADOPTED-WITH-MODIFICATION under the original three-way schema.

7.9 Worked example: distinguishing trivial from substantial modification. Consider a synthetic
decision-flow event: an advisory system recommends a specific remediation sequence for a flagged
system; the consuming system adopts the recommendation but reorders two non-dependent steps for its
own scheduling convenience, with no change to which downstream systems the remediation ultimately
touches — this reorder is TRIVIAL, correctly remaining ADOPTED-WITH-MODIFICATION. A second synthetic
event: the same advisory system recommends escalating a finding to IAS-D1; the consuming system instead
routes the finding to a narrower, informal review with no IAS-D1 involvement at all — the downstream
FEEDS edge the original recommendation implied (a flow into IAS-D1) never materializes, and a
different one (a flow into the informal review) appears instead — this is SUBSTANTIAL-MODIFICATION, a
materially more consequential finding than a trivial reorder, because the recommendation's actual
institutional effect was redirected even though the consuming system's own summary could honestly
describe itself as having "adopted, with adjustments," the original advice.

FINAL LAW — PART VII. Authority on paper and authority in practice are two different facts, and the
gap between them — an advisory system obeyed as if final, a final authority routinely worked around —
is invisible to every system that only reads the registry; the decision-flow observatory exists to
make that specific, consequential gap visible, and to hand the finding to the system with the actual
power to close it.

===============================================================================================
PART VIII — THE FLOW OBSERVATORY III: EVIDENCE FLOW
===============================================================================================

8.1 What evidence flow tracks, and why it is the fabric's own self-audit surface. Where Parts VI and
VII track the movement of facts and decisions, this Part tracks the movement of EVIDENCE ITSELF — the
provenance and evidence-level markers (IAS-F1 Article V's vocabulary: OBSERVED, DERIVED, HYPOTHESIZED,
and the rest) as they propagate, or fail to propagate, from a claim's origin through every downstream
system that consumes and re-asserts it. This is, in a precise sense, the fabric's own self-audit
surface, because the discipline Article V mandates estate-wide (every load-bearing claim marked) is
only as strong as its WEAKEST propagation link, and no existing system before E1 tracked whether an
evidence level survives a hop from producer to consumer or silently upgrades itself along the way.

8.2 The evidence-decay and evidence-inflation problem. Two failure shapes recur when evidence
propagates across system boundaries without a tracking mechanism. EVIDENCE DECAY: a claim marked
OBSERVED at its origin is re-asserted by a downstream consumer with no evidence marking at all,
losing its epistemic status silently as it moves — the fact survives, but the reader two hops
downstream has no way to know it was ever checked against reality. EVIDENCE INFLATION, the more
dangerous inverse: a claim marked HYPOTHESIZED or DERIVED at its origin is re-asserted by a downstream
consumer AS IF it were OBSERVED, because the consumer's own author, working quickly, dropped the
qualifier and stated the claim as settled fact. (DERIVED: inflation is more dangerous than decay
because decay merely loses information, while inflation actively manufactures false confidence, and a
downstream allocation decision (IAS-C1) built on an inflated claim is a decision built on a fiction
wearing the clothing of a fact — precisely the failure IAS-F1 Part IV §4.6 and Part XII both warn
against at the single-record level, now tracked at the cross-system-propagation level.)

8.3 The evidence-flow record schema. Each observed evidence-flow event, DEFINED here, carries: ORIGIN
CLAIM (a reference to the specific claim and its original evidence level, at its first-observed
assertion); SOURCE SYSTEM and TARGET SYSTEM for this specific propagation hop; SOURCE EVIDENCE LEVEL
and TARGET EVIDENCE LEVEL (the level as marked at each end of the hop); and a DRIFT CLASSIFICATION —
STABLE (level unchanged), DECAYED (level dropped or omitted), or INFLATED (level rose without a
corresponding new reconciliation event, per IAS-F1 Part IV §4.3's reconciliation discipline). A hop
is classified INFLATED specifically, not merely "changed," only when the upgrade is unaccompanied by
a fresh reconciliation record — a claim genuinely re-verified by the target system and legitimately
upgraded from DERIVED to OBSERVED is STABLE-WITH-RECONCILIATION, a fourth classification this Part
adds precisely so that legitimate epistemic progress is not confused with unearned inflation.

8.4 Worked example: a three-hop evidence chain showing both failure modes. Consider a synthetic
chain: a claim originates at a system as HYPOTHESIZED ("this pattern likely reduces rework, not yet
tested"). It propagates to a second system, which cites it, correctly, still as HYPOTHESIZED — a
STABLE hop. It then propagates to a third system, whose own output text drops the qualifier and states
the pattern "reduces rework" as an unqualified fact — an INFLATED hop, because no reconciliation event
appears in the provenance chain between the second and third systems. The evidence-flow observatory
flags this specific hop, naming the origin claim, the point of inflation, and the two systems involved,
which is a materially more actionable finding than a generic "some claims may be overstated" warning,
because it identifies exactly where in the ensemble the discipline broke down and gives IAS-B2's
insight-synthesis process (the ledger-designated owner of cross-dataset knowledge quality) a concrete
correction target.

8.5 Aggregation: the evidence-integrity score per system. At the ensemble level, the observatory
computes, per system, an EVIDENCE-INTEGRITY RATIO: the share of that system's outbound evidence-flow
hops classified STABLE or STABLE-WITH-RECONCILIATION versus DECAYED or INFLATED, over the observation
window. A system with a persistently low ratio is not thereby judged unreliable in its own primary
function — the ratio measures propagation discipline, a distinct property from the quality of the
system's own first-order output — but it is a specific, checkable signal for whichever process
(governance-overlay, per the estate's existing discipline) audits documentation and output quality
across the estate.

8.6 Edge case: legitimate reconciliation misclassified as inflation. A system that performs genuine,
first-party verification of a previously HYPOTHESIZED claim — running the actual test the claim
carries, per IAS-F1 Article V's EXPERIMENTAL vocabulary entry ("proposed to be tested; carries its
test") — and then correctly upgrades the claim to OBSERVED is, by design, exempt from the INFLATED
classification via the reconciliation-event check of §8.3, but this exemption depends entirely on the
reconciliation event itself being recorded and traceable. Where a system performs real verification
but fails to log it as a distinct reconciliation event (a documentation gap, not a truth gap), the
observatory will misclassify a legitimate upgrade as INFLATED — a known, named FALSE-POSITIVE risk
this dataset does not claim to eliminate, only to bound: the observatory's own confidence in an
INFLATED classification is itself marked DERIVED rather than OBSERVED whenever the target system's
reconciliation-logging discipline is itself unaudited, a recursive application of the honesty
discipline this whole facet exists to enforce on everyone else.

8.7 Why evidence flow is the highest-leverage of the four flow facets. HYPOTHESIZED, stated as a
forward claim this dataset does not yet have the volume of evaluation data (Part XXI) to confirm: of
the four flow facets (knowledge, decision, evidence, resource), evidence flow is expected to have the
highest leverage per finding, because a single INFLATED hop, left uncorrected, can silently
misinform every DOWNSTREAM knowledge-flow (Part VI) and decision-flow (Part VII) event that builds on
it — evidence integrity is upstream of the other two facets' own reliability, not merely a peer
measurement alongside them.

8.8 Cumulative decay across long chains, and why a per-hop ratio understates the real risk. Part VIII
§8.5's evidence-integrity ratio is computed PER SYSTEM, per hop, but a claim's real institutional risk
compounds across the FULL LENGTH of the chain it travels, not at any single hop in isolation. A claim
that passes through five systems, each individually showing a respectable ninety-percent STABLE rate
at that hop, has a compounded probability of surviving all five hops STABLE that is meaningfully lower
than ninety percent — DERIVED directly from treating each hop's stability as, to a first approximation,
an independent event, the same reasoning any reliability chain applies to a series of components. This
matters concretely for provenance records (Part XII) with long branches: a HIGH-CONFIDENCE provenance
record four hops deep, per Part XII §12.4's worked example, carries a materially higher cumulative-
decay risk than a two-hop HIGH-CONFIDENCE record, even where every individual hop in both chains shows
identical per-hop integrity ratios, and this Part's own schema (§8.3) is extended to carry a CHAIN-
LENGTH field precisely so that a downstream consumer comparing two HIGH-CONFIDENCE findings can weigh
the longer chain's greater cumulative exposure rather than treating both confidence grades as
equivalent.

8.9 Why this Part does not attempt to compute a single chain-level confidence discount. This dataset
deliberately declines to specify a numeric discount formula (for instance, multiplying per-hop rates
across a chain) as part of E1's own specification, because Article VIII forbids implementation-level
precision this mission is not scoped to calibrate, and a specific formula asserted without real
retrodiction data (Part XXI) would be exactly the guessed, unearned precision Article V warns against.
Instead, this Part specifies only the OBSERVABLE INPUT (per-hop integrity ratio and chain length,
§8.8) a future implementation's own discount formula would need, leaving the formula itself, like the
Bottleneck Map's centrality algorithm (Part XI §11.3), a translation-time decision for whichever
mission builds against this specification.

FINAL LAW — PART VIII. A claim's evidence level is not a permanent property of the claim; it is a
property of a specific assertion at a specific hop, and it can decay through carelessness or inflate
through haste at every boundary it crosses — the evidence-flow observatory exists because Article V's
discipline, honored perfectly at every system's own origin, is worthless if nothing tracks whether it
survives the trip to the next one.

===============================================================================================
PART IX — THE FLOW OBSERVATORY IV: RESOURCE FLOW
===============================================================================================

9.1 What resource flow tracks, closing the four-facet flow taxonomy. The fourth and final flow facet
tracks RESOURCE — the cost (in the sense IAS-F1 Part IV §4.2 defines the COST property: token share,
session overhead, human maintenance burden) that actually moves as a consequence of one system
invoking another, as distinct from the fact moved (knowledge flow), the decision moved (decision
flow), or the epistemic status moved (evidence flow). A RESOURCE FLOW, DEFINED here, is the observed
cost increment charged to, or absorbed by, the CONSUMING mission when it invokes a given system,
aggregated across the observation window into a per-system, per-consumer cost attribution record.

9.2 Why resource flow is E1's object and not CO's, restated precisely. The boundary here mirrors Part
VI's boundary against GK-09: the Cognitive OS (CO) DECIDES what a session should cost (a resource-
level, session-scoped authority, per IAS-F1's own worked Shape One example, Part VIII §8.7 of that
dataset: "given a session's stated budget, pick the model tier — Level(X) = RESOURCE"). This Part
does not decide cost, set budgets, or route model tiers; it OBSERVES, after the fact, how cost
actually distributed itself across the ensemble's systems once CO's decisions were made and missions
executed — an ensemble-level aggregation over many resource-level CO decisions, not a resource-level
decision itself. (DERIVED, applying IAS-F1's majority-object rule, Part VIII §8.4: resource flow's
object is the DISTRIBUTION of cost across systems, which is not contained in any single session's
scope, even though every individual data point it aggregates originates from a CO-owned, session-
scoped cost figure.)

9.3 The resource-flow record schema. Each observed resource-flow event, DEFINED here, carries: SYSTEM
(the system whose invocation incurred the cost); CONSUMING MISSION TYPE; COST FIGURE (read directly
from CO's own existing telemetry, per IAS-F1 Part IV §4.2's explicit instruction that E1 "does not
build a second cost meter, it names the field the existing meter fills" — applied here to a flow
record rather than a static property); TIMESTAMP; and an ATTRIBUTION CONFIDENCE flag, because not
every cost increment can be cleanly attributed to a single invoked system when a mission chains
several systems together in one execution — the attribution confidence flag distinguishes SOLE
(the entire cost increment is attributable to one system's invocation) from SHARED (the increment
covers a chain of several systems and is apportioned by a stated, auditable method, never silently
divided evenly by default).

9.4 Worked example: resource flow revealing a cost-concentration finding. Consider a synthetic
aggregation over a ninety-day window: five systems account, jointly, for eighty-one percent of the
ensemble's total observed resource flow, while the remaining sixty-five addressable systems in the
synthetic estate share the remaining nineteen percent. Read alongside Part III's usage distribution,
this concentration finding becomes actionable in a way neither facet alone would permit: if the same
five systems ALSO account for a proportionate eighty-plus percent of total invocations, the
concentration is unremarkable — expensive systems are expensive because they are heavily used, a
healthy correlation. If, instead, one of the five high-cost systems shows LOW invocation volume in
the usage distribution, the resource-flow observatory has surfaced a specific, checkable finding —
disproportionate cost per invocation — that IAS-C1's portfolio math is the designated consumer for,
per the Compounding Graph's stated edge, without E1 itself rendering the value judgment of whether
that disproportion is justified by the system's criticality (a DEPENDS-ON edge for a rare but
essential mission type) or is genuine waste.

9.5 The resource-versus-value distinction, stated as a hard boundary. This Part deliberately does
not compute, name, or imply a RETURN figure for any system — cost observed against value produced is
explicitly IAS-C1's object (Capability ROI, per the Non-Duplication Ledger's Family C row, candidate
23, MERGE-routed into IAS-C1) and this dataset does not duplicate it. E1's resource-flow observatory
supplies the COST half of that ratio, evidenced and attributed; IAS-C1 supplies the VALUE half from
its own portfolio and ROI mechanisms and computes the ratio itself. (NORMATIVE: stating this boundary
explicitly, rather than letting E1's cost data drift toward an implied ROI verdict, is the specific
discipline that keeps E1 from quietly re-litigating IAS-C1's object under an observability name — the
same discipline Part II §2.6 states in general terms, applied here at its most tempting point, because
cost data sitting next to usage data is only one small step from looking like a value judgment even
when none is intended.)

9.6 Edge case: shared-cost apportionment for chained invocations. When a mission invokes System A,
which internally invokes System B as part of fulfilling the mission, the total resource cost recorded
by CO's telemetry may not cleanly separate A's own overhead from B's — a SHARED attribution case
under §9.3. The observatory's apportionment method, DERIVED as the simplest defensible default rather
than a sophisticated allocation model this mission is not scoped to build, is to apportion by each
system's own declared cost property (IAS-F1 Part IV §4.2) as a weighting prior, flagged explicitly as
an ESTIMATE rather than an OBSERVED figure, with the caveat that a system whose declared cost property
is itself UNRESOLVED (Part IV §4.6 of F1) cannot receive a weighted share and is recorded as
UNAPPORTIONED, a residual category IAS-C1 is warned, in this dataset's own text, not to silently treat
as zero cost.

9.7 Adversarial case: cost-hiding through undeclared subinvocation. A system could reduce its own
apparent resource-flow footprint by invoking a second, less-visible system to perform expensive work
on its behalf without that subinvocation generating its own clean Control Bus event — effectively
laundering cost through an underinstrumented pathway. The defense mirrors Part V's dark-matter
detection directly: a system whose declared cost property (a static, self-reported figure) diverges
significantly from the SUM of resource-flow events actually attributed to missions that invoke it is
flagged for reconciliation, because a persistent, one-directional divergence (self-reported cost
consistently lower than the aggregated observed flow attributable to its invocations) is the specific
signature cost-hiding would produce, as distinct from ordinary measurement noise, which this dataset
expects to be roughly symmetric rather than one-directional.

9.8 Closing the four-facet flow taxonomy. Parts VI through IX, read together, give IAS-D1, IAS-C1, and
IAS-G1 four independently evidenced lenses on the same underlying relation graph — what fact moved
(knowledge), what decision moved (decision), what epistemic status survived the move (evidence), and
what it cost to move (resource) — and the deliberate separation of the four, rather than one blended
"activity" metric, is what lets a downstream consumer distinguish a system that is expensive but
epistemically sound from one that is cheap but propagating inflated claims, or a system whose
decisions are obeyed but whose knowledge is ignored from the reverse pattern — distinctions a single
composite score would erase.

9.8b Worked example: apportionment in practice, distinguishing SOLE from SHARED attribution. Consider a
synthetic mission that invokes a single, self-contained analysis system with no internal
subinvocation — the entire cost figure CO's telemetry records for that mission attaches cleanly to
one system, a SOLE attribution requiring no apportionment logic at all. Contrast a second synthetic
mission that invokes an orchestration system which internally, and transparently, delegates a portion
of its work to two further systems before returning a combined result — here the mission's total
recorded cost must be split three ways. Using the declared-cost-property weighting method of §9.6, if
the orchestration system's own declared cost property is LOW, and the two delegated systems' declared
properties are MEDIUM and HIGH respectively, the apportionment assigns cost shares roughly
proportional to that ordering rather than splitting the total evenly across three systems by default —
an even split would misattribute cost away from the genuinely more expensive delegated system, exactly
the kind of unearned precision this dataset's apportionment method is designed to avoid, while still
being explicitly flagged, per §9.6's own discipline, as an ESTIMATE rather than an OBSERVED figure in
every downstream consumption of the record.

9.9 Why resource-flow data, read alongside usage data, is the single most likely E1 output to attract
an implicit value judgment despite Part IX's own explicit boundary. This Part closes by naming, plainly,
the specific institutional temptation its own §9.5 boundary exists to resist: cost and usage figures,
presented side by side, are visually and narratively adjacent to a value judgment even when neither
figure alone claims to render one, because a reader's own intuition tends to complete the "is this
system worth its cost" inference automatically. This dataset's own discipline — repeating the boundary
explicitly at §9.5 and again here — is a deliberate redundancy, not an oversight, because a boundary
stated once, in a Part a reader may skim past, is more easily forgotten than one restated at the Part's
own close, immediately before the FINAL LAW a reader is most likely to carry forward as the Part's own
summary.

FINAL LAW — PART IX. Cost observed is not value judged; the resource-flow observatory's entire
discipline is to supply the cost half of a ratio it does not compute, honestly attributed and
flagged wherever attribution is an estimate rather than a fact, leaving the verdict to the system
whose object the verdict actually is.

PART X — THE LATENCY OBSERVATORY
===============================================================================================

10.1 What latency measures that flow and usage do not. Parts III through IX establish WHETHER a
system was used and WHAT moved across its boundaries; neither establishes HOW LONG the movement
took. A LATENCY OBSERVATORY, DEFINED here as E1's tenth engine, measures the time elapsed between a
mission's need for a capability and that capability's output becoming available for downstream
consumption — a distinct and, in institutional terms, often more consequential measurement than
either usage count or flow volume, because a system that is heavily used and confirms its knowledge
flow perfectly but takes three times as long as a comparable alternative to produce its output is
imposing a real institutional cost (delay compounds into missed windows, stale context by the time
consumption occurs, and cascading delay in any downstream DEPENDS-ON chain) that neither the usage
observatory nor the flow observatory is built to surface.

10.2 The three latency components. E1 decomposes observed latency into three components rather than
reporting one blended figure, because the three have different causes and different owners for
remediation. QUEUE LATENCY: the time between a mission's request reaching the routing layer (IAS-A1)
and the target system beginning execution — typically a resource-contention signal, actionable by
IAS-C1's allocation decisions. EXECUTION LATENCY: the time the target system itself takes, once
started, to produce output — a signal about the system's own internal efficiency, actionable by
whichever process maintains that system directly. PROPAGATION LATENCY: the time between a system's
output becoming available and a downstream CONSUMES or FEEDS edge's target actually incorporating it
(per Part VI's consumption-confirmation discipline) — a signal about the ensemble's own wiring rather
than about either endpoint system, actionable by IAS-A2's control-bus mechanism or by IAS-G1's
topology analysis.

10.3 The latency record schema and its dependence on Part VI-IX's confirmation discipline. Each
latency record, DEFINED here, carries the three component durations above plus a TOTAL END-TO-END
LATENCY, computed only where all three components have OBSERVED (not estimated) values across a
complete, traceable chain — a partial chain (for instance, where propagation latency cannot be
computed because Part VI's consumption-confirmation flag is DELIVERED-UNCONFIRMED rather than
CONFIRMED) yields a PARTIAL record explicitly marked as such rather than a total latency figure
silently computed from an incomplete chain, which would understate the true end-to-end delay by
excluding the unconfirmed segment.

10.4 Worked example: decomposing a slow mission. Consider a synthetic mission whose Owner-visible
total delay was noticeably longer than the estate's typical range. The latency observatory's
decomposition shows: queue latency accounting for eleven percent of the total delay, execution
latency accounting for eighteen percent, and propagation latency accounting for the remaining
seventy-one percent — the bulk of the delay was not the invoked system being slow to compute, nor
contention for resources, but the TIME BETWEEN the system's output being ready and a downstream
system actually picking it up and acting on it. This decomposition redirects the remediation target
entirely: a naive "the mission was slow" complaint would likely have prompted an incorrect look at
the invoked system's own execution efficiency, when the real finding points at IAS-A2's event-delivery
mechanism or at a structural wiring gap IAS-G1's topology analysis is positioned to examine.

10.5 Aggregation: the latency baseline and drift detection. At the ensemble level, the observatory
maintains a rolling BASELINE latency distribution per system, per component, and flags any system
whose current-window latency for a given component exceeds its own historical baseline by a stated
margin — a DRIFT flag, distinguished deliberately from an ABSOLUTE threshold, because different
systems have genuinely different natural latency profiles (a system performing a deep multi-step
analysis is expected to have higher execution latency than a lightweight lookup, and comparing the two
against one shared absolute threshold would either falsely flag the analytical system or fail to
catch a genuine regression in the lightweight one). (DERIVED: this per-system, self-relative baseline
design directly mirrors IAS-F1's staleness discipline, Part IV §4.3 of that dataset — a property
compared against ITS OWN prior state, not against an estate-wide constant.)

10.6 Edge case: legitimate latency increase from a genuine complexity increase. A system that has
recently absorbed new capability (an EXTEND disposition under D2A, folding a duplicate candidate's
scope into itself) will legitimately show higher execution latency than its own pre-extension
baseline, and the drift detector's naive application would flag this as a regression. The resolution,
DERIVED from the same principle as Part IV §4.5's seasonal-system edge case, is that a DRIFT flag is
demoted to INFORMATIONAL rather than ACTIONABLE when it coincides, within the observation window, with
a recorded capability-signature change on the system's own IAS-F1 addressing record (Part IX §9.2 of
that dataset — the capability-signature component is explicitly expected to change as a system
evolves) — the latency observatory checks for this coincidence before escalating a drift flag, rather
than reporting every latency increase as if it were unexplained.

10.7 Adversarial case: latency masking through asynchronous fire-and-forget invocation. A system could
report artificially low execution latency by returning a preliminary, near-empty acknowledgment
immediately while the substantive work continues asynchronously, with the real output arriving much
later through a separate, less-instrumented channel — effectively hiding true latency behind a fast
but content-free response. The defense is the same consumption-confirmation discipline Part VI
establishes: the latency observatory's TOTAL END-TO-END figure is computed against the CONFIRMED
consumption event (the point at which a downstream system's behavior actually reflects the output),
not against the point at which a response of any kind was first returned, which means a fire-and-
forget acknowledgment with no substantive content generates, at most, a very short queue-and-execution
latency figure paired with a long, honestly-reported propagation latency once the real output
eventually confirms — the gaming strategy does not actually hide the true delay, it merely relocates
it into the propagation component, where §10.2's decomposition still surfaces it.

10.8 Aggregation beyond the single system: percentile reads and why an average hides tail risk. Every
worked example in this Part so far has read a single system's own latency record against its own
baseline. At the ensemble level, the observatory additionally maintains a PERCENTILE DISTRIBUTION per
component (queue, execution, propagation) across all systems — the p50 (median), p90, and p99 latency
for each component, computed across the full population of observed invocations in the window rather
than a simple estate-wide average. This distinction matters concretely: an estate-wide AVERAGE
propagation latency can look acceptable while a small number of systems sit at the extreme tail (p99),
imposing real institutional cost on the missions unlucky enough to route through them, and an average
figure alone would hide exactly the systems Part XI's bottleneck map most needs to examine. (DERIVED:
this is the same reasoning that underlies Part XIV §14.2's rejection of a single composite health
score, applied here at the latency-distribution level rather than the cross-facet level — a summary
statistic that collapses a distribution into one number hides the tail, and the tail is frequently
where the institutional cost concentrates.)

10.9 Worked example: a p99 finding invisible to the mean. Consider a synthetic ensemble-wide
propagation-latency distribution with a p50 of eleven minutes (a healthy median) and a mean of
fourteen minutes (only modestly higher, and unremarkable on its own) but a p99 of nine hours — a
small population of edges where propagation takes dramatically longer than the typical case. Because
the mean is barely elevated (the tail population is small relative to the whole), an average-only
report would never surface this finding, while the percentile read isolates it immediately and hands
the specific edges responsible for the p99 tail to the Bottleneck Map (Part XI) as a candidate
STRUCTURAL CHOKEPOINT investigation, since a small number of edges carrying disproportionately extreme
latency is frequently the same population carrying disproportionate structural criticality.

FINAL LAW — PART X. A slow mission has a cause with an owner, and the owner is almost never the system
the Owner's own intuition first suspects; the latency observatory exists to decompose delay into
queue, execution, and propagation components precisely because remediating the wrong component wastes
institutional effort on a fix that was never going to move the number that mattered.

===============================================================================================
PART XI — THE BOTTLENECK MAP
===============================================================================================

11.1 From per-edge measurement to graph-wide structure. Parts VI through X each measure a property of
individual edges or systems — a flow's confirmation rate, a system's usage count, a component's
latency. The BOTTLENECK MAP, DEFINED here as E1's eleventh engine, is the first facet that reads the
TYPED RELATION GRAPH (IAS-F1 Part VI) as a whole structure rather than edge-by-edge, identifying
where the ensemble's actual throughput concentrates, stalls, or single-points-of-failure gather —
findings that are, by construction, invisible to any per-edge measurement no matter how precise,
because a bottleneck is a property of the GRAPH's topology under load, not of any one edge in
isolation.

11.2 The three bottleneck signatures. CONVERGENT BOTTLENECK: a single system that sits on an
abnormally high share of DEPENDS-ON edges across the graph — many other systems cannot function
without it — combined with latency (Part X) or usage (Part III) data showing it operating near a
capacity ceiling, such that its own queue latency is rising across the observation window. A
convergent bottleneck is the ensemble-level analogue of a single point of failure, and its remediation
(redundancy, load redistribution, or architectural decomposition) is IAS-G1's object, with E1 supplying
only the evidenced finding. STRUCTURAL CHOKEPOINT: an edge, or small cluster of edges, through which a
disproportionate share of TOTAL graph flow (summed across all four flow facets, Parts VI-IX) must pass
because no alternative path exists between two otherwise well-connected regions of the graph — distinct
from a convergent bottleneck because the constraint is the EDGE's structural uniqueness, not any one
system's capacity. STARVED REGION: a cluster of systems with real, observed DEPENDS-ON relationships
among themselves but abnormally low inbound flow from the rest of the graph relative to their combined
capability signatures — a region of the estate that is structurally isolated from the ensemble's main
traffic, whether by genuine narrow applicability or by an undiscovered routing gap IAS-A1's own
mechanism should be examining.

11.3 The detection method: graph centrality over the observed-load-weighted relation graph. The
bottleneck map is computed, DERIVED here as the specific mechanical procedure, by weighting every edge
in IAS-F1's typed relation graph with its OBSERVED flow volume and latency (from Parts VI-X, not from
any system's self-reported importance) and then applying standard graph-centrality analysis — a system
with high inbound DEPENDS-ON centrality AND rising latency drift (Part X §10.5) is flagged CONVERGENT;
an edge whose removal, in a counterfactual graph traversal, would disconnect two otherwise well-
connected regions AND which carries above-baseline flow volume is flagged CHOKEPOINT; a cluster with
internal DEPENDS-ON density well above its inbound-flow-from-outside-the-cluster ratio is flagged
STARVED. (NORMATIVE: this dataset specifies the METHOD, per Article VIII's specification-only mandate,
not an implementation of the centrality algorithm itself — the specific algorithm choice is a
translation-time decision for whichever mission builds the fabric, named as a candidate in this
dataset's closing Dataset-to-Runtime section.)

11.4 Worked example: a convergent bottleneck surfaced from synthetic data. Consider a synthetic
finding: one system shows inbound DEPENDS-ON edges from fourteen other systems — the highest count in
the synthetic estate — and its queue-latency baseline (Part X §10.5) has risen thirty-four percent
over the observation window with no corresponding capability-signature change (ruling out the
legitimate-complexity-increase edge case of Part X §10.6). The bottleneck map flags this system
CONVERGENT with HIGH severity, and the finding, read alongside Part III's usage distribution (this
system also shows the highest raw invocation count in the estate) and Part IX's resource-flow data
(disproportionate cost concentration, per the worked example of Part IX §9.4), assembles into a
coherent institutional picture: a single, heavily-relied-upon, increasingly strained system that the
estate has, likely without deliberate decision, made structurally central — precisely the finding
IAS-G1's architecture intelligence and IAS-D1's ecology are positioned to act on, whether through
redundancy investment, load redistribution, or a deliberate decomposition into a less centralized
topology.

11.5 Worked example: a starved region surfaced from synthetic data. A second synthetic finding: a
cluster of six systems shows dense internal DEPENDS-ON relationships (each depends on at least two
others within the cluster) but a combined inbound-flow-from-outside ratio in the bottom decile of the
synthetic estate — the cluster is functioning, internally coherent, but almost entirely disconnected
from the rest of the ensemble's traffic. Cross-referenced against Part V's dark-matter detector, the
bottleneck map's starved-region finding and the dark-matter detector's pattern-recurrence finding are
DISTINCT but potentially RELATED signals worth joint review: a starved region might indicate a genuine
niche capability serving a narrow, legitimate mission type (in which case low outside-flow is expected
and healthy), or it might indicate a routing gap where IAS-A1 simply never discovers the cluster as a
candidate for missions that would benefit from it — the bottleneck map does not adjudicate between
these two readings, flagging the finding and its ambiguity for IAS-D1's fitness-landscape analysis and
IAS-A1's own routing-coverage review.

11.6 Edge case: a bottleneck that is bottleneck by design. A system deliberately built as a single,
centralized authority for a decision class explicitly requiring singular authority (IAS-F1 Invariant
Three, Part V §5.4 of that dataset) will, correctly, show convergent-bottleneck signatures — high
inbound DEPENDS-ON centrality is not a defect in this case, it is the invariant working as intended.
The bottleneck map's severity classification therefore checks a system's DECLARED AUTHORITY TYPE
(Part VII §7.3's schema field) before assigning severity: a system with declared final authority
showing convergent centrality is downgraded from HIGH to INFORMATIONAL severity by default, because
centralization of a genuinely singular decision class is the invariant's correct operation, not a
structural risk — only a RISING LATENCY signal on such a system (evidence the singular authority is
becoming an operational strain, not merely a structural fact) escalates the finding back to actionable
severity.

11.7 Why the bottleneck map depends on every prior engine in this Part-sequence. Stated explicitly
because it is the clearest demonstration of E1's own internal architecture (formalized fully in Part
XVI): the bottleneck map cannot be computed from the relation graph alone — IAS-F1 already defines
the graph's topology, unweighted, in its own Part VI — it requires the OBSERVED-LOAD weighting Parts
III through X supply, meaning the bottleneck map is, by construction, the synthesis engine of this
dataset's first eleven Parts rather than an independent measurement, and any implementation that tried
to build the bottleneck map without first building the usage, flow, and latency engines beneath it
would be building on an unweighted, and therefore uninformative, graph.

11.8 Worked example: a structural chokepoint surfaced from synthetic data, distinguished from the
convergent-bottleneck case. Where §11.4's worked example showed a single high-centrality SYSTEM, this
example demonstrates the second bottleneck signature named in §11.2: a structural chokepoint at the
EDGE level. Consider a synthetic estate divided, by its own genuine functional structure, into two
well-connected regions — an "analysis" cluster of eleven systems and a "delivery" cluster of nine —
with real, observed FEEDS and DEPENDS-ON traffic flowing in both directions between the two clusters,
but ALL of that cross-cluster traffic routing through exactly one edge, a single FEEDS relation between
one analysis-cluster system and one delivery-cluster system. Neither endpoint system, individually,
shows unusual centrality within its own cluster (ruling out a convergent-bottleneck reading, §11.2's
first signature), but the counterfactual graph traversal named in §11.3 shows that removing this one
edge would fully disconnect the two clusters, and the edge carries eighty-eight percent of total
observed cross-cluster flow volume across the four flow facets combined — a textbook STRUCTURAL
CHOKEPOINT. The remediation implication differs materially from the convergent case: where a
convergent bottleneck's typical remedy is redundancy or load redistribution at the SYSTEM level, a
structural chokepoint's typical remedy is a NEW EDGE — either a second, parallel connection between the
two clusters, or a restructuring of which specific systems within each cluster interface directly —
and the bottleneck map's own classification (CONVERGENT versus CHOKEPOINT) is precisely what lets
IAS-G1's topology analysis (Part XXII §22.4) select the correct remedy category rather than applying a
system-level fix to what is, in fact, an edge-level structural gap.

FINAL LAW — PART XI. A bottleneck is never a property of one system or one edge in isolation; it is a
property of the graph's shape under real, observed load, and an estate that maps its systems without
weighting the map by actual traffic has drawn a map of where things ARE, not a map of where the
institution's real strain actually lives.

===============================================================================================
PART XII — DECISION PROVENANCE TRACKING
===============================================================================================

12.1 The question this engine answers. "Why was this decision made, and which systems actually
contributed to it?" is a question the estate needs answered honestly for reasons ranging from routine
audit to post-hoc failure analysis, and it is a question none of E1's prior engines directly answer:
the usage observatory says a system was invoked; the decision-flow observatory (Part VII) says an
authority's output was adopted; neither reconstructs the full causal chain of a SPECIFIC decision,
end to end, across every system that fed it. DECISION PROVENANCE, DEFINED here as E1's twelfth engine,
is the reconstructed causal graph — a subgraph of IAS-F1's typed relation graph, restricted to the
edges that actually contributed evidence, knowledge, or constraint to one named decision — walkable
backward from the decision to its root inputs.

12.2 Why provenance tracking is a distinct engine, not a byproduct of the flow observatories. Parts VI
through IX each record individual flow events; provenance tracking is the ASSEMBLY of those individual
records into one coherent, decision-scoped narrative, which requires a distinct data structure (a
directed acyclic subgraph rooted at the decision, per §12.3) and a distinct query pattern (backward
traversal from one node, rather than the aggregate, ensemble-wide statistics the other engines
compute). (DERIVED: this mirrors the general software-engineering distinction between an event log
and a trace — the flow observatories are the event log; provenance tracking is the trace assembled
from it on demand.)

12.3 The provenance record schema. A provenance record, DEFINED here, is rooted at a single DECISION
NODE (a specific, timestamped output from a system holding authority, per IAS-F1 Part X, over the
relevant decision class) and contains every edge, walked backward, that the flow observatories (Parts
VI-IX) recorded as CONFIRMED (Part VI's consumption-confirmation standard, applied uniformly across
all four flow types) contributing to that decision node, terminating at ROOT INPUTS — nodes with no
further confirmed inbound edges within the observation window, which may be either a genuinely
original system-generated judgment or simply the edge of what the observation window and instrumentation
can see (a distinction the record marks explicitly as OBSERVATION-BOUNDARY versus GENUINE-ORIGIN,
because conflating the two would falsely present an incompletely-instrumented chain as a fully-
resolved one).

12.4 Worked example: tracing a synthetic decision to its roots. Consider a synthetic decision node: a
capital-allocation recommendation issued by a portfolio-facing system. Walking backward through
CONFIRMED edges, the provenance record shows the recommendation drew on: a usage-distribution figure
(Part III, this dataset, itself a root input — genuine origin, since it derives directly from Control
Bus observation with no further upstream system feeding it); a cost-attribution figure (Part IX, also
a root input by the same reasoning); and a fitness score from an ecology system, which itself, walked
one hop further back, drew on a symbiosis finding that in turn drew on two DEPENDS-ON edges recorded
in the base relation graph. The full provenance record is four hops deep at its longest branch, and it
terminates cleanly at genuine-origin root inputs on every branch — a HIGH-CONFIDENCE provenance
reconstruction, per the confidence grading of §12.5.

12.5 Confidence grading of provenance records. A provenance record's confidence, DEFINED here, is
graded by the WEAKEST evidence level among all edges in its walked subgraph (per Article V's own
discipline that a chain is only as strong as its weakest link, applied here structurally rather than
per-claim) and by whether any branch terminates at an OBSERVATION-BOUNDARY rather than a GENUINE-
ORIGIN root. A record with every edge CONFIRMED and OBSERVED, terminating only at genuine-origin
roots, is HIGH-CONFIDENCE. A record with any edge at DELIVERED-UNCONFIRMED status (Part VI §6.3) or
any branch terminating at an observation boundary is MEDIUM-CONFIDENCE, with the specific weak point
named rather than the whole record silently downgraded without explanation. A record that cannot be
assembled at all — no confirmed inbound edges recorded for the decision node — is NO-PROVENANCE, a
distinct and separately important finding from a low-confidence one, because it means either the
decision was made with no traceable system input at all (a governance concern in its own right) or the
instrumentation genuinely failed to capture real inputs that existed.

12.6 Why provenance tracking is the engine most exposed to the evidence-inflation risk of Part VIII.
Because a provenance record is, by its nature, presented as a clean narrative ("decision D drew on
inputs X, Y, Z"), it is the E1 output most likely to be read and cited by a human Owner or a
downstream system as settled fact, which makes it the single highest-leverage point for Part VIII's
evidence-inflation failure mode to do institutional damage if the confidence grading of §12.5 is
dropped in re-telling. This dataset's specific mitigation, NORMATIVE, is that every provenance record's
confidence grade is a MANDATORY, non-optional field in any presentation of the record — a provenance
narrative presented without its confidence grade is, per this dataset's own discipline, an incomplete
and therefore non-compliant output, not a convenience omission.

12.7 Integration with decision-flow and evidence-flow. Provenance tracking is explicitly the assembly
layer sitting atop Part VII's decision-flow records (which supply the authority-and-outcome context
for each node) and Part VIII's evidence-flow records (which supply the drift classification for each
edge, letting a provenance record show not just WHAT fed a decision but whether the evidence
INFLATED along the way) — a provenance record that includes an INFLATED edge, per Part VIII's
schema, carries that inflation forward into its own confidence grading, demoting the record even where
every individual link would otherwise look CONFIRMED, because a confirmed-but-inflated chain is a
worse foundation for trust than an honestly-marked medium-confidence one.

12.8 Retention and the revised-decision edge case. A provenance record is not necessarily static once
assembled — a decision node's own downstream effects can continue to accrue new CONFIRMED edges after
the decision itself was made (a recommendation adopted, then cited again weeks later as input to a
second, related decision), and a decision can itself be formally REVISED (superseded by a later
decision from the same authority, per IAS-F1 Part X's own decision-class vocabulary) without its
original provenance record ceasing to be historically true. This Part's resolution, DERIVED from the
same lineage-preserving discipline Part XVIII §18.3 specifies for merged systems, is that a provenance
record is retained as an IMMUTABLE historical artifact once its observation window closes — a revision
does not edit the original record, it creates a NEW decision node with its own provenance chain,
which may, and typically will, cite the original decision as one of its own root or intermediate
inputs. This means a single institutional question — "why do we currently do X" — can require walking
TWO provenance records in sequence (the original decision's, and the revision's), and this Part
specifies that a REVISION-CHAIN query, distinct from a single-record walk, is the correct mechanism for
answering that broader question rather than attempting to retrofit the original record with the
revision's own later reasoning.

12.9 Worked example: a two-record revision chain. Consider a synthetic case: a capital-allocation
decision (Part XII §12.4's own worked example) is revised eleven weeks later after the Bottleneck Map
(Part XI) surfaces a CONVERGENT finding on one of the systems the original allocation favored. The
revision's own provenance record shows a HIGH-CONFIDENCE chain rooted partly in fresh Bottleneck Map
evidence and partly in a CONFIRMED citation of the ORIGINAL decision node itself, treated as a single
root input rather than re-walked in full. A consumer asking "why does the estate currently allocate
capital this way" retrieves the revision's own record plus, by following its citation edge, the
original's — a two-hop REVISION-CHAIN read that preserves both decisions' own historical integrity
rather than silently overwriting the first with the second.

FINAL LAW — PART XII. A decision's provenance is not a story told after the fact to make the decision
sound well-founded; it is a walkable, confidence-graded subgraph of what the ensemble can actually
prove fed the decision, and the honest answer is sometimes NO-PROVENANCE — a finding the fabric must
be willing to report exactly as readily as a clean four-hop chain, or its provenance records become
theater wearing a rigorous name.

PART XIII — THE KNOWLEDGE-CONSUMPTION TRACKER
===============================================================================================

13.1 Closing the gap Part VI opened, and folding candidate 90's yield metric. Part VI's knowledge-flow
observatory tracks individual flow events and their confirmation status; this Part, E1's thirteenth
engine, aggregates that stream into a per-KNOWLEDGE-ASSET view — for a given distilled deposit,
principle, or context pack (the estate's own knowledge production units, owned by FD, ACIS, and the
Graphify Kernel's supply chain), what share of its downstream FEEDS edges across its full lifetime
resulted in confirmed consumption versus delivered-unconfirmed status. This is, precisely, the
"measured knowledge yield" metric the Non-Duplication Ledger's Family I row (candidates 81-90)
instructs be folded into IAS-E1 rather than built as its own dataset, because the knowledge supply
chain's OWN systems (GK-01 through GK-08, FD, CO-05, akos_knowledge) already own production,
canonicalization, routing, freshness, and writeback — but none of them, as OBSERVED, closes the loop
by measuring whether what they produce is actually CONSUMED downstream, which is the specific gap
this engine closes using machinery this dataset already built for Part VI's own purposes.

13.2 The distinction from recall_roi, restated precisely per the Strengthening Matrix. recall_roi (one
of E1's five strengthened parents) measures per-ASSET reuse — did this specific distilled deposit get
reused in a later, different context. The knowledge-consumption tracker measures something adjacent
but distinct: not merely WHETHER an asset was reused, but the full CONFIRMATION-RATE distribution
across every asset the knowledge supply chain has produced, aggregated to expose systemic patterns
(which PRODUCING system's deposits confirm at a healthy rate versus which producing system's deposits
are routinely delivered-but-inert) that a single-asset reuse metric cannot surface on its own. (DERIVED:
this is the specific mechanism by which E1 strengthens recall_roi per the matrix's own row — E1
supplies the cross-asset, cross-producer aggregate pattern; recall_roi's own per-asset figure becomes
one confirmed data point feeding that aggregate, rather than E1 rebuilding recall_roi's own per-asset
measurement.)

13.3 The consumption-tracker record schema. Per PRODUCING SYSTEM (the system whose FEEDS edges
originate the knowledge asset — FD, ACIS, or a GK writeback pathway), the tracker maintains: TOTAL
ASSETS PRODUCED in the observation window; CONFIRMED-CONSUMPTION COUNT (assets with at least one
CONFIRMED flow event, per Part VI §6.3, within a stated grace period after production); DELIVERED-
UNCONFIRMED COUNT; NO-DELIVERY COUNT (assets with zero recorded FEEDS edges at all — produced but never
even routed to a candidate consumer, a distinct and more concerning finding than delivered-but-
unconfirmed, because it suggests a routing gap rather than a consumption gap); and the resulting YIELD
RATIO (confirmed-consumption count divided by total assets produced), the specific figure the ledger
names as the folded metric.

13.4 Worked example on synthetic data, distinguishing a routing problem from a consumption problem. A
synthetic producing system shows: 140 assets produced, 19 confirmed-consumption, 88 delivered-
unconfirmed, and 33 no-delivery, over a ninety-day window — a yield ratio of roughly fourteen percent.
Read at the yield-ratio level alone, this looks like an undifferentiated knowledge-quality problem. The
tracker's decomposition redirects the finding: the thirty-three no-delivery assets point at a ROUTING
gap (IAS-A1's object — these assets were never even offered to a candidate consumer), while the
eighty-eight delivered-unconfirmed assets point at either a genuine RELEVANCE gap (the assets were
offered but not useful enough to change downstream behavior) or an INSTRUMENTATION gap (real
consumption occurred but was not traceably confirmed, per Part VIII §8.6's analogous false-positive
caution). The tracker does not resolve which of the two explains the delivered-unconfirmed population
— that judgment requires either deeper sampling (Part XXI's evaluation protocol) or a qualitative
review this dataset does not automate — but decomposing "low yield" into three structurally distinct
causes, each with a different owner, is the tracker's specific contribution over a single blended
percentage.

13.5 Aggregation to the ensemble level, and the anti-Goodhart caution this engine requires most
sharply. An ensemble-wide YIELD DISTRIBUTION, ranked by producing system, is the tracker's headline
output — and it is, of every metric this dataset defines, the one most vulnerable to the watched-
metric-gaming failure mode formalized in Part XIX, because "produce fewer, more targeted assets"
trivially raises a producing system's yield ratio without any genuine improvement in institutional
knowledge quality, simply by shrinking the denominator. Part XX's countermetric design pairs the yield
ratio with an ABSOLUTE confirmed-consumption count specifically to close this gap: a producing system
optimizing its ratio by starving its own output would show a rising ratio alongside a falling absolute
count, a divergence the paired-metric design is built to make visible rather than let the ratio alone
disguise.

13.6 Integration: the tracker as GK-09's and recall_roi's downstream consumer of consumers. Per the
Strengthening Matrix's evidence requirement for IAS-E1 ("a dead-capability list and a dark-matter item,
each with evidence"), this engine additionally supplies GK-09 with a finding GK-09's own substrate-
scoped telemetry cannot generate on its own: a knowledge node that GK-09 reports as fresh and
well-indexed but that this tracker shows has a NO-DELIVERY status across its full observation
lifetime — freshness and findability, per GK-09's own object, do not guarantee the node has ever
actually been offered to a consuming mission, and the tracker's finding is the specific, checkable
evidence of that gap, handed back to GK-09 as a strengthening input rather than GK-09 rebuilding it
independently.

13.7 Reconciling the tracker's aggregate against recall_roi's own per-asset figure, and what a
divergence would mean. Because recall_roi already measures per-asset reuse independently (Part XIII
§13.2's stated boundary), the tracker's own yield distribution can be cross-checked against a
SEPARATE, independently-sourced expectation: summing recall_roi's own per-asset reuse events for a
given producing system's output should, in a well-instrumented estate, track closely with this
tracker's own CONFIRMED-CONSUMPTION count for the same system and window, since both are, at bottom,
counting the same underlying phenomenon — a distilled asset changing downstream behavior — through two
independently-built measurement paths. A persistent, one-directional DIVERGENCE between the two counts
(recall_roi consistently reporting materially more reuse than this tracker's own confirmed-consumption
count for the same producing system) is not dismissed as measurement noise; it is flagged as an
INSTRUMENTATION-DISCREPANCY finding, because it indicates either this tracker's own consumption-
confirmation mechanism (inherited from Part VI §6.3) is missing real confirmed flow events that
recall_roi's own, differently-built mechanism is catching, or recall_roi is over-counting relative to a
stricter confirmation standard — either explanation is a genuine, actionable finding about the
estate's own measurement infrastructure, not a finding to be silently averaged away.

13.8 Why this cross-check is the specific, concrete mechanism behind the Strengthening Matrix's
recall_roi row. Part XIII §13.2 already states the general strengthening claim; this cross-check is
its precise operational form — two independently-sourced measurements of a closely related phenomenon,
reconciled against each other, are jointly more trustworthy than either alone, and a divergence between
them is a genuinely new class of finding (an instrumentation gap) that neither recall_roi's own
per-asset scope nor this tracker's own aggregate scope could surface working in isolation. This is the
same general pattern Part III §3.2 applies at the usage-observatory level (bus-sourced versus
incidental-evidence reconciliation) and Part IX §9.7 applies at the resource-flow level
(self-reported versus aggregated-observed cost reconciliation) — cross-checking two independently-
sourced signals against each other, rather than trusting either alone, recurs across this dataset as
its general defense against both honest measurement gaps and adversarial gaming alike.

FINAL LAW — PART XIII. Knowledge produced is not knowledge yielded; a distilled deposit that is fresh,
well-indexed, and perfectly canonicalized but never routed to a consumer, or routed and never
confirmed changing a downstream decision, has cost the estate real production effort for a return of
zero — and the only honest way to know the difference between a healthy knowledge supply chain and one
quietly producing for no one is to track consumption with the same discipline the chain already
applies to production.

PART XIV — THE INSTITUTIONAL HEALTH OBSERVATORY
===============================================================================================

14.1 Why a composite view is needed, and why it is built last among the measurement engines. Parts III
through XIII each answer a narrow, well-scoped question — usage, deadness, dark matter, four flow
facets, latency, bottlenecks, provenance, consumption yield. An Owner or a consuming IAS dataset asking
"how healthy is the ensemble, overall, right now" cannot be answered by handing over eleven separate
reports; the INSTITUTIONAL HEALTH OBSERVATORY, DEFINED here as E1's fourteenth and final measurement
engine (deliberately built last, consuming every prior engine's output rather than computing anything
independently), synthesizes the eleven facets into one composite, evidence-graded READ of ensemble
health — explicitly a READ, not a SCORE, for reasons this Part develops at length, because a single
scalar health number is precisely the artifact most exposed to the observability-theater failure mode
Part XIX names.

14.2 Why the health observatory refuses a single scalar score. A single composite health number,
however carefully weighted, collapses eleven independently meaningful, differently-evidenced facets
into one figure that necessarily hides which facet is driving it — a healthy-looking aggregate score
could mask a severe bottleneck finding (Part XI) sitting alongside excellent usage and flow numbers, or
a rising evidence-inflation rate (Part VIII) sitting alongside a strong yield ratio (Part XIII). This is
not a hypothetical risk; it is DERIVED directly from the general property of any weighted-sum metric —
information is lost in the weighting, and the loss is invisible to a reader who sees only the final
number. (NORMATIVE: this dataset therefore defines the health READ as a STRUCTURED PROFILE — one
graded value per facet, plus a small number of CROSS-FACET CORRELATION findings, Part XIV §14.4 — never
a single number an Owner could quote without the underlying structure.)

14.3 The health profile schema. A health-read record, DEFINED here, carries, for the stated
observation window: the USAGE DISTRIBUTION summary and concentration figure (Part III); DEAD and
LIVE-SUSPECT counts (Part IV); DARK-MATTER finding count by signature type (Part V); per-flow-facet
CONFIRMATION RATES (knowledge, decision, evidence, resource — Parts VI-IX); the EVIDENCE-INTEGRITY
RATIO distribution (Part VIII §8.5); LATENCY DRIFT flags outstanding (Part X); BOTTLENECK findings by
severity (Part XI); NO-PROVENANCE rate among sampled recent decisions (Part XII); and the YIELD
DISTRIBUTION (Part XIII). Each field carries its own evidence level inherited from its source engine —
the health record does not upgrade or homogenize evidence levels in the act of compiling them, per the
same discipline Part VIII exists to police at the cross-system level, now applied reflexively to E1's
own compilation step.

14.4 Cross-facet correlation as the health observatory's genuine value-add. The observatory's actual
synthesis work is not the compilation of §14.3's profile (a straightforward roll-up) but the detection
of CORRELATED findings across facets that, read separately, would each look like isolated, moderate
concerns, but together indicate a compounding institutional risk. A worked synthetic example: a system
showing rising latency drift (Part X) AND convergent-bottleneck flagging (Part XI) AND a declining
evidence-integrity ratio (Part VIII) simultaneously is a materially more urgent finding than any one of
the three alone — the pattern suggests a single, overloaded, increasingly central system that is
ALSO, under that load, cutting corners on its own evidence discipline, a compounding-risk shape the
health observatory names explicitly (a CONVERGENT-STRAIN correlation) rather than leaving the three
findings to be manually cross-referenced by a reader.

14.5 Worked example: a full health-read synthesis on the synthetic estate. Drawing together the
synthetic findings threaded through Parts III-XIII of this dataset — Sigma-22's DEAD verdict (Part IV
§4.4), the eleven-times-recurring unregistered pattern (Part V §5.6), the low-confirmation knowledge
edge (Part VI §6.5), the ninety-one-percent as-stated advisory-adoption drift (Part VII §7.4), the
inflated-evidence hop (Part VIII §8.4), the propagation-dominated slow mission (Part X §10.4), the
convergent bottleneck (Part XI §11.4), and the fourteen-percent yield producing system with a routing-
versus-relevance ambiguity (Part XIII §13.4) — the health observatory's synthesis does not average
these into one number; it presents them as a ranked list of DISTINCT findings, each retaining its own
facet, evidence level, and consuming-system routing, with only the convergent-bottleneck-plus-latency-
drift pair (§14.4's correlation logic, since the same synthetic system appears in both findings)
elevated to a CONVERGENT-STRAIN correlation flag. This worked synthesis demonstrates the observatory's
actual output shape: a structured, cross-referenced brief, not a dashboard number.

14.6 Integration: the health read as the primary artifact IAS-D1 and IAS-G1 consume. Per the
Compounding Graph's stated loop ("IAS-E1 reveals which systems are used/dead/dark, and real flow +
bottlenecks... IAS-D1... outcomes update fitness/symbiosis/parasitism... IAS-G1... fitness informs
topology"), the health read is the specific artifact form in which E1's synthesis reaches IAS-D1 and
IAS-G1 — not fourteen separate raw engine outputs each sibling dataset must independently reassemble,
but one structured, correlation-annotated brief per observation window, which is the concrete mechanism
this dataset uses to discharge its Compounding Graph obligations rather than merely gesturing at the
edges the graph names.

14.7 Retention and trend reading across successive health reads. A single window's health read, however
well-structured, cannot itself answer whether the ensemble is improving or deteriorating — that
question requires comparing successive reads across TIME, which this Part specifies as a distinct
capability rather than assuming it falls out of the per-window schema automatically. The Health
Observatory retains its own structured profile (§14.3) across every past observation window it has
computed, and offers a TREND READ — for each field in the profile, whether the current window's value
represents an IMPROVEMENT, a DETERIORATION, or NO-MATERIAL-CHANGE relative to the prior window's same
field, using each field's own natural direction of improvement (a rising confirmation rate is an
improvement; a rising DEAD count is, absent an explicit retirement decision, a deterioration; a rising
LATENCY-DRIFT count is a deterioration unless explained by Part X §10.6's legitimate-complexity
exception). The trend read is explicitly NOT a second composite score — it is the same eleven-facet
structure, each field now carrying a directional arrow alongside its absolute value, preserving §14.2's
no-single-number discipline at the trend level exactly as it applies at the single-window level.

14.8 Why a trend read is the specific mechanism that makes Part XIX's theater-drift signature checkable
in practice. Part XIX §19.4 names the observability-theater detection signature as a rising share of
CONFIRMED/OBSERVED/HIGH-CONFIDENCE findings alongside a falling rate of STALENESS/UNRESOLVED/NO-
PROVENANCE flags — a pattern that is, by definition, only visible by comparing successive windows, which
is exactly what this Part's trend read computes. This is named explicitly as the concrete link between
Part XIV's own retention capability and Part XIX's theater-detection requirement, because a fabric that
computed only a single-window health read, with no retained trend history, would have no mechanism
capable of surfacing the theater-drift signature at all — the trend read is not a convenience feature,
it is the load-bearing prerequisite Part XIX's own detection method depends on.

14.9 Edge case: a trend read across a system merge or D2A disposition. Where Part XVIII §18.3's
lineage-preserving rule keeps a merged system's historical usage continuous under its surviving
system's own identity, the trend read inherits the same continuity by construction — a health-profile
field that references the merged system's own contribution (its usage count, its flow confirmation
rate) continues to resolve correctly across the merge boundary because the underlying per-system
records already carry the lineage forward, meaning the trend read requires no special-case handling of
its own for a D2A disposition event, only correct inheritance from the engines beneath it, which is the
concrete payoff of Part XVI's dependency-ordered architecture surfacing at the top of the fabric's own
synthesis stack.

FINAL LAW — PART XIV. Institutional health is not a number; it is a structured pattern across eleven
independently evidenced facets, and the single most dangerous thing this observatory could do is
collapse that pattern into a score an Owner could feel reassured by without ever seeing the specific,
correlated strain the score was computed to hide.

PART XV — THE TELEMETRY-BEFORE-CLAIMS CONTRACT
===============================================================================================

15.1 The single rule every other Part in this dataset presupposes. Parts III through XIV each produce
findings — a DEAD verdict, a dark-matter flag, a confirmation rate, a bottleneck classification, a
health read. None of those findings is trustworthy unless one prior discipline holds without
exception: no metric this dataset, or any consumer of this dataset's engines, emits may exist without
a traceable (SOURCE, VALUE) pair — a named producing mechanism and the specific observation that
generated the number. This Part, DEFINED here as the TELEMETRY-BEFORE-CLAIMS CONTRACT, states that
rule as a binding, checkable discipline rather than an implicit assumption, because every prior Part
in this dataset has relied on it silently, and a fabric whose own foundational rule is never written
down is a fabric one careless implementation choice away from becoming exactly the observability
theater Part XIX warns against.

15.2 The contract, stated precisely. NORMATIVE: any claim of the form "system X has property/verdict
Y" that this fabric emits, or that any consumer of this fabric's engines re-asserts, MUST carry a
(SOURCE, VALUE) pair where SOURCE names the specific engine, edge type, and observation window that
produced VALUE, and VALUE is either a directly OBSERVED figure (Part III's bus-sourced counts, Part
IX's cost figures read from CO's telemetry) or a DERIVED figure whose derivation chain is itself
traceable to OBSERVED figures (Part IV's verdicts, computed from Part III's records). A claim with no
SOURCE — a number simply asserted, a verdict rendered with no cited engine, a health-read finding
presented without its provenance record (Part XII) — is, under this contract, not a metric at all; it
is an unsupported assertion wearing a metric's grammar, and this dataset's own discipline requires it
be marked UNRESOLVED (per IAS-F1 Article V) rather than silently treated as fact.

15.3 Why the contract is named "before claims" rather than "with evidence." The naming is deliberate:
a discipline that requires evidence ALONGSIDE a claim still permits the claim to be drafted first and
justified after, which is precisely the sequence that produces retrofitted, cherry-picked evidence —
a claim is decided, then a supporting (SOURCE, VALUE) pair is found or manufactured to back it.
TELEMETRY-BEFORE-CLAIMS inverts the sequence structurally: the (SOURCE, VALUE) pair must exist and be
queried FIRST, and the claim is whatever that query honestly returns, including a null or UNRESOLVED
result. (DERIVED, generalizing IAS-F1's own claimed-until-observed discipline, Part III §3.3 of that
dataset, from the ensemble-state level to the individual-metric level, and making the sequencing
requirement explicit in a way F1's own text leaves implicit.)

15.4 The contract's three enforcement points within this dataset's own architecture. First, at
EMISSION: every engine specified in Parts III-XIV is required, by this Part's own discipline, to
attach its source record (which Control Bus event, which reconciliation pass, which flow-observatory
record) to every value it produces — this is not a separate mechanism but a restatement of a
requirement already implicit in every worked example this dataset has presented, made explicit here so
it cannot be silently dropped in a future implementation. Second, at AGGREGATION: Part XIV's health
observatory, when compiling eleven facets into one profile, must preserve each facet's own SOURCE
chain rather than replacing it with a generic "per E1" citation — a health-read consumer who wants to
verify one alarming finding must be able to walk back to the specific bus event or reconciliation pass
that produced it, not merely trust the fabric's own aggregate authority. Third, at CONSUMPTION: any
downstream IAS dataset (C1, D1, G1, per Part XXII) that re-asserts an E1 finding in its own text or
decision record inherits the obligation to preserve the (SOURCE, VALUE) pair rather than laundering it
into an unattributed fact of its own — this is the specific mechanism that prevents the evidence-
inflation failure mode of Part VIII from occurring not just within E1's own graph but across the
family's own dataset boundaries.

15.5 Worked example: a claim that fails the contract, and its correction. Consider a hypothetical
draft finding: "the ensemble's knowledge supply chain is underperforming." Under the Telemetry-Before-
Claims Contract, this sentence, as drafted, is non-compliant — it names no source engine, no
observation window, no specific figure. The corrected form, compliant with the contract: "producing
system Phi-04's yield ratio (Part XIII §13.3, computed from 140 produced assets against 19 confirmed-
consumption events, ninety-day window ending [date], source: Control Bus consumption-confirmation
log) is fourteen percent, below the ensemble's [N]-system median of [M] percent for the same window."
The corrected form is longer, but every clause traces to a specific engine and a specific observation
— the difference between an institutional claim and an institutional finding.

15.6 The contract's relationship to the UNRESOLVED default. Where §15.2 requires a claim with no
traceable source be marked UNRESOLVED rather than asserted, this Part inherits, without modification,
IAS-F1's own Part IV §4.6 discipline ("a fabricated 'medium' cost is worse than an honest blank") and
Part XII §12.5 of that dataset's provenance-honesty requirement, applying both, by name, to every
metric this dataset itself produces. (NORMATIVE: this is the specific discharge of this dataset's own
obligation, stated in Part II §2.5, to inherit F1's evidence vocabulary rather than reinventing a
looser one for its own convenience — a temptation this Part names explicitly because a fabric under
schedule pressure to "show results" is exactly the actor most likely to loosen its own evidence
standard first.)

15.7 Edge case: a claim traceable to a source that is itself stale. A (SOURCE, VALUE) pair can satisfy
§15.2's structural requirement — a named engine, a named observation window — while resting on a
source record that has gone stale under IAS-F1's own Part IV §4.3 staleness discipline. The contract's
resolution is that a stale source does not disqualify the claim from having a SOURCE (the traceability
requirement is satisfied), but the claim itself inherits the source's staleness flag and must carry it
forward — a metric reported without its staleness flag, even where its SOURCE citation is otherwise
complete, is a PARTIAL compliance with this contract, not full compliance, because staleness is itself
part of the honest value of the claim, not a separate concern.

15.8 Why this contract is the fabric's most important export to the estate beyond E1's own engines. Of
everything this dataset produces, the Telemetry-Before-Claims Contract is the one artifact this
dataset expects, HYPOTHESIZED pending Part XXI's evaluation, to be adopted BEYOND E1's own engines —
as a general discipline any PP system, existing or future, could apply to its own claim-making, not
merely to observability findings. This is named explicitly because it is the clearest instance in this
dataset of a second-order capability (per IAS-F1 Part VII) producing a norm rather than a mechanism —
E1 does not enforce this contract on other systems (it holds no authority to, per Part II §2.6), but
by naming and modeling the discipline consistently across fourteen prior Parts, it makes the norm
available for voluntary adoption, which is the precise, non-coercive mechanism by which a second-order
capability strengthens an ensemble's own epistemic culture without claiming authority it was never
granted.

FINAL LAW — PART XV. A metric with no traceable source is not a conservative estimate and not a
rounding-up of a probable truth; it is an assertion wearing a number's clothing, and the fabric's own
credibility — the only asset an observability system actually has — depends entirely on never
publishing one.

PART XVI — ARCHITECTURE: ENGINES, SUBENGINES, AND THE FABRIC'S OWN TOPOLOGY
===============================================================================================

16.1 Why this dataset states its own architecture explicitly, rather than leaving it implicit across
fourteen prior Parts. Parts III through XIV each specified one engine; this Part, DEFINED here,
assembles those fourteen specifications into one coherent architectural statement — the DEPENDENCY
ORDER among E1's own engines, which of them are PRIMARY (reading raw evidence) versus SYNTHESIS
(reading other engines' output), and which of IAS-F1's own object classes (system, subsystem,
component, per that dataset's Part II) each engine corresponds to. This matters concretely because a
future implementer building against this specification needs to know not just WHAT each engine does
but in what ORDER they must be built, since several engines this dataset defines cannot function
correctly if built before their own dependencies exist.

16.2 The two-tier engine classification. TIER-ONE (PRIMARY) engines read the estate's raw evidence
streams directly — the Control Bus event log, incidental evidence (file-modification patterns, log
lines), and IAS-F1's own property and relation-graph records — without depending on any other E1
engine's output. Three engines are Tier-One: the Usage Observatory (Part III), whose input is the
Control Bus and incidental-evidence streams directly; the four Flow Observatories (Parts VI-IX,
counted here as one Tier-One family sharing the same CONFIRMED/DELIVERED-UNCONFIRMED evidence
discipline but reading four distinct payload types), whose input is likewise the Control Bus's
recorded edges; and the Latency Observatory (Part X), whose input is timestamp data on the same edges
the flow observatories read. TIER-TWO (SYNTHESIS) engines consume one or more Tier-One engines' output
as their own primary input and produce no new raw-evidence reads of their own. Six engines are Tier-
Two: the Dead-Capability Detector (Part IV, consuming Usage); the Dark-Matter Detector (Part V,
consuming incidental evidence directly but ALSO consuming Usage's registry-matching logic, making it a
hybrid case named explicitly in §16.4); the Bottleneck Map (Part XI, consuming Usage, all four Flow
facets, and Latency together, per Part XI §11.7's own explicit dependency statement); Decision
Provenance (Part XII, consuming Decision Flow and Evidence Flow); the Knowledge-Consumption Tracker
(Part XIII, consuming Knowledge Flow); and the Institutional Health Observatory (Part XIV, consuming
all thirteen prior engines).

16.3 Why the dependency order matters for a real build, stated as a build-sequence specification (per
Article VIII, specification-level only). A correct build sequence, DERIVED directly from §16.2's
tiering, is: (1) establish the addressing and relation-graph substrate this dataset consumes from
IAS-F1 (a precondition, not an E1 engine itself); (2) build the three Tier-One families (Usage, the
four Flow facets, Latency) in parallel, since none depends on another; (3) build the Dead-Capability
Detector and the Knowledge-Consumption Tracker, each of which depends on exactly one Tier-One family;
(4) build Decision Provenance, which depends on two Tier-One families (Decision Flow and Evidence
Flow); (5) build the Bottleneck Map, which depends on the widest Tier-One base (Usage plus all four
Flow facets plus Latency); (6) build the Dark-Matter Detector, whose hybrid dependency (§16.4) makes it
usable in a degraded, lower-confidence mode as early as step 2 but reaches full designed confidence
only once Usage's registry-matching logic (step 2/3) is mature; (7) build the Institutional Health
Observatory last, as the synthesis of everything else. A build that attempted the Health Observatory
or the Bottleneck Map before their Tier-One dependencies existed would produce, at best, a hollow
shell reporting on evidence streams that do not yet exist — the exact Scaffold Illusion failure this
corpus's own governing discipline (external to this dataset, per the estate's Reality Contract) warns
against in implementation contexts generally, named here as an architectural reason to respect the
build order rather than merely a general caution.

16.4 The Dark-Matter Detector's hybrid classification, explained. Named as a genuine architectural
nuance rather than smoothed over: Part V's detector reads incidental evidence directly (a Tier-One
characteristic) but its core comparison — is there a REGISTERED system accounting for this observed
pattern — depends on the Usage Observatory's own registry-matching logic (a Tier-Two characteristic,
since it reads Usage's own definition of what counts as a "registered, addressable system" per Part
III). This dataset resolves the classification honestly rather than forcing it into one tier: the Dark-
Matter Detector is TIER-ONE-AND-A-HALF, capable of a degraded standalone mode (flagging recurring
unmatched patterns without a mature registry to check against, useful for early bring-up) and reaching
its full designed function only once the registry-matching dependency is satisfied — a distinction a
future implementer needs to know explicitly rather than discover by a failed early deployment.

16.5 Cross-cutting infrastructure shared by every engine, named once rather than repeated per Part.
Three cross-cutting concerns apply uniformly across all fourteen engines and are specified here rather
than re-derived in each Part: the EVIDENCE-LEVEL TAGGING requirement (every value carries its Article
V level, per Part XV's contract); the OBSERVATION-WINDOW PARAMETERIZATION requirement (every engine's
output is scoped to a stated window, never an implicit "all time" default, per Part III §3.3's
explicit reasoning); and the STABLE-ID ANCHORING requirement (every system-scoped record keys to
IAS-F1's Part IX addressing triple, never to a file path or a display name, so that engine output
survives the refactors IAS-F1's own addressing model exists to survive). These three requirements are
the specific, checkable content of what it means for E1's fourteen engines to be "one fabric" rather
than fourteen independently-built tools that happen to share a directory.

16.6 What E1 explicitly does not build, stated as an architectural boundary matching Part II §2.6's
authority boundary. E1 builds no dashboard rendering layer, no alerting or notification mechanism, no
retirement or allocation executor, and no automated remediation of any finding it produces — every
engine in this Part-sequence terminates at a RECORD or a FLAG, never at an ACTION. (NORMATIVE,
restating Article VIII's specification-only mandate at the architecture level: this dataset specifies
what a fabric implementation would need to compute and how the pieces depend on one another; it does
not specify, and does not imply the existence of, any executing system that acts on the computed
findings — that translation gap is named explicitly in this dataset's closing Dataset-to-Runtime
section, and closing it is future-mission work, not a gap in this dataset's own completeness.)

16.7 The fabric's own resource footprint, and why this Part addresses it rather than deferring it
entirely to Part IX. A fourteen-engine fabric reading the full Control Bus event log, the incidental-
evidence stream, and IAS-F1's own relation graph on a recurring cadence is not a free measurement — it
consumes real cognitive capital (per IAS-F1 Part IV §4.2's own COST property), and Part II §2.7
already established that E1's own engines are, themselves, addressable systems subject to Part IX's
resource-flow observatory applied reflexively. This Part names the architectural consequence directly:
the widest-dependency engines (the Bottleneck Map and the Institutional Health Observatory, per §16.2's
own tiering) are also the most resource-intensive to recompute, because each re-reads the full output
of every engine beneath it rather than a narrow slice, meaning a real implementation's own
reconciliation CADENCE (how often each engine's output is refreshed) should, as a NORMATIVE design
guidance rather than a numeric commitment this dataset is not positioned to make, run Tier-One engines
on a tighter cadence than Tier-Two synthesis engines, since the synthesis engines' own value does not
meaningfully improve from being recomputed more often than their slowest Tier-One dependency changes.

16.8 Scaling consideration: engine cost growth as the estate itself grows. As the addressable system
count grows beyond the seventy-plus OBSERVED at this dataset's own authorship (Part I §1.1), the
Bottleneck Map's graph-centrality computation (Part XI §11.3) and the Dark-Matter Detector's pattern-
recurrence clustering (Part V §5.3) are the two engines this dataset expects, HYPOTHESIZED pending
real implementation experience, to show the steepest cost growth relative to estate size, because both
operate over structures (the full relation graph; the full incidental-evidence stream) that grow
faster than linearly with system count in a densely-interconnected estate. This dataset does not
specify a mitigation (sampling, incremental recomputation, or graph-partitioning are each plausible
implementation-level responses, per Article VIII's specification-only boundary) but names the scaling
risk explicitly so a future implementer budgets for it rather than discovering it only once the
estate's own growth makes a full recomputation cadence impractical.

FINAL LAW — PART XVI. A fabric of fourteen engines is not fourteen tools that happen to read the same
estate; it is one dependency-ordered structure where the synthesis engines are provably downstream of
the primary ones, and a future builder who does not respect that order will build a health observatory
that reports on evidence streams that were never wired beneath it.

PART XVII — A COMPREHENSIVE WORKED EXAMPLE: ONE SYNTHETIC QUARTER, END TO END
===============================================================================================

17.1 Why a single, connected narrative is needed beyond the per-engine worked examples. Parts III
through XIV each carried their own worked example, illustrating one engine in isolation on synthetic
data invented for that Part alone. This Part exists to do something those examples deliberately did
not: trace ONE synthetic institutional quarter through all fourteen engines in sequence, showing how a
finding at one engine becomes an input to the next, and how the fourteen separate mechanisms this
dataset has specified actually compose into the single fabric Part XVI's architecture claims they are.
Every figure in this Part is synthetic, illustrative only, and does not assert real measurements of any
actual Claude Power Pack module, per this dataset's standing convention.

17.2 The synthetic estate snapshot at quarter open. The synthetic estate carries seventy-eight
addressable systems at the start of the observation quarter, four of them sealed in the final two
weeks of the prior quarter (falling under Part III §3.6's new-system edge case for roughly the first
third of the current window). The Usage Observatory's Tier-One read (Part III) produces a usage
distribution showing the expected long tail: a top decile accounting for seventy-six percent of total
bus-sourced invocations, a middle band of moderately-used systems, and a bottom dozen systems with
sparse-to-zero invocation counts. The Flow Observatories (Parts VI-IX) and Latency Observatory (Part
X), reading the same window, produce their own independent Tier-One records — a knowledge-flow
confirmation rate estate-wide of sixty-two percent, a decision-flow as-stated-adoption rate of
seventy-nine percent, an evidence-integrity ratio estate-wide of eighty-eight percent, and a latency
baseline with no systemic drift outside three flagged systems.

17.3 The Tier-Two synthesis, walked in dependency order. Following Part XVI §16.3's build-and-read
sequence: the Dead-Capability Detector (Part IV), reading the bottom dozen from §17.2, resolves nine of
the twelve to DORMANT (usage in the PRIOR window, none in the current one — a seasonal pattern
consistent with Part IV §4.5's cadence exception once their declared cadences are checked) and three to
DEAD outright, past grace period with no incidental evidence. The Knowledge-Consumption Tracker (Part
XIII), reading the sixty-two-percent estate-wide confirmation rate at the per-producing-system
granularity, isolates two producing systems responsible for a disproportionate share of the
unconfirmed population — one with a routing gap (high no-delivery count) and one with a relevance
ambiguity (high delivered-unconfirmed count), the same two-cause decomposition worked in Part XIII
§13.4 now shown occurring simultaneously within one quarter's real aggregate rather than as an isolated
illustration. Decision Provenance (Part XII), sampled across eleven notable decisions made in the
quarter, resolves nine to HIGH-CONFIDENCE, one to MEDIUM-CONFIDENCE (a branch terminating at an
observation boundary rather than a genuine root), and one to NO-PROVENANCE — a capital-allocation
decision with no traceable inbound edge at all, flagged for governance review rather than assumed
benign.

17.4 The Bottleneck Map's synthesis, and the correlation it exposes before Part XIV even runs. Reading
Usage, all four Flow facets, and Latency together (per its Tier-Two dependency, Part XVI §16.2), the
Bottleneck Map identifies one CONVERGENT bottleneck — a system with the highest DEPENDS-ON inbound
count in the synthetic estate, showing a twenty-nine percent queue-latency rise over the quarter with
no coincident capability-signature change (ruling out Part X §10.6's legitimate-complexity exception)
— and one STARVED region, a four-system cluster with dense internal DEPENDS-ON edges but bottom-decile
inbound flow from the rest of the graph. Cross-referenced against the Dark-Matter Detector's own
pattern-recurrence finding for the same quarter (an eight-times-recurring unregistered workaround,
below Part V's conservative threshold at week six but crossing it by week eleven), the starved region
and the dark-matter finding share a partial system overlap — two of the four starved-region systems
also appear among the dark-matter finding's structurally-similar unmatched evidence — a genuinely new
observation this comprehensive trace surfaces that no single per-engine worked example in Parts III-XIV
could have shown, because it depends on reading two engines' outputs against each other rather than
either in isolation.

17.5 The Health Observatory's final synthesis for the quarter. Compiling all thirteen prior engines'
quarter-end output per Part XIV's schema, the Health Observatory's structured profile shows: usage
concentration at seventy-six percent (top decile), three DEAD systems and nine DORMANT, one dark-matter
finding crossing threshold at week eleven, knowledge-flow confirmation at sixty-two percent with two
named producing-system outliers, decision-flow adoption at seventy-nine percent with no INVARIANT-
THREE practice violations detected, evidence-integrity at eighty-eight percent estate-wide, three
latency-drift flags (one explained by legitimate complexity increase, two unexplained), one CONVERGENT
bottleneck and one STARVED region with a partial dark-matter overlap, provenance at nine HIGH / one
MEDIUM / one NO-PROVENANCE among sampled decisions, and a yield distribution with two named outlier
producers. The correlation layer (Part XIV §14.4) elevates exactly one CONVERGENT-STRAIN flag — the
bottleneck system, which also shows one of the two unexplained latency drifts — and separately notes
the starved-region/dark-matter overlap of §17.4 as a second, distinct correlation type this dataset had
not previously named: a STRUCTURAL-BLINDSPOT correlation, where a region of the graph is simultaneously
underserved by registered flow AND host to unregistered recurring work, suggesting the two facts may
share one root cause (a genuine capability gap the estate has been informally patching around) rather
than being two unrelated findings.

17.6 What this comprehensive trace demonstrates that no isolated example could. Three things, named
explicitly. First, TEMPORAL SEQUENCING: findings that look isolated at week six (the dark-matter
pattern below threshold) become actionable by week eleven, demonstrating why every engine's output
must be re-queryable across the observation window rather than computed once and frozen. Second,
CROSS-ENGINE CORRELATION beyond what Part XIV's own worked example (§14.5, a simpler single-system
correlation) showed: the STRUCTURAL-BLINDSPOT pattern of §17.5 required reading the Bottleneck Map
against the Dark-Matter Detector directly, a combination Part XIV's schema accommodates but does not
itself compute by default, suggesting the correlation logic of §14.4 is extensible and this dataset's
own specification should be read as a floor, not a ceiling, on which correlations a real
implementation surfaces. Third, GOVERNANCE ROUTING: of the quarter's findings, exactly one (the
NO-PROVENANCE capital-allocation decision) required routing outside the standard C1/D1/G1 consumers
this dataset's Compounding Graph edges name, because a decision with no traceable input is, in the
estate's own governance terms, a different kind of problem than a slow system or a dead capability —
this dataset notes the routing gap honestly rather than inventing a consumer for it that no governing
document currently names.

17.7 What the quarter's trend read would show against a hypothetical prior quarter. Applying Part XIV
§14.7's trend-reading discipline retrospectively to this synthetic quarter against a hypothetical prior
one: the usage concentration figure (seventy-six percent top-decile) would be read as an IMPROVEMENT or
DETERIORATION only relative to the prior window's own concentration figure, not against any absolute
estate-wide constant — this dataset deliberately does not invent a prior-quarter baseline for this
worked example, because doing so would manufacture a comparison this synthetic trace has no genuine
grounding for, and a fabricated trend direction would violate the same UNRESOLVED-over-guessed
discipline this dataset applies throughout (Part III §3.6, Part XVIII §18.2). What this Part can state
honestly is the STRUCTURE the trend read would apply, not the direction it would find: each of the
eleven profile fields from §17.5 would carry its own directional comparison, computed independently,
with the three DEAD verdicts and the CONVERGENT-STRAIN correlation flag receiving particular scrutiny
under Part XIX's theater-drift check, since a fabric under pressure to present a clean quarter would be
most tempted to soften exactly those two findings in any narrative summary built on top of this trace.

17.8 The comprehensive trace as a template, not a forecast. This Part's synthetic quarter should be
read by a future implementer as a TEMPLATE for what a real quarterly cycle through this dataset's
fourteen engines looks like structurally — the dependency order, the kinds of cross-engine correlation
worth watching for, the shape of a governance-routing gap — rather than as a forecast of what any real
estate's own numbers will show. Every figure in Parts 17.2 through 17.6 is synthetic and invented for
illustration, and this dataset's own Reality Contract discipline requires that distinction be restated
here, at the close of the one Part most likely to be mistaken for an empirical claim precisely because
of how concrete and connected its narrative reads.

FINAL LAW — PART XVII. Fourteen engines proven correct in isolation are not yet proven to compose; only
a trace that walks a single synthetic quarter through all fourteen, in dependency order, watching one
engine's flag become the next engine's input, demonstrates that the fabric is one fabric and not
fourteen adjacent tools — and even that trace surfaced a correlation type (STRUCTURAL-BLINDSPOT) this
dataset's own per-Part specifications had not individually anticipated, which is itself evidence that
composition, not enumeration, is where a fabric's real value is found.

PART XVIII — EDGE CASES AND ADVERSARIAL CASES AT THE ENSEMBLE SCALE
===============================================================================================

18.1 Why this Part exists beyond the per-engine edge and adversarial cases already specified. Parts III
through XIV each named at least one edge case and one adversarial case scoped to that engine — the
new-system grace period, heartbeat-gaming, the seasonal-cadence exception, cost-hiding through
subinvocation, and others. This Part addresses cases that arise only at the ENSEMBLE scale, across
multiple engines or across the fabric's own lifecycle, which no single-engine treatment could surface
because each is, by construction, a property of the whole fabric rather than of one measurement.

18.2 Edge case: the cold-start estate with no observation history. A fabric deployed against an estate
with no prior observation window has no baseline for Part X's drift detection, no PRIOR window for
Part IV's DORMANT classification, and no historical yield distribution for Part XIII's median
comparison. The resolution, DERIVED from the same UNRESOLVED-over-guessed principle this dataset has
applied throughout (Part III §3.6, Part IV §4.6 of F1): a cold-start fabric's first observation window
produces ONLY absolute figures (raw counts, not drift or dormancy classifications, which structurally
require a comparison point that does not yet exist) and marks every comparative field UNRESOLVED
rather than defaulting to a synthetic zero baseline that would make the first window's systems look
artificially anomalous the moment any activity is observed. A second window is the earliest point at
which Part IV's three-state taxonomy and Part X's drift detection can be meaningfully computed, and
this dataset states that lag explicitly as a known limitation rather than a solvable gap.

18.3 Edge case: a system merge mid-observation-window. When D2A resolves a DUPLICATES relation (IAS-F1
Invariant Two) with a MERGE-INTO-FAMILY disposition during an active observation window, the merged-
away system's usage history, flow records, and latency baseline do not simply vanish — they become
part of the surviving system's own historical record, per a LINEAGE-PRESERVING rule this dataset
specifies: every engine's historical query, from the moment of merge forward, resolves both the
surviving system's stable ID and the merged-away system's prior stable ID (IAS-F1 Part IX's addressing
triple, which the merged system retains as a HISTORICAL identity even after ceasing to be independently
addressable) as one continuous record. Without this rule, a merge would produce a false DROP in the
surviving system's own apparent usage trajectory at exactly the moment institutional consolidation
should show a healthy continuity, an artifact this dataset explicitly guards against because it would
otherwise penalize the exact D2A disposition (MERGE) Article I's supreme law encourages.

18.4 Adversarial case: coordinated multi-engine gaming. Parts III through XIV each name a single-engine
gaming defense (heartbeat-gaming, fragmentation, cost-hiding, authority laundering). A more
sophisticated adversarial actor could combine several simultaneously — invoking a system just often
enough to avoid a DEAD verdict (Part IV) while ALSO ensuring those invocations show CONFIRMED
knowledge-flow (Part VI) by pre-arranging a trivial downstream behavior change, defeating both
single-engine defenses at once with a coordinated, low-effort pattern. This dataset's defense against
the coordinated case is structural rather than per-engine: the Health Observatory's correlation layer
(Part XIV §14.4) is specifically positioned to catch what individual engines cannot, because a
coordinated gaming pattern, even where each individual engine's own defense is defeated, tends to
produce a STATISTICALLY UNUSUAL correlation signature — invocations and confirmations clustering with
unnaturally low variance, structurally identical payload shapes repeating (the same LIVE-SUSPECT
signature named in Part IV §4.6, generalized here to a cross-engine pattern) — which the correlation
layer is built to surface as a named finding type even where it cannot yet prove intent. (HYPOTHESIZED:
this defense's real-world effectiveness against a genuinely sophisticated coordinated actor is not
established by this dataset's synthetic reasoning alone and is named as a candidate for Part XXI's
red-team evaluation protocol, consistent with IAS-F1's own Protocol Three precedent, Part XVI §16.4 of
that dataset.)

18.5 Adversarial case: institutional pressure to suppress an unflattering finding. The most consequential
adversarial case this dataset names is not technical gaming of the metrics but SOCIAL pressure on
whoever operates the fabric to withhold, soften, or delay a finding that reflects poorly on a system's
owner, a recent decision, or an ongoing initiative — a NO-PROVENANCE flag on a high-visibility decision
(§17.3's worked instance), a CONVERGENT-STRAIN correlation on a system someone has publicly championed,
a DEAD verdict on a recently-celebrated build. This dataset's defense is structural rather than
behavioral, because a norm alone cannot resist genuine institutional pressure: every finding this
fabric produces carries, by Part XV's Telemetry-Before-Claims Contract, its own traceable (SOURCE,
VALUE) pair independent of any narrative wrapped around it, meaning a suppressed or softened PRESENTATION
of a finding does not alter the underlying record — the raw evidence remains queryable by anyone with
access to the fabric's own engines, and a later audit can always reconstruct what the honest finding
was regardless of how it was initially reported. (NORMATIVE: this is the specific, non-negotiable
reason Part XV's contract requires the SOURCE be preserved at every consumption point, Part XV §15.4 —
not merely as a data-quality discipline but as an integrity defense against exactly this adversarial
case.)

18.6 Edge case: partial instrumentation across a heterogeneous estate. Not every system in a real
estate will have equally rich Control Bus integration — older systems, systems built before IAS-A2's
own sealing, or systems maintained outside the estate's primary tooling may generate sparse or no bus
events, relying entirely on Part III §3.2's incidental-evidence channel. A fabric that silently treated
sparse-instrumentation systems identically to richly-instrumented ones would systematically misjudge
the former as less active than they truly are, purely as an artifact of instrumentation coverage rather
than real institutional behavior — a MEASUREMENT-COVERAGE BIAS this dataset names explicitly. The
resolution, consistent with every prior evidence-honesty discipline in this dataset, is that every
per-system record (Part III §3.3) carries an INSTRUMENTATION-COVERAGE field alongside its usage
figures, and any downstream comparison (a ranking, a concentration figure) that does not account for
coverage disparity is flagged, by this dataset's own discipline, as a comparison of unequal confidence
rather than a clean ranking — a caution this dataset expects IAS-C1's portfolio math, in particular, to
respect when ranking systems whose instrumentation maturity differs.

18.7 Edge case: the fabric observing itself. E1's own fourteen engines are, per IAS-F1's ontology,
themselves systems, addressable and subject to the same usage, flow, and health measurement every
other system in the estate receives — including from E1's own engines applied reflexively. This is not
a paradox but a genuine, useful case: a Usage Observatory record for the Institutional Health
Observatory itself (Part XIV) would show who consumes E1's own health reads, and a DEAD verdict on one
of E1's own fourteen engines (if, for instance, the Dark-Matter Detector's findings were never actually
consumed by any downstream governance process) would be a legitimate, and specifically pointed, finding
about the fabric's own institutional value — this dataset states plainly that it does not exempt itself
from its own measurement, and a future fabric implementation that quietly special-cased E1's own
engines out of the usage observatory's scope would be violating this dataset's own stated discipline.

FINAL LAW — PART XVIII. The edge cases that matter most are not the ones a single engine's own logic
anticipates but the ones that only appear when the whole fabric, and the institution around it,
are considered together — cold starts with no baseline, merges that must preserve lineage, coordinated
gaming no single defense catches alone, and the social pressure to soften a true finding — and a
fabric that has not named these has not yet earned the trust its own findings ask the estate to place
in it.

===============================================================================================
PART XIX — FAILURE MODES: OBSERVABILITY THEATER AND WATCHED-METRIC GAMING
===============================================================================================

19.1 Why failure modes get their own Part rather than remaining scattered across Parts III-XVIII.
Every prior engine-specific Part named at least one local failure risk — this dataset has not been
silent about its own fragility. This Part exists to name the two SYSTEMIC failure modes that recur
across every engine rather than belonging to any one of them, because a systemic failure mode is not
caught by strengthening any single engine's own defense; it requires a different kind of vigilance,
aimed at the fabric's own operating culture rather than at any one measurement.

19.2 OBSERVABILITY THEATER, defined precisely. OBSERVABILITY THEATER, DEFINED here, is the condition
in which a fabric produces abundant, confident-looking output — dashboards, ranked lists, health
reads, correlation flags — that satisfies the INSTITUTIONAL APPEARANCE of visibility without
satisfying its SUBSTANCE, because the underlying evidence chains have quietly degraded, gone stale, or
been replaced by convenient defaults nobody is checking. Theater is distinguished from honest partial
coverage by one test: an honestly partial fabric MARKS its gaps (per Part XV's UNRESOLVED discipline
and Part XVIII §18.6's coverage-bias flag), while a fabric drifting into theater PRESENTS its gaps as
resolved — a health read that shows a clean, all-green profile not because the ensemble is genuinely
healthy but because the engines producing the unflattering findings have silently stopped running, or
their outputs have silently stopped being checked against fresh evidence.

19.3 The specific mechanism by which theater develops, traced across this dataset's own engines. Theater
rarely arrives as a single decision; it accretes through small, individually defensible shortcuts. A
reconciliation cadence (IAS-F1 Part IV §4.3) is quietly extended under schedule pressure, so a
STALENESS flag that should have fired does not. A DELIVERED-UNCONFIRMED count (Part VI §6.3) is
reclassified as CONFIRMED without a genuine provenance link, because "it's probably being used" feels
close enough to save the effort of tracing it. A NO-PROVENANCE finding (Part XII §12.5) is quietly
omitted from a presented health read because it is inconvenient and its absence is unlikely to be
noticed. None of these, in isolation, looks like a lie — each looks like a reasonable, minor
convenience — but their accumulation is exactly the gap between APPEARANCE and SUBSTANCE §19.2 defines,
and this dataset names the accretive pattern explicitly because a fabric operator watching for one
dramatic falsification will miss the slow accumulation of a hundred small ones.

19.4 The detection signature of observability theater, and why it is structurally hard to self-detect.
A fabric drifting into theater shows a specific, checkable signature: a rising share of CONFIRMED,
OBSERVED, HIGH-CONFIDENCE findings across the health profile (Part XIV §14.3) with a FALLING or FLAT
rate of new STALENESS, UNRESOLVED, and NO-PROVENANCE flags — an ensemble that appears to be getting
healthier and better-instrumented at the same time is, empirically, a rarer and more suspicious pattern
than one where instrumentation gaps and health findings both persist honestly over time, because real
institutions do not typically improve on every evidenced axis simultaneously. This detection signature
is, deliberately, itself a candidate metric this dataset names for the fabric's own self-audit (Part
XXI), because theater is structurally hard to self-detect from WITHIN the fabric's own engines — an
engine that has drifted into theater is, by definition, no longer trustworthy evidence about its own
drift, which is why this dataset recommends the signature be checked by an INDEPENDENT process (a
periodic external audit, per Part XXI's evaluation protocol) rather than folded into the fabric's own
routine health read.

19.5 WATCHED-METRIC GAMING, defined precisely, and its relationship to Goodhart's Law. WATCHED-METRIC
GAMING, DEFINED here, is the condition in which an institutional actor — a system's maintainer, an
Owner under delivery pressure, a downstream consumer optimizing for a good-looking number — alters
BEHAVIOR to improve a metric this fabric reports WITHOUT improving the underlying institutional
property the metric was built to proxy for. This is the specific, ensemble-scale instance of the
general principle known informally in the estate's own prior documentation as Goodhart's Law ("when a
measure becomes a target, it ceases to be a good measure"), and every per-engine adversarial case named
across Parts III-XIII — heartbeat-gaming (Part IV §4.6), payload fragmentation (Part VI §6.7), authority
laundering (Part VII §7.6), cost-hiding (Part IX §9.7), yield-ratio starvation (Part XIII §13.5) — is a
specific instance of this general failure mode, named individually where it was first encountered and
named here, once, as the unifying pattern.

19.6 Why every individual per-engine defense named in Parts III-XIII is necessary but not sufficient.
Each per-engine defense (§19.5's list) closes one specific gaming pathway for one specific metric. But
a sufficiently motivated actor, aware that this dataset's own text names each individual defense, could
in principle search for a gaming pathway THIS dataset has not yet anticipated — a structural
possibility this dataset does not claim to have fully closed, per the epistemic honesty Article V
requires. (HONEST ADMISSION, CONTESTED: whether the specific defenses named in this dataset are
JOINTLY sufficient to prevent all forms of watched-metric gaming, or whether they merely close the
specific pathways this dataset's authors were able to anticipate, is not resolved by this dataset's own
reasoning and is named as a standing question for Part XXI's red-team evaluation protocol, rather than
asserted as a closed problem.)

19.7 The general countermeasure this dataset applies across every engine, stated once as the unifying
principle. Read across Parts III-XIII, one design choice recurs in every per-engine defense: preferring
a RATIO or a STRUCTURAL SIGNATURE (a divergence between two independently-sourced figures, a
consumer-list cross-check, a correlation across facets) over a single, directly-optimizable COUNT.
This is not coincidental; it is the deliberate, DERIVED application of a general principle — a metric
computed from TWO OR MORE independently-sourced signals is harder to game than one computed from a
single signal, because gaming the composite requires manipulating multiple, structurally distinct
evidence streams in a consistent direction simultaneously, which is a strictly harder problem for an
adversarial actor than manipulating one stream. Part XX formalizes this principle into the countermetric
design this dataset applies to every headline metric it names.

19.8 Why the fabric's own credibility, once damaged by either failure mode, is expensive to restore.
Both failure modes share a common consequence, named here as the closing reason this Part matters more
than any single engine's own correctness: an observability fabric's entire value rests on being
TRUSTED enough that its findings actually change downstream decisions (per the decision-flow discipline
of Part VII). A fabric caught in observability theater, or a fabric whose metrics are known to be
gameable, loses that trust — and unlike a single wrong finding, which can be corrected by a
reconciliation event, a loss of institutional trust in the fabric's own honesty propagates into every
downstream consumer (IAS-C1, IAS-D1, IAS-G1) treating EVERY future finding with elevated skepticism,
which is a strictly worse outcome than the fabric simply not existing, because a distrusted fabric
still consumes the resource cost of running (Part IX) while returning degraded institutional value.

FINAL LAW — PART XIX. An observability fabric does not fail by producing a wrong number once; it fails
by drifting, one small convenience at a time, into producing confident numbers nobody is checking, or
by becoming a target institutional actors learn to optimize instead of the reality it was built to
reveal — and the only durable defense against either is preferring structural, multi-source signatures
over single, directly-optimizable counts, everywhere, as a standing discipline rather than a one-time
design choice.

PART XX — METRICS AND ANTI-GOODHART COUNTERMETRICS
===============================================================================================

20.1 Why this Part formalizes what Parts III-XIX have already been doing informally. Every prior
engine-specific Part has named at least one metric and, in several cases, a paired countermetric
designed to resist the gaming pattern §19.5-19.7 generalize. This Part collects those metrics into one
formal table-in-prose, states each metric's PAIRED countermetric explicitly, and states the general
design rule (DERIVED from Part XIX §19.7) that governs every pairing this dataset defines, so that a
future implementer building against this specification has one authoritative reference rather than
needing to re-derive the countermetric design from scattered per-engine mentions.

20.2 The general pairing rule. NORMATIVE: no metric this dataset names may be reported, in any
consumer-facing form, WITHOUT its paired countermetric in the same emission — a metric reported alone
is, under this dataset's own discipline, an incomplete and non-compliant output, exactly as Part XII
§12.6 requires a provenance record's confidence grade be mandatory rather than optional. This rule
generalizes that specific requirement to every metric-countermetric pair this Part defines.

20.3 Usage metrics (Part III). PRIMARY: bus-sourced invocation count per system, per window. PAIRED
COUNTERMETRIC: incidental-evidence invocation count, reported alongside, never merged into one blended
figure (Part III §3.2's dual-source discipline) — the pairing exposes a system whose bus-sourced count
alone would look dead but whose incidental evidence shows real use (Sigma-14 from the worked example,
Part III §3.4).

20.4 Dead-capability metrics (Part IV). PRIMARY: DEAD-verdict count, ensemble-wide. PAIRED
COUNTERMETRIC: LIVE-SUSPECT rate as a share of total LIVE verdicts (Part IV §4.7) — a rising DEAD count
alongside a rising LIVE-SUSPECT rate signals genuine retirement pressure combined with gaming pressure
simultaneously, a compound reading a single DEAD count would miss.

20.5 Flow metrics (Parts VI-IX). PRIMARY: confirmation rate per flow facet, per system pair. PAIRED
COUNTERMETRIC: raw event VOLUME reported alongside the ratio (Part VI §6.7's fragmentation defense,
generalized to all four facets) — a rising confirmation rate on a collapsing volume base indicates the
system is being asked to do less, not doing more of what matters, a distinction the ratio alone hides.

20.6 Latency metrics (Part X). PRIMARY: per-component latency (queue, execution, propagation) against
self-relative baseline. PAIRED COUNTERMETRIC: the CONFIRMED-consumption-gated total end-to-end figure
(Part X §10.7's fire-and-forget defense) — a system reporting fast component latencies but a widening
gap to its confirmed-total figure signals a masking pattern the component figures alone would not
surface.

20.7 Bottleneck metrics (Part XI). PRIMARY: centrality score per system (Part XI §11.3). PAIRED
COUNTERMETRIC: the DECLARED AUTHORITY TYPE cross-check (Part XI §11.6) — a rising centrality score on a
system with no declared final authority is a materially different finding than the same rise on a
system whose singular authority is structurally expected to concentrate load, and reporting centrality
alone erases that distinction.

20.8 Evidence-integrity metrics (Part VIII). PRIMARY: evidence-integrity ratio per system (Part VIII
§8.5). PAIRED COUNTERMETRIC: the ABSOLUTE count of INFLATED hops, reported alongside the ratio — a
system with a stable high ratio on a small absolute sample size is a weaker signal than the same ratio
on a large one, and the absolute count prevents a small, cherry-picked sample from producing a falsely
reassuring ratio.

20.9 Knowledge-yield metrics (Part XIII), the sharpest anti-Goodhart case in this dataset. PRIMARY:
yield ratio per producing system (Part XIII §13.3). PAIRED COUNTERMETRIC: the ABSOLUTE confirmed-
consumption count, reported alongside the ratio, specifically because Part XIII §13.5 names this
pairing as resisting the single most tempting gaming pathway in the entire fabric — a producing system
that shrinks its own output to raise its ratio shows a rising ratio against a FALLING absolute count,
a divergence the pairing is built explicitly to make visible, and this dataset treats this specific
pairing as the canonical worked instance of the general rule stated in §20.2.

20.10 Health-observatory metrics (Part XIV), where the pairing rule becomes a structural prohibition
rather than a paired figure. Because Part XIV §14.2 already prohibits a single scalar health score
outright, the "countermetric" for the health observatory is not a second number but the STRUCTURED
PROFILE itself — every field's own paired countermetric (§20.3-20.9) carried forward into the profile
unreduced, which is the specific mechanism by which Part XIV's no-single-score discipline and this
Part's pairing rule are the same design principle applied at two different scales: no single number,
anywhere in this fabric, stands alone without the second figure that would reveal its gaming.

20.11 The one metric this dataset explicitly declines to pair, and why. The Telemetry-Before-Claims
Contract's own compliance rate (what share of this fabric's own emitted claims carry a valid (SOURCE,
VALUE) pair, per Part XV) is deliberately left WITHOUT a paired countermetric in this dataset's own
specification, because pairing it with anything risks implying a tolerable non-compliance rate exists
— this dataset's position, NORMATIVE, is that Telemetry-Before-Claims compliance is binary and
absolute at the level of individual claims (a claim either has a traceable source or it does not,
per §15.2), and the correct ensemble-wide measurement is not a ratio to be balanced against a
countermetric but a hard floor: any non-compliant claim discovered is itself the finding, full stop,
with no accompanying figure that could soften it.

20.12 Worked numeric example: the pairing rule applied end to end using this dataset's own comprehensive
trace. Returning to the synthetic quarter of Part XVII §17.3, the yield-outlier producing system (140
assets, 19 confirmed-consumption) illustrates the full pairing discipline in one concrete case: reported
ALONE, the fourteen-percent yield ratio invites exactly one narrative — "this producer is underperforming
and should produce less until relevance improves" — a narrative that, if acted on by shrinking output to
raise the ratio, is the precise gaming pathway §20.9 names. Reported WITH its paired countermetric (the
absolute confirmed-consumption count of 19, tracked across the following window), a materially different
and more accurate narrative becomes visible: if the following window shows a ratio RISE to twenty-two
percent alongside a confirmed-consumption count FALL to 14, the pairing immediately flags the
divergence §20.9 predicts, correctly identifying output-shrinkage rather than genuine relevance
improvement as the driver — a distinction the ratio alone, reported in isolation across the same two
windows, would have presented as unambiguous good news.

20.13 Why the pairing discipline is cheaper to apply consistently than to apply selectively. A design
that paired only the metrics judged, in advance, to be "most gameable" would require an ongoing editorial
judgment about which metrics deserve the extra reporting overhead — a judgment this dataset declines to
make, because the entire history of Goodhart's Law failures, as informally documented in the estate's own
prior session lessons, shows that the metric judged least likely to be gamed is frequently the one that
eventually is, precisely because it was never watched as closely as the ones flagged high-risk in
advance. Applying the pairing rule UNIFORMLY, to every metric this dataset names without exception
(§20.2's own wording — "no metric... may be reported... WITHOUT its paired countermetric"), is therefore
not merely a stylistic consistency choice; it is the specific design decision that removes the need for
that fallible, forward-looking editorial judgment altogether.

FINAL LAW — PART XX. A metric without a paired countermetric is a target waiting to be gamed, because
every metric this dataset has named can be improved by behavior that does not improve the underlying
reality it proxies for — and the only design discipline that closes that gap, applied uniformly across
fourteen engines rather than invented ad hoc per metric, is to make every reported figure a pair, never
a singleton, except for the one claim this dataset holds to an absolute rather than a balanced
standard: that a metric have a source at all.

PART XXI — EVALUATION PROTOCOL AND BENCHMARKS
===============================================================================================

21.1 Why this dataset specifies its own evaluation rather than merely asserting its mechanisms work.
Per Article VIII, this dataset builds no code and runs no evaluation against the live estate; every
worked example in Parts III-XVIII is synthetic and illustrative, and this dataset does not claim its
own mechanisms have been empirically validated. This Part specifies the PROTOCOL a future
implementation mission would run to validate this dataset's own claims — a direct application of
IAS-F1's own precedent (that dataset's Part XVI, four numbered protocols) applied here to E1's specific
engines, so that "does the fabric actually work" has a checkable answer rather than remaining a
permanent assertion.

21.2 Protocol One — Coverage Retrodiction. Given a stated historical window of the estate's real
activity (once instrumentation exists), run the Usage Observatory (Part III) and Dark-Matter Detector
(Part V) against it, then independently audit a random sample of systems and a random sample of
manually-identified real institutional work (interviews, artifact review, or existing informal records)
to check whether the fabric's OBSERVED usage and dark-matter findings match what independent human
review finds. PASS CRITERION, EXPERIMENTAL: the fabric's dark-matter detector surfaces at least the
majority of independently-identified unregistered recurring work within its conservative recurrence
threshold (Part V §5.3), and its usage observatory's DEAD verdicts (Part IV) do not contradict
independently-confirmed real usage the audit finds. A failure here — the fabric missing real dark
matter an independent audit finds easily, or flagging DEAD a system independent review confirms is
genuinely active — is the single most damaging failure this protocol could surface, because it would
mean the fabric's foundational evidence layer (Parts III-V) cannot be trusted by anything built on top
of it.

21.3 Protocol Two — Countermetric Stress Test. For each metric-countermetric pair specified in Part XX,
construct a synthetic or sandboxed gaming scenario matching the adversarial case named in that metric's
originating Part (heartbeat-gaming for Part IV, fragmentation for Part VI, and so on) and verify the
paired countermetric detects the manipulation — that is, verify the divergence Part XX predicts (rising
ratio, falling absolute count; low component latency, widening confirmed-total gap) actually appears in
the sandboxed data rather than being a plausible-sounding but empirically absent prediction. PASS
CRITERION, EXPERIMENTAL: every one of the nine paired metrics named in Part XX §20.3-20.9 shows its
predicted divergence signature under its corresponding gaming scenario; any pair that fails this test
is a candidate for revision under this dataset's own future-versioning discipline (per IAS-F1 Part
XVII precedent), not evidence the whole fabric is unsound.

21.4 Protocol Three — Red-Team Coordinated Gaming. Following IAS-F1's own Protocol Three precedent
(that dataset's Part XVI §16.4, red-team evaluation of the second-order test), an independent red team
is tasked with defeating as many of this dataset's fourteen engines SIMULTANEOUSLY as possible within a
sandboxed estate, specifically targeting the coordinated-gaming adversarial case named in Part XVIII
§18.4. PASS CRITERION, EXPERIMENTAL: the Health Observatory's correlation layer (Part XIV §14.4)
surfaces at least one anomalous correlation flag in response to the red team's coordinated activity,
even where individual per-engine defenses are each partially defeated — this is a WEAKER pass bar than
"no engine can be gamed," deliberately, because Part XIX §19.6 already admits this dataset does not
claim its per-engine defenses are jointly exhaustive; the correlation layer's own detection is the
fallback this protocol is actually testing.

21.5 Protocol Four — Theater Drift Detection. Over an extended observation period (multiple
consecutive windows), run the observability-theater detection signature named in Part XIX §19.4 (rising
CONFIRMED/OBSERVED/HIGH-CONFIDENCE share alongside falling STALENESS/UNRESOLVED/NO-PROVENANCE rate)
against the fabric's own real operating history, checked by a process INDEPENDENT of the fabric itself,
per Part XIX §19.4's own stated requirement. PASS CRITERION, EXPERIMENTAL: no sustained multi-window
instance of the theater-drift signature is found without a genuine, externally-verifiable improvement
explaining it (a real registry cleanup, a real instrumentation upgrade) — an unexplained instance of
the signature is treated as a finding requiring investigation, not as evidence the protocol itself
failed.

21.6 Benchmark targets, stated as EXPERIMENTAL calibration points rather than committed thresholds. This
dataset names indicative benchmark RANGES for three of its own headline metrics, explicitly marked
EXPERIMENTAL and subject to revision once Protocol One generates real retrodiction data: a knowledge-
flow confirmation rate (Part VI §6.5) below roughly one-third across an entire producing system's
output, sustained across two or more windows, is a candidate threshold for escalation to IAS-B2's
insight-synthesis review; a dark-matter pattern-recurrence count (Part V §5.3) crossing its
conservative threshold is, by construction, always escalated to D2A rather than calibrated to a
severity band, since dark matter's very definition (Part V §5.1) makes any confirmed instance
noteworthy; and a Telemetry-Before-Claims non-compliance rate (Part XV §20.11) above zero, for any
single audited claim, is treated as an immediate finding rather than calibrated against any threshold
at all, consistent with that metric's own stated absolute-standard design. These three benchmark
statements are the ONLY numeric thresholds this dataset commits to, and each is explicitly named
EXPERIMENTAL rather than NORMATIVE, because Article V forbids asserting a calibrated threshold as
settled fact before real retrodiction data (Protocol One) exists to calibrate it.

21.7 Why this dataset declines to specify pass/fail thresholds for the remaining metrics. Every metric
named in Part XX beyond the three in §21.6 is deliberately left WITHOUT a committed numeric threshold in
this dataset's own text, because a threshold asserted without real estate data behind it would be
exactly the kind of guessed, unearned precision Article V and this dataset's own Part IV §4.6 (via F1)
both warn against — a threshold invented to sound authoritative is worse than an honestly-stated
UNRESOLVED calibration gap, and this dataset would rather name the gap than manufacture a number.
(NORMATIVE: threshold-setting for the remaining metrics is explicitly deferred to whichever future
mission runs Protocol One against real data, per Article VIII's implementation-is-later-work boundary.)

21.8 Sequencing the four protocols against the build order Part XVI already specifies. The four
protocols are not intended to run in an arbitrary order once implementation begins; DERIVED from Part
XVI §16.3's own build sequence, Protocol One (Coverage Retrodiction) is the earliest runnable, because
it validates only Tier-One-adjacent engines (Usage, Dark-Matter) that Part XVI's own sequence builds
first, and its results are a precondition for trusting anything Protocol Two tests downstream of them.
Protocol Two (Countermetric Stress Test) follows once the Tier-Two engines carrying the nine named
metric pairs (Part XX §20.3-20.9) are built, since a countermetric cannot be stress-tested before the
engine computing it exists. Protocol Three (Red-Team Coordinated Gaming) requires the Health
Observatory's correlation layer, the LAST engine in Part XVI's own build order, and therefore cannot
run meaningfully before every other engine — and every other protocol — has already been exercised at
least once. Protocol Four (Theater Drift Detection) is explicitly the LONGEST-RUNNING of the four,
requiring multiple consecutive observation windows rather than a single sandboxed run, and this dataset
expects it to be the last protocol to yield a first result even though nothing prevents it from being
STARTED early, in parallel with the other three, once the fabric's earliest engines are live.

21.9 What a failed protocol means for this dataset's own SEALED status, stated honestly rather than
left implicit. Per Article XI, this dataset's completion state is judged on inspected content, and
Article VIII already establishes that no protocol in this Part has been run — meaning this dataset's
own claims about its mechanisms' effectiveness are, in their entirety, EXPERIMENTAL or HYPOTHESIZED
per Article V, never OBSERVED, until a future mission executes Parts XXI §21.2-21.5 against a real
estate. A protocol failure, once run, is not evidence this dataset's authorship was flawed in the sense
of containing an error the authors should have caught — it is the ordinary, expected outcome of a
falsifiable specification meeting reality for the first time, and this dataset's own versioning
discipline (inherited from IAS-F1 Part XVII, per Decision D5 of this dataset's own Decision Log) is the
correct mechanism for absorbing a protocol failure into a revised specification, not evidence the
fabric's foundational design should be abandoned.

FINAL LAW — PART XXI. A dataset that specifies its own evaluation protocol before a single line of
implementing code exists is stating, in writing, exactly what would have to be true for its own claims
to survive contact with the real estate — and the honest calibration gap this Part leaves open on most
of its own metrics is not a weakness in the specification, it is the specification refusing to claim a
precision it has not yet earned.

PART XXII — INTEGRATION: FEEDING IAS-C1, IAS-D1, IAS-G1, AND THE FABRIC'S OWN EVOLUTION
===============================================================================================

22.1 Why integration is specified last, and why it closes rather than opens the measurement sequence.
Every prior Part has named, in passing, which downstream IAS dataset consumes a given finding. This
Part assembles those scattered references into one authoritative statement of E1's outbound edges,
per the Compounding Graph's own admission requirement (Article IV: "a dataset with no inbound and no
outbound edge is an orphan"), and closes with the fabric's own evolution path — how E1 itself is
expected to change as the estate it observes changes.

22.2 The edge to IAS-C1 (Capability Portfolio), specified precisely. Per the Compounding Graph's stated
edge ("IAS-E1 usage / dead-capability / dark-matter → IAS-C1 → allocation"), E1 supplies IAS-C1 with:
the Usage Distribution and Concentration figures (Part III §3.5), which IAS-C1's portfolio math treats
as an input alongside its own ROI and reuse-horizon calculations, never as a ranking E1 itself renders;
the DEAD-verdict list (Part IV), which IAS-C1 weighs as one input into allocation-versus-retirement
decisions, alongside IAS-D1's own ecology judgment; the Resource-Flow cost-attribution data (Part IX),
supplying the COST half of any ROI ratio IAS-C1 computes, per the explicit boundary Part IX §9.5 states;
and the Knowledge-Yield distribution (Part XIII), which IAS-C1's capital-allocation-board subengine
(per the Non-Duplication Ledger's candidate 25) can weigh when deciding whether to invest further in a
producing system's own capacity. (NORMATIVE, restating Part II §2.6's authority boundary: every one of
these four feeds is EVIDENCE into an IAS-C1 decision, never a decision E1 renders on IAS-C1's behalf.)

22.3 The edge to IAS-D1 (System Ecology), specified precisely. Per the Compounding Graph's primary loop
("IAS-E1... IAS-D1... outcomes update fitness/symbiosis/parasitism"), E1 supplies IAS-D1 with: the full
Health Read (Part XIV), read by IAS-D1's fitness-landscape analysis as its primary observational input;
the Bottleneck Map's CONVERGENT and STARVED findings (Part XI), which IAS-D1's ecology reads to judge
whether a convergent system represents healthy specialization or dangerous monoculture, and whether a
starved region represents a legitimate niche or an undiscovered symbiosis opportunity — judgments Part
XI §11.4-11.5 explicitly decline to render themselves; and the DUPLICATES-adjacent findings the Dark-
Matter Detector surfaces (Part V), which feed IAS-D1's own speciation and merger-intelligence subengines
(Non-Duplication Ledger candidates 36-37) as raw material for a D2A disposition once formally reviewed.
The STRUCTURAL-BLINDSPOT correlation named in this dataset's own comprehensive worked example (Part
XVII §17.5) is the clearest single instance of an E1 finding requiring IAS-D1's own judgment to resolve
— E1 names the pattern; IAS-D1 decides whether it reflects a genuine niche or a genuine gap.

22.4 The edge to IAS-G1 (Architecture Intelligence), specified precisely. Per the Compounding Graph's
stated edge ("IAS-D1... IAS-G1... fitness informs topology + complexity-budget decisions"), E1 also
feeds IAS-G1 DIRECTLY where a finding is architectural in nature rather than requiring IAS-D1's
ecological judgment first: the Bottleneck Map's CONVERGENT findings (Part XI), read by IAS-G1's own
topology and complexity-budget analysis as a candidate for decomposition or redundancy investment; the
Latency Observatory's PROPAGATION-dominated findings (Part X §10.4's worked example), which point
specifically at wiring gaps IAS-G1's topology mechanism, not any single system's own maintainer, is
positioned to close; and the evidence-integrity drift patterns (Part VIII §8.5), which inform IAS-G1's
own debt-market subengine (Non-Duplication Ledger candidate 69) when assessing which parts of the
estate's documentation and knowledge architecture carry accumulating, unaddressed quality debt.

22.5 The inbound edges E1 itself depends on, stated for completeness against Article IV's orphan test.
E1 is not a pure source; per Article IV's requirement that every dataset name what it RECEIVES as well
as produces, E1's own inbound dependencies are: IAS-F1's ontology, addressing model, property model,
and typed relation graph (Part II of this dataset, resolved in full); IAS-A2's Control Bus event log,
the primary evidentiary substrate every Tier-One engine reads (Part XVI §16.2); and, indirectly, the
outputs of every system E1 observes, since a fabric with nothing to observe produces nothing — a

DECISION LOG
===============================================================================================

D1. DECISION: Split the flow observatory into four separate, independently-schemaed facets — knowledge,
decision, evidence, resource (Parts VI-IX) — rather than one unified "activity flow" record. ALTERNATIVES
CONSIDERED: (a) one blended flow schema tagged by a payload-type field, letting downstream consumers
filter; (b) two facets only (a "content" flow merging knowledge and decision, and a "cost" flow merging
evidence and resource). WHY THIS: Part IX §9.8 states the reasoning directly — a single blended metric
would erase the specific distinctions (expensive-but-sound versus cheap-but-inflated, obeyed-but-ignored
versus the reverse) that make each facet independently actionable by a different downstream consumer;
collapsing them costs exactly the information the four-facet design exists to preserve. TRADE-OFFS: four
independent schemas is more specification surface than one blended record, and a future implementation
must maintain four confirmation-tracking pipelines rather than one. REVERSAL CONDITIONS: if Protocol One
(Part XXI §21.2) finds real-estate data where the four facets are so highly correlated in practice that
the separation adds cost without adding distinguishing signal, a MERGE of two facets becomes a candidate
for this dataset's own future revision.

D2. DECISION: Define the dead-capability detector's DEAD threshold as two consecutive reconciliation
windows past a cadence-aware grace period (Part IV §4.2), rather than a single-window zero-count or a
fixed calendar duration. ALTERNATIVES CONSIDERED: (a) single-window zero-count, flagging DEAD
immediately on first silence; (b) a fixed calendar threshold (for instance, ninety days) applied
uniformly regardless of a system's own declared cadence. WHY THIS: a single-window rule generates
false positives on any system with a natural usage rhythm longer than one window (Part IV §4.5's
seasonal-system edge case), and a fixed calendar threshold ignores that different system types have
genuinely different natural cadences, penalizing legitimately infrequent-but-real capabilities.
TRADE-OFFS: the two-window, cadence-aware rule is slower to flag a genuinely abandoned system than a
single-window rule would be, meaning true dead capabilities persist, undetected, for longer under this
design. REVERSAL CONDITIONS: if Protocol One's retrodiction finds the two-window lag causes material
institutional harm (capital or attention wasted on capabilities that a faster-flagging rule would have
caught sooner), the threshold is a revision candidate, traded off explicitly against the false-positive
cost this decision was built to avoid.

D3. DECISION: Require the Institutional Health Observatory to report a structured, multi-field profile
(Part XIV §14.3) rather than a single composite health score. ALTERNATIVES CONSIDERED: (a) a weighted
composite score (0-100 or similar), the most common industry pattern for a "health dashboard"; (b) a
simple traffic-light (green/yellow/red) summary per system, one level up from a raw score but still a
collapsing operation. WHY THIS: Part XIV §14.2 develops the reasoning at length — any composite
necessarily hides which facet drives it, and Part XIX names exactly this collapsing operation as the
specific artifact most exposed to observability theater. A structured profile costs more to read but
cannot silently hide a severe finding behind a reassuring aggregate. TRADE-OFFS: a structured profile
is harder for a time-pressured Owner to consume at a glance than a single score, and this dataset
accepts that readability cost as the necessary price of honesty. REVERSAL CONDITIONS: none anticipated
— this decision is treated as close to load-bearing as any in the dataset, on the same footing as
IAS-F1's own D3 (per-fact rather than per-record evidence marking), because reversing it would
reopen the exact theater risk Part XIX exists to close.

D4. DECISION: Make the Telemetry-Before-Claims Contract's compliance standard absolute (Part XX
§20.11) rather than a target ratio with an acceptable tolerance band. ALTERNATIVES CONSIDERED: (a) a
target compliance rate (for instance, ninety-five percent of claims must carry a valid source, with
the remaining five percent treated as acceptable overhead); (b) a graduated severity scale where
non-compliance is weighted by the claim's own institutional consequence. WHY THIS: any tolerance band,
however small, creates exactly the space observability theater (Part XIX) grows in — a five-percent
allowance becomes, over time, the five percent an operator under pressure quietly routes their least
defensible claims through. An absolute standard removes that space entirely, at the cost of the
dataset being unable to offer a single reassuring compliance percentage. TRADE-OFFS: an absolute
standard means a single missed source on a low-stakes claim is treated with the same formal severity
as one on a high-stakes decision, which some readers may find disproportionate. REVERSAL CONDITIONS:
none stated — this decision, like D3, is treated as structurally load-bearing to the fabric's own
credibility, not a calibration choice subject to routine revision.

D5. DECISION: Classify the Dark-Matter Detector as TIER-ONE-AND-A-HALF (Part XVI §16.4) rather than
forcing it into a clean Tier-One or Tier-Two bucket. ALTERNATIVES CONSIDERED: (a) classify it Tier-One,
since its primary evidence source is incidental data read directly; (b) classify it Tier-Two, since its
core comparison logic depends on the Usage Observatory's registry. WHY THIS: forcing a clean binary
classification when the engine's real dependency structure is genuinely mixed would misrepresent its
actual build requirements to a future implementer, who would either under-build it (Tier-One
classification, missing the registry dependency) or over-delay it (Tier-Two classification, missing the
degraded-standalone-mode opportunity). An honest hybrid classification, though less tidy, is more useful.
TRADE-OFFS: a two-tier taxonomy with one named exception is architecturally less clean than a strict
binary, and future engines this dataset has not anticipated may require their own hybrid
classifications, complicating the taxonomy further over time. REVERSAL CONDITIONS: if a future revision
of this dataset adds enough hybrid-classified engines that the two-tier-plus-exceptions model becomes
unwieldy, a three-tier taxonomy (Primary / Hybrid / Synthesis) is the natural revision path.

D6. DECISION: Fold the "measured knowledge yield" metric (Non-Duplication Ledger candidate 90) into the
Knowledge-Consumption Tracker (Part XIII) rather than treating it as a standalone addition bolted onto
the flow observatory. ALTERNATIVES CONSIDERED: (a) build it as a fully separate fifteenth engine, sitting
alongside but architecturally independent of Part VI's knowledge-flow machinery; (b) decline the fold
entirely and leave the yield metric to whichever future mission implements the Family I REFERENCE
routing. WHY THIS: Part XIII §13.1 states the reasoning directly — the yield metric is, mechanically, an
aggregation of the same CONFIRMED/DELIVERED-UNCONFIRMED distinction Part VI's knowledge-flow observatory
already tracks per-event, and building it as a separate engine would duplicate that underlying evidence
pipeline rather than reusing it, violating Article III's strengthen-never-replace discipline at the
engine-architecture level, not merely at the dataset level. TRADE-OFFS: folding the metric into Part
XIII means the Knowledge-Consumption Tracker carries slightly more schema surface (the producing-system
aggregation, Part XIII §13.3) than a minimal per-event flow tracker would otherwise need. REVERSAL
CONDITIONS: none anticipated; the ledger's own routing instruction is treated as a binding input to this
dataset's architecture, not a suggestion open to reconsideration by this dataset alone.

ANTI-PATTERNS
===============================================================================================

ANTI-PATTERN 1 — COLLAPSING A METRIC-COUNTERMETRIC PAIR INTO ONE NUMBER FOR CONVENIENCE. The most
likely implementation error, once a real fabric is built against this specification, is a downstream
dashboard or report quietly dropping the paired countermetric Part XX mandates for every headline
figure this dataset names — reporting the yield ratio (Part XIII) without the absolute confirmed-
consumption count, or the centrality score (Part XI) without the declared-authority cross-check —
because a single number is simpler to display and the pairing feels like bookkeeping overhead to a
builder under schedule pressure. This is dangerous specifically because the resulting single number is
still technically sourced and technically accurate; it is not a fabrication, only an incomplete
presentation, which makes it far more likely to survive a casual review than an outright false claim
would. PREVENTION: treat Part XX §20.2's pairing rule as a presentation-layer hard requirement, not a
data-layer suggestion — any interface, report, or query that surfaces one of this dataset's named
metrics without its paired countermetric in the same emission is, per this dataset's own discipline,
producing a non-compliant output regardless of how accurate the single number is.

ANTI-PATTERN 2 — BUILDING THE SYNTHESIS ENGINES BEFORE THE PRIMARY ONES ARE MATURE. The second most
likely error, especially under pressure to show a working health dashboard quickly, is a build sequence
that reaches for Part XIV's Institutional Health Observatory or Part XI's Bottleneck Map before Parts
III, VI-IX, and X have accumulated enough real observation windows to be trustworthy — producing a
composite or graph-centrality analysis that LOOKS complete and authoritative while actually resting on
thin, immature, or partially-cold-start data (Part XVIII §18.2's cold-start edge case). This is the
specific architectural instance of the observability-theater risk (Part XIX): a synthesis engine built
ahead of its own dependencies does not fail loudly, it produces confident-looking output on an
unweighted or under-weighted graph, exactly the failure Part XI §11.7 names for the bottleneck map
specifically. PREVENTION: follow Part XVI §16.3's specified build sequence strictly — Tier-One engines
first, in parallel, across at least two full reconciliation windows, before any Tier-Two synthesis
engine is trusted for a real institutional decision, even if the synthesis engine's own code is ready
to run earlier.

ANTI-PATTERN 3 — TREATING A DEAD OR DARK-MATTER FLAG AS A VERDICT INSTEAD OF EVIDENCE. The third likely
error, particularly for an Owner or a consuming system unfamiliar with this dataset's own authority
boundary (Part II §2.6, restated in Part IV §4.3 and Part XVI §16.6), is acting directly on a DEAD
verdict or a dark-matter finding as if it were itself a retirement or registration decision — quietly
deprecating a flagged system, or hastily formalizing an unregistered pattern into a new system, without
routing the finding through D2A's own disposition process (BUILD, MERGE-INTO-FAMILY, SUBENGINE, EXTEND,
COMPOSITION, REFERENCE, or RETIRE) as this dataset's own text repeatedly insists. This converts E1,
silently, from a witness into a judge — the precise failure Part IV §4.3's closing line names directly.
PREVENTION: any process consuming an E1 finding (DEAD, dark-matter, NO-PROVENANCE, or otherwise) must
route it through the named downstream authority (IAS-D1, D2A, IAS-A2) before any irreversible action is
taken, and a future implementation should make this routing structurally mandatory — a DEAD flag with
no corresponding D2A review ticket should itself be flagged as a process gap, not treated as an
implicit invitation to act.
