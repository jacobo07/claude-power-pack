PART I — THE THESIS: WHY THE ENSEMBLE NEEDS A ROUTER ABOVE THE RESOURCE ROUTERS
===============================================================================================

1.1 The founding observation, restated at the routing layer. Claude Power Pack, as OBSERVED
across its own indices, already contains at least three working routers: CO-03 selects which
model tier serves a session by cost and capability fit; skill_router selects which skill file's
instructions activate for an incoming prompt by keyword and description match; GK-05 selects
which knowledge node answers a query by coordinate proximity and freshness. Each router is
excellent at its own resource and blind past its own boundary. None of the three can answer the
question a real mission actually poses on its first minute: "given what this Owner just asked
for, which of the estate's seventy-plus SYSTEMS — not which model, not which skill file, not
which knowledge node, but which whole governing systems, in what sequence — should even be
invoked." A mission that needs a spec gate, then a model, then three skills, then a knowledge
lookup, then a pane dispatch, then a governance check has no existing system that decides that
sequence; each resource router only answers "given that I am already the one deciding, what do
I pick." (OBSERVED: three resource routers exist and are each scoped to one resource class;
DERIVED: no existing router's scope includes deciding whether it should be consulted at all.)

1.2 The gap made concrete: a mission with no home. Consider an Owner request of ordinary
complexity: "harden this module against the top five predictable failure modes, then ship it."
Read literally, that single sentence implicates at minimum a spec-tier gate (is this L or XL),
a code-review pass, an adversarial-hardening pass, a deploy governance check, and a hard-rules
audit — five systems, none of which the sentence names, and none of which volunteers itself.
CO-03 will happily pick a model once something asks it to. skill_router will happily activate a
skill once a prompt resembling its description arrives. Neither decides that a spec-tier check
must run BEFORE the model is even engaged, or that the deploy governance check must run AFTER
the code exists but BEFORE anything ships. That ordering decision — which systems, in what
sequence, with what handoffs between them — is the object no resource router owns, because it
is not a resource; it is a decision about SYSTEMS, which is precisely the ensemble-level object
IAS-F1 Part III defines. (DERIVED from IAS-F1 Part I §1.4's resource-level/ensemble-level
distinction, applied here to the specific case of mission-to-system dispatch.)

1.3 The trap of building the seventy-first resource router. As with IAS-F1's own founding
argument, the natural but wrong reflex is to build one more router that behaves like CO-03,
skill_router, and GK-05 — a fourth sibling that picks "the right system" the way the other three
pick "the right model / skill / node." That reflex fails the Non-Duplication Ledger's test on
inspection: a fourth resource router, scoped to picking one system per call, still cannot answer
the sequencing question of §1.2, because sequencing is a property of a MISSION across MULTIPLE
systems, not a property of any single dispatch decision. A router that only ever answers "which
one system next" one call at a time, with no memory of the mission's shape, degenerates into the
DELEGATOR NEAR-MISS the Federation Ontology names in Part VII §7.7 — a lookup table dressed as a
router, first-order despite sitting structurally "above" the resource routers. IAS-A1 must clear
that bar by holding the MISSION as its unit of work, not the single dispatch. (NORMATIVE, DERIVED
from IAS-F1 Part VII §7.7's near-miss catalogue.)

1.4 The two absorbed problems that make a router insufficient on its own. The Non-Duplication
Ledger routes two additional candidates into this same dataset precisely because a router alone
cannot close the gap of §1.2: Candidate 2, the Cross-System Mission Composer, and Candidate 6,
Workflow Synthesis. A router decides WHICH systems; a composer decides HOW their outputs combine
into one mission result; workflow synthesis decides the TEMPORARY SEQUENCE that only this
mission needs and that dissolves once the mission completes. The three absorbed candidates are
not three datasets awkwardly forced together; they are three facets of one indivisible act —
composing a mission from capabilities distributed across systems requires first identifying
which systems hold the needed capabilities (routing), then deciding the order and combination
rule for their outputs (composition), and doing so via a structure that exists only for this
mission's duration (synthesis). (DERIVED: the Ledger's own verdict for candidates 1, 2, and 6
converges on IAS-A1 because their objects are inseparable in practice, however separable they
are in name.)

1.5 The third absorbed problem: handoff without state loss. Candidate 8, the Capability Handoff
Protocol, closes the remaining gap: once IAS-A1 has routed a mission across, say, four systems
in sequence, what happens at each seam — does System 2 know what System 1 already established,
or does it start cold? A mission composer that decides sequence but loses state at every seam
produces a chain of amnesiac dispatches, each re-deriving context the previous system already
held, which the estate's own Session Resilience family has already OBSERVED as a costly failure
pattern at the pane level (PM-02's handoff problem, one level down). IAS-A1 inherits the same
structural risk one level up — at the SYSTEM level rather than the pane level — and Candidate 8
is absorbed here specifically because handoff-without-loss is not a separable concern from
routing; a route that arrives at the right system with no context is barely better than a route
to the wrong system. (DERIVED from the Ledger's Candidate 8 disposition; the mechanism itself is
specified in Part VIII.)

1.6 What IAS-A1 is not: the boundary this dataset must hold from its first sentence. IAS-A1 does
not select a model (CO-03's object, never rebuilt here). It does not select a skill file
(skill_router's object). It does not locate a knowledge node (GK-05's object). It does not decide
whether a proposal is a duplicate (D2A's object). It does not compile a single mission's contract
in isolation (one_shot's object, which A1 CONSUMES rather than replaces once a system has been
selected). Every one of these boundaries is load-bearing, because the instant IAS-A1 reaches into
any of them, it stops being an ensemble-level system and becomes a resource-level duplicate under
its own second-order name — the exact near-miss pattern IAS-F1 Part VIII §8.6 warns against.
IAS-A1's boundary is drawn, and held throughout every later Part of this dataset, at exactly one
line: A1 decides WHICH SYSTEMS act and in what sequence; it never decides what any single system
does once dispatched. (NORMATIVE, DEFINED — this line is the operational form of the Article II
boundary applied to this specific dataset.)

1.7 A worked trace of the founding gap, before the mechanism is specified. Returning to the
mission of §1.2 ("harden this module against the top five predictable failure modes, then ship
it") on synthetic data makes the founding observation concrete rather than assertion-only.
OBSERVED under a simulated dispatch: a naive execution path that hands the sentence directly to
CO-03 produces a single model call with a system prompt improvised ad hoc from the sentence's
surface words, no spec-tier check, no adversarial-hardening pass, and no deploy-governance gate —
because CO-03's object is model tier, not system sequence, so nothing in its scope registers that
five prior systems were implicated at all. IAS-A1's mechanism, specified fully in Parts V through
VIII, instead decomposes the sentence into five capability requirements (spec-tier classification,
code-review, adversarial-hardening, deploy-governance, hard-rules-audit), matches each against the
capability index of Part III, and emits a five-entry route with explicit handoff state carried at
each seam: spec-tier's verdict becomes code-review's input, code-review's diff becomes adversarial-
hardening's target, and adversarial-hardening's finding list becomes deploy-governance's pre-check.
(HYPOTHESIZED for this specific synthetic trace, pending the full worked mission of Part XXI;
DERIVED in mechanism from Parts V-VIII.) The comparison is not a claim that CO-03 or skill_router
are broken — each performs its own object correctly, on its own terms — it is a claim that NOTHING
in the ensemble, absent IAS-A1, performs the sequencing object at all, which is the precise gap
this Part exists to name before any later Part specifies how the gap is closed.

FINAL LAW — PART I. Three resource routers can each answer "which one thing, given that I am
the one deciding," and none of them can answer "which of the estate's systems, in what order, for
this whole mission" — that unanswered question is not a fourth resource router's job but a
second-order capability's, and the fourth-router reflex is the exact naming-only-novelty trap the
federation exists to reject.

PART II — ONTOLOGY RESOLUTION: RESOLVING IAS-A1's TERMS AGAINST IAS-F1
===============================================================================================

2.1 Why a resolution Part, not a redefinition Part. Article XI of the Federation Ontology
(IAS-F1) requires every later IAS dataset to resolve its objects against F1 with zero undefined
terms; this Part performs that resolution explicitly rather than assuming it, because a dataset
that silently reuses F1's vocabulary without stating the mapping leaves its own claims
unverifiable against the shared standard. Every term this dataset uses in a load-bearing way is
listed here with its F1 origin and, where IAS-A1 needs a term F1 does not supply, the new term is
introduced with the same discipline F1 used for its own primitives — justified by an OBSERVED
gap, not invented for convenience. (NORMATIVE.)

2.2 SYSTEM, CAPABILITY, SECOND-ORDER CAPABILITY — imported unchanged. IAS-A1 uses SYSTEM exactly
as IAS-F1 Part II §2.1 defines it: a bounded unit with a distinct object, authority, interface,
and address. IAS-A1 uses CAPABILITY exactly as Part II §2.4 defines it: a nameable thing a system
can do, independent of the system that provides it. IAS-A1 uses SECOND-ORDER CAPABILITY exactly
as Part VII §7.2 defines it, and IAS-A1's own status as second-order was already established in
this dataset's header block by direct application of that test. No redefinition occurs at any of
these three terms; IAS-A1 is a consumer of F1's vocabulary, not a competing source of it.
(DEFINED, imported.)

2.3 CONSUMES — the relation type this dataset most directly operationalizes. IAS-F1 Part VI §6.7
defines CONSUMES as the relation a routing decision creates at mission time: System A invokes
System B as a service within a mission, transient, not a standing structural relationship. IAS-A1
is, in the ontology's own terms, THE SYSTEM WHOSE CAPABILITY IS TO CREATE CONSUMES EDGES. Every
successful route this dataset produces is, formally, the instantiation of one or more consumes
edges in the ensemble's typed relation graph, timestamped to the mission that created them. This
resolution matters operationally: it means IAS-A1's own output is directly legible to IAS-E1
(Observability Fabric), which reads the consumes subgraph to compute usage, and to IAS-D1
(System Ecology), which reads consumes-edge density as one input to a system's fitness. (DERIVED
from IAS-F1 Part VI §6.7 and Part VI §6.8's claim that emergent properties are computed over the
typed graph.)

2.4 FEEDS and DEPENDS-ON — read, not created, by this dataset. Where consumes edges are IAS-A1's
own output, FEEDS and DEPENDS-ON edges (IAS-F1 §6.2, §6.3) are INPUT this dataset reads before
routing: a mission that would route to System X but System X depends-on a currently-degraded
System K should not be routed there without accounting for the propagated failure risk, and Part
X of this dataset (routing under uncertainty) specifies exactly how the depends-on subgraph
informs a route's confidence. IAS-A1 does not modify the standing feeds/depends-on graph; it
reads it as a precondition, the same way a shipping route reads a road network without owning
the roads. (DERIVED, NORMATIVE boundary.)

2.5 MISSION — a term F1 does not supply, introduced here with justification. IAS-F1's object
model stops at systems, capabilities, and their relations; it does not define the unit of work
that arrives from outside the ensemble and triggers a routing decision. IAS-A1 DEFINES MISSION as
a bounded unit of Owner (or upstream-system) intent that names, explicitly or by inference, one or
more desired outcomes, and that requires zero or more of the ensemble's systems to be invoked, in
some order, to be satisfied. A mission is external to the ensemble in the sense that it originates
outside any single system (IAS-F1 Part III §3.4's boundary discipline: the ensemble is bounded to
the estate, and a mission is the estate's unit of externally-triggered work crossing that
boundary), but internal to the ensemble's reasoning the instant IAS-A1 begins decomposing it,
because decomposition (Part V) expresses the mission entirely in terms of F1's own object model —
required capabilities, candidate systems, and the relation types that will connect them.
(DEFINED, justified by an OBSERVED gap: no existing IAS-F1 object names the unit IAS-A1's own
input arrives in.)

2.6 ROUTE — the second new term, and its relation to CONSUMES. A ROUTE is DEFINED as an ordered
sequence of (system, capability-invoked, handoff-state) triples that jointly satisfy a mission's
decomposed capability requirements. A route is not identical to a set of consumes edges; it is the
ORDERED, STATEFUL structure that, once executed, PRODUCES those consumes edges as its ensemble-
level trace. The distinction matters because a route can be planned, evaluated, and even rejected
before a single consumes edge is instantiated (Part IX, arbitration; Part X, uncertainty), while a
consumes edge, per F1, only exists once the invocation has actually occurred. IAS-A1's core
operation, across every later Part, is producing and validating routes before they become edges.
(DEFINED.)

2.7 CAPABILITY INDEX — the third new term, specified fully in Part III. A CAPABILITY INDEX is
DEFINED as the standing, queryable record of which systems provide which capabilities, built from
the property model IAS-F1 Part IV specifies (each system's capability signature, per Part IX
§9.2's addressing triple) plus a route-specific extension this dataset adds: applicability
conditions, cost-to-invoke, and typical latency. The index is what makes routing tractable without
re-deriving, on every mission, which of seventy-plus systems could plausibly serve it. (DEFINED,
elaborated in Part III.)

2.8 AUTHORITY and DECISION CLASS — imported from F1 Invariant Three, applied to dispatch. IAS-F1
Part V §5.4 (Invariant Three) establishes that exactly one system holds final authority per
decision class, with others holding advisory input. IAS-A1's own decision class is precisely
DEFINED here as "which system(s) act on a given mission, in what order" — and Part XI of this
dataset addresses, directly, whether IAS-A1 itself should hold final or advisory authority over
that class, resolving the question the header block's boundary section only gestures at. This
resolution is deferred to Part XI deliberately, because the authority question requires the
arbitration and uncertainty machinery of Parts IX and X to be specified first. (DERIVED,
forward-referenced.)

2.9 The resolution table as a governance artifact, not decoration. Every later Part of this
dataset uses SYSTEM, CAPABILITY, SECOND-ORDER CAPABILITY, CONSUMES, FEEDS, DEPENDS-ON exactly as
resolved above, and MISSION, ROUTE, CAPABILITY INDEX exactly as newly defined above, with no
silent drift. This discipline is what lets IAS-E1's later observability sweep, or IAS-D1's later
ecology map, read IAS-A1's own routing records without a translation layer — the entire point of
F1 existing at all, made concrete for this one dataset rather than asserted in the abstract.
(NORMATIVE.)

2.10 Two terms considered and rejected during resolution, named for the record. Before settling on
MISSION and ROUTE as the only two genuinely new terms (§2.9), two candidate terms were considered and
rejected specifically because IAS-F1 already supplies an adequate equivalent, and introducing a
synonym would have violated the resolution discipline this Part exists to enforce. "DISPATCH TARGET"
was considered as a name for what a route ultimately resolves to, and rejected because it is fully
subsumed by IAS-F1's own SYSTEM (Part II §2.1) — a dispatch target is simply a system, and naming it
separately would imply a distinction that does not exist. "CAPABILITY BUNDLE" was considered as a
name for a system's full set of offerings, and rejected because IAS-F1's CAPABILITY (Part III §3.1),
applied at the plural, already covers the same ground without needing a bundling concept this
dataset would then have to separately define, maintain, and keep synchronized with the singular term
it merely aggregates. Naming both rejected candidates explicitly, rather than silently not using
them, is itself part of the resolution discipline: a future reader encountering either term
elsewhere in the estate's own documentation should not assume IAS-A1 independently coined it.
(NORMATIVE, DERIVED — negative resolution, stated for completeness.)

2.11 Why MISSION is not merely a renamed "task" or "request." A skeptical reading might ask whether
MISSION is genuine terminological necessity or a stylistic preference for a grander word than the
estate's existing vocabulary already uses informally ("task," "request," "job"). The distinction is
substantive, not stylistic: none of those informal terms carries, as part of its own definition, the
property of decomposing into MULTIPLE (intent, object) pairs routed across MULTIPLE systems in a
specific sequence with handoffs between them (Part V, Part VI, Part VIII) — a "task" in the estate's
ordinary usage is typically scoped to a single system's own unit of work (a task CO-03 completes, a
task a skill executes), while MISSION is DEFINED here specifically as the unit that spans systems.
Using "task" for both the single-system and the cross-system unit would blur exactly the distinction
Part I §1.3 built this entire dataset's founding argument around — that the ensemble needs a router
whose unit of work is NOT any single system's own dispatch. The terminological choice is therefore
load-bearing, not decorative. (DERIVED, NORMATIVE, defending the necessity of the one term §2.6 most
directly introduces.)

FINAL LAW — PART II. A dataset that reuses a shared ontology's terms without stating the mapping
has not actually inherited the ontology, only borrowed its authority; IAS-A1 resolves every term
it uses against IAS-F1 explicitly, introduces only the two terms — mission and route — that the
foundation genuinely does not supply, and justifies each by a gap the foundation itself leaves
open rather than by convenience.

PART III — THE CAPABILITY INDEX OVER ALL SYSTEMS
===============================================================================================

3.1 Why an index precedes a router, structurally. A router that re-derives, on every mission,
which of the estate's seventy-plus systems could plausibly serve it is not a router; it is a
brute-force search dressed as one, and at estate scale that search is both slow and unreliable,
because "could plausibly serve it" depends on properties — cost, evidence level, model-dependence,
current health — that are expensive to compute fresh each time and cheap to look up if maintained
standing. The CAPABILITY INDEX (Part II §2.7) is therefore the load-bearing data structure this
entire dataset is built on top of, and its correctness is a precondition for every later Part's
correctness: a route computed from a stale or incomplete index is a route computed from a
fiction, however sound the routing LOGIC applied to it. (DERIVED, NORMATIVE.)

3.2 What the index contains, per entry. Each capability-index entry is a record keyed by
(system stable ID, capability name) — deliberately not by system alone, because IAS-F1 Part II
§2.4 already establishes that one system may provide many capabilities, and routing must resolve
at the capability granularity, not the system granularity, or a mission needing only one of a
system's five capabilities would over-invoke the whole system. Each entry carries: the
CAPABILITY SIGNATURE component of the system's address (IAS-F1 Part IX §9.2); the six canonical
properties of IAS-F1 Part IV §4.2 (cost, evidence level, reuse horizon, model-dependence,
consumers, authority) read by reference rather than re-measured, exactly as F1's property model
requires; an APPLICABILITY PREDICATE — a statement of the mission-shape conditions under which
this capability is a candidate at all (e.g., "applicable when the mission names code as its
object and requests hardening"); an invocation COST-TO-ROUTE distinct from the system's own
operating cost (the overhead of dispatching TO this system, which for a heavy subagent spawn is
non-trivial and for a lightweight in-process check is near zero); and a LAST-VALIDATED
timestamp, inheriting IAS-F1 Part IV §4.3's staleness discipline at the routing-specific
granularity of "was this capability actually successfully invoked recently." (DEFINED,
elaborated from F1's property model plus three route-specific fields.)

3.3 How the index is built: extraction, not invention. Consistent with F1's own founding
discipline (IAS-F1 Part I §1.5: "an ontology extracted from the systems that already exist is a
map; an ontology invented for the mechanisms one hopes to build is a fiction"), the capability
index is built by EXTRACTION from each system's own declared interface — its documented inputs,
outputs, and invocation contract — rather than by IAS-A1 inventing what each system "should"
provide. This extraction is a standing maintenance operation, not a one-time build: every new
system admitted to the estate (per IAS-F1's own systemhood test) contributes new index entries;
every system retired (D2A's RETIRE disposition, Candidate 38 in Family D) removes its entries;
every system whose capability signature changes (a refactor, a merge, IAS-F1 Part IX §9.3's
identity-preservation-through-change) triggers a re-extraction of that system's entries alone,
never a full rebuild. (DERIVED, NORMATIVE — extraction over invention, incremental over batch.)

3.4 The taxonomy of capability kinds the index must distinguish. Not every entry is
interchangeable in kind, and collapsing the distinction produces routing errors Part XII
catalogues later. IAS-A1 taxonomizes index entries into four kinds. ADVISORY capabilities produce
a recommendation another system or the Owner may override (e.g., a spec-tier suggestion) — routing
to one of these is low-risk because a wrong route is correctable downstream. GATING capabilities
produce a pass/fail verdict that blocks further mission progress if failed (e.g., a hard-rules
check, a scaffold audit) — routing to one of these MUST occur before any capability whose output
it would gate, and Part V's decomposition algorithm treats gating capabilities as hard sequencing
constraints, not soft preferences. TRANSFORMING capabilities consume mission state and produce
changed mission state (e.g., a code-generation skill, a distillation engine) — these are the
capabilities whose output typically becomes another capability's input, and their ordering is
governed by data dependency, not authority. TERMINAL capabilities produce a mission's final
deliverable and, once invoked successfully, end the route (e.g., a deploy governance check that
ships). A route that never reaches a terminal capability, or reaches one prematurely while gating
capabilities remain unsatisfied, is a malformed route by this taxonomy alone, independent of
whether any individual capability was correctly matched. (DEFINED, elaborated for routing
purposes; DERIVED that this taxonomy, not the F1 relation types, is what Part V's sequencing
logic actually operates on.)

3.5 Worked example: three index entries for a synthetic mission shape. To illustrate without
re-narrating any real module by name, consider three synthetic index entries relevant to the
§1.2 hardening-then-ship mission. Entry Alpha: capability "classify mission size tier," kind
GATING, applicability predicate "mission requests code change of unknown scope," cost-to-route
LOW (in-process, no subagent spawn), evidence level OBSERVED, authority ADVISORY (per IAS-F1
Part IV §4.2's worked-example pattern — the tier classification can be overridden by an Owner
who already knows the tier). Entry Beta: capability "enumerate top-five predictable failure
modes for a given code shape," kind TRANSFORMING, applicability predicate "mission requests
hardening AND a code object is named," cost-to-route MEDIUM (loads a domain-specific archetype
catalogue), evidence level OBSERVED, authority ADVISORY. Entry Gamma: capability "verify deploy
readiness against governance checklist," kind GATING, applicability predicate "mission requests
shipping AND code changes exist," cost-to-route LOW, evidence level OBSERVED, authority FINAL
(this entry cannot be overridden by a lower-authority route — a route that reaches "ship" without
Gamma passing is, by IAS-F1 Invariant Three, not a legal ensemble action). Reading these three
entries side by side already constrains the route before Part V's algorithm runs: Alpha must
precede Beta (a GATING entry establishing scope precedes a TRANSFORMING entry whose behavior
depends on that scope), and Gamma must be the route's terminal step because its authority is
FINAL and its applicability names "shipping" as the mission's terminal intent. (Illustrative
only, not an assertion about any specific real PP module's exact interface.)

3.6 Index health and the dead-entry problem, previewed. An index entry that was extracted once
and never revalidated decays, per IAS-F1 Part IV §4.3's staleness discipline, from a trustworthy
routing candidate into a liability: a route computed against Entry Beta six months after its
providing system was quietly refactored may dispatch to a capability signature that no longer
matches reality. This specific failure — DEAD-CAPABILITY ROUTING — is common enough and costly
enough to warrant its own full Part (Part XII) rather than a subsection here; this Part
establishes only that the index's LAST-VALIDATED field (§3.2) is the mechanism by which staleness
becomes visible rather than silent, which is the precondition Part XII's fuller treatment builds
on. (Forward-referenced, NORMATIVE.)

3.7 Why the index is read by, not owned by, downstream siblings. The capability index is IAS-A1's
own artifact, built and maintained for routing, but its structure — capability-keyed records over
the property model — makes it directly reusable by IAS-C1 (Capability Portfolio), which needs a
per-capability, not merely per-system, view to allocate honestly, and by IAS-E1 (Observability
Fabric), which needs the applicability-predicate field to distinguish "unused because inapplicable
to any recent mission" from "unused despite being applicable" — the exact distinction between a
healthy narrow specialist and true dead capability that Part XII will need for its own diagnosis.
This reuse is not incidental; it is the specific mechanism by which IAS-A1 strengthens IAS-C1 and
IAS-E1 beyond the routing function alone (Strengthening Matrix evidence, elaborated fully in the
Existing-System Reinforcement Map at this dataset's close). (DERIVED.)

FINAL LAW — PART III. A router without a maintained, capability-granular, staleness-aware index
is not routing; it is guessing with better vocabulary — the index is the map, extracted from the
territory and never invented ahead of it, and every later Part of this dataset is only as correct
as the index it queries.

===============================================================================================
PART IV — SYSTEM-SELECTION VERSUS MODEL/SKILL/KNOWLEDGE SELECTION: THE EXPLICIT BOUNDARY
===============================================================================================

4.1 Why this boundary needs its own Part rather than a paragraph. The header block already
states the boundary in one sentence; a one-sentence boundary is cheap to state and expensive to
hold once real missions arrive with ambiguous shape, and the Non-Duplication Ledger's own
disposition for Candidate 1 depends on this exact boundary surviving contact with a real mission,
not merely surviving a definition. This Part therefore works the boundary against each of the
three resource routers individually, because "system-selection is different from resource-
selection" is true in the abstract and each concrete pairing has its own specific failure shape
if the boundary slips. (NORMATIVE.)

4.2 IAS-A1 versus CO-03: the model boundary. CO-03 decides which model tier serves a session once
a system has already determined that a model call is needed at all — its object is DEFINED,
per the Federation Ontology's worked Shape One (IAS-F1 Part VIII §8.7), as RESOURCE-level: the
model tier assigned to one session. IAS-A1 never decides model tier; when a route's capability
entry happens to be provided by a system whose internals invoke a model, IAS-A1's routing decision
stops at "invoke this system" and CO-03's own logic governs everything past that boundary,
including which model the invoked system uses internally. The failure shape if this boundary
slips is SCOPE CREEP INTO RESOURCE ARBITRATION: a version of IAS-A1 that also second-guesses
CO-03's model choice ceases to be ensemble-level by the majority-object rule (IAS-F1 Part VIII
§8.4), because it would now be deciding upon a resource, not upon systems. (DERIVED, applying
IAS-F1's own worked example to this dataset's specific boundary.)

4.3 IAS-A1 versus skill_router: the skill boundary, and its sharper edge. This boundary is
sharper than the model boundary because skills and systems can, superficially, look like the
same kind of thing — both are "capabilities a prompt can activate." The distinguishing test,
DERIVED directly from IAS-F1's systemhood criteria (Part II §2.1), is that a skill is typically a
COMPONENT or, at most, a SUBSYSTEM of a larger system's capability surface — it lacks independent
authority and is addressed only through the system that loads it — while the object IAS-A1 routes
across is the SYSTEM layer: whether to invoke the governance-overlay system at all, not which of
its constituent skill files fires once invoked. Where a mission's shape genuinely requires
choosing among several standalone skills with no unifying parent system, that choice remains
skill_router's object in full; IAS-A1's involvement is limited to having already decided, at the
system level, that "skill selection is in scope for this mission," a decision skill_router itself
does not make about its own invocation. (DERIVED, NORMATIVE — the boundary tracks the
system/subsystem/component distinction of IAS-F1 Part II §2.3, not a topical similarity between
"skill" and "system.")

4.4 IAS-A1 versus GK-05: the knowledge boundary. GK-05 decides which knowledge node answers a
query once IAS-A1 (or any other system) has already established that a knowledge lookup belongs
in this mission's route at all. The boundary here is the cleanest of the three, because knowledge
nodes are unambiguously RESOURCE-level under IAS-F1's own Shape One pattern — a coordinate is
GK's owned resource class, stated explicitly in IAS-F1 Part III §3.4's boundary catalogue. IAS-A1
treats "consult the knowledge graph" as one CAPABILITY entry in its index (kind TRANSFORMING,
typically, since a knowledge lookup changes mission state by adding grounded context), and once
that entry is selected as part of a route, GK-05's own internal node-selection logic governs
everything past the system boundary, exactly as with CO-03. (DERIVED, symmetric with §4.2.)

4.5 The composite failure shape: a mission that plausibly touches all three boundaries at once.
The boundary is easiest to state pairwise and hardest to hold when a single mission's route
crosses all three resource routers in sequence, because at each seam there is a live temptation
for IAS-A1 to "help" by reaching one level down — recommending a model tier alongside the system
selection, or pre-filtering which skill within a selected system should fire. Every such reach-
down is, individually, small and individually plausible-sounding ("I already know the mission
needs a cheap model, why not say so"), and the cumulative effect across many missions is a router
that has quietly absorbed three resource routers' decision rights one recommendation at a time —
the SLOW BOUNDARY EROSION failure named explicitly here so that Part XVIII's adversarial-case
catalogue can reference it as a known shape rather than a hypothetical one. The structural
defense, DERIVED from the capability index's own taxonomy (Part III §3.4), is that IAS-A1's
output is never anything but a route — an ordered sequence of (system, capability, handoff-
state) triples — and a route object has no field for "suggested model tier" or "suggested skill
within system"; the schema itself is the boundary enforcement, not vigilance alone. (DERIVED,
NORMATIVE — schema-as-boundary is stronger than discipline-as-boundary, echoing IAS-F1's own
preference for structural invariants over intention.)

4.6 What IAS-A1 legitimately DOES read from the resource routers, without deciding for them. The
boundary is not a wall against all interaction; IAS-A1's routing decisions are genuinely better
when informed by resource-level signals, provided those signals are read as INPUT (IAS-F1's
majority-object rule, Part VIII §8.4) rather than acted on as output. IAS-A1 may read CO-03's
current cost posture (is the estate in a constrained-budget window) as a factor in preferring a
cheaper route among several capability-equivalent candidates; it may read skill_router's recent
activation frequency as a signal of which systems are "warm" versus cold-start; it may read GK-05's
freshness metadata as a factor in whether a knowledge-dependent capability entry is currently
trustworthy. In every case the READ informs a decision whose PRODUCED object remains the route
itself — the mixed-capability case IAS-F1 Part VIII §8.4 explicitly licenses, applied here
concretely to all three resource routers at once. (DERIVED, applying the majority-object rule.)

4.7 Why this boundary is the specific mechanism of the Strengthening Matrix claim. The
Strengthening Matrix states IAS-A1's gain as "fewer wrong-system dispatches; missions reach the
right capability first," evidenced by "a mission trace where A1 selects a system the parent
routers could not see." This Part's boundary work is what makes that evidence possible at all:
because IAS-A1 never reaches into CO-03's, skill_router's, or GK-05's resource-level decisions,
its own routing trace is legible as a genuinely SEPARATE decision layer, one whose correctness
or error can be audited independently of whether the resource routers, once invoked, chose well
within their own scope. A router that blurred the boundary would produce traces where "wrong
system" and "wrong model" were indistinguishable, making the strengthening claim unfalsifiable —
exactly the failure Article III warns against (a strengthening claim without checkable evidence
is HYPOTHESIZED, never fact). (DERIVED, NORMATIVE.)

4.8 The edge case where a resource router is itself the routed-to system. The boundary of §4.2-
§4.4 assumes CO-03, skill_router, and GK-05 always sit one level below IAS-A1's routing decision,
invoked internally by whichever system A1 selects. A genuine edge case breaks that assumption: a
mission whose capability requirement IS, in full, "audit the knowledge graph's node freshness
across all coordinates" — here GK-05 is not a resource silently consumed by some other system but
the entire capability the mission needs, addressed directly by IAS-A1's route. CONTESTED whether
this collapses the boundary: on one reading, IAS-A1 has now "selected GK-05," which looks
indistinguishable from GK-05 selecting itself and therefore looks like exactly the scope-creep
§4.2 warns against. The resolution, DERIVED from the majority-object rule of §4.6, is that IAS-A1's
routing decision in this case remains ensemble-level, because its output is still "invoke the
GK-05 SYSTEM as a whole, for this capability," never "return coordinate X" — the latter would be
GK-05's own resource-level object, reached in this edge case without any intermediate skip only
because the mission's capability requirement was itself system-shaped rather than node-shaped.
The distinguishing test, usable mechanically on any ambiguous route entry: did IAS-A1's route
entry name a coordinate, a model tier, or a skill file (resource-level, boundary violated), or did
it name the system as an addressable whole with the capability description left to that system's
own internals to resolve (ensemble-level, boundary held)? (NORMATIVE, DERIVED.)

FINAL LAW — PART IV. The boundary between routing systems and routing a system's own resources is
not a courtesy to the resource routers; it is what keeps IAS-A1's own decisions auditable as a
distinct layer, and the schema of a route — which has no field for model tier, skill choice, or
knowledge node — enforces that boundary structurally, so that it need not be re-litigated by
discipline on every mission.

===============================================================================================
PART V — MISSION DECOMPOSITION TO CAPABILITY MATCHING
===============================================================================================

5.1 The problem this Part solves. Part I established that a mission is the unit IAS-A1 receives;
Part III established the capability index it queries. Between the two sits the actual mechanism:
how an Owner's sentence, or an upstream system's structured request, becomes a set of capability
requirements the index can be matched against. This is DECOMPOSITION, and it is the step where
most of the routing error this dataset's later Parts diagnose actually originates — not in the
index (Part III) and not in the matching arithmetic (this Part's second half), but in a
decomposition that mis-states what the mission actually needs. (DERIVED from the general pattern
that upstream errors dominate downstream error in any pipeline; NORMATIVE that decomposition
therefore receives the most careful specification of this dataset's mechanical Parts.)

5.2 Decomposition as extraction of intents, objects, and constraints — not free-form parsing.
IAS-A1 DEFINES decomposition as the extraction of three structured elements from a mission: its
INTENTS (the verbs — harden, ship, summarize, migrate, evaluate — each mapped, where possible, to
the applicability-predicate vocabulary the capability index already uses, per Part III §3.2, so
that decomposition output and index structure share one vocabulary rather than requiring a
translation step); its OBJECTS (the nouns the intents act upon — a module, a dataset, a session, a
decision — read against the resource classes IAS-F1 Part VIII §8.2 already enumerates, so that
decomposition can distinguish a resource-level object a single system should handle from an
ensemble-level object this dataset itself must handle); and its CONSTRAINTS (explicit bounds the
Owner stated — a budget ceiling, a deadline, an exclusion of a named approach — plus IMPLICIT
constraints inherited from standing governance, such as the estate's own hard-rules triggers,
which decomposition must surface as constraints even when the Owner's sentence never mentions
them). (DEFINED.)

5.3 The GATING-capability pre-scan, run before full decomposition completes. Because Part III
§3.4's taxonomy establishes that GATING capabilities constrain sequencing hard, decomposition
runs a pre-scan specifically for gating triggers before attempting full intent/object/constraint
extraction: does this mission's stated intent or object pattern-match any known gating condition
(a size-tier gate, a spec-tier gate, a deploy-governance gate, a secret-firewall trigger)? A
gating trigger found in the pre-scan is treated as a HARD CONSTRAINT from the moment it is found,
even before the rest of decomposition completes, because a gating capability's entire purpose is
to potentially halt the mission before further capability-matching work is spent on a mission
that should not proceed as stated. (DERIVED, NORMATIVE — this ordering is the routing-layer
analogue of "fail fast," applied to gating capabilities specifically because their authority, per
Part III §3.4, is the strongest kind the index records.)

5.4 Capability matching: from extracted intent/object pairs to index candidates. Once intents,
objects, and constraints are extracted, matching queries the capability index (Part III) for
entries whose APPLICABILITY PREDICATE is satisfied by the extracted (intent, object) pairs,
filtered by any hard constraints from §5.3. A single mission typically produces MULTIPLE
(intent, object) pairs — the §1.2 hardening-then-ship example yields at minimum (harden, module)
and (ship, module) — and each pair independently queries the index, producing a CANDIDATE SET per
pair rather than one candidate set for the whole mission; this separation is what allows Part VI's
compose engine to reason about combination and Part IX's arbitration to reason about conflicting
candidates within a single pair, rather than conflating both problems into one search. (DEFINED,
DERIVED that per-pair candidate sets are the correct granularity for the reasons given.)

5.5 Confidence scoring at the match level, feeding but not replacing Part X's uncertainty
handling. Each (pair, candidate) match carries a MATCH CONFIDENCE, DERIVED from three inputs: the
lexical/semantic closeness between the extracted intent/object and the candidate's applicability
predicate (a HYPOTHESIZED score pending an actual similarity measure's specification, marked
HYPOTHESIZED here deliberately because no concrete measure is asserted as chosen); the
candidate's own evidence level from the property model (IAS-F1 Part IV §4.2) — an OBSERVED
capability scores higher confidence than a HYPOTHESIZED one for the same lexical match; and the
candidate's LAST-VALIDATED recency (Part III §3.2), because a stale entry's applicability
predicate may no longer describe the system's actual current behavior. This confidence score is
consumed whole by Part X's routing-under-uncertainty machinery; decomposition and matching
PRODUCE the uncertainty signal, they do not themselves resolve it, which is a deliberate
separation of concerns — matching should not silently discard a low-confidence candidate, because
a discarded low-confidence candidate is unavailable to Part IX's arbitration even in situations
where every other candidate is worse. (DERIVED, NORMATIVE separation of concerns.)

5.6 Worked example: decomposing the hardening-then-ship mission end to end. Returning to §1.2's
synthetic mission, "harden this module against the top five predictable failure modes, then ship
it": the pre-scan (§5.3) fires on "ship," surfacing the deploy-governance gating trigger as a hard
constraint before anything else is extracted. Full decomposition then yields two (intent, object)
pairs: (harden, module) and (ship, module), plus one implicit pair the Owner's sentence never
states but standing governance requires — (classify-tier, module), because the estate's own
spec-gate discipline (HR-SPEC-001, referenced by name only as a governance fact this worked
example inherits, not re-derived here) requires a tier classification before an L/XL-shaped
mission proceeds. Matching against the synthetic index entries of Part III §3.5 finds Entry Alpha
for (classify-tier, module) at high confidence (OBSERVED evidence, LOW cost-to-route, exact
applicability match), Entry Beta for (harden, module) at high confidence, and Entry Gamma for
(ship, module) at high confidence with FINAL authority correctly flagged as a hard sequencing
constraint. No arbitration is needed in this worked instance because each pair produced exactly
one high-confidence candidate; Part IX's arbitration machinery is reserved for the harder case
where a pair produces two or more candidates of comparable confidence, illustrated there rather
than here to keep this Part's worked example focused on decomposition proper. (Illustrative,
continuing the synthetic example established in Part III.)

5.7 Failure shape specific to decomposition: the UNDER-DECOMPOSED MISSION. A mission stated too
tersely for reliable extraction — "fix it," with no named object, no named intent beyond the
vaguest possible verb — produces empty or near-empty (intent, object) pairs, and matching against
an empty pair either returns no candidates (a legitimate empty route, surfaced to the Owner as
insufficient information rather than guessed at) or, worse, returns EVERY candidate whose
applicability predicate is unconditionally true, a degenerate case this dataset treats as a
decomposition failure requiring Owner clarification rather than a routing decision to be made
regardless. IAS-A1 does not attempt to guess a plausible decomposition from an under-specified
mission, because a guessed decomposition that happens to route correctly is indistinguishable,
in its own audit trail, from a guess that happened to route incorrectly, undermining Part XIII's
provenance discipline before it can even begin. (NORMATIVE, DERIVED from the provenance
requirement stated fully in Part XIII.)

FINAL LAW — PART V. Decomposition is where most routing error actually originates, not in the
index and not in the matching arithmetic; a mission's intents, objects, and constraints —
including the implicit constraints standing governance imposes and the Owner's sentence never
states — must be extracted explicitly and gating triggers pre-scanned before any candidate is
matched, because a decomposition that guesses produces a route indistinguishable, on inspection,
from one that reasoned.

PART VI — THE CROSS-SYSTEM COMPOSE ENGINE (ABSORBING CANDIDATE 2)
===============================================================================================

6.1 What composition adds once matching is done. Part V ends with, per (intent, object) pair, one
or more candidate capability-index entries. A mission is not satisfied by matching alone; it is
satisfied by COMPOSING the matched candidates into a single ordered structure whose combined
output is the mission's deliverable. This is the object the Non-Duplication Ledger names for
Candidate 2, the Cross-System Mission Composer: "composes a mission from capabilities distributed
across systems," explicitly distinguished from one_shot's object, which composes ONE plan within
a single system's or a single mission's internal scope once that mission has already been
resolved to one governing contract. IAS-A1's compose engine operates one level above: it decides
how OUTPUTS FROM DIFFERENT SYSTEMS combine, an object one_shot's contract compiler does not and
should not own, because one_shot presupposes the systems are already known. (DERIVED from the
Ledger's Candidate 2 disposition and its explicit contrast with one_shot's overlap.)

6.2 The three composition operators. IAS-A1 DEFINES three operators sufficient to combine any two
matched candidates' outputs, chosen because every observed mission shape in this dataset's worked
examples decomposes into some combination of the three, and because a larger operator set risks
the same over-specification IAS-F1 Part II §2.5 warns against for primitive objects. SEQUENCE(A,
B) means B's invocation begins only after A's completes, and B receives A's output as part of its
own input context (this is the default operator for any pair connected by a data dependency —
Entry Alpha's tier classification feeding Entry Beta's hardening pass in the §5.6 worked example
is a sequence). PARALLEL(A, B) means A and B may invoke concurrently because neither's input
depends on the other's output, and their outputs are independently retained for a later
composition step — used when a mission genuinely needs two unrelated capabilities whose order
carries no meaning (e.g., a knowledge lookup and a cost estimate that neither informs the other).
MERGE(A, B, rule) means both A's and B's outputs are combined into one composite result according
to an explicit merge rule (concatenation, precedence-by-authority, conflict-flagging for Owner
review) — used when two candidates were matched to the SAME (intent, object) pair at comparable
confidence and both are invoked rather than arbitrated away, a deliberate alternative to Part IX's
arbitration for the specific case where invoking both is cheaper and safer than choosing one
wrongly. (DEFINED — three operators, chosen for coverage and against over-specification.)

6.3 The compose plan as the artifact that becomes the route. The compose engine's output is a
COMPOSE PLAN: a tree whose leaves are (system, capability) pairs from Part V's matched candidates
and whose internal nodes are SEQUENCE, PARALLEL, or MERGE operators. This tree, once validated
against Part III §3.4's taxonomy constraints (gating capabilities must precede what they gate;
terminal capabilities must be leaves with no further sequencing after them), IS the ROUTE defined
in Part II §2.6 — the compose plan and the route are the same object viewed from two angles, plan
before execution, trace after. (DERIVED, DEFINED — this identity is what lets Part XIII's
provenance discipline treat the compose plan as the route's own audit record without a separate
translation.)

6.4 Building the compose plan: dependency-first, not confidence-first. A composer that assembled
candidates in order of match confidence (Part V §5.5) without regard to data dependency would
produce plans that violate the taxonomy's own sequencing constraints — a high-confidence
TERMINAL capability composed before a lower-confidence GATING capability it should have waited
for. The compose engine therefore builds the plan DEPENDENCY-FIRST: gating capabilities are
placed at the earliest legal position their applicability allows (per §5.3's pre-scan); data
dependencies among TRANSFORMING capabilities are resolved into SEQUENCE nodes wherever one
candidate's applicability predicate names another candidate's output as part of its own input;
independent TRANSFORMING capabilities with no shared dependency become PARALLEL nodes by default,
because parallel execution is strictly cheaper in wall-clock terms than an unnecessary sequence
and IAS-F1's own economy-of-primitives discipline (Part II §2.5) favors the cheaper structure
absent a reason to prefer the costlier one; and the mission's TERMINAL capability, once identified
by the object model of §5.2, is always the tree's root, never an intermediate node. (DERIVED,
NORMATIVE ordering discipline.)

6.5 Worked example: composing the hardening-then-ship mission. Continuing §5.6's three matched
candidates — Entry Alpha (classify-tier, GATING), Entry Beta (harden, TRANSFORMING), Entry Gamma
(ship, TERMINAL, FINAL authority) — the compose engine produces SEQUENCE(Alpha, SEQUENCE(Beta,
Gamma)): Alpha's gating output (the tier classification) is required before Beta's hardening pass
begins, because Beta's own applicability predicate in this synthetic example names "known scope"
as a precondition; Beta's hardened-module output then feeds Gamma's deploy-governance check as
its input; and Gamma, correctly, is the tree's root and its terminal leaf simultaneously, since
this mission has exactly one terminal capability. Had the mission also required, independently,
"summarize the change for the Owner" — a capability with no data dependency on Alpha, Beta, or
Gamma's internals, only on the fact that they ran — the compose engine would instead produce
SEQUENCE(Alpha, PARALLEL(Beta, summarize-capability)) feeding into Gamma, illustrating the
PARALLEL operator's legitimate use once a second, independent (intent, object) pair enters the
mission. (Illustrative, continuing the running synthetic example.)

6.6 Why composition, not routing alone, is what the Strengthening Matrix's "missions reach the
right capability first" gain actually depends on. A router that correctly matched every candidate
in Part V but composed them naively — say, always sequencing in the order candidates were
discovered rather than in dependency order — would still deliver wrong-shaped missions: Gamma
invoked before Beta completes, a deploy check run against an unhardened module, technically
"routing to the right systems" while producing an operationally wrong mission. The compose engine
is therefore not a cosmetic ordering step; it is the mechanism that converts a correct SET of
routing decisions into a correct SEQUENCE of routing decisions, and the Strengthening Matrix's
evidence requirement (a mission trace where A1 selects a system the parent routers could not see)
is only meaningful once composition guarantees that "selects" also means "at the right moment."
(DERIVED, NORMATIVE.)

6.7 Failure mode specific to composition: OVER-EAGER PARALLELIZATION. §6.4's cheaper-structure
preference for PARALLEL over an unnecessary SEQUENCE carries its own risk if applied mechanically:
two capabilities with no EXPLICIT data dependency in their applicability predicates may still share
an IMPLICIT dependency the predicates do not encode — Entry Beta's hardening pass and a hypothetical
"summarize the change" capability might appear independent by predicate inspection alone, yet the
summary is more useful, and sometimes only correct, once the hardening pass's actual findings exist
to summarize. A compose engine that parallelizes on predicate-visible dependency alone would produce
a summary written against pre-hardening state, silently stale the instant hardening completes. IAS-A1
guards against this by treating TRANSFORMING capabilities whose OUTPUT NAME textually overlaps a
sibling capability's own declared inputs (Part III §3.2's signature fields) as a WEAK dependency
signal even absent an explicit predicate reference — not strong enough to force a SEQUENCE outright,
but strong enough to demote the pair from automatic PARALLEL to a flagged CANDIDATE-PARALLEL that
the Owner or the invoking system may confirm or override before dispatch. This is a deliberate,
NORMATIVE friction: the cost of an occasional unnecessary confirmation prompt is small relative to
the cost of a mission silently completing on stale, out-of-order state. (DERIVED, NORMATIVE.)

6.8 Worked contrast: when a compose plan must be REJECTED rather than merely reordered. Extend the
running hardening-then-ship example with a synthetic Entry Theta whose applicability predicate for
"ship" REQUIRES a signed attestation Entry Gamma alone does not produce, and suppose no matched
candidate in the current mission's candidate set provides that attestation capability at all. A
compose engine that composed anyway — SEQUENCE(Alpha, SEQUENCE(Beta, Gamma)) — would produce a
structurally valid TREE that is nonetheless an INVALID mission plan, because Gamma's own precondition
is unmet by anything upstream of it. Part VI's validation step (§6.3) therefore checks not only
acyclicity and taxonomy ordering but also PRECONDITION SATISFACTION per node: every input a node's
applicability predicate names as required must be traceable to either the mission's own initial
input or a preceding node's output, and a node whose precondition traces to nothing is a validation
FAILURE, not a warning — the compose plan is rejected outright and the missing precondition is
surfaced through Part X's DECOMPOSITION GAP reporting for that specific sub-requirement, rather than
dispatched incomplete and left to fail expensively at Gamma's own invocation. (DERIVED, closing a
validation gap the base mechanism of §6.3 implies but does not itself enumerate explicitly.)

FINAL LAW — PART VI. Matching finds which systems could serve a mission; composition decides how
their outputs combine into one mission — and a router that gets the set right while composing the
sequence wrong has not routed correctly, it has merely assembled the right ingredients in an order
that ruins the dish.

PART VII — WORKFLOW SYNTHESIS: THE TEMPORARY, MISSION-SCOPED STRUCTURE (ABSORBING CANDIDATE 6)
===============================================================================================

7.1 What distinguishes workflow synthesis from the compose plan itself. Part VI's compose plan is
a STATIC tree, valid the instant it is built from Part V's matched candidates. WORKFLOW SYNTHESIS,
the object the Ledger routes here as Candidate 6 (Workflow Synthesis, SUBENGINE → IAS-A1,
overlapping one_shot and ultra only at "compose one workflow" while its unique object is temporary
workflows built from CROSS-SYSTEM capabilities), is the RUNTIME STRUCTURE that actually executes
the compose plan, carries live state between nodes, and — critically — DISSOLVES once the mission
completes, leaving no standing artifact in the ensemble beyond the route trace Part XIII records
for provenance. The distinction matters because a compose plan is reusable ACROSS missions of the
same shape (Part III's index-driven matching will, for a structurally identical future mission,
produce the same or a near-identical plan), while the synthesized workflow instance is not — it is
built fresh, holds this mission's specific state, and is discarded. (DEFINED, DERIVED from the
Ledger's own distinction between "one workflow" (one_shot/ultra's object) and "temporary workflows
from cross-system capabilities" (this dataset's absorbed object).)

7.2 Why a temporary structure, never a standing one. IAS-F1's Invariant One (Part V §5.2, no
orphan system) and the estate's own biodiversity concerns (Part XV of the Federation Ontology,
referenced there, not re-derived here) together imply a caution against IAS-A1 quietly
accumulating standing infrastructure for every mission shape it has ever routed — a "workflow
library" that grows without bound would itself become a system requiring its own governance, cost
accounting, and staleness discipline, effectively spawning a seventy-first-plus system by
accretion rather than by deliberate admission. Workflow synthesis therefore builds and discards by
design: the workflow INSTANCE lives only for the mission's duration; what persists is the
capability index (Part III, standing, maintained) and the route trace (Part XIII, standing,
append-only), never the workflow instance itself. (NORMATIVE, DERIVED — the discipline is a direct
application of the estate's general aversion to unbounded standing infrastructure, applied to this
specific dataset's own temptation to accumulate.)

7.3 The runtime state a synthesized workflow carries. Each workflow instance carries: a MISSION
HANDLE (a reference back to the originating mission, for provenance); a POSITION pointer into the
compose plan tree (which node is currently executing, which have completed, which remain); an
ACCUMULATED CONTEXT — the running set of outputs produced by completed nodes, structured so that a
not-yet-executed node's input requirements (per §6.4's dependency resolution) can be satisfied by
querying this accumulated context rather than re-deriving it; and a FAILURE LEDGER, empty at
synthesis time, populated if any node's invocation fails, consumed by Part X's uncertainty
handling and Part XIX's failure-mode catalogue. This runtime state is exactly what makes Part
VIII's handoff protocol possible: a handoff is, formally, the accumulated context being made
available to the next node in position order, without loss. (DEFINED, forward-referenced to Part
VIII.)

7.4 Synthesis versus mere execution: the workflow's adaptive property. A naive reading of "execute
the compose plan" suggests workflow synthesis is nothing more than a tree-walk interpreter. It is
more than that, and the extra property is what earns the name SYNTHESIS rather than EXECUTION:
the workflow instance may, within limits Part VII §7.5 bounds explicitly, RE-COMPOSE its remaining
structure mid-mission if a completed node's actual output diverges materially from what Part V's
decomposition predicted. If Entry Alpha's tier classification (§6.5) returns a tier the original
decomposition did not anticipate — say, a mission initially read as MICRO-scoped in fact requires
an XL-tier spec-gate check the pre-scan of §5.3 did not surface because the mission's stated
intent undersold its own scope — the synthesized workflow may insert the newly-required gating
node into its remaining plan before continuing, rather than either failing outright or silently
proceeding on a stale plan. This adaptive re-composition is bounded (§7.5) precisely so it does
not become an unbounded, ungoverned mid-flight redesign. (DERIVED, HYPOTHESIZED mechanism pending
the bound's own validation in practice — marked HYPOTHESIZED because no live instance of this
specific re-composition trigger has yet been OBSERVED against the real estate; the mechanism is
proposed on the strength of the dependency logic already established in Parts V and VI, not on
empirical confirmation.)

7.5 The bound on adaptive re-composition: gating-only, single-pass. To prevent the adaptive
property of §7.4 from becoming an ungoverned loop, IAS-A1 NORMATIVELY restricts mid-mission
re-composition to two conditions, both required: the newly-required node must itself be a GATING
capability (Part III §3.4) discovered via a completed node's actual output, never a TRANSFORMING
or TERMINAL capability inserted on a hunch; and re-composition may occur at most ONCE per
workflow instance — a workflow that would need a second re-composition is treated instead as
evidence the original decomposition (Part V) was too unreliable to trust further, and the mission
is surfaced to the Owner for re-decomposition from scratch rather than iteratively patched. This
bound is what keeps workflow synthesis a SUBENGINE of routing rather than an independent planning
system re-litigating the mission indefinitely — exactly the containment the Ledger's SUBENGINE
verdict for Candidate 6 implies, as distinct from a BUILD verdict that might have licensed
unbounded synthesis logic. (NORMATIVE, DERIVED from the Ledger's own SUBENGINE disposition.)

7.6 Why workflow synthesis strengthens one_shot and ultra rather than competing with them. Once a
compose plan node is reached whose capability is provided by a system that itself runs one_shot's
contract-compilation logic or ultra's seven-phase protocol, workflow synthesis does not re-
implement or bypass that logic; it INVOKES the system at that node exactly as it would invoke any
other, passing the accumulated context (§7.3) as that system's input, and receiving that system's
own internally-composed output back into the accumulated context for the next node. one_shot and
ultra remain fully in control of what happens WITHIN their own invocation; workflow synthesis
controls only the SEQUENCE of invocations across the whole mission. This is the same boundary
discipline Part IV established for the resource routers, applied here to two mission-scoped
compilers instead of two resource routers, and it is the specific mechanism by which the
Strengthening Matrix's claim about one_shot ("IAS-A1 routes at the system layer above each
parent's resource router") extends to one_shot's mission-scoped role without contradiction —
one_shot is not a resource router in the CO-03 sense, but it is still a single-system-scoped
compiler that IAS-A1's ensemble-scoped composition sits above. (DERIVED, NORMATIVE.)

7.7 The cost of synthesis itself: why a temporary structure is not a free structure. Building a
workflow instance — allocating a mission handle, initializing an empty accumulated context and
failure ledger, positioning a pointer into the compose plan — is not zero-cost merely because the
result is discarded at mission end. For a CLEAN-MATCH mission (Part XV §15.2) with a short compose
plan, synthesis overhead is negligible relative to the capabilities it coordinates; for a
MULTI-HANDOFF mission chaining many PARALLEL and MERGE nodes, the accumulated context itself can
grow large enough that its own maintenance — reconciling branch-local copies at each MERGE point,
per §8.4's fan-out discipline — becomes a measurable cost IAS-A1 must account for in its own
COST-TO-ROUTE arbitration criterion (Part IX §9.2), rather than treating synthesis as a free
substrate beneath the capabilities it coordinates. This is named explicitly because a specification
that treats its own coordination mechanism as costless risks under-pricing complex missions relative
to simple ones in exactly the arbitration and evaluation layers that later Parts build on top of
this one. (DERIVED, NORMATIVE — an honest cost acknowledgment rather than an implicit free-lunch
assumption.)

7.8 Workflow instance identity versus mission identity: a distinction the re-composition bound
depends on. §7.5's single-pass re-composition bound is meaningful only if "the same workflow
instance" is a well-defined notion across a re-composition event — otherwise a router could evade
the bound by quietly instantiating a fresh workflow instance under the same mission handle whenever
a second re-composition is tempting, technically obeying "at most once per instance" while violating
its evident intent. IAS-A1 therefore DEFINES workflow instance identity as bound to the MISSION
HANDLE for the instance's entire lifetime: a mission handle may have at most one active workflow
instance at any moment (the same invariant Part XIX §19.3's split-brain defense depends on), and
re-composition is counted against that handle's own history, not against whichever instance object
happens to currently hold it. A second re-composition attempt against a handle that has already
re-composed once is rejected at the mission-handle level, forcing the escalation §7.5 already
specifies, regardless of whether the attempt originates from a freshly-instantiated object. (DEFINED,
NORMATIVE, closing an evasion path the bound's literal wording alone would not have closed.)

FINAL LAW — PART VII. A synthesized workflow is a scaffold, not a building: it carries this
mission's state faithfully from node to node, may adapt exactly once and only for a newly-
discovered gating requirement, and dissolves the instant the mission completes — because an
ensemble that let its router accumulate standing workflow infrastructure would have built a
seventy-first system by accident, one temporary structure at a time.

===============================================================================================
PART VIII — THE HANDOFF PROTOCOL: PASSING CONTROL BETWEEN SYSTEMS WITHOUT STATE LOSS
===============================================================================================

8.1 The absorbed object, restated precisely. Candidate 8, the Capability Handoff Protocol, is
routed to IAS-A1 as a SUBENGINE with the unique object "mission handoff between systems without
state loss," distinguished from its closest parents PM-02 (pane/session handoff, one level down
at the pane granularity) and session_resilience (recovery of a SINGLE session's continuity, not a
cross-system relay). This Part specifies the mechanism precisely, because §7.3 already named the
accumulated context as the substrate a handoff moves, and this Part is where "moves without loss"
is actually defined and defended against the specific ways loss occurs in practice. (DERIVED from
the Ledger's Candidate 8 disposition.)

8.2 The three loss modes a handoff must prevent, named individually. TRUNCATION LOSS occurs when
the accumulated context exceeds a receiving system's input capacity and is silently clipped rather
than summarized or flagged — the receiving system proceeds on a partial context believing it to be
complete. TRANSLATION LOSS occurs when the accumulated context's structure does not match the
receiving system's expected input schema and a lossy ad-hoc conversion occurs — a structured
tier-classification result flattened into a single string a downstream system can only partially
parse. AUTHORITY LOSS occurs when a handoff drops the FINAL-versus-ADVISORY authority marker (Part
III §3.4) a prior node's output carried — a receiving system that treats a FINAL gating verdict as
merely advisory input has lost the single most consequential bit of the handoff, because it may
then proceed past a block the ensemble's own invariants (IAS-F1 Part V, Invariant Three) required
to be respected. (DEFINED — three named loss modes, each independently preventable and each
requiring its own defense.)

8.3 The defense against each loss mode. Against TRUNCATION LOSS: the handoff protocol requires the
accumulated context to be passed as a STRUCTURED RECORD (per node: capability invoked, output
summary, authority level, evidence level) rather than a flattened transcript, and a receiving
system whose input capacity cannot hold the full record receives a COMPRESSED record with an
explicit "N prior nodes summarized, full record available on request" marker — never a silent
truncation. Against TRANSLATION LOSS: the structured record's schema is fixed by this Part (the
same four fields — capability, output summary, authority, evidence — for every node, regardless of
which system produced or receives it), so no receiving system needs a bespoke parser for each
possible predecessor; the schema itself is the interoperability layer, the same structural-
boundary-over-discipline principle Part IV §4.5 already established for the resource-router
boundary, applied here to the handoff's own data shape. Against AUTHORITY LOSS: the authority field
is NEVER optional in the structured record and a receiving system's own invocation contract must
explicitly acknowledge it before proceeding past any node whose authority was FINAL — an
acknowledgment gap here is treated, per Part XIX's failure catalogue, as a routing failure
attributable to IAS-A1's own handoff mechanism, not to either system individually. (DEFINED,
NORMATIVE — one defense per named loss mode.)

8.4 Handoff at PARALLEL and MERGE nodes: the harder case SEQUENCE alone does not cover. §8.2's
loss modes are easiest to reason about for SEQUENCE nodes, where exactly one predecessor hands off
to exactly one successor. PARALLEL nodes (Part VI §6.2) require the handoff protocol to fan the
SAME accumulated context out to multiple concurrent receivers without any one of them corrupting
the shared record for the others — achieved by treating the accumulated context as read-only at
the moment of a PARALLEL fan-out, with each concurrent branch appending its own new node record
to its own branch-local copy, reconciled only at the MERGE point that follows. MERGE nodes (Part
VI §6.2) require the handoff protocol to combine two branch-local accumulated contexts into one,
governed by the same explicit merge rule the compose plan already specified for that MERGE node
(concatenation, precedence-by-authority, or conflict-flagging) — a merge rule that was never
specified at compose time is treated as a compose-plan defect (Part VI §6.3's validation step),
never resolved ad hoc at handoff time. (DERIVED from Part VI's operator definitions, extended
here to the handoff mechanism specifically.)

8.5 Why handoff-without-loss is what makes composition's value real rather than nominal. Part VI
argued that composition, not routing alone, delivers the Strengthening Matrix's gain. That
argument is incomplete without this Part: a compose plan that is structurally correct but whose
handoffs lose authority, truncate context, or mistranslate structure delivers, in practice, the
same operational failure as a naively-ordered plan — a downstream system acting on information it
should not trust, or failing to act on a constraint it should have respected. The handoff protocol
is therefore not a peripheral concern bolted onto routing; it is the mechanism that makes the
distinction between "IAS-A1 correctly decided the sequence" and "IAS-A1's mission actually
succeeded" collapse into the same fact, which is the standard Part XIV's evaluation Part later
holds this dataset to. (DERIVED, NORMATIVE.)

8.6 A fourth loss mode, named for completeness: STALENESS LOSS. §8.2 named three loss modes at
handoff time — truncation, translation, authority. A fourth exists at a slightly different point in
the mission's timeline: the accumulated context, once handed to a receiving system, may sit
unconsumed for a materially long interval if that system's own queue or invocation schedule is
backed up, and by the time it is actually acted upon, the WORLD the context describes may have
changed underneath it — a tier classification computed against a module's state at minute zero,
consumed only at minute forty after the module was independently modified by a concurrent process
outside this mission's own scope. STALENESS LOSS is distinct from the three §8.2 modes because
nothing about the record's structure, translation, or authority marking was ever wrong; the record
was simply true when written and may no longer be true when read. The defense is a TIMESTAMP field
added to every structured record entry (alongside capability, output summary, authority, and
evidence), and a receiving system whose own invocation occurs more than a bounded interval after
the record's timestamp is required to treat the record as PROVISIONAL — re-confirming the relevant
fact if the cost of re-confirmation is low, or explicitly flagging the staleness risk in its own
output if re-confirmation is not feasible, rather than silently trusting a record whose currency it
cannot itself verify. (DEFINED, NORMATIVE — a fourth loss mode and its defense, extending §8.2's
catalogue for completeness.)

8.7 Handoff under partial failure: what a downstream node receives when an upstream node's own
invocation only partially succeeds. §8.2 through §8.4 assume each node's invocation cleanly succeeds
or cleanly fails; in practice a system may return a PARTIAL result — three of five requested
hardening checks completed, two timed out — and the handoff protocol must specify what the next
node in SEQUENCE receives in that case, rather than leaving it ambiguous whether a partial result
counts as a completed handoff at all. IAS-A1 requires every structured record entry's output summary
to carry an explicit COMPLETENESS marker (FULL or PARTIAL, with the partial case naming which
sub-components are missing), and a downstream node whose own applicability predicate requires a
FULL upstream result is blocked from proceeding on a PARTIAL one — the mission's workflow instance
treats this exactly as a node-level failure for FAILURE LEDGER purposes (Part VII §7.3), even though
the upstream system itself did not report outright failure, because "some of what was required" is,
for a downstream consumer with no way to distinguish a missing check from a check that passed
silently, operationally equivalent to not knowing. A downstream node whose predicate can tolerate a
PARTIAL result (rare, but real for advisory-outcome, best-effort capabilities) may proceed, with the
partial marker propagated forward into ITS OWN output summary rather than silently upgraded to FULL.
(DEFINED, NORMATIVE — closing an ambiguity the base mechanism's binary success/failure framing left
open.)

8.8 Why the handoff protocol is itself a candidate for the capability index, and why it is not
listed there. A reader familiar with Part III might reasonably ask whether the handoff protocol
mechanism described in this Part should itself appear as a capability index entry, matchable and
dispatchable like any other system's offering. It should not, and the reason is structural rather
than incidental: the capability index catalogues what SYSTEMS the ensemble can route MISSIONS to;
the handoff protocol is not a system a mission is ever routed TO, it is the substrate BY WHICH every
routing decision's own effects propagate, invoked implicitly at every SEQUENCE, PARALLEL, and MERGE
transition rather than explicitly matched against an (intent, object) pair. Indexing it would
misrepresent it as one candidate among many competing for a mission's business, when it is instead
infrastructure every mission depends on uniformly — the same distinction Part IV draws between a
resource router's object and the boundary IAS-A1 must not cross into. (DERIVED, NORMATIVE, a
boundary clarification specific to this Part's own mechanism.)

FINAL LAW — PART VIII. A route that reaches the right system with the wrong, truncated, or
mis-authorized context has not actually reached the right system; the structured, schema-fixed,
authority-preserving handoff record is what makes "the sequence was correct" and "the mission
succeeded" the same claim rather than two claims that happen to coincide when nobody is watching.

===============================================================================================
PART IX — MULTI-MATCH ARBITRATION: WHEN MORE THAN ONE SYSTEM COULD SERVE
===============================================================================================

9.1 The case Part V deferred. Part V §5.6's worked example was chosen deliberately to need no
arbitration — each (intent, object) pair matched exactly one high-confidence candidate. Real
missions are not always so accommodating: an estate with, as OBSERVED, redundant capability
coverage in places (the ecology dataset IAS-D1's own object includes judging whether such
redundancy is healthy diversity or wasteful competition) will regularly present IAS-A1 with two
or more candidates of comparable MATCH CONFIDENCE (Part V §5.5) for the same pair. This Part
specifies how that comparable-confidence case is resolved, because Part VI's MERGE operator only
covers the subset of cases where invoking both is the right answer; many cases require choosing
exactly one, and that choice needs a defensible procedure rather than an arbitrary tie-break.
(DERIVED, NORMATIVE.)

9.2 The arbitration criteria, in strict priority order. IAS-A1 arbitrates a tied candidate set by
four criteria applied in strict, non-averaged priority order — never a weighted blend, because a
blended score obscures WHICH criterion decided, undermining Part XIII's provenance requirement
that a route's reasoning be inspectable. First, AUTHORITY PRECEDENCE: a candidate whose property-
model authority (Part III §3.2, imported from IAS-F1 Part IV §4.2) is FINAL for the relevant
decision class is preferred over one that is merely ADVISORY, because IAS-F1's Invariant Three
already establishes that only one system may hold final authority per decision class — if two
candidates both claim FINAL authority over the same decision class, this is not an arbitration
case at all but an Invariant Three violation requiring escalation (Part XI), not a routing choice
IAS-A1 may resolve unilaterally. Second, EVIDENCE LEVEL: among candidates of equal authority, an
OBSERVED capability is preferred over a HYPOTHESIZED one, and a HYPOTHESIZED capability over an
UNRESOLVED one, directly inheriting IAS-F1 Part IV's epistemic vocabulary at the routing decision
point. Third, COST-TO-ROUTE: among candidates equal on the first two criteria, the cheaper
invocation is preferred, consistent with Part VI §6.4's general preference for the cheaper
structure absent a reason to prefer the costlier one. Fourth, and only if the first three remain
tied, RECENCY OF SUCCESSFUL INVOCATION (the capability index's LAST-VALIDATED field, Part III
§3.2): the more recently and successfully exercised candidate is preferred, on the DERIVED
reasoning that recent success is weak but real evidence against undetected staleness. (DEFINED,
NORMATIVE — four criteria, strict order, never blended.)

9.3 Why strict order rather than a weighted score. A weighted-average arbitration score is
tempting because it collapses four criteria into one comparable number, but it purchases that
convenience at the cost of exactly the auditability Part XIII requires: a route chosen by a
composite score of 0.71 versus 0.69 answers "which criterion mattered" with "an opaque blend,"
while a route chosen because Candidate A held FINAL authority and Candidate B did not answers
the same question in one sentence. This is the same structural preference for legible mechanism
over convenient aggregation that IAS-F1 itself models throughout — the ontology's own invariants
(Part V) are stated as a conjunctive checklist, never a weighted health score, for the identical
reason. (DERIVED, NORMATIVE, cross-referencing IAS-F1's own methodological choice.)

9.4 The genuinely-tied case: when all four criteria exhaust without resolution. Two candidates
that remain tied after authority, evidence, cost, and recency have all been checked represent a
case IAS-A1 does not resolve by an arbitrary tiebreak (a coin-flip arbitration would be
unfalsifiable and unauditable, violating the same provenance discipline §9.3 already invoked).
Instead, a genuinely-tied pair is surfaced as a MERGE candidate per Part VI §6.2's third operator
— both are invoked, with a conflict-flagging merge rule, and the resulting composite output is
routed to the Owner (or, for advisory-authority pairs, to the mission's own downstream nodes) with
both candidates' outputs attached and the tie explicitly disclosed, rather than resolved silently.
This is a deliberate design choice: an unresolvable tie is information the mission's consumer
should have, not noise IAS-A1 should hide behind a false appearance of confident selection.
(NORMATIVE, DERIVED from the provenance discipline of Part XIII, forward-referenced.)

9.5 Worked example: arbitrating a genuine tie in the synthetic index. Extend the running example
with a synthetic Entry Delta, near-identical in applicability predicate to Entry Beta (§3.5): both
match (harden, module) at comparable lexical confidence, both carry OBSERVED evidence, both carry
ADVISORY authority. Entry Beta's cost-to-route is MEDIUM (domain-catalogue load); Entry Delta's is
LOW (a lighter, narrower archetype set). By criterion three (cost-to-route), Delta is preferred
over Beta without needing criterion four at all — arbitration resolves cleanly at the third
criterion, illustrating that most real ties are not genuine four-criterion exhaustions but resolve
at authority, evidence, or cost well before recency is ever consulted; the genuinely-tied case of
§9.4 is, by this dataset's own HYPOTHESIZED expectation, the minority shape rather than the
common one, though no live frequency measurement against the real estate is asserted here.
(Illustrative, HYPOTHESIZED frequency claim marked as such.)

9.6 Arbitration's relationship to the DUPLICATES relation. A pair of candidates that arbitration
repeatedly resolves identically, mission after mission, with one candidate never once winning
regardless of context, is itself a signal worth surfacing outward: a capability that never wins
arbitration against another may be a genuine DUPLICATES relation (IAS-F1 Part VI §6.5) that D2A
has not yet resolved, rather than a healthy competing alternative. IAS-A1 does not itself render
a duplicates verdict — that remains D2A's object, imported by reference per Part II — but its
arbitration history is exactly the kind of OBSERVED evidence a duplicates review would want, and
Part XVI's integration Part specifies this signal's path outward to the systems that do own that
judgment. (DERIVED, forward-referenced to Part XVI.)

9.7 Arbitration cost: why a strict four-criterion checklist does not scale linearly with candidate
count. §9.2's four criteria are described for a PAIR of tied candidates, but a genuinely crowded
(intent, object) pair — one an active, growing ensemble may plausibly present with three, four, or
more comparable-confidence candidates as the estate's own capability surface expands — requires the
same strict ordering applied across the whole SET, not merely a pair, and a naive pairwise
tournament (compare A against B, the winner against C, and so on) risks a non-transitive surprise if
COST-TO-ROUTE or RECENCY happen to be measured on drifting live data between comparisons rather than
a single frozen snapshot. IAS-A1 therefore requires arbitration across a candidate set of size
greater than two to evaluate all four criteria against a SINGLE frozen snapshot of the capability
index taken at the moment arbitration begins, then rank the full set by that snapshot in one pass,
rather than a sequence of pairwise comparisons against a index that may itself be updating mid-
arbitration (Part III §3.3's incremental re-extraction does not pause for an in-flight arbitration).
This closes a specific, narrow failure: a candidate that would have lost to A on cost but wins
against A after A's own cost was updated mid-comparison by an unrelated re-extraction event. (DEFINED,
NORMATIVE, extending §9.2 from the pairwise case to the general N-candidate case.)

9.8 Arbitration and the Owner's own override. Every criterion in §9.2 and every disclosed tie in
§9.4 produces a route IAS-A1 itself considers correctly reasoned, but "correctly reasoned by IAS-A1's
own criteria" is not the same claim as "correct for this Owner's actual, possibly idiosyncratic,
preference this one time" — an Owner may hold institutional knowledge about a candidate's real-world
reliability that has not yet propagated into the capability index's own EVIDENCE LEVEL or LAST-
VALIDATED fields. IAS-A1 therefore treats every arbitrated route, for missions where the Owner is
directly present rather than the mission originating from an upstream automated system, as carrying
an OVERRIDE WINDOW: the Owner may substitute a different candidate from the same original candidate
set before dispatch proceeds, and that substitution is recorded in the provenance record (Part XIII
§13.2) as an OWNER OVERRIDE, distinct from and never conflated with an ordinary arbitration outcome,
because a pattern of repeated overrides against the same criterion is itself valuable evidence
(per Part XX §20.5's evidence-to-governance path) that the criterion's own weighting may deserve
Owner-gated revision. (DERIVED, NORMATIVE, tying arbitration to the Owner-authority discipline
established generally in Part XI.)

FINAL LAW — PART IX. Arbitration is a strict, ordered, legible checklist — authority, then
evidence, then cost, then recency — never a blended score, because a route's provenance must be
able to name the one criterion that decided; and a tie that survives all four criteria is
disclosed as a tie, never resolved by an unauditable coin-flip dressed as confidence.

PART X — ROUTING UNDER UNCERTAINTY AND ABSENT CAPABILITY
===============================================================================================

10.1 The harder case: not too many candidates, but too few or none. Part IX addressed the case of
excess — multiple plausible candidates. This Part addresses its inverse: a (intent, object) pair
for which the capability index (Part III) returns zero candidates, or returns candidates whose
MATCH CONFIDENCE (Part V §5.5) falls below a usability threshold. This is not a rare edge case;
it is the routing-layer manifestation of the estate's own general incompleteness — no federation
of seventy-plus systems covers every conceivable mission shape, and a router that pretends
otherwise, by forcing a low-confidence match rather than admitting absence, produces the single
most damaging routing error this dataset can commit: a confidently wrong route. (DERIVED,
NORMATIVE.)

10.2 The three-tier uncertainty response, calibrated to confidence band. IAS-A1 DEFINES three
confidence bands with distinct responses, rather than a single binary match/no-match decision.
HIGH CONFIDENCE (above a threshold this dataset does not numerically fix, marking the exact
cutoff HYPOTHESIZED pending calibration against real routing outcomes) proceeds directly to
composition (Part VI) with no special handling. MODERATE CONFIDENCE proceeds to composition but
the resulting route carries an explicit UNCERTAINTY FLAG in its provenance record (Part XIII),
visible to any downstream consumer of the route trace, so that a mission that succeeds despite a
moderate-confidence match is distinguishable, in the historical record, from one that succeeded
on a clean high-confidence match — a distinction Part XIV's metrics need to avoid conflating luck
with reliability. LOW CONFIDENCE OR ABSENT does not proceed to composition at all; it is surfaced
as a DECOMPOSITION GAP, reported to the Owner (or the upstream mission-issuing system) as "no
system in the current federation reliably covers this requirement," which is itself valuable
institutional information — a demand signal Part XVII's integration with IAS-C2 (Demand
Forecasting) is specifically built to consume. (DEFINED, NORMATIVE — three bands, three distinct
responses, none of which silently guesses.)

10.3 Why "no route found" is a valid and valuable output, not a router failure. A router judged
solely on "percentage of missions successfully routed" has a perverse incentive to lower its own
confidence threshold until every mission finds SOME candidate, however poor a fit — the exact
Goodhart failure Part XIV's metrics Part must guard against explicitly. IAS-A1's design instead
treats an honest DECOMPOSITION GAP report as a successful exercise of the router's actual job:
distinguishing "this mission is servable by the current federation" from "it is not, yet." A gap
report is not merely a graceful failure mode; per §10.2, it is itself the raw signal from which
IAS-C2 forecasts future capability demand, meaning an honestly-reported gap has MORE downstream
institutional value than a forced low-confidence match that happens to squeak by, because the
forced match generates no actionable signal for what the estate should build next. (DERIVED,
NORMATIVE — the incentive design is the load-bearing claim here.)

10.4 Partial coverage: when a mission decomposes into pairs of mixed confidence. The common real
case is neither uniformly high nor uniformly absent but MIXED: a hardening-then-ship mission
where the classify-tier and ship pairs match at high confidence but the harden pair, for a
sufficiently novel code shape, matches only at moderate or low confidence. IAS-A1 does not reject
the whole mission for one weak pair; it composes the route with the confident pairs proceeding
normally and the weak pair either flagged (moderate case, per §10.2) or, if genuinely absent
(low/absent case), presented to the Owner as a partial route with an explicit gap at that specific
position — "the tier classification and deploy check are ready to run; no reliable hardening
capability was found for this code shape; proceed with the two confident steps and handle
hardening manually, or wait for capability development." This graceful partial-routing behavior
is DERIVED directly from the per-pair candidate-set granularity Part V §5.4 established
specifically to make it possible — a mission-level all-or-nothing match/no-match design could not
support this partial behavior at all. (DERIVED, elaborating why Part V's earlier granularity
choice pays off here.)

10.5 Confidence decay under repeated failure, distinct from the index's own staleness field. A
candidate that is matched at moderate or high confidence but whose actual invocation, once
executed, FAILS (populates the workflow instance's FAILURE LEDGER per Part VII §7.3) should not
retain its pre-failure confidence on the next mission that queries the same pair. IAS-A1 applies a
CONFIDENCE DECAY: each observed invocation failure for a given (capability, applicability
pattern) reduces that entry's effective match confidence for future queries by a bounded amount,
recovering gradually on subsequent successful invocations — distinct from, and layered on top of,
the index's own LAST-VALIDATED staleness field (Part III §3.2), because a capability can be
perfectly fresh by validation recency while still exhibiting a live, currently-elevated failure
rate the staleness field alone does not capture. (DERIVED, HYPOTHESIZED mechanism — the specific
decay function and bound are not asserted numerically here, marked HYPOTHESIZED pending
calibration, consistent with §10.2's treatment of the confidence-band thresholds themselves.)

10.6 The relationship between absent-capability reporting and dead-capability routing (previewed).
This Part's DECOMPOSITION GAP (a capability that was never present) is the structural inverse of
Part XII's DEAD-CAPABILITY ROUTING (a capability the index believes is present but which no longer
functions as recorded). Both are failure-adjacent conditions this dataset must distinguish sharply
in its own vocabulary, because the remedy differs: a decomposition gap is closed by BUILDING a new
capability (an IAS-C2/D-family concern, outside this dataset's own authority), while a dead
capability is closed by RE-VALIDATING or RETIRING an index entry (an IAS-A1 maintenance concern,
specified fully in Part XII). Confusing the two in reporting would misdirect institutional effort
— sending a "build this" signal for a capability that already exists but has silently broken, or a
"fix your stale index" signal for a genuine coverage gap that requires new construction. (DEFINED,
forward-referenced to Part XII, NORMATIVE distinction.)

10.7 The threshold calibration problem, named honestly rather than hidden behind a number. §10.2
deliberately does not fix a numeric cutoff between HIGH, MODERATE, and LOW/ABSENT confidence bands,
and this subsection explains why leaving it unfixed here is the correct specification choice rather
than an oversight. A threshold set too high converts genuinely serviceable moderate-confidence
matches into false DECOMPOSITION GAPS, inflating the OVER-CAUTIOUS GAP RATE countermetric Part XIV
§14.3 defines specifically to catch this; a threshold set too low converts genuinely unreliable
matches into false HIGH-confidence dispatches, inflating WRONG-SYSTEM RATE. The correct threshold is
therefore not a constant this specification can derive from first principles — it is an empirical
question the SYNTHETIC MISSION BENCHMARK of Part XV is built partly to surface, and the honest
position this dataset takes is that the threshold is CALIBRATED, not decreed, with its initial value
treated as a HYPOTHESIZED starting point subject to revision through the same Owner-gated evidence
path Part XX §20.4 specifies for every other tunable parameter in this dataset's mechanism.
(NORMATIVE, HYPOTHESIZED, an explicit refusal to overclaim precision this specification cannot yet
earn.)

10.8 Uncertainty compounding across a multi-pair mission: why MODERATE-confidence pairs are not
independent risks. A mission decomposing into several pairs, each individually at MODERATE
confidence, is not merely as risky as any one MODERATE pair considered alone — the PROBABILITY that
at least one of several independently-uncertain pairs turns out wrong compounds across the mission,
a statistical reality this Part names explicitly because §10.4's partial-coverage mechanism handles
each weak pair's OWN uncertainty flag individually but does not, on its own, surface the mission-
level compounded risk. IAS-A1 therefore adds, to the provenance record's mission-level summary
(distinct from any single node's own confidence band), a COMPOUND UNCERTAINTY marker whenever two or
more pairs in the same compose plan carry MODERATE flags simultaneously — not a numeric probability
this dataset is not positioned to compute reliably, but a qualitative signal that the mission's
overall reliability should be read as lower than any single pair's confidence band would suggest in
isolation, giving the Owner or downstream consumer the compounded picture rather than a set of
locally-accurate but globally-misleading individual flags. (DERIVED, NORMATIVE.)

FINAL LAW — PART X. A router that always finds a candidate has stopped being honest about the
federation's actual coverage; the three-tier confidence response — proceed, flag, or report the
gap — is what keeps "no reliable system covers this" a valuable, actionable signal instead of a
failure the router is incentivized to paper over.

===============================================================================================
PART XI — THE AUTHORITY TO DISPATCH A SYSTEM
===============================================================================================

11.1 The question deferred since Part II. Part II §2.8 deferred a specific question: does IAS-A1
itself hold FINAL or ADVISORY authority over its own decision class — "which system(s) act on a
given mission, in what order" — per IAS-F1's Invariant Three (Part V §5.4). This Part answers it,
now that Parts III through X have specified the mechanism whose authority is in question. (DERIVED,
resolving the forward reference.)

11.2 The answer: SPLIT authority, by decision sub-class, never a single blanket grant. IAS-A1
does NOT hold uniform final authority over all routing decisions; it holds authority that SPLITS
along the same taxonomy Part III §3.4 already established. Over ADVISORY-outcome routing (which
candidate serves an ADVISORY pair, per Part IX's arbitration), IAS-A1 holds FINAL authority — a
route to an advisory-authority capability may proceed without further sign-off, because the
capability's own output remains overridable downstream regardless, so an incorrect ROUTE to it
carries bounded risk. Over routing that would DISPATCH a GATING or TERMINAL capability whose
property-model authority is itself FINAL (Part III §3.2), IAS-A1 holds only ADVISORY authority
over the DISPATCH decision — it proposes the route, but the actual invocation of a FINAL-authority
gating or terminal capability requires the acknowledgment discipline Part VIII §8.3 already
specified (an explicit receiving-system acknowledgment) functioning here as the accountability
mechanism, rather than IAS-A1 unilaterally triggering a consequential, hard-to-reverse action.
(DEFINED, NORMATIVE — split authority, not blanket, DERIVED from the taxonomy of Part III §3.4
applied to IAS-A1's own decision class rather than to the capabilities it routes among.)

11.3 Why split authority, not blanket final authority, is the correct resolution. A router with
blanket final authority over every dispatch — including dispatching a FINAL-authority deploy-
governance gate — would concentrate an outsized share of the ensemble's consequential decision
power into a single second-order system whose own correctness (Parts V through X) rests on
extracted, staleness-prone data (the capability index) and confidence heuristics IAS-A1 itself
marks HYPOTHESIZED in several places (§7.4, §10.5). Granting that same system unchecked final
authority over dispatching the estate's highest-consequence gates would be a bet that the router's
own uncertainty machinery is more reliable than it honestly claims to be. Split authority instead
matches the router's actual epistemic position: confident and final where the downstream risk of
a routing error is bounded (advisory outcomes, self-correcting downstream), and advisory itself
where the downstream risk is not bounded (final-authority gates, whose entire purpose is to
prevent an unrecoverable action). (DERIVED, NORMATIVE — this is the specific, mechanical form
Article IX's "governed self-evolution" discipline takes for this dataset's own authority claim,
applied not to self-modification but to dispatch authority.)

11.4 Escalation: what happens when IAS-A1's advisory dispatch proposal is contested. If the
receiving system's acknowledgment (Part VIII §8.3) at a FINAL-authority node does not occur — the
system declines, times out, or flags a disagreement with the proposed route reaching it — IAS-A1
does not retry silently or substitute a different candidate unilaterally; per the arbitration
discipline of Part IX, a disagreement at a FINAL-authority boundary is escalated to the Owner (or
the mission's originating authority) as an explicit CONTESTED DISPATCH, carrying the proposed
route, the receiving system's stated objection, and IAS-A1's own confidence trail (Part V §5.5,
Part X §10.2's confidence bands) as the record the Owner needs to resolve it. This is the ensemble-
level analogue of IAS-F1's own Invariant Three resolution mechanism (arbitration between competing
final authorities routes to IAS-A2 per the Compounding Graph), applied here to the specific case
of a dispatch proposal contested at its point of delivery rather than two systems independently
claiming the same decision class. (DERIVED, NORMATIVE.)

11.5 Authority is not static: how a decision sub-class's authority split can change. The split of
§11.2 is bound to the CURRENT property-model authority markers in the capability index (Part III
§3.2), which are themselves subject to change as systems evolve (D2A dispositions, IAS-F1 Part IX
§9.2's evolving capability signature). A capability that begins as ADVISORY and is later promoted
to FINAL authority (a legitimate system-maturation path the estate's own governance may grant)
automatically shifts IAS-A1's own dispatch authority over routes to it from final to advisory,
without any change to IAS-A1's own code or configuration — the split is DERIVED live from the
index's current state, never hard-coded per capability, which is what lets IAS-A1's authority
claims stay honest as the ensemble evolves around it rather than requiring a synchronized update
every time a capability's authority is revised elsewhere. (DERIVED, NORMATIVE — authority
computed from current state, not memorized.)

11.6 The relationship to spec_gate, made explicit. spec_gate's own object — determining whether
an L/XL task requires an established spec before coding — is a FINAL-authority gating decision in
this dataset's own taxonomy (Part III §3.4), and IAS-A1's dispatch of spec_gate itself is
therefore, by §11.2's split, an ADVISORY proposal requiring spec_gate's own acknowledgment before
a mission proceeds past it — IAS-A1 does not decide FOR spec_gate whether the spec-gate check
should be honored; it only decides WHETHER AND WHEN to route the mission to spec_gate at all, and
once routed, spec_gate's own final authority over its verdict is fully preserved. This concrete
instance is named explicitly because spec_gate is one of the six systems the Strengthening Matrix
names IAS-A1 as strengthening, and the strengthening mechanism specifically is that missions now
reliably REACH spec_gate at the correct point in their sequence (Part V §5.3's pre-scan, Part VI's
dependency-first composition) rather than spec_gate depending on each calling context to remember
to invoke it — a reliability gain that does not require, and must not require, IAS-A1 second-
guessing spec_gate's own verdict once reached. (DERIVED, concrete instance of §11.2's general
rule, tying directly to the Strengthening Matrix's named evidence requirement.)

11.7 What happens when authority itself is UNRESOLVED for a candidate, rather than cleanly ADVISORY
or FINAL. The split of §11.2 presupposes every candidate's property-model authority is already
classified. A newly-extracted capability signature (Part III §3.3, PROVISIONAL status per Part XIX
§19.4) may not yet carry a confirmed authority marker at all, because authority classification
itself may depend on observing the capability's actual behavior under real invocation, not merely
its declared interface. IAS-A1 treats a candidate with UNRESOLVED authority as though it carried
FINAL authority for the specific, narrow purpose of §11.2's dispatch-authority split — the more
conservative assumption — until the authority marker resolves one way or the other, because
mistakenly treating an actually-FINAL capability as ADVISORY (and therefore self-dispatching without
acknowledgment) is a strictly worse error than the reverse: an unnecessary acknowledgment request
for what turns out to be an advisory-authority capability costs a small delay, while an unchecked
dispatch to what turns out to be a FINAL-authority gate risks exactly the unrecoverable action §11.3
argues split authority exists to prevent. (DEFINED, NORMATIVE — conservative default under
authority-classification uncertainty.)

11.8 The relationship between IAS-A1's dispatch authority and the ORIGINATING mission's own
authority. §11.2 through §11.5 specify IAS-A1's authority over WHICH candidate is dispatched; a
separate question is whether IAS-A1 has authority to act on the MISSION at all — a mission may
itself originate from a system or process that lacks standing to request certain classes of action
(an automated upstream process requesting a mission that would, if fully routed, culminate in a
FINAL-authority deploy action, without any human Owner in the loop at all). IAS-A1 does not itself
adjudicate originating-mission authority — that determination belongs to whatever governance layer
admits missions into the ensemble in the first place, outside this dataset's own object — but it
DOES propagate the originating mission's own authority level into the provenance record (Part XIII
§13.2) as a distinct field from any node's own capability authority, so that a downstream reviewer
can distinguish "IAS-A1 correctly required acknowledgment at a FINAL-authority node" from "the
mission itself should never have reached IAS-A1 with standing to request that node at all" — two
different failures, requiring two different fixes, that a provenance record silent on originating
authority would conflate. (DERIVED, NORMATIVE, a boundary clarification distinguishing this
dataset's own authority object from a mission-admission authority it does not hold.)

FINAL LAW — PART XI. IAS-A1's authority is never a single blanket grant; it is final exactly
where a routing error's downstream risk is bounded and advisory exactly where the destination
capability's own authority is final — because a router confident enough to dispatch unchecked
into the estate's highest-consequence gates has mistaken its own routing logic for the judgment
those gates exist to protect.

PART XII — DEAD-CAPABILITY ROUTING: THE FAILURE MODE OF A STALE INDEX
===============================================================================================

12.1 The failure named and deferred since Part III. Part III §3.6 previewed this failure and
deferred its full treatment here: DEAD-CAPABILITY ROUTING is DEFINED as a route successfully
composed and dispatched against a capability-index entry that no longer accurately describes the
system it claims to describe — the system was refactored, retired, or degraded since the entry's
LAST-VALIDATED timestamp (Part III §3.2), and the index, uncorrected, continues to present it as a
live, matchable candidate. This is distinct from the DECOMPOSITION GAP of Part X (a capability
that was never present); dead-capability routing is a capability the index CLAIMS is present, and
the failure surfaces only at dispatch, often expensively, after Parts V through IX have already
spent their reasoning on a candidate that will not actually deliver. (DEFINED, distinguishing from
Part X §10.6's explicit contrast.)

12.2 The three shapes dead-capability routing takes in practice. SILENT REFACTOR DRIFT: the
providing system's interface changed (new required inputs, a changed output shape) without the
index's capability signature being re-extracted (Part III §3.3's incremental extraction failing to
trigger) — the route composes cleanly, dispatches, and fails only at the receiving system's own
input validation, if that validation exists at all; if it does not, the failure may not even be
loud, producing a WRONG rather than a REJECTED result, the more dangerous variant. QUIET
RETIREMENT: the providing system was formally retired (D2A's RETIRE disposition) but its index
entries were not correspondingly removed, per the maintenance discipline Part III §3.3 requires
but does not itself enforce — a governance gap between D2A's own retirement action and IAS-A1's
index maintenance, named explicitly here as a genuine cross-system coordination risk this dataset
cannot resolve unilaterally (Part XVI specifies the integration that must close it). DEGRADED-BUT-
UNMARKED: the providing system still exists and still technically responds, but its actual success
rate has fallen (the confidence-decay mechanism of Part X §10.5 is designed to catch this over
repeated failures, but a first occurrence, or a system degraded in a way that fails silently rather
than loudly, may not yet have accumulated enough decay to be caught before a mission is exposed to
it). (DEFINED — three named shapes, each with a distinct root cause and a distinct primary
defense.)

12.3 Defense layering: no single mechanism catches all three shapes alone. SILENT REFACTOR DRIFT is
primarily defended by Part III §3.3's incremental re-extraction trigger, which this Part now makes
explicit is IMPERFECT — it depends on the refactoring system's own change signal reaching the
index maintenance process, a dependency this dataset cannot guarantee holds for every one of the
seventy-plus systems. QUIET RETIREMENT is primarily defended by Part XVI's integration with IAS-A2
(Control Bus), which is specified there to carry retirement EVENTS the index subscribes to, closing
the D2A-to-index gap named in §12.2. DEGRADED-BUT-UNMARKED is primarily defended by Part X §10.5's
confidence decay, which is explicitly probabilistic and slow rather than immediate — it catches a
PATTERN of failure, not a first occurrence. Because no single defense is complete against any
single shape, and because the three shapes have distinct primary defenses, IAS-A1 treats dead-
capability routing as a risk that is REDUCED, never eliminated, by this dataset's own mechanisms
alone — full closure requires the cross-system integration of Part XVI, an honest limit stated here
rather than asserted away. (DERIVED, NORMATIVE, honest limit stated per Article V's epistemic
discipline.)

12.4 The recovery path once dead-capability routing is detected. A dispatch that fails due to a
dead capability populates the workflow instance's FAILURE LEDGER (Part VII §7.3) and triggers two
distinct responses at two distinct time horizons. IMMEDIATE: the workflow instance's bounded
adaptive re-composition (Part VII §7.5) is invoked, treating the dead capability's absence as
equivalent to Part X's LOW-CONFIDENCE-OR-ABSENT case for that specific pair — the mission attempts
to proceed via the next-best arbitrated candidate (Part IX) if one exists, or surfaces a
DECOMPOSITION GAP if none does, rather than the whole mission failing outright on one dead node.
STANDING: the index entry itself is flagged for RE-VALIDATION (its LAST-VALIDATED timestamp is not
merely stale but now actively CONTESTED), removing it from future high-confidence matching until
either re-extraction confirms it is current again or D2A's retirement process confirms it should be
removed — this standing correction is what prevents the SAME dead entry from causing the SAME
failure on the next mission that happens to match it. (DERIVED, DEFINED — two-horizon recovery,
immediate mission continuity plus standing index correction.)

12.5 Why dead-capability routing is a strengthening-relevant failure, not merely an operational
nuisance. The Strengthening Matrix's evidence requirement for IAS-A1 is "a mission trace where A1
selects a system the parent routers could not see" — and a capability index riddled with
undetected dead entries would make that same trace ambiguous in the other direction: was a
selection correct because IAS-A1 genuinely reasoned across the ensemble, or merely lucky that the
particular entry it matched happened not to be dead this time? Systematic dead-capability defense
is therefore not a peripheral reliability concern; it is a precondition for the Strengthening
Matrix's own evidence being trustworthy, because an evidence trace built on a corrupted index
proves nothing about the router's actual reasoning quality. (DERIVED, NORMATIVE, tying back to the
dataset's own strengthening claim.)

12.6 Measuring dead-capability prevalence: the DARK-ENTRY SWEEP. Because none of the three defenses
in §12.3 is individually complete, IAS-A1 specifies a periodic, standing maintenance procedure
distinct from the reactive per-mission defenses already described: a DARK-ENTRY SWEEP that
periodically samples index entries which have NOT been matched-and-dispatched within a bounded
recency window (distinct from LAST-VALIDATED, which tracks extraction currency, not usage recency)
and actively re-validates a sample of them against their providing systems, rather than waiting
passively for a live mission to stumble into a dead entry and surface the problem expensively. An
entry that has gone unmatched for a long window is not necessarily dead — it may simply describe a
rare capability few missions need — but the intersection of LONG-UNMATCHED and LONG-SINCE-VALIDATED
is exactly the population where dead-capability risk concentrates, because neither ordinary mission
traffic nor ordinary re-extraction triggers has touched the entry recently enough to catch drift.
The sweep is deliberately SAMPLED rather than exhaustive — re-validating every index entry on every
cycle would itself impose a standing cost disproportionate to the risk for entries that ARE matched
frequently and therefore self-validate through ordinary use — and its sampling rate is itself an
EXPERIMENTAL parameter this dataset does not fix numerically, left to calibration once a live index
exists to calibrate against. (DEFINED, EXPERIMENTAL, a fourth, proactive defense layer beyond the
three §12.3 already names.)

12.7 Why the DARK-ENTRY SWEEP and the arbitration RECENCY criterion (Part IX §9.2) must not be
allowed to reinforce each other into a blind spot. A naive reading might assume the two mechanisms
are redundant — both, in some sense, care about recency. They are not redundant, and treating them
as such would create a specific compounding gap worth naming explicitly: arbitration's RECENCY
criterion (fourth priority, tie-break only) rewards a candidate for RECENT SUCCESSFUL invocation,
which means an entry that consistently LOSES arbitration to a stronger sibling (per §9.2's first
three criteria) may never accumulate the recent-invocation history that would make it a RECENCY
winner even if it were sound — and that same consistently-losing entry is exactly the kind of
rarely-matched entry the DARK-ENTRY SWEEP of §12.6 is designed to catch. The two mechanisms are
therefore complementary by design, not overlapping: arbitration's recency criterion rewards proven
freshness among CONTENDING candidates, while the sweep actively checks entries arbitration rarely
gets the chance to exercise at all, and an implementer who treated the sweep as unnecessary because
"recency is already handled in arbitration" would have reintroduced exactly the blind spot this
subsection exists to close — the entries arbitration structurally never reaches are the ones a
usage-driven recency signal can never validate. (DERIVED, NORMATIVE, an explicit non-redundancy
clarification between two superficially similar mechanisms.)

FINAL LAW — PART XII. A capability index that is never wrong about what it no longer knows is a
fiction; dead-capability routing is reduced, in three distinct shapes, by three distinct and
individually imperfect defenses, and the honest claim this dataset makes is reduction, layered
across mission-time recomposition and standing index correction, never elimination.

PART XIII — THE PROVENANCE OF A ROUTE
===============================================================================================

13.1 Why provenance is a first-class Part, not a logging afterthought. Every mechanism specified
in Parts V through XII has, at some point, invoked "the route's provenance record" as the place a
decision's reasoning is captured — the uncertainty flag (Part X §10.2), the arbitration criterion
that decided (Part IX §9.3), the acknowledgment gap that triggered escalation (Part XI §11.4).
This Part specifies what that record actually IS, because a router whose every mechanism claims
to be auditable but which never defines the artifact that carries the audit trail has made a
promise it cannot keep. (DERIVED, NORMATIVE.)

13.2 The ROUTE PROVENANCE RECORD, structured. IAS-A1 DEFINES the provenance record as a
structured, append-only artifact attached to every route (Part II §2.6), carrying, per node in
the compose plan: the DECOMPOSITION trace (which intent/object pair, and what implicit constraints
§5.2 surfaced beyond the Owner's literal sentence); the CANDIDATE SET considered (not merely the
one selected — Part IX's arbitration is only auditable if the rejected candidates are visible
too); the ARBITRATION CRITERION that decided, by name, per §9.3's discipline against blended
scores; the CONFIDENCE BAND (Part X §10.2) at match time; the AUTHORITY SPLIT (Part XI §11.2)
governing whether the dispatch was final or advisory; and, once execution occurs, the ACTUAL
OUTCOME (success, failure populating the FAILURE LEDGER, or a contested-dispatch escalation).
This record is exactly the compose plan of Part VI §6.3 annotated with its own decision history —
the same object, the same identity Part VI §6.3 already established between plan and route,
carrying now the WHY alongside the WHAT. (DEFINED, DERIVED from the identity established in
Part VI.)

13.3 Provenance as the substrate IAS-D2, IAS-E1, and IAS-C2 each read differently. A single
provenance record is not written for one downstream consumer; per the Compounding Graph's own
edge table, IAS-A1's mission execution flows onto IAS-A2 (Control Bus), which distributes it to
IAS-D2 (failure fingerprints), IAS-E1 (usage/dead-capability/dark-matter observation), and IAS-C2
(demand signal). Each of these three reads the SAME provenance record for a DIFFERENT field:
IAS-D2 reads the ACTUAL OUTCOME field, specifically failures, to fingerprint recurring failure
shapes; IAS-E1 reads the CANDIDATE SET and CONFIDENCE BAND fields across many routes to compute
which capability-index entries are frequently matched-but-never-selected (a dark-matter signal,
Part XIV) versus never matched at all (a dead-capability or genuinely unneeded entry); IAS-C2
reads DECOMPOSITION GAP reports (Part X §10.3) specifically as raw forecasting input. One record,
one write, three independently legible reads — the specific mechanism, DERIVED here, by which a
single well-structured artifact avoids the alternative of three separate logging systems each
capturing a partial, differently-shaped view of the same routing event. (DERIVED, elaborating the
Compounding Graph's edge table concretely for this dataset's own output.)

13.4 Provenance is append-only and never retroactively edited. Consistent with IAS-F1's own
staleness discipline (Part IV §4.3) applied here to the route's own history rather than to a
system's property record, a provenance record, once written, is never edited to reflect a later
understanding — if a route later turns out to have dispatched a since-discovered dead capability
(Part XII), the ORIGINAL record still shows the confidence and reasoning available AT THE TIME,
and the correction (the index entry's re-validation or removal) is a NEW, separate record,
timestamped after the fact. This append-only discipline is what makes IAS-D2's cross-project
failure fingerprinting possible at all: a fingerprint is only meaningful if it reflects what was
actually believed and done at the time, not a retroactively cleaned-up version of events. (DERIVED,
NORMATIVE — append-only is a precondition for the Immune System's own function, not a stylistic
preference.)

13.5 The limits of provenance: what it does not and cannot capture. A provenance record captures
IAS-A1's OWN reasoning — decomposition, matching, arbitration, dispatch decision, and the outcome
IAS-A1 itself observes. It does NOT capture the internal reasoning of the invoked system once
control has passed to it (Part IV's boundary discipline: IAS-A1 does not reach into CO-03's model
choice, and correspondingly its provenance record does not claim to explain CO-03's own internal
decision, only that CO-03 was dispatched, with what context, and what it returned). A provenance
record is therefore a PARTIAL audit trail of the whole mission — complete for the routing layer,
silent by design past each system's own boundary — and any downstream consumer treating it as a
complete mission audit trail (rather than a routing-layer audit trail specifically) is making the
same category error Part IV §4.5 already named as boundary erosion, applied here to interpretation
rather than to decision-making. (NORMATIVE, honest limit stated per Article V.)

13.6 Provenance record size discipline: the same tension §7.7 named for workflow synthesis applies
here at the standing-record layer. A full structured record per node (§13.2), retained indefinitely
(§13.4's append-only discipline), for a mission with a long MULTI-HANDOFF compose plan (Part XV
§15.2's archetype) accumulates a genuinely large artifact, and "retained indefinitely" cannot mean
"retained at full node-level granularity forever" without the provenance store itself becoming an
unbounded standing cost the estate must account for. IAS-A1 resolves this the same way IAS-F1's own
staleness discipline resolves an analogous tension for property records: the record is retained at
FULL granularity for a bounded RECENT window (long enough to serve IAS-D2's failure-fingerprinting,
which needs fresh detail most), and beyond that window is COMPRESSED into a summary record — mission
shape, terminal outcome, aggregate confidence, and any flags (uncertainty, split-brain, contested-
dispatch) — that preserves what Part XIV's metrics need to compute historical trends without
preserving every individual node's full structured entry indefinitely. Compression is itself an
event, timestamped and never silent, consistent with §13.4's prohibition on retroactive editing: the
compressed record does not overwrite the full one's meaning, it summarizes it, and the summarization
event is itself provenance-recorded. (DEFINED, NORMATIVE, a size-discipline extension to the base
append-only mechanism.)

13.7 Cross-mission provenance linkage: when one mission's route depends on a PRIOR mission's outcome.
Not every mission is independent of every other; a mission that decomposes to (re-attempt, prior-
failed-hardening-pass) explicitly references an earlier mission's own provenance record as part of
its own input context — the accumulated context §7.3 defines is normally scoped to ONE mission's own
nodes, but a re-attempt mission legitimately needs visibility into what the PRIOR mission's
provenance record already established, rather than re-deriving that history from scratch. IAS-A1
supports this via an explicit PRIOR-MISSION REFERENCE field in the provenance record, distinct from
the accumulated context proper, which links forward (never backward — the prior mission's own record
is never retroactively modified to know about the later one, preserving §13.4's append-only
discipline in both directions) to any mission whose provenance a later mission's decomposition
explicitly cites. This is what allows IAS-D2's failure-fingerprinting (Part XIII §13.3) to recognize
a RECURRING failure shape across nominally separate missions as one pattern rather than several
unconnected incidents, closing a linkage gap the base per-mission record, considered alone, would
leave implicit. (DEFINED, DERIVED, extending §13.2's base record schema for the cross-mission case.)

FINAL LAW — PART XIII. A route's provenance record is not a log kept for its own sake; it is the
single written artifact that lets IAS-D2, IAS-E1, and IAS-C2 each read a different truth from the
same event without IAS-A1 maintaining three separate logging systems, and it is honest only
because it is append-only and stops, by design, exactly where another system's own boundary
begins.

===============================================================================================
PART XIV — METRICS: WRONG-SYSTEM RATE, FIRST-HIT RATE, AND THEIR ANTI-GOODHART COUNTERMETRICS
===============================================================================================

14.1 Why metrics require a countermetric in the same breath they are proposed. A metric proposed
without its own countermetric is an invitation to game the metric rather than improve the thing it
measures — Part X §10.3 already named the specific perverse incentive a naive success-rate metric
would create (lower the confidence threshold until everything "routes"). This Part generalizes
that discipline: every metric IAS-A1 proposes for its own evaluation is paired, in the same
subsection, with the countermetric that catches its most likely gaming vector. (NORMATIVE, per the
task's own anti-Goodhart requirement.)

14.2 FIRST-HIT RATE, and its countermetric. FIRST-HIT RATE is DEFINED as the proportion of
missions whose eventual successful route was IAS-A1's FIRST-proposed candidate set (Part V), with
no arbitration reversal (Part IX), no contested-dispatch escalation (Part XI §11.4), and no
mid-mission re-composition (Part VII §7.4-§7.5). A high first-hit rate is evidence the
decomposition and matching layers (Parts V, III) are working well together — the router's first
read of a mission is usually its correct read. The countermetric, GAMED-SIMPLICITY RATE, DEFINED
as the proportion of missions IAS-A1 routes at HIGH first-hit rate but which are, independently,
below-median in mission COMPLEXITY (measured by decomposed pair count, Part V §5.4), guards
against the degenerate case where first-hit rate is improved by the router systematically
under-decomposing complex missions into artificially simple, easily-first-hit shapes — the
UNDER-DECOMPOSED MISSION failure Part V §5.7 already named, here made visible as a metric
artifact rather than only a per-mission failure. A first-hit rate that rises while mission
complexity distribution simultaneously flattens toward simple missions is not genuine improvement;
it is the router quietly avoiding hard missions. (DEFINED, paired countermetric per §14.1's
discipline.)

14.3 WRONG-SYSTEM RATE, and its countermetric. WRONG-SYSTEM RATE is DEFINED as the proportion of
dispatched routes whose ACTUAL OUTCOME (Part XIII §13.2) records a failure attributable to the
ROUTING decision itself (a genuinely inapplicable capability was matched and dispatched) rather
than to the invoked system's own internal execution failure (a boundary already drawn in Part IV
and preserved here: a system that was correctly and applicably routed to, but which then failed
for its own internal reasons, is not a wrong-system-rate failure, it is that system's own
reliability concern, IAS-E2's object). A low wrong-system rate is the most direct evidence for
the Strengthening Matrix's core claim. The countermetric, OVER-CAUTIOUS GAP RATE, DEFINED as the
proportion of missions reported as DECOMPOSITION GAPS (Part X §10.3) that a subsequent, independent
review determines COULD have been served by an existing capability at genuinely usable confidence,
guards against the degenerate case where wrong-system rate is driven toward zero by the router
simply refusing to route anything it is not overwhelmingly certain about — trivially minimizing
wrong routes by minimizing routes attempted at all, which would also, not incidentally, tank the
mission-throughput value the whole router exists to provide. A wrong-system rate that falls while
over-cautious gap rate simultaneously rises is not genuine improvement; it is the router trading
false positives for false negatives without net gain. (DEFINED, paired countermetric.)

14.4 A third metric pair, added for completeness beyond the two the task specifies: ARBITRATION
STABILITY and its countermetric. ARBITRATION STABILITY is DEFINED as the proportion of repeated,
structurally-identical (intent, object) pairs across different missions that arbitration (Part IX)
resolves to the SAME candidate — instability here (the same pair type resolving to different
winners across missions with no underlying index change) is a signal that the arbitration
criteria (§9.2) are being fed noisy or under-differentiated inputs, most likely a MATCH CONFIDENCE
computation (Part V §5.5) that is not discriminating well between genuinely different candidates.
The countermetric, FALSE STABILITY RATE, DEFINED as the proportion of stably-arbitrated pairs
where the consistently-losing candidate's own property-model record (evidence level, cost) has
MEANINGFULLY IMPROVED since the arbitration history began without ever winning even once, guards
against stability that is actually the router failing to notice a legitimately improving
alternative — stability that has calcified into never re-considering a candidate that has earned
reconsideration. (DEFINED, third metric pair, HYPOTHESIZED as valuable pending its own live
calibration, marked explicitly since it extends beyond the two metrics the founding brief names.)

14.5 What these metrics are measured AGAINST: the synthetic mission benchmark, forward-referenced.
None of the three metric pairs in this Part are meaningful without a stable population of missions
to measure across, and a live estate's real mission stream is both too sparse, early in this
family's life, and too uncontrolled (mission complexity varies) to serve as a clean baseline.
Part XV specifies the synthetic mission benchmark this dataset uses instead, and every metric
named here is understood, per Article V's epistemic discipline, as EXPERIMENTAL — proposed to be
tested, carrying its own test in Part XV — rather than as an already-validated measurement
apparatus. (Forward-referenced, EXPERIMENTAL marking per Article V.)

14.6 A fourth metric pair, closing the handoff-quality gap the first three pairs do not cover:
HANDOFF INTEGRITY RATE and its countermetric. Nothing in §14.2 through §14.4 directly measures the
handoff protocol's own performance (Part VIII) — a mission can score well on first-hit rate and
wrong-system rate while still suffering repeated TRUNCATION, TRANSLATION, AUTHORITY, or STALENESS
LOSS (Part VIII §8.2, §8.6) at its internal handoffs, because those losses degrade the QUALITY of a
correctly-routed mission's execution without necessarily flipping its outcome to outright failure.
HANDOFF INTEGRITY RATE is DEFINED as the proportion of node-to-node handoffs, across all dispatched
routes, whose structured record (Part VIII §8.3) required no downstream re-confirmation, no
completeness-marker downgrade (§8.7), and no staleness-triggered re-confirmation (§8.6) — a clean
handoff, end to end. The countermetric, SUPPRESSED-RECONFIRMATION RATE, DEFINED as the proportion of
handoffs where a receiving system's own downstream behavior suggests it silently proceeded on
context it should, by the completeness or staleness rules, have flagged or re-confirmed but did not
report doing so, guards against the degenerate case where handoff integrity rises because receiving
systems have simply stopped enforcing the very checks §8.6 and §8.7 require, rather than because
handoffs have genuinely improved — a receiving system that silently drops its own enforcement
obligation would make IAS-A1's own handoff protocol look healthier than it is. (DEFINED, paired
countermetric, extending Part XIV's metric family to cover Part VIII's own mechanism.)

14.7 Metric interaction: reading the four pairs as a system, not four independent dashboards. A
router could, in principle, score well on each of the four pairs individually while still exhibiting
a compounding problem invisible to any single pair read in isolation — Part XIX §19.5's own failure
interaction matrix already names one such case (a dead capability winning a near-tied arbitration).
This subsection makes explicit the reading discipline Part XIX's interaction matrix implies but does
not itself frame as a metrics concern: the four pairs of this Part should be read TOGETHER, per
mission archetype (Part XV §15.2), rather than pooled across archetypes into one estate-wide number,
because a pooled number can mask a pair moving in the wrong direction for one archetype while
improving in aggregate due to volume from a different, easier archetype — the same GAMED-SIMPLICITY
risk §14.2 already names, generalized here to apply across the metric family as a whole rather than
to first-hit rate alone. A metrics dashboard that reports one blended number per pair, without the
archetype breakdown, reintroduces at the REPORTING layer the exact loss of legibility §9.3 already
argued against at the ARBITRATION layer — a single number that cannot say which archetype, or which
mechanism, actually moved it. (DERIVED, NORMATIVE, a reporting-discipline extension tying this
Part's metric family back to the legibility principle established earlier in Part IX.)

FINAL LAW — PART XIV. A metric proposed alone is a target waiting to be gamed; first-hit rate
needs gamed-simplicity rate, wrong-system rate needs over-cautious gap rate, and arbitration
stability needs false stability rate, because a router optimized against one number in isolation
will find the cheapest path to that number, and the cheapest path is rarely the honest one.

PART XV — EVALUATION ON SYNTHETIC MISSIONS
===============================================================================================

15.1 Why synthetic, not live, missions ground this dataset's evaluation. Article VIII's Reality
Contract forbids production code this mission and confines IAS-A1, like every CPP-IAS dataset, to
specification; a specification cannot be validated against a live execution history that does not
yet exist for a mechanism not yet implemented. IAS-A1 therefore specifies a SYNTHETIC MISSION
BENCHMARK — a designed population of illustrative missions, explicitly not real Owner requests,
constructed to exercise every mechanism this dataset defines, so that Parts XIV's metrics have a
defined population to be computed against once IAS-A1 moves from specification to implementation
(Article VIII's later mission, outside this dataset's own scope). (NORMATIVE, DERIVED from
Article VIII.)

15.2 The benchmark's structure: mission archetypes, not a single difficulty axis. Rather than a
list of missions varying only in size, the benchmark is structured around SIX ARCHETYPES, each
targeting a specific mechanism this dataset specifies, because a benchmark that only varies
difficulty would systematically under-exercise the arbitration, uncertainty, and handoff
machinery that a purely size-scaled mission set would rarely trigger. CLEAN-MATCH missions (like
§1.2's hardening-then-ship example) exercise decomposition and composition with no arbitration or
uncertainty needed, establishing a baseline first-hit rate. TIED-CANDIDATE missions are
constructed to deliberately produce comparable-confidence candidates for at least one pair,
exercising Part IX's arbitration machinery through all four criteria. GAP missions are constructed
with at least one pair that NO index entry, however loosely read, actually covers, exercising Part
X's decomposition-gap reporting and its distinction from a merely low-confidence match. STALE-
INDEX missions are constructed against a deliberately corrupted synthetic index (an entry marked
current but describing a capability that has, in the synthetic scenario, since changed),
exercising Part XII's dead-capability detection and recovery. MULTI-HANDOFF missions chain four or
more nodes with at least one PARALLEL and one MERGE operator, exercising Part VIII's handoff
protocol under its harder fan-out and reconciliation cases. CONTESTED-AUTHORITY missions are
constructed so that a FINAL-authority node's simulated acknowledgment is deliberately withheld,
exercising Part XI's escalation path. (DEFINED — six archetypes, each targeting a named Part's
mechanism.)

15.3 Worked instance: a TIED-CANDIDATE archetype mission, traced end to end. Consider a synthetic
mission, "review this dataset for internal consistency," decomposed (§5.2) into a single (intent,
object) pair (review, dataset) with no gating pre-scan trigger. The synthetic index is populated,
for this benchmark instance, with two entries of deliberately comparable confidence: Entry Epsilon
(a consistency-checking capability, OBSERVED evidence, ADVISORY authority, MEDIUM cost-to-route)
and Entry Zeta (a broader quality-audit capability that includes consistency checking as one of
several checks, OBSERVED evidence, ADVISORY authority, HIGH cost-to-route). Arbitration (Part IX
§9.2) proceeds: authority precedence ties (both ADVISORY); evidence level ties (both OBSERVED);
cost-to-route resolves in Epsilon's favor (MEDIUM beats HIGH) — arbitration concludes at criterion
three, and the benchmark records this as an EXPECTED outcome the metric harness checks the
implementation against once built. A benchmark VARIANT of this same instance, with Epsilon's cost
artificially raised to equal Zeta's, forces the tie into criterion four (recency), and a further
variant with LAST-VALIDATED timestamps also tied forces the genuinely-tied disclosure path of Part
IX §9.4 — three variants of one mission shape, each targeting a different arbitration depth, which
is the general pattern the benchmark uses to turn one archetype into several targeted test
instances rather than one instance per archetype. (Illustrative, defining the benchmark's own
variant-generation pattern.)

15.4 Retrodiction as the evaluation method for the synthetic benchmark, before any live
implementation exists. Consistent with IAS-F1 Part XVI §16.3's retrodiction protocol (referenced
by name, not re-derived here), the synthetic benchmark's value at the specification stage is that
its EXPECTED outcomes (§15.3's "arbitration concludes at criterion three") are derivable directly
from this dataset's own stated mechanisms without needing an implementation to run — the benchmark
IS the retrodiction target: an implementation is correct, at minimum, if it reproduces every
archetype's expected outcome exactly, and any divergence is either an implementation bug or a
specification ambiguity this dataset failed to resolve tightly enough, both findings being useful
regardless of which. (DERIVED, NORMATIVE — the benchmark doubles as a specification-completeness
check even before implementation.)

15.5 What the benchmark cannot validate: the honest limit. A synthetic benchmark validates that
the MECHANISM behaves as specified against DESIGNED inputs; it cannot validate that real Owner
missions actually decompose (Part V) the way the benchmark assumes, nor that the capability
index's real extraction (Part III §3.3) stays as clean as the synthetic index's deliberately
controlled entries. The benchmark is therefore a NECESSARY but not SUFFICIENT evaluation layer —
sufficient evaluation requires the live mission trace evidence the Strengthening Matrix itself
demands, which by construction cannot exist until this dataset moves past specification. This
limit is stated explicitly rather than allowed to be implied, per Article V's discipline against
overclaiming a validation method's reach. (NORMATIVE, honest limit.)

15.6 Benchmark maintenance: why six archetypes is a floor, not a ceiling, and how new archetypes are
admitted. §15.2's six archetypes were derived by mapping each to a specific mechanism this dataset
specifies — but Parts XVIII and XIX, written after the original archetype set was conceived,
introduce failure shapes (route flapping, circular routing, split-brain routing, index corruption
cascade) that none of the six original archetypes directly exercises. Rather than treating the
benchmark as frozen at six, this Part specifies the ADMISSION RULE for a new archetype: any
mechanism this dataset defines with its own NAMED failure mode or NORMATIVE defense (dead-capability
routing, route flapping, circular routing, and so on) is entitled to its own archetype, constructed
analogously to §15.3's worked instance — a minimal synthetic mission plus a deliberately-shaped
synthetic index engineered to trigger exactly that mechanism and no other, isolating the mechanism
under test the same way §15.3's three variants isolated arbitration's four criteria one depth at a
time. Applying this rule retroactively, this dataset's own later Parts already imply at least four
further archetypes beyond the original six — STALE-INDEX already covers dead-capability routing
(Part XII) adequately, but FLAPPING-PAIR (Part XVIII §18.3), CIRCULAR-DEPENDENCY (Part XIX §19.2),
SPLIT-MISSION (Part XIX §19.3), and CASCADE-ORIGIN (Part XIX §19.4) each warrant their own dedicated
instance, named here as an explicit BENCHMARK EXPANSION this dataset commits to rather than leaving
implicit. (DERIVED, NORMATIVE, an explicit expansion the benchmark's own admission rule licenses.)

15.7 The benchmark's relationship to the D2A evidence standard: reuse, not parallel invention. A
reader might ask whether constructing six-plus synthetic mission archetypes duplicates evaluation
machinery the estate may already possess elsewhere (D2A's own best-adjacent-capability search, or
SQI's baseline-guardian pattern referenced in the estate's own knowledge base). It does not: D2A and
SQI evaluate whether a PROPOSED system duplicates an EXISTING one, a Non-Duplication Ledger concern
this dataset's own header block already resolves for IAS-A1 itself; the synthetic mission benchmark
evaluates whether A DATASET'S OWN SPECIFIED MECHANISM behaves as specified, a correctness concern
one level more specific than duplication-checking and not a service either D2A or SQI is positioned
to provide, because neither holds this dataset's own arbitration criteria, confidence bands, or
composition operators as an object to check consistency against. The two evaluation layers are
complementary, not overlapping, and this Part borrows D2A's and SQI's general METHODOLOGICAL
posture — evidence over assertion, falsifiability over declared confidence — without borrowing or
duplicating either's actual mechanism. (DERIVED, NORMATIVE, a boundary clarification against a
plausible but mistaken duplication reading.)

FINAL LAW — PART XV. A specification's evaluation, before any implementation exists, is a
retrodiction target, not a proof — six archetypes, each targeting one mechanism, let this dataset's
own claims be checked for internal consistency before a single line of the router is built, and
that check is necessary groundwork, never a substitute for the live mission evidence the
Strengthening Matrix ultimately requires.

PART XVI — INTEGRATION WITH THE IAS-A2 CONTROL BUS
===============================================================================================

16.1 The edge this Part specifies concretely. The Compounding Graph names the primary loop's
step from mission execution to IAS-A2: "(mission executes across systems via IAS-A2 Control Bus)
→ produces outcomes + events." This Part specifies exactly what IAS-A1 places onto that bus and
what IAS-A1, in turn, reads FROM it, closing the loop the header block's Compounding Graph
position already claims in summary. (DERIVED, elaborating a named Compounding Graph edge.)

16.2 What IAS-A1 PUBLISHES to the Control Bus. Every completed or failed route's provenance
record (Part XIII §13.2) is published, in full, as an ensemble-level EVENT the instant its
mission concludes — not merely a summary, because Part XIII §13.3 already established that three
distinct downstream consumers each need a different field from the same record, and a bus that
received only a lossy summary would force IAS-D2, IAS-E1, and IAS-C2 to each request the fuller
record separately, re-introducing the exact translation-loss risk Part VIII §8.3 defends against
for mission-internal handoffs, now at the ensemble-event layer instead. In addition to completed
routes, IAS-A1 publishes DECOMPOSITION GAP reports (Part X §10.3) as their own event type
immediately upon detection, because IAS-C2's forecasting value from a gap report decays with
delay — a gap reported promptly is a fresher demand signal than one batched and published later.
(DEFINED, NORMATIVE.)

16.3 What IAS-A1 SUBSCRIBES to from the Control Bus. IAS-A1 subscribes to RETIREMENT events
(closing the QUIET RETIREMENT gap Part XII §12.3 named as requiring exactly this integration —
a D2A RETIRE disposition, published to the bus, triggers immediate index entry removal rather
than waiting for the slower incremental re-extraction path), CAPABILITY-SIGNATURE-CHANGE events
(closing part of the SILENT REFACTOR DRIFT gap, though §12.3's honest limit stands: this closes
the case where the refactoring system DOES emit a change signal, not the case where it silently
changes without one), and AUTHORITY-CHANGE events (the mechanism behind Part XI §11.5's claim
that IAS-A1's own dispatch authority split updates live as a capability's authority marker
changes elsewhere in the estate — that claim is only true because this specific subscription
exists to carry the change). (DEFINED, tying three earlier-named gaps and claims to their actual
carrying mechanism.)

16.4 Why the bus relationship is genuinely bidirectional, not merely a reporting pipe. A weaker
design would have IAS-A1 only publish outward, treating the Control Bus as a one-way telemetry
sink. The subscription relationship of §16.3 is what makes IAS-A1 itself a beneficiary of the
ensemble's own event fabric, not merely a contributor to it — the index's own currency (Part III's
central precondition for every later Part's correctness) is, on inspection, MAINTAINED PARTLY BY
IAS-A2, not solely by IAS-A1's own extraction discipline. This is the concrete mechanism by which
IAS-A2 strengthens IAS-A1 in return for IAS-A1 strengthening IAS-A2's event volume and diversity —
a genuinely mutual STRENGTHENS relation (IAS-F1 Part VI §6.6) in both directions, worth naming
explicitly because the Strengthening Matrix's own row for IAS-A2 lists IAS-A1's routing traces as
part of what makes IAS-A2 valuable, and this Part is where that value flows back. (DERIVED,
NORMATIVE.)

16.5 Failure mode specific to the bus integration: EVENT STARVATION. If IAS-A2 itself degrades
or its event delivery lags, IAS-A1's subscriptions (§16.3) silently stop receiving retirement,
signature-change, and authority-change events, and the index quietly reverts to depending solely
on its own imperfect incremental extraction (Part III §3.3) without any signal that its secondary
defense layer has gone dark. IAS-A1 defends against this specific failure by treating an
unusually long gap since the LAST received bus event (of any subscribed type, across the whole
estate) as itself a signal worth surfacing — not a routing decision, but a maintenance alert that
the index's currency guarantee has silently weakened, distinct from any single entry's own
staleness (Part III §3.2) because it is a signal about the DEFENSE MECHANISM's health, not about
any one system's health. (DERIVED, NORMATIVE — naming a specific integration-failure mode rather
than assuming the bus integration is itself infallible.)

16.6 Event ordering and idempotency: what the bus integration must guarantee for IAS-A1 to trust it
at all. §16.3's subscriptions are only as reliable as the ORDER in which their events arrive and the
integration's own tolerance for a DUPLICATE delivery. A RETIREMENT event for a capability that
arrives AFTER a CAPABILITY-SIGNATURE-CHANGE event for the same entry, out of the order the two
actions actually occurred in the estate, could cause the index to apply the signature change to an
entry it should already have removed — a small but real correctness risk this Part names explicitly
rather than assuming the bus delivers events in causal order for free. IAS-A1 therefore requires
every subscribed event to carry, at minimum, a monotonically increasing SEQUENCE MARKER per
providing system (not a global clock, which the estate's own distributed nature makes unreliable to
depend on, but a per-system counter that at least orders that one system's own events relative to
each other), and the index applies events in sequence-marker order per system, buffering an
out-of-order arrival briefly rather than applying it immediately, with an unresolved gap in the
sequence itself becoming, after a bounded wait, the same kind of maintenance alert §16.5 already
specifies for outright starvation. IDEMPOTENCY is required separately: a duplicate delivery of the
same event (the same providing system, same sequence marker) must be a no-op on second application,
because the bus's own delivery guarantee is not assumed to be exactly-once, only at-least-once, and
an integration that assumed exactly-once would silently double-apply a change on the more common
real-world failure mode of a retried delivery. (DEFINED, NORMATIVE — ordering and idempotency
requirements this dataset places on the bus integration rather than assuming for free.)

16.7 What IAS-A1 does NOT subscribe to, and why the subscription list is deliberately narrow. The
Control Bus, per IAS-A2's own object, carries far more event traffic than the three types §16.3
names — pane-level session events, deploy-governance outcomes, hard-rules audit results, and many
other categories entirely outside IAS-A1's own decision class. A subscription to every event type
the bus carries would be simpler to specify but would flood the index maintenance process with
traffic it has no mechanism to act on, and — more importantly — would blur the same boundary
discipline Part IV establishes for the resource routers, this time applied to event consumption
rather than resource selection: IAS-A1 subscribes only to events that DIRECTLY inform its own three
named concerns (index currency via retirement and signature-change, authority-split currency via
authority-change), and deliberately does not subscribe to event categories that belong to a sibling
system's own reasoning even when those events might be tangentially interesting. A future maintainer
tempted to "just subscribe to everything, it's cheap" is the specific temptation this subsection
exists to name and reject — cheap at the bus layer, but corrosive to the legibility of what IAS-A1's
own maintenance process is actually reacting to and why. (DERIVED, NORMATIVE, a deliberate
narrowness matching the boundary discipline established elsewhere in this dataset.)

16.8 Worked instance: tracing one D2A retirement through the bus to a corrected index. A synthetic
providing system, "legacy-tier-classifier," is formally retired via D2A's RETIRE disposition after a
successor capability (Entry Alpha, the running example's own tier-classification entry) supersedes
it. D2A's own retirement action, per its own object, publishes a RETIREMENT event onto the Control
Bus carrying the retired system's identifier and a sequence marker. IAS-A1's subscription (§16.3)
receives the event, applies §16.6's ordering check (no prior signature-change event pending for the
same system, so no buffering needed), and removes the legacy-tier-classifier's index entries
immediately — not queued for the next incremental re-extraction cycle (Part III §3.3), which might
otherwise run hours or days later, but applied the moment the event arrives, closing the QUIET
RETIREMENT gap (Part XII §12.2) in the specific, concrete case this worked instance traces end to
end. Had the bus integration been absent, this same retirement would have left a dead entry
matchable at full confidence until the next re-extraction cycle happened to touch it, or until a
mission-time dispatch failure surfaced it expensively — the exact cost this Part's entire mechanism
exists to avoid. (Illustrative, tracing the mechanism concretely against the running synthetic
example.)

FINAL LAW — PART XVI. IAS-A1's relationship to the Control Bus is not publish-only telemetry; it
is a genuine two-way strengthening — full provenance records out, retirement and change events
in — and the specific gaps named in Parts III and XII (quiet retirement, live authority updates)
are closed only because this subscription exists, not by IAS-A1's own extraction discipline alone.

PART XVII — INTEGRATION WITH THE IAS-C1 CAPABILITY PORTFOLIO: ALLOCATION INFORMS ROUTING
===============================================================================================

17.1 The primary loop's second-most-cited edge, specified. The Compounding Graph's primary loop
names this edge directly: "IAS-C1 Capability Portfolio... uses usage+ROI data to allocate cognitive
capital → IAS-A1 Capability Router... allocation informs which systems a mission routes to." This
Part specifies the mechanism by which an allocation decision made in IAS-C1's own portfolio layer
actually changes a routing outcome in IAS-A1's own arbitration or matching layer, because "informs"
is, left unspecified, exactly the kind of vague cross-reference Article VI warns against as padding.
(DERIVED, elaborating the named edge concretely.)

17.2 The mechanism: allocation as a fifth, optional arbitration input, never a fifth arbitration
criterion. Part IX §9.2 fixed four arbitration criteria in strict priority order and §9.3 argued
explicitly against a blended score. Introducing IAS-C1's allocation signal as a literal fifth
criterion in that same strict ordering would either dominate the existing four (if placed early,
undermining authority and evidence precedence with a portfolio judgment that has nothing to do
with THIS mission's correctness) or be nearly inert (if placed last, rarely reached). IAS-A1
instead treats allocation as a TIE-BREAK PREFERENCE applied only within §9.4's genuinely-tied
case — where all four existing criteria have already exhausted without resolution, IAS-C1's
current allocation ranking of the tied candidates' PROVIDING SYSTEMS (not the capabilities
themselves, since portfolio allocation is a system-level, not capability-level, judgment per
IAS-C1's own object) breaks the tie in favor of the system IAS-C1 has currently allocated more
cognitive capital toward, on the DERIVED reasoning that a system the estate has deliberately
invested in is, all else being genuinely equal, the one whose continued exercise the estate has
already decided is worth the marginal cost. (DEFINED, DERIVED — allocation as tie-break, not as a
fifth blended criterion, preserving Part IX's own legibility discipline.)

17.3 Why this integration is bounded to the tied case, and not extended further. A stronger
integration — allocation influencing MATCH CONFIDENCE itself (Part V §5.5), or overriding
authority precedence — would make IAS-A1's routing decisions partly a function of IAS-C1's own
economic judgment, blurring the boundary Part IV worked hard to establish between "which system
acts" (this dataset's object) and "how much capital the estate commits to a system" (IAS-C1's
object). Confining the integration to the narrowest case where it adds value without contaminating
the other three criteria's legibility is a deliberate, NORMATIVE restraint, consistent with Part
II §2.9's discipline against every later Part introducing scope creep into a sibling's object.
(NORMATIVE, DERIVED from the boundary discipline of Part IV, applied to a sibling IAS dataset
rather than a resource router this time.)

17.4 The reverse direction, closing the loop concretely: what IAS-A1 gives IAS-C1. The primary
loop does not stop at IAS-A1; per the Compounding Graph, mission execution flows onward through
IAS-A2 to produce outcomes that eventually inform IAS-D1's ecology and, through the loop's own
closure, IAS-C1's allocation again. IAS-A1's specific contribution to that return path, beyond
the general provenance-to-bus flow already specified in Part XVI, is the CONSUMERS property (IAS-
F1 Part IV §4.2) itself: because IAS-A1's routing IS what creates consumes edges (Part II §2.3),
IAS-A1's own operation is the primary source of the observed-consumer-count data IAS-C1's
portfolio math depends on as one of its six inputs (Strengthening Matrix row for IAS-C1: "CO-01,
D2A-5, token_irr, recall_roi, FD, owner_queue" — consumer counts specifically flow from routing
activity, closing the loop the Federation Ontology's own DOUBLE-BOOKED CONSUMER edge case (IAS-F1
Part IV §4.6) already anticipated needing a falsifiable source: IAS-A1's own event log, per Part
XVI §16.2, is exactly that falsifiable source). (DERIVED, tying this dataset's own mechanism
directly to a specific edge case IAS-F1 left as a forward reference.)

17.5 Worked example: allocation breaking a genuine tie. Return to Part IX §9.5's Entry Beta versus
Entry Delta case, but construct a variant where cost-to-route is ALSO tied (both MEDIUM), forcing
the arbitration into recency (criterion four) — and suppose recency, too, ties (both entries
successfully invoked within the same recent window). This is now a genuine §9.4 tie. IAS-C1's
current portfolio allocation, in this synthetic instance, has ranked Beta's providing system
above Delta's providing system for the current allocation cycle (perhaps because Beta's providing
system also serves several other high-value capabilities the portfolio has independently chosen
to invest in, a judgment entirely outside IAS-A1's own visibility). The tie-break preference of
§17.2 resolves the mission's routing to Beta, and — critically — the provenance record (Part XIII
§13.2) explicitly names "IAS-C1 allocation tie-break" as the resolving factor, distinct from and
never conflated with the four PRIMARY arbitration criteria, preserving the auditability §9.3
already established as non-negotiable. (Illustrative, extending the running synthetic example one
final time.)

17.6 What happens when IAS-C1's allocation ranking itself changes mid-cycle. §17.2's tie-break
preference reads IAS-C1's CURRENT allocation ranking at the moment a genuine tie is being resolved,
and this subsection addresses a timing question the base mechanism leaves implicit: IAS-C1's own
allocation cycle (Strengthening Matrix row for IAS-C1) does not run continuously; it recomputes on
its own cadence, and a tie-break decided against one ranking may, minutes or hours later, be decided
differently had arbitration occurred just after the next allocation cycle completed instead of just
before it. IAS-A1 does not treat this as an error requiring correction — the tie-break preference is
NORMATIVE with respect to allocation AS OF THE MOMENT the tie is resolved, not a promise that the
same tie resolved twice at different times would resolve identically, because allocation ranking is,
by IAS-C1's own object, a live economic judgment expected to shift as the portfolio's own inputs
shift. What IAS-A1 DOES guarantee is that the provenance record (§17.5) captures WHICH allocation
snapshot informed the tie-break, so that a later reviewer comparing two instances of the same
structurally-identical tie resolved differently across two missions can trace the difference to a
genuine, timestamped allocation shift rather than to an inconsistency in IAS-A1's own mechanism.
(DERIVED, NORMATIVE — a timing clarification distinguishing legitimate drift from an inconsistency.)

17.7 The failure mode specific to this integration: ALLOCATION LAG POISONING. If IAS-C1's own
allocation computation is itself degraded or delayed — feeding on stale usage-and-ROI data per its
own six named inputs (§17.4) — a tie-break preference computed from that stale ranking could
systematically favor a providing system the portfolio would no longer rank highly if its own inputs
were current, quietly biasing IAS-A1's tie-break behavior toward yesterday's economics rather than
today's. IAS-A1 defends against this narrow, integration-specific risk by treating the AGE of the
allocation ranking it reads (distinct from the allocation VALUE itself, which is IAS-C1's own object
to compute correctly) as a gating condition on whether the tie-break preference is applied at all: a
ranking older than a bounded freshness window is treated as UNAVAILABLE for tie-break purposes for
that specific arbitration, falling through to §9.4's disclosed-tie path instead of applying a
preference that provenance could not honestly label current. This mirrors, at the IAS-C1 integration
seam specifically, the same staleness discipline Part III §3.2 applies to the capability index's own
entries — a value's own correctness is one axis, but a value's own CURRENCY is a separate axis this
dataset checks independently before trusting either. (DERIVED, NORMATIVE, naming and defending
against an integration-specific failure mode analogous to, but distinct from, index staleness.)

FINAL LAW — PART XVII. Allocation informs routing at exactly one narrow, clearly-labeled seam —
breaking a tie that survives all four primary arbitration criteria — never as a silent fifth
criterion blended into the rest, because the primary loop's own value depends on each system
contributing its own distinct judgment at its own distinct layer, not on IAS-C1's economics
quietly overriding IAS-A1's routing logic from the inside.

PART XVIII — ADVERSARIAL CASES: GAMING, SPOOFING, AND BOUNDARY REACH
===============================================================================================

18.1 Why an ensemble-level router is a specifically attractive target for adversarial pressure.
A system that decides which of seventy-plus governing systems act on a mission sits at a
structural chokepoint: a proposal that games IAS-A1 does not need to defeat any single downstream
gate on its own merits, only to arrange for IAS-A1's routing decision to skip or misdirect around
it. This Part catalogues the adversarial shapes DERIVED from that structural exposure, distinct
from the ordinary failure modes Part XIX catalogues, because an adversarial case assumes an actor
(human or system) actively exploiting the mechanism, not merely a mechanism degrading on its own.
(NORMATIVE, DERIVED.)

18.2 CAPABILITY SIGNATURE SPOOFING. A system whose actual behavior does not match its declared
capability signature (Part III §3.2) — deliberately or through careless self-description — can
cause the index to match it against missions it should not serve, effectively hijacking routing
traffic intended for a better-suited candidate. The defense is the same extraction-not-invention
discipline of Part III §3.3 made adversarially aware: capability signatures are not merely
extracted once and trusted; the confidence-decay mechanism (Part X §10.5) applies with EQUAL
force to a spoofed signature as to an honestly degraded one, because from IAS-A1's own vantage
point the two are observationally identical — repeated failure against the declared signature
decays effective confidence regardless of whether the mismatch was honest or engineered. This is
a DELIBERATE non-distinction: IAS-A1 does not attempt to adjudicate INTENT behind a signature
mismatch (that adjudication belongs to a governance or immune-system process, IAS-D2's object,
reading IAS-A1's failure-pattern events per Part XVI), only to STOP TRUSTING a signature that
repeatedly fails to deliver what it claims, regardless of why. (DERIVED, NORMATIVE — the defense
is structural indifference to intent, not intent-detection.)

18.3 ROUTE FLAPPING as an adversarial-adjacent instability. A pair of candidates whose match
confidence (Part V §5.5) sits close enough together that ordinary index churn (routine re-
extraction, Part III §3.3) flips which one arbitration selects mission to mission, even absent
any actual change in either candidate's real suitability, produces ROUTE FLAPPING: a mission type
that should route consistently instead alternates unpredictably. Whether engineered (an actor
deliberately triggering re-extraction cycles timed to exploit near-tied confidence) or organic
(genuine measurement noise near a threshold), the OBSERVABLE signature is identical, and Part
XIV's ARBITRATION STABILITY metric (§14.4) is the detection mechanism: flapping shows up directly
as instability in that metric, and the FALSE STABILITY RATE countermetric's inverse — GENUINE
instability without a correspondingly improving loser candidate — is the specific signature that
distinguishes flapping from healthy re-ranking. The remedy is a HYSTERESIS margin: once
arbitration resolves a tie, per §9.4, or a near-tie via any of the four criteria, the winning
candidate retains a small confidence advantage for subsequent identical-pair queries until a
MATERIAL change (not routine noise) in either candidate's property record is observed — a
NORMATIVE mechanism this Part introduces specifically to close the flapping vector rather than
leaving it as an unaddressed consequence of Part IX's otherwise-correct strict-priority
arbitration. (DERIVED, NORMATIVE, introducing hysteresis as this Part's specific contribution
beyond Part IX's base mechanism.)

18.4 THE IMPERIAL-BOUNDARY REACH, applied to routing specifically. IAS-F1 Part V §5.5 (Invariant
Four) and Part XIV name the imperial-boundary adversarial case generally: a system claiming
authority past the ensemble's own boundary. For IAS-A1 specifically, this shape manifests as a
mission decomposition (Part V §5.2) that names an OBJECT outside the ensemble boundary — the host
machine, the model provider's own infrastructure, a project the estate does not govern — as
though it were a routable capability requirement. IAS-A1's defense is that the capability index
(Part III) is, by construction, populated only from systems that pass IAS-F1's own systemhood
test (Part II §2.1), which is itself bounded to the estate; an object outside the boundary
therefore has, structurally, NO matching index entry, and decomposition of such a mission
correctly terminates in a DECOMPOSITION GAP (Part X) rather than a route — the boundary is
enforced not by a special adversarial check but by the simple fact that nothing outside the
estate is indexed. This is named explicitly as an adversarial case, rather than left as an
implicit consequence of Part III, because a future maintainer under pressure to "just make it
work" for an out-of-boundary request might be tempted to hand-populate a special-case index entry
for exactly such an object — the specific temptation this subsection exists to forbid. (DERIVED,
NORMATIVE — the defense is structural absence, and the explicit naming here is to prevent that
structural absence from being patched away.)

18.5 THE AUTHORITY-LADDER CLIMB: a mission crafted to accumulate advisory dispatches into a
de facto final action. Because Part XI §11.2 grants IAS-A1 final authority over advisory-outcome
dispatches, a sufficiently long SEQUENCE of individually-advisory dispatches (Part VI §6.2) could,
in principle, be composed such that their CUMULATIVE effect approximates a consequential, hard-
to-reverse action no single node's authority marker would have permitted alone — each step
individually low-risk, the chain as a whole not. The defense is that Part XIII's provenance
record (§13.2) captures the FULL compose plan before dispatch begins, not merely each node in
isolation, and Part XI's escalation path (§11.4) is triggered not only by an individual FINAL-
authority acknowledgment gap but by a compose-plan-level check: does the CUMULATIVE route, read
as a whole, cross into terminal-capability territory (Part III §3.4's fourth kind) despite no
single node individually claiming FINAL authority? A route whose TERMINAL node is reached without
any node along the way ever having FINAL authority is treated, per this Part's NORMATIVE
extension of Part XI, as itself requiring the CONTESTED DISPATCH escalation, because a terminal
action reached entirely through advisory steps is exactly the shape this adversarial case
exploits. (DERIVED, NORMATIVE, extending Part XI's escalation trigger to cover a shape Part XI's
own original specification did not explicitly name.)

18.6 DECOMPOSITION GAMING: a mission crafted to force an artificially favorable decomposition. Part
V §5.2's decomposition surfaces implicit constraints beyond an Owner's literal sentence — a mechanism
that is itself an attack surface if a mission is phrased specifically to avoid triggering a gating
pre-scan (Part V §5.3) that would otherwise route it through a costlier, more scrutinized path — a
mission that avoids the literal word "ship" or "deploy," for instance, while still ultimately
producing a shippable artifact through a chain of individually innocuous-sounding steps, attempting
to exploit the pre-scan's own keyword-adjacent trigger surface rather than its intent. The defense
is that the pre-scan's trigger set (Part V §5.3) is DERIVED from the mission's decomposed OBJECTS
and their eventual TERMINAL capability (Part III §3.4's fourth kind), not from surface keyword
matching against the Owner's literal phrasing alone — a decomposition that traces forward to a
terminal capability requiring FINAL authority triggers the same gating and escalation discipline
(Part XI) regardless of what words the originating sentence used to get there, closing the same
class of surface-level evasion Part XVIII §18.4's boundary-reach defense closes for the structurally
similar case of an out-of-boundary object. (DERIVED, NORMATIVE, naming a gaming vector this Part's
existing mechanisms already structurally close but had not previously named explicitly.)

18.7 Why this Part's adversarial catalogue is deliberately not exhaustive, and what governs additions
to it. Article V's epistemic discipline requires this dataset to avoid the overclaim that five named
adversarial shapes (§18.2 through §18.6) constitute a complete threat model — an ensemble-level
chokepoint invites pressure in proportion to its own leverage, and leverage grows as the ensemble
itself grows, meaning new adversarial shapes are expected to emerge as new systems, new capability
types, and new mission patterns enter the federation over time. This Part therefore closes not with
a claim of completeness but with an ADMISSION RULE structurally identical to Part XV §15.6's
benchmark-expansion rule: any newly observed exploitation of a routing mechanism, once confirmed as
a genuine pattern rather than a one-off anomaly, is named, catalogued in this Part under its own
subsection, and given a structural defense consistent with §18.5's closing principle — indifference
to intent, hysteresis, index absence, or cumulative-route checking, never a vigilance promise alone.
(NORMATIVE, honest limit stated per Article V, closing the catalogue with its own extension
mechanism rather than a false claim of completeness.)

FINAL LAW — PART XVIII. An ensemble-level router is a chokepoint, and a chokepoint invites
pressure at exactly its own seams — spoofed signatures, flapping ties, boundary-testing missions,
and authority accumulated one advisory step at a time — and every defense in this Part is
structural (indifference to intent, hysteresis, index absence, cumulative-route checking) rather
than a vigilance promise, because a chokepoint defended only by vigilance is a chokepoint waiting
for the day vigilance lapses.

PART XIX — THE FAILURE MODE CATALOGUE
===============================================================================================

19.1 Why a dedicated catalogue, beyond the failures already named in passing. Parts III through
XVIII have each named a failure mode local to their own mechanism — dead-capability routing (XII),
route flapping (XVIII), the under-decomposed mission (V), event starvation (XVI). This Part
collects the failures that are NOT local to any single mechanism but arise from the INTERACTION of
two or more Parts, because interaction failures are the ones most likely to be missed by a review
that checks each Part in isolation. (NORMATIVE, DERIVED — this Part exists specifically to catch
what per-Part review structurally cannot.)

19.2 CIRCULAR ROUTING: a compose plan whose dependency resolution (Part VI §6.4) produces a cycle
rather than a tree. If Capability A's applicability predicate names Capability B's output as a
precondition, and Capability B's applicability predicate names Capability A's output as a
precondition — a mutual dependency the index's own per-entry structure (Part III §3.2) does not,
on its own, prevent from being expressed — the compose engine's SEQUENCE-building logic (§6.4)
has no legal starting point for either. This is DEFINED as a COMPOSE-PLAN VALIDATION failure,
caught at the validation step Part VI §6.3 already specifies ("once validated against Part III
§3.4's taxonomy constraints"), extended here explicitly to include acyclicity as a validation
requirement this dataset's earlier Parts implied but did not state outright: a compose plan that
fails to validate as acyclic is treated exactly as Part X treats a LOW-CONFIDENCE-OR-ABSENT pair —
surfaced as a gap, specifically a CIRCULAR-DEPENDENCY GAP, rather than resolved by an arbitrary
break in the cycle, because an arbitrary break would silently choose which of the two mutually-
dependent capabilities runs on stale input from the other. (DERIVED, closing a gap in Part VI's
own stated validation step.)

19.3 SPLIT-BRAIN ROUTING: two workflow instances (Part VII §7.3) for what should be one mission,
arising from a mission being independently decomposed twice — once by IAS-A1 proper, and once by
an upstream system that itself performs a partial local decomposition before handing the mission
to IAS-A1 (a plausible interaction given that one_shot and ultra, per Part VII §7.6, retain full
internal control once invoked, and either could, in principle, re-decompose a sub-mission IAS-A1
already decomposed, rather than treating IAS-A1's accumulated context, per Part VIII, as
authoritative). The defense is that the MISSION HANDLE (Part VII §7.3) is required to be unique
per originating mission and is propagated, per the handoff protocol's structured record (Part
VIII §8.3), to every invoked system as part of its input — a system receiving a mission handle it
did not itself originate is expected to treat that handle's existing decomposition as authoritative
rather than independently re-decomposing, and a SECOND workflow instance observed against an
ALREADY-ACTIVE mission handle is itself the detection signal for this failure, surfaced to Part
XIII's provenance layer as a SPLIT-BRAIN FLAG rather than silently allowed to proceed as two
parallel, uncoordinated missions. (DERIVED, NORMATIVE, cross-referencing Part VII's mission-handle
mechanism as the specific defense.)

19.4 INDEX CORRUPTION CASCADE: a single bad re-extraction (Part III §3.3) that writes an
INCORRECT capability signature — not stale, but actively wrong from the moment it was written —
which then propagates through Part IX's arbitration (winning ties it should not win, because its
now-incorrect applicability predicate happens to lexically over-match a broad range of missions),
through Part VI's composition (producing structurally invalid plans downstream systems reject),
and eventually through Part XVI's event publication onto the Control Bus, where IAS-D2, IAS-E1,
and IAS-C2 each ingest the corrupted signal as though it were reliable OBSERVED data. This is
DEFINED as the dataset's own single most consequential failure mode, because it is the one
interaction failure capable of contaminating THREE sibling systems' own data through the very
integration Part XVI designed to strengthen them. The defense, DERIVED from the general principle
that a cascade is best stopped at its origin rather than chased through its propagation, is that
re-extraction (Part III §3.3) is never trusted as a single-write operation: a newly re-extracted
signature is held at PROVISIONAL status, matched only at reduced confidence (analogous to, but
distinct from, Part X's MODERATE band) until it has survived at least one successful real
invocation under the new signature, converting to full standing only then — a deliberate friction
this Part introduces specifically because the cost of a slower-to-trust index update is small
relative to the cost of a cascade reaching three sibling systems' own data before detection.
(DERIVED, NORMATIVE, the dataset's own most important single defense mechanism.)

19.5 Failure interaction matrix: which local failures compound which. Beyond the three named
interaction failures above, this dataset names explicitly, as a table in prose rather than a
diagram, the compounding pairs most worth a future implementer's attention: a DEAD CAPABILITY
(Part XII) that is ALSO the winner of a near-tied arbitration (Part IX) compounds into a
confidently-wrong route rather than a merely-uncertain one, because arbitration's own legibility
(§9.3) does not itself check for staleness — the two mechanisms must be composed, not merely
run in sequence, meaning arbitration candidates should be filtered by re-validation status BEFORE
the four criteria are applied, an ordering this Part makes explicit rather than assumed. An EVENT
STARVATION period (Part XVI §16.5) that coincides with a QUIET RETIREMENT (Part XII §12.2)
compounds into the single worst-case shape this dataset's defenses cover: a retired system,
un-flagged because the retirement event never arrived, matched at full confidence because no
starvation-period downgrade was applied to it specifically — the general starvation signal of
§16.5 being ESTATE-WIDE rather than PER-ENTRY is itself, on this reflection, an incomplete
defense, and this Part records that incompleteness honestly as an UNRESOLVED gap for Part XXII
to carry forward rather than silently patching it here beyond this dataset's own remaining scope.
(DERIVED, honest naming of an unresolved compounding gap, per Article V.)

19.6 A fourth interaction failure: ARBITRATION-ALLOCATION DESYNC. Part XVII §17.6 already names the
timing tension between an arbitration's own moment and IAS-C1's allocation cycle as a legitimate,
non-error drift. This subsection names the interaction failure that drift can compound into when it
coincides with §18.3's ROUTE FLAPPING: a pair whose match confidence sits close enough to flap on
routine index churn, ALSO tie-broken by allocation (Part XVII §17.2) whose own ranking is itself
shifting cycle to cycle, can produce a route that changes not because either underlying candidate's
real suitability changed, but because two independently-drifting signals (index churn and allocation
cycling) happen to cross each other's thresholds at different moments. Read separately, neither
Part XVII's timing clarification nor Part XVIII's hysteresis defense fully addresses this compound
case, because the hysteresis margin of §18.3 was specified against confidence churn alone, not
against a tie-break preference's own independent drift. The defense this Part specifies is to extend
§18.3's hysteresis margin to cover the ALLOCATION tie-break specifically: once an allocation
tie-break has resolved a given pair, that resolution, too, retains a small preference against
reversal until a MATERIAL allocation shift (not routine cycle-to-cycle noise) is observed, the same
NORMATIVE friction §18.3 already introduces for the primary criteria, extended here to the seam Part
XVII introduces. (DERIVED, NORMATIVE, a fourth named interaction failure closing a gap between Parts
XVII and XVIII.)

19.7 Why interaction failures, once catalogued, feed back into Part XIV's metrics rather than
remaining a static list. A catalogue entry that is never checked against live or benchmark evidence
degrades into documentation nobody re-reads. Each interaction failure named in this Part (§19.2
through §19.6) is therefore paired, per this dataset's own general anti-Goodhart discipline (Part
XIV §14.1), with a DETECTION SIGNAL already available in an existing metric or provenance field
rather than requiring a bespoke new measurement apparatus: CIRCULAR ROUTING is detected by a spike
in CIRCULAR-DEPENDENCY GAP reports (§19.2) against the DECOMPOSITION GAP baseline Part X §10.3
already tracks; SPLIT-BRAIN ROUTING is detected by SPLIT-BRAIN FLAG frequency (§19.3) against active
mission-handle count; INDEX CORRUPTION CASCADE is detected by an unusual concentration of PROVISIONAL
entries (§19.4) winning arbitration disproportionately relative to STANDING entries; and the
ARBITRATION-ALLOCATION DESYNC of §19.6 is detected by ARBITRATION STABILITY (Part XIV §14.4) computed
SEPARATELY for allocation-tie-broken routes versus primary-criteria-resolved routes, with a
meaningfully lower stability score in the former group being the specific signature this compound
failure leaves in existing metric data. (DERIVED, NORMATIVE, tying every catalogued interaction
failure to a concrete, already-specified detection path rather than leaving detection implicit.)

FINAL LAW — PART XIX. The failures worth cataloguing separately are not the ones any single Part
already names, but the ones that live in the seams between Parts — circular dependencies
validation missed, two decompositions racing under one mission, and a single bad write cascading
through three sibling systems' own data — and the defense in every case is to compose the
relevant mechanisms deliberately rather than trust that running them in sequence is the same
thing as running them together.

PART XX — LIFECYCLE: HOW A ROUTE IS BORN, HOW AN INDEX ENTRY AGES, HOW THE ROUTER EVOLVES
===============================================================================================

20.1 Three distinct lifecycles this Part must not conflate. IAS-A1 contains three genuinely
different things that age and change at different rates, and a lifecycle discussion that
conflates them produces exactly the kind of blurred reasoning Part II's resolution discipline
exists to prevent: the ROUTE lifecycle (minutes to hours, one mission's duration, per Part VII
§7.2's temporary-workflow discipline), the CAPABILITY INDEX ENTRY lifecycle (days to months, tied
to the providing system's own evolution), and the ROUTER'S OWN MECHANISM lifecycle (this dataset's
own specification evolving over the estate's longer institutional time). (DEFINED, NORMATIVE
separation.)

20.2 The route lifecycle: born, executed, provenance-sealed, discarded. A route is born at the
moment decomposition (Part V) and matching produce a candidate set; it moves through arbitration
(Part IX) and composition (Part VI) into a compose plan; it is dispatched, node by node, through
the handoff protocol (Part VIII), accumulating its provenance record (Part XIII) as it goes; it
concludes at its terminal node (or at a decomposition gap, or a contested-dispatch escalation);
its provenance record is SEALED (append-only, per §13.4) and published to the Control Bus (Part
XVI); and its RUNTIME workflow instance (Part VII §7.3) is then discarded, per Part VII §7.2's
explicit no-standing-infrastructure discipline, while its sealed provenance record persists
indefinitely as the estate's own historical trace. The route's "death" is therefore not loss —
the instance dissolves, but the record it earned survives it, the same distinction a biological
metaphor would draw between an organism and the fossil record it may leave, invoked here only as
an illustrative aid, not as an ontological claim this dataset makes about the ensemble. (DEFINED,
DERIVED from Parts VII and XIII together.)

20.3 The capability index entry lifecycle: extracted, provisional, standing, decaying, retired.
An entry begins PROVISIONAL (Part XIX §19.4's cascade defense) upon initial extraction (Part III
§3.3) or re-extraction after a signature change; it graduates to STANDING after surviving at
least one successful real invocation under its current signature; it may accrue CONFIDENCE DECAY
(Part X §10.5) through repeated invocation failures without ever leaving standing status outright,
because decay is continuous and reversible, not a discrete state transition; it may be flagged
CONTESTED (Part XII §12.4) upon a dead-capability detection, removing it from high-confidence
matching pending re-validation; and it is REMOVED upon a D2A RETIRE disposition received via the
Part XVI §16.3 subscription, or upon a maintenance sweep confirming a long-CONTESTED entry was
never re-validated. This five-state path (provisional → standing → [decaying] → contested →
removed, with decay as a continuous overlay rather than a discrete fifth state) is the concrete
mechanism by which Part III §3.6's staleness discipline and Part XII's dead-capability defense
are unified into one coherent entry lifecycle rather than two separately-described mechanisms
that happen to both touch the same record. (DERIVED, unifying two earlier Parts' mechanisms into
one lifecycle.)

20.4 The router's own mechanism lifecycle: how IAS-A1's specification itself may evolve. Per
Article IX (governed self-evolution), IAS-A1 does not evolve its own mechanism — its arbitration
criteria order (Part IX §9.2), its confidence bands (Part X §10.2), its authority split logic
(Part XI §11.2) — autonomously; any change to these mechanisms is a PROPOSAL requiring the
evidence threshold, validation, versioning, and rollback discipline Article IX mandates, with the
Owner retaining promotion authority. What IAS-A1's OWN operation legitimately produces, without
requiring this heavier governance, is EVIDENCE for such a proposal: Part XIV's metrics,
accumulated over enough missions, are the specific evidentiary substrate a future proposal to
revise, say, the arbitration criteria's priority order would need to cite. IAS-A1 therefore
PROPOSES via its own observed metrics; it does not self-modify. (NORMATIVE, DERIVED directly from
Article IX, applied concretely to this dataset's own mechanism.)

20.5 Worked instance: a metric-driven evolution proposal, traced from signal to governance. Should
Part XIV's WRONG-SYSTEM RATE metric, measured over a sufficient mission population (Part XV's
benchmark, later supplemented by live evidence per Part XV §15.5's own stated limit), reveal that
COST-TO-ROUTE (arbitration criterion three, Part IX §9.2) is winning ties in cases where the
LOSING candidate's higher cost would have been justified by a materially higher long-run success
rate the current criteria never surface — a genuine possible finding, HYPOTHESIZED here as an
illustration, not asserted as an observed result — the correct response is NOT for IAS-A1 to
quietly re-order its own criteria. It is to package the metric evidence, the specific mission
traces exhibiting the pattern (Part XIII's provenance records), and a PROPOSED criteria revision
into the governance artifact Article IX requires, routed to the Owner for evaluation, versioned
so the prior criteria order remains available for rollback if the revision underperforms once
live. This worked instance is the concrete answer to "how does a router that specifies its own
arbitration logic ever legitimately improve that logic" — through evidence and Owner-gated
promotion, never through silent self-tuning. (HYPOTHESIZED illustration, NORMATIVE process.)

20.6 A fourth clock, named for completeness: the BENCHMARK's own lifecycle. Parts XV §15.6 and
XVIII §18.7 both introduced admission rules for their respective catalogues (new archetypes, new
adversarial shapes) to grow over the estate's institutional time — a fourth clock, distinct from the
three §20.1 names, because the synthetic mission benchmark and the adversarial catalogue are neither
a single route's ephemeral lifetime, nor an index entry's system-tied evolution, nor the router's own
mechanism requiring Article IX's heavier governance. This fourth clock moves at the pace new
mechanisms are ADDED to this dataset's own specification — each new Part, or each new subsection
naming a new failure mode or defense, mechanically entitles a corresponding benchmark archetype or
adversarial catalogue entry per those two Parts' own admission rules, WITHOUT requiring the full
Article IX evidence-and-promotion process §20.4 reserves for changing an EXISTING mechanism's actual
behavior — adding a test for a mechanism that already exists is a lower-stakes action than revising
the mechanism itself, and treating the two identically would either over-gate benign test-coverage
growth or under-gate genuine behavior changes by lumping them into one governance lane. (DERIVED,
NORMATIVE, naming and distinguishing a fourth clock from the three the base mechanism specifies.)

20.7 Worked instance: an index entry's full lifecycle traced through all five states in sequence.
Consider a synthetic new capability, "cross-mission dependency linter," entering the estate for the
first time. It is EXTRACTED (Part III §3.3) upon its providing system's own admission to the
federation, immediately entering PROVISIONAL status (§20.3, Part XIX §19.4's cascade defense) rather
than being trusted at full confidence from its first appearance. Its first real invocation, dispatched
at PROVISIONAL's reduced confidence per a mission that happens to need exactly this capability,
SUCCEEDS, and the entry graduates to STANDING. Over the following weeks, three further invocations
each fail for reasons traced, via the FAILURE LEDGER (Part VII §7.3), to a genuine reliability issue
in the providing system rather than a routing error — CONFIDENCE DECAY (Part X §10.5) accrues with
each failure, continuously lowering the entry's effective match confidence while it remains, in
STATE terms, still STANDING rather than transitioning to CONTESTED, because decay alone does not
cross the CONTESTED threshold in this instance. A subsequent mission's dispatch fails in a way that
DOES meet Part XII §12.4's dead-capability detection criteria (not merely a low-confidence match,
but a route that composed cleanly and failed only at dispatch) — the entry is flagged CONTESTED,
removed from high-confidence matching pending re-validation. Re-extraction (Part III §3.3) confirms
the providing system's interface genuinely changed underneath the entry (SILENT REFACTOR DRIFT, Part
XII §12.2) — rather than re-validating the OLD signature, the entry is REMOVED and a NEW provisional
entry is extracted under the corrected signature, restarting the lifecycle at PROVISIONAL rather than
attempting to salvage the stale one. This full traversal — provisional, standing, decaying,
contested, removed-and-reborn — is included here because no single earlier subsection walks all five
states for one entry end to end, and a lifecycle description that only ever discusses states in the
abstract risks understating how much real institutional history one entry's own timeline can
accumulate before it is finally retired. (Illustrative, tracing §20.3's full state machine against
one synthetic entry.)

FINAL LAW — PART XX. A route is born and dissolves within a mission's lifetime, an index entry
ages across a system's own evolution, and the router's own mechanism changes only through
Owner-gated, evidence-backed proposal — three different clocks, and confusing any two of them
produces either a router that forgets too fast or one that never legitimately learns.

DECISION LOG
===============================================================================================

D1. DECISION: Arbitration (Part IX) uses a strict, ordered, four-criterion checklist rather than a
weighted composite score. ALTERNATIVES CONSIDERED: (a) a single weighted-average score combining
authority, evidence, cost, and recency into one comparable number; (b) a machine-learned ranking
function trained on historical arbitration outcomes. WHY THIS: strict ordering keeps every
arbitration decision legible to a one-sentence explanation ("Candidate A won on cost-to-route"),
which Part XIII's provenance record and Part XIV's metrics both depend on for auditability; a
blended score cannot be decomposed after the fact into "which criterion actually decided."
TRADE-OFFS: strict ordering can produce a criterion-one winner that a genuinely well-calibrated
weighted score might have ranked second when the higher criteria are near-ties (e.g., a
marginally-FINAL-authority candidate beating a vastly-cheaper, vastly-more-evidenced ADVISORY
candidate purely on authority precedence) — a cost this dataset accepts because Part XI's split-
authority discipline already treats FINAL-authority dispatch as a distinct, higher-scrutiny path,
softening the practical impact of this edge. REVERSAL CONDITION: if Part XX §20.5's governed-
evolution cycle accumulates evidence that strict ordering systematically underperforms a weighted
approach for a well-defined mission class, a weighted variant may be proposed through Article IX's
governance path — never adopted by silent self-tuning.

D2. DECISION: Workflow synthesis (Part VII) permits mid-mission re-composition, but bounded to
gating-capability discoveries and a single pass. ALTERNATIVES CONSIDERED: (a) no re-composition at
all — any decomposition mismatch fails the mission outright and requires a fresh route; (b)
unbounded re-composition — the workflow may re-plan its remaining structure any number of times as
new information arrives. WHY THIS: zero re-composition wastes the productive work already
completed by the time a gating mismatch is discovered; unbounded re-composition risks an
ungoverned mid-flight redesign loop that never reaches a terminal node. A single bounded pass
captures most of the practical value (a missed gating requirement, the most common and most
consequential mismatch shape) without the open-ended risk. TRADE-OFFS: a mission that genuinely
needs a second legitimate re-composition is forced into full re-decomposition (Part VII §7.5),
which is more expensive than a second bounded pass would have been, in the rare cases where the
second need is legitimate rather than a symptom of unreliable decomposition. REVERSAL CONDITION: if
live evidence (post-implementation) shows a meaningful fraction of missions legitimately need a
second bounded re-composition rather than full re-decomposition, the pass limit may be raised to
two via governed proposal, with the same append-only provenance discipline extended accordingly.

D3. DECISION: IAS-A1 holds SPLIT authority (final for advisory-outcome dispatch, advisory for
final-authority-node dispatch) rather than blanket final authority over all routing. ALTERNATIVES
CONSIDERED: (a) blanket final authority, letting IAS-A1 dispatch anything it routes without
downstream acknowledgment; (b) blanket advisory authority, requiring Owner sign-off on every
single dispatch regardless of risk, which would eliminate the router's own value by reintroducing
manual orchestration at every step. WHY THIS: split authority matches dispatch risk to dispatch
consequence, concentrating oversight exactly where an error is hardest to reverse (Part XI §11.3).
TRADE-OFFS: split authority requires the acknowledgment-and-escalation machinery of Part VIII §8.3
and Part XI §11.4 to be implemented correctly for the design to hold in practice — a partially-
implemented acknowledgment step would silently degrade toward blanket final authority without
anyone noticing, an implementation risk this dataset names explicitly in the Anti-Patterns section
below. REVERSAL CONDITION: none contemplated as a reversal to blanket authority in either
direction; a future revision might instead ADD finer authority gradations (e.g., a third tier for
moderate-consequence dispatches) rather than collapsing the split back to one level.

D4. DECISION: The capability index (Part III) is built by extraction from each system's declared
interface, maintained incrementally, rather than hand-authored or rebuilt in full on each change.
ALTERNATIVES CONSIDERED: (a) a hand-curated index maintained by a governance process independent
of the systems it describes; (b) a full rebuild triggered on any single system's change, rather
than incremental per-entry re-extraction. WHY THIS: hand-curation drifts from reality the moment
any system evolves without a curator noticing (the exact staleness IAS-F1 Part IV §4.3 warns
against generally); full rebuild is correct but wasteful at estate scale, re-deriving seventy-plus
systems' worth of entries to correct one. Incremental extraction, triggered by IAS-A2 change
events (Part XVI §16.3) plus a periodic backstop sweep, balances currency against cost. TRADE-OFFS:
incremental extraction depends on change signals actually being emitted and received (Part XII
§12.3's honest limit on SILENT REFACTOR DRIFT), a dependency a full rebuild would not have.
REVERSAL CONDITION: if the periodic backstop sweep's own cadence, once implemented, proves
insufficient to catch silent drift at an acceptable rate, the backstop frequency is a tunable
parameter adjustable without a full governance cycle, since it does not change this dataset's own
arbitration or authority logic, only a maintenance cadence.

D5. DECISION: Allocation (IAS-C1) informs routing only as a tie-break within the already-tied
arbitration case, never as a blended fifth criterion. ALTERNATIVES CONSIDERED: (a) allocation as a
fifth strict-priority criterion, inserted at some fixed rank among the existing four; (b) no
integration at all, keeping IAS-A1 and IAS-C1 fully independent. WHY THIS: a fixed-rank fifth
criterion would either dominate the existing four (if ranked early) or be nearly inert (if ranked
late), and in either case would blur the boundary between "which system correctly serves this
mission" (IAS-A1's object) and "which system the estate has chosen to invest in" (IAS-C1's object)
— two genuinely different judgments that should remain separately legible in the provenance record.
No integration at all would leave the Compounding Graph's own named primary-loop edge
unimplemented. Tie-break-only integration realizes the edge's value in exactly the cases where the
first four criteria are genuinely uninformative, without contaminating the cases where they are
not. TRADE-OFFS: in the (expected, per Part IX §9.5's illustrative frequency claim) minority of
missions that reach a genuine four-criterion tie, allocation's influence is real but narrow; a
stronger integration would matter in more missions but at the cost of the boundary this decision
protects. REVERSAL CONDITION: if IAS-C1's own maturation (a later mission, outside this dataset's
scope) demonstrates allocation data reliable and stable enough to serve as a genuine fifth
criterion without the blending risk named in Part XVII §17.3, a governed proposal may widen the
integration — but the burden of proof sits with the proposal, not with this dataset's current,
narrower default.

ANTI-PATTERNS
===============================================================================================

ANTI-PATTERN 1 — THE SILENT AUTHORITY COLLAPSE. The most likely implementation error is a partial
or missing acknowledgment step (Part VIII §8.3, Part XI §11.2) at FINAL-authority dispatch nodes,
implemented as a best-effort check that silently proceeds when the receiving system does not
respond in time, rather than as a hard blocking requirement. The consequence is that Decision D3's
split-authority design DEGRADES, invisibly, toward the blanket-final-authority alternative it was
deliberately built to avoid — every dispatch effectively becomes final regardless of the
destination capability's own declared authority, because the check that should have distinguished
them was implemented as advisory to itself. PREVENTION: the acknowledgment requirement at a FINAL-
authority node must be implemented as a hard block with no timeout-to-proceed default, and Part
XIV's metrics harness should include a specific check — the ACKNOWLEDGMENT GAP RATE, a metric this
dataset did not name in Part XIV proper but records here as a required implementation-time
addition — counting any FINAL-authority dispatch that proceeded without a logged acknowledgment,
treated as a CRITICAL implementation defect rather than a routing-quality metric, the instant that
rate is non-zero.

ANTI-PATTERN 2 — THE PROVISIONAL-STATUS SKIP. Because Part XIX §19.4's provisional-staging defense
against index corruption cascades adds latency (a newly re-extracted signature is not immediately
trusted at full confidence), an implementer under throughput pressure is likely to special-case it
away for "obviously safe" re-extractions — a routine cosmetic rename, say — reasoning that the
provisional friction is unnecessary overhead for low-risk changes. This is the exact failure shape
Part XIX §19.4 exists to prevent, because the corruption cascade's danger is not correlated with
how routine a re-extraction LOOKS; a signature written incorrectly due to a tooling bug during a
"routine" rename is observationally identical, at the point of arbitration, to a deliberately
corrupted one. PREVENTION: provisional staging applies uniformly to every re-extraction with no
"obviously safe" exemption path; if the latency cost proves genuinely material once implemented,
the correct governed response (per Article IX) is to shorten the provisional-to-standing
qualification bar (fewer required successful invocations before graduation), never to bypass the
stage entirely for a subjectively-judged-safe subset.

ANTI-PATTERN 3 — METRIC-COUNTERMETRIC DECOUPLING. Because Part XIV pairs every metric with a
countermetric specifically to prevent gaming, the most likely long-run implementation error is an
observability dashboard, built after this specification, that surfaces FIRST-HIT RATE or WRONG-
SYSTEM RATE prominently while quietly dropping GAMED-SIMPLICITY RATE or OVER-CAUTIOUS GAP RATE
from the same view — not through any decision to abandon the countermetrics, but through the
ordinary erosion of a dashboard that started complete and had fields trimmed for readability over
time. The consequence is that the router's own maintainers begin, in practice, to optimize the
metric they can see against the countermetric they can no longer see, reproducing exactly the
Goodhart failure Part XIV was built to prevent, despite the countermetric still existing
somewhere in the underlying data. PREVENTION: every dashboard or report surfacing a Part XIV metric
is required, as a structural rule rather than a style preference, to display its paired
countermetric in the SAME view, never in a separate or optional panel — the pairing is enforced at
the presentation layer specifically because presentation-layer erosion, not deliberate metric
abandonment, is the realistic failure vector.
