===============================================================================================
PART I — WHY PER-SESSION IMMUNITY CANNOT SEE ACROSS PROJECTS
===============================================================================================

1.1 The founding observation. The estate, as OBSERVED across the parent systems named in the header,
already contains substantial failure-handling machinery. cascade_prevention refuses a dangerous
command mid-session. osa/CEPS engineers context so a session does not compound one small error into a
cascade of larger ones. secret_firewall blocks a Write carrying a credential pattern. hard_rules
seals a bug that already happened into a standing STOP-trigger. refcheck verifies a plan's premises
against the real repository before code is written against them. sweep_enforcer executes a governance
sweep when one is scheduled. Each of these six systems is, in the vocabulary of IAS-F1, a real system:
it owns a distinct object, holds authority within its scope, exposes an interface, and is addressable.
None of them, individually or collectively, owns the object this dataset exists to define, because
each of them is bounded, by its own design, to a single session inside a single project. (OBSERVED:
every one of the six parent mechanisms fires, checks, or blocks within the transcript of one running
Claude Code session; DERIVED: none of the six persists a record that a second, unrelated project,
running its own session tomorrow, could consult before its own failure occurs.)

1.2 The specific gap, stated precisely. Consider the concrete shape of the gap rather than an
abstract description of it. Project Alpha ships a hard rule this week because a deploy shipped
without running tests — the rule is sealed, cascade_prevention's deploy-gate is updated, and Alpha is,
from that point forward, protected against a recurrence of exactly that failure. Project Beta, an
unrelated repository built by the same estate, sharing none of Alpha's code, has the identical
structural exposure: a deploy path that does not hard-require a test pass before shipping. Beta has
not yet had the incident. Under the estate's existing machinery, Beta remains exposed until Beta
either (a) suffers the same incident independently and seals its own local rule, or (b) an Owner or
agent happens to notice the parallel and manually ports Alpha's rule across. Both paths are OBSERVED
failure modes of the current estate: path (a) means the estate pays the cost of the same bug twice
before it learns once; path (b) depends on a human or an agent holding both projects in working
memory simultaneously, which does not scale past a handful of repositories and does not happen at all
for projects the acting agent is not currently working in. (DERIVED: the estate's learning curve, at
present, is per-project and reactive, not cross-project and preventive — the exact deficit IAS-D2 is
built to close.)

1.3 Why this is not merely "hard_rules needs a sync script." A shallow reading of 1.2 concludes the
fix is mechanical: copy Alpha's new rule file into Beta's rule directory. That reading is wrong for
three reasons, each of which is a reason the fix must be a SYSTEM and not a script. First, EXPOSURE IS
NOT UNIFORM: Beta may not have a deploy path at all, or may have one that already hard-requires tests
through a different mechanism, so a blind copy either does nothing useful or produces a false
positive that blocks legitimate work — Part X names this failure autoimmune. Second, RULES MUTATE:
the surface form of the next incident on the same underlying cause is rarely byte-identical to the
first (Part V), so a copy-the-rule-file approach immunizes only against an exact repeat, not against
the family of failures the incident actually represents. Third, THE DIRECTION OF PROPAGATION IS NOT
KNOWN IN ADVANCE: a sync script presumes Alpha is always the source and Beta always the destination,
but in a federation of many projects any project can be the first to discover any given pathogen, and
the propagation graph must be computed from exposure, not hardcoded from precedent. (DERIVED: a
correct solution requires a fingerprint that survives mutation, an exposure model that is per-project
and evidence-based rather than blanket, and a propagation mechanism that is symmetric across the
federation rather than source-fixed — Parts II through VI supply each of these in turn.)

1.4 The biological metaphor, adopted deliberately and bounded precisely. This dataset borrows its
vocabulary — pathogen, fingerprint, antibody, quarantine, immunization, autoimmune, immunity decay —
from immunology because the structural problem the estate faces (a hostile pattern recurs across
independent hosts that share a common vulnerability but no shared tissue) is the same structural
problem biological immune systems solve, and importing a mature vocabulary avoids re-deriving from
scratch a set of distinctions (innate vs. acquired response, antigen vs. antibody, autoimmune
overreaction) immunology already sharpened over a long history. (NORMATIVE: the metaphor is a naming
convenience, not a claim that the mechanisms are biologically accurate; every biological term used in
this dataset is re-DEFINED in software terms at first use, per Part II onward, and the metaphor is
abandoned wherever it would mislead rather than clarify.) The metaphor's specific value is that it
supplies a vocabulary for the exact asymmetry Part 1.2 exposed: a biological immune system does not
wait for every cell in the body to be independently infected before responding — it fingerprints a
pathogen once, in one exposure, and distributes resistance system-wide. That is precisely, and only,
the capability the estate currently lacks across its own projects.

1.5 Boundary restated against the two nearest parents. Because IAS-D2's object sits adjacent to two
parents with genuinely similar-sounding names, the boundary must be drawn twice, explicitly.
Against cascade_prevention: cascade_prevention's object is "should THIS command, in THIS session,
proceed" — a resource-level, in-session decision. IAS-D2's object is "which OTHER projects share the
exposure that made this command dangerous here" — an ensemble-level, cross-project decision that only
exists once cascade_prevention (or one of its four siblings) has already produced a first detection.
Against osa/CEPS: osa/CEPS's object is "how should THIS session's context be engineered so THIS
session does not compound a small error" — again resource-level and in-session. IAS-D2 consumes
osa/CEPS's detected-error events as raw material (Part III) but does not re-engineer any session's
context; it engineers the estate's cross-project exposure map instead. (DERIVED, applying IAS-F1 Part
VIII's formal level test: IAS-D2's direct object is the SET of projects and their exposure relations,
not any single session's command or context — Level(IAS-D2) = ENSEMBLE by the formal test, satisfying
the Non-Duplication Ledger's admission criterion.)

1.6 What "immunity" means operationally, defined ahead of the fuller apparatus. IMMUNITY, as DEFINED
in this corpus, is a property of a (project, pathogen) pair: a project is IMMUNE to a pathogen when
either (a) the project has been verified not to carry the structural exposure the pathogen requires,
or (b) the project carries a verified preventive protection — an antibody, Part IX — that intercepts
the pathogen's mechanism before it reaches production. Immunity is never asserted from the mere
existence of a fingerprint in a registry; it requires either an exposure-negative scan (Part VI) or a
deployed antibody (Part VIII-IX), each producing its own evidence level. A project with a matching
fingerprint on file but no scan and no antibody is not immune — it is merely KNOWN AT RISK, a
distinct and weaker state named explicitly in Part VI so the two are never confused in a report.

1.7 The founding claim of the whole dataset. A federation of N independently built projects, sharing
one estate of governance mechanisms but no shared deployment, currently learns from a production
failure exactly once per project that happens to suffer it — an O(N) learning cost for a single root
cause. IAS-D2's founding claim, carried through every subsequent Part, is that the same root cause
should cost the estate exactly ONE full incident, with every other exposed project immunized before
it independently pays the same cost — an O(1) learning cost, amortized across however many projects
the fingerprint's exposure model finds vulnerable. (HYPOTHESIZED as a general claim; DERIVED as a
specific mechanism once Parts II through IX are read together; the claim's truth for the estate is an
empirical matter Part XVI's evaluation protocol exists to check, not something this Part asserts as
already proven.)

FINAL LAW — PART I. Six parent systems already detect and block failures, but every one of them is
bounded to one session inside one project; the estate's cross-project immunity is owned by none of
them, and until a system owns it explicitly, the same root cause will keep costing the estate a full
incident in every project unlucky enough to be next.

===============================================================================================
PART II — THE PATHOGEN OBJECT: A TAXONOMY OF INSTITUTIONAL FAILURE
===============================================================================================

2.1 Why a taxonomy precedes a mechanism. Before a fingerprint can be computed (Part III) or an
exposure scan run (Part VI), the dataset must answer a prior question: what KINDS of thing can even
be a pathogen? A mechanism built against an unstated, ad hoc notion of "bad thing" will fit whichever
failure motivated its author and miss every structurally different one. F1's own method — DEFINE the
object classes before specifying operations over them — is imported here at the domain-specific level:
this Part DEFINES a closed, six-member taxonomy of institutional pathogens, derived from the actual
failure shapes the estate's own hard-rule and postmortem history exhibits, not from an abstract list
of "things that could go wrong" in general software.

2.2 Class One — FALSE-DONE. A FALSE-DONE pathogen is a claim of task completion that is not backed by
observed evidence: a "done," "shipped," or "ready" assertion where the underlying verification (a
test run, a build, a real deploy check) either did not happen or happened and was ignored. (DEFINED,
matching the estate's own Reality Contract and HR-OUTPUT-002.) The mechanism of harm is specific: a
false-done claim propagates downstream as if it were true, and every system that trusts a "done"
status without re-verifying — a dependent task, a release note, an Owner's mental model — inherits the
falsehood. False-done is the single most estate-general pathogen class because it can attach to any
artifact type (a feature, a migration, a fix, a dataset) and its detection signature (a completion
claim with no adjacent evidence token) is uniform across attachment sites, which is precisely why it
recurs across unrelated projects independently rather than needing a shared root cause.

2.3 Class Two — PROXY-EVIDENCE. A PROXY-EVIDENCE pathogen substitutes a weaker, easier-to-produce
signal for the evidence a claim actually requires, and then treats the proxy as if it were the real
thing: static analysis passing treated as proof of runtime correctness (the estate's own Mistake #16
and #17), a file's existence treated as proof a feature is wired end-to-end (the Scaffold Illusion), a
compiled-without-error state treated as proof the compiled code is called by anything. (DEFINED.) The
distinguishing feature separating proxy-evidence from false-done is that proxy-evidence pathogens
involve genuine, real evidence — the compile really succeeded, the file really exists — the corruption
is entirely in the INFERENCE drawn from that evidence, not in the evidence's existence. This
distinction matters operationally (Part III) because a proxy-evidence fingerprint must encode the
specific weak-to-strong evidence substitution being made, not merely "insufficient evidence" in
general, or the fingerprint will be too broad to be actionable.

2.4 Class Three — INEFFECTIVE RULE. An INEFFECTIVE RULE pathogen is a governance rule — a hard rule, a
hook, a gate — that exists, is correctly triggered, but fails to actually prevent the harm it was
sealed to prevent, either because its STOP condition is too narrow (matches the exact original
incident but not its near neighbors), its enforcement mechanism is advisory when the harm requires
hard blocking, or its trigger fires after the harmful action rather than before it. (DEFINED.)
Ineffective-rule pathogens are the most institutionally embarrassing class because they masquerade as
solved problems — the estate's own governance registry shows a rule exists, a postmortem shows the
rule was written in direct response to an incident, and yet the incident's underlying cause continues
to produce harm because the rule's coverage is narrower than the cause. This class is why Part XIV
names "false immunity" as a first-class failure mode: an ineffective rule is precisely a pathogen the
estate believes it is already immune to.

2.5 Class Four — CONTAMINATED PROMPT. A CONTAMINATED PROMPT pathogen is an instruction, template, or
context artifact — a slash command body, a hook's injected context, a subagent prompt, a copy-pasted
prior session's output — that carries an unintended directive, an injected instruction from untrusted
data, or a stale assumption presented as current fact, and which, when reused, reproduces its
contamination in every session that consumes it. (DEFINED, aligned with the estate's own Prompt
Defense Baseline.) This class differs from the other five in propagation mechanism: while
false-done, proxy-evidence, ineffective-rule, and stale-knowledge pathogens propagate because
independent projects share a STRUCTURAL exposure, a contaminated prompt propagates because it is
LITERALLY COPIED — a template, a snippet, a "here's how we did it last time" reuse — carrying its
defect verbatim into every project that reuses the artifact rather than merely sharing the
vulnerability class.

2.6 Class Five — PARASITIC COMPLEXITY. A PARASITIC COMPLEXITY pathogen is a mechanism, abstraction, or
process step that consumes more ensemble value — tokens, maintenance attention, cognitive load on the
next agent that must reason around it — than it returns, expressed in IAS-F2's Institutional Advantage
Unit as a persistently negative or near-zero score, and which nonetheless persists because no single
project-level review catches it in isolation. (DEFINED, cross-referencing IAS-D1's "parasitic
complexity detector" candidate at the single-system level; IAS-D2's distinct concern is a parasite
PATTERN — an entire class of overhead, such as a specific over-engineered scaffolding habit — recurring
across multiple projects' codebases independently, which no single project's local review is
positioned to notice as a pattern rather than as one file's problem.)

2.7 Class Six — STALE KNOWLEDGE. A STALE KNOWLEDGE pathogen is a fact, assumption, or cached summary
that was true when recorded and is no longer true, and which continues to be trusted past its
reconciliation window — precisely the property-honesty failure IAS-F1's Invariant Five names at the
system level, applied here to any knowledge artifact: a vault entry, an audit-cache summary, a
memorized API signature, a "this is how the deploy pipeline works" note that predates a pipeline
change. (DEFINED.) Stale knowledge is the slowest-acting of the six pathogen classes — it rarely
causes an immediate crash — and the most corrosive over time, because it erodes silently until an
agent acts on it with full confidence and produces a failure that looks, superficially, like a fresh
bug rather than the symptom of an old fact quietly going wrong.

2.8 Why exactly six, and the taxonomy's closure property. The six classes were not chosen to be
exhaustive of all conceivable software failure — that would be an unbounded, ungovernable list — they
were DERIVED from the actual failure shapes recorded in the estate's own hard-rule and mistake
registry (HR-OUTPUT-001/002/003, Mistake #16/#17, the Prompt Defense Baseline, the Reality Contract,
IAS-D1's parasitic-complexity candidate, IAS-F1's property-honesty invariant), grouped by their
propagation mechanism rather than by their surface symptom, until every OBSERVED estate incident fit
exactly one class without remainder. (DERIVED, with the closure claim itself CONTESTED: new incident
types may surface a seventh class the estate has not yet suffered; Part XXII names this the novel-
pathogen edge case and specifies how the taxonomy accepts an amendment rather than forcing a
misclassification.) A pathogen record (Part III) is required to declare exactly one primary class from
this six, with a secondary class permitted where a real incident genuinely straddles two — a
false-done claim resting on proxy evidence is common enough that the taxonomy allows dual tagging
rather than forcing an arbitrary primary choice.

2.9 Worked example distinguishing the six on one synthetic incident family. Consider a synthetic,
illustrative incident: a project's deploy script reports "deployed successfully" (a false-done claim)
because it checks that the deploy COMMAND exited zero (proxy-evidence — command-exit-zero substituted
for service-actually-responds), a hard rule exists requiring a post-deploy health check but only
triggers on a specific deploy command name that this script does not use (ineffective rule — narrow
trigger), the deploy script itself was copy-pasted from a template that never included a health check
step (contaminated prompt/template — literal propagation), the script has grown three additional retry
layers over eighteen months that no longer correspond to any real failure mode still occurring
(parasitic complexity), and the vault note describing "how deploys are verified here" still describes
the pre-health-check process (stale knowledge). One incident, five of the six classes present
simultaneously, each requiring a DIFFERENT remediation — this is precisely why a single "bug" cannot
be treated as one pathogen: it must be decomposed into its constituent classes before a fingerprint
(Part III) can be computed for each, or the resulting single fingerprint will be too coarse to match
any other project's partial exposure.

FINAL LAW — PART II. A pathogen is not "a bug" in general; it is one of six named, mechanism-derived
classes — false-done, proxy-evidence, ineffective rule, contaminated prompt, parasitic complexity,
stale knowledge — and a failure record that does not resolve to at least one of these six classes has
not yet been diagnosed, only observed.

===============================================================================================
PART III — THE FAILURE FINGERPRINT: A CANONICAL, MUTATION-TOLERANT REPRESENTATION
===============================================================================================

3.1 Why the raw incident is not the unit of federation. A postmortem, a hard-rule entry, or a
session's error transcript is rich, narrative, and specific to the one incident that produced it —
exactly the properties that make it useless as a federation unit. If IAS-D2 federated raw incident
text, every consuming project would need to re-read and re-interpret a paragraph of prose to decide
whether its own exposure matches, an operation too expensive to run automatically across every
project on every incident and too subjective to produce a reliable match. The FAILURE FINGERPRINT, as
DEFINED here, is the compressed, structured, comparison-ready representation extracted from an
incident, built specifically so that two independently produced fingerprints can be tested for a
match by a cheap, mechanical operation rather than by re-reading either incident's narrative.

3.2 The fingerprint schema. A failure fingerprint carries seven fields, each chosen because at least
one downstream mechanism in this dataset reads it directly. PRIMARY CLASS and SECONDARY CLASS (Part
II's six-member taxonomy, one required, one optional) situate the pathogen in the taxonomy.
MECHANISM SIGNATURE is a structured, abstracted description of HOW the pathogen operates — not the
specific file or command involved, but the general pattern (example, abstracted from Part 2.9: "a
completion signal derived from process-exit-code rather than from post-action service verification"),
because the mechanism, not the surface syntax, is what a mutation (Part V) preserves. STRUCTURAL
PRECONDITION is the property a project must have for the pathogen to be able to occur at all (example:
"has a deploy step producing a completion signal"), and is the field IAS-D2's exposure scan (Part VI)
tests against directly. SEVERITY is drawn from the same conjunctive vocabulary the estate's hard rules
already use (CRITICAL/HIGH/MEDIUM/LOW), never re-invented. ORIGIN records the project, session, and
approximate date the fingerprint was first extracted, so provenance is never lost. REMEDIATION CLASS
names the shape of fix that resolved the originating incident (example: "add a post-action
verification step gating the completion signal") without prescribing the exact code, because the
remediation's exact implementation is project-specific while its shape is what other projects reuse.
(DEFINED, all seven fields.)

3.3 Mechanism signature as the load-bearing field. Of the seven, MECHANISM SIGNATURE carries the
fingerprint's entire discriminating power, and its construction is the dataset's most delicate
specification. A mechanism signature must be abstract enough to match a mutated recurrence (Part V)
and concrete enough not to match unrelated incidents that merely share a taxonomy class — "false-done"
alone is far too broad a signature, since nearly every project could exhibit SOME false-done pathogen
at SOME point, and a fingerprint that matches at that granularity floods every exposure scan with
false positives. The signature is therefore specified as a SLOT-FILLED TEMPLATE: a short structural
sentence with typed slots (ACTION-KIND, VERIFICATION-KIND, SUBSTITUTED-SIGNAL) filled from the
originating incident, so that two incidents match when their slot-fills match under a controlled
similarity tolerance (Part V's mutation-distance function), not when their prose happens to overlap.
(DERIVED: this is the specific technical answer to "how does a fingerprint survive being rephrased by
a different author in a different project" — it survives because the fingerprint was never prose to
begin with.)

3.4 Where a fingerprint is extracted FROM, and why IAS-D2 does not invent detection. Every fingerprint
in this dataset's registry has, by construction, an origin in one of the six parent detectors named in
the header, or in a hard-rule seal, a postmortem, or an Owner-flagged incident report — never in
IAS-D2 independently deciding a session went wrong. (NORMATIVE, restating the header's boundary: IAS-D2
extracts and federates fingerprints; it does not detect the underlying failures.) The extraction step
itself is a defined, mechanical transform — read the incident's evidence (a hard-rule STOP entry, a
cascade_prevention block log, a secret_firewall match, a refcheck premise failure) and populate the
seven fields — and because the extraction reads only what a parent already produced, IAS-D2 strengthens
those parents (per the Strengthening Matrix) by giving their local, one-off detections a second life
across the federation, without ever second-guessing whether the parent's original detection was
correct.

3.5 Fingerprint identity and the deduplication problem. Two independently extracted fingerprints, from
two unrelated incidents in two unrelated projects, may describe the same underlying pathogen using
different slot-fills if the two authors abstracted the mechanism differently. This is the FINGERPRINT
DEDUPLICATION problem, and it is solved not by textual equality (which would fail on the very variance
just described) but by a similarity check over the mechanism-signature slots combined with the
structural-precondition field: two fingerprints with matching ACTION-KIND and VERIFICATION-KIND slots
and an equivalent structural precondition are treated as ONE pathogen with two independent origins, a
stronger form of evidence than either alone (a pathogen independently rediscovered in two projects is
more likely to be a real, general vulnerability class than one reported once). (DERIVED: independent
rediscovery is evidence, not noise, and the dedup step is designed to capture that evidence — Part
XVI's evaluation protocol treats multi-origin fingerprints as higher-confidence than single-origin
ones for exactly this reason.)

3.6 The fingerprint's evidence level and its epistemic lifecycle. A freshly extracted fingerprint
carries evidence level OBSERVED with respect to its origin incident (the incident really happened, the
extraction transform is mechanical) but only HYPOTHESIZED with respect to any OTHER project's exposure
(Part VI has not yet run against that project). This distinction matters because a fingerprint
propagated across the federation (Part IV) before any exposure scan runs is a WARNING, not a VERDICT —
treating an unscanned fingerprint match as proof of exposure is the exact error Part XIV's false-
immunity failure mode inverts (believing you are protected when you are not); the mirror error, believing
you are exposed when you are not, is equally real and is why Part VI's exposure scan, not the
fingerprint's mere existence, is what upgrades a project's status from "flagged" to "confirmed exposed"
or "confirmed clear."

3.7 Worked example: extracting a fingerprint from the Part 2.9 synthetic incident. Applying the schema
to the false-done/proxy-evidence component of the synthetic deploy incident: PRIMARY CLASS =
false-done, SECONDARY CLASS = proxy-evidence. MECHANISM SIGNATURE = "a completion signal
[ACTION-KIND: deploy] is derived from [VERIFICATION-KIND: process-exit-code] rather than from
[SUBSTITUTED-SIGNAL: post-action service health check]." STRUCTURAL PRECONDITION = "project has an
automated or semi-automated deploy step that emits a completion signal to a consumer (Owner, CI log,
downstream task) without a gating health check between the deploy action and the signal." SEVERITY =
HIGH (a false-done deploy claim can mask a production outage). ORIGIN = the synthetic project, session,
date, marked illustrative. REMEDIATION CLASS = "insert a post-action verification step between the
deploy action and the completion signal, gating the signal on the verification's result." This single
fingerprint, once federated, is testable against any OTHER project by asking exactly one question —
does this project's deploy path emit a completion signal without a gating health check — which is a
mechanical exposure test (Part VI), not a re-reading of the original incident's narrative.

3.8 Boundary against over-fingerprinting. Not every session error merits a fingerprint. A one-off
mistake with no structural precondition — an agent misread a file path once, corrected immediately, no
governance gap implicated — has no MECHANISM SIGNATURE worth extracting, because there is no
STRUCTURAL PRECONDITION for a second project to share; extracting a fingerprint anyway would pollute
the registry with noise that dilutes the signal of genuine, structurally-recurring pathogens. (NORMATIVE:
fingerprint extraction is gated on the incident producing evidence of a STRUCTURAL, not merely
INCIDENTAL, cause — the same discipline hard_rules already applies when deciding whether an incident
merits a sealed rule versus a one-time fix, and IAS-D2 inherits rather than relaxes that discipline.)

FINAL LAW — PART III. A failure fingerprint is not the incident's story; it is the incident's
mechanism, abstracted into typed slots precisely so that a second project, sharing none of the first
project's code, can still be tested against it by a cheap structural question rather than a narrative
re-reading.

===============================================================================================
PART IV — CROSS-PROJECT FEDERATION: HOW A FINGERPRINT TRAVELS
===============================================================================================

4.1 The federation problem, stated as a graph problem. Given a set of projects P and a newly extracted
fingerprint F with origin in project p0 ∈ P, federation is the process of determining, for every other
project p ∈ P \ {p0}, whether p should receive F for exposure scanning, and doing so without requiring
a human to enumerate P by hand or an agent to hold every project in working memory at once. (DEFINED.)
This is precisely the object the Non-Duplication Ledger names for candidates 41-50 taken together, and
it is why IAS-D2 cannot be built as a purely local artifact inside any one project's own governance
folder — federation requires a channel that reaches across project boundaries, which no single
project's local hooks or hard-rule files can provide on their own.

4.2 The registry as the federation substrate. IAS-D2 specifies a PATHOGEN REGISTRY: a shared,
estate-level store of fingerprint records (Part III's schema) that is NOT scoped to any one project,
analogous in placement to the estate-level knowledge_vault the estate already OBSERVED to exist
outside any single project's directory. (DERIVED, importing the estate's own existing pattern of
estate-scoped versus project-scoped storage rather than inventing a new storage tier.) The registry
is the passive half of federation: it is where a fingerprint, once extracted, becomes available to
every project's exposure scan, present or future, without requiring the originating project to know
in advance which siblings are exposed.

4.3 The active half: event-carried federation via the Control Bus. A registry alone is federation by
polling — a project's exposure scan must periodically re-read the whole registry to notice a new
fingerprint, which is workable but slow. IAS-D2's ACTIVE federation path consumes the ensemble-level
event mechanism the Compounding Graph's defense loop names explicitly: "IAS-A2 Control Bus carries
failure events → IAS-D2 Immune System fingerprints and federates them cross-project." (OBSERVED from
the Compounding Graph; DERIVED operationally: when a parent detector — cascade_prevention,
secret_firewall, hard_rules, refcheck — produces a qualifying incident, IAS-D2 extracts a fingerprint
per Part III and PUBLISHES it as an event on IAS-A2's control bus; any project's exposure-scan
subsystem, if it SUBSCRIBES to the pathogen-fingerprint event class, receives the new fingerprint
without polling.) This is the specific, checkable mechanism the Strengthening Matrix requires for
IAS-D2's row: IAS-A2 is strengthened because it gains a concrete, high-value event class to carry, and
IAS-D2 is strengthened because it inherits a transport it did not have to build.

4.4 Push versus pull, and why federation needs both. Pure push (event-only) federation loses any
project that was offline, newly onboarded, or simply not yet listening at the moment a fingerprint was
published — a real and common condition given the estate's own session-based, non-always-on
architecture (sessions start and end; there is no permanently running listener in every project by
default). Pure pull (registry-only) federation is correct eventually but slow, and provides no signal
about WHICH fingerprints are new since a project's last scan without an explicit bookmark. IAS-D2
therefore specifies BOTH: the registry is the durable source of truth every exposure scan may
re-consult in full, and the control-bus event is the low-latency notification that a scan SHOULD run
now rather than waiting for its next scheduled pass. (NORMATIVE: an exposure scan that only ever
listens to events and never reconciles against the full registry will silently miss fingerprints
published while the project was between sessions — Part VI's exposure-scan specification requires a
periodic full-registry reconciliation precisely to close this gap.)

4.5 Federation direction is symmetric, not source-fixed. Part 1.3 already established that any project
can be the first to discover any given pathogen; the federation mechanism reflects this by having no
notion of a "source project" and a "destination project" baked into its architecture — every project
is, simultaneously, a potential origin (any of its sessions may trigger a parent detector that yields a
new fingerprint) and a potential recipient (its exposure scan subscribes to the same event class every
other project's origin events are published on). (DERIVED from 4.3's mechanism: publication and
subscription are symmetric operations over the same bus and the same registry; there is no
architectural asymmetry between "the project that found it" and "the project that receives it" beyond
the ORIGIN field Part 3.2 already records for provenance.)

4.6 Federation scope and the boundary invariant. Federation is bounded to the ensemble as IAS-F1 Part
III §3.4 defines it: projects the estate governs. A fingerprint is never published to, or scanned
against, a system outside the estate's boundary — a third party's repository, a public dependency's
upstream source, the model provider's infrastructure — because IAS-D2's authority, like every IAS
system's, stops at the boundary IAS-F1's Invariant Four names. (NORMATIVE, direct application of the
boundary invariant; Part XI returns to this boundary when discussing containment, and Part XIV's
imperial-boundary adversarial case is the failure mode this restatement exists to forestall.)

4.7 Federation latency as a first-class, measured quantity. Because the entire value of federation is
that project p receives immunizing knowledge BEFORE p independently suffers the same incident, the
TIME between a fingerprint's extraction at p0 and its availability for exposure scanning at every other
project is not a cosmetic performance detail — it is the quantity Part XV's Mean-Time-To-Federate
metric exists to measure, and it directly bounds how much of the "O(1) learning cost" claim from Part
1.7 the estate actually realizes versus how much degrades back toward the O(N) baseline because
federation was too slow to beat the next incident's occurrence. (DERIVED: MTTF is not incidental to
federation's design, it is federation's central performance question, and the push/pull hybrid of 4.4
is specified the way it is BECAUSE pull-only federation has an MTTF bounded below by the exposure
scan's polling interval, which the header's design goal treats as unacceptably slow for a HIGH or
CRITICAL severity fingerprint.)

4.8 Worked example: a fingerprint's journey across three projects. Project Alpha's deploy incident
(Part 3.7) produces fingerprint F-2201, published to the registry and to the control bus the same
session it is extracted. Project Beta, mid-session at the moment of publication and subscribed to the
pathogen-fingerprint event class, receives F-2201 as an event within the session, triggers an
immediate targeted exposure scan (Part VI) against just F-2201's structural precondition, and confirms
exposure within minutes — this is federation's push path performing as designed. Project Gamma has no
active session running when F-2201 is published; Gamma's next session, three days later, begins with a
scheduled full-registry reconciliation (per 4.4's requirement) that surfaces F-2201 as unseen, and
Gamma's exposure scan runs then — slower than Beta's, but not lost, which is the pull path's
guarantee. Project Delta, upon inspection, has no deploy step at all and is, by Part VI's structural
test, simply not a candidate host for F-2201's precondition; Delta correctly never runs a targeted scan
for this fingerprint, illustrating that federation reaching every project does not mean every project
is scanned identically — the fingerprint reaches all three live projects' registries, but the exposure
TEST, not the federation event, is what determines which projects actually invest scan effort.

FINAL LAW — PART IV. A fingerprint federates through two channels at once — a durable registry every
project may reconcile against, and a low-latency event every subscribed project receives immediately —
because a federation mechanism graded only on eventual correctness has already conceded the race
against the next incident it exists to prevent.

===============================================================================================
PART V — FAILURE-MUTATION DETECTION: SAME CAUSE, CHANGED SURFACE
===============================================================================================

5.1 The mutation problem, stated. A pathogen rarely recurs byte-identical to its first occurrence. The
second project's deploy script uses a different language, a different CI tool, a different variable
name for its completion signal, and a naive fingerprint match — string equality, or even a loose
keyword match over the incident's prose — will fail to recognize the second incident as the same
pathogen, because the SURFACE has changed even though the MECHANISM (Part 3.3's signature) has not.
(DEFINED: a FAILURE MUTATION is a recurrence of a fingerprint's mechanism signature under a changed
surface realization — different code, different names, different language, different tool — such that
a surface-level comparison would score it as unrelated while a mechanism-level comparison correctly
scores it as the same pathogen family.) This Part specifies how the registry distinguishes a genuine
mutation (same cause) from a genuinely new, unrelated pathogen (different cause), because conflating
the two in either direction is a distinct, named failure mode.

5.2 Why mutation detection is harder than the fingerprint match of Part III alone. Part 3.5's
deduplication logic already handles the case of two INDEPENDENTLY EXTRACTED fingerprints describing the
same pathogen with different slot-fills — that is a matching problem over two already-abstracted
signatures. Mutation detection is a different, harder problem: it must decide whether a NEW, raw
incident (not yet fingerprinted) is a mutation of an EXISTING registry fingerprint, before the new
incident's own fingerprint has even been extracted, so that the extraction step itself can be informed
by the candidate match rather than treating the new incident as if it had no history. (DERIVED: mutation
detection sits temporally BEFORE Part 3.5's dedup step in the pipeline, feeding it a candidate match to
confirm or reject, rather than operating independently after the fact.)

5.3 The mutation-distance function. IAS-D2 specifies a MUTATION DISTANCE as a bounded score in [0, 1]
computed between a new incident's provisional mechanism-signature slots and every registry fingerprint's
slots, where 0 means an exact mechanism match (the incident IS the pathogen, surface variance aside)
and 1 means no shared structural preconditions at all. The distance is computed per-slot (ACTION-KIND,
VERIFICATION-KIND, SUBSTITUTED-SIGNAL) with the STRUCTURAL PRECONDITION field weighted most heavily,
because two incidents that share a structural precondition but differ in surface action are far more
likely to be the same mutating pathogen than two incidents that happen to share a verification-kind
slot by coincidence. (DEFINED, with the specific weighting scheme marked EXPERIMENTAL: the estate has
not yet accumulated enough multi-mutation pathogen histories to calibrate the weights against outcome
data, and Part XVI's evaluation protocol names weight calibration as an explicit, ongoing evaluation
target rather than a value fixed here by assertion.)

5.4 The mutation threshold and its two-sided risk. A LOW mutation distance (below a threshold τ,
provisionally 0.25, EXPERIMENTAL) classifies the new incident as a mutation of the matched fingerprint
— the registry updates that fingerprint's MUTATION LINEAGE (5.6) rather than creating a new entry. A
HIGH distance (above 1 − τ) classifies it as unrelated. The interval between the two thresholds is
AMBIGUOUS and is never auto-resolved; it is surfaced for Owner or agent review with both candidate
interpretations shown, because forcing a binary classification inside the ambiguous band risks the two
symmetric errors this dataset must avoid equally: classifying a genuine mutation as unrelated (which
LOSES the immune memory the estate already paid for) and classifying an unrelated incident as a
mutation (which POLLUTES an existing fingerprint's lineage with a spurious variant, degrading its
future matching precision for everyone). (NORMATIVE: the ambiguous band is a designed feature, not a
gap — an automatic tie-break in either direction is worse than an honest UNRESOLVED classification
awaiting more evidence.)

5.5 Why the same-root-cause claim requires a REMEDIATION CLASS check, not just a mechanism match. A
mechanism-signature match alone is necessary but not sufficient to confirm a mutation, because two
genuinely distinct pathogens can coincidentally share slot values while differing in what actually
fixes them. IAS-D2 therefore requires a SECOND confirming check before a mutation lineage is finalized:
the candidate incident's eventual remediation, once known, must be an instance of the matched
fingerprint's REMEDIATION CLASS (Part 3.2) — if the fix that resolves the new incident is structurally
different from what resolved the original, the mutation hypothesis is REJECTED regardless of how close
the mechanism-signature distance scored, because "the same fix works" is stronger evidence of a shared
root cause than "the description sounds similar." (DERIVED: this two-check design — signature distance
as the candidate filter, remediation-class match as the confirming test — is what keeps mutation
detection from becoming a purely linguistic-similarity exercise divorced from the causal claim it is
actually making.)

5.6 The mutation lineage as a first-class object. A PATHOGEN LINEAGE, DEFINED, is the ordered set of
all fingerprints (an original plus zero or more confirmed mutations) sharing one root cause, together
with the specific slot-level deltas each mutation introduced relative to the lineage's earliest member.
A lineage is more informative than any single fingerprint in it, because the PATTERN of what varies
across confirmed mutations (which slots change, which stay fixed) predicts what the NEXT mutation is
likely to look like — a lineage where every confirmed mutation has varied ACTION-KIND while
STRUCTURAL PRECONDITION stayed fixed tells the exposure scan (Part VI) to weight structural precondition
even more heavily for that specific lineage's future matches, a lineage-specific refinement Part V's
global weighting (5.3) cannot express on its own.

5.7 Worked example: a three-member lineage across mutating surfaces. Fingerprint F-2201 (Part 3.7,
Alpha's deploy false-done). Eight weeks later, Project Epsilon's incident: not a deploy script but a
database-migration runner that reports "migration applied" based on the migration tool's exit code
without querying the schema afterward to confirm the change landed — ACTION-KIND has changed from
"deploy" to "migrate," VERIFICATION-KIND is identical ("process-exit-code"), SUBSTITUTED-SIGNAL is
analogous ("post-action schema check" in place of "post-action service health check"), and
STRUCTURAL PRECONDITION generalizes cleanly ("has an automated action step emitting a completion
signal without a gating post-action check"). Mutation distance scores LOW; the eventual fix (insert a
post-action verification gating the signal) matches F-2201's REMEDIATION CLASS exactly. F-2201's
lineage now has two members. A further six weeks on, Project Zeta's incident superficially resembles
the pattern (a completion claim, a process-exit-code check) but its eventual fix is unrelated —
Zeta's actual defect was a race condition between two concurrent processes, not a missing verification
step — so despite an initially LOW-scoring mechanism-signature distance, the remediation-class check
(5.5) rejects the mutation hypothesis, and Zeta's incident is fingerprinted as a new, unrelated entry
rather than incorrectly folded into F-2201's lineage. This worked trace demonstrates both halves of
5.5's two-check design operating on realistic, adjacent-looking candidates.

5.8 Mutation detection's dependency on the exposure scan's own history. A subtle but important
property: mutation detection improves as the lineage accumulates members, because each new confirmed
mutation refines the lineage's slot-weighting (5.6). This means the SAME new incident, presented to the
registry early in a lineage's life versus late, may receive a different mutation-distance score purely
because the registry has learned more about which slots that lineage's mutations actually vary — a
property Part XIII (immunity decay) must account for in reverse: a lineage that stops accumulating new
members for a long period is not necessarily "cured," it may simply be that the estate has stopped
producing incidents in that structural region, and Part XIII specifies how the registry distinguishes
those two very different explanations for silence.

FINAL LAW — PART V. A pathogen that has mutated once will mutate again, and a fingerprint registry that
can only recognize a pathogen's ORIGINAL surface has already lost the war against its second
occurrence; mutation detection exists because the mechanism, not the syntax, is what must be matched,
and even a mechanism match is provisional until the remediation confirms the causal claim it implies.

===============================================================================================
PART VI — EXPOSURE SCANNING: WHICH PROJECTS ARE VULNERABLE
===============================================================================================

6.1 The exposure question, distinguished from the federation question. Part IV answered "how does a
fingerprint REACH a project"; this Part answers a different question entirely — "given that project p
HAS a fingerprint F in reach, is p actually exposed to F's pathogen." The two questions are
independent and must not be conflated: a project can receive every fingerprint the federation
publishes and still be exposed to none of them (if it never carries any of the structural
preconditions), and a project's exposure to a given fingerprint does not depend on whether federation
has reached it yet — exposure is a property of the project's own structure, federation is merely how
the project LEARNS its exposure. (DERIVED, restating Part 3.6's distinction between a fingerprint
match being a warning versus an exposure scan being a verdict.)

6.2 The exposure scan as a structural-precondition test. An EXPOSURE SCAN, DEFINED, is the operation of
testing a project's actual structure — its files, its configuration, its process, its governance
coverage — against a fingerprint's STRUCTURAL PRECONDITION field, producing one of three verdicts:
EXPOSED (the precondition is present and no antibody, Part IX, currently intercepts the mechanism),
CLEAR (the precondition is absent, or present but already gated by an existing, independently-verified
control), or UNRESOLVED (the scan could not determine the precondition's status — commonly because the
relevant artifact was not readable, or the project's structure does not map cleanly onto the
fingerprint's abstracted precondition language). (DEFINED, three-valued verdict, deliberately not
binary — collapsing UNRESOLVED into either EXPOSED or CLEAR by default is a named failure mode, Part
XIV, because a default-to-CLEAR silently under-protects and a default-to-EXPOSED silently floods
Part VII's quarantine mechanism with false alarms.)

6.3 Why exposure scanning is per-fingerprint, not a single global sweep. It is tempting to imagine
exposure scanning as one large sweep that checks a project against "all known pathogens" in a single
pass. IAS-D2 specifies against this design deliberately: each fingerprint carries its own, narrow
structural-precondition test, and a scan is most efficient, most accurate, and cheapest to reason
about when it is TARGETED — run against the specific fingerprint(s) a federation event just delivered,
or against the full registry only on the periodic reconciliation cadence 4.4 requires. (DERIVED: a
single monolithic "scan everything" sweep re-tests every precondition on every run regardless of
whether anything changed, which is the specific inefficiency sweep_enforcer's existing scheduling
discipline already avoids for its own sweep classes, and IAS-D2 inherits rather than reinvents that
discipline by delegating scan EXECUTION, once specified, to sweep_enforcer per the Strengthening
Matrix.)

6.4 The evidence hierarchy for an exposure verdict. Not every exposure verdict carries equal weight.
IAS-D2 specifies an evidence hierarchy matching Article V's vocabulary: a verdict is OBSERVED when the
scan directly read the relevant artifact (the deploy script's actual content, confirmed absent a
health-check step); DERIVED when the verdict follows from a related, already-OBSERVED fact (a project
confirmed to have no deploy automation at all is DERIVED-clear for every deploy-preconditioned
fingerprint without each needing its own direct read); and HYPOTHESIZED when the scan relied on a
project-type heuristic without reading the specific artifact (example: "projects of this framework
type typically have this exposure," asserted without confirming this specific project's actual
configuration). A quarantine action (Part VII) may be triggered from an OBSERVED or DERIVED EXPOSED
verdict; a HYPOTHESIZED verdict may only trigger a scan request, never a quarantine, because acting on
an unconfirmed hypothesis is precisely the autoimmune risk Part X exists to prevent.

6.5 Cost-bounded scanning and the scan-priority function. Running a full, artifact-reading exposure
scan against every fingerprint for every project is not free — it consumes agent attention, tokens, and
time proportional to the registry's size and the estate's project count, a cost the header's boundary
against re-implementing per-session mechanisms does not exempt IAS-D2 from bearing honestly. A
SCAN-PRIORITY FUNCTION orders pending scans by SEVERITY (Part 3.2) descending, then by the fingerprint's
lineage size (Part 5.6 — a pathogen confirmed to mutate repeatedly is a higher scanning priority than a
single-occurrence one), then by the receiving project's own prior exposure history (a project
previously EXPOSED to fingerprints in the same lineage is scanned before one with no history in that
region). (NORMATIVE: priority ordering exists because scan capacity is finite and Part XV's metrics
must be read against this ordering — a low completion rate on LOW-severity, single-occurrence
fingerprints is a healthy sign of correct prioritization, not a coverage failure, and conflating the
two is a specific instance of the ratio-gaming failure mode Part XV's countermetrics guard against.)

6.6 The exposure map as the registry's second core artifact. Alongside the pathogen registry (Part
4.2), IAS-D2 specifies an EXPOSURE MAP: a matrix, conceptually, of (project, fingerprint) pairs each
carrying the three-valued verdict of 6.2, its evidence level, and a last-reconciled timestamp — the
project-property-record discipline of IAS-F1 Part IV §4.3 applied at the (project, pathogen)
granularity rather than the single-system granularity. The exposure map is what makes a question like
"which projects remain exposed to CRITICAL-severity fingerprints published this quarter" answerable in
one read rather than requiring a fresh scan of every project on every query, and it is the artifact
IAS-E1's Observability Fabric and IAS-E2's Cognitive Reliability both consume as raw material for their
own downstream reasoning (Part XVII-XVIII).

6.7 Worked example: exposure scanning the F-2201 lineage across the four synthetic projects. Continuing
Part 4.8's trace, once F-2201's lineage includes the migration-runner mutation from Epsilon (Part 5.7),
a subsequent targeted scan re-runs against ALL structural preconditions the lineage now covers, not
merely the original deploy-specific one. Beta's original scan (4.8) tested only the deploy
precondition and returned CLEAR after remediation; re-scanning against the generalized lineage
precondition ("any automated action step emitting an unGated completion signal") surfaces that Beta
ALSO has a database-backup step with the same unguarded pattern — a exposure the original,
narrower scan could not have found, because the lineage's precondition generalized only after
Epsilon's mutation was confirmed. This is the direct operational payoff of Part V's lineage mechanism
feeding Part VI's scan: exposure scanning is not a one-time test per fingerprint, it is a test that
must be RE-RUN as a lineage's structural-precondition field generalizes with each confirmed mutation,
and a project scored CLEAR against an early lineage member is not permanently cleared against later
generalizations of the same lineage.

6.8 Boundary: exposure scanning never mutates the scanned project. An exposure scan is read-only by
definition — it tests, it does not fix. The moment a scan confirms EXPOSED with sufficient evidence to
act, authority passes to Part VII's quarantine mechanism or Part VIII's immunization mechanism, each of
which is a distinct, separately-authorized operation; IAS-D2's scanning subsystem itself never writes
to a scanned project's files, precisely mirroring IAS-A2's advisory-versus-final authority distinction
(IAS-F1 Part IV §4.2) — an exposure verdict is, by construction, advisory information about a project's
state, never a final action against it.

FINAL LAW — PART VI. Exposure is a property of a project's own structure, not of whether a fingerprint
has reached it; a scan that defaults an unresolved precondition to either CLEAR or EXPOSED has replaced
evidence with a guess, and a lineage's exposure test must be re-run as the lineage generalizes, because
the precondition a project was correctly cleared against yesterday may no longer be the precondition
that pathogen represents today.

===============================================================================================
PART VII — QUARANTINE: ISOLATING AN ACTIVE PATHOGEN
===============================================================================================

7.1 What quarantine is, and is not, in this ontology. QUARANTINE, DEFINED, is a temporary, reversible
restriction placed on a specific capability or code path within a confirmed-EXPOSED project, scoped as
narrowly as the fingerprint's structural precondition allows, that prevents the pathogen's mechanism
from completing while leaving the rest of the project's normal operation untouched. Quarantine is
explicitly NOT a project-wide freeze, NOT a revert of prior work, and NOT a punishment — it is a
surgical containment measure whose entire justification is the exposure verdict of Part VI, and it
expires automatically the moment Part VIII's immunization mechanism deploys a durable antibody or the
exposure verdict is revised to CLEAR on re-scan. (NORMATIVE: scoping quarantine to the narrowest
capability that satisfies the precondition, rather than to the whole project, is what separates a
functioning immune response from the autoimmune overreaction Part X names — an immune system that
quarantines indiscriminately is functionally indistinguishable, from the Owner's perspective, from the
pathogen it was meant to contain.)

7.2 The quarantine decision gate. Quarantine is authorized only when three conditions hold jointly:
the exposure verdict is EXPOSED with evidence level OBSERVED or DERIVED (never HYPOTHESIZED, per Part
6.4); the fingerprint's SEVERITY is HIGH or CRITICAL (a LOW or MEDIUM severity exposure is routed to
Part VIII's immunization queue directly, without the interim quarantine step, because the cost of a
quarantine action — even a narrow one — exceeds the risk of leaving a low-severity path open for the
short interval until a permanent antibody deploys); and no existing antibody already covers the
fingerprint's lineage (checked against Part IX's immune memory before quarantine is even considered,
so the system never quarantines a path that is already durably protected). (DEFINED, conjunctive
gate; DERIVED from the header's cost discipline — quarantine is reserved for the cases where the
exposure is both confirmed and dangerous enough that the narrower, cheaper, permanent fix cannot be
awaited safely.)

7.3 Quarantine's four legal actions, and no others. To keep quarantine mechanical and auditable rather
than an open-ended discretionary power, IAS-D2 specifies exactly four legal quarantine actions, each
mapped to a specific pathogen class from Part II. ADVISORY-GATE inserts a mandatory confirmation step
before the vulnerable action proceeds (appropriate for false-done and proxy-evidence pathogens, where
the fix is "require the missing verification," and a human or agent confirmation substitutes for it
temporarily). SIGNAL-DOWNGRADE marks the vulnerable path's output as unverified rather than trusted,
propagating that downgrade to any consumer of the signal (appropriate where the pathogen's harm is
downstream trust in a false signal). ROUTE-SUSPEND removes a specific automated trigger from firing
(appropriate for contaminated-prompt pathogens where a specific template or hook is the carrier —
suspend the template's automated reuse, not the project's ability to operate). SCOPE-FREEZE halts
further growth of a specific parasitic-complexity pattern (new instances of the pattern are blocked;
existing instances are left for Part VIII's remediation) without touching unrelated code. (DEFINED,
four actions, closed set; NORMATIVE: a proposed quarantine action outside this set is not authorized —
Part XIV's over-quarantine failure mode is specifically the failure of exceeding this closed set's
scope.)

7.4 Quarantine is never silent. Every quarantine action, on activation, produces a visible record —
the equivalent of a hard-rule STOP entry — naming the fingerprint, the exposure evidence, the specific
action taken, and the condition under which it will lift. (NORMATIVE, direct application of the
estate's Reality Contract at the immune-system layer: an invisible restriction that blocks legitimate
work without explanation is indistinguishable, to the person or agent it blocks, from a bug in the
governance layer itself, and is precisely the condition that produces an Owner's loss of trust in
governance broadly — the same dynamic the estate's own false-positive-catalog discipline already
guards against for code review findings.) The visible record is what lets an Owner distinguish a
correctly-functioning quarantine from a misfire, and it is the primary input to Part X's autoimmune
detection, which reads exactly this record stream to look for patterns of quarantine blocking
legitimate work.

7.5 Quarantine duration and the escalation clock. A quarantine action carries a default REVIEW WINDOW
(EXPERIMENTAL default: 14 days for HIGH severity, 3 days for CRITICAL) after which, if no antibody has
deployed and no re-scan has cleared the exposure, the quarantine escalates — not by tightening further
(quarantine does not compound on itself) but by surfacing to the Owner queue as a stalled remediation,
because a quarantine that persists past its review window without progress toward a durable fix is
evidence the narrow containment has become a substitute for the permanent one rather than a bridge to
it, a drift this dataset treats as a governance debt requiring visibility rather than as a stable
resting state. (NORMATIVE.)

7.6 Interaction with cascade_prevention: quarantine as an upstream input, not a duplicate gate. Where a
quarantine action's scope overlaps a decision cascade_prevention would independently evaluate (example:
a deploy-related ADVISORY-GATE overlapping cascade_prevention's own deploy-tests-passed check),
IAS-D2 does not install a second, competing gate — it PUBLISHES the quarantine condition as an input
cascade_prevention's existing gate consumes, per IAS-F1's singular-authority invariant (Invariant
Three): cascade_prevention retains final authority over the in-session block/proceed decision;
IAS-D2's quarantine record is advisory information that makes cascade_prevention's existing decision
better-informed, never final authority.

7.7 Worked example: quarantining Beta's newly-discovered exposure. Continuing the trace from Part 6.7,
Beta's database-backup step is confirmed EXPOSED to the generalized F-2201 lineage with OBSERVED
evidence (the backup script was directly read and confirmed to lack a post-action verification), and
the lineage's severity is HIGH. Quarantine gate (7.2) is satisfied: EXPOSED/OBSERVED, HIGH severity, no
existing antibody for this lineage covers backup steps yet (Part IX's memory only has a deploy-scoped
antibody so far). The action selected is ADVISORY-GATE, scoped narrowly to the backup script's
completion-signal step: any session that would mark a backup "complete" based on the backup tool's
exit code alone is now required to confirm an explicit post-action check before the signal is trusted,
while every other part of Beta's project — including its already-remediated deploy path — is
untouched. The quarantine record is published, visible in Beta's governance surface, with a 14-day
review window and an explicit note that it lifts automatically once Part VIII deploys a
backup-scoped antibody or the lineage's remediation-class fix is applied directly.

7.8 What quarantine deliberately does not attempt. Quarantine does not attempt to auto-fix the
underlying code — the four legal actions all restrict or downgrade trust in an output, none of them
write a durable remediation, because auto-writing a fix into a project IAS-D2 does not own crosses the
governed-self-evolution boundary (Constitution Article IX) that reserves estate-mutating changes for
Owner-promoted action. (NORMATIVE: this is a deliberate, load-bearing limitation, not an unfinished
feature — Part VIII's immunization mechanism specifies a proposed antibody, but even immunization stops
at PROPOSAL; deployment of an antibody that changes a project's actual code remains an Owner-gated
action, consistent with the header's absorption of candidates 41-50 as an intelligence-and-coordination
layer rather than an autonomous-mutation layer.)

FINAL LAW — PART VII. Quarantine is a narrow, visible, self-expiring restriction on the exact capability
a confirmed, severe exposure implicates — never a project-wide freeze, never silent, and never a
substitute for the durable fix it exists only to bridge toward.

===============================================================================================
PART VIII — IMMUNIZATION: FROM INCIDENT TO PREVENTIVE PROTECTION
===============================================================================================

8.1 Immunization as the durable half of the immune response. Where quarantine (Part VII) is a temporary
bridge, IMMUNIZATION, DEFINED, is the process of converting a confirmed pathogen's REMEDIATION CLASS
(Part 3.2) into a project-specific, durable protection — a preventive check, a governance rule, or a
structural change specification — deployed to an EXPOSED project BEFORE that project independently
suffers the pathogen's harm. Immunization is the mechanism that actually delivers on Part 1.7's founding
claim: the incident cost the estate exactly once; every other exposed project receives protection as a
PROPOSAL derived from that one incident's remediation, never as a repeat of the incident itself.

8.2 The immunization pipeline, four stages. IMMUNIZATION PROPOSAL: given a confirmed EXPOSED verdict
(Part VI) and the fingerprint's REMEDIATION CLASS, IAS-D2 drafts a project-specific instantiation of the
remediation class — not generic advice ("add verification") but a concrete, scoped proposal referencing
the exact capability the exposure scan located ("gate the backup script's completion signal on a
post-action schema-row-count check"). PROPOSAL REVIEW: the proposal is surfaced through the estate's
existing Owner-decide channel (owner_queue, or direct Owner review for CRITICAL severity), never
auto-applied, per the governed-self-evolution boundary already invoked in Part 7.8. DEPLOYMENT: upon
Owner approval, the concrete implementation is a normal, project-scoped code or governance change — made
by whatever mechanism the estate already uses to make such changes (a session, hard_rules sealing,
sweep_enforcer executing a scoped sweep) — IAS-D2 does not introduce a new deployment mechanism, it
supplies the SPECIFICATION deployed through an existing one. CONFIRMATION: once deployed, the exposure
scan (Part VI) is re-run against the same fingerprint, and a CLEAR verdict with the antibody's presence
noted converts the project's status from EXPOSED to IMMUNE (Part 1.6's operational definition), closing
the loop.

8.3 The antibody record as the durable artifact. An ANTIBODY, DEFINED, is the confirmed, deployed
instantiation of a remediation class in a specific project — the record that a specific fingerprint's
mechanism is, in this project, intercepted before it can complete. An antibody record carries the
fingerprint or lineage it addresses, the project it protects, the concrete mechanism deployed
(referenced, not duplicated — a pointer to the actual governance artifact, hook, or code change, since
IAS-D2 does not maintain a shadow copy of project code), a deployment date, and a CONFIRMATION status
(re-scanned CLEAR, or pending). Antibody records, in aggregate across the federation, constitute the
IMMUNE MEMORY (Part IX) — the federation's accumulating, checkable record of which pathogens which
projects are now durably protected against.

8.4 Why immunization must be project-specific and never a blind template copy. Part 1.3 already
established, at the founding-gap level, that a blind copy-the-fix-file approach fails because Beta's
actual deploy mechanism may differ structurally from Alpha's even when both share the same
STRUCTURAL PRECONDITION. Immunization's PROPOSAL stage exists specifically to close this gap: the
REMEDIATION CLASS (Part 3.2) is deliberately specified as a SHAPE of fix ("gate the completion signal on
a post-action check") rather than a literal artifact, and the proposal stage is where that shape is
instantiated against the RECEIVING project's own concrete structure, discovered by that project's own
exposure scan (Part VI) — never against the originating project's implementation details, which the
receiving project may not share at all.

8.5 Prioritizing the immunization queue. Not every EXPOSED verdict can be immunized simultaneously
across every project at once; the immunization queue is prioritized by the same severity-first,
lineage-size-second ordering Part 6.5 already establishes for scanning, extended with one additional
factor: PROXIMITY TO RECURRENCE, an estimate (marked HYPOTHESIZED where no better signal exists) of how
soon the exposed project's own operational cadence is likely to exercise the vulnerable path — a
project that deploys daily and carries an unremediated deploy-related exposure is prioritized above an
equally-severe exposure in a project whose corresponding path runs quarterly, because the expected
TIME-TO-HARM, not merely the severity label, is what the estate is racing against. (NORMATIVE: this
priority factor is explicitly why Part XV's Mean-Time-To-Federate metric is paired with a companion
Mean-Time-To-Immunize metric — federation speed alone does not capture whether the SLOWER stage,
immunization, actually beats the exposed project's own operational clock.)

8.6 Immunization and the second-order-capability test. Applying IAS-F1 Part VII §7.2's mechanical test
to immunization itself: its direct object is not any single project's vulnerable code (a resource-level
object cascade_prevention or a project's own governance already owns) — its direct object is the
REMEDIATION CLASS as a reusable, cross-project specification, and the decision of WHICH projects
receive an instantiation of it. This confirms immunization is a genuinely ensemble-level operation, not
merely a batch of first-order fixes performed serially — the batching itself, and the shared shape
underlying every instance in the batch, is exactly the second-order object Part VII of F1 requires.

8.7 Worked example: immunizing Gamma and a fourth project preemptively. Returning to the F-2201 lineage,
Gamma's delayed reconciliation (Part 4.8) eventually confirms EXPOSED on its own migration-runner path
(matching Epsilon's mutation, Part 5.7). An immunization proposal is drafted: gate Gamma's migration
tool's completion signal on a post-migration schema query, an instantiation of the SAME remediation
class already deployed at Alpha and Beta but concretely different in implementation because Gamma's
migration tooling differs from either. Separately, a fifth synthetic project, Theta — never directly
implicated in any of the four prior incidents — is scanned as part of the routine full-registry
reconciliation (Part 4.4) and found to carry the generalized structural precondition on an entirely
different capability (a report-generation step). Theta's immunization is genuinely PREVENTIVE in the
strongest sense: no incident has ever occurred at Theta involving this pathogen, and Theta receives
protection purely from the federation-and-scan pipeline, before any harm — the clearest demonstration
available of Part 1.7's founding claim actually operating end to end.

8.8 Immunization's limits: what it cannot promise. Immunization confirms that a SPECIFIC, matched
mechanism is intercepted; it is never claimed to make a project immune to the pathogen CLASS in
general (Part II's six classes), only to the specific fingerprint or lineage matched. A project immune
to F-2201's lineage remains fully exposed to an unrelated false-done pathogen with a different
mechanism signature, and reporting an immunized project as "protected against false-done pathogens"
rather than "protected against this specific lineage" is a specific overclaim this dataset forbids —
Part XIV names the general shape of this error as false immunity, and immunization's own record
(8.3) is scoped narrowly precisely to prevent the record itself from inviting the overclaim.

8.9 Antibody revocation: when the protection itself fails. An antibody is not exempt from the taxonomy
that produced it. IAS-D2 specifies a REVOCATION path for the case where a deployed, CONFIRMED antibody
is later discovered to be an INEFFECTIVE RULE (Part 2.4) in its own right — its gate too narrow for a
mutation the lineage subsequently accumulates (Part V), or the receiving project's structure changed
after deployment so the gate no longer sits on the path it was built to guard (a refactor moved the
completion signal upstream of the antibody's check without anyone re-running Part VI's exposure scan
against the moved path). REVOCATION, DEFINED, is the operation of downgrading an antibody's status from
CONFIRMED to SUSPECT, which in turn downgrades every project holding that antibody from IMMUNE back to
KNOWN AT RISK (Part 1.6) for the affected fingerprint or lineage, and re-queues a fresh exposure scan at
elevated priority — an antibody discovered ineffective is treated as MORE urgent than a fresh, never-
immunized exposure, because a project trusting a revoked antibody has been operating with a false sense
of protection, which is strictly worse than knowing it is exposed. (NORMATIVE: revocation is triggered
by evidence — a new mutation the antibody does not cover, or a re-scan finding the gate no longer wired
— never by mere suspicion or the passage of time alone; Part XIII's immunity-decay monitor is the
distinct, weaker signal for time-based re-verification, while revocation is reserved for a CONFIRMED
gap.) Worked example: Beta's backup-step antibody (Part 6.7) gates the backup script's completion
signal on a schema-row-count check; eleven months later Beta's team migrates the backup step to a
managed cloud job that emits its own completion webhook upstream of the original script entirely — the
antibody's code is still present and still passes its own unit test, but the completion signal it was
built to gate no longer flows through the path it guards. A routine lifecycle audit (Part XXIII) flags
the antibody as SUSPECT pending re-scan; the re-scan confirms the gate is orphaned, revocation fires,
Beta's status for the F-2201 lineage reverts to KNOWN AT RISK, and a fresh immunization proposal is
drafted against the new managed-job path. Without an explicit revocation path, Beta's exposure map would
have continued reporting IMMUNE indefinitely — a silent regression exactly as dangerous as never having
been immunized at all, and one Part VI's evidence hierarchy alone cannot catch, because a stale CONFIRMED
antibody record is not, by itself, evidence of anything; it is merely evidence that once, at deployment
time, a re-scan was CLEAR.

8.10 Immunization cost accounting and the diminishing-return floor. Drafting a project-specific proposal
(8.2) is not free, and the estate's own cost discipline (HR-COST-002) applies to immunization exactly as
it applies to any other agent-executed task: a proposal's drafting cost is tracked against the
fingerprint's SEVERITY-weighted expected harm, and IAS-D2 specifies a floor below which further
immunization spend on a single lineage is DEFERRED rather than pursued immediately — a LOW-severity,
single-occurrence fingerprint whose only remaining unimmunized project shows HYPOTHESIZED-level exposure
(Part 6.4) is queued behind every CONFIRMED, higher-severity proposal rather than drafted on demand,
because the expected harm avoided no longer clears the drafting cost at that confidence level.
(DERIVED: this is the immunization-side mirror of Part 6.5's scan-priority function, and the two
priority orderings are kept deliberately consistent — a fingerprint that was low priority to SCAN for is,
for the same underlying reason, low priority to IMMUNIZE against once exposure is confirmed, so an
agent reading either queue draws the same conclusion about where the estate's attention is actually
warranted.)

FINAL LAW — PART VIII. Immunization is not the incident's fix copied elsewhere; it is the incident's
remediation SHAPE, re-instantiated against each exposed project's own structure, deployed only with
Owner approval, and confirmed only by a re-scan — and a project's antibody protects against exactly the
lineage it was built for, no broader, no matter how the record is later summarized, and no longer than
the antibody's gate actually sits on the path it was built to guard.

===============================================================================================
PART IX — THE ANTIBODY RECORD AND THE IMMUNE MEMORY STORE
===============================================================================================

9.1 Why immune memory is a separate object from the pathogen registry. Part 4.2's pathogen registry
answers "what pathogens does the estate know about." A separate, equally necessary question is "what
protections does the estate already have deployed" — and collapsing the two into one store would force
every consumer that only cares about CURRENT PROTECTION STATE to filter a much larger stream of
historical incident data on every read. IAS-D2 therefore specifies IMMUNE MEMORY as its own store: the
full set of antibody records (Part 8.3) across every project, indexed by fingerprint/lineage and by
project, queryable in either direction — "which pathogens is project p immune to" and "which projects
are immune to lineage L" are both single-index lookups, not derived joins computed fresh on every
query.

9.2 The two-directional index and its operational payoff. The project-indexed direction of immune
memory is what Part 7.2's quarantine gate consults ("does an antibody already cover this exposure") and
what Part VI's exposure scan short-circuits against (a project with a confirmed, current antibody for a
lineage does not need re-scanning against that lineage on every reconciliation pass, only on the
antibody's own staleness cadence, Part XIII). The lineage-indexed direction is what a NEW project —
onboarding into the estate for the first time — consults to receive a PRE-POPULATED exposure baseline:
rather than scanning a brand-new project against the entire historical registry from zero knowledge,
the immune-memory lookup surfaces which lineages have PROVEN, reusable remediation classes ready to
propose immediately if the new project's own scan confirms the matching structural precondition,
compressing the new project's time-to-baseline-immunity relative to an estate with no memory at all.

9.3 Antibody staleness and the reconciliation obligation. An antibody record is not permanently valid
merely because it was once confirmed CLEAR; the protected code path can itself change (a refactor
removes the gating check without anyone realizing it was serving an immunizing function, because the
check's origin as an antibody was not visible at the point of the unrelated refactor). IAS-D2 therefore
specifies that every antibody record inherits IAS-F1's property-honesty discipline (Invariant Five)
directly: a CONFIRMATION status ages, and Part XIII's immunity-decay monitor is the mechanism that
periodically re-validates a sample of antibody records against their still-live code paths rather than
trusting the original confirmation indefinitely.

9.4 Immune memory as a shared artifact across sibling IAS datasets. Immune memory is not consumed by
IAS-D2 alone. IAS-E1's Observability Fabric reads it to report which pathogen classes the estate has
durably closed versus which remain open exposure. IAS-E2's Cognitive Reliability reads a project's
antibody coverage as one input to that project's failure-budget calculation (Part XVIII) — a project
with dense antibody coverage over its high-traffic paths carries a different reliability posture than
an equally-sized project with sparse coverage. IAS-C1's Capability Portfolio, per the Compounding
Graph's defense loop, eventually benefits from the freed capacity lower recurrence produces, though it
consumes that benefit indirectly through IAS-E1's usage data rather than reading immune memory
directly. (DERIVED, cross-referencing the Compounding Graph's defense-loop and cross-dataset feedback
table.)

9.5 The provenance chain from incident to antibody, preserved end to end. A defining design property of
immune memory is that every antibody record carries an unbroken provenance chain back to its
originating incident: antibody → immunization proposal → exposure scan verdict → fingerprint →
originating parent-detector event (Part 3.4). This chain matters for a concrete operational reason: when
an antibody is later found to have failed (Part XIV's false-immunity failure mode, or a genuine novel
mutation the antibody's narrower scope did not anticipate), the chain lets an investigator walk
backward to the exact originating incident and remediation-class definition, distinguishing "the
antibody was correctly built but the pathogen mutated past its scope" (a Part V mutation-detection
matter, feeding a lineage update) from "the antibody was incorrectly instantiated against this
project's structure" (an immunization-pipeline defect, Part 8.2).

9.6 Memory capacity and pruning discipline. Immune memory grows monotonically with the estate's
incident history, and an ever-growing, never-pruned store degrades query performance and signal density
over a long enough horizon. IAS-D2 specifies that antibody records are never DELETED (a deleted record
would erase provenance the estate might need later) but may be ARCHIVED — moved to a lower-priority tier
excluded from routine scan short-circuiting (9.2) but still queryable — once a lineage has shown no new
mutations (Part V) and no confirmation failures for a sustained period, EXPERIMENTAL default two years,
long enough that seasonal or infrequent-cadence projects are not prematurely archived out of active
protection status.

9.7 Worked example: a new project's onboarding baseline. Project Iota joins the estate with no incident
history of its own. Its onboarding exposure scan runs against the full pathogen registry (per 4.4's
reconciliation logic) but is INFORMED by immune memory's lineage-indexed view: for every lineage with
one or more confirmed antibodies elsewhere in the estate, Iota's scan is flagged HIGH-CONFIDENCE-USEFUL,
because a matched exposure there has an immediately available, already-proven remediation class ready
to propose — versus a lineage with fingerprints but zero antibodies yet, where a matched exposure would
trigger a fresh immunization-proposal cycle with no prior instantiation to draw on. Iota's scan confirms
exposure to the F-2201 lineage on a report-export step (a fourth mechanism realization of the same
generalized precondition); because three prior antibody instantiations already exist in immune memory,
Iota's immunization proposal (Part 8.2) is drafted with unusually high confidence and unusually low
review burden, a direct, measurable benefit of the memory store's existence that a registry-only design
(Part 9.1's rejected alternative) would not provide.

9.8 The boundary between immune memory and hard_rules. Immune memory records CROSS-PROJECT protection
state at the (project, pathogen) granularity; hard_rules records STANDING, PROJECT-LOCAL STOP
conditions at the (trigger, action) granularity, most commonly the exact enforcement mechanism an
antibody's deployment stage (8.2) may itself take the FORM of. The two are not competitors: a single
antibody's concrete implementation may literally BE a new hard_rules entry in the receiving project,
in which case immune memory's antibody record and hard_rules' STOP entry are two views (federation-scoped
and enforcement-scoped, respectively) of the same underlying protection, linked by the provenance chain
of 9.5 rather than duplicated as independent facts.

9.9 Scoring an antibody's standing value under IAS-F2's shared unit. Once confirmed (Part 8.2's fourth
stage), an antibody is not merely a binary protected/unprotected flag — it is an ongoing asset with a
maintenance cost and a continuing benefit, and both are expressible in IAS-F2's Institutional Advantage
Unit (referenced, not re-narrated, per this dataset's ontology-resolution clause). An antibody's IAU
numerator is DERIVED from the exposure it closes — weighted by the lineage's severity (Part 3.2) and the
count of projects it protects, whether directly (its own deployment) or by proving a remediation class
other projects' antibodies subsequently reused (Part 9.2's memory dividend). Its denominator is the
maintenance burden the deployed mechanism itself imposes on the receiving project: an ADVISORY-GATE that,
once deployed, never requires further human attention costs less to carry than a SCOPE-FREEZE whose
boundary must be periodically re-verified against code that keeps growing around it (Part 13.3's
code-drift sampling). A lineage's aggregate IAU, summed across every antibody it has produced, is the
single number IAS-C1's Capability Portfolio needs to weigh "keep investing in closing this lineage's
remaining exposure" against every other claim on the estate's cognitive capital, on the same shared scale
every other capability in the estate is scored on. (DERIVED for the existence of the mapping; HYPOTHESIZED
for the specific numerator/denominator split proposed here — IAS-F2 owns the authoritative formula, and
IAS-D2's contribution is limited to supplying the raw exposure-closed and maintenance-cost facts an IAU
calculation over an antibody would require, never to computing the IAU itself.) This is the concrete,
checkable mechanism behind the Compounding Graph's own claim that immune memory eventually feeds IAS-C1's
allocation math: the feed runs through IAS-F2's shared unit, not through immune memory's own ad hoc
scoring, which is exactly why Part 9.4 describes the IAS-C1 relationship as indirect — indirect through a
shared formula is a stronger integration than a direct but incompatible one would be.

FINAL LAW — PART IX. A pathogen registry that grows without a matching memory of what has already been
cured re-litigates its own history on every query; immune memory is the federation's answer to "are we
protected right now," queryable in both directions, provenance-linked back to the incident that earned
it, and archived, never erased, so the estate's immunity is never quietly forgotten.

===============================================================================================
PART X — AUTOIMMUNE DETECTION: WHEN GOVERNANCE ATTACKS LEGITIMATE WORK
===============================================================================================

10.1 The autoimmune risk, named precisely. An AUTOIMMUNE EVENT, DEFINED, is a quarantine action (Part
VII) or an immunization deployment (Part VIII) that blocks, degrades, or gates a code path that was
never actually exposed to the pathogen its fingerprint describes — a false positive at the exposure or
mechanism-matching layer that manifests, to the person or agent it affects, exactly as governance
attacking legitimate work. This is not a hypothetical risk invented for completeness; it is the
structural mirror of every prior Part's design choice, because every mechanism capable of correctly
blocking a real pathogen is, by the same construction, capable of incorrectly blocking a non-pathogen
that merely resembles one closely enough to satisfy the same matching logic.

10.2 The four autoimmune entry points, mapped to their originating Parts. OVER-BROAD MECHANISM SIGNATURE
(Part 3.3): a signature abstracted too loosely matches legitimate patterns that share surface features
with the pathogen but not its actual harmful mechanism — the specific risk Part 3.3 already names and
guards against with slot-typing, but which remains a live risk if a signature is authored carelessly.
FALSE MUTATION CONFIRMATION (Part 5.4-5.5): the ambiguous-band or remediation-class check fails to
reject a coincidental resemblance, folding an unrelated, legitimate pattern into a lineage it does not
belong to. HYPOTHESIZED EXPOSURE ACTED UPON (Part 6.4): a quarantine or immunization action triggered
from unconfirmed, heuristic exposure evidence rather than an OBSERVED or DERIVED verdict, violating Part
7.2's gate. STALE ANTIBODY MISFIRING (Part 9.3): a protection mechanism that was correctly scoped at
deployment continues to gate a path whose surrounding code has since legitimately changed such that the
original exposure no longer applies, but the antibody was never re-validated to notice.

10.3 The autoimmune detection mechanism: reading the quarantine and immunization record stream for
harm-pattern signatures. IAS-D2 does not detect autoimmune events by re-running the original matching
logic more carefully in isolation — a system cannot reliably audit its own single decision for a false
positive using the same reasoning that produced the decision. Instead, autoimmune detection is a
SEPARATE, downstream read of the visible quarantine/immunization record stream (Part 7.4's mandatory
visibility requirement is precisely what makes this downstream read possible) for a specific behavioral
signature: a protection mechanism that is repeatedly OVERRIDDEN, DISPUTED, or MANUALLY BYPASSED by
Owners or agents across multiple independent instances is exhibiting the autoimmune pattern, because a
correctly-targeted protection blocking genuinely harmful work should rarely need overriding, while an
autoimmune one systematically does. (DERIVED: this is a behavioral, evidence-based detection strategy —
override frequency as the signal — rather than a re-inspection of the original matching decision, and it
is deliberately analogous to how a biological autoimmune condition is diagnosed by its symptomatic
pattern rather than by re-examining the immune cell's original targeting logic in isolation.)

10.4 The override-rate threshold and its Goodhart risk. A protection mechanism whose override rate
exceeds a threshold (EXPERIMENTAL default: three independent overrides across three distinct sessions or
projects within a rolling window) is flagged CANDIDATE-AUTOIMMUNE and routed for review — not
automatically disabled, because a genuinely correct but inconvenient protection could also accumulate
overrides from Owners simply choosing to accept risk consciously, and auto-disabling on override
frequency alone would let a determined pattern of legitimate risk-acceptance train the immune system to
stand down exactly the protections it should be enforcing hardest. (NORMATIVE: this is why 10.4's
threshold triggers REVIEW, never automatic deactivation — the override-rate signal is evidence a human
should examine, not a self-correcting control loop the system is trusted to close unsupervised, directly
consistent with Constitution Article IX's governed-self-evolution boundary.)

10.5 Distinguishing autoimmune from a merely inconvenient but correct protection. The review step (10.4)
must answer a specific question the override-rate metric alone cannot: was each override an instance of
"this protection is wrong" (autoimmune) or "this protection is right but this specific case is an
accepted exception" (correct, if inconvenient)? IAS-D2 specifies that this distinction is made by
re-examining, for each override, whether the OVERRIDDEN CASE actually satisfied the fingerprint's
structural precondition (Part 6.2) — if it did, and the pathogen's mechanism genuinely applied, the
override was risk-acceptance, not evidence of autoimmunity, however inconvenient; if it did not — the
precondition test itself was a false positive — the override is confirmed autoimmune, and the underlying
mechanism signature or exposure test is the actual defect requiring correction, not the individual
project's response to it.

10.6 Remediation of a confirmed autoimmune mechanism. Once confirmed, an autoimmune mechanism is
corrected at its SOURCE — the over-broad signature is narrowed (Part 3.3), the mutation-confirmation
check is tightened (Part 5.5), the acted-upon evidence threshold is raised (Part 6.4), or the stale
antibody is re-validated and re-scoped (Part 9.3) — never merely by disabling the specific instance that
triggered the review, because disabling only the local instance leaves the same defective mechanism free
to misfire identically against the next, different project, which is itself a second-order pathogen: an
ineffective-rule pathogen (Part 2.4) in the immune system's OWN governance, a recursive case Part XIV
returns to directly.

10.7 Worked example: a candidate autoimmune event in the F-2201 lineage. Suppose a sixth synthetic
project, Kappa, has a code-review-report generator that emits a "review complete" signal derived from
its own process exit code — matching F-2201's mechanism signature closely on the surface. An exposure
scan flags Kappa EXPOSED, and, because the lineage's severity is HIGH, a quarantine ADVISORY-GATE
deploys, requiring confirmation before the review-complete signal is trusted. Kappa's maintainer
overrides the gate twice within a week, each time with the same annotation: the report generator's
process already INCLUDES a full post-generation validation step internally, before it ever exits — the
"process-exit-code" signal in Kappa's case is not a proxy for a missing check, it IS the check,
integrated into the same process rather than a separate gating step the fingerprint's slot template
assumed would be external. Review (10.5) confirms: Kappa's structural precondition does NOT actually
match F-2201's — the precondition assumed the verification step is separate from and after the action,
and Kappa's design collapses the two into one atomic step, a distinction the original slot-fill did not
anticipate. This is confirmed autoimmune, and the remedy (10.6) is to refine F-2201's STRUCTURAL
PRECONDITION field to explicitly exclude the atomic-verification case, protecting every other project
sharing Kappa's architecture from the same future misfire, not merely lifting Kappa's own gate.

10.8 Autoimmune detection's own failure mode, named to close the loop. A detection mechanism built to
catch a specific class of error can itself fail in the mirrored direction: an override-rate threshold
set too low flags genuinely correct protections as autoimmune merely because they are inconvenient
enough to be overridden occasionally by an impatient session, degrading trust in the protection for the
wrong reason. This is why 10.4 specifies review rather than automatic action, and why Part XV's metrics
(autoimmune false-positive rate, tracked as its own quantity) hold the autoimmune detector itself to the
same evidentiary discipline it applies to the pathogens it is meant to catch — the immune system does
not get an exemption from the standards it enforces on everything else.

FINAL LAW — PART X. A protection mechanism repeatedly overridden is not automatically wrong and not
automatically right; it is a signal demanding the same evidence discipline as the original exposure
verdict, and an autoimmune mechanism, once confirmed, must be fixed at its structural source — never
merely silenced at the one project unlucky enough to have surfaced it first.

===============================================================================================
PART XI — FAILURE-PROPAGATION CONTAINMENT: BOUNDING THE BLAST RADIUS ACROSS THE ENSEMBLE
===============================================================================================

11.1 The distinct question containment answers. Every prior Part in this dataset addresses a pathogen
that has already been discovered and fingerprinted. CONTAINMENT, DEFINED, addresses a different,
time-critical question: while a pathogen is ACTIVELY PROPAGATING — mid-incident, before a fingerprint
even exists yet, or immediately after one is extracted but before federation (Part IV) has reached every
exposed project — how does the estate bound the damage in flight, rather than merely learning from it
after the fact? Containment is the dataset's most time-pressured mechanism, and it is specified
separately from quarantine (Part VII) because quarantine presupposes a completed exposure scan against
a stable fingerprint, while containment must act under partial, still-forming information.

11.2 The blast-radius object. A BLAST RADIUS, DEFINED, is the estimated set of ensemble entities — other
projects, shared artifacts (a common template, a shared prompt, a widely-reused code snippet), or active
sessions — that could be reached by an actively propagating pathogen given its CURRENT known propagation
mechanism, distinguished explicitly from the exposure map's (Part 6.6) STRUCTURAL, resting-state exposure
by being a live, time-bounded estimate computed at the moment of an active incident rather than a
standing property re-scanned periodically. The blast radius for a CONTAMINATED-PROMPT pathogen (Part
2.5) in particular can be computed with unusual precision, because a literally-copied template's
propagation path is traceable through the estate's own artifact-reuse records, whereas a
STRUCTURAL-EXPOSURE pathogen's (the other five classes) blast radius is necessarily a broader estimate
bounded by the exposure map's DERIVED and HYPOTHESIZED entries, not yet confirmed OBSERVED ones.

11.3 The containment action ladder, ordered by cost and reversibility. IAS-D2 specifies a four-rung
containment ladder, applied in ascending order only as lower rungs prove insufficient. RUNG ONE —
NOTIFY: publish an urgent, high-priority federation event (Part 4.3) ahead of the normal fingerprint-
extraction pipeline, carrying a provisional, unconfirmed mechanism description, so every subscribed
project's active sessions receive an early-warning signal even before a full fingerprint exists. RUNG
TWO — SUSPEND SHARED CARRIER: for a contaminated-prompt pathogen specifically, suspend the SPECIFIC
shared artifact (the template, the snippet, the prompt) from further automated reuse across the
federation, the cross-project generalization of Part 7.3's ROUTE-SUSPEND action. RUNG THREE — TARGETED
QUARANTINE: once a provisional fingerprint permits it, apply Part VII's quarantine mechanism against the
specific projects the blast-radius estimate names, ahead of a full exposure scan's confirmation, under
an explicitly PROVISIONAL evidence flag distinct from Part 7.2's normal OBSERVED/DERIVED gate. RUNG FOUR
— ESCALATE TO OWNER: for a blast radius estimated to exceed a severity-weighted threshold (EXPERIMENTAL:
three or more CRITICAL-severity-implicated projects), surface directly to Owner attention rather than
letting the automated ladder proceed further unsupervised, consistent with Article IX.

11.4 Why containment tolerates lower evidence standards than quarantine, and the compensating control
that makes this acceptable. Rungs one and two act on PROVISIONAL, not yet OBSERVED-or-DERIVED evidence
— a deliberate relaxation of Part 6.4's evidence hierarchy, justified only because rungs one and two are
themselves low-cost and highly reversible (a notification imposes no restriction; suspending one shared
artifact from further automated reuse blocks only NEW propagation, never existing, already-functioning
code). Rung three's relaxation to provisional evidence is the more consequential one, and it is bounded
by an explicit compensating control: any provisional-evidence quarantine MUST be re-evaluated against
Part 6.4's full evidence hierarchy within a short window (EXPERIMENTAL: 48 hours) and either confirmed
(converted to a standard Part VII quarantine) or LIFTED — a provisional quarantine that is neither
confirmed nor lifted within its window is a governance defect in its own right, tracked identically to
Part 7.5's stalled-quarantine escalation.

11.5 Containment's relationship to the boundary invariant. Containment, like every mechanism in this
dataset, is bound by IAS-F1's boundary invariant (Invariant Four): the blast-radius estimate and every
rung of the containment ladder are scoped to the estate's own projects and shared artifacts, never to
external dependencies, the model provider, or systems outside the estate's authority, even when a
propagating pathogen's mechanism happens to involve an external tool or service as an intermediate
carrier — containment acts on the ESTATE'S OWN exposure to that carrier, never on the carrier itself,
which the estate has no authority to quarantine.

11.6 The relationship between containment and the defense loop's timing. The Compounding Graph's defense
loop names the sequence "Control Bus carries failure events → Immune System fingerprints and federates
→ Cognitive Reliability turns recurring failures into SLOs → Digital Twin simulates the failure before it
recurs → Immune System immunizes preventively." Containment is the mechanism that operates in the GAP
this sequence's first two steps leave open — between "a failure event occurs" and "a fingerprint is
fully extracted and federated" — a gap the defense loop's own timing does not otherwise address, because
extraction (Part III) and federation (Part IV) are specified as careful, evidence-respecting processes
that take real time to execute correctly, time an actively propagating contaminated-prompt pathogen does
not necessarily grant the estate. (DERIVED: containment is not a redundant fifth stage bolted onto the
defense loop; it is the loop's necessary bridge across its own first transition, made explicit because
an active-incident response cannot wait for the deliberate pipeline to complete.)

11.7 Worked example: containing a contaminated-prompt outbreak in flight. A shared onboarding template
used across the estate's newer projects is discovered, mid-incident at Project Lambda, to carry an
instruction fragment that silently disables a safety confirmation step whenever the template is invoked
verbatim — a genuine contaminated-prompt pathogen (Part 2.5), still active in at least one other
project's recent session at the moment of discovery (confirmed by a quick check of the template's known
reuse log). RUNG ONE fires immediately: an urgent notification reaches every subscribed project. RUNG
TWO fires within the same response: the template itself is suspended from further automated reuse across
the federation, blocking any NEW session from picking it up, while existing deployments that already
copied it remain until individually addressed. The blast-radius estimate, computed from the template's
reuse log rather than a general structural-exposure guess, names exactly four projects that used the
template in the prior thirty days — a precise, traceable list rung three's provisional quarantine targets
directly, each confirmed or lifted within the 48-hour window as their individual exposure scans complete.
This worked trace demonstrates containment's core value proposition concretely: the four affected
projects receive a notification and a targeted, provisional response measured in the same session the
outbreak was discovered, not after the multi-stage fingerprint-and-federation pipeline (Parts III-IV)
completes its normal, more deliberate course.

11.8 A metric for containment discipline, and the countermetric that prevents it from being gamed.
CONTAINMENT LATENCY, DEFINED, is the elapsed time between a failure event's first appearance on the
Control Bus and Rung One's notification firing for the same lineage (OBSERVED, measurable directly from
the bus's own event log). A shrinking latency is the intended signal of a healthier defense loop, but
optimized alone it rewards firing Rung One reflexively on every failure event regardless of whether the
event is genuinely propagating — a system that notifies on every single-project incident achieves
near-zero latency by construction while flooding subscribers with undiscriminating alerts, the
alert-fatigue failure Part 14.3 already names for a sibling mechanism. The countermetric is
OVER-CONTAINMENT RATE, DEFINED as the proportion of Rung One or Rung Two actions whose blast-radius
estimate (11.2) is later confirmed, once federation (Part IV) completes, to have named zero additional
exposed projects beyond the origin. Falling latency paired with rising over-containment rate is not
improvement — it is the metric being gamed — and the two figures must be read together, never singly.

11.9 Interface to IAS-B1: ranking containment actions by systems-amplified. When a pathogen's propagation
touches several shared artifacts at once, the containment ladder (11.3) can face more than one viable
Rung Two suspension target, a prioritization problem IAS-D2's own scope — exposure to a pathogen, not
general intervention leverage — has no native way to resolve. IAS-B1's Cognitive Multiplier, whose
function is ranking candidate interventions by how many systems each amplifies, is CONSUMED (IAS-F1's
sixth relation) for exactly this sub-problem: candidate suspension targets are handed to IAS-B1 as
candidate interventions, and its systems-amplified scoring returns the ordering Rung Two applies, so the
estate suspends whichever shared artifact protects the most downstream systems first. (DERIVED: IAS-D2
supplies B1 a concrete, safety-relevant ranking problem it did not previously need to solve; B1 supplies
IAS-D2 a principled tie-break its own ladder lacks.)

FINAL LAW — PART XI. A pathogen actively propagating does not wait for a careful fingerprint before it
reaches its next host; containment exists precisely to act, cheaply and reversibly, in the gap between
discovery and full federation — and a containment action that cannot be confirmed or lifted within its
window is itself a governance defect the estate must see, not a quiet, indefinite restriction.

===============================================================================================
PART XII — PREVENTIVE SWEEP INTELLIGENCE: DECIDING WHEN A SWEEP IS AN IMMUNE RESPONSE
===============================================================================================

12.1 The distinct question this Part answers. sweep_enforcer, as an existing parent system, already
knows HOW to execute a scheduled governance sweep across a project or the estate. What it does not
decide, and what the Non-Duplication Ledger's candidate 49 (preventive sweep) names as the gap, is WHEN
a sweep should be triggered as a specifically PREVENTIVE, immunology-motivated action — distinct from
sweep_enforcer's routine, calendar-driven cadence — and WHAT that sweep should actually check for. IAS-D2
supplies the decision layer; sweep_enforcer remains the sole execution mechanism, per the header's
boundary and the Strengthening Matrix's row for this dataset.

12.2 Three triggers for a preventive sweep, each mapped to an upstream mechanism. TRIGGER ONE —
LINEAGE GENERALIZATION: whenever a pathogen lineage's structural precondition generalizes following a
confirmed mutation (Part 5.6), every project previously scored CLEAR against an EARLIER, narrower
version of that lineage's precondition is a candidate for a preventive re-scan, because Part 6.7 already
demonstrated concretely that a lineage's generalization can surface exposure a narrower scan missed.
TRIGGER TWO — HIGH-CONFIDENCE IMMUNE-MEMORY MATCH: whenever a new project onboards, or an existing
project's structure changes significantly (a new deployment path added, a new integration adopted), a
preventive sweep runs against every lineage with an existing, proven antibody (Part 9.2's lineage-indexed
view), because these are the highest-confidence, lowest-cost sweeps available — a proven remediation is
ready to propose immediately on a match, unlike a sweep against a lineage with no prior instantiation.
TRIGGER THREE — SEVERITY ESCALATION: whenever an existing fingerprint's SEVERITY field is revised upward
(a LOW-severity pathogen later found, through a subsequent incident elsewhere, to have caused CRITICAL
harm under a condition the original assessment did not anticipate), every project previously scanned and
scored CLEAR or left UNRESOLVED under the lower severity is re-prioritized for a sweep, because the
scan-priority function (Part 6.5) that originally deprioritized it was computed against a severity value
that has since changed.

12.3 The sweep proposal, not the sweep itself, is IAS-D2's output. Consistent with the header's boundary
and Part 7.6's pattern for cascade_prevention, IAS-D2's preventive-sweep-intelligence subsystem produces
a SWEEP REQUEST — naming the trigger, the target lineage or lineages, the candidate project set, and the
priority ordering — and hands it to sweep_enforcer for actual execution, scheduling, and resource
management, exactly as sweep_enforcer already handles its own routine sweep classes. (DERIVED: this is
the concrete mechanism behind the Strengthening Matrix's claim that IAS-D2 strengthens sweep_enforcer —
sweep_enforcer gains a new, well-specified class of sweep request it did not previously know how to
generate on its own, without IAS-D2 duplicating sweep_enforcer's scheduling or execution logic.)

12.4 Distinguishing a preventive sweep from routine housekeeping, operationally. sweep_enforcer's
existing routine sweeps typically check for drift against a static ruleset on a fixed calendar cadence.
A preventive sweep, by contrast, is EVENT-TRIGGERED (by one of 12.2's three triggers, never by a
calendar alone), TARGETED (scoped to the specific lineage(s) the triggering event implicates, never a
full-registry check unless Trigger Two's onboarding case genuinely requires one), and CARRIES A REASON
(the sweep request names which trigger fired and why, so the resulting findings, when reviewed, are
legible as "this sweep exists because lineage F-2201 generalized on this date" rather than appearing as
an unexplained routine artifact). (NORMATIVE: the reason field is what lets Part XV's metrics attribute a
sweep's findings back to the specific immune-system trigger that caused it, which is necessary for
evaluating whether preventive sweeps are earning their cost, Part XVI.)

12.5 Cost discipline: preventive sweeps compete for the same scan capacity Part 6.5 already bounds.
Preventive sweep requests are not exempt from the scan-priority function; they enter the same queue
Part 6.5 governs, ranked by the same severity-first, lineage-size-second ordering, with Trigger Three's
severity-escalation sweeps typically ranking highest because a revised-upward severity is, by
definition, newly urgent information the estate did not previously have. (DERIVED: unifying the queue
prevents preventive sweeps from silently starving routine or reactive exposure scans of capacity, a
specific instance of the resource-competition discipline IAS-F1's competes-with relation type exists to
make visible rather than let run unaccounted.)

12.6 Worked example: Trigger One firing across the estate following Epsilon's mutation confirmation.
Immediately upon Part 5.7's confirmation that Epsilon's migration-runner incident is a genuine mutation
of the F-2201 lineage, generalizing its structural precondition from "deploy step" to "any automated
action step emitting an unGated completion signal," Trigger One fires a sweep request naming every
project previously scored CLEAR against the lineage's PRE-generalization precondition — which, per Part
6.7's own worked trace, is exactly how Beta's previously-unnoticed backup-step exposure was actually
found. This closes the loop between Part V's mutation mechanism and Part VI's scanning mechanism
explicitly through Part XII's triggering logic: the mutation confirmation does not merely update a
registry record passively, it actively re-opens the scanning question for every project the
generalization newly implicates, which is the specific, checkable mechanism (a mutation event producing
a sweep request producing a re-scan producing a new confirmed exposure) that justifies calling this
sweep intelligence "preventive" rather than merely reactive.

12.7 The failure mode unique to preventive sweeping: trigger fatigue. A federation with a large,
actively-mutating registry can generate preventive sweep requests fast enough to overwhelm sweep_
enforcer's execution capacity or an Owner's review attention, at which point the mechanism's own success
(catching more lineage generalizations) produces a new form of institutional cost (sweep-request volume)
that must itself be governed. IAS-D2 specifies that trigger fatigue is monitored as a named quantity
(Part XV) and that, if sustained, the response is to raise the SEVERITY threshold at which Trigger Three
fires and to batch Trigger One requests across multiple same-day lineage generalizations into a single
combined sweep request rather than firing one per event — a rate-limiting discipline applied to the
triggering logic itself, never to the underlying detection quality.

12.8 The relationship between preventive sweeps and decay monitoring, made explicit ahead of Part XIII.
Trigger One (12.2) fires from a mutation-driven lineage generalization; a fourth, related trigger belongs
here in the interest of completeness even though its full mechanism is specified only in Part XIII: a
CONFIRMED DECAY EVENT (Part 13.5) against one antibody in a lineage is itself grounds for a preventive
sweep request against every OTHER project holding an antibody in the same lineage, on the reasoning that
whatever caused one project's antibody to decay — an infrastructure migration pattern, an unrelated
refactor convention, a shared tooling upgrade rolled out estate-wide — may plausibly have caused, or be
about to cause, the identical decay at sibling projects that share the same underlying tooling. (DERIVED:
this is the specific bridge Part 13.6's Alpha worked example gestures toward when it notes the migration-
induced decay is "itself published as a signal" — the mechanism that consumes that signal and turns it
into a sweep request IS this fourth trigger, named here as TRIGGER FOUR — DECAY PROPAGATION, completing
the set Part 12.2 opened with three.) Trigger Four enters the same unified scan-priority queue (Part 6.5,
Part 12.5) as its three siblings, and is deliberately NOT given elevated priority over Trigger Three's
severity-escalation sweeps by default, because a decay event, while informative, is evidence of a
PAST protection lapsing rather than a NEWLY discovered severity, and the priority ordering already
established privileges newly-discovered danger over previously-known-but-now-possibly-lapsed protection,
a distinction this dataset holds deliberately rather than by oversight.

FINAL LAW — PART XII. A sweep executed on a calendar checks for drift; a sweep triggered by a lineage's
generalization, a proven antibody's new relevance, a severity revision, or a sibling project's confirmed
decay checks for a SPECIFIC, newly understood exposure the estate did not know to look for yesterday —
and the difference between the two is not effort, it is timing married to a reason.

===============================================================================================
PART XIII — IMMUNITY DECAY MONITORING: PROTECTION IS NOT PERMANENT
===============================================================================================

13.1 The decay problem, stated against the temptation to treat immunity as a one-time achievement. Every
mechanism through Part XII converges on a single moment — an antibody confirmed CLEAR, a project marked
IMMUNE — that reads, if taken naively, as a terminal state: the problem is solved, move on. IAS-D2
specifies explicitly that this reading is wrong, because three independent processes erode a
confirmed antibody's validity over time without any single one of them announcing itself as a
regression: the protected CODE can change (9.3's refactor case), the PATHOGEN can mutate past the
antibody's original scope (Part V, discovered only when a NEW incident occurs despite the antibody's
presence), and the ESTATE'S OWN UNDERSTANDING of the pathogen's mechanism can improve, revealing that
the original antibody's scope was narrower than the threat it was meant to close even without any code
or pathogen change at all. IMMUNITY DECAY, DEFINED, is the general name for any of these three
processes reducing an antibody's actual protective coverage below what its CONFIRMATION status still
claims.

13.2 Why decay cannot be caught by re-running the original confirmation check alone. The confirmation
step (Part 8.2) that originally validated an antibody tests the SAME exposure question the antibody was
built to answer, using the SAME structural-precondition test — and a test that already passed once will
continue to report CLEAR indefinitely unless something makes the underlying artifact re-readable and
re-evaluated, which nothing in the confirmation step itself schedules to happen again automatically.
(DERIVED: this is the same underlying limitation Part IV §4.3 of IAS-F1 names for any property record —
staleness is a property of NOT being re-checked, and a check that is never re-run cannot, by
construction, ever notice its own obsolescence.) Decay monitoring is therefore specified as a distinct,
scheduled process, not a side effect of any other mechanism in this dataset.

13.3 The three decay signals, one per erosion process. CODE-DRIFT SIGNAL: a periodic, sampled
re-verification (not a full re-scan of every antibody on every cycle, which Part 6.5's cost discipline
would forbid at scale, but a rotating sample weighted toward higher-severity, longer-standing antibodies)
that re-reads the protected artifact and confirms the gating mechanism the antibody recorded is still
present and still wired to the path it was meant to gate. PATHOGEN-MUTATION SIGNAL: any NEW incident,
anywhere in the federation, whose fingerprint's mutation-distance test (Part 5.3) scores LOW against a
lineage that already has one or more antibodies deployed is itself evidence that at least one deployed
antibody's scope did not anticipate this mutation — every such incident automatically raises a decay flag
against every antibody in the affected lineage, prompting a targeted re-check of whether each already-
protected project is ALSO exposed to the new mutation's specific surface. UNDERSTANDING-REVISION SIGNAL:
when Part X's autoimmune review (10.6) or a direct Owner correction narrows or reshapes a fingerprint's
STRUCTURAL PRECONDITION or MECHANISM SIGNATURE, every antibody built against the PRIOR, less precise
version of that field is flagged for re-validation against the corrected version, because a correction
made to fix a false positive elsewhere may equally reveal a false negative in an antibody presumed
sufficient.

13.4 The immunity half-life concept and its epistemic status. IAS-D2 introduces IMMUNITY HALF-LIFE as a
per-lineage estimate of how long, on average, an antibody in that lineage remains fully valid before one
of the three decay signals is expected to fire against it — a quantity explicitly marked HYPOTHESIZED
and EXPERIMENTAL for any lineage with fewer than a minimum sample of decay events (provisionally three),
because a half-life estimated from zero or one observed decay event is not yet a measured rate, it is a
guess wearing a rate's clothing. (NORMATIVE: the half-life concept exists to let Part XV's monitoring
distinguish a lineage that decays fast — meaning its antibodies need frequent re-verification priority —
from one that has proven durably stable, so decay-monitoring resources are allocated by evidence rather
than spread uniformly thin across every lineage regardless of its actual volatility.)

13.5 Response to a confirmed decay event: re-entering the immunization pipeline, not restarting from
zero. When a decay signal is confirmed (the code-drift check finds the gate genuinely removed, the
mutation signal finds a genuinely uncovered surface, or the understanding-revision signal finds a
genuinely narrower-than-needed antibody), the affected (project, lineage) pair does not revert to
UNKNOWN status and start over — it re-enters Part VIII's immunization pipeline at the PROPOSAL stage,
carrying its full provenance history (the original antibody record, the specific decay signal that
fired, and the delta between what was protected and what needs to be protected now), so the re-
immunization proposal is informed by everything the estate already learned rather than repeating the
original diagnostic work. (DERIVED: this is the operational reason 9.5's provenance chain is
specified as unbroken — decay response is the mechanism that most directly depends on that chain being
walkable.)

13.6 Worked example: a code-drift decay event in Alpha's original antibody. Eighteen months after
Alpha's original deploy-path antibody (Part 3.7's incident, remediated and confirmed) deployed, Alpha
undergoes an unrelated infrastructure migration that replaces its entire deploy tooling. The migration
was reviewed for its own stated purpose and passed its own checks, but the reviewer, unaware the old
deploy script's post-action health check was ALSO serving as F-2201's antibody, did not carry the
gating logic forward into the new tooling — a genuine, unintentional CODE-DRIFT decay event. A routine
decay-monitoring sample (13.3), weighted toward this antibody for its age and the lineage's HIGH
severity, re-reads Alpha's current deploy path six weeks after the migration and finds the gate absent.
Alpha's status reverts from IMMUNE to EXPOSED for F-2201's lineage, a re-immunization proposal is
drafted carrying the original antibody's remediation class plus a note naming the migration as the
decay's specific cause, and — critically — this decay event is ALSO published as a signal (per 4.3's
event mechanism) because an infrastructure-migration-drops-embedded-governance pattern is itself a
candidate new fingerprint under Part II's ineffective-rule or stale-knowledge classes, potentially
protecting every OTHER project against the same category of migration-induced antibody loss, not merely
re-fixing Alpha alone.

13.7 Decay monitoring's own cost discipline and the sampling trade-off. Decay monitoring, like exposure
scanning, competes for finite scan capacity, and IAS-D2 specifies that its sampling rate is itself
tunable against Part XV's observed decay rate: a lineage with a long-observed half-life (13.4) is
sampled less frequently, freeing capacity toward lineages with shorter observed half-lives or higher
severity, a self-adjusting allocation the dataset marks EXPERIMENTAL pending enough estate history to
validate that the adjustment actually improves overall detection rate rather than merely reallocating
blind spots from one lineage to another.

13.8 The relationship between decay monitoring and false immunity. Part XIV names FALSE IMMUNITY as a
first-class failure mode of the whole system — a project believed protected that is not. Decay
monitoring is the dataset's primary DEFENSE against false immunity becoming permanent and undetected:
every decay signal is, definitionally, an instance of false immunity CAUGHT before it causes a real
recurrence, and the gap between decay monitoring's actual sampling coverage and full, continuous
verification of every antibody at every moment is the RESIDUAL false-immunity risk the dataset can never
fully close, only bound and shrink through better sampling and faster signal response — a limitation
this dataset states honestly rather than implying decay monitoring, once specified, eliminates the risk
outright.

FINAL LAW — PART XIII. A confirmed antibody is a claim about the past, not a guarantee about the
present; immunity decays through code drift, pathogen mutation, and improved understanding alike, and a
federation that never re-checks its own confirmed protections is accumulating false immunity it cannot
yet see.

===============================================================================================
PART XIV — FAILURE MODES OF THE IMMUNE SYSTEM ITSELF
===============================================================================================

14.1 Why this Part exists, and why it is not redundant with the failure discussions already embedded in
Parts III through XIII. Every prior Part named its own local failure risk where relevant — over-broad
signatures in Part III, false mutation confirmation in Part V, autoimmune events in Part X, decay in
Part XIII. This Part exists to assemble those local risks into a single, deliberately adversarial
catalogue, examined together rather than scattered, because a reader — or an implementer — who
encounters each risk once, in passing, inside the Part that introduces its corresponding mechanism, may
never see the PATTERN that connects them: every one of IAS-D2's core mechanisms has a corresponding
over-eager and under-eager failure pole, and the dataset is only honestly specified once both poles of
every mechanism are named side by side.

14.2 FALSE IMMUNITY — the under-eager pole of the whole system. Already introduced structurally in Part
1.6 and Part 13.8, FALSE IMMUNITY is the condition of a (project, pathogen) pair recorded as IMMUNE when
the actual protective coverage has lapsed or was never as broad as recorded. Its three specific causes
are the three decay signals of Part 13.3, but false immunity can also arise at ORIGIN, independent of
decay, if an immunization proposal (Part 8.2) is confirmed CLEAR by a re-scan that itself suffers a
false negative — the re-scan reads the deployed gate's PRESENCE but not its actual EFFECTIVENESS under
every real invocation path, a gap that is structurally identical to Part II's proxy-evidence pathogen
class applied reflexively to the immune system's own confirmation step. (CONTESTED: whether confirmation
should require a live, adversarial test of the deployed gate — attempting to trigger the pathogen's
mechanism and confirming the gate actually intercepts it — versus a structural presence check alone is
an open design question this dataset does not resolve; Part XXIV names it explicitly as an unresolved
governance question for future refinement.)

14.3 OVER-QUARANTINE — the over-eager pole of the containment and quarantine mechanisms. OVER-QUARANTINE
is the condition of a quarantine or containment action (Parts VII, XI) whose scope exceeds the narrowest
restriction the confirmed exposure actually requires — either because the fingerprint's structural
precondition was drawn too broadly (a Part III authoring defect), because containment's provisional
evidence (Part 11.4) over-estimated the blast radius, or because a quarantine action, once deployed, was
never re-narrowed as more precise information became available during its review window (Part 7.5).
Over-quarantine's harm is not merely inconvenience; a federation whose immune system repeatedly
over-restricts legitimate work trains its own Owners and agents to distrust and route around EVERY
future quarantine, including the correctly-scoped ones — the specific institutional cost that makes
over-quarantine a more corrosive long-run failure than an occasional missed exposure, because missed
exposures degrade the estate's protection while over-quarantine degrades the estate's WILLINGNESS to be
protected at all.

14.4 SIGNATURE COLLISION — a failure mode distinct from both false immunity and over-quarantine. A
SIGNATURE COLLISION occurs when two genuinely DIFFERENT pathogens, with different root causes and
different correct remediations, produce mechanism signatures (Part 3.3) similar enough to be treated as
one lineage by the deduplication (3.5) or mutation-confirmation (5.4-5.5) logic. Unlike over-quarantine
(too broad a restriction on one real pathogen) or false immunity (too narrow a protection on one real
pathogen), signature collision corrupts the REGISTRY'S OWN TAXONOMY — an antibody built for the
COLLIDED, merged lineage may correctly address one of the two underlying pathogens while leaving the
other entirely unaddressed under the false confidence of a shared confirmed-CLEAR status. Part 5.5's
remediation-class confirming check is the primary defense against this specific failure mode, and its
absence — treating mechanism-signature similarity alone as sufficient for lineage merger — is precisely
what Part 5.7's Zeta worked example was constructed to illustrate failing correctly.

14.5 IMPERIAL BOUNDARY CREEP — the failure mode of authority exceeding the ensemble boundary. Named
already at the ontology level by IAS-F1 Part V §5.5 and Part 8.6, this dataset's specific instance is an
immune-system mechanism — most plausibly containment's blast-radius estimation (Part XI) or a
preventive-sweep request (Part XII) — that names an entity outside the estate's boundary (a third-party
dependency, the model provider, a non-estate repository) as a quarantine or sweep target, on the
reasoning that "the pathogen touches it too." IAS-D2 forbids this explicitly: the correct response to a
pathogen mechanism that involves an external carrier is to quarantine or sweep the ESTATE'S OWN
exposure TO that carrier (per 11.5), never to attempt any action against the carrier itself, which the
estate holds no authority over and for which IAS-D2's own quarantine or sweep mechanisms have no
legitimate jurisdiction to act.

14.6 REGISTRY POLLUTION — the failure mode of Part 3.8's boundary against over-fingerprinting, realized
at scale. If the incidental-versus-structural discipline of 3.8 is not honestly applied, the pathogen
registry accumulates a large volume of low-value, single-occurrence, no-structural-precondition entries
that dilute every downstream mechanism's signal: exposure scans waste capacity testing preconditions
that will never recur, decay monitoring samples against antibodies protecting non-recurring incidents,
and the scan-priority function (6.5) is forced to work harder to separate genuine, structurally-recurring
threats from registry noise. Registry pollution is a slow-onset failure mode — no single polluting entry
causes visible harm — and is precisely why Part XV specifies a registry SIGNAL DENSITY metric rather
than relying on registry SIZE alone as an implicit health indicator.

14.7 RECURSIVE INEFFECTIVE RULE — the immune system diagnosing itself. Part 10.6 already named this
explicitly: an autoimmune-detection mechanism, a decay-monitoring sampler, or a mutation-confirmation
check that is ITSELF an ineffective rule (Part 2.4's taxonomy applied reflexively) — correctly triggered,
but too narrow to catch the harm it exists to catch. This is the dataset's own worst-case failure mode,
because a defect here does not merely miss one pathogen, it degrades the estate's confidence in the
entire immune system's verdicts, and it is why Part XVI's evaluation protocol treats the immune system's
own mechanisms as subject to the same evidence-and-recheck discipline as any first-order pathogen —
IAS-D2 does not exempt itself from the taxonomy it built.

14.8 Reading the eight named failure modes as pairs, and the discipline that follows. Six of the eight
failure modes named across this Part and its cross-references (false immunity / over-quarantine,
signature collision's two sub-failures, imperial boundary creep, registry pollution, recursive
ineffective rule) resolve into a recognizable structure: for nearly every mechanism this dataset
specifies, there is an under-eager failure (missing a real pathogen) and an over-eager failure (acting
against a non-pathogen), and the correct discipline, stated once here for the whole dataset, is never to
tune a mechanism's threshold toward one pole without measuring its effect on the other — Part XV's
paired metrics exist expressly to make this trade-off visible rather than letting an implementer quietly
optimize false-immunity risk downward while over-quarantine risk rises unmeasured, or vice versa.

FINAL LAW — PART XIV. Every mechanism in this dataset that can correctly catch a real pathogen can, by
the identical construction, incorrectly catch a non-pathogen or miss a mutated one; an immune system
audited only for the pathogens it caught, never for the legitimate work it blocked or the mutations it
missed, is grading itself on half the ledger.

===============================================================================================
PART XV — METRICS AND ANTI-GOODHART COUNTERMETRICS
===============================================================================================

15.1 Why every metric in this Part is issued in a pair. Part XIV established that nearly every
mechanism has two failure poles. A metrics scheme that measures only one pole is not neutral — it is an
active invitation to optimize toward the unmeasured pole, the precise Goodhart dynamic the estate's own
governance already names as a recurring risk across its scoring systems (IAS-F2's non-inflation
property, Part 1.2). IAS-D2 therefore issues no metric without a paired countermetric measuring the
opposite failure pole, and no metric in this Part is offered as a target to be maximized in isolation.

15.2 CROSS-PROJECT IMMUNIZATION RATE (CPIR) and its countermetric AUTOIMMUNE FALSE-POSITIVE RATE. CPIR,
DEFINED, is the proportion of confirmed-EXPOSED (project, fingerprint) pairs that reach CONFIRMED
antibody status within a target window, per severity class. A rising CPIR is, on its face, evidence the
immunization pipeline (Part VIII) is functioning. Its countermetric, the AUTOIMMUNE FALSE-POSITIVE RATE
(Part X's confirmed autoimmune events divided by total quarantine/immunization actions over the same
window), must be read alongside CPIR because a naive way to raise CPIR is to loosen the exposure gate
(Part 6.4) so more (project, fingerprint) pairs qualify as EXPOSED in the first place — which would also
raise the autoimmune rate in lockstep, and the pairing makes that trade-off visible rather than letting
CPIR be reported as an unqualified success metric.

15.3 MEAN TIME TO FEDERATE (MTTF) and its countermetric FEDERATION PRECISION. MTTF, DEFINED, is the
median elapsed time between a fingerprint's extraction (Part III) and its availability at every
subscribed project (Part IV), the direct measure of 4.7's federation-latency concern. Its countermetric,
FEDERATION PRECISION, is the proportion of federated fingerprints that go on to match a CONFIRMED
exposure somewhere in the estate within a bounded window versus those that never match anything —
because the cheapest way to drive MTTF toward zero is to federate every raw incident immediately without
the extraction discipline of Part 3.8, which would also drive federation precision toward the registry-
pollution failure mode of Part 14.6, and the pairing prevents MTTF from being optimized at precision's
expense.

15.4 IMMUNITY HALF-LIFE (Part 13.4) and its countermetric DECAY-DETECTION LATENCY. Half-life measures
how long protection tends to last; its countermetric measures how long, on average, elapses between an
actual decay event occurring (the code changing, the mutation occurring) and the corresponding decay
signal (Part 13.3) firing — because a lineage could show an apparently long half-life not because its
antibodies are genuinely durable but because the decay-monitoring sampler is simply too infrequent to
have caught the decay yet, a specific instance of the "zero cannot fall" measurement trap the estate's
own governance history already names: an unmeasured decay reads as no decay, and no decay never falls.

15.5 OUTBREAK CONTAINMENT RATIO and its countermetric CONTAINMENT OVER-TRIGGER RATE. The containment
ratio, DEFINED, is the proportion of Part XI's blast-radius estimates that, upon later confirmation
(Part 6.4's full evidence hierarchy applied post-hoc), are found to have correctly identified the
actually-exposed project set, versus estimates that were substantially too narrow (missed a real
exposure) or too broad (Part 14.3's over-quarantine risk realized at the containment stage). The over-
trigger rate specifically isolates the too-broad direction, tracked separately from the too-narrow
direction, because a containment mechanism tuned purely to avoid ever missing an exposure (driving the
too-narrow failure toward zero) will mechanically drive the over-trigger rate up, and reporting only the
combined "containment ratio" would hide which direction any given tuning change actually moved.

15.6 REGISTRY SIGNAL DENSITY as the direct countermetric to unconstrained registry growth. Rather than
pairing registry size with a second metric, IAS-D2 specifies SIGNAL DENSITY directly as the proportion
of registry entries that are part of a lineage with two or more confirmed origins (Part 3.5) OR at least
one deployed antibody (Part 8.3) — entries that have EARNED their place in the registry through either
independent rediscovery or actual remediation — against the total registry size. A declining signal
density over time is the direct, quantitative symptom of Part 14.6's registry pollution failure mode,
and is monitored as a standing health indicator rather than only investigated reactively after pollution
is suspected.

15.7 SWEEP-REQUEST VOLUME as the direct countermetric to preventive-sweep coverage. Paired against no
single "coverage" metric (preventive sweep coverage is better read through CPIR and MTTF combined),
sweep-request volume per period is tracked on its own as the direct measure of Part 12.7's trigger-
fatigue risk, with a rising trend, independent of any single trigger's correctness, treated as sufficient
grounds to invoke the batching and threshold-raising response 12.7 already specifies, because volume
itself — regardless of whether each individual request was well-justified — is the resource constraint
that determines whether sweep_enforcer and Owner review capacity remain functional.

15.8 The prohibition on any single composite immune-health score. Given the pairing discipline of
15.1-15.7, IAS-D2 explicitly declines to define a single composite "immune health score" combining CPIR,
MTTF, half-life, containment ratio, and signal density into one number, even though IAS-F2's
Institutional Advantage Unit exists as a general-purpose composite for exactly this kind of aggregation
elsewhere in the family. (NORMATIVE, and explicitly CONTESTED against the general utility of composite
scoring: a composite here would necessarily weight several paired metrics whose correct trade-off point
is a genuine, context-dependent governance judgment — how much autoimmune risk is acceptable to buy how
much CPIR, for a given severity class — and collapsing that judgment into one number would hide the
exact trade-off Part XIV exists to keep visible; each metric pair is reported and reviewed jointly
instead, an explicit design choice this dataset defends over the more conventional single-score
approach.)

15.9 Reporting cadence and audience-specific views over one shared metric substrate. The paired metrics of
15.2 through 15.7 are computed once, against the shared registry, exposure map, and immune-memory store —
but three distinct audiences read them at three distinct grains, and specifying those views prevents a
future implementer from either flooding an Owner with per-lineage detail or starving sweep_enforcer of the
operational detail it actually needs. An OWNER-FACING VIEW aggregates by severity class and reports
trend direction (is CPIR rising alongside a stable autoimmune rate, or rising because the autoimmune rate
is quietly rising too) rather than per-lineage figures, consistent with 15.8's rejection of a single
composite score while still respecting an Owner's limited review bandwidth. A SWEEP_ENFORCER-FACING VIEW
reports sweep-request volume (15.7) and containment over-trigger rate (15.5) at the granularity
sweep_enforcer's own capacity-planning needs — per period, per trigger class — because that system's
own scheduling logic, not IAS-D2's, consumes this view directly. An IAS-E1-FACING VIEW exposes the full,
un-aggregated registry and exposure-map state, because Observability Fabric's own object (Part 9.4) is
the estate's usage and health picture writ large, and IAS-E1 is the one consumer positioned to fold
IAS-D2's detail into a broader cross-system report rather than needing it pre-summarized. (NORMATIVE:
one substrate, three views, no view computed by a separate methodology — the specific discipline that
keeps 15.1's pairing requirement intact across every audience, since a view that silently dropped a
countermetric for brevity would reintroduce exactly the Goodhart risk the pairing exists to prevent.)

FINAL LAW — PART XV. No metric in this dataset is reported alone; every measure of how well the immune
system catches real pathogens is paired with a measure of how often it catches something that was never
a pathogen at all, because an immune system graded on only one side of that ledger will learn to win the
grade instead of the war.

PART XVI — EVALUATION PROTOCOL: HOW TO PROVE THE IMMUNE SYSTEM WORKS
===============================================================================================

16.1 Why an evaluation protocol is a separate Part from the metrics of Part XV. Part XV defines WHAT is
measured. This Part defines HOW those measurements are turned into a defensible claim that the system,
as specified, actually delivers Part 1.7's founding claim — because a metric that is well-defined but
never actually computed against real or realistic data is not evidence, it is a specification of what
evidence would look like if collected, and the Constitution's Article XI (only SEALED counts) requires
this dataset to name the concrete evaluation path rather than merely asserting the mechanisms would
work if run.

16.2 The retrodiction protocol. The primary evaluation method IAS-D2 specifies is RETRODICTION: given
the estate's own historical incident record (hard-rule seals, postmortems, sealed session lessons — all
OBSERVED artifacts already existing prior to this dataset), extract fingerprints (Part III) from a
sample of PAST incidents using the schema specified here, and test whether the resulting fingerprints,
had they existed at the time, would have correctly flagged OTHER, independently-occurring past incidents
in the estate's own history as exposure matches or confirmed mutations — incidents that, at the time,
were treated as unrelated because no federation mechanism connected them. (DERIVED, generalizing IAS-F1
Part VIII §8.7's retrodiction protocol from the level-test domain to the fingerprint-matching domain; a
successful retrodiction — finding two or more historical incidents that the fingerprint schema would
correctly have linked, where the estate in fact paid the cost of both independently — is the single
strongest form of evidence available for the founding claim, because it requires no new incident to
occur, only a correct, honest re-reading of incidents that already did.)

16.3 The synthetic stress-test protocol. Where retrodiction is unavailable (a mechanism, such as
containment's blast-radius ladder, that the estate's historical record may not contain a clean past
instance of), IAS-D2 specifies constructing SYNTHETIC, clearly-labeled-illustrative incident sequences —
exactly the kind used throughout Parts II through XIV's worked examples — run through the full pipeline
(extraction, federation, exposure scan, quarantine, immunization, decay) end to end, with the SAME
adversarial rigor a real audit would apply: deliberately including near-miss cases designed to trigger
signature collision (14.4), autoimmune events (Part X), and decay (Part XIII), and checking that the
specified mechanisms correctly resolve each rather than only checking the easy, unambiguous cases. (
EXPERIMENTAL: synthetic stress-testing is a necessary complement to retrodiction, not a substitute for
it, because a synthetic test can only validate that the SPECIFIED mechanism behaves as specified — it
cannot validate that the specification itself matches the real distribution of pathogen shapes the
estate will actually encounter, which only retrodiction against real history, or eventual live operation,
can confirm.)

16.4 The confirmation-triad requirement for any claimed immunization. Mirroring the estate's own Code
Review Doctrine's Proof Triad for HIGH/CRITICAL findings, IAS-D2 specifies that no (project, lineage)
pair may be reported as IMMUNE in any summary, dashboard, or Owner-facing claim without three
co-present pieces of evidence: the ANTIBODY RECORD (Part 8.3, what was deployed), the CONFIRMATION SCAN
(Part 8.2's final stage, that the deployment actually cleared re-scan), and the PROVENANCE CHAIN (Part
9.5, traceable back to a real originating incident or a real synthetic stress-test entry, never an
unattributed assertion). A claim missing any leg of the triad is downgraded in any report to "proposed"
or "pending," never presented as achieved immunity — the direct, dataset-specific application of
Article V's epistemic-honesty requirement to this dataset's central deliverable.

16.5 Evaluating the mutation-detection weights and thresholds specifically. Part 5.3 marked its slot-
weighting scheme, and Part 5.4 marked its distance threshold, EXPERIMENTAL, pending accumulated
multi-mutation history. The evaluation protocol names the specific update path: as each new confirmed
mutation (via the remediation-class check of 5.5) is added to a lineage, the weighting and threshold are
re-fit against the growing set of confirmed-mutation and confirmed-non-mutation (Part 5.7's Zeta-type)
examples, treating the accumulating registry itself as the estate's own evolving evaluation dataset for
this specific sub-mechanism — an explicit instance of the dataset governing its own calibration through
evidence rather than fixing the weights permanently at authoring time.

16.6 Benchmarking against the O(N)-versus-O(1) founding claim directly. Part 1.7's founding claim — a
root cause costs the estate one full incident, not one per exposed project — is directly testable, over
time, by comparing, for each confirmed pathogen lineage, the number of projects that suffered an
INDEPENDENT incident (a genuine recurrence the estate paid for) against the number that received
IMMUNIZATION before any incident occurred (Part 8.7's Theta case, the strongest form of the claim). A
lineage with a high independent-incident-to-preemptive-immunization ratio is evidence the federation and
scanning pipeline (Parts IV, VI) is running too slowly relative to the pathogen's own real-world
recurrence rate — direct, actionable evidence pointing back at Part 4.7's MTTF metric and Part 8.5's
proximity-to-recurrence prioritization as the specific levers to adjust.

16.7 The evaluation protocol's own limit, stated honestly. No evaluation method specified here can prove
the immune system catches every future pathogen shape the estate will ever encounter — retrodiction is
bounded by the estate's actual past incident diversity, and synthetic stress-testing is bounded by the
adversarial imagination of whoever constructs the synthetic cases. (UNRESOLVED: the estate cannot know,
from inside this dataset alone, how large the space of pathogen mechanisms not yet represented in
either the historical record or the synthetic test suite actually is; Part XXII's novel-pathogen edge
case names the operational response to this limit — an UNMATCHED incident is never silently absorbed
into an existing lineage merely to avoid admitting the taxonomy's coverage gap, it is flagged as
genuinely novel and handled by the taxonomy-amendment path Part 2.8 already reserves.)

16.8 A third evaluation method: scheduled Owner-in-the-loop spot audits, distinct from both automated
protocols. Retrodiction (16.2) and synthetic stress-testing (16.3) are both mechanized, repeatable, and
therefore cheap to re-run — but both are also bounded by the same authorship the mechanisms themselves
were bounded by: retrodiction can only find what the estate's own past record contains, and synthetic
stress-tests can only find what their author thought to construct adversarially (Part 16.7's named
limit). IAS-D2 therefore specifies a third, deliberately non-automated evaluation method: a periodic
(EXPERIMENTAL default, quarterly) OWNER SPOT AUDIT, in which a small, random sample of CONFIRMED antibody
records — drawn independent of severity or lineage size, specifically to avoid the same prioritization
bias Part 6.5's scan-priority function otherwise legitimately applies everywhere else — is manually walked
through its full provenance chain (Part 9.5) by an Owner or Owner-delegated reviewer, checking not
whether the mechanism WOULD have worked as specified, but whether the reviewer, reading the incident,
the fingerprint, and the antibody with fresh eyes, would independently reach the same confirmation
verdict the pipeline reached automatically. (NORMATIVE: a spot audit's sample size is deliberately small
— it is not a substitute for retrodiction's or stress-testing's systematic coverage, it is a check against
the specific risk that BOTH systematic methods share the same blind spot, because both were authored by
the same process; an independent human read is the one evaluation source in this Part not subject to that
shared-blind-spot risk.) A spot audit finding — confirmed-correct, or a disagreement worth investigating —
is itself fed back into Part 16.5's calibration loop exactly as a retrodiction or stress-test finding
would be, closing this third method into the same evidence-accumulation discipline the other two already
follow.

FINAL LAW — PART XVI. A specified mechanism is a hypothesis about how failures propagate until it is run
against the estate's own real history, a genuinely adversarial synthetic stress test, or an independent
human's fresh read of a real record; retrodiction, synthetic stress-testing, Owner spot audits, and
honest calibration against accumulating evidence are what convert that hypothesis into a claim the estate
may actually act on.

PART XVII — INTEGRATION WITH IAS-A2: THE CONTROL BUS AS THE IMMUNE SYSTEM'S NERVOUS SYSTEM
===============================================================================================

17.1 Restating the defense loop's first edge with full mechanical precision. The Compounding Graph names
"IAS-A2 Control Bus carries failure events → IAS-D2 Immune System fingerprints and federates them
cross-project" as the defense loop's opening edge. Parts III and IV of this dataset already specified
the fingerprinting and federation halves of that edge in detail; this Part specifies the OTHER half —
what IAS-D2 requires FROM the Control Bus, and what it RETURNS to it — completely, because a
strengthening claim (per the System-Strengthening Matrix's own evidentiary requirement) must name the
concrete mechanism, not merely gesture at the relationship.

17.2 What IAS-D2 consumes from IAS-A2: the failure-event class. IAS-D2 requires the Control Bus to carry
a FAILURE EVENT class — a typed event, distinct from the general-purpose event types IAS-A2 already
carries for other IAS datasets, whose payload includes at minimum the originating parent detector's
identity (cascade_prevention, osa/CEPS, secret_firewall, hard_rules, refcheck), a reference to the
incident's evidence artifact, and a timestamp. (DERIVED: IAS-D2 does not require the Control Bus to
UNDERSTAND pathogen taxonomy or fingerprint schemas — that interpretation is entirely IAS-D2's own
responsibility, per IAS-F1's relation-typing discipline that a FEEDS edge carries raw data the receiving
system interprets, not pre-interpreted data the producing system must understand. This is precisely why
IAS-A2 is strengthened rather than duplicated: it gains one new, well-specified event class to transport,
without needing to absorb any of IAS-D2's own domain logic.)

17.3 What IAS-D2 publishes back to IAS-A2: the pathogen-fingerprint and containment-alert event classes.
Two event classes flow the opposite direction. The PATHOGEN-FINGERPRINT event (Part 4.3) carries a newly
extracted or newly generalized (Part 5.6) fingerprint to every subscribing project's exposure-scan
subsystem. The CONTAINMENT-ALERT event (Part 11.3's Rung One) carries the urgent, provisional
notification ahead of a full fingerprint, distinguished from the pathogen-fingerprint event class by its
explicitly PROVISIONAL evidence flag (matching 11.4's relaxed evidence standard) so that a subscribing
project's own governance can correctly treat a containment-alert with appropriately lower confidence
than a fully-fingerprinted event. (NORMATIVE: keeping these two event classes distinct, rather than
merging them into one generic "immune system says something" event, is what lets a subscribing project's
own logic apply Part 6.4's evidence hierarchy correctly at the point of receipt, rather than needing to
inspect the payload to infer which evidentiary standard applies.)

17.4 The consumes relation, applied to IAS-D2's own use of exposure-scan requests. When IAS-D2's
scan-priority function (Part 6.5) or preventive-sweep-intelligence (Part XII) needs an exposure scan
actually executed against a specific project, per the header's boundary IAS-D2 does not execute the scan
itself against a project it does not own — it CONSUMES (IAS-F1's sixth relation type) the receiving
project's own exposure-scan capability, via a request carried over the same Control Bus, transient by
IAS-F1 Part VI §6.7's definition of the consumes relation rather than a standing structural dependency.
This is the precise mechanism by which the header's boundary claim ("IAS-D2 does not re-implement
per-session detection") is kept true even where a scan must, mechanically, touch a specific project's own
files — the touching is delegated, requested via the bus, never performed by IAS-D2 reaching directly
into a project it does not administer.

17.5 Event ordering and the guarantee IAS-D2 requires from the bus. Because Part XI's containment ladder
depends on rung ordering (notify, then suspend, then quarantine, then escalate) and Part 13.5's decay
response depends on a decay signal being processed before a stale re-immunization proposal is drafted,
IAS-D2 requires the Control Bus to guarantee AT-LEAST-ONCE delivery with PER-LINEAGE ordering — events
concerning the SAME lineage are guaranteed to arrive at each subscriber in the order published, even if
events concerning DIFFERENT lineages may interleave arbitrarily. (NORMATIVE, a specific, checkable
requirement on IAS-A2's own design, offered here as the concrete evidence the Strengthening Matrix
requires: IAS-A2 is strengthened not merely by carrying more traffic, but by IAS-D2 naming a specific,
non-trivial ordering guarantee that makes IAS-A2's own architecture more rigorously specified than a
generic best-effort bus would be, a requirement IAS-A2's other consuming datasets — IAS-D2's own defense-
loop siblings — benefit from once it exists.)

17.6 Worked example: the full event trace for the F-2201 lineage across its lifetime, read through the
bus. Origin: cascade_prevention fires a deploy-gate block at Alpha, publishing a FAILURE EVENT. IAS-D2's
extraction subscriber consumes it, produces fingerprint F-2201, publishes a PATHOGEN-FINGERPRINT event.
Beta's exposure-scan subsystem, subscribed, consumes it, later publishes its own scan-result event (an
IAS-A2 event type owned by the scanning subsystem's own project-side implementation, outside IAS-D2's own
publication scope but read back by IAS-D2's exposure-map updater). Epsilon's mutation incident
independently triggers a second FAILURE EVENT, IAS-D2's mutation-detection subscriber consumes it
alongside the existing F-2201 fingerprint, confirms the mutation, publishes a LINEAGE-GENERALIZED event
(a variant of the pathogen-fingerprint class carrying the updated structural precondition), which
Part XII's Trigger One consumes to produce a sweep request. Every step in this trace is a publish or
consume operation over the Control Bus, and the trace itself — recoverable in full from the bus's own
event log, per 17.5's ordering guarantee — is the literal, checkable artifact the Strengthening Matrix's
evidence requirement for IAS-D2's row (a failure fingerprint matched across two or more projects) names.

17.7 Failure modes of the bus integration itself, and the fallback that bounds them. Every dependency
this dataset takes on IAS-A2 is a DEPENDS-ON relation in IAS-F1's typed sense (Part VI §6.3 of F1), not a
mere FEEDS relation, precisely because Part 4.3's active federation path genuinely cannot function at all
if the bus is unavailable — which is exactly why Part 4.4 specifies the durable registry as a co-equal,
independent channel rather than an optimization: a Control Bus outage, a dropped subscription, or a
violation of 17.5's ordering guarantee degrades federation from event-driven (low-MTTF) to
reconciliation-driven (bounded by the periodic full-registry sweep's own cadence) but never to zero —
the registry has no dependency on the bus's own availability. (DERIVED: this is the specific, concrete
reason 4.4's push/pull hybrid was specified as mandatory rather than optional — a bus-only design would
make IAS-D2's entire federation function DEPENDS-ON a single piece of shared infrastructure with no
fallback, exactly the fragility IAS-E2's Cognitive Reliability discipline, Part XVIII, would need to
budget against as a standing risk rather than treat as bounded by design.) A bus-outage event, once the
bus recovers, is itself worth fingerprinting under this dataset's own taxonomy if the outage pattern
recurs — an infrastructure dependency whose own failure propagates degraded (not broken) federation
performance is a textbook parasitic-complexity or ineffective-rule candidate (Part 2.4, 2.6) for the
estate's OWN infrastructure, illustrating that IAS-D2's taxonomy is reflexively applicable to its own
supporting infrastructure exactly as honestly as to any first-order project.

17.8 A failure mode specific to the ordering guarantee, and its detection signal. Section 17.5's
per-lineage ordering guarantee is a promise ABOUT the bus, not a promise IAS-D2 can itself enforce —
IAS-D2 is a consumer of that guarantee, not its implementer, and a bus-side defect (a partition, a replay
bug, a misconfigured consumer group) can violate it silently, delivering, for instance, a
LINEAGE-GENERALIZED event to a subscriber before the PATHOGEN-FINGERPRINT event it generalizes. OBSERVED
SIGNAL: a subscriber's exposure-scan subsystem receiving a generalization event that references a
fingerprint ID it has never itself received is the detection signature for exactly this violation — the
reference resolves to nothing locally, an anomaly distinguishable from ordinary event loss (which
produces a missing event, not a dangling reference) precisely because ordering violation and event loss
leave different, non-confusable traces in a subscriber's own local event log. The specified response is
not to attempt local reordering (a subscriber cannot reconstruct a guarantee the bus itself failed to
honor) but to treat the affected lineage as PROVISIONALLY UNRELIABLE, re-request the missing predecessor
event directly rather than trusting the generalization's own payload, and — because a single ordering
violation is exactly the kind of infrastructure-reflexive incident 17.7 already names — fingerprint the
violation itself if the pattern recurs across more than one lineage, feeding the estate's own bus
reliability back into IAS-D2's taxonomy exactly as an outage pattern would.

FINAL LAW — PART XVII. IAS-D2 does not carry its own events; it borrows the Control Bus's nervous system,
in exchange for handing that nervous system a concrete, well-specified event vocabulary and a non-trivial
ordering guarantee it did not previously need to provide — the exact shape a genuine strengthening
relation takes, bounded by a registry fallback so that dependency never becomes fragility.

PART XVIII — INTEGRATION WITH IAS-E2: FROM RECURRING FAILURE TO SLO AND FAILURE BUDGET
===============================================================================================

18.1 The defense loop's second edge, specified. "IAS-D2 Immune System fingerprints and federates them
cross-project → IAS-E2 Cognitive Reliability turns recurring failures into SLOs and failure budgets."
This Part specifies what, concretely, IAS-D2 hands to IAS-E2 to make that conversion possible, honoring
the Strengthening Matrix's requirement that IAS-D2's row name a checkable mechanism, not an aspiration.

18.2 What a recurring-failure signal actually consists of, handed to IAS-E2. IAS-D2 hands IAS-E2 three
specific artifacts per lineage, none of which IAS-E2 could construct on its own from a single project's
local telemetry: the LINEAGE'S RECURRENCE COUNT (how many independent, confirmed origins the lineage has
accumulated across the federation, Part 3.5), the LINEAGE'S SEVERITY (Part 3.2, informing how tight an
SLO the recurring failure justifies), and — the most operationally distinctive artifact — the EXPOSURE
MAP'S CURRENT STATE for that lineage (Part 6.6: which projects remain EXPOSED, which are IMMUNE, which
are UNRESOLVED). A project's individual failure history alone tells IAS-E2 how often THAT project has
suffered a given failure; only IAS-D2's cross-project view tells IAS-E2 whether a given project's failure
is part of a WIDER pattern the estate has already characterized, or a genuinely local anomaly — and that
distinction changes what SLO discipline is appropriate.

18.3 Why exposure state, not just recurrence count, determines the right failure budget. A lineage with
a high recurrence count but a shrinking EXPOSED set (most projects already immunized) justifies a
LOOSENING failure budget over time for that lineage — the pathogen is being actively closed out, and a
static, permanently-tight budget would over-penalize newly-onboarded or lower-priority projects for a
risk the estate is already retiring. Conversely, a lineage with a low recurrence count but a large,
static EXPOSED set (many projects vulnerable, none yet immunized, immunization queue backlogged per Part
8.5) justifies a TIGHTENING budget, because the low recurrence so far may simply reflect that the
vulnerable capability is rarely exercised yet, not that the risk is small. (DERIVED: this is the specific
reasoning that makes IAS-D2's exposure map more valuable to IAS-E2's SLO-setting than a bare incident
count would be — IAS-E2 sets a forward-looking budget, and a forward-looking budget needs a
forward-looking exposure signal, which only the immune system's own live map, not a historical tally
alone, supplies.)

18.4 Antibody coverage as a direct input to a project's own reliability posture. Per Part 9.4, a
project's aggregate antibody coverage — the proportion of its high-traffic paths that carry a confirmed,
current antibody against at least one known lineage — is a direct input IAS-E2 may weight into that
project's OWN failure-budget calculation independent of any specific lineage's SLO: a project with dense,
current antibody coverage has, by construction, already closed a meaningful share of the estate's known
recurring-failure surface, and IAS-E2's reliability model treats that coverage as a form of demonstrated,
evidence-backed resilience rather than an unverified claim, precisely because Part 16.4's confirmation-
triad discipline is what makes an antibody record trustworthy enough to feed a downstream reliability
calculation at all.

18.5 The graceful-degradation path immunization enables, versus the hard-failure path quarantine
temporarily accepts. IAS-E2's own object, per the System-Strengthening Matrix, includes "graceful
degradation instead of hard failure." IAS-D2 contributes to this specific goal in two distinct ways
mapped to two distinct Parts: Part VII's quarantine actions (ADVISORY-GATE, SIGNAL-DOWNGRADE specifically)
are themselves instances of graceful degradation — a vulnerable path is not hard-blocked, its output is
downgraded or gated, which is precisely the softer failure mode IAS-E2's own design goal names, applied
at the immune-system layer before IAS-E2's own reliability mechanisms would otherwise need to react to a
harder failure. Part VIII's completed immunizations, over time, shrink the set of paths that ever need a
degraded mode at all, converting IAS-E2's graceful-degradation problem into a smaller, better-bounded
one for the lineages the immune system has already closed out.

18.6 What IAS-D2 does not attempt to specify on IAS-E2's behalf. Consistent with the header's boundary
discipline applied here reflexively, IAS-D2 does not specify the actual SLO formula, the failure-budget
depletion mechanics, or the chaos-testing protocol IAS-E2 owns — those remain IAS-E2's own distinct
object, per the Non-Duplication Ledger's Family E disposition. IAS-D2's contribution stops at handing
IAS-E2 the three artifacts named in 18.2, well-specified and evidence-graded, and trusting IAS-E2's own
mechanism to convert them into reliability discipline; the reverse relationship — IAS-E2 informing
IAS-D2 — is not part of the defense loop's stated direction and is left, honestly, as an open
integration question rather than asserted without evidence.

18.7 Worked example: the F-2201 lineage's SLO trajectory across its lifecycle. At Part 3.7's origin,
F-2201 has recurrence count one, severity HIGH, and an exposure map with one project (Alpha) confirmed
remediated and an unknown number of others UNRESOLVED — IAS-E2, receiving this early state, has
insufficient information to set anything beyond a provisional, wide budget. By Part 8.7's stage
(Gamma and Theta immunized, Beta remediated, Epsilon's mutation confirmed and folded into the same
lineage), recurrence count is four (three independent origins plus one mutation-confirmed recurrence),
severity remains HIGH, and the exposure map shows a majority of scanned projects now IMMUNE — IAS-E2's
own mechanism, consuming this state, is positioned to set a materially tighter, better-evidenced SLO for
this specific lineage's failure class than it could have at origin, and to do so without ever needing to
independently rediscover the cross-project pattern IAS-D2's federation and scanning pipeline already
characterized in full.

18.8 Cascading budget effects along a DEPENDS-ON chain, and why IAS-D2's exposure map must be read at
the right granularity for this case. IAS-F1's DEPENDS-ON relation (Part VI §6.3 of F1) is stronger than
FEEDS precisely because it propagates failure rather than merely degrading it, and a lineage's exposure
map interacts with this distinction in a way IAS-E2's budget-setting needs surfaced explicitly: if
Project A DEPENDS-ON Project B, and B carries a CONFIRMED, current antibody against a lineage while A
remains EXPOSED to the same lineage through its own, separate code path, A's own failure budget cannot
be set as if B's protection extends to A — antibodies are project-scoped by construction (Part 8.8's
explicit limit), and IAS-E2 must read the exposure map PER PROJECT along the dependency chain, never
infer that a downstream dependency's immunization implies an upstream dependent's safety. Conversely, if
A DEPENDS-ON B and B is EXPOSED (not yet immunized), A's own failure budget for any capability that
routes through B should be tightened even if A's own code has no direct structural precondition for the
lineage at all, because B's exposure is, via the depends-on edge, a genuine risk to A's own reliability
that A's own exposure scan alone would never surface. (DERIVED: this is the specific, concrete reason
IAS-D2 hands IAS-E2 the full exposure map rather than a per-project summary — a summary collapsed to
"this project's own status" would lose exactly the dependency-chain information 18.8 shows IAS-E2 needs
to set a genuinely accurate budget for a project whose OWN code is clean but whose dependencies are not.)

FINAL LAW — PART XVIII. A reliability discipline set from one project's own history is guessing at a
pattern it cannot fully see; IAS-D2 hands IAS-E2 the federation's own exposure map, evidence-graded,
lineage-scoped, and legible along dependency chains, so that the failure budget IAS-E2 sets reflects what
the estate actually knows about a recurring failure's true, cross-project shape — not merely what one
project happened to suffer, and not merely what one project's own code, in isolation, appears to show.

PART XIX — INTEGRATION WITH IAS-F3: SIMULATING THE PATHOGEN BEFORE IT RECURS
===============================================================================================

19.1 The defense loop's third edge. "IAS-E2 Cognitive Reliability turns recurring failures into SLOs and
failure budgets → IAS-F3 Digital Twin simulates the failure before it recurs → IAS-D2 immunizes exposed
projects preventively." This Part specifies the two touchpoints where IAS-D2 and IAS-F3 exchange
material directly, distinguishing what IAS-D2 SUPPLIES to a simulation from what IAS-D2 CONSUMES back
from one, because the loop's own diagram shows F3 sitting between E2 and IAS-D2's own preventive
immunization step, and a dataset that only describes one direction of that adjacency has not honestly
specified the edge.

19.2 What IAS-D2 supplies to a Digital Twin simulation: the pathogen's mechanism, not its raw incident.
For IAS-F3 to simulate a pathogen's future recurrence against a candidate project's structure BEFORE that
project suffers a real incident, it needs exactly the artifact Part III was built to produce — the
MECHANISM SIGNATURE and STRUCTURAL PRECONDITION fields, abstracted away from any one project's specific
implementation, precisely because a simulation run against the ORIGINATING incident's literal code would
only ever validate that the originating project's own fix works, telling IAS-F3 nothing useful about a
DIFFERENT candidate project's exposure. (DERIVED: this is the specific reason Part 3.3's slot-typed,
mutation-tolerant signature design pays a second dividend beyond mutation detection itself — the same
abstraction that lets a fingerprint match a mutated surface is what lets it be simulated against a
project's structure the originating incident never touched.)

19.3 What IAS-D2 consumes back: simulated exposure as a pre-scan signal, never a substitute for a real
scan. A Digital Twin simulation, run against a fingerprint's mechanism signature and a candidate
project's MODELED structure (not its live, current structure — a twin is, by IAS-F3's own definition, a
representation, and a representation can be stale or incomplete in ways a direct exposure scan, Part VI,
is not), can return a PREDICTED exposure verdict. IAS-D2 treats a predicted verdict as, at most,
evidence level HYPOTHESIZED (per Part 6.4's hierarchy) — sufficient to raise a candidate project's
priority in the scan-priority queue (Part 6.5), sufficient to justify Part XII's Trigger Two-style
preventive sweep request, but never sufficient on its own to authorize a quarantine action (Part 7.2's
gate explicitly excludes HYPOTHESIZED evidence) or an immunization deployment (Part 8.2's confirmation
stage still requires a real, direct exposure scan before an antibody is recorded as CONFIRMED). (NORMATIVE:
this boundary is the direct application of Part VI's evidence discipline to a new evidence SOURCE, and it
exists specifically so that a Digital Twin's own representational fidelity limits — which are IAS-F3's
concern to characterize, not IAS-D2's to assume away — cannot silently downgrade the rigor of an actual
immunization decision.)

19.4 Why simulation is most valuable precisely where retrodiction and live incidents are both silent. Part
16.2's retrodiction protocol can only evaluate mechanisms against incidents that already happened; Part
16.3's synthetic stress-testing can only validate specified mechanisms behave as specified. Neither can
tell the estate whether a NOVEL candidate project — one that has never suffered any incident in a given
lineage and has not yet been scanned — would in fact be exposed if a mutation the lineage has not yet
produced were to occur. A Digital Twin simulation is the one mechanism in this dataset's integration
surface capable of exploring that specific, forward-looking question, which is precisely why the defense
loop places it between reliability discipline (which reacts to what has already recurred) and preventive
immunization (which acts before a specific project's own recurrence) — simulation is the loop's
speculative-but-structured bridge between the two.

19.5 The scenario-construction handoff, concretely. When IAS-D2's mutation-lineage data (Part 5.6) shows
a pattern in HOW a lineage's structural precondition has generalized across its confirmed mutations so
far (Part 5.7's example: ACTION-KIND varies, STRUCTURAL PRECONDITION stays fixed), that PATTERN is
exactly the input IAS-F3 needs to construct a plausible NEXT-MUTATION scenario worth simulating, rather
than simulating an arbitrary, unconstrained hypothetical. IAS-D2 hands IAS-F3 the lineage's own observed
mutation pattern as scenario-generation material; IAS-F3's own object — constructing and running the
scenario itself — remains entirely IAS-F3's, per the Non-Duplication Ledger's Family K disposition, and
IAS-D2 does not specify simulation mechanics here.

19.6 Worked example: simulating F-2201's next plausible mutation before it occurs. Given the F-2201
lineage's observed pattern (ACTION-KIND has varied across deploy, migrate, backup, report-export;
STRUCTURAL PRECONDITION has stayed fixed at "automated action step, unGated completion signal"), a
Digital Twin simulation is constructed against a project whose modeled structure includes an
action-step type the lineage has not yet exhibited — an automated NOTIFICATION-DISPATCH step whose
"sent" signal is likewise derived from a process-exit-code alone. The simulation predicts EXPOSED with
HYPOTHESIZED evidence. This predicted exposure is not acted on directly; it raises the affected project
to the top of Part 6.5's scan-priority queue for a REAL exposure scan, which either confirms the
prediction (upgrading evidence to OBSERVED and proceeding through the normal immunization pipeline) or
refutes it (in which case the discrepancy — why did the twin's model diverge from the live structure — is
itself useful feedback to IAS-F3's own representational-fidelity concerns, outside IAS-D2's own scope to
resolve but worth surfacing as an observed instance of twin-versus-reality divergence).

19.7 The boundary this integration must not cross: IAS-D2 does not become a simulation consumer for
routine, non-mutation-pattern-informed hypotheticals. A Digital Twin is, in principle, capable of
simulating almost any imaginable scenario against almost any project; IAS-D2's integration surface with
IAS-F3 is scoped narrowly to scenarios directly informed by an existing lineage's own observed mutation
pattern (19.5), never a general request to "simulate whatever pathogens might exist" — an unscoped
request of that shape would have no natural stopping point, would compete unboundedly for IAS-F3's own
simulation capacity, and would violate the same targeted-not-blanket discipline Part 6.3 already
establishes for exposure scanning itself, applied here to the simulation integration specifically.

19.8 Cost accounting for simulation requests, and why IAS-D2 self-bounds its own consumption of IAS-F3's
capacity. Because a Digital Twin simulation is, per IAS-F3's own object, itself a resource-intensive
operation, IAS-D2 specifies an explicit request budget for its own simulation consumption rather than
issuing a scenario request for every lineage generalization unconditionally: a simulation request (Part
19.5) is issued only for a lineage that has ALREADY generalized at least once (Part 5.6 — a single-
occurrence fingerprint with no confirmed mutation has no observed pattern to extrapolate from, and
requesting a simulation against it would be speculation with no grounding, not the pattern-informed
scenario 19.5 requires) AND whose severity is HIGH or CRITICAL, mirroring Part 7.2's quarantine-gate
discipline applied here to a different, but structurally analogous, resource-competition problem: IAS-F3's
simulation capacity, like sweep_enforcer's execution capacity (Part 12.5) and IAS-D2's own scan capacity
(Part 6.5), is finite, and every mechanism in this dataset that consumes a shared, finite resource is
specified with an explicit, severity-and-evidence-gated admission rule rather than an open tap. (NORMATIVE:
this is the same discipline, applied a fourth time across this dataset, that Part XV names as the
resource-competition visibility IAS-F1's competes-with relation exists to make explicit — IAS-D2 does not
merely strengthen its downstream and upstream siblings, it disciplines its own consumption of each one
it depends on, which is itself part of what "strengthen, never burden" means in practice.)

FINAL LAW — PART XIX. A Digital Twin simulation earns its place in the defense loop by answering the one
question retrodiction and stress-testing cannot — what would happen to a project that has not yet been
touched — but a predicted exposure is a reason to look harder, never a reason to act as if the looking
were already done, and IAS-D2 requests that simulation only where a real, observed mutation pattern
justifies the cost of asking.

PART XX — WORKED CASE: A CROSS-PROJECT OUTBREAK, END TO END
===============================================================================================

20.1 Purpose of this Part. Parts I through XIX each demonstrated their own mechanism in isolation, using
fragments of a single running synthetic example (the F-2201 lineage) built up incrementally. This Part
assembles one complete, self-contained trace — deliberately a DIFFERENT synthetic pathogen from F-2201,
so that this Part does not merely recap prior worked examples but demonstrates the full dataset operating
as one coherent system against fresh material, illustrative only, with no correspondence to any real PP
incident.

20.2 The originating incident. Session work in Project Nu, mid-task, invokes a subagent to summarize a
completed migration's outcome for an Owner-facing changelog. The subagent's output is accepted and
appended to the changelog without the invoking session re-reading the subagent's actual transcript — the
subagent's own "summary complete" signal is treated as sufficient grounds to trust the summary's factual
content. osa/CEPS's context-engineering discipline flags this pattern during a routine session review: a
delegated summary was trusted without a verification read of the delegate's underlying work, and the
changelog entry, on inspection, contains a factual error the actual migration transcript would have
revealed. This is fingerprinted (Part III): PRIMARY CLASS = false-done (secondary: proxy-evidence, the
delegate's completion signal substituted for content verification). MECHANISM SIGNATURE: "a delegated
task's [ACTION-KIND: summarization] output is trusted based on [VERIFICATION-KIND: delegate-reported
completion] rather than [SUBSTITUTED-SIGNAL: invoker re-reading the delegate's underlying transcript]."
STRUCTURAL PRECONDITION: "project's workflow delegates a content-generating subtask to a subagent or
tool and consumes its output without a re-verification read by the invoking context." SEVERITY: MEDIUM
(factual drift in a changelog, not a production outage). Fingerprint F-3107 is extracted and published.

20.3 Federation and initial scan. F-3107 propagates over the Control Bus (Part XVII) to every subscribed
project. Because severity is MEDIUM, Part 7.2's quarantine gate is not triggered even if exposure is
later confirmed — this lineage routes directly to the immunization queue per 7.2's design, illustrating
that not every worked case in this dataset needs to touch every mechanism to be a complete, correct trace.
Project Xi, an active subscriber, runs a targeted scan and confirms EXPOSED/OBSERVED: Xi's own release-
notes workflow delegates summarization identically. Project Omicron scans CLEAR/OBSERVED: Omicron's
workflow already requires a second-pass human read before any delegate summary is committed, an existing
control the exposure scan correctly recognizes as already gating the mechanism.

20.4 Immunization and a confirmed mutation. Xi's immunization proposal (Part VIII) instantiates the
remediation class — "require an explicit re-verification step against the delegate's underlying output
before its summary is trusted downstream" — concretely as a checklist gate in Xi's own changelog
workflow, approved and deployed, later confirmed CLEAR. Six weeks later, Project Pi's incident: not a
changelog but a code-review delegation, where a subagent's "review complete, no issues found" signal is
trusted without the invoking session reading the subagent's actual review transcript, and a real defect
the subagent's transcript had in fact flagged in a caveat sentence is missed because only the terminal
"no issues found" line was read. Mutation-distance scoring (Part V) against F-3107: ACTION-KIND changes
from "summarization" to "review," VERIFICATION-KIND and SUBSTITUTED-SIGNAL are structurally identical,
STRUCTURAL PRECONDITION generalizes cleanly to "any delegated content-generating or content-evaluating
subtask." Pi's eventual fix matches F-3107's remediation class exactly (require a re-verification read).
Confirmed mutation; F-3107's lineage now spans two members and its precondition generalizes.

20.5 Preventive sweep and a preemptive save. Per Part XII's Trigger One, the lineage's generalization
fires a sweep request against every project previously scanned CLEAR or UNRESOLVED against the narrower,
pre-generalization precondition — which includes Project Omicron, originally cleared (20.3) only against
the summarization-specific case. Re-scanning Omicron against the generalized precondition finds that
while Omicron's changelog workflow is genuinely gated, its SEPARATE code-review delegation workflow — a
different capability the narrower original scan never tested — carries the identical unguarded pattern
Pi just suffered. Omicron's immunization proceeds preemptively, closing the gap before any Omicron
session actually suffers the review-delegation variant of the incident — a second, independently
confirmed instance of Part 1.7's founding claim operating end to end within a single worked case.

20.6 Autoimmune review of a legitimate override. Project Rho's exposure scan flags a candidate match on
its own automated dependency-update summarizer, which reports "update summary generated" based on
generation completing. Rho's maintainer overrides the resulting immunization-queue flag, noting the
summarizer's output is never consumed by a human or downstream system without independent verification —
it is discarded after thirty days if unread, and no decision is ever made from it directly. Review (Part
10.5) confirms: Rho's structural precondition genuinely does not match, because F-3107's precondition
presumes the summary is CONSUMED as a trust input somewhere, and Rho's is not — this is legitimate
risk-context, not autoimmunity, and no correction to the fingerprint is needed, illustrating the healthy
branch of Part 10.5's distinction alongside Part 10.7's Kappa case, which illustrated the confirmed-
autoimmune branch.

20.7 Reading the outbreak's arc against Part XV's metrics. Across the six-project, two-fingerprint-
member trace: CPIR is high (three of four exposed projects reach confirmed immunization within the
observed window; Rho is correctly excluded as never truly exposed). MTTF for the initial F-3107
federation event was same-session for Xi (an active subscriber) and the periodic-reconciliation interval
for slower-onboarding projects. The autoimmune false-positive rate for this lineage is zero — Rho's
override was correctly resolved as risk-context, not autoimmunity, at review, never escalated as a
confirmed autoimmune event. Registry signal density benefits: F-3107 now carries two independent
confirmed origins plus a generalized precondition, placing it well above the single-occurrence threshold
Part 15.6 treats as earning registry weight.

20.8 What this worked case demonstrates that no single-mechanism example could. Every prior Part's
worked example validated its own mechanism narrowly. This trace shows the mechanisms COMPOSING correctly
across a full lifecycle — extraction feeding federation feeding scanning feeding immunization feeding
mutation detection feeding preventive sweep feeding a second immunization feeding autoimmune review — with
no step requiring a hand-off outside the specification already given in Parts II through XVIII, which is
the concrete, checkable sense in which this dataset's Parts form one coherent system rather than fourteen
independently-motivated but disconnected mechanisms.

20.9 What this worked case deliberately does not demonstrate, named honestly. The F-3107 trace was chosen
because its MEDIUM severity let it route entirely through the immunization queue without ever engaging
Part VII's quarantine gate or Part XI's containment ladder (20.3 already notes this explicitly) — a
deliberate choice that keeps the trace legible, but that also means this worked case, on its own, provides
no direct evidence that the quarantine action set (Part 7.3), the containment ladder's four rungs (Part
11.3), or the CRITICAL-severity Owner-escalation path (Part 11.3's Rung Four) compose correctly in
practice, only that they are separately specified and separately worked in their own originating Parts'
narrower examples. (NORMATIVE: naming this gap explicitly, rather than letting a reader over-generalize
from one comprehensive-looking trace to "the whole system has been demonstrated end to end at every
severity," is itself an application of this dataset's own epistemic-honesty discipline to its own
strongest piece of evidence — Part XVI's evaluation protocol, not this single worked case, is what
actually licenses a claim about the HIGH/CRITICAL, quarantine-and-containment-engaging path, and a future
evaluation pass should construct a second full-lifecycle trace specifically choosing a CRITICAL-severity,
contaminated-prompt origin precisely to exercise the rungs and gates this trace's MEDIUM severity never
touched.)

FINAL LAW — PART XX. A pathogen's full lifecycle — from one delegated task's quiet trust failure to a
second, independently-caught mutation, to a preemptive save at a third project that never suffered either
incident — is the concrete shape Part 1.7's founding claim takes when every mechanism this dataset
specifies is allowed to run its full course, and a single trace, however complete along its own severity
path, is evidence for that path alone, not for every path this dataset specifies.

PART XXI — ADVERSARIAL CASES
===============================================================================================

21.1 Purpose: cases constructed specifically to probe where the specification, if implemented carelessly,
would fail — distinguished from Part XIV's failure-mode catalogue by being scenario-driven rather than
mechanism-driven, and from Part XX's worked case by being deliberately hostile to the specification
rather than illustrative of its normal operation.

21.2 Adversarial Case One — the gamed remediation-class match. A project, under review pressure to close
an EXPOSED flag quickly, deploys a fix that is engineered to satisfy Part 5.5's remediation-class check
LITERALLY (it technically inserts a step named "verification" between the action and the completion
signal) without actually implementing the substantive check the remediation class requires (the inserted
"verification" step always returns success without examining anything). This is a proxy-evidence
pathogen (Part 2.3) committed AGAINST the immunization pipeline itself — the antibody record's
CONFIRMATION status would read CLEAR under a structural-presence-only confirmation check (Part 14.2's
named limitation), while the actual protection is nil. Defense: Part 16.4's confirmation-triad
requirement is necessary but, on its own, insufficient against this specific attack, because all three
legs (antibody record, confirmation scan, provenance chain) can be technically satisfied by a fake gate;
the dataset's honest answer, per 14.2's CONTESTED note, is that closing this gap fully requires an
ADVERSARIAL confirmation test — attempting to actually trigger the pathogen's original mechanism and
confirming the gate substantively intercepts it, not merely confirming the gate's syntactic presence —
which this dataset names as a required future refinement rather than claiming is already fully
specified.

21.3 Adversarial Case Two — federation timing exploited to outrun containment. An actor (or a session
under time pressure, not necessarily malicious intent) deliberately propagates a contaminated-prompt
artifact (Part 2.5) rapidly across multiple projects within the containment ladder's 48-hour provisional
window (Part 11.4), specifically betting that Rung Three's confirm-or-lift deadline will lapse before
enough projects' individual exposure scans complete, causing the provisional quarantine to be LIFTED by
default across projects that are, in fact, genuinely exposed. Defense: 11.4's specification already
requires the window's default outcome, on ambiguous or incomplete evidence, to escalate rather than
silently lift — a gap in the literal text of Part XI as originally stated, and this Part corrects it
explicitly: the 48-hour window's non-confirmation default MUST be treated as Rung Four escalation to
Owner review, never as an implicit CLEAR, precisely because a default-to-lift creates exactly the
exploitable race this adversarial case constructs.

21.4 Adversarial Case Three — signature-laundering across taxonomy classes. An incident is deliberately
(or carelessly) fingerprinted under a SECONDARY class only, with the PRIMARY class chosen to be the
taxonomy member least likely to trigger a HIGH-severity quarantine gate (Part 7.2) — for instance,
tagging a genuinely severe false-done pathogen primarily as stale-knowledge (Part 2.7), a class whose
typical severity distribution runs lower, specifically to route the incident away from quarantine and
toward the slower, less scrutinized immunization-only path. Defense: Part 2.8's dual-tagging allowance
was designed for genuine ambiguity, not for this exploit; the corrective this dataset specifies is that
SEVERITY (Part 3.2) is assessed from the incident's actual observed or estimated HARM, independent of
which taxonomy class is chosen as primary, so that reclassifying a pathogen's taxonomy tag cannot, on its
own, downgrade the severity-driven gate at Part 7.2 — severity and taxonomy class are deliberately
decoupled fields precisely to close this specific manipulation path.

21.5 Adversarial Case Four — antibody-record forgery through unverified self-report. A project's own
tooling self-reports an antibody deployment and CONFIRMATION status directly into immune memory (Part
IX) without IAS-D2's own confirmation-scan step (Part 8.2's fourth stage) ever independently re-running
the exposure test — a shortcut that could be adopted under time or resource pressure to make CPIR (Part
15.2) look better without the underlying protection existing. Defense: Part 16.4's confirmation-triad
requirement already names the CONFIRMATION SCAN as a required, independent leg distinct from the antibody
record itself; this adversarial case is the concrete justification for treating that independence as
load-bearing rather than a formality — an antibody record whose confirmation scan was performed by the
SAME process that authored the deployment, without any independent re-scan, does not satisfy the triad,
and IAS-D2's evaluation protocol (Part XVI) treats self-reported-only antibody records as PROPOSED, never
CONFIRMED, until an independently-triggered exposure scan re-validates them.

21.6 Adversarial Case Five — trigger-fatigue exploited as a denial-of-attention attack. An actor
deliberately manufactures a large volume of low-severity, loosely-matched incidents specifically to
trigger Part XII's Trigger One repeatedly, flooding sweep_enforcer's execution queue and Owner review
attention (Part 12.7's named trigger-fatigue risk) to the point that a genuinely severe, concurrently
federated fingerprint is buried in the noise and receives delayed attention. Defense: Part 12.7's own
rate-limiting response (raising severity thresholds, batching same-day requests) is a partial mitigation
but, read adversarially, is itself exploitable if the threshold-raise applies uniformly — an attacker
could specifically target the threshold's blind spot. This dataset's honest correction: the batching and
threshold-raising response of 12.7 must be applied PER-SOURCE (grouping sweep requests by their
originating incident pattern or session identity where available) rather than globally, so that a flood
from one narrow pattern does not raise the threshold for genuinely unrelated, independently-arriving
severe fingerprints — a refinement this Part adds to Part XII's original specification in direct response
to this adversarial construction.

21.7 Adversarial Case Six — a compromised or malfunctioning parent detector forging a failure event. One
of the six named parent systems (Part 3.4's origin list) publishes a FAILURE EVENT (Part 17.2) that does
not correspond to a genuine detection — a bug in the detector itself fabricates an event, or, in the
adversarial framing this Part requires, an actor with write access to a project's session transcript
crafts input specifically designed to trigger a false detection and get a fabricated fingerprint
federated estate-wide, potentially as a vector for Part 21.3's timing exploit or as pure disruption
(flooding the registry, Part 14.6). Defense: this dataset's extraction discipline (Part 3.4) already
requires a fingerprint's origin to reference a real evidence artifact — a hard-rule STOP entry, a
cascade_prevention block log, a secret_firewall match record — never a bare claim; the corrective this
adversarial case adds explicitly is that the extraction stage must ALSO verify the referenced evidence
artifact is independently reconstructible from the originating project's own persistent record (the
actual STOP log entry, the actual block transcript), not merely present in the event payload itself,
because an event payload alone is exactly as forgeable as the fabricated claim this case constructs — the
artifact REFERENCE must be checked against the artifact's OWN persistent store, a verification step this
Part adds to Part 3.4's specification as a closing refinement, mirroring 21.5's antibody-forgery defense
at the earlier, extraction end of the pipeline rather than the later, confirmation end.

FINAL LAW — PART XXI. A specification untested against deliberate hostility is a specification tested
against cooperation only; each adversarial case here — including one aimed at the pipeline's own point of
origin — found a real gap between a mechanism's honest intent and its literal text, and each gap is closed
here explicitly rather than left for the estate to discover the hard way, which is the entire reason this
Part exists as a required companion to Part XX's cooperative worked case rather than an optional addendum.

===============================================================================================
PART XXII — EDGE CASES
===============================================================================================

22.1 Purpose and distinction from Part XXI. Adversarial cases probe deliberate or incidental hostility to
the specification. EDGE CASES probe genuine AMBIGUITY at the boundary of the specification's own defined
terms — situations the mechanisms must handle correctly not because someone is attacking them, but
because the real estate will generate situations the clean, worked examples of Parts II through XX did
not anticipate.

22.2 The novel-pathogen edge case. An incident occurs whose mechanism, on honest analysis, does not fit
cleanly into any of Part II's six taxonomy classes, even as a secondary tag. Per Part 2.8's closure
caveat, the taxonomy is CONTESTED as exhaustive, and this edge case is its operational test: the correct
handling is neither to force a misclassification into the nearest available class (which would corrupt
that class's future matching precision, a self-inflicted instance of Part 14.4's signature-collision
failure) nor to silently drop the incident from fingerprinting entirely (which would violate Article V's
epistemic honesty by pretending no gap exists). The specified response is a SEVENTH PROVISIONAL CLASS —
UNCLASSIFIED — carrying the incident's mechanism signature and structural precondition exactly as any
other fingerprint would, explicitly flagged as taxonomy-incomplete, and reviewed on an accumulating basis
(a threshold of, provisionally, three or more UNCLASSIFIED fingerprints sharing a similar mechanism
pattern is grounds to propose a genuine seventh taxonomy class, an amendment this dataset explicitly
authorizes as a future, evidence-gated addition rather than freezing the taxonomy as permanently fixed
at six).

22.3 The benign-mutation edge case: when a "mutation" is actually a legitimate, superior alternative
implementation. Project Sigma's incident superficially mutation-matches an existing lineage (mechanism
signature scores LOW distance), but Sigma's implementation is not a bug at all — it is a deliberately
chosen, well-reasoned design where the "unguarded" completion signal is, on inspection, provably
equivalent in safety to the guarded pattern because of a property specific to Sigma's architecture (for
instance, the action itself is provably idempotent and self-verifying by construction, making a separate
post-action check genuinely redundant rather than merely inconvenient — a stronger and more specific
claim than Part 10.7's Kappa case, which showed an existing but differently-shaped check; Sigma
demonstrates no check is needed at all). This is distinguished from Part X's autoimmune review by scale
and permanence: it is not a one-off override, it is a standing architectural pattern. The specified
response is a LINEAGE EXCLUSION RECORD — not a correction to the fingerprint's precondition (which
remains correct for the general case) but an explicit, provenance-linked exception noting that Sigma's
specific architectural pattern is verified exempt, distinguished from a routine autoimmune false positive
precisely because it documents a genuine structural reason rather than merely resolving one project's
disputed instance.

22.4 The orphaned-fingerprint edge case: a fingerprint whose originating project no longer exists. A
fingerprint's ORIGIN field (Part 3.2) names a project that is later retired, merged, or extinguished
(per IAS-D1's own extinction protocol, referenced not re-narrated here). The fingerprint itself, its
lineage, and any antibodies it produced elsewhere in the federation remain fully valid and continue to be
federated and scanned against normally — a pathogen's validity as a cross-project threat does not depend
on its originating project's continued existence, precisely because Part III's extraction discipline
already abstracts the mechanism away from any one project's specific implementation. The ORIGIN field is
retained for historical provenance (Part 9.5) even after the named project no longer exists, marked with
a retired-origin flag rather than deleted, consistent with Part 9.6's never-delete discipline.

22.5 The multi-origin-conflicting-severity edge case. Two independent origins of what deduplication (Part
3.5) correctly identifies as the same underlying pathogen assign genuinely different severities to their
respective incidents, because the SAME mechanism caused MEDIUM harm in one project's context and CRITICAL
harm in another's (a false-done pathogen in a low-stakes internal tool versus the structurally identical
pathogen in a project whose output feeds a higher-stakes downstream decision). The specified resolution:
a lineage's SEVERITY field is set to the MAXIMUM across its confirmed origins, never an average and never
the most recent, because Part 7.2's quarantine gate exists to catch the worst-case exposure, and a lineage
scored by average severity would systematically under-trigger quarantine for the specific projects whose
context makes the same mechanism most dangerous — the exact projects the gate most needs to reach.

22.6 The concurrent-quarantine-and-immunization edge case. A project's exposure scan confirms EXPOSED
while, unknown to the scanning subsystem, an immunization proposal for the SAME (project, lineage) pair
is already mid-review from an earlier-triggered path (for instance, Part XII's Trigger Two fired from a
structural-change event just before Trigger One independently fired from a lineage generalization). The
specified resolution: Part 7.2's antibody-memory check ("no existing antibody already covers the
fingerprint's lineage") is extended explicitly to also check for an IN-FLIGHT immunization proposal, not
only a CONFIRMED antibody — a pending proposal is sufficient grounds to suppress a redundant quarantine
action or a duplicate immunization proposal, avoiding the specific institutional waste of two independent
response pipelines converging on the same fix without either being aware of the other, a coordination gap
resolved by treating the immune-memory index (Part 9.1) as covering PROPOSED as well as CONFIRMED
antibody states.

22.7 The simultaneous-onboarding edge case: many new projects joining the estate within a short window.
Part 9.7's Iota worked example illustrated one new project's onboarding baseline computed against an
already-populated immune memory. A genuinely different situation arises when SEVERAL new projects onboard
within a short window relative to Part 6.5's scan-priority throughput — a common real-world shape (a
batch import of several repositories into estate governance at once, rather than the estate's usual
one-at-a-time growth). Under 6.5's existing priority ordering (severity first, lineage-size second), a
naive implementation would interleave every new project's onboarding scans with the estate's ordinary,
ongoing exposure and decay scanning, and a large enough onboarding batch could, without a specific
accommodation, temporarily starve routine scanning capacity across the whole federation. The specified
resolution: onboarding scans for a batch are treated as their own priority BAND, scheduled within the
existing severity ordering but capacity-capped as a group (EXPERIMENTAL default: no more than a fixed
fraction, provisionally one-third, of any scanning period's total capacity may be consumed by onboarding
scans regardless of batch size), so that a large onboarding event degrades gracefully — new projects take
longer to reach their full baseline immunity picture — rather than silently starving the routine scanning
Part XIII's decay monitoring and Part XII's preventive sweeps depend on continuing at their own accustomed
cadence. (DERIVED: this is the same graceful-degradation-over-hard-failure discipline Part 18.5 already
names as IAS-D2's contribution to IAS-E2's own design goal, applied here reflexively to IAS-D2's own
internal capacity management rather than to a downstream project's protection.)

22.8 The topology-drift-collision edge case: a pathogen whose structural precondition is itself a symptom
of drift IAS-G1 already tracks. A fingerprint's structural precondition (Part 3.3) sometimes turns out, on
inspection, to describe not an isolated coding mistake but a SYSTEM-LEVEL topology shape — for instance, a
precondition naming a completion-signaling call reachable without a preceding validation call may recur
across an unusually large fraction of the estate's projects not because each project independently made
the same local error, but because the estate's own architecture has drifted toward a topology IAS-G1's own
drift detection would separately flag as a complexity-budget violation. Treating this purely as a pathogen
with an unusually wide blast radius, under Part VI's exposure-scanning logic alone, would immunize every
individual instance while leaving the underlying topological cause — the reason the same precondition
keeps recurring — entirely unaddressed, a durable instance of Part 14.6's whack-a-mole failure mode
operating at architecture scale rather than at the level of one recurring bug. The specified handling:
when a lineage's exposed-project count (Part 6.6) exceeds an EXPERIMENTAL threshold (provisionally,
exposure confirmed across five or more independent projects sharing no common shared artifact per 11.2's
contaminated-prompt distinction), IAS-D2 CONSUMES IAS-G1's drift-detection capability directly, submitting
the fingerprint's structural precondition as a candidate topology-drift signal rather than treating the
wide blast radius as coincidence. A confirmed topological match does not replace immunization at the
individual-project level (Part VIII proceeds normally, each exposed project still receives its antibody)
but ADDS a topology-level finding to IAS-G1's own drift ledger, so the estate's architecture review sees
the aggregate pattern a per-project view could never surface on its own — the edge case's genuine
ambiguity being that a sufficiently widespread pathogen is not always merely a pathogen.

FINAL LAW — PART XXII. A taxonomy, a severity field, a memory index, and a scan-priority queue under
onboarding pressure are only as honest as their handling of the case that does not fit cleanly; the
estate is better served by a seventh provisional class, a documented exception, a maximum-not-average
severity rule, and a capacity-capped onboarding band than by forcing every real incident and every real
growth surge into the shape the worked examples already anticipated.

PART XXIII — THE LIFECYCLE OF A PATHOGEN RECORD
===============================================================================================

23.1 Why a lifecycle Part, after twenty-two Parts of individual mechanisms. Parts III through XIII each
specified one STAGE a pathogen record passes through — extraction, federation, mutation confirmation,
exposure scanning, quarantine, immunization, memory, decay — but no prior Part assembled the STATE
MACHINE those stages compose into, the object an implementer or an auditor actually needs to answer "what
state is THIS fingerprint in, right now, and what transitions are legal from here." This Part supplies
that state machine as the dataset's own closing structural artifact, mirroring the Constitution's own
dataset-level state machine (DISCOVERED → ... → SEALED) at the finer grain of a single pathogen record.

23.2 The eight states. EXTRACTED (Part III: a fingerprint exists, evidence OBSERVED at origin only).
FEDERATING (Part IV: published to registry and bus, not yet confirmed at any second project). CANDIDATE-
MUTATION (Part V: a new incident scored LOW distance against this lineage, pending remediation-class
confirmation). EXPOSURE-CONFIRMED (Part VI: at least one non-origin project scanned EXPOSED with OBSERVED
or DERIVED evidence). QUARANTINED (Part VII: an active containment action exists against at least one
confirmed-exposed project — a state that may coexist with EXPOSURE-CONFIRMED at other projects
simultaneously, since quarantine is per-project while the fingerprint's own state reflects its most
advanced stage anywhere in the federation). IMMUNIZING (Part VIII: at least one immunization proposal is
in review or deployed-pending-confirmation). IMMUNIZED (Part VIII-IX: at least one CONFIRMED antibody
exists; the lineage as a whole may still show other projects in earlier states simultaneously). DECAYING
(Part XIII: at least one previously-CONFIRMED antibody has an active, unresolved decay signal). Each
state names a per-lineage AGGREGATE condition — because, as 23.2's own definitions make explicit,
different member projects of the same lineage can honestly occupy different individual states at once,
the lineage's own reported state is its most-advanced-currently-active condition, with the full per-
project breakdown always available in the exposure map (Part 6.6) for anyone who needs finer resolution
than the aggregate state provides.

23.3 Legal transitions and the two transitions requiring independent confirmation. Most transitions
(EXTRACTED to FEDERATING, FEDERATING to EXPOSURE-CONFIRMED, EXPOSURE-CONFIRMED to QUARANTINED or
IMMUNIZING) are STRAIGHTFORWARD — each is the direct, mechanical output of the Part that governs it, no
separate confirmation logic required beyond what that Part already specifies. Two transitions are
DELIBERATELY GATED behind independent confirmation, and are named explicitly here because their gating is
the load-bearing defense against Part XXI's Adversarial Case One and Case Four: CANDIDATE-MUTATION to
IMMUNIZING requires the remediation-class check (Part 5.5) to pass independently of the mechanism-
signature distance score; IMMUNIZING to IMMUNIZED requires the independent confirmation scan (Part 16.4's
triad, reinforced by 21.5's forgery defense) rather than accepting a self-reported deployment claim.
Every other transition in the state machine may be automated in full per the mechanisms Parts III-XIII
already specify; these two remain the state machine's explicit choke points precisely because they are
where an unverified claim would otherwise convert directly into a false institutional record.

23.4 Reverse transitions and the honesty of a state machine that permits regression. Unlike many
state-machine designs that model only forward progress, this one explicitly permits IMMUNIZED to
DECAYING (Part XIII, already specified) and, from DECAYING, back to EXPOSURE-CONFIRMED or IMMUNIZING
depending on the decay signal's resolution (Part 13.5) — the state machine does not pretend immunity, once
achieved, is a terminal, upward-only condition, because Part XIII's entire argument is that it is not.
Similarly, QUARANTINED reverts to EXPOSURE-CONFIRMED (never silently to a lower state) if a quarantine's
review window (Part 7.5) lapses without progress, ensuring the aggregate state always reflects a currently
justified condition rather than a historical peak the lineage may no longer actually be in.

23.5 The terminal state, and why it is provisional rather than absolute. The closest this state machine
comes to a terminal condition is a lineage where every known exposed project (per the current exposure
map) shows a CONFIRMED, non-decaying antibody and no CANDIDATE-MUTATION is pending — reported, for
convenience, as FULLY-IMMUNIZED. This label is explicitly provisional, never absolute, for two reasons
stated plainly: Part 22.2's novel-pathogen mechanism means a genuinely new project could always onboard
tomorrow and discover a fresh exposure the current lineage's precondition, however well-generalized,
still does not cover a further mutation of; and Part XIII's decay signals mean a FULLY-IMMUNIZED lineage
can silently regress to DECAYING at any time a code-drift, mutation, or understanding-revision signal
fires. A FULLY-IMMUNIZED report is therefore always dated and always understood, per Article V, as
OBSERVED-as-of-that-date rather than as a permanent verdict.

23.6 The lifecycle as the object Part XVI's evaluation protocol actually audits. Every evaluation
question Part XVI poses — MTTF, CPIR, half-life, the confirmation triad — is, at bottom, a question about
how a population of pathogen records moves through exactly this eight-state lifecycle: MTTF measures the
EXTRACTED-to-FEDERATING transition's speed; CPIR measures the EXPOSURE-CONFIRMED-to-IMMUNIZED transition
rate; half-life measures the expected dwell time in IMMUNIZED before a transition to DECAYING. Naming the
lifecycle explicitly in this closing Part retroactively clarifies that every metric in Part XV was always
a measurement over this state machine's transition rates, even though the state machine itself was not
assembled as an explicit object until here — a sequencing choice this dataset acknowledges directly: the
mechanisms were specified stage by stage because that is how an implementer builds them, but the state
machine is how an auditor or an Owner should actually READ the system's health at a glance.

23.7 Auditing the lifecycle state machine against the exposure map directly, as a standing consistency
check. Because a lineage's aggregate state (23.2) is defined as its most-advanced-currently-active
condition across all member projects, while the exposure map (Part 6.6) carries the full, per-project
breakdown beneath that aggregate, the two artifacts can DRIFT apart if an implementation updates one
without the other — a lineage reported IMMUNIZED in aggregate while its own exposure map still shows one
project genuinely EXPOSED with no in-flight proposal is not a nuanced edge case, it is a bookkeeping
defect, and IAS-D2 specifies a standing RECONCILIATION CHECK, run on the same cadence as Part 4.4's
full-registry reconciliation, that recomputes each lineage's aggregate state fresh from its own exposure
map and antibody records rather than trusting a separately-maintained aggregate field to have been updated
correctly on every individual transition. (NORMATIVE: this mirrors, at the lifecycle-state layer, the
exact discipline IAS-F1's Invariant Five applies to any property record generally — an aggregate is a
DERIVED fact, and a derived fact that is cached rather than recomputed is a staleness risk by
construction; the reconciliation check exists specifically so that "lineage state" is always, at audit
time, verifiably a function of the underlying per-project records, never an independently-drifting
summary someone forgot to update after a quarantine lifted or a decay signal resolved.) A reconciliation
mismatch, when found, is itself worth fingerprinting under this dataset's own taxonomy as a
stale-knowledge pathogen (Part 2.7) in IAS-D2's own bookkeeping — the reflexive application Part 17.7
already demonstrated for bus-outage patterns, extended here to the lifecycle layer.

FINAL LAW — PART XXIII. A pathogen record's life is not a straight line from discovered to cured; it is
an eight-state machine with two independently-gated choke points and at least one honest path backward,
and a federation that reports only forward progress on this machine — or that lets its own aggregate
state drift from the per-project record beneath it — is reporting a fiction the decay mechanism it also
specifies was built specifically to contradict.

PART XXIV — GOVERNANCE, EVOLUTION, AND THE BOUNDARY RESTATED
===============================================================================================

24.1 Why the dataset closes on governance rather than on a further mechanism. Twenty-three Parts have
specified what the institutional immune system detects, federates, contains, quarantines, immunizes,
remembers, and audits. This final Part closes the dataset the way the Constitution requires every IAS
dataset to close: by stating explicitly how IAS-D2 itself may evolve, who is authorized to change it, and
by restating — one final time, with the full apparatus now visible — the exact boundary that keeps this
dataset from becoming the imperial coordinator Part I §1.2 warned against building.

24.2 Governed self-evolution, applied to IAS-D2's own thresholds and defaults. Every numeric default this
dataset names — the 14-day and 3-day quarantine review windows (7.5), the 48-hour containment
confirmation window (11.4), the two-year archival threshold (9.6), the three-independent-overrides
autoimmune candidate threshold (10.4), the mutation-distance threshold τ (5.4) — is explicitly marked
EXPERIMENTAL and is a candidate for revision as Part XVI's evaluation protocol accumulates real evidence.
Per Constitution Article IX, no such revision may be self-applied by any IAS-D2 mechanism without Owner
promotion: a mechanism that observes its own threshold is miscalibrated (via Part XV's metrics) may
PROPOSE a revised value, with the observed evidence attached, but the revision itself is deployed only
through the same Owner-decide channel Part 8.2 already specifies for immunization proposals — threshold
tuning is, in this precise sense, treated as just another instance of a proposal requiring promotion,
never a self-modifying parameter.

24.3 What a future extension of this dataset may add, and what it may never add. Legitimate future
extensions, consistent with everything specified here, include: a seventh-plus taxonomy class (Part
22.2's amendment path), refined mutation-distance weights (Part 16.5's calibration path), an adversarial
confirmation-scan upgrade (Part 21.2's named future refinement), and additional integration touchpoints
with sibling IAS datasets not yet built at this dataset's authoring time. What may never be added, under
any evidence or Owner authorization, without amending the Constitution itself first: authority to act
against a system outside the ensemble boundary (Article II, Part 14.5), authority to auto-deploy an
antibody without Owner approval (Article IX, Part 8.2), or a composite immune-health score that collapses
Part XV's deliberately-paired metrics into one number (15.8's explicit prohibition) — these three are not
merely defaults awaiting recalibration, they are structural boundaries this dataset holds fixed regardless
of how compelling a future efficiency argument for crossing them might appear.

24.4 The boundary restated, once more, now that every mechanism is visible. Having specified fingerprint
extraction, federation, mutation detection, exposure scanning, quarantine, immunization, immune memory,
autoimmune detection, containment, preventive sweeps, decay monitoring, and three sibling integrations,
it is worth confirming directly that none of these twenty-three Parts ever specified IAS-D2 detecting a
failure inside a single session on its own authority, blocking a command inside one project's live
execution on its own authority, or writing a durable code change to any project without Owner approval —
the three specific capabilities that would have made IAS-D2 a duplicate of cascade_prevention, osa/CEPS,
or a self-authorizing mutation engine, respectively. Every mechanism this dataset actually specifies
operates strictly on the ensemble-level object Part I §1.6 and Part 8.6 name: which projects are exposed
to which fingerprinted mechanisms, and how protection propagates between them — never on any single
project's resource-level execution.

24.5 The founding claim, restated as a closing accounting. Part 1.7 opened this dataset with a claim: a
root cause should cost the estate one full incident, with every other exposed project immunized before
independently paying the same cost. Twenty-three Parts later, that claim is no longer asserted alone — it
is backed by a named taxonomy (Part II), a mutation-tolerant fingerprint schema (Part III), a dual-channel
federation mechanism (Part IV) with a named latency metric (Part XV), a three-valued exposure scan
(Part VI), a narrowly-scoped quarantine mechanism with a closed action set (Part VII), a project-specific
immunization pipeline with an Owner-gated deployment stage (Part VIII), a bidirectional immune memory with
provenance (Part IX), a behaviorally-grounded autoimmune detector (Part X), a time-pressured containment
ladder (Part XI), an event-triggered sweep-intelligence layer (Part XII), a decay-monitoring discipline
that refuses to treat immunity as permanent (Part XIII), an assembled eight-failure-mode catalogue
(Part XIV), paired anti-Goodhart metrics (Part XV), a retrodiction-and-stress-test evaluation protocol
(Part XVI), two fully-specified sibling integrations (Parts XVII-XIX), one end-to-end worked lifecycle
(Part XX), five adversarial cases with their gaps closed in text (Part XXI), five edge cases with their
resolutions specified (Part XXII), and one closing state machine (Part XXIII). What remains, honestly, is
what Part XVI §16.7 already names as UNRESOLVED: the space of pathogen shapes the estate has not yet
encountered, which no dataset authored before those shapes exist can fully anticipate — and the correct
institutional response to that residual uncertainty is not a claim of completeness this dataset does not
make, but the taxonomy-amendment, threshold-recalibration, and evaluation-protocol machinery specified
across Parts XVI and XXII, built so that the estate's immune system, like the one it borrows its
vocabulary from, gets more precise the longer it operates, without ever needing this text amended to let
it do so.

24.6 What this dataset explicitly leaves for its nearest sibling and for future Family-D work to resolve.
Consistent with Part 14.8's reading of paired failure modes and Part 21's honest accounting of gaps closed
in text rather than left implicit, this closing Part names three items explicitly OUT of IAS-D2's own
scope, owed to a sibling or to future work rather than silently assumed solved. First, the boundary
between IAS-D2's PARASITIC-COMPLEXITY pathogen class (Part 2.6, a recurring PATTERN of overhead across
projects) and IAS-D1's own single-system parasitic-complexity detector (Non-Duplication Ledger candidate
33) is drawn here only at the definitional level — IAS-D2 detects the pattern's cross-project recurrence,
IAS-D1 judges any one instance's fitness within its own project's ecology — and a fully worked joint
protocol for the two systems handing a specific parasite case back and forth has not been specified in
either dataset's text and is named here as the concrete integration gap a future revision of either
dataset should close. Second, Part 21.2's adversarial confirmation gap (structural-presence-only
verification cannot fully distinguish a genuine antibody from a gamed one) remains open at this dataset's
sealing, and closing it fully likely requires a capability — safe, bounded, adversarial re-triggering of
a pathogen's original mechanism against a live antibody — that does not yet exist anywhere in the estate
and that this dataset does not attempt to specify from scratch. Third, the relationship between IAS-D2's
per-lineage IMMUNITY HALF-LIFE (Part 13.4) and IAS-F3's own simulation-based forecasting (Part XIX) is
gestured at but not fully joined — a mature integration might let a Digital Twin simulate a lineage's
LIKELY future decay pattern from its observed half-life, rather than IAS-D2 waiting passively for decay
signals to fire, and this dataset names that possibility as a natural next extension rather than building
it prematurely against an evidence base (Part 13.4's own EXPERIMENTAL half-life estimates) not yet mature
enough to simulate against responsibly.

FINAL LAW — PART XXIV. An institutional immune system is judged not by the pathogens it has already
named, but by whether the mechanism that named them can honestly recognize the ones it has not — and this
dataset closes having built that mechanism, named its own remaining seams to its nearest siblings
explicitly, and not having claimed the naming, or the federation, is finished.

PART XXV — CLASS SEVEN: THE ADVERSARIAL PATHOGEN
===============================================================================================

25.1 Why a twenty-fifth Part, and why it is the amendment Part II anticipated. Part II sealed a
closed six-member pathogen taxonomy but marked its own closure CONTESTED (§2.8): "new incident
types may surface a seventh class the estate has not yet suffered," resolved not by forcing a
misclassification but through the amendment path Part XXII §22.2 specifies. Part XXIV §24.3 names
"a seventh-plus taxonomy class" first among the legitimate future extensions of this dataset, and
the Decision Log's D2 records the "explicit seventh-class amendment path" as a deliberate design
choice rather than an oversight. This Part exercises exactly that sanctioned path. The six original
classes are left sealed and unedited; Class Seven is added here, and it is a full Part rather than a
seventh entry beside §2.2-§2.7 because it differs from all six on an axis the six silently share —
and naming that axis is the whole justification for a new class rather than a stretched old one.
(NORMATIVE, DERIVED from §2.8's CONTESTED closure, §22.2's amendment path, and §24.3's named
extension.)

25.2 The axis the six share and Class Seven breaks: no adversary. Every one of the six original
classes is SELF-INFLICTED. False-done is the estate's own unbacked claim; proxy-evidence its own
inference; ineffective-rule its own governance; contaminated-prompt its own reused artifact;
parasitic-complexity its own overhead; stale-knowledge its own cached fact. Not one has an
adversary — each arises from the estate acting upon itself, accidentally, and drifting as its own
code and notes drift. The whole immune model inherits that assumption: the fingerprint (Part III)
is deliberately mutation-TOLERANT against INCIDENTAL drift (Part V, "same cause, changed surface"),
because a self-inflicted pathogen does not try to change its surface. Class Seven, the ADVERSARIAL
pathogen, is DEFINED as a pathogen introduced by an actor whose intent is hostile to the estate and
whose defining property is that it ADAPTS AGAINST THE IMMUNE RESPONSE ITSELF — an adversary who
observes the deployed defense and reshapes the pathogen to evade it. That single property breaks
the no-adversary assumption the other six license, and every mechanism built on that assumption
inherits the break. (DEFINED.)

25.3 The definition, and the boundary against Class Four. Class Seven is not merely "a contaminated
prompt that came from outside." Class Four (§2.5) already covers an artifact carrying an injected
instruction that propagates by literal copying — but a Class-Four pathogen is PASSIVE: it carries
its defect verbatim and does not adapt, and its mutation is the incidental drift of the six. Class
Seven is ACTIVE, and the distinguishing test is precise: is there an adversary who, upon the estate
deploying a defense, MODIFIES the pathogen specifically to defeat that defense? A prompt-injection
payload sitting inert in fetched data is a Class-Four contaminated prompt; the same payload becomes
a Class-Seven adversarial pathogen the moment its surface is observed to be reshaped against the
estate's filters. The class is individuated by the ADAPTATION-AGAINST-DEFENSE relationship, not by
the arrival surface — injection, a poisoned dependency, a hostile contribution are all merely how
it arrives; intent plus observed adaptation is what makes it Class Seven. Drawing the boundary on
the relationship rather than the surface is what keeps Class Seven from either swallowing Class Four
or being swallowed by it. (DEFINED, NORMATIVE boundary.)

25.4 Why Part V's mutation model cannot absorb it by recalibration. Part V's failure-mutation
detection is built for "same cause, changed surface" where the change is INCIDENTAL — a project
refactors and the pathogen's surface drifts without anyone trying to hide it, so a mutation-tolerant
fingerprint within a distance threshold τ (§5.4) still matches. An adversarial pathogen's mutation
is DELIBERATE and DEFENSE-DIRECTED: the adversary reads or infers the fingerprint the estate matches
on and constructs the next variant to fall just OUTSIDE it. Against incidental drift, a fixed τ
converges — the surface stops drifting once the refactor settles. Against a deliberate adversary, τ
does not converge, because every variant is engineered to sit just past the previous one's match
radius, so a threshold tuned on the estate's self-inflicted history systematically UNDER-detects
Class Seven, and tightening τ to compensate raises the autoimmune rate (Part X) against the six
benign classes. This is not a tuning error correctable within Part V's model; it is a category
error, and it is the concrete reason Class Seven must be its own class rather than a hard case of
the existing mutation machinery. (DERIVED.)

25.5 The distinct immune requirement: adversarial confirmation, not structural presence. Part XXI
§21.2 and Part XXIV §24.6 already named the estate's honest open gap — structural-presence-only
confirmation cannot distinguish a genuine antibody from a gamed one, and closing it "requires an
ADVERSARIAL confirmation test — attempting to actually trigger the pathogen's original mechanism and
confirming the gate substantively intercepts it." For the six self-inflicted classes that adversarial
confirmation is a desirable upgrade the dataset defers as future work. For Class Seven it is
MANDATORY and part of the class definition, because an adversarial pathogen is defined precisely by
its ability to satisfy a structural check while defeating its substance — satisfying-the-check-while-
defeating-the-substance is exactly what an adaptive adversary optimizes for. A Class-Seven antibody
is therefore UNCONFIRMED until an independently-triggered, safe, bounded re-exercise of the
pathogen's mechanism against the live defense OBSERVES substantive interception; confirmed by
structural presence alone, it is by the class's own definition presumed defeated. (NORMATIVE,
DERIVED — promoting §21.2's named-future-refinement from optional to definitional for this class.)

25.6 Containment under an adversary who exploits the containment ladder. Part XI's containment
ladder and Part XXI's adversarial cases already show an adversary weaponizing the immune system's
own timing (§21.3, the confirm-window race) and rate limits (§21.6, the denial-of-attention flood).
For the six self-inflicted classes those are edge cases; for Class Seven they are the EXPECTED
operating condition, because an adversary who adapts against detection adapts against containment
too. The corrective §21.3 already sealed — the 48-hour window's non-confirmation default MUST
escalate, never lift — is, for Class Seven, promoted from a correction into an INVARIANT: any
ambiguity in a Class-Seven containment decision defaults to the most conservative outcome, continued
quarantine and Owner escalation, never an implicit clear. The reasoning is that the one variable an
adversary reliably controls is the estate's own uncertainty, so a default-to-lift on ambiguity hands
the adversary the win by construction. Default-to-contain on ambiguity is the Class-Seven containment
law, and it is the direct application of §21.3's fix as a standing rule rather than a case-specific
patch. (NORMATIVE, DERIVED.)

25.7 Attribution by observed adaptation, never by naming the attacker. A self-inflicted pathogen's
origin is a real evidence artifact (§3.4: a hard-rule STOP entry, a block log). A Class-Seven
pathogen's origin includes an adversary, and the tempting error is to attribute the adversary — to
record who did it and why. Class Seven explicitly does NOT require identifying the attacker or
proving intent, because attribution of identity is usually impossible and attribution of motive is
unreliable, and a class that demanded either would be inert in exactly the cases that matter most.
What Class Seven requires recorded is the ADAPTATION EVIDENCE: the observed fact that a pathogen's
surface changed against a deployed defense — variant A was intercepted, and variant B, of materially
the same mechanism but a defense-evading surface, appeared after the defense was deployed.
Adaptation-against-defense is an OBSERVABLE property of the pathogen's own history; the adversary's
identity and motive are not, and the class is built strictly on the observable, exactly as §3.4
builds a fingerprint's origin on a reconstructible artifact and never on a bare claim. (NORMATIVE,
DERIVED, mirroring §3.4's extraction discipline.)

25.8 The failure shape this class exists to prevent: the LEARNED BLIND SPOT. The specific harm Class
Seven closes is the immune system TEACHING ITS OWN ADVERSARY. Every defense the estate deploys and
describes is, to an adaptive adversary, a specification of what to evade — so an immune system that
publishes its fingerprints and confirmation checks in full operational detail is, against Class
Seven, training the next variant to fall past exactly those defenses, and the more diligently it
immunizes, the more precisely the adversary is shaped around its protection. This is the adversarial
dual of stale-knowledge (§2.7): stale-knowledge is the estate trusting an old fact of its own; the
learned blind spot is the adversary exploiting the estate's own published defense against it. The
structural defenses are the two already stated — adversarial confirmation (§25.5), so a defeated
defense is detected rather than trusted, and default-to-contain (§25.6), so an ambiguous outcome
fails CLOSED with the pathogen presumed live. Failing closed is the whole point: the immune system's
uncertainty about a Class-Seven pathogen must cost the adversary, not the estate. (DERIVED,
NORMATIVE.)

25.9 The boundary: Class Seven recognizes and contains, it never counter-attacks. Held to the exact
boundary Part XXIV §24.3-§24.4 hold for the whole dataset: recognizing an adversarial pathogen
authorizes DETECTION, CONTAINMENT, adversarial CONFIRMATION, and immunization of exposed projects —
it authorizes no action whatever against any actor or system outside the ensemble boundary, which
§24.3 names as a structural limit unamendable without amending the Constitution itself. The immune
system's entire response to Class Seven is to fail closed, escalate to Owner, and immunize the
exposed — never to retaliate against, probe, or act upon an external party, and this Part specifies
no technique for doing so and licenses none. A class that permitted counter-action would cross the
precise line §24.3 seals as permanent; Class Seven widens only what the estate can RECOGNIZE, and
widens nothing about what the estate may DO to anyone outside itself. The seventh class is a sharper
set of eyes, never a weapon, and the distinction is load-bearing rather than rhetorical: the whole
value of naming the adversary is to see the pathogen more honestly, not to license the estate to
become an adversary in turn. (NORMATIVE, DEFINED — the boundary that keeps this appended class a
recognition object rather than an escalation into offence.)

FINAL LAW — PART XXV. The six original classes are pathogens the estate inflicts upon itself, and
their whole immune model quietly assumes there is no one on the other side trying to win; Class Seven
is the pathogen that has an adversary, defined not by where it arrives but by the one property the
six lack — it adapts against the defense itself — so its fingerprint threshold never converges, its
antibody is UNCONFIRMED until a safe adversarial re-exercise observes substantive interception, its
containment defaults to contain-and-escalate on any ambiguity so that the estate's uncertainty costs
the adversary rather than the estate, and its attribution rests on the observable fact of adaptation
and never on naming an attacker; and the class recognizes and contains and immunizes and fails
closed, but it never counter-attacks beyond the ensemble boundary, because an immune system that
answers an adversary by becoming one has not defended the estate, only enlisted it.

DECISION LOG
===============================================================================================

D1. DECISION: Federate at the MECHANISM-SIGNATURE level (slot-typed abstraction, Part 3.3) rather than
at the raw-incident-text level. ALTERNATIVES CONSIDERED: (a) federate the full incident narrative and let
each receiving project's own agent interpret it; (b) federate only a taxonomy-class tag with no further
structure. WHY THIS: (a) is too expensive to match automatically at federation scale and too subjective
for consistent cross-project comparison; (b) is too coarse to distinguish a genuine pathogen match from
mere taxonomy-class coincidence, as Part 10.2's over-broad-signature autoimmune entry point demonstrates
directly. TRADE-OFFS: slot-typed signatures require disciplined extraction (Part 3.8's incidental-versus-
structural gate) and impose authoring overhead per fingerprint. REVERSAL CONDITIONS: if retrodiction
(Part 16.2), once run against real estate history, shows slot-typed signatures systematically miss
genuine matches that a looser, embedding-based similarity measure would catch, the signature schema
should be revisited — this is an empirical question Part XVI leaves open, not a closed one.

D2. DECISION: Specify a closed, six-member pathogen taxonomy (Part II) with an explicit seventh-class
amendment path (Part 22.2), rather than an open, unbounded classification scheme. ALTERNATIVES
CONSIDERED: (a) no taxonomy at all, classify each pathogen ad hoc; (b) a large, granular taxonomy
covering dozens of specific failure shapes from the outset. WHY THIS: (a) makes cross-fingerprint
comparison and metric aggregation (Part XV) impossible, because nothing is held constant across records;
(b) risks Part 14.6's registry-pollution dynamic at the taxonomy level itself — a taxonomy so granular
that every incident gets its own class provides no real compression. TRADE-OFFS: six classes will
occasionally force a judgment call on dual-tagging (Part 2.9's worked example shows five classes present
in one incident) and the UNCLASSIFIED provisional class (22.2) adds a small amount of process overhead.
REVERSAL CONDITIONS: three or more UNCLASSIFIED fingerprints sharing a mechanism pattern is the named,
concrete trigger for proposing a genuine seventh class — reversal is built into the design, not an
exception to it.

D3. DECISION: Quarantine authorization requires OBSERVED or DERIVED evidence only (Part 6.4, Part 7.2),
explicitly excluding HYPOTHESIZED evidence including Digital Twin predictions (Part 19.3). ALTERNATIVES
CONSIDERED: (a) allow quarantine from any sufficiently high-confidence prediction, including simulated
exposure; (b) require unanimous multi-signal confirmation before any quarantine, even for CRITICAL
severity. WHY THIS: (a) directly enables Part 14.3's over-quarantine failure mode at scale, because a
simulation's representational fidelity is, by Part 19.3's own argument, bounded below the fidelity of a
direct scan; (b) is too slow for CRITICAL-severity exposures where Part XI's containment ladder exists
precisely because full confirmation cannot always be awaited. TRADE-OFFS: some genuinely exposed
projects will experience delayed quarantine while a direct scan is arranged, a cost this dataset accepts
explicitly in exchange for bounding autoimmune risk. REVERSAL CONDITIONS: if Part 21.2's adversarial gap
(structural-presence-only confirmation) is closed by a future adversarial confirmation test, the evidence
hierarchy could be revisited to admit a HIGH-confidence simulated verdict as DERIVED rather than
HYPOTHESIZED for a narrow, well-validated simulation class — explicitly left open, not decided here.

D4. DECISION: No composite immune-health score (Part 15.8); every metric is reported paired with its
countermetric. ALTERNATIVES CONSIDERED: (a) a single weighted composite, analogous to IAS-F2's
Institutional Advantage Unit, for at-a-glance reporting; (b) a composite reported alongside, never
instead of, the paired metrics. WHY THIS OVER (a): a composite here would bake in a governance judgment
(how much autoimmune risk buys how much CPIR) that varies legitimately by severity class and by an
Owner's actual risk tolerance, and collapsing it into one number would hide exactly the trade-off Part
XIV exists to keep visible. (b) was seriously considered and rejected only because an available composite
number, however clearly labeled as secondary, tends in practice to become the de facto reported figure —
a specific, named risk of composite scoring the estate has learned to distrust from other domains, and
this dataset chooses to close that path entirely rather than manage it. TRADE-OFFS: dashboards and
reports built against this dataset's metrics require slightly more sophisticated presentation (paired
figures, not a single number) than a composite would allow. REVERSAL CONDITIONS: if IAS-E1's
Observability Fabric later demonstrates a proven, non-gameable technique for safely compositing paired
metrics without hiding their trade-off, this decision is a candidate for revisiting jointly with that
sibling dataset — not unilaterally by IAS-D2 alone.

D5. DECISION: Immunization stops at PROPOSAL; deployment always requires Owner approval (Part 8.2, Part
24.3). ALTERNATIVES CONSIDERED: (a) auto-deploy antibodies for LOW/MEDIUM severity lineages with an
existing, already-proven remediation class (Part 9.2's high-confidence case), reserving Owner review for
HIGH/CRITICAL only; (b) require Owner approval uniformly, as specified. WHY THIS OVER (a): Constitution
Article IX's governed-self-evolution boundary applies without a severity carve-out, and Part 21.5's
antibody-forgery adversarial case demonstrates concretely why even a "proven" remediation class,
self-deployed without independent confirmation, is exploitable — the severity of the ORIGINAL pathogen
does not bound the risk of a SILENT, ungated deployment mechanism, which is a different risk surface
entirely. TRADE-OFFS: even high-confidence, low-severity immunizations wait on Owner or Owner-delegated
review, adding latency Part 8.5's proximity-to-recurrence prioritization exists partly to manage.
REVERSAL CONDITIONS: none within this dataset's own authority — per 24.3, this boundary is held fixed
unless the Constitution itself is amended.

D6. DECISION: Containment (Part XI) is specified as a SEPARATE mechanism from quarantine (Part VII)
rather than as quarantine's early, low-evidence variant folded into the same Part. ALTERNATIVES
CONSIDERED: (a) a single "quarantine" mechanism with a variable evidence threshold depending on urgency;
(b) two genuinely distinct mechanisms, as specified. WHY THIS OVER (a): a single mechanism with a
variable threshold blurs the exact evidence-level distinction Part 6.4 and Part 7.2 depend on for their
own gating logic, risking a drift where "urgent" quietly becomes the default justification for acting on
weaker evidence generally, rather than remaining the narrow, time-boxed exception Part 11.4's 48-hour
compensating control requires it to be. TRADE-OFFS: two mechanisms mean two specifications, two action
vocabularies (Part 7.3's four quarantine actions versus Part 11.3's four containment rungs), and a
transition point (rung three to standard quarantine) that must itself be kept correct. REVERSAL
CONDITIONS: if operational experience shows the two mechanisms are consistently used identically in
practice with no cases actually exercising containment's lower-evidence rungs independently, a future
revision could consider unification — a genuinely open, evidence-driven question this dataset does not
foreclose.

GAP CLOSED: prior to this dataset, the estate's failure-learning curve was per-project and reactive — a
root cause recurred independently in every project unlucky enough to suffer it, with no mechanism
propagating a fix from the project that discovered it to the projects that had not yet been harmed. This
dataset closes that gap by defining the ensemble-level object "cross-project immunity" and specifying the
complete pipeline — fingerprint, federate, scan, contain, quarantine, immunize, remember, decay-monitor —
that converts one incident's cost into protection for every structurally exposed sibling.

SYSTEMS STRENGTHENED: cascade_prevention (its in-session blocks now feed a cross-project federation
channel, Part 7.6, Part XVII); osa/CEPS (its per-session context-error detections become fingerprint
origins, Part 3.4, demonstrated concretely in Part XX's worked case); secret_firewall (its credential-

federated pattern class); hard_rules (immune-memory antibodies are frequently instantiated AS new
hard-rule entries in receiving projects, Part 9.8, and a hard-rule seal is a canonical fingerprint-
extraction trigger, Part 3.4); refcheck (a premise-verification failure is a qualifying origin event,

===============================================================================================
ANTI-PATTERNS
===============================================================================================

ANTI-PATTERN ONE — blind template propagation ("just copy Alpha's fix to Beta"). The most likely
implementation error, because it is the most intuitive shortcut: treating a remediation class (Part 3.2)
as a literal artifact to copy-paste across projects rather than a shape to re-instantiate against each
receiving project's own discovered structure (Part 8.4). PREVENTION: the immunization pipeline's
PROPOSAL stage (Part 8.2) must always be driven by the RECEIVING project's own exposure-scan findings,
never by the originating project's implementation details; any implementation that skips straight from
"fingerprint matched" to "apply Alpha's diff" has collapsed Part VIII's four-stage pipeline into one step
and reintroduced the exact founding-gap failure (Part 1.3) this dataset exists to close.

ANTI-PATTERN TWO — treating fingerprint existence as exposure proof. The second most likely error,
because the fingerprint and the exposure verdict are easy to conflate once a federation event has been
received: acting (quarantining, immunizing, or reporting a project as at-risk in an Owner-facing summary)
on the mere presence of a matching fingerprint in a project's federation feed, without running the actual
exposure scan (Part VI) that tests the project's real structure. PREVENTION: enforce Part 6.1's explicit
distinction — a fingerprint reaching a project is a WARNING requiring a scan, never a VERDICT — as a
structural gate in any implementation: no downstream action (Part VII, Part VIII) may read directly off
a federation event, every action must read off an exposure-map entry with an explicit evidence level
attached (Part 6.4).

ANTI-PATTERN THREE — auto-tuning thresholds against a single metric without its paired countermetric. The
third likely error, because it is the natural failure mode of any operator under pressure to "improve"
a visible number: raising CPIR (Part 15.2), lowering MTTF (Part 15.3), or increasing containment ratio
(Part 15.5) by adjusting a threshold or gate without checking the paired countermetric's simultaneous
movement — the precise Goodhart dynamic Part XV's entire pairing discipline exists to prevent.
PREVENTION: no threshold or gate defined anywhere in this dataset (Parts 5.4, 6.4, 7.2, 10.4, 11.3, 12.2)
may be revised, even as an Owner-approved proposal per Part 24.2, without the revision proposal explicitly
reporting BOTH members of its relevant metric pair, before and projected after — a revision proposal
naming only the metric it improves, silent on the countermetric, does not meet Part 24.2's evidence-
attachment requirement and should be rejected on that basis alone.
