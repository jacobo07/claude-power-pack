===============================================================================================
PART I — THE LEVERAGE OBJECT: DEFINING THE INSTITUTIONAL MULTIPLIER
===============================================================================================

1.1 The founding gap this dataset closes. Claude Power Pack, as OBSERVED across the estate audit
that produced the Non-Duplication Ledger, contains dozens of mechanisms that find and record a
single true thing: FD distills one session's delta into one deposit, hard_rules records one sealed
bug as one kill-switch, ACIS promotes one claim through one evidence ladder, compound-learnings
converts one recurring pattern into one skill or rule. Each of these mechanisms is, by IAS-F1's
Part VII test, first-order or borderline-second-order over a narrow object: it decides upon one
deposit, one rule, one claim. None of them asks a different, harder question that only becomes
askable once the estate is large enough to have real cross-system structure: of all the candidate
changes the estate could make this week, WHICH ONE, if made, would raise the output or reliability
of the largest number of OTHER systems at once? (DERIVED: this question cannot be phrased inside
any single learner, because the answer requires reading the relation graph across systems, which
is exactly the ensemble-level object IAS-F1 Part III reserves for the CPP-IAS family.) The gap is
not "we lack a place to record learnings" — the estate over-provisions recording mechanisms, as the
Non-Duplication Ledger's Family B candidates 11-20 attest, nine of ten of which duplicate an
existing recorder under a new name. The gap is a ranking and search function over the candidates
those recorders already produce: a function that asks not "is this true" (ACIS's question) or "is
this reusable" (FD's question) but "how many systems get better if this is acted on."

1.2 Leverage point, defined precisely. A LEVERAGE POINT, DEFINED in this corpus, is a candidate
intervention — a rule change, a schema fix, a shared vocabulary correction, a routing adjustment,
a newly-recombined capability — whose adoption is PREDICTED or OBSERVED to improve the function of
two or more systems that did not each independently require that exact intervention to be
discovered. The definition has three load-bearing clauses. First, "candidate intervention" ties
leverage to an ACTION, not to a fact: a leverage point is always a proposed or enacted change,
never a static observation, which is the precise line separating B1 from IAS-B2 (Insight
Synthesis), whose object is the insight itself rather than the change the insight licenses.
Second, "two or more systems" is the floor, not the target: a change that improves exactly one
system is not thereby disqualified from existing, but it is a first-order fix, and B1's distinct
object (§1.1) is the search for changes ABOVE that floor. Third, "did not each independently
require that exact intervention to be discovered" excludes coincidental multi-system benefit —
if System A and System B each independently proposed the identical fix through their own local
learners, the multi-system benefit is not leverage, it is convergent first-order maintenance, and
Part XVII's false-leverage catalogue names this the CONVERGENCE MIRAGE explicitly.

1.3 Multiplier and amplification factor, defined. Given a leverage point L, its MULTIPLIER, DEFINED,
is the count of DISTINCT systems (per IAS-F1 Part II's systemhood test) whose function is measurably
or predictably improved by L's adoption, where "measurably improved" is evidenced per Part IV's
measurement theory and "predictably improved" carries a HYPOTHESIZED marking until measured. A
multiplier of one is not leverage under §1.2; a multiplier of two is the minimum admissible leverage
point; multipliers observed in the Family B candidate review ranged, in illustrative synthetic
examples only, from two to the low double digits, because the estate's own strengthens-edge density
(IAS-F1 Part VI §6.10) is sparse by construction — most systems strengthen few others, so a leverage
point touching even five or six systems at once is, DERIVED, already in the tail of the estate's own
distribution, not a routine occurrence. The AMPLIFICATION FACTOR, DEFINED, is a second, independent
quantity: not how many systems L touches, but how much each touched system's own output quality or
cost improves, aggregated. A leverage point can have a high multiplier and a low amplification factor
(touches ten systems, improves each barely) or a low multiplier and a high amplification factor
(touches two systems, transforms each); Part IX's ranking model treats these as two axes of one
score rather than collapsing them prematurely, because collapsing them early is exactly how a
ranking function invites the false-leverage failure mode of Part XVII.

1.4 Why leverage is not size, and the size-vs-reach confusion named early. A natural but wrong
intuition equates "big change" with "high leverage": a rewrite of a core module feels more
consequential than a one-line vocabulary fix, and an institution that ranks by effort spent
mistakes the rewrite for the higher-value action. B1's object explicitly separates SIZE (lines
changed, hours spent, files touched — a property of the intervention itself) from LEVERAGE (systems
amplified — a property of the intervention's position in the relation graph). The clearest
illustrative case, described generically to respect Article VII's domain purity, is a single shared
vocabulary correction analogous to what IAS-F1 itself performs for the whole estate: F1 is, by word
count, a modest foundation document, but its multiplier — every system that resolves its own object
model against F1's definitions — is, by construction, ALL of them. A thousand-line refactor confined
to one module's internals, by contrast, has a multiplier of exactly one no matter how skillfully
executed, because IAS-F1 Part VIII's majority-object rule applies here too: what the refactor
DECIDES upon is one system's internals, not a relation between systems. (DERIVED: the size/leverage
independence is a direct corollary of the resource-level/ensemble-level distinction B1 inherits
from F1 Part VIII — a leverage point is, definitionally, closer to ensemble-level than a same-sized
resource-level fix, even when the two require identical engineering effort.)

1.5 Is B1 itself second-order, and why the answer matters for its own admissibility. Applying
IAS-F1 Part VII §7.2's mechanical test to B1's own capability: B1's direct object is not a resource
(a session, a rule, a deposit) but a SET OF CANDIDATE INTERVENTIONS considered in relation to the
systems they would touch — a decision over other capabilities' future states, which satisfies the
second-order test cleanly, unlike F1's own self-referential edge case (F1 Part VII §7.6, left
CONTESTED). B1 does not merely aggregate learnings (the AGGREGATOR NEAR-MISS of F1 Part VII §7.7);
it decides a RANKING over them and, per Part IX, can recommend one over another, which is precisely
the "held accountable for the choice" test F1 uses to separate a genuine second-order decision from
a lookup table. This matters operationally, not just taxonomically: because B1 clears the second-
order bar, its overlap with FD, ACIS, hard_rules, and the learners (§ header) is convertible under
Article I rather than a genuine duplication requiring deletion — B1 does not compete with FD's
distillation or ACIS's promotion, it reads their OUTPUT as candidate leverage points and ranks
across them, strengthening both without replacing either, exactly as the second-order capability
definition (F1 Part VII §7.4) predicts a properly-disposed duplication should behave.

FINAL LAW — PART I. A leverage point is a candidate intervention whose adoption improves two or
more systems that did not each discover it independently; its value is measured by how many systems
it amplifies and by how much, never by how large the change itself is — and the search for such
points, not the recording of any single true thing, is the object no existing PP learner owns.

===============================================================================================
PART II — WHY AMPLIFICATION IS NOT DISTILLATION: THE FD/B1 BOUNDARY, MADE EXPLICIT
===============================================================================================

2.1 The confusion this Part exists to foreclose, stated as a question. If FD already distills a
session's delta into a reusable deposit, and B1 ranks deposits (among other candidates) by how many
systems they amplify, is B1 not simply "FD with an extra sorting step"? This is the single most
likely misclassification of B1 among readers familiar with the estate, because both systems touch
"what did we learn and what should we do with it," and the Non-Duplication Ledger's own listing of
candidate 11 ("Cognitive Multiplier | FD (distillation) | distills one delta | finds the small
change amplifying many systems") shows the overlap was real enough to require an explicit verdict
rather than an automatic REJECT. This Part answers the question directly, mechanically, with a
worked trace, because a family whose founding distinction is asserted rather than demonstrated is
not trustworthy at the scale CPP-IAS operates.

2.2 FD's object, restated precisely and not re-narrated. FD's distinct object, as OBSERVED from its
own sealed documentation, is the SESSION-TO-DEPOSIT pipeline: a single session produces a single
delta (what changed, what was learned, what surprised the operator), and FD's job is to compress
that delta into a portable, reusable knowledge unit with provenance, without losing the delta's
substance and without inflating it with editorializing. FD's unit of work is ONE session; its unit
of output is ONE deposit; its success criterion is fidelity (did the deposit preserve the delta) and
portability (can the deposit be reused outside the session that produced it). FD does not, by design,
ask whether the deposit is more valuable than the deposit sitting next to it in the vault, and it
does not ask how many OTHER systems would benefit from the deposit being acted upon; that question
is outside FD's scope by the same ensemble-level/resource-level boundary (IAS-F1 Part VIII) that
separates every resource-level system from its ensemble-level sibling.

2.3 B1's object, restated against the same boundary. B1's unit of work is not a session; it is the
CANDIDATE POOL — every deposit FD has produced, every rule hard_rules has sealed, every claim ACIS
has promoted, every pattern a learner has surfaced, considered TOGETHER as a set of interventions
competing for scarce deep-work attention. B1's unit of output is not a deposit; it is a RANKING, and
the ranking's success criterion is not fidelity to any one input (each input is already trusted, per
§2.4, as FD's or ACIS's own responsibility) but ACCURACY OF PREDICTED MULTIPLIER — did the top-ranked
leverage point, once acted upon, actually improve as many systems as B1 predicted. This is a
categorically different success criterion from FD's, and DERIVED: a system cannot be "FD with sorting"
if its success criterion is not expressible in FD's terms at all — fidelity and multiplier-accuracy
are not the same axis, and optimizing one does not optimize the other.

2.4 The dependency direction, stated once and never reversed. B1 CONSUMES FD's deposits (and ACIS's
promoted claims, and hard_rules' sealed entries, and every learner's output) as RAW MATERIAL for its
candidate pool; it does not re-derive them, re-verify their internal fidelity, or duplicate FD's
compression work. This is a FEEDS relation in IAS-F1 Part VI §6.2 terms — FD feeds B1's candidate
pool — not a DEPENDS-ON relation, because B1 can still function, in a degraded sense, if FD's flow
of new deposits pauses (it re-ranks the existing pool); and it is emphatically not a DUPLICATES
relation, because B1's distinct-object test (§2.3) resolves to an answer FD's own object model never
touches. (NORMATIVE: any implementation of B1 that re-implements deposit compression, delta
extraction, or fidelity checking is out of scope and is, by this Part's own logic, duplicating FD —
the correct implementation surface for B1 is strictly downstream of FD's output.)

2.5 The reverse question: does FD need B1, or is B1 optional decoration? Before FD existed, and before
the estate had enough sealed deposits to have real cross-system structure, a ranking function over
"which deposit matters most" would have had nothing to rank — a candidate pool of one or two items
does not need a multiplier-ranking engine, it needs a human glance. B1's own strengthening claim on
FD (Strengthening Matrix row IAS-B1) is therefore conditioned on scale: FD strengthens best from B1
once its deposit count is large enough that manual triage of "which learning matters most this week"
exceeds what a human can hold in attention, which the estate's own OBSERVED deposit counts (dozens of
sealed session-lesson files, per the MEMORY.md ledger referenced by this mission's grounding) already
exceed. (DERIVED: B1's value to FD is a function of FD's OWN output volume, which is exactly the
"strengthens" relation IAS-F1 Part VI §6.6 describes — B1 does not feed FD data in the course of
normal operation, it makes FD's accumulated output more valuable by surfacing which parts of it are
worth acting on, a strengthens edge distinct from any feeds edge between the two.)

2.6 Worked trace: the same raw material, two different products. Consider a synthetic, illustrative
trace only. A session produces the observation "a hook that checks file staleness before Reading
also silently prevents three other hooks from firing, because they short-circuit on the same guard
clause." FD's product from this session is ONE deposit: "staleness-check short-circuit — hooks sharing
a guard clause can silently starve each other; verify the guard is scoped to the intended hook only."
That deposit is FD's entire contribution — compressed, provenance-tagged, portable. B1's product, taking
this deposit as ONE item in a pool of (illustratively) forty candidate deposits accumulated that
month, is a RANKING statement: "this deposit's predicted multiplier is four, because the guard-clause
pattern it names recurs, per the relation graph, in the secret-firewall hook chain, the scaffold-
auditor hook chain, the anti-thrash counter, and the output-contract validator — all four use a shared
guard-clause idiom the deposit's fix would correct in one pass — versus a sibling deposit this month
whose predicted multiplier is one." FD never produces that ranking statement; it is not FD's object.
B1 never produces the underlying deposit; it is not B1's object. The two systems' outputs are
different in kind, not merely in scope, which is the concrete, falsifiable form of the abstract
distinction §2.1-2.5 establish.

2.7 The reverse-boundary temptation: could FD simply rank its own deposits and skip B1 entirely? A
design alternative considered and rejected during this dataset's authoring, recorded here because
Article VI's design-decision discipline treats a seriously-considered-and-rejected alternative as
load-bearing context, not throwaway narration. The proposal: rather than standing up a separate
ensemble-level ranker, extend FD's own sealing step with a lightweight "how many other systems does
this deposit's guard-clause/vocabulary/schema fix touch" tag, computed at deposit-creation time and
sorted on later. This looks, superficially, like it would save a whole dataset's worth of engineering.
It fails for a reason DERIVED directly from §2.3's success-criterion argument: at the moment FD seals
a deposit, the candidate pool §2.3 defines does not yet exist in a form a single-session process can
see — FD's own object is bounded to ONE session's delta, and computing a defensible multiplier
requires comparing THIS candidate against the full accumulated pool of every other candidate currently
competing for the same scarce deep-work attention, a comparison that is, by construction, invisible
from inside any single session. A per-deposit tag computed at seal-time would therefore either be
a static, never-revisited guess (violating Part IV §4.3's reconciliation discipline, since the estate's
relation graph and candidate pool keep changing after the deposit is sealed) or would require FD to
re-open and re-score every prior deposit whenever a new one arrives — turning FD into a continuously-
running ranking engine in addition to its own compression job, which is precisely the SCOPE-CREEP this
dataset's own §21.5 later names explicitly when B1 declines a symmetric expansion into B2's territory.
The rejected alternative also fails a narrower, more concrete test: two FD deposits sealed in the same
week, each locally plausible as "high leverage" by its own author's intuition, frequently turn out,
once compared against the FULL pool including deposits from OTHER sessions the first deposit's author
never saw, to rank very differently — one dominates because its target system has since gained three
new strengthens edges from unrelated work, the other's predicted fan-out turns out, on graph traversal,
to terminate at depth one. Only a ranker with pool-wide visibility, updated on the graph's current
state rather than frozen at seal-time, can make that call, which is exactly the ensemble-level
visibility FD's own resource-level object was never built to hold. The rejected design is preserved
in the Decision Log below as DL-7, not because it was a bad-faith proposal, but because the boundary
it would have blurred is the single most likely misreading of this Part by a future maintainer under
time pressure.

FINAL LAW — PART II. FD compresses one session's delta into one portable deposit; B1 ranks the
accumulated pool of such deposits — and every other candidate intervention the estate produces — by
how many systems each would amplify if acted upon; the two success criteria, fidelity and multiplier-
accuracy, are not the same axis, and a system optimizing one is not thereby doing the other's work.

===============================================================================================
PART III — ONTOLOGY RESOLUTION: LEVERAGE POINTS IN THE F1 RELATION GRAPH
===============================================================================================

3.1 Why B1 must resolve against F1's typed graph rather than invent its own. IAS-F1 Part VI fixes
six typed relations (feeds, depends-on, competes-with, duplicates, strengthens, consumes) as the
CLOSED set the federation reasons with. A dataset that invents a seventh relation type to express
"amplifies" would fracture the very vocabulary F1 exists to unify, repeating the exact failure Part
I §1.2 of F1 itself warns against — three parallel measurement systems where one shared record would
do. B1 therefore does not add a relation type; it DEFINES leverage as a property of a PATTERN over
the existing six relations, specifically over the strengthens and feeds subgraphs, which is a
disciplined act of composition rather than an act of ontology inflation.

3.2 A leverage point as a strengthens-edge fan-out pattern. Formally, DEFINED here and consumed
throughout the rest of this dataset: given a candidate intervention L targeting a system S, L is a
leverage point of multiplier k if enacting L is predicted or observed to create or improve k distinct
strengthens edges outbound from S (or from whichever system L directly modifies) to k distinct
recipient systems, where each edge independently satisfies IAS-F1 Part VI §6.6's strengthens test
(the recipient produces more value, decides better, or costs less BECAUSE L was enacted). This
formalization is deliberately narrow: a leverage point is not any change that happens to touch many
files, and it is not any change with many feeds edges (feeds is the highest-cardinality, least
decision-critical relation per F1 Part VI §6.10, and a leverage-ranking function that counted feeds
fan-out would be dominated by noise). Anchoring leverage specifically to the strengthens subgraph —
the sparsest, most decision-critical relation type in F1's own taxonomy — is what keeps B1's
detection engine (Part VI) computationally tractable and semantically meaningful at once.

3.3 The second-order capability lens on the same pattern. Read through IAS-F1 Part VII instead of
Part VI, the same object appears from a different angle: a leverage point is, definitionally, a
candidate FIRST-ORDER change (it modifies one system's internals — a rule, a schema, a guard clause)
whose EFFECT is second-order (it alters the strengthens relation between that system and others).
This dual reading resolves a question that would otherwise recur throughout this dataset: is the
INTERVENTION itself second-order, or only its consequence? DERIVED, from F1 Part VII §7.2's own
"direct object" test applied strictly: the intervention's direct object is typically first-order
(fix a guard clause), but B1's OWN capability — deciding which first-order intervention to
prioritize based on its second-order consequence — is what clears the second-order bar (already
established in Part I §1.5). A leverage point is therefore best understood not as "a second-order
change" but as "a first-order change with a second-order shadow," and B1's engine is built to detect
the shadow, not to reclassify the change itself.

3.4 Resolving "multiplier" against F1's property model. IAS-F1 Part IV fixes six canonical
properties per system (cost, evidence level, reuse horizon, model-dependence, consumers, authority).
B1's multiplier is not a seventh property bolted onto every system; it is a property of the
CANDIDATE INTERVENTION, computed by counting how many systems' CONSUMERS-weighted strengthens fan-out
would change if the intervention were enacted — meaning B1 reads F1's existing property model rather
than extending it. (NORMATIVE: any implementation that adds a "leverage score" field to the system
property record itself, rather than keeping it as a property of the CANDIDATE, conflates a static
system attribute with a dynamic, intervention-specific prediction, and risks staleness exactly as
F1 Part IV §4.3 warns against for every other property — a system's leverage exposure changes with
every new candidate proposed against it, far faster than its cost or evidence level changes.)

3.5 The invariants B1 must respect, inherited from F1 Part V. B1's ranking function operates entirely
within F1's five invariants and violates none of them by design: it does not create orphan systems
(Invariant One) — a leverage point by definition creates or strengthens edges, moving a system away
from orphan status, never toward it; it does not adjudicate duplicates (Invariant Two) — that
remains D2A's and, within CPP-IAS, the ledger's job, and B1 treats an unresolved duplicate in its
candidate pool as a candidate whose multiplier is UNRESOLVED rather than guessed, per F1 Part IV
§4.6's treatment of unmeasured properties; it does not grant or revoke authority (Invariant Three)
— B1 recommends, it does not decide with final authority, which Part X's decision model states
explicitly; and it respects the boundary invariant (Invariant Four) by scoring only systems inside
the estate's ensemble boundary, never the host or the model provider. The one invariant B1 actively
SERVES, rather than merely respects, is Invariant Five (property honesty): because a leverage
score is only as good as the property records (cost, consumers, evidence level) it is computed
from, B1 has a direct institutional interest in those records being freshly reconciled, and Part
XIX's metrics include a check on the staleness of the inputs a leverage claim was computed from.

3.6 Worked resolution: naming one synthetic candidate fully in F1's terms. Consider, illustratively,
a candidate intervention L: "standardize the epistemic-marking vocabulary check across every dataset-
authoring pipeline so DEFINED/OBSERVED/DERIVED/HYPOTHESIZED/NORMATIVE/EXPERIMENTAL/CONTESTED/
UNRESOLVED is validated the same way everywhere." Resolved against F1: L's direct object is a
validation rule (first-order, targets one shared component). Its predicted strengthens fan-out
touches, illustratively, ACIS (whose entire law schema depends on consistent evidence-level marking),
hard_rules (whose digest compiler assumes a stable vocabulary), FD (whose deposits carry evidence
levels), and B1 itself (whose own multiplier calculations are only as trustworthy as the evidence
levels feeding them) — four strengthens edges, multiplier four, none of them a feeds edge and none
a duplicate. Its level, per F1 Part VIII's formal test, is ENSEMBLE, because what it decides upon —
a shared validation rule consumed across systems — is not enumerable within any single system's
owned resource class. This worked resolution is the template Part VI's detection engine automates
at scale: name the direct object, trace the predicted strengthens fan-out, count distinct recipients,
check the level test, and only then assign a provisional multiplier.

3.7 The cold-start problem: a leverage point targeting a system with no recorded strengthens edges
yet. A newly-onboarded system, or one whose relation-graph entries have not yet been reconciled since
onboarding, may show ZERO outbound strengthens edges in F1's graph — not because it genuinely
strengthens nothing (a violation of Invariant One, per Part III §3.5, which the estate treats as rare
and worth flagging), but because the graph itself has not yet caught up to the system's real position
in the estate. Left unhandled, the scanner (Part VII) would report a provisional multiplier of zero
for any candidate targeting such a system, which reads, to Part IX's ranker, identically to "this
candidate has no leverage" — a false negative indistinguishable from a true one purely because the
INPUT graph is incomplete rather than because the CANDIDATE is weak. B1's resolution, NORMATIVE: a
target system with zero recorded strengthens edges AND a LAST-RECONCILED timestamp (F1 Part IV §4.3)
older than the estate's own onboarding-grace-period is flagged GRAPH-INCOMPLETE rather than scored as
multiplier zero, and a GRAPH-INCOMPLETE candidate is excluded from Part IX's confident ranking tiers
entirely — not penalized as low-leverage, but marked UNSCOREABLE pending graph reconciliation, which
is a materially different disposition from REJECTED (Part V §5.3) and must never be silently collapsed
into it. This is the ontology-resolution-level instance of the same discipline Part VII §7.5 applies at
the scanner-mechanism level to contested or unresolved graph nodes: an absence of evidence about the
graph's own completeness is not evidence of the candidate's absence of leverage, and the two must
remain distinguishable all the way through the ranking pipeline, from ingestion (Part VI §6.3) through
to the leverage record's own status field. (DERIVED: this follows directly from F1 Part IV §4.6's
treatment of unmeasured properties — a system whose CONSUMERS or strengthens-fan-out property is
simply unreconciled is UNRESOLVED, not zero, and B1 inherits that same discipline rather than
re-deriving a laxer one of its own.) Worked illustration: a system onboarded three days prior, still
within the estate's typical reconciliation window, shows zero outbound edges; a candidate targeting it
is flagged GRAPH-INCOMPLETE and re-queued for scanning once the system's next reconciliation cycle
completes, rather than being scored, ranked, and silently buried at the bottom of Part IX's tiers as if
the estate had already concluded the system amplifies nothing — a conclusion the graph's own youth
never actually supports.

FINAL LAW — PART III. A leverage point is not a new relation type layered onto F1's ontology; it is
a pattern read from the existing strengthens subgraph — a first-order intervention with a second-
order shadow — and every multiplier B1 reports must be traceable to named, typed edges in F1's graph,
never asserted as a bare number with no graph behind it.

===============================================================================================
PART IV — AMPLIFICATION MEASUREMENT THEORY: HOW ONE CHANGE RAISES N SYSTEMS
===============================================================================================

4.1 The measurement problem, stated honestly. Counting strengthens edges (Part III) tells B1 how
MANY systems a leverage point touches, but it does not, by itself, tell B1 whether the touch is real
or predicted, large or marginal, durable or transient. A ranking function built on unqualified edge
counts is exactly the kind of ratio-blind, evidence-blind measurement this corpus's own governing
doctrine (imported by reference from the estate's broader governance, never re-litigated here) warns
against: a count with no evidence level attached is a hypothesis wearing a number's clothing. This
Part specifies HOW amplification is measured, not merely what is counted, because the measurement
method is what separates B1 from a spreadsheet that sums strengthens edges and calls the sum
"leverage."

4.2 Three measurement modes, ordered by evidentiary strength. B1 recognizes three distinct modes for
establishing that a leverage point amplifies a given recipient system, and marks every claim with
the mode that produced it. RETRODICTIVE measurement, the strongest, applies to a leverage point
already enacted: B1 compares the recipient system's OBSERVED property values (cost, consumers,
evidence level — F1 Part IV) before and after the intervention, against the same system under
otherwise-unchanged conditions, and a positive delta attributable to the intervention is marked
OBSERVED. PROSPECTIVE-STRUCTURAL measurement applies to a not-yet-enacted candidate: B1 traces the
relation graph (Part III §3.2) to identify which systems WOULD receive a strengthens edge if the
intervention were made, without yet measuring the edge's magnitude, and marks the resulting
multiplier DERIVED (it follows by stated graph-traversal reasoning from OBSERVED graph structure)
but not yet OBSERVED as an effect. ANALOGICAL-HISTORICAL measurement, the weakest admissible mode,
applies when a candidate resembles a past leverage point closely enough that the past point's
retrodictive multiplier is used as a prior; this mode is always marked HYPOTHESIZED, never promoted
silently to DERIVED, because resemblance is not identity and the estate has, per Part XVII, an
OBSERVED history of analogical over-confidence.

4.3 Why retrodiction is the only mode that can promote a candidate to a trusted ranking tier. Part IX's
tier model reserves its top tier for leverage points with at least one RETRODICTIVE data point,
because prospective-structural measurement, however rigorously graph-traced, is a prediction about a
strengthens edge that has not yet been tested against the recipient system's actual behavior — and
F1's own property model (Part IV §4.3) treats an unreconciled claim as inherently staler than a
reconciled one. This is not a preference for caution over ambition; it is DERIVED directly from the
false-leverage failure mode (Part XVII): a structurally-plausible fan-out that never gets confirmed
against real system behavior is indistinguishable, from inside the ranking function, from a fan-out
that was never real in the first place, and only retrodiction closes that gap.

4.4 The counterfactual requirement: amplification versus mere co-occurrence. A recipient system's
metrics improving in the same period a leverage point was enacted is not, by itself, evidence of
amplification — the estate changes continuously, and attributing an improvement to one intervention
among several concurrent ones is the classic co-occurrence trap. B1's measurement protocol therefore
requires, wherever feasible, a COUNTERFACTUAL COMPARISON: the recipient system's trajectory is
compared not merely to its own past, but to a baseline period, or to a structurally similar system
NOT touched by the intervention, before a delta is attributed. (NORMATIVE: where no counterfactual
is available — a common situation in a single evolving estate with no control group — B1 marks the
attribution CONTESTED rather than OBSERVED, and Part XX's evaluation protocol treats an unqualified
before/after claim with no counterfactual as insufficient evidence for a top-tier ranking, exactly
the discipline the estate's DRK precedent work already applies to decision attribution elsewhere.)

4.5 Aggregation: from N individual edge measurements to one leverage score. Once each candidate
strengthens edge carries its own measurement mode and, where available, magnitude, B1 must aggregate
them into the ranking inputs Part IX consumes. The aggregation is NOT a simple sum, for two reasons
DERIVED from §4.2-4.4: first, edges of different evidentiary strength are not fungible — four
HYPOTHESIZED edges do not equal one OBSERVED edge, and summing them as if they did would let a
speculative candidate with a wide, unverified fan-out outrank a modest candidate with a narrow,
proven one; second, edges to systems with high existing consumer counts (F1 Part IV property) carry
more institutional weight than edges to lightly-consumed systems, because amplifying a widely-relied-
upon system compounds further than amplifying an edge system. B1's aggregation therefore weights each
edge by (evidentiary-mode weight) × (recipient consumer count), and reports the aggregate alongside
its own confidence interval rather than as a single bare number — the confidence interval itself
being a first-class output, not a footnote, because Part IX's decision model reads it directly.

4.6 Worked example: two candidates compared honestly under the measurement protocol. Candidate M has
a multiplier of six computed entirely by prospective-structural graph tracing (DERIVED, zero
retrodictive points, moderate confidence). Candidate N has a multiplier of two, but one of the two
edges is OBSERVED via retrodiction against a near-identical intervention made three months prior, and
the other is prospective-structural with high consumer-count weighting. Under naive edge-counting, M
outranks N three-to-one. Under B1's weighted, evidence-leveled aggregation, N's score is not
automatically higher, but its confidence interval is materially tighter, and Part IX's decision model
(§9.4) treats "high confidence, modest score" and "low confidence, high score" as genuinely different
recommendations — the first is safe to act on now, the second is a candidate for the Recombination
Lab's controlled-experiment protocol (Part XIII) rather than immediate adoption. This is the concrete
payoff of treating measurement mode as a first-class dimension rather than collapsing everything into
one number: two candidates that would tie or invert under naive counting are correctly distinguished
by what the estate actually knows about each.

4.7 Why measurement mode, not raw multiplier, is the field Part IX's ranker treats as primary. A
reasonable design alternative to this Part's whole framework would rank candidates on raw multiplier
first, using measurement mode only as a secondary sort key among ties. This dataset rejects that
ordering, NORMATIVE, for a reason DERIVED directly from §4.3's promotion logic: a raw-multiplier-first
ranking would, by construction, treat a wide, entirely prospective-structural fan-out as more valuable
than a narrow, retrodictively confirmed one whenever the former's edge count exceeds the latter's — the
exact inversion §4.6's worked M-vs-N comparison exists to prevent. Measurement mode is therefore not a
tiebreaker layered on top of the score; it is baked into the weighted aggregation itself (§4.5), meaning
the two candidates are never actually tied on the raw axis before mode-weighting is applied — the
weighting is prior to, not subsequent to, the comparison. This ordering decision is revisited explicitly
in the Decision Log, because it is one of the few points in this dataset's design where a naive,
intuitively simpler alternative was seriously considered and rejected on stated evidentiary grounds
rather than dismissed without argument.

FINAL LAW — PART IV. Amplification is measured, never assumed, along three evidentiary modes —
retrodictive, prospective-structural, analogical-historical — and aggregated by weighting each edge
by its evidentiary strength and its recipient's consumer count, mode-weighting applied prior to
comparison rather than after; a leverage score with no measurement mode attached is not a measurement,
it is a guess wearing a decimal point.

===============================================================================================
PART V — THE AMPLIFICATION LEDGER: OBJECTS, PROPERTIES, AND THE LEVERAGE RECORD
===============================================================================================

5.1 Why a ledger, and why it is a record type, not a database implementation. Parts I-IV establish
what a leverage point is and how its amplification is measured; this Part specifies the shared
RECORD SCHEMA those measurements are written into — the LEVERAGE RECORD — because a measurement
method with no fixed record shape produces the same fragmentation F1 Part IV warned about for system
properties: every consumer of B1's output (Part IX's ranker, Part XI's synergy detector, Part XXI's
integrations) needs one interchange format, not one format per consumer. As Article VIII requires,
this is specified as a record schema — fields, types, and constraints in prose — never as
implementation code.

5.2 The leverage record's fields. Every leverage record, DEFINED, carries: a STABLE CANDIDATE ID
(assigned at first detection, never reused, mirroring F1 Part IX's system-address discipline so a
candidate can be tracked across re-scoring cycles); a SOURCE REFERENCE (which FD deposit, ACIS claim,
hard_rules entry, or learner output the candidate originated from, preserving provenance back to the
resource-level system that produced the raw material, per Part II §2.4's feeds relation); a TARGET
SYSTEM (the system the intervention directly modifies); a PREDICTED-RECIPIENT SET (the systems
Part III's graph trace identifies as receiving a strengthens edge, each tagged with its own
measurement mode per Part IV §4.2); an AGGREGATE MULTIPLIER SCORE with CONFIDENCE INTERVAL (Part IV
§4.5's weighted output); a STATUS field (PROPOSED, UNDER-EVALUATION, ADOPTED, REJECTED, or
SUPERSEDED); and a LAST-RECONCILED timestamp, inherited directly from F1's property-record
discipline (Part IV §4.3), because a leverage score computed from stale system properties is itself
stale the moment any input property is reconciled to a new value.

5.3 Why status is a first-class field, and the state transitions it governs. A leverage record's
STATUS is not decoration; it is what lets B1 avoid re-scoring the same candidate on every cycle and
what lets Part XX's evaluation protocol distinguish a claim that was ACTED ON from one that merely
sits in the pool. The legal transitions, NORMATIVE: PROPOSED → UNDER-EVALUATION when Part IX's
ranker selects it for deeper (possibly retrodictive) measurement; UNDER-EVALUATION → ADOPTED when
the Owner or an authorized process enacts the intervention (per Article IX, IAS never self-adopts);
UNDER-EVALUATION → REJECTED when deeper measurement fails to confirm the predicted fan-out or when
Part XVII's false-leverage checks flag the candidate; ADOPTED → SUPERSEDED when a later candidate
subsumes or replaces an adopted one. A record that skips UNDER-EVALUATION and moves directly from
PROPOSED to ADOPTED has bypassed the measurement discipline Part IV exists to enforce, and Part XV's
anti-pattern catalogue names this the SKIP-THE-EVALUATION-TIER failure explicitly.

5.4 The ledger as a shared interchange, and the duplication it prevents. Without this shared schema,
the natural failure mode mirrors F1 Part IV §4.4 exactly: Part IX's ranker would compute its own
ad hoc multiplier representation, Part XI's synergy detector would compute a separate one for pairs
of candidates, and Part XIII's recombination lab would track experimental candidates in a third,
incompatible shape — three parallel bookkeeping systems inside ONE dataset, which is the intra-
dataset analogue of the cross-dataset duplication Article I exists to prevent. By fixing one leverage-
record schema all of B1's own subengines read and write, this Part converts what would have been
three internal measurement systems into three consumers of one shared record, the same mechanism
F1 uses at the ensemble scale, applied here at the dataset-internal scale.

5.5 Provenance chains and the multi-hop candidate. Some candidates are not traceable to one source
but to a CHAIN: an FD deposit that itself synthesizes three prior session lessons, later promoted by
ACIS into a claim, later still surfaced by a learner as a recurring pattern. B1's source reference
field, DEFINED, is not restricted to the immediate producer; it carries the FULL provenance chain
back to the earliest resource-level origin, because a leverage point's multiplier calculation in
Part III sometimes depends on knowing whether the candidate has already been independently
strengthened along its chain (in which case some of its apparent multiplier may already be realized,
not prospective) — a distinction Part XVII's DOUBLE-COUNTED AMPLIFICATION failure mode depends on the
ledger being able to draw.

5.6 Worked example: one candidate's full leverage record over its lifecycle. A candidate born as an
FD deposit ("guard-clause starvation," Part II §2.6) enters the ledger PROPOSED, source-referenced to
the originating deposit, target system the shared hook-dispatcher component, predicted-recipient set
of four systems each tagged prospective-structural. Part IX's ranker selects it for deeper evaluation;
one of the four predicted edges gets a retrodictive check against a near-identical prior fix (Part
IV §4.6's Candidate N pattern), tightening its confidence interval; status moves to UNDER-EVALUATION.
The Owner reviews and adopts it; status moves to ADOPTED, and the record's last-reconciled timestamp
resets. Six weeks later, a broader fix supersedes it; status moves to SUPERSEDED, but the record is
never deleted — per Part XIV's lifecycle discipline, a superseded record remains in the ledger as a
retrodictive data point for FUTURE candidates resembling it, which is precisely how Part IV's
analogical-historical measurement mode gets a growing evidence base over time rather than starting
from zero on every cycle.

5.7 The duplicate-candidate merge problem, and why the ledger must resolve it before ranking sees it.
Two independent producers occasionally surface the SAME underlying leverage point through different
channels — an FD deposit and, separately, a learner's pattern capture, both naming what is, on
inspection, one intervention against one target system. Left unresolved, both enter the ledger as
distinct candidate IDs (Part V §5.2), and Part IX's ranker would then see two records competing for
attention where only one genuine leverage point exists — not a duplication of SYSTEMS in F1's Part V
Invariant Two sense, but a duplication of CANDIDATES, a narrower, ledger-internal analogue of the same
problem. B1's resolution, NORMATIVE: before a new candidate is scanned (Part VII), the ledger checks its
stated target system and its source description against EXISTING PROPOSED or UNDER-EVALUATION records
targeting the same system; a sufficiently close match (same target, materially overlapping predicted-
recipient description) triggers a MERGE, not a second independent record — the two provenance chains
(Part V §5.5) are both preserved under one candidate ID, crediting both originating producers, but the
scanning, estimation, and ranking work proceeds once, against the merged record, never twice against
two records that would have produced two slightly different, competing multiplier estimates for what is
actually one true leverage point. This merge discipline is the ledger-internal application of the same
principle Part XI's synergy detector uses to distinguish real joint effects from coincidence: a shared
target and a shared predicted mechanism is evidence of ONE thing observed twice, not two things, and
treating it as two inflates the ledger's own apparent candidate volume — a subtle instance of the
volume-versus-value confusion Part XVI's local-optimum catalogue warns against, occurring here at
ledger-ingestion time rather than at ranking time. A merge that turns out, on deeper scanning, to have
been premature — the two producers' candidates in fact target subtly different components with
genuinely different predicted-recipient sets — is reversible: the ledger splits the merged record back
into two, each retaining its own half of the shared provenance history, rather than being locked into an
incorrect merge simply because reversing a ledger operation feels more costly than making one.

FINAL LAW — PART V. Every leverage claim lives in one shared record — candidate ID, provenance chain,
target, predicted-recipient set with measurement modes, aggregate score with confidence interval,
status, and reconciliation timestamp — and a multiplier asserted outside this record is not yet a
claim the ledger, or any downstream consumer, can be held to.

FINAL LAW — HEADER SECTION CLOSE. Parts I through V establish the object, its boundary against FD,
its resolution against F1's ontology, its measurement theory, and its record schema; Parts VI onward
specify the engine that finds, ranks, and acts on leverage points at scale.

===============================================================================================
PART VI — THE MULTIPLIER DETECTION ENGINE: ARCHITECTURE AND SIGNAL COLLECTORS
===============================================================================================

6.1 Why detection is an engine and not a query. Finding leverage points is not a single lookup
against the ledger of Part V, because the ledger only holds candidates ALREADY surfaced by some
resource-level producer (FD, ACIS, hard_rules, a learner); the pool of things that COULD be leverage
points but have not yet been noticed by any first-order system is, by construction, larger than the
pool already recorded. The MULTIPLIER DETECTION ENGINE, DEFINED, is the subsystem that (a) ingests
newly-produced candidates from the estate's resource-level producers, (b) actively searches the F1
relation graph for structural patterns that PREDICT undiscovered leverage even where no producer has
yet flagged one, and (c) writes both kinds of finding into the leverage-record schema of Part V. It
is an engine, not a query, because step (b) is an ongoing search process over a graph that changes
continuously, not a one-shot lookup with a fixed answer.

6.2 The two-collector architecture, and why detection is split rather than unified. B1's detection
engine is composed of exactly two signal collectors, each covering a distinct evidentiary source, per
the SUBENGINE disposition the Non-Duplication Ledger assigns to the recombination lab (candidate 19)
and the merge disposition it assigns to the synergy detector (candidate 20) — both of which are, in
fact, specialized collectors under this same engine rather than freestanding systems, which is the
concrete sense in which B1 "absorbs" rather than merely references them. The FAN-OUT SCANNER (Part
VII) is the passive collector: it watches the ledger's inbound candidate stream and the F1 relation
graph for structural fan-out, per Part III's pattern definition, without requiring any human or
learner to have already suspected leverage. The COUNTERFACTUAL ESTIMATOR (Part VIII) is the active
collector: it is invoked, not continuously running, against a specific candidate already flagged by
the scanner, and it performs the deeper retrodictive and counterfactual measurement work of Part IV
§4.3-4.4 that is too expensive to run against every candidate unconditionally.

6.3 The ingestion boundary: what counts as a candidate worth scanning. Not every FD deposit, ACIS
claim, or hard_rules entry is worth the engine's attention; an intervention whose direct object
(IAS-F1 Part VIII §8.2) is unambiguously resource-level, touching one system with no plausible
cross-system relation, is filtered before it reaches the fan-out scanner, because running the full
detection pipeline against every first-order fix would drown the engine's signal in noise and would,
per Part XVII's failure catalogue, invite the AGGREGATOR NEAR-MISS at engine scale — treating volume
of candidates processed as a proxy for leverage found. The filter, NORMATIVE, is deliberately
permissive rather than restrictive at this stage (a false negative here means a real leverage point
is missed entirely, while a false positive merely costs the scanner one extra graph traversal), so
the ingestion boundary excludes only candidates whose target system is OBSERVED, per F1's property
model, to have zero outbound strengthens or feeds edges at all — a genuinely isolated system, which
Part III §3.5 already notes should be rare under Invariant One.

6.4 The engine's relationship to the ledger: read-heavy, write-disciplined. The detection engine
reads the full leverage ledger and the full F1 relation graph on every cycle, but it writes only new
or updated leverage records, never mutating a record's provenance chain (Part V §5.5) once written,
because a provenance chain is a historical fact, not a live computation. (NORMATIVE: an
implementation that allows the detection engine to silently rewrite a candidate's source reference
when a new, more convenient producer is found is destroying the audit trail Part XX's evaluation
protocol depends on; the correct operation is to add a NEW leverage record cross-referencing the
prior one, never to overwrite.) This write discipline is the same append-preferred posture the
estate already applies to its hard_rules archive and its session-lesson ledger, imported here by
analogy rather than by mechanism, because the underlying reason — provenance must survive revision —
is the same reason in both cases.

6.5 Cadence and the cost of running the engine continuously. Running the fan-out scanner against
every new candidate as it arrives is cheap (a bounded graph traversal per Part III §3.2's pattern),
but running the counterfactual estimator against every scanner hit is expensive (it requires
comparing recipient system trajectories, per Part IV §4.4, against a baseline or control), so the
engine's cadence is two-speed by design: the scanner runs on EVERY new candidate, continuously; the
estimator runs only on candidates the scanner has flagged AND that Part IX's ranker has selected for
deeper evaluation, converting the ledger's UNDER-EVALUATION status transition (Part V §5.3) into the
literal trigger for the estimator's invocation. This two-speed cadence is DERIVED from the same
evidentiary-strength asymmetry Part IV §4.3 establishes: cheap, wide, prospective-structural coverage
first; expensive, narrow, retrodictive confirmation second, only where the ranking model judges the
confirmation worth its cost.

6.6 Failure isolation: what happens when the engine is wrong about a fan-out. The scanner's structural
prediction (Part IV's prospective-structural mode) is, by definition, unconfirmed until the estimator
or a real adoption event checks it; the engine must therefore be built to fail SAFELY when a predicted
edge turns out not to exist — the wrong prediction downgrades that one edge's status in the leverage
record (Part V §5.2) without invalidating the candidate's other predicted edges or corrupting the
ledger's other records. This is a straightforward consequence of Part V's record-per-candidate,
edge-per-recipient schema, but it is worth stating explicitly as an engine requirement because a
naive implementation that computes one monolithic multiplier score per candidate, rather than
per-edge scores that aggregate, would have no way to isolate a single wrong prediction from the
candidate's overall credibility — one bad edge would poison the whole score, which is precisely the
opposite of the graceful, edge-level correction this Part requires.

6.7 The engine's own leverage: does the detection engine amplify itself. Applying B1's own object
(Part I §1.2) reflexively: the detection engine is a system inside B1, and an improvement to the
engine — a better graph-traversal heuristic, a cheaper estimator invocation strategy — is itself a
candidate leverage point, because it would improve every future scoring cycle across every candidate
the engine ever processes. B1 does not exempt itself from its own ranking discipline; an internal
engine improvement enters the SAME leverage ledger, source-referenced to B1's own operational
history rather than to an external producer, and competes for evaluation priority against every
other candidate exactly as Part IX's ranker requires. (NORMATIVE: this reflexivity is a deliberate
design choice, not an accident — a leverage-discovery engine that treated its own improvement as
categorically special, outside its own ranking discipline, would be claiming an exemption its own
object model gives it no principled basis to claim.)

6.8 Worked example on synthetic data: one candidate through both collectors. Consider a synthetic
candidate, OBSERVED-for-illustration only, generated when hard_rules seals a rule stating that a
particular retry-backoff constant must never be set below 200ms, with a provenance note citing three
prior incidents across unrelated hosts. On ingestion (§6.3), the candidate clears the filter because
its direct object — the backoff constant — is referenced, per the F1 relation graph, by outbound
strengthens edges from the hard_rules system into four other systems that each embed their own retry
logic: a deployment gate, a background-task scheduler, a session-recovery watchdog, and a hook
dispatcher. The fan-out scanner (§6.5, continuous speed) traverses those four edges in a single bounded
pass and writes a leverage record with four PROSPECTIVE-STRUCTURAL edges, per Part IV §4.3's mode
taxonomy, because none of the four recipient systems has yet actually adopted the corrected constant —
the scanner has only found that the graph SAYS they could. Part IX's ranker (consumed, not restated,
here) then scores the four-edge record highly enough to trigger the counterfactual estimator, which is
invoked (§6.5, active speed) against each of the four recipients in turn: it finds that the deployment
gate and the session-recovery watchdog already independently converged on a 250ms floor through
unrelated bug fixes (downgrading those two edges to CONFIRMED-REDUNDANT, a null but honestly-recorded
result), while the background-task scheduler and the hook dispatcher had not, and adopting the corrected
constant in both measurably reduced their retry-storm incident rate in the following month
(CONFIRMED-POSITIVE, per Part IV §4.4's retrodictive mode). Per §6.6's failure-isolation requirement,
the two redundant edges do not lower the record's overall credibility below what the two confirmed
edges independently earn — the record correctly reports "2 of 4 predicted edges confirmed, 2 already
redundant," never a single blended score that would misleadingly average a real leverage finding
against a coincidental non-finding.

6.9 A detection-engine metric and its anti-Goodhart countermetric. The natural engine-health metric —
candidates fully processed per cycle — is exactly the metric §6.3 already warns against optimizing for
directly, because a scanner tuned to maximize throughput will favor cheap, shallow fan-out patterns
over the rarer, deeper ones the counterfactual estimator is built to confirm, silently converging the
engine toward volume over leverage quality even though no single design decision intended that drift.
The corrective countermetric, DEFINED, is CONFIRMED-EDGE YIELD PER ESTIMATOR INVOCATION — the fraction
of estimator-invoked edges that resolve CONFIRMED-POSITIVE (§6.8) rather than CONFIRMED-REDUNDANT or
CONFIRMED-ABSENT, tracked as a trailing ratio rather than a raw count so that it cannot be inflated by
simply invoking the estimator more often. A healthy engine keeps candidates-processed rising over time
(more of the estate's activity is being scanned) while confirmed-edge yield stays roughly flat or rises
(the estimator's invocations remain worth their cost); a throughput-gamed engine shows candidates-
processed rising while yield falls, which is the detection signal, per Part XIX's countermetric
convention, that the scanner's filter (§6.3) or the ranker's threshold (Part IX) has drifted toward
noise rather than signal and needs recalibration before the ledger accumulates low-value records that
crowd out genuine leverage points in Part IX's evaluation queue.

FINAL LAW — PART VI. The detection engine is two collectors at two speeds — a continuous, cheap
fan-out scanner and an invoked, expensive counterfactual estimator — reading the shared ledger and
writing to it append-preferred, filtering only genuinely isolated candidates, and subject to its own
ranking discipline exactly as every external candidate is; an engine exempt from its own rules is not
a leverage-discovery engine, it is a privileged first-order system wearing a second-order name.

===============================================================================================
PART VII — SUBENGINE ONE: THE FAN-OUT SCANNER
===============================================================================================

7.1 The scanner's precise job, distinguished from the engine's job. Part VI names the fan-out
scanner as the continuous, cheap collector; this Part specifies its mechanism. The scanner's job,
DEFINED narrowly, is: given a new or updated candidate intervention with a stated or inferable
target system, traverse the F1 relation graph outward from that target system along STRENGTHENS
and (secondarily) FEEDS edges to a bounded depth, and report every distinct system reached as a
PREDICTED recipient, each tagged prospective-structural (Part IV §4.2) with a provisional multiplier
equal to the count of distinct systems reached. The scanner does not measure magnitude, does not run
counterfactuals, and does not decide whether the candidate is worth adopting — those are, respectively,
the estimator's job (Part VIII), the ranker's job (Part IX), and the Owner's job (Article IX).

7.2 Why the traversal is bounded, and what the bound protects against. An unbounded traversal along
strengthens and feeds edges would, in a sufficiently connected estate, eventually reach nearly every
system through some chain of intermediate relations, producing an inflated multiplier for almost
every candidate — the exact SIZE-INFLATION failure IAS-F1 Part XIV names at the ontology level,
recurring here at the engine level. The scanner therefore bounds its traversal to a small fixed
depth (illustratively, two or three hops from the target system), NORMATIVE, chosen because a
strengthens relation two hops removed from the direct target is already a materially weaker claim of
amplification than a direct edge, and Part IV §4.5's aggregation weighting further discounts edges
found at greater depth, so the bound and the weighting work together rather than either alone
carrying the full burden of preventing inflation.

7.3 Direct edges versus chained edges, and why they are marked differently. A system reached at
depth one (a direct strengthens edge from the target) is a materially stronger claim than a system
reached at depth two (the target strengthens an intermediate system, which itself strengthens the
recipient) — the chained claim inherits uncertainty from BOTH edges in the chain, and the scanner
marks each predicted recipient with its traversal depth as part of the leverage record's predicted-
recipient entry (Part V §5.2), so Part IX's ranker can, and does, weight depth-one hits more heavily
than depth-two hits without needing to re-traverse the graph itself. (DERIVED: this is the scanner-
level instantiation of the general principle, established in Part IV §4.5, that not all edges are
fungible — depth is simply a second axis of the same evidentiary-strength discipline, alongside
measurement mode and recipient consumer count.)

7.4 The scanner's blind spots, named honestly. The scanner can only find leverage along edges the
F1 relation graph ALREADY records; a genuinely novel strengthens relation — one no system has yet
observed, because the two systems in question have simply never before been considered together —
is invisible to a pure graph traversal, no matter how the traversal is tuned. This is not a defect
to be engineered away within the scanner's own scope; it is the precise reason Part XIII's
Recombination Lab exists as a SEPARATE mechanism, one that does not traverse the existing graph but
deliberately proposes NEW relations between systems that have no recorded edge at all. The scanner
finds leverage in the graph as it stands; the lab finds leverage the graph does not yet know exists.
(NORMATIVE: an implementation that expects the scanner alone to find all leverage in the estate has
mistaken a graph-traversal tool for a creativity tool, and Part XV's minimal-sufficient-doctrine
material returns to exactly this expectation-boundary in its own terms.)

7.5 Handling contested and unresolved graph structure. The F1 relation graph itself is not always
settled: an edge may be CONTESTED (Part V, F1's own worked example of an unresolved duplicates
relation) or a system's properties UNRESOLVED (F1 Part IV §4.6). The scanner's traversal, encountering
a contested or unresolved node mid-path, does not silently skip it (which would understate the
candidate's true multiplier) and does not silently treat it as confirmed (which would overstate it);
it reports the affected predicted-recipient entry with an explicit UNRESOLVED-PATH flag, and Part
IX's ranker treats an UNRESOLVED-PATH entry as excluded from the confident aggregate score while
still surfacing it as an "if resolved, this candidate's multiplier could rise" annotation — preserving
information the estate might act on (resolve the underlying contest) rather than discarding it.

7.6 The scanner's output format and its relation to the ledger write. Every scanner run against a
candidate produces a complete predicted-recipient set (Part V §5.2), written to that candidate's
leverage record as a batch, not incrementally, so that a partially-completed scan is never mistaken
for a completed one — a candidate's ledger status only advances past its initial PROPOSED entry
once the scanner has completed a full bounded traversal, and a scan interrupted mid-traversal is
discarded and re-run in full on the next cycle rather than resumed from a partial state, which
trades a small amount of redundant computation for the guarantee that every recorded predicted-
recipient set is internally consistent.

7.7 Worked trace: the scanner applied to the guard-clause candidate from Part II §2.6. Target system:
the shared hook-dispatcher component. Depth-one strengthens edges found: the secret-firewall hook
chain, the scaffold-auditor hook chain (two direct edges, high confidence). Depth-two edge found: the
anti-thrash counter, reached via the scaffold-auditor hook chain's own downstream strengthens edge to
it (one chained edge, discounted per §7.3). A fourth candidate recipient, the output-contract
validator, is reached only through a relation currently marked CONTESTED in the graph (its own
duplicates-disposition against a sibling validator is still under review) — the scanner reports it
with an UNRESOLVED-PATH flag rather than folding it into the confident count. The scanner's raw output
for this candidate is therefore: two depth-one direct recipients, one depth-two chained recipient, one
unresolved-path recipient — a provisional multiplier of three confident plus one contingent, which is
the exact input Part VIII's estimator and Part IX's ranker consume next, rather than a bare "four."

7.8 Scanner determinism and why re-running it against an unchanged graph must produce an identical
result. A property this Part has not yet stated explicitly but which every downstream consumer (the
estimator, the ranker, the audit trail) depends on: the scanner is DETERMINISTIC — given the same F1
relation graph state and the same candidate target, two scans produce the identical predicted-recipient
set, identical depths, identical UNRESOLVED-PATH flags, every time. This determinism is not an
implementation detail incidental to the scanner's purpose; it is NORMATIVE and load-bearing, because
Part XX §20.6's regression discipline depends on being able to re-run the scanner against the synthetic
suite's fixed graph state and compare the output to a stipulated ground truth bit-for-bit, and Part
XVII §17.7's adversarial re-check depends on a "fresh, skeptical re-scan" meaning something precise —
if the scanner's own traversal order or tie-breaking behavior were non-deterministic, a re-scan
producing a different predicted-recipient set from the original would be genuinely ambiguous as to
whether it reveals a real problem with the original scan or merely reflects the scanner's own internal
randomness, collapsing exactly the diagnostic value Part XVII §17.7's worked trace depends on. Any
non-determinism that does legitimately enter a scan result — a graph state that changed between the
original scan and a later re-scan, per Part VII §7.6's completed-scan discipline — is therefore always
attributable to a genuine, timestamped CHANGE in the underlying graph, never to the scanner's own
traversal logic, which is the precise sense in which "the graph changed" and "the scanner behaved
differently" must remain two distinguishable claims rather than one conflated symptom.

FINAL LAW — PART VII. The fan-out scanner finds leverage the graph already knows about, at a bounded,
deterministic depth, marking direct edges more heavily than chained ones and flagging contested paths
rather than silently resolving them either way; what the graph has never recorded, no traversal,
however careful, can find — that gap belongs to the Recombination Lab, not to a deeper scan.

===============================================================================================
PART VIII — SUBENGINE TWO: THE COUNTERFACTUAL ESTIMATOR
===============================================================================================

8.1 The estimator's precise job, distinguished from the scanner's. Where the scanner answers "which
systems COULD this candidate touch," the counterfactual estimator answers "how much did (or would)
each touched system's actual behavior change." It is invoked only against candidates the ranker has
selected (Part VI §6.5), and its output upgrades a candidate's predicted-recipient entries from
prospective-structural to, where possible, retrodictive (Part IV §4.2-4.3), tightening the confidence
interval Part IV §4.5 requires.

8.2 The two estimator modes, mirrored to the two candidate states. For an ADOPTED candidate (Part V
§5.3's status field), the estimator runs a full retrodictive comparison per Part IV §4.2: recipient
system property values before and after adoption, against a counterfactual baseline per Part IV §4.4.
For a candidate still UNDER-EVALUATION but not yet adopted, the estimator cannot retrodict an effect
that has not happened; instead it runs an ANALOGICAL-HISTORICAL estimate (Part IV §4.2's third mode),
searching the ledger's SUPERSEDED and ADOPTED records (Part V §5.6) for the closest prior candidate
by target system, predicted-recipient overlap, and intervention shape, and reports that prior
candidate's retrodictive multiplier as a HYPOTHESIZED prior for the current one — never as a
substitute for eventual retrodiction, only as a better-than-nothing estimate to inform Part IX's
prioritization of WHICH candidates deserve the scarcer, more expensive full retrodictive treatment
once adopted.

8.3 Constructing a counterfactual baseline honestly. The estimator's hardest technical problem, named
rather than glossed: the estate has no control group in the experimental sense — every system is
live, and the same time period in which a candidate was adopted also contains every OTHER change made
across the estate. The estimator's baseline construction, NORMATIVE, therefore prefers, in order:
(a) a structurally similar system that was NOT touched by the candidate, observed over the same
period, as the closest available control; (b) the touched system's own trend over a comparable prior
period of equal length, extrapolated forward, where no structurally similar untouched system exists;
(c) explicit marking as CONTESTED attribution, per Part IV §4.4, where neither (a) nor (b) is
available with sufficient confidence — and the estimator NEVER falls back silently to a bare
before/after comparison with no stated baseline, because that comparison is exactly the co-occurrence
trap Part IV §4.4 exists to prevent.

8.4 The estimator's own evidence-level discipline on its outputs. Every retrodictive measurement the
estimator produces carries its OWN evidence marking, inherited from which baseline mode (§8.3) was
used: mode (a) baselines support an OBSERVED marking on the resulting delta; mode (b) baselines
support DERIVED (the delta follows from stated extrapolation reasoning over OBSERVED trend data,
which is a weaker claim than a true control comparison); mode (c) attributions are marked CONTESTED
and excluded from Part IX's confident-tier scoring entirely, appearing only as a footnote candidate
for the Recombination Lab's controlled-experiment protocol (Part XIII §13.4) where a cleaner
counterfactual can potentially be engineered going forward.

8.5 Cost discipline: why the estimator is expensive and must be spent carefully. Constructing a
counterfactual baseline, per §8.3, requires identifying and characterizing a comparison system or
period — work that does not parallelize cheaply across the estate's full candidate pool, which is
precisely why Part VI §6.5 gates the estimator's invocation behind the ranker's selection rather than
running it on every scanner hit. The practical consequence, stated as a design constraint rather than
a mere efficiency note: the estimator's invocation budget is itself a scarce institutional resource,
and Part IX's ranking model must account for "cost of confirming this candidate's leverage" as an
input to prioritization, not treat estimator invocation as free just because the estimator, once run,
produces a more trustworthy answer than the scanner alone.

8.6 What the estimator cannot resolve, and the honest escalation path. Some candidates' recipient
systems have such thin OBSERVED history — a newly built system with few prior measurements, per F1's
UNMEASURED PROPERTY case (Part IV §4.6) — that no baseline construction mode in §8.3 produces a
trustworthy estimate regardless of effort spent. The estimator's honest output in this case is not a
forced number; it is a report that the candidate's leverage claim on that specific recipient remains
UNRESOLVED, carried in the leverage record exactly as such, and Part IX's ranker treats an UNRESOLVED
recipient the same way it treats the scanner's UNRESOLVED-PATH flag (Part VII §7.5) — excluded from
the confident aggregate, retained as an annotation, never silently defaulted to zero or to average.

8.7 Worked trace: estimating the guard-clause candidate after adoption. Continuing Part VII §7.7's
trace: the guard-clause candidate is adopted. Six weeks later the estimator runs a full retrodictive
pass. For the secret-firewall hook chain (a depth-one direct recipient), a structurally similar,
untouched hook chain from a sibling project provides a mode-(a) baseline; the delta is OBSERVED and
positive, confirming the predicted edge. For the scaffold-auditor hook chain, no untouched sibling
exists, so the estimator falls to mode (b), extrapolating the auditor's own prior trend; the delta is
DERIVED and positive but with a wider confidence interval. For the anti-thrash counter (the depth-two
chained recipient), the estimator finds insufficient history to construct any baseline and reports
UNRESOLVED. The candidate's final leverage record, after estimation, therefore shows one OBSERVED
confirmed edge, one DERIVED confirmed edge, and one UNRESOLVED edge — a materially more honest and
more useful record than the scanner's original bare "three confident plus one contingent," and this
progressive refinement, scan then estimate, is the concrete mechanism by which Part IV's evidentiary
ladder is climbed one candidate at a time rather than asserted in the abstract.

8.8 What the estimator does when a recipient system it is mid-measurement on becomes a contested
duplicate. A recipient system named in a candidate's predicted-recipient set can, between the scanner's
initial traversal (Part VII) and the estimator's later invocation, become the subject of a fresh D2A
duplicates review (F1 Part V Invariant Two) — a second system is proposed with a near-identical object,
and the two systems' futures, including whether they remain separately addressable at all, are now
genuinely uncertain. The estimator's protocol, NORMATIVE: it does not silently pick one of the two
contested systems to measure against and ignore the other, and it does not pause indefinitely awaiting
the duplicates resolution either, because Part VIII §8.5's cost discipline forbids treating estimator
capacity as free to hold in reserve. Instead, the estimator records the affected predicted-recipient
entry as CONTESTED-TARGET, computes its baseline against whichever of the two systems currently holds
authority over the shared decision class (if any does, per F1 Invariant Three), and flags the entry for
mandatory re-measurement once the duplicates review concludes — mirroring, at the estimator's own layer,
the same structural-before-relational resolution ordering F1 Part V §5.7 establishes for its own worked
invariant-conflict example, because a candidate's leverage claim on a recipient whose very identity is
unsettled cannot be more confident than the identity question itself. This is a genuinely different
situation from Part VII §7.5's UNRESOLVED-PATH handling, which concerns an edge along a contested
RELATION; here the recipient SYSTEM itself, not merely the edge reaching it, is what the D2A process has
not yet settled, and conflating the two would let a system-identity question be quietly resolved by a
graph-traversal shortcut that has no authority to resolve it. The estimator's honest output in this case
— a CONTESTED-TARGET flag rather than a forced number against either candidate system — is the same
discipline §8.6 already requires for genuinely unmeasured properties, extended here to the harder case
where the uncertainty is not about a VALUE but about which SYSTEM the value even belongs to.

FINAL LAW — PART VIII. The estimator upgrades a candidate's predicted edges toward retrodiction using
the best available counterfactual baseline, marks its own outputs with the evidence level the baseline
mode actually supports, and reports UNRESOLVED honestly where no baseline can be built — because an
estimate manufactured to fill a gap is worse for the ledger than an acknowledged gap.

===============================================================================================
PART IX — LEVERAGE-POINT RANKING: THE DECISION MODEL
===============================================================================================

9.1 The ranker's job, and its authority boundary. The ranker consumes every candidate's leverage
record (Part V) and produces a PRIORITIZED ORDERING for scarce deep-work attention — which candidates
merit estimator invocation (Part VI §6.5), and which merit surfacing to the Owner for adoption
(Article IX). The ranker's authority, per F1 Part V Invariant Three (singular authority per decision
class) and this corpus's own inherited Article IX, is strictly ADVISORY: it recommends an ordering,
it does not adopt, and no leverage record transitions to ADOPTED status (Part V §5.3) without an
Owner-authorized action outside B1's own authority. This boundary is not a limitation grudgingly
accepted; it is DERIVED directly from the governed-self-evolution constraint every IAS system
inherits — B1 proposes leverage, the estate's own governance promotes it.

9.2 The ranking inputs, restated as a checklist against Parts IV-VIII. Each candidate entering the
ranker carries: an aggregate multiplier score with confidence interval (Part IV §4.5); a full
predicted-recipient set with per-edge measurement mode, depth, and evidence level (Parts VII-VIII);
a provenance chain (Part V §5.5); and a current status (Part V §5.3). The ranker does not re-derive
any of these; it reads them as given inputs, which keeps the ranker itself a comparatively simple,
auditable function over a rich, already-disciplined input space — the complexity of leverage
discovery lives in Parts VI-VIII, not in the ranker, and this separation of concerns is itself a
minimal-sufficient design choice in the spirit of Part XV.

9.3 The two-axis score, and why it is not collapsed to one number prematurely. Consistent with Part
I §1.3's distinction between multiplier (how many systems) and amplification factor (how much each
system improves), the ranker's core scoring surface is TWO-DIMENSIONAL, not one: reach (the
confidence-weighted count of recipients) on one axis, and depth-of-effect (the confidence-weighted
magnitude of improvement per recipient, where measured) on the other. A candidate high on reach but
low on measured depth-of-effect (touches many systems, each barely) and a candidate low on reach but
high on depth-of-effect (touches few systems, each substantially) are NOT automatically ranked against
each other by a collapsed product or sum; the ranker reports both axes, and Part X's tier model is
where a NORMATIVE judgment about which axis matters more for a given institutional moment is applied
— explicitly, visibly, and revisable, rather than baked silently into an opaque formula.

9.4 Confidence as a gating filter before ranking, not merely a tiebreaker. A candidate whose aggregate
confidence interval is too wide — dominated by HYPOTHESIZED or UNRESOLVED edges (Parts VII-VIII) — is
not ranked alongside confidently-measured candidates at all; it is routed instead to either the
counterfactual estimator (if adopted or adoptable cheaply enough to generate real retrodictive data)
or the Recombination Lab (Part XIII), if it is speculative enough that controlled experimentation, not
graph analysis, is the appropriate next step. This gating is DERIVED from Part IV §4.6's worked
example: ranking a wide, unverified candidate directly against a narrow, proven one on a single
combined score is exactly how a naive ranker would have inverted M and N's correct treatment, and the
gate exists specifically to prevent that inversion structurally rather than relying on careful manual
review to catch it every time.

9.5 The recommendation object, and what the ranker actually outputs. The ranker's output, DEFINED, is
not a single "best" candidate but a RECOMMENDATION SET: a short list of high-confidence candidates
ready for Owner review, tagged with their two-axis score and a plain-language statement of which
systems each would strengthen and on what evidence; a longer list of candidates awaiting estimator
invocation, prioritized by estimated cost-to-confirm against expected score if confirmed; and a
residual list of speculative candidates routed to the lab. This three-part output structure is what
lets B1's recommendation be genuinely actionable rather than a single opaque score an Owner must
either trust blindly or ignore — the plain-language recipient statement is a NORMATIVE requirement,
not an optional nicety, because an unexplainable ranking is a ranking the estate cannot audit, and
an un-auditable ranking is precisely the kind of governance theater Article XI's SEALED discipline
exists to exclude.

9.6 Re-ranking cadence and the staleness problem inherited from F1. Because every input to the ranker
(Part IV §4.5's weighting, in particular, on recipient consumer count) is itself a property subject to
F1's own staleness discipline (F1 Part IV §4.3), a ranking computed once and never refreshed decays in
trustworthiness even if no new candidates are added — the ESTATE around the candidates keeps changing.
The ranker therefore re-runs on a cadence tied to ledger and graph reconciliation events rather than on
a fixed calendar schedule, NORMATIVE, so that a ranking is always computed against the freshest
available property data, and a recommendation set older than its own last reconciliation timestamp
(Part V §5.2) is marked stale in exactly the same sense F1 marks a system property stale, rather than
being presented to the Owner as current when it is not.

9.7 Worked example: two candidates through the full ranking pipeline. Candidate A: reach 3 (all
depth-one, two OBSERVED via retrodiction from Part VIII §8.7's guard-clause example, one still
prospective), depth-of-effect moderate and OBSERVED on the confirmed edges. Candidate B: reach 7, all
prospective-structural, no retrodictive confirmation, wide confidence interval. Under §9.4's gate,
Candidate B does not enter the confident ranking tier despite its larger raw reach — it is routed to
the estimator-invocation queue, prioritized moderately given its size. Candidate A enters the Owner-
facing recommendation set directly, with a plain-language statement naming its three recipients and
their measured deltas. This ordering — the smaller, better-evidenced candidate surfaced first — is
the ranker functioning exactly as designed, and it is the concrete refutation of the intuition Part
XVII's false-leverage catalogue names: that a bigger predicted number is automatically the better
recommendation.

9.8 The empty-cycle case, and why an honest null output is a legitimate ranker result. Some ranking
cycles will find the ledger's confident tier genuinely empty — every PROPOSED or UNDER-EVALUATION
candidate this cycle is either gated out by §9.4's confidence filter, still awaiting estimator
invocation, or routed to the lab as too speculative for direct ranking. A ranker under institutional
pressure to always produce a recommendation might be tempted to relax §9.4's gate just enough to
surface SOMETHING, manufacturing an appearance of continuous productivity where none currently exists.
This Part forbids that relaxation explicitly, NORMATIVE: an empty confident-tier recommendation set is
a valid, honest output, reported as such — "no candidate cleared the confidence gate this cycle" — rather
than papered over by a quiet loosening of the gate's own bar, because a gate that flexes under pressure
to always produce output is not a gate, it is theater wearing a gate's name, and the specific
institutional harm is that a relaxed gate this cycle sets a precedent for a further relaxation next
cycle, the same slow-drift failure Part XVII §17.5's metric-reflexivity analysis names in a different
guise. An empty cycle is, moreover, informative in its own right: several consecutive empty cycles are
themselves a signal worth surfacing to the systems this dataset feeds (Part XXI) — it may mean the
candidate pool's producers (FD, ACIS, hard_rules, the learners) have genuinely slowed, or it may mean
the confidence bar itself has drifted too strict relative to the estate's real evidentiary reach, a
question Part XX's evaluation protocol, not the ranker's own discretion, is the correct venue to
adjudicate.

FINAL LAW — PART IX. The ranker recommends, never adopts; scores on two axes, reach and depth-of-
effect, never collapsed prematurely; gates low-confidence candidates out of the confident tier before
scoring rather than scoring everything alike; and explains every recommendation in plain language
naming the specific systems and evidence behind it — an unexplainable ranking is not a ranking the
estate can be governed by.

PART X — TIERS OF LEVERAGE: FROM LOCAL FIX TO FEDERATION-WIDE LEVER
===============================================================================================

10.1 Why a tier model, given the ranker already produces a continuous two-axis score. Part IX's
recommendation set is precise but not, by itself, легible at a glance to an Owner triaging many
candidates across many sessions; a small number of named TIERS, each with a plain-language
description and a minimum evidentiary bar, converts the continuous score into an institutional
vocabulary the estate can use in conversation — "this is a Tier Two lever" carries meaning a raw
confidence-weighted reach-and-depth pair does not, without discarding the underlying precision the
tier is built from. The tiers are a PRESENTATION layer over Part IX's model, never a replacement for
it; every tier assignment remains traceable to the specific score and evidence it was computed from.

10.2 Tier Zero — Local Fix (multiplier one, not a leverage point at all). Per Part I §1.2's floor,
a candidate touching exactly one system is not admitted to the leverage ledger's ranking pipeline as
a leverage point; it is correctly the province of whichever first-order producer (FD, a learner)
surfaced it, and B1's role toward a Tier Zero candidate is limited to confirming, via the scanner
(Part VII), that it genuinely has no fan-out — a negative result that is itself useful, because it
tells the requesting system "this fix is worth doing on its own merits, but do not expect it to
compound," closing off a false expectation before deep-work time is spent chasing multiplier that
does not exist.

10.3 Tier One — Local Lever (multiplier two to three, direct edges, at least one retrodictively or
analogically supported). The minimum admissible leverage tier: a small, well-evidenced fan-out,
typically within one hop, typically inside one family of related systems (the guard-clause example
of Parts II, VII, and VIII is illustratively a Tier One lever). Tier One candidates are the ranker's
most common confident-tier output, NORMATIVE, because the estate's strengthens-edge density (F1 Part
VI §6.10) makes wide fan-outs comparatively rare — most real leverage, OBSERVED across the illustrative
worked examples this dataset develops, is Tier One, and an institution that under-values Tier One
levers in favor of chasing rarer, higher tiers is, per Part XVII, at risk of the local-optimum-sold-
as-global failure in its inverted form: dismissing genuine, modest, well-proven leverage because it
is not dramatic enough.

10.4 Tier Two — Cross-Family Lever (multiplier four to six, spanning systems that do not share an
obvious functional family, at least partial retrodictive confirmation). A Tier Two lever's defining
feature is not size alone but SPAN: it strengthens systems that a casual observer, without the
relation graph, would not have grouped together — the vocabulary-standardization example of Part III
§3.6 (touching ACIS, hard_rules, FD, and B1 itself) is illustratively Tier Two, because epistemic-
marking consistency is not an obviously shared concern across those four systems until the graph
makes the shared dependency visible. Tier Two candidates are, per Part IX §9.4's gating discipline,
held to a materially higher evidentiary bar before Owner-facing recommendation than Tier One, because
a wider, less obviously-related fan-out is exactly the shape a false-leverage claim tends to take
(Part XVII), and the tier's own name should never be granted on reach alone without confirmation
proportionate to the claim's ambition.

10.5 Tier Three — Federation-Wide Lever (multiplier spanning a majority of active systems, or
touching a foundational shared resource, always requiring retrodictive confirmation before the tier
name is used). This is the rarest tier, and F1's own worked contrast in Part I §1.4 (a shared
vocabulary correction touching every consuming system) is the canonical illustrative shape: a
Federation-Wide Lever typically operates on a shared substrate — a vocabulary, a schema, an ontology
term, a widely-imported check — rather than on any one system's internals, which is DERIVED directly
from the level test (F1 Part VIII): the majority-object rule almost always classifies a candidate
this widely consumed as ensemble-level in its own right, meaning a Tier Three lever sits very close
to B1's own object boundary with F1 itself, a proximity Part XXII returns to as an open question
about B1's evolution.

10.6 The tier-assignment discipline, and the anti-inflation rule. NORMATIVE: no candidate may be
assigned a tier above what its CONFIRMED (retrodictive or partially-retrodictive) evidence supports,
regardless of its prospective-structural reach; a candidate with a prospective reach of eight but zero
retrodictive confirmation is presented as "Tier Two-provisional, pending confirmation," never as a
bare "Tier Two" or, worse, "Tier Three," because presenting an unconfirmed wide fan-out at its full
prospective tier is the single most direct institutional pathway into the false-leverage failure mode
Part XVII exists to prevent — Owners triaging by tier name alone, rather than by reading the
underlying evidence, are exactly the audience this anti-inflation rule protects.

10.7 Worked comparison across all four tiers on illustrative synthetic candidates. Tier Zero: a fix to
one system's internal logging format, confirmed by the scanner to have zero fan-out — correctly
routed back to its originating producer with no B1 involvement beyond the negative confirmation.
Tier One: the guard-clause candidate, multiplier three, two edges OBSERVED via retrodiction. Tier
Two-provisional: the vocabulary-standardization candidate, prospective multiplier four, zero
retrodictive confirmation yet, correctly held at provisional status per §10.6 until at least one
edge is confirmed. Tier Three: illustratively, a correction to how the ensemble's own property-
staleness reconciliation cadence is computed (F1 Part IV §4.3's own mechanism), which if confirmed
would improve every downstream consumer of every system's property record simultaneously — named
here only as the SHAPE such a lever would take, since no synthetic worked instance of it is claimed
as confirmed, consistent with §10.5's rarity claim.

10.8 Tier erosion: what happens when a SEALED, confirmed lever's measured effect decays over time.
A candidate confirmed at Tier Two or Three, adopted, and re-confirmed (mirroring Part XII §12.4's
re-confirmation requirement for synergy pairings) can still, over a longer horizon, show its confirmed
recipients' amplification eroding — a strengthens edge that was genuinely real at confirmation time
weakens as the recipient system itself evolves away from the conditions that made the original
intervention effective, or as a later, unrelated change to the recipient dilutes the earlier gain.
B1's tier assignment, per this Part's own definitions, is not a permanent institutional medal; it is a
claim tied to its LAST-RECONCILED evidence (Part V §5.2), and an eroded tier is re-computed, NORMATIVE,
on the same reconciliation cadence Part IX §9.6 already applies to the ranker's inputs generally — a
Tier Two lever whose confirmed delta on one of its two recipients has fallen to statistical
insignificance on re-measurement is demoted to Tier One, with its full prior history (the original
confirmation, the erosion, the demotion) preserved in the leverage record exactly as Part V §5.6's
append-preferred discipline requires for every other status transition. This erosion path matters
institutionally because a tier catalogue that only ever moves candidates UP, never down, would
systematically overstate the estate's accumulated leverage over time, silently converting Part XVI's
false-leverage risk into a slow-motion version of the same failure — not a claim inflated at
presentation, but a claim never DEFLATED as its own supporting evidence weakens, which is functionally
indistinguishable from inflation to anyone reading the tier catalogue without the full reconciliation
history behind it. Tier erosion is therefore treated, NORMATIVE, with the same seriousness as tier
promotion: both are ordinary, expected consequences of a property-honesty discipline (F1 Part V
Invariant Five) applied continuously rather than only at the moment a candidate first earns its tier.

FINAL LAW — PART X. Four tiers, Zero through Three, each a plain-language name over Part IX's
continuous score, each gated by confirmed evidence rather than prospective reach alone — a tier name
is a claim the estate will act on, and a claim the estate acts on must never outrun what has actually
been confirmed.

PART XI — THE COGNITIVE SYNERGY DETECTOR: WHEN TWO CAPABILITIES BEAT THEIR SUM
===============================================================================================

11.1 Absorbing candidate 20 as a specialized collector, not a freestanding system. The Non-Duplication
Ledger's disposition for candidate 20, Cognitive Synergy Detector ("measures when 2 capabilities beat
their sum," verdict MERGE → IAS-B1), names an object adjacent to but distinct from single-candidate
leverage: SYNERGY, DEFINED here, is the property of a PAIR (or small set) of capabilities whose
COMBINED strengthens fan-out, or combined amplification factor, exceeds the sum of their INDEPENDENT
fan-outs or factors measured separately. Where Parts I-X score one candidate at a time, the synergy
detector scores CANDIDATE PAIRS drawn from the same leverage ledger (Part V), which is why it is
architected as a third collector alongside the scanner and estimator (Part VI §6.2) rather than as an
independent dataset — its input is the same ledger, its output writes to the same record schema,
extended (§11.3) with a pairing field.

11.2 Why synergy cannot be detected by summing two independent multiplier scores. The naive approach —
compute candidate P's multiplier, compute candidate Q's multiplier, and flag the pair as synergistic
if their sum is large — measures co-occurrence of size, not synergy, and is a direct instance of the
CONVERGENCE MIRAGE named in Part I §1.2: two large, unrelated candidates summing to a large number
tells the estate nothing about whether adopting BOTH together produces value neither would produce
alone. Genuine synergy, DEFINED precisely, requires that the JOINT strengthens fan-out — the set of
recipients that benefit only when BOTH P and Q are adopted, not from either alone — is non-empty and
independently confirmed; a pair whose joint fan-out is empty, however large each individual multiplier
is, is not synergistic, it is merely two good candidates that happen to share a ledger.

11.3 The pairing record and its relationship to the leverage record. A PAIRING RECORD, DEFINED, extends
Part V's schema with: the two (or more) constituent candidate IDs; the JOINT-ONLY recipient set (systems
strengthened by the pair together but by neither candidate independently, per §11.2's non-empty
requirement); a SYNERGY DELTA (the aggregate score, per Part IV §4.5's weighting, of the joint-only
set alone, reported separately from either constituent's own score); and the same status and
reconciliation fields as a single leverage record. A pairing record without a non-empty joint-only
recipient set is not written to the ledger as a synergy claim at all — this is a hard gate, NORMATIVE,
because a "synergy" record with an empty joint-only set is definitionally not synergy under §11.2, and
allowing it into the ledger would corrupt Part IX's ranker with a category of claim indistinguishable
from noise.

11.4 How joint-only recipients are found: the counterfactual-of-counterfactuals problem. Establishing
that a recipient system benefits from P-and-Q together but not from P alone or Q alone is a harder
measurement problem than Part IV's single-candidate counterfactual, because it requires, in principle,
three comparisons: the recipient with neither adopted, with only P adopted, with only Q adopted, and
with both — a four-way comparison where the estate can rarely observe all four states for the same
recipient at different times with a clean counterfactual for each. The synergy detector's practical
protocol, NORMATIVE, therefore prioritizes candidates where P and Q were adopted at DIFFERENT times
(so a with-P-only period and a with-both period both exist in the estate's own history), using the
estimator's baseline-construction discipline (Part VIII §8.3) at each transition rather than attempting
a simultaneous four-way experimental design the estate's live, uncontrolled environment cannot support.

11.5 The false-synergy trap: correlated adoption without causal joint effect. Two candidates adopted in
the same period, whose recipient system happens to improve in that period, can appear synergistic under
a naive joint-window comparison even when the improvement is attributable to a THIRD, unrelated change
made in the same window — the co-occurrence trap of Part IV §4.4, compounded, because now there are two
candidates' worth of plausible alternative explanations rather than one. The detector's mitigation,
DERIVED directly from Part VIII §8.3's baseline hierarchy: a joint-only claim is only promoted past
HYPOTHESIZED when a genuine counterfactual — a structurally similar recipient that received P but not
Q, or Q but not P, in the same period — exists and shows the joint-adopting recipient outperforming
both single-adoption comparisons; absent that comparison, a joint-window improvement is marked
CONTESTED, exactly as an unqualified single-candidate attribution is under Part IV §4.4, never promoted
to a confident synergy claim on co-occurrence alone.

11.6 Synergy and B1's own self-reflexivity, revisited. Part VI §6.7 established that B1's own engine
improvements enter its own ledger as ordinary candidates; the same reflexivity applies to synergy — a
pairing of "the fan-out scanner's depth-bound tuning" and "the estimator's baseline-selection heuristic"
is itself a candidate pairing record if the two together produce joint-only recipients (for instance,
faster convergence to confident tiers estate-wide) that neither tuning change produces alone. This is
not a hypothetical corner case invented for completeness; it is the concrete mechanism by which B1's
OWN operational improvements are held to the same synergy discipline it applies externally, which
matters because an engine that discovers synergy everywhere except within itself would be exempting
its own evolution from the very rigor it demands of every other candidate in the estate.

11.7 Worked example: a synthetic synergy pair traced end to end. Candidate P: standardizes a shared
error-classification vocabulary across the estate's hooks (illustratively Tier One, multiplier two,
per Part X). Candidate Q: adds a cross-hook event bus that lets one hook's finding be consumed by
another without re-derivation (illustratively Tier One, multiplier two, a different pair of recipients
than P's). Adopted separately, each produces its own confirmed, modest improvement. Adopted TOGETHER,
a joint-only recipient emerges: a downstream auditor that could not previously consume another hook's
classified finding at all (Q alone gives it a channel; P alone gives it nothing to send over that
channel usefully classified; only together does the auditor's own false-positive rate measurably drop,
confirmed via a structurally similar sibling auditor that received only Q as its counterfactual). The
pairing record's synergy delta is therefore a genuine joint-only effect — one recipient, but a
recipient neither P's nor Q's individual leverage record could have predicted, which is the precise,
falsifiable shape of "two capabilities beating their sum" this Part exists to formalize.

11.8 Adversarial case: a joint-only recipient that turns out to be reachable by one constituent alone
through a path the scanner missed. Consider, illustratively, the §11.7 pair once more, but with the
downstream auditor's apparent joint-only dependency later re-examined under Part XVII §17.6's
independent adversarial review. The re-check discovers that candidate P's own depth-two chained edge
(Part VII §7.3) — initially excluded from the scanner's confident count because the intermediate node
sat behind a CONTESTED relation (Part VII §7.5's UNRESOLVED-PATH treatment) — in fact resolves,
once the contest is settled, to the SAME auditor the pairing record had flagged as joint-only. The
apparent synergy was therefore not genuine: P alone, via a path the scanner had correctly but
conservatively excluded rather than confidently confirmed, already reached the recipient; Q's
contribution was real but not NECESSARY in the strict sense §11.2 requires, because the joint-only
test demands a recipient unreachable by either constituent ALONE, and P alone, once its own
contested path resolves, turns out to reach it. B1's resolution, NORMATIVE, mirrors Part VII §7.5's
own discipline rather than inventing a separate one: while the underlying path remains CONTESTED, the
pairing record legitimately treats the recipient as joint-only, because at that point in time neither
constituent's OWN confident count included it; the moment the contest resolves in a way that gives P
sole credit, the pairing record's joint-only recipient set is corrected — not silently, but via the
same append-preferred, cross-referencing discipline Part VI §6.4 requires for any provenance
correction — and the synergy delta (Part XI §11.3) is recomputed with that recipient removed. This is
not a failure of the detector; it is the detector behaving exactly as its own evidentiary discipline
demands when new information resolves a previously-contested path, and it is precisely why Part XI
§11.3's non-empty joint-only gate is checked continuously against the CURRENT graph state rather than
frozen at the pairing record's creation time. The broader lesson this adversarial case generalizes,
consistent with Part XVII's own catalogue: a synergy claim's confidence is only as durable as the
weakest CONTESTED or UNRESOLVED-PATH edge feeding either constituent's individual multiplier, and a
pairing record built while either constituent still carries an unresolved path is itself provisionally
weaker than its synergy delta alone would suggest — a caveat Part IX's ranker surfaces alongside any
pairing record whose constituents are not both fully resolved, so a reviewer sees the provisional
status rather than a bare, falsely-confident score.

FINAL LAW — PART XI. Synergy is not the sum of two multipliers; it is a non-empty, independently-
confirmed JOINT-ONLY recipient set — a system that benefits from both candidates together and from
neither alone — and a pairing record with an empty joint-only set is not a synergy claim, it is two
good candidates sharing a ledger by coincidence.

PART XII — SYNERGY WORKED EXAMPLES AND THE FALSE-SYNERGY TRAP
===============================================================================================

12.1 Why this Part exists separately from Part XI. Part XI specifies the synergy detector's mechanism;
this Part is devoted entirely to adversarial and edge-case worked material, because synergy claims are,
DERIVED from §11.5's own analysis, the single category of leverage claim most exposed to false-positive
risk in the whole B1 system — a pairing record inherently invites a narrative ("these two things
obviously work together") that a single-candidate leverage record does not, and narrative plausibility
is precisely what the estate's broader evidence discipline exists to discount in favor of confirmed
measurement.

12.2 Adversarial case one: the shared-cause illusion. Two candidates, P and Q, are both proposed by the
same learner in the same session, both target systems that happen to share a common upstream cause
(both systems recently had a schema migration that temporarily degraded their metrics for unrelated
reasons). When both P and Q are adopted shortly after the migration settles, both recipients' metrics
improve — not because of P, not because of Q, not because of P-and-Q together, but because the shared
migration-induced degradation was already resolving. A naive synergy read sees "adopted together,
improved together" and reports high-confidence synergy. The detector's mitigation, DERIVED from §11.5's
baseline discipline: the counterfactual baseline must be drawn from a period AFTER the shared cause
(the migration) has stabilized on both sides, or from a structurally similar recipient pair that did
NOT share the same upstream migration; a synergy claim whose baseline period overlaps a known shared
external event on both recipients is automatically demoted to CONTESTED regardless of the apparent
improvement's size.

12.3 Adversarial case two: the narrative-plausible non-synergy. Two candidates, R and S, are
STRUCTURALLY plausible synergy partners — both touch adjacent stages of the same pipeline, and a
reasonable Owner reading their descriptions would expect them to compound. Measured, their joint-only
recipient set is empty: each candidate's benefit is fully realized independently, and adopting both
produces exactly the sum of their individual effects, no more. This is not a detector failure; it is
the detector correctly reporting NO SYNERGY despite strong narrative plausibility, and Part XI §11.3's
hard gate (no pairing record without a non-empty joint-only set) is precisely what prevents this
narrative-plausible pair from entering the ledger as a false synergy claim just because a human reading
the two candidate descriptions would have expected one.

12.4 Adversarial case three: synergy that reverses under a longer observation window. A pairing T-U
shows a genuine, confirmed joint-only improvement in the six weeks immediately following adoption, but
by six months, the joint-only recipient's metric has regressed below its pre-adoption baseline, because
the interaction between T and U introduced a subtle new failure mode that only manifests at higher load
— a case this dataset marks explicitly as an open evidentiary risk rather than a solved one. The
detector's partial mitigation, NORMATIVE: a synergy delta's status (mirroring Part V §5.3) includes a
RE-CONFIRMATION requirement at a longer horizon before the pairing is treated as durably ADOPTED-
CONFIRMED rather than merely ADOPTED-PROVISIONAL, and Part XIX's metrics track re-confirmation failure
rate as a countermetric specifically because a synergy program that only ever measures short-horizon
effects will systematically over-claim durability it has not actually observed.

12.5 The false-synergy trap generalized: correlation of ADOPTION TIMING as the root vulnerability. Every
adversarial case above shares one structural root: candidates adopted close together in time create
apparent joint effects whether or not a real interaction exists, simply because temporal proximity is
easy to observe and causal interaction is hard to observe. DERIVED, this is the single highest-priority
risk the synergy detector must design against, more than any individual adversarial scenario, and it is
the reason §11.4's protocol specifically prioritizes candidates adopted at DIFFERENT times — not as an
arbitrary preference, but because temporal separation is the one structural feature that makes a genuine
counterfactual (a with-only-one-adopted period) available at all.

12.6 A worked non-adversarial validation case, to show the detector functioning correctly under
pressure. Returning to Part XI §11.7's P/Q/auditor trace, consider a challenge: could the auditor's
improvement instead be explained by a THIRD concurrent change to the auditor itself, made in the same
window as P and Q's adoption? The detector's protocol, applied honestly, requires checking exactly this
— and in the illustrative trace, the structurally similar sibling auditor used as Q-only counterfactual
received the SAME third concurrent change (if any existed) as the joint-adopting auditor, because both
are drawn from the same deployment wave; the sibling's failure to improve as much as the joint-adopting
auditor, despite sharing any such third change, is what isolates P-and-Q's joint effect specifically —
this is the counterfactual doing exactly the work §12.5 requires of it, and it is the difference between
a synergy claim that survives adversarial questioning and the three cases in §12.2-12.4 that do not.

12.7 What the false-synergy trap implies for Part IX's ranker. Because synergy claims carry
systematically higher false-positive risk than single-candidate leverage claims (§12.1's opening
observation, now demonstrated across three concrete cases), Part IX's ranker applies a STRICTER
confidence gate to pairing records than to single-candidate records before either enters the confident
recommendation tier — NORMATIVE, and DERIVED directly from this Part's adversarial material rather than
from an arbitrary caution: a synergy claim requires not merely a confirmed joint-only recipient but a
confirmed joint-only recipient whose baseline has been checked against shared-cause contamination
(§12.2), narrative-plausibility bias (§12.3), and short-horizon-only measurement (§12.4) before it is
presented to an Owner as a Tier-rated recommendation.

12.8 Adversarial case four: reviewer confirmation bias on ambiguous joint-effect evidence. The three
adversarial cases above (§12.2-12.4) concern the DATA the synergy detector reasons over; this fourth
case concerns the HUMAN or process reviewing a borderline pairing record before it reaches the Owner.
A joint-only recipient's measured delta sometimes sits close to the boundary between a genuine, if
modest, confirmed effect and statistical noise indistinguishable from zero — precisely the case where
Part IV §4.4's confidence-interval discipline should mark the finding CONTESTED rather than confidently
confirmed. A reviewer who already finds the pairing narratively appealing (§12.3's plausibility bias, now
acting on a human rather than on the detector's own algorithm) is systematically more likely to round a
borderline delta UP into confirmed synergy than a reviewer with no prior narrative attachment to the
pairing, an asymmetry this dataset names rather than assumes away. The mitigation, NORMATIVE: the
confidence-interval threshold that separates CONTESTED from confirmed (Part IV §4.4) is fixed and
published BEFORE any specific pairing is reviewed against it, never set or adjusted on a case-by-case
basis by whoever happens to be reviewing a given candidate, precisely because a threshold set after
seeing the data it will judge is a threshold already contaminated by the same confirmation pressure it
exists to guard against. Where a genuinely borderline case does arise, Part XVII §17.6's adversarial-
review requirement applies with an additional discipline specific to this case: the adversarial reviewer
is, where feasible, someone other than the pairing's original proposer or its most enthusiastic internal
advocate, mirroring the estate's own broader code-review doctrine that a reviewer with authorship
investment in the artifact under review is a weaker check than an independent one — imported here by
direct analogy, because the underlying reason (authorship investment biases judgment on ambiguous cases)
is identical whether the artifact under review is code or a synergy claim.

FINAL LAW — PART XII. Every synergy claim is guilty of the shared-cause illusion, the narrative-
plausible non-synergy, and the short-horizon reversal until its counterfactual specifically rules each
out — temporal proximity of adoption manufactures the appearance of synergy for free, and only a
genuinely separated, re-confirmed counterfactual earns the claim the right to be believed.

PART XIII — THE KNOWLEDGE RECOMBINATION LAB: CONTROLLED RECOMBINATION PROTOCOL
===============================================================================================

13.1 Absorbing candidate 19 as a generative collector for what the graph does not yet know. Part VII
§7.4 named the fan-out scanner's core limitation honestly: it can only find leverage along edges the
relation graph already records, and it is structurally blind to a genuinely novel relation between two
systems that have never before been considered together. The KNOWLEDGE RECOMBINATION LAB, absorbing
the Non-Duplication Ledger's candidate 19 (Knowledge Recombination Lab, verdict SUBENGINE → IAS-B1,
overlap with evolution_engine's single-dataset "recombine" operation), is the GENERATIVE counterpart to
the scanner's traversal: rather than reading the existing graph for known fan-out, it deliberately
PROPOSES new pairings of capabilities across systems that have no recorded relation, and tests whether
a controlled trial reveals a genuine strengthens edge worth adding to the graph.

13.2 The lab's object, distinguished from evolution_engine's single-dataset recombine operation. The
Non-Duplication Ledger's overlap note is precise and worth honoring exactly: evolution_engine's
recombine operation acts on ONE dataset (recombining elements within a single knowledge deposit or
capability), a resource-level operation squarely within evolution_engine's own scope. The lab's object
is different in kind, not merely in scale: it recombines CAPABILITIES ACROSS DIFFERENT SYSTEMS,
searching for a novel strengthens relation between two previously-unconnected parts of the ensemble —
an ensemble-level object under IAS-F1 Part VIII's formal test, because what the lab decides upon is a
proposed relation between systems, not the internal content of any one system's dataset. This is the
same resource-level/ensemble-level boundary Part I §1.1 and Part III apply throughout this dataset,
applied here to justify the SUBENGINE disposition rather than a REJECT.

13.3 The recombination hypothesis, and how candidates for recombination are generated. A RECOMBINATION
HYPOTHESIS, DEFINED, is a proposed pairing of two capabilities (Part II's F1-inherited definition —
provider-independent, nameable separately from the system that supplies them) from systems with no
existing strengthens or feeds edge between them, where the hypothesis states a specific, falsifiable
predicted joint effect. Hypotheses are generated by three sources: STRUCTURAL ANALOGY (two capabilities
that each independently resemble a capability pairing elsewhere in the estate that IS known to be
synergistic, per Part XI's confirmed pairing records — an analogical-historical seed, Part IV §4.2's
weakest mode, appropriately used here only to generate hypotheses, never to confirm them); GAP-ADJACENT
PROPOSAL (a capability whose recipient system has an OBSERVED unmet need — per F1's own dead-capability
and dark-matter observability concepts, referenced not re-derived — paired experimentally with a
capability from elsewhere in the estate that structurally resembles what the gap requires); and DIRECT
OWNER PROPOSAL (a human-originated hypothesis, entered directly, carrying no automated provenance but
subject to the same trial protocol as any other).

13.4 The controlled-trial protocol, and why it is the lab's central discipline. A recombination
hypothesis is not adopted on the strength of its plausibility; it enters a bounded TRIAL: the proposed
capability pairing is enacted in a LIMITED, REVERSIBLE scope (a single project, a single mission class,
a bounded time window — the specific scoping mechanism is an implementation decision outside this
dataset's remit, per Article VIII, but the REQUIREMENT of limited, reversible scope is NORMATIVE and
non-negotiable), and the trial's outcome is measured against a counterfactual exactly as Part VIII's
estimator would measure any other candidate — the lab does not invent a separate measurement standard,
it FEEDS its trial outcomes into the same leverage-record and pairing-record schemas (Parts V, XI) once
a trial concludes, entering the ledger as an ordinary candidate or pairing record from that point
forward, evidence-leveled per Part IV like every other entry.

13.5 Why reversibility is the load-bearing constraint, not merely a caution. A recombination hypothesis
is, by definition, an UNPROVEN pairing — its predicted joint effect is HYPOTHESIZED at best before the
trial runs, and a lab that enacted hypotheses at full, irreversible scope would be gambling the estate's
real capability on structural analogy alone, precisely the failure Article IX's governed-self-evolution
boundary exists to prevent. The reversibility requirement is DERIVED directly from that Article: IAS
proposes, the Owner promotes, and a trial that cannot be cleanly reverted if it fails is not a proposal
at all — it is an unauthorized adoption wearing a trial's name, and Part XVII's anti-pattern catalogue
names exactly this failure the IRREVERSIBLE-TRIAL trap.

13.6 Trial outcomes and their disposition. A concluded trial has three possible honest outcomes,
NORMATIVE: CONFIRMED (the predicted joint effect is measured, evidence-leveled per Part IV, and the
pairing enters the ledger as a new candidate or pairing record eligible for Part IX's ranker exactly
like any other); REFUTED (the predicted effect does not appear; the hypothesis is logged, per Part V
§5.6's append-preferred discipline, as a REJECTED record retained for future analogical reasoning about
what does NOT work, which is itself valuable negative evidence the lab's structural-analogy generation
source, §13.3, should weight against); and INCONCLUSIVE (the trial's scope was too limited, or its
duration too short, or its counterfactual too weak, to support either confirmation or refutation — an
honest outcome the lab must be willing to report rather than forcing a false confirm or refute simply
because a trial concluded).

13.7 Worked example: a synthetic recombination trial traced through the full protocol. A structural-
analogy hypothesis proposes pairing the epistemic-marking validation capability (originally built for
dataset-authoring pipelines, per Part III §3.6) with a capability from an entirely different system —
illustratively, the hard_rules digest compiler's staleness-detection logic — on the theory that a
"claim's evidence level can go stale the same way a rule's applicability can go stale" is a genuine
structural analogy worth testing. The trial is scoped to one project's dataset-authoring pipeline for
one month, reversible by a single configuration flag. Measured against a counterfactual sibling
pipeline without the pairing, the trial CONFIRMS a modest joint effect: claims flagged for staleness
review are caught measurably earlier than either capability alone would catch them. The confirmed
pairing enters the ledger as a new Tier One candidate (Part X), available for Part IX's ranker on the
estate's next ranking cycle, and the trial's provenance chain (Part V §5.5) records its origin as a
LAB-GENERATED hypothesis rather than an FD deposit or ACIS claim — preserving the honest fact that this
leverage point was found by deliberate recombination, not by any resource-level producer noticing it
first.

13.8 The lab's own leverage accounting: does running the lab itself constitute a leverage point. Applying
Part I §1.2's own definition reflexively, as Part VI §6.7 and Part XI §11.6 already do for the scanner,
estimator, and synergy detector: the LAB ITSELF, as an operating capability of the estate, has its own
strengthens fan-out — every project or mission class that gains access to a SEALED-CAPABILITY pairing
(Part XIV §14.4) the lab discovered is, in the aggregate, a recipient of the lab's own existence, not
merely of any one trial's specific result. This is worth naming explicitly because it clarifies a
boundary that could otherwise be read as circular: the lab's OPERATION is not itself re-entered into the
leverage ledger as a giant, standing candidate competing against every other candidate for ranking
attention (that would conflate an ongoing capability with a discrete, adoptable intervention, a category
error Part II §2.2's distinct-object discipline would flag); rather, each of the lab's INDIVIDUAL trial
outcomes enters the ledger as its own bounded candidate or pairing record, exactly as Parts XIII-XIV
already specify, and the lab's aggregate institutional value is an emergent property of the SUM of those
individual records over time, not a first-class candidate the lab must rank against itself. This
resolves the same question Part VI §6.7 raises for the detection engine's own tuning improvements,
applied here to the lab's continued existence rather than to a specific parameter change: a capability
that generates candidates is evaluated through the candidates it generates, never by asserting its own
standing leverage score independent of that evidence trail.

FINAL LAW — PART XIII. The lab proposes what the graph does not yet know, tests it in a limited,
reversible trial, and reports CONFIRMED, REFUTED, or INCONCLUSIVE honestly, its own aggregate value
always traceable to the sum of its individual trial records rather than asserted independently — a
recombination hypothesis enacted at full, irreversible scope on structural analogy alone is not a
trial, it is an unauthorized bet against the estate's own governance.

PART XIV — RECOMBINATION LIFECYCLE: FROM HYPOTHESIS TO SEALED CAPABILITY
===============================================================================================

14.1 Why the lifecycle needs its own Part beyond the single-trial protocol. Part XIII specifies one
trial; this Part specifies the FULL lifecycle a recombination hypothesis travels through, because a
single confirmed trial (§13.6) is not, by itself, sufficient grounds for estate-wide adoption — a
trial confirmed in one limited, reversible scope has answered "does this work HERE" but not yet "does
this work EVERYWHERE this pairing could apply," and collapsing those two questions is a specific,
nameable failure this Part exists to prevent.

14.2 The five lifecycle stages. HYPOTHESIZED (Part XIII §13.3's generation, no trial yet run) →
TRIALED (Part XIII §13.4-13.6's bounded, reversible trial, concluded with one of three outcomes) →
CONFIRMED-LOCAL (a trial outcome of CONFIRMED, but validated only within its original limited scope) →
GENERALIZATION-PENDING (a CONFIRMED-LOCAL pairing proposed for a WIDER scope — additional projects,
mission classes, or a permanent estate-wide relation — itself entering a SECOND, wider trial before
full adoption) → SEALED-CAPABILITY (a pairing confirmed at estate-wide scope, formally added to the F1
relation graph as a genuine new strengthens edge between the two previously-unconnected systems, no
longer a lab artifact but a permanent part of the ensemble's own structure). Each transition, NORMATIVE,
requires its own evidence, and no stage may be skipped — a pairing cannot move from CONFIRMED-LOCAL
directly to SEALED-CAPABILITY without a generalization trial, for the same reason Part V §5.3 forbids
a leverage record from skipping UNDER-EVALUATION: local confirmation and estate-wide confirmation are
different evidentiary claims, and treating one as proof of the other is a scope-generalization error.

14.3 Why generalization requires its own trial rather than inheriting the local trial's confidence. A
pairing confirmed in one project's dataset-authoring pipeline (Part XIII §13.7's worked example) may
depend, invisibly, on features specific to that project — a particular deposit volume, a particular
staleness cadence, a particular team's review rhythm — that do not hold estate-wide. The
generalization-pending trial, DERIVED from this risk, is scoped wider than the original (multiple
projects or mission classes rather than one) and measured against its own counterfactual, exactly as
the original trial was; a CONFIRMED-LOCAL pairing that FAILS its generalization trial is not a
contradiction or a lab error, it is the lifecycle functioning correctly — revealing that the pairing's
value was real but scope-bound, a finding worth recording (per §13.6's REFUTED-at-this-scope treatment)
even though it was never wrong at its original scope.

14.4 What SEALED-CAPABILITY status actually changes, and who has authority to grant it. Reaching
SEALED-CAPABILITY is not merely a ledger status change; it is, DEFINED, the formal addition of a new
STRENGTHENS edge to the F1 relation graph itself (Part III §3.1-3.2), meaning every future leverage
scan (Part VII) treats this pairing as an EXISTING relation rather than a hypothesis to be
re-discovered. Per Article IX's governed-self-evolution boundary and F1 Part V's Invariant Three
(singular authority per decision class), B1 does not grant SEALED-CAPABILITY status to itself; the
authority to add a new permanent relation to the shared ontology belongs to whatever estate-level
governance process F1 itself specifies for graph mutation (a process F1 owns, not B1), and B1's role
is to supply the evidence package — the full lifecycle trail from HYPOTHESIZED through the
generalization trial — that such a process would require before granting the status.

14.5 The retirement path: what happens to a pairing that fails at any stage. A hypothesis REFUTED at
the initial trial, or CONFIRMED-LOCAL but refuted at generalization, does not vanish from the ledger;
per Part V §5.6's append-preferred discipline, it is retained with its terminal status and full
evidence trail, because a refuted recombination hypothesis is itself a leverage-adjacent fact: it
tells future structural-analogy generation (Part XIII §13.3) that this specific pairing, or pairings
structurally similar to it, did not produce a joint effect where a reasonable analogy suggested one
might, which is exactly the kind of negative evidence that keeps the lab's hypothesis-generation
honest over time rather than repeatedly proposing variations on an already-refuted idea.

14.6 Lifecycle cost accounting: why not every CONFIRMED-LOCAL pairing pursues generalization. A
CONFIRMED-LOCAL pairing that is valuable within its original scope but expensive to test at wider
scope (§14.3's generalization trial is not free) may simply REMAIN at CONFIRMED-LOCAL indefinitely,
NORMATIVE, if its local value already justifies its cost and estate-wide value is speculative or
low-priority relative to other candidates in Part IX's ranking queue. This is not a lifecycle failure;
it is the same scarcity discipline Part VIII §8.5 applies to estimator invocation, applied here to
generalization trials — the lab does not owe every confirmed local pairing a path to estate-wide
status, only a path that is available WHEN the ranker judges the generalization worth its cost.

14.7 Worked trace: the epistemic-marking-and-staleness pairing through the full lifecycle. Continuing
Part XIII §13.7: the pairing reaches CONFIRMED-LOCAL after its one-month, one-project trial. Part IX's
ranker, reviewing the ledger's next cycle, judges the pairing's local confirmation strong enough and
its estimated generalization cost low enough (the two capabilities involved are already shared
components across most projects, not project-specific) to prioritize a GENERALIZATION-PENDING trial
across three additional projects. The wider trial confirms the joint effect holds at the broader
scope, with one caveat: the effect is measurably smaller in a project with a much lower deposit
volume, a scope-dependency noted in the trial's own record rather than smoothed over. The pairing
reaches SEALED-CAPABILITY, and the new strengthens edge is submitted, with its full evidence package,
to F1's graph-mutation process — at which point this specific pairing exits B1's own active
ledger-management scope and becomes, per §14.4, simply a fact the fan-out scanner (Part VII) will
find on every future traversal without needing to rediscover it.

14.8 Convergent hypotheses: when two generation sources independently propose the same pairing at the
same time. Part XIII §13.3 names three hypothesis-generation sources — structural analogy, gap-adjacent
proposal, direct Owner proposal — and nothing prevents two of them from independently converging on the
identical capability pairing in the same ledger cycle: a structural-analogy search and a gap-adjacent
search, reasoning from entirely different starting premises, can land on the same proposed relation
between the same two systems. Left unhandled, this produces the lifecycle analogue of Part V §5.7's
duplicate-candidate merge problem, but with an added wrinkle the leverage ledger's merge protocol does
not fully cover: two INDEPENDENT hypotheses reaching the same proposed pairing is not merely an
ingestion-time coincidence to be merged for efficiency, it is itself a WEAK, additional piece of evidence
that the pairing has organic merit, because two different reasoning paths agreeing is a lower-probability
event, under a null hypothesis of random proposal, than one path proposing it alone. B1's protocol,
NORMATIVE: when a new hypothesis matches an existing HYPOTHESIZED or TRIALED record's target pairing
closely enough to merge (per Part V §5.7's general rule, applied here to pairing rather than single-
candidate records), the merge additionally upgrades the merged record's priority in Part XIII §13.4's
trial-scheduling queue, reflecting the convergent-generation signal, but explicitly does NOT upgrade the
pairing's evidentiary STATUS itself — convergent generation is a reason to trial a hypothesis sooner, not
a substitute for the trial, and a pairing still moves through HYPOTHESIZED, TRIALED, CONFIRMED-LOCAL,
GENERALIZATION-PENDING, and SEALED-CAPABILITY in the same required order (§14.2) regardless of how many
independent sources proposed it — because independent proposal, however suggestive, is still
analogical-historical-grade evidence at best (Part IV §4.2's weakest mode), never a substitute for the
trial's own retrodictive confirmation.

FINAL LAW — PART XIV. A recombination hypothesis earns SEALED-CAPABILITY status one stage at a time —
hypothesized, trialed, confirmed-local, generalization-pending, sealed — and a pairing that never
advances past confirmed-local because its cost to generalize outweighs its estimated value has not
failed; it has simply found its true, honestly-bounded scope.

PART XV — MINIMAL-SUFFICIENT-DOCTRINE: THE COMPRESSION PRINCIPLE FOLDED IN
===============================================================================================

15.1 Why compression belongs inside the leverage dataset, and the ledger's own fold disposition. The
Non-Duplication Ledger routes Family N — Compression (candidates 131-140) to REFERENCE for most of its
content (GK-06's context pack, CO-04's virtual memory, FD-04's portability, ACIS's principle compiler
each already own a piece of the compression story) but names one specific exception: "Minimal-
sufficient-doctrine folded into IAS-B1." This Part honors that fold explicitly, and explains WHY
minimal-sufficient-doctrine belongs here rather than with any of the compression family's REFERENCE
owners: compression, in general, asks "how small can a representation of X be while preserving X's
content" — a question about FIDELITY, squarely FD-04's and GK-06's territory. MINIMAL-SUFFICIENT-
DOCTRINE asks a different question: "how small can an INTERVENTION be while preserving its FULL
leverage" — a question about AMPLIFICATION EFFICIENCY, which is B1's territory, not compression's,
even though the two questions rhyme in structure.

15.2 The doctrine, defined precisely against B1's own objects. MINIMAL-SUFFICIENT-DOCTRINE, DEFINED,
is the normative principle that among two or more candidate interventions predicted or confirmed to
achieve the SAME multiplier and depth-of-effect (Part IX §9.3's two-axis score), the ranker prefers
the SMALLER intervention — measured in scope of what it directly modifies, not in the number of
systems it strengthens — because a smaller intervention carries lower adoption cost, lower risk of
unintended side effects on systems OUTSIDE its predicted-recipient set, and a shorter path through
Part V's UNDER-EVALUATION status before it can responsibly reach ADOPTED. This is the leverage-
specific instantiation of a discipline the estate already applies elsewhere in different forms (the
principle compiler's minimal sufficient schema, GK-06's minimal context pack), imported here by
analogy and re-derived for B1's own object rather than re-narrated from those siblings.

15.3 Why minimality is checked AFTER leverage, never before it. A doctrine that searched for the
smallest possible intervention FIRST, then asked what it amplifies, would systematically bias toward
Tier Zero and Tier One candidates (Part X) purely because small changes are easier to conceive of than
large ones, starving the search space Part XIII's recombination lab exists to explore. NORMATIVE: B1's
minimal-sufficient check applies ONLY as a tiebreaker among candidates already scored equivalently on
Part IX's two-axis leverage score — it never substitutes for, precedes, or discounts the leverage
search itself. This ordering is DERIVED directly from Article VI's own depth-without-padding
discipline applied reflexively: a system that optimized for minimality before establishing sufficiency
would be padding its OWN search process with a false economy, cutting candidates before their true
leverage was known rather than after.

15.4 The sufficiency half of the doctrine, and why it is not merely "smaller is always better." The
doctrine's name pairs MINIMAL with SUFFICIENT deliberately: a candidate is not preferred merely for
being smaller, it is preferred for being the smallest candidate that is STILL SUFFICIENT to achieve
the full predicted multiplier and depth-of-effect. A candidate that shrinks its own scope so far that
it no longer reaches all of its predicted recipients (Part VII's fan-out) is not a minimal-sufficient
candidate; it is an under-scoped candidate whose actual multiplier is lower than the original's,
and the two are not comparable under §15.3's tiebreaker rule at all — they occupy different points on
Part IX's own scoring surface, and the doctrine's tiebreaker only ever fires between candidates the
ranker has already confirmed are score-equivalent, never between a full-scope and a deliberately-
shrunk, lower-scoring variant of the same idea.

15.5 The relationship to the recombination lab's reversibility requirement. Part XIII §13.5 requires
every trial to be scoped for reversibility; minimal-sufficient-doctrine strengthens that requirement
rather than duplicating it, because a SMALLER intervention is, all else equal, easier to make cleanly
reversible than a larger one touching more of a system's internals — fewer moving parts to restore on
rollback. This is a genuine synergy in the Part XI sense between two of B1's own internal disciplines
(reversibility-by-design and minimality-as-tiebreaker), named here explicitly as a self-referential
instance of the very phenomenon Part XI formalizes: the two disciplines together make trial design
cheaper than either would alone, because a minimal-sufficient trial candidate is, structurally, a
lower-risk reversible trial candidate before any explicit reversibility engineering is even applied.

15.6 The failure mode minimal-sufficient-doctrine specifically guards against: leverage inflation
through scope creep. Without this doctrine, a candidate's scope tends to GROW during its
UNDER-EVALUATION period (Part V §5.3) as each new stakeholder proposes "while we're touching this
component, let's also fix..." — a pattern this dataset names SCOPE CREEP UNDER EVALUATION, and one
that directly threatens Part IV's measurement discipline, because a candidate whose scope changed
between its initial scan (Part VII) and its eventual adoption invalidates the scan's original
predicted-recipient set without anyone necessarily noticing the invalidation. NORMATIVE: any material
scope change during UNDER-EVALUATION triggers a mandatory re-scan (Part VII) before the candidate may
advance to ADOPTED, and minimal-sufficient-doctrine's tiebreaker preference is precisely what makes
resisting scope creep the institutionally cheaper choice — the smaller, unmodified candidate remains
preferred over its creeped, unrescanned sibling by the doctrine's own logic, not merely by process
discipline imposed from outside.

15.7 Worked example: minimal-sufficient-doctrine breaking a tie between two equivalently-scored
candidates. Two independently-generated candidates, both targeting the same root cause (illustratively,
an inconsistency in how "evidence level" is spelled across three related schemas), both scoring
identically on Part IX's two-axis surface (multiplier three, moderate depth-of-effect, comparable
confidence intervals) after independent scanning. Candidate V proposes correcting the inconsistency at
its single point of origin (one shared constant definition all three schemas import). Candidate W
proposes correcting it independently in each of the three schemas' own local definitions, achieving the
identical predicted multiplier through three separate edits rather than one. Minimal-sufficient-
doctrine's tiebreaker prefers Candidate V: identical predicted leverage, smaller total scope, lower
adoption risk (one edit to verify rather than three, one point of future drift rather than three), and,
per §15.5, a structurally easier reversible trial. This is the doctrine functioning exactly as designed
— not discovering NEW leverage, but choosing more wisely among leverage already found.

15.8 Why minimal-sufficient-doctrine is folded into B1 rather than left as a general estate-wide
principle every dataset applies independently. A natural objection: if minimality-after-sufficiency is
good judgment, why does it need a home in B1 specifically rather than being restated identically inside
every IAS dataset that makes tradeoffs? The answer is DERIVED from this dataset's own distinct object
(Part I): minimal-sufficient-doctrine, as folded here, is not a generic preference for small things — it
is specifically a tiebreaker over B1's OWN two-axis leverage score (Part IX §9.3), defined in terms of
predicted-recipient sets and confirmed multipliers that only B1's own machinery produces. A sibling
dataset facing its own tradeoff between two equally-valued options (IAS-C1 choosing between two
equally-scored portfolio allocations, for instance) faces a structurally similar judgment call, but not
the SAME one — its own "equally valued" is defined by its own scoring surface, not by Part IX's reach-
and-depth-of-effect axes, and a sibling borrowing B1's doctrine wholesale would be importing a tiebreaker
defined over the wrong objects. What DOES generalize across the estate, and what this dataset makes no
claim to own, is the higher-level NORMATIVE instinct — prefer the smaller sufficient option once
sufficiency is otherwise equal — which each dataset is free to instantiate against its own scoring
surface independently, exactly as the compression family's other REFERENCE owners (GK-06, CO-04, FD-04,
the principle compiler) each already instantiate a version of the same instinct against their own
distinct objects, per the Non-Duplication Ledger's own fold disposition (§15.1). B1 owns the LEVERAGE-
SPECIFIC instantiation and no other.

FINAL LAW — PART XV. Minimal-sufficient-doctrine is a tiebreaker, never a search strategy, and its
instantiation here is specific to B1's own two-axis score, not a generic principle every sibling dataset
inherits wholesale: among candidates already confirmed equivalent in leverage, the smaller one wins,
because smallness earned after sufficiency is established lowers risk and cost for free — smallness
sought before sufficiency is established is not economy, it is a search space quietly narrowed by
convenience.

PART XVI — FAILURE MODES I: FALSE LEVERAGE AND THE LOCAL-OPTIMUM-SOLD-AS-GLOBAL TRAP
===============================================================================================

16.1 Why B1 needs a dedicated failure-mode catalogue rather than scattered warnings. Parts I-XV each
named specific risks in the context of the mechanism they threaten (the convergence mirage in Part I,
the co-occurrence trap in Part IV, the false-synergy adversarial cases in Part XII). This Part and the
next collect the two most institutionally dangerous failure shapes — FALSE LEVERAGE and the LOCAL-
OPTIMUM-SOLD-AS-GLOBAL trap — into a single, named catalogue with diagnostic signatures, because a
system whose entire purpose is directing scarce deep-work attention toward the "highest-leverage"
candidate does more damage when wrong than a system with a narrower, lower-stakes scope: a false
leverage claim does not merely waste the time spent on it, it displaces a genuinely high-leverage
candidate that a correct ranking would have surfaced instead.

16.2 FALSE LEVERAGE, defined and diagnosed. FALSE LEVERAGE, DEFINED, is a leverage record whose
reported multiplier or tier materially exceeds what its actual, eventually-confirmed evidence
supports — not merely an honest HYPOTHESIZED claim later refuted (which the ledger's status
transitions, Part V §5.3, handle gracefully as REJECTED), but a claim that was PRESENTED at a
confidence or tier higher than its measurement mode (Part IV §4.2) justified at the time it was
presented. The diagnostic signature: a false-leverage record, traced backward through its own leverage
record (Part V §5.2), shows a tier assignment (Part X) that violates §10.6's anti-inflation rule —
predicted, unconfirmed edges counted as if confirmed, or an aggregate score computed without the
confidence-interval discount Part IV §4.5 requires. A false-leverage FINDING is therefore, by
construction, always traceable to a specific point where this dataset's OWN stated discipline was
violated, which is the precise, falsifiable sense in which false leverage is a process failure, not
merely an unlucky prediction.

16.3 The LOCAL-OPTIMUM-SOLD-AS-GLOBAL trap, defined and distinguished from false leverage. This trap
is subtler than false leverage because every individual measurement in it may be entirely honest: a
candidate's predicted-recipient set (Part VII) is correctly and conservatively scanned, its edges are
correctly measured (Part VIII), its tier is correctly assigned (Part X) — and yet the candidate is
presented, implicitly or explicitly, as "the highest-leverage thing the estate could do right now" when
it is, in fact, only the highest-leverage candidate WITHIN A NARROW SLICE of the candidate pool that
happened to be scanned, generated, or considered in a given cycle. DERIVED: this trap is a SAMPLING
failure, not a measurement failure — the ranker (Part IX) can only rank what has entered the ledger
(Part V), and a ledger that under-samples an entire region of the estate's true intervention space
(because no FD deposit, ACIS claim, or lab hypothesis happened to surface a candidate from that region
this cycle) will confidently, honestly, and wrongly present its best LOCAL candidate as though it were
the estate's best GLOBAL one.

16.4 Why the recombination lab does not fully solve the sampling problem. It is tempting to believe
Part XIII's lab, precisely because it generates novel hypotheses rather than merely ranking what
first-order producers surface, closes the sampling gap §16.3 names. It narrows the gap but does not
close it: the lab's own hypothesis-generation sources (Part XIII §13.3 — structural analogy, gap-
adjacent proposal, direct Owner proposal) are themselves biased toward regions of the estate that are
STRUCTURALLY SIMILAR to already-known synergies, or toward gaps the estate's own observability (an
IAS-E1 concern, referenced not rebuilt here) has already surfaced. A genuinely novel leverage point in
a region of the estate with no structural analogy to anything previously confirmed, and no OBSERVED
gap flagging it, remains invisible to the lab exactly as it is invisible to the scanner — this is
UNRESOLVED as an open limitation of B1's current mechanism set, named honestly rather than
papered over with a claim of completeness the design does not support.

16.5 Diagnostic signature of the local-optimum trap, and the partial mitigation. Because the trap is a
sampling failure rather than a measurement failure, it cannot be caught by re-checking any individual
candidate's evidence — the candidate presented IS correctly evidenced. The diagnostic signature is
instead structural: a ranking output (Part IX §9.5) whose recommendation set has been stable across
many consecutive ranking cycles despite the estate's candidate pool continuing to grow, or whose
recommended candidates cluster heavily around one or two systems' neighborhoods in the relation graph
while leaving large regions of the graph untouched by any candidate at all. Part XIX's metrics include
a POOL COVERAGE countermetric specifically to surface this signature — not a countermetric on any
single candidate's score, but on the DIVERSITY of the candidate pool's origin across the estate's
systems, because a ranker producing excellent recommendations from a narrow, repeatedly-sampled corner
of the estate is, by this Part's own logic, a ranker whose "best" claim needs the word "locally"
attached whether or not it says so.

16.6 The honest labeling requirement, and why it is the primary institutional defense. Given that
§16.4 establishes the sampling gap cannot be fully closed by B1's own mechanisms, the NORMATIVE
requirement this Part imposes is not "eliminate the local-optimum trap" (an unachievable standard
given an inherently incomplete candidate pool) but "never present a local optimum AS a global one." Every
recommendation set (Part IX §9.5) carries an explicit POOL COVERAGE STATEMENT alongside its ranked
candidates — a plain-language note on which regions of the F1 relation graph the current candidate
pool draws from and which it does not — so that an Owner reading "Candidate A is our top recommendation
this cycle" also reads "drawn from a pool concentrated in the hook-dispatcher and validation-schema
neighborhoods; the portfolio, ecology, and reliability system neighborhoods contributed no candidates
this cycle." This labeling does not solve the sampling problem; it prevents the sampling problem from
becoming a false institutional claim of global optimality, which is the specific, achievable standard
this Part holds B1 to.

16.7 Worked contrast: a false-leverage finding versus a local-optimum finding on the same candidate
shape. Consider two synthetic candidates presented as Tier Two (Part X §10.4). Candidate X's tier
assignment, on audit, is found to rest on three prospective-structural edges with no retrodictive or
partial confirmation at all — a direct violation of §10.6's anti-inflation rule, and therefore a FALSE
LEVERAGE finding requiring the record be corrected to Tier Two-provisional per §10.6's own remedy.
Candidate Y's tier assignment, on audit, is found to be entirely correctly evidenced — every edge
properly measured and weighted — but the entire candidate pool that cycle contained only six
candidates, all sourced from two adjacent systems' recent FD deposits, with zero candidates from any
other region of the estate; Candidate Y is not mislabeled, but its presentation as "this cycle's top
leverage point" without the §16.6 coverage statement would be a LOCAL-OPTIMUM-SOLD-AS-GLOBAL failure,
remedied not by correcting Candidate Y's own record but by attaching the missing coverage statement to
the recommendation set as a whole.

FINAL LAW — PART XVI. False leverage is a measurement discipline violated on one candidate and is
fixed by correcting that candidate's record; the local-optimum trap is a sampling limitation of the
whole candidate pool and is fixed only by an honest coverage statement — conflating the two, or
believing either can be eliminated rather than managed, is itself the deeper failure this Part exists
to prevent.

PART XVII — FAILURE MODES II: ADVERSARIAL CASES, GAMING THE MULTIPLIER SCORE, AND METRIC REFLEXIVITY
===============================================================================================

17.1 Why adversarial framing matters for a ranking system with institutional consequence. Any function
that determines where scarce deep-work attention flows creates an incentive, even inside a
well-governed estate, to describe a candidate in terms that score well rather than terms that are
merely accurate — not necessarily through bad faith, but through the ordinary human and automated
tendency to frame proposals in their most favorable light. This Part catalogues the specific ways
B1's own scoring surface (Parts IV, VI-XI) can be gamed, deliberately or not, and the countermeasure
each gaming vector requires.

17.2 GAMING VECTOR ONE — the manufactured fan-out. A candidate's target system is described broadly
enough, or its predicted-recipient set is populated with systems related only tenuously (a real but
weak feeds edge stretched to look like a strengthens edge), to inflate its reach axis (Part IX §9.3)
without the underlying intervention actually being that consequential. Countermeasure, NORMATIVE:
Part III §3.2's formal definition — a leverage point's recipient must satisfy the FULL strengthens
test (recipient produces more value, decides better, or costs less BECAUSE the intervention was made),
not merely share a feeds or consumes relation with the target — is the structural defense, and Part
VII §7.2's bounded, depth-weighted traversal is the mechanical defense; a candidate whose predicted-
recipient set is populated mostly by weakly-weighted, deep-chain edges (§7.3) is a signature this
gaming vector leaves behind, visible to an auditor even before any measurement is disputed.

17.3 GAMING VECTOR TWO — provenance laundering. A candidate with a genuinely thin or already-refuted
evidentiary history (Part XIII §13.6's REFUTED records, or a candidate whose earlier version failed
Part IX's confidence gate) is re-proposed with cosmetic changes and a fresh candidate ID, severing the
provenance chain (Part V §5.5) that would otherwise let the ranker or an auditor recognize it as a
repeat of previously-rejected reasoning. Countermeasure, NORMATIVE: the ledger's structural-analogy
generation source (Part XIII §13.3) is required to check new hypotheses against the FULL REFUTED
record set for structural similarity before generation, not merely before ranking, and Part XIX's
metrics include a REPEAT-REJECTION RATE countermetric that flags when structurally similar candidates
recur across multiple cycles under different candidate IDs — a pattern that, even without proving
deliberate laundering, warrants a provenance-chain audit before the candidate proceeds.

17.4 GAMING VECTOR THREE — confidence interval narrowing through selective baseline choice. Because
Part IV §4.5's aggregation and Part IX §9.4's confidence gate both reward tighter confidence intervals,
an actor constructing a counterfactual baseline (Part VIII §8.3) has an incentive to select the
comparison system or period that produces the MOST FAVORABLE, not the most APPROPRIATE, baseline —
choosing a structurally similar system that happened not to improve, making the candidate's delta look
larger, rather than the truly closest structural match. Countermeasure, NORMATIVE: Part VIII §8.3's
baseline-selection hierarchy (a, b, c, in that stated preference order) is a REQUIRED order of
operations, not a menu, and any baseline selection that skips a higher-preference option in favor of a
lower one requires an explicit, auditable justification recorded alongside the leverage record — an
unexplained skip down the hierarchy is, itself, the diagnostic signature this gaming vector leaves.

17.5 GAMING VECTOR FOUR — metric reflexivity, or optimizing the score instead of the object. Once
Part XIX's metrics are published and consulted, a subtle risk emerges that is distinct from the three
deliberate gaming vectors above: candidates begin to be SHAPED, even honestly, to score well on B1's
own visible metrics (multiplier count, tier, confidence interval width) rather than to genuinely
maximize amplification — Goodhart's dynamic operating on B1's own ranking surface. This is DERIVED as
an inevitable pressure on any published scoring system, not a hypothetical, and Part XIX's own
anti-Goodhart countermetrics (§19.4-19.6) are the primary defense; this Part's contribution is naming
the specific shape the pressure takes inside B1: a drift toward candidates that are easy to score
well (small, well-precedented, cheap to baseline) at the expense of candidates that are genuinely
high-value but structurally harder to measure (Part XVI §16.4's sampling-gap regions), which is the
metric-reflexivity RESTATEMENT of the local-optimum trap — the trap can now be self-inflicted by the
scoring system's own visibility, not merely inherited from an incomplete candidate pool.

17.6 The adversarial-review requirement for Tier Two and Tier Three recommendations. Given the four
gaming vectors above concentrate their risk on exactly the candidates with the widest claimed reach
(Tier Two and Three, Part X §10.4-10.5), this Part imposes a NORMATIVE adversarial-review step before
any such candidate reaches the Owner-facing recommendation set: an independent traversal check (does
the predicted-recipient set survive a fresh, skeptical re-scan per §17.2's signature check), a
provenance-chain check (§17.3's repeat-rejection check), and a baseline-hierarchy check (§17.4's
skip-justification check), performed as a distinct pass from the original scoring, mirroring the same
separation-of-concerns discipline Part IX §9.2 already applies between measurement and ranking.

17.7 Worked adversarial trace: a candidate that passes ordinary scoring but fails adversarial review.
A candidate reports Tier Two status (reach four, moderate confidence) with a plain-language recipient
statement (Part IX §9.5) naming four systems. Adversarial review finds: two of the four predicted
edges are depth-two chained edges through a single already-CONTESTED intermediate relation (§17.2's
signature, compounding Part VII §7.5's unresolved-path handling, which should have excluded them from
the confident count but did not due to an implementation gap); one edge's baseline was selected as a
system that had recently undergone an unrelated migration, skipping a more appropriate, available
comparison without documented justification (§17.4's signature). The candidate, on adversarial review,
is demoted to Tier One-provisional pending re-measurement — not rejected outright, because its
remaining single clean edge may still constitute genuine, smaller leverage, but its original Tier Two
presentation is corrected before ever reaching the Owner, which is the adversarial-review requirement
(§17.6) functioning exactly as designed: catching an honest-but-flawed inflation before it becomes an
institutional decision made on bad information.

17.8 GAMING VECTOR FIVE — false urgency framing to bypass the adversarial-review queue. A fifth vector,
distinct from the four above because it targets the PROCESS surrounding scoring rather than the scoring
mechanism itself: a candidate is framed, in its plain-language recipient statement (Part IX §9.5) or its
accompanying description, as time-sensitive — "this must be adopted before system X's next migration" —
creating institutional pressure to fast-track it past Part XVII §17.6's adversarial-review step on the
grounds that the review's own latency would itself cause harm. This vector is particularly dangerous
because it does not require the candidate's underlying evidence to be manipulated at all (unlike vectors
one through three); it manipulates the PROCESS TIMING around evidence that may be entirely honest but
unconfirmed, converting Part XX §20.2's proof standard from a requirement into an inconvenience to be
argued around under time pressure. Countermeasure, NORMATIVE: adversarial review's requirement for
Tier Two/Three candidates (§17.6) is NOT waivable by an urgency claim internal to the candidate's own
description — an urgency claim is, at most, a reason to prioritize a specific candidate's review ahead
of others in the review queue (a legitimate, first-in-line reordering), never a reason to skip the review
step itself, and any workflow implementation that allows an "urgent" flag to bypass rather than merely
reorder the adversarial-review requirement has built a gaming vector directly into its own process
architecture. The diagnostic signature this vector leaves, for a later audit: a disproportionate share
of ADOPTED records whose leverage record shows no completed adversarial-review artifact despite
qualifying for one by tier, clustered around candidates whose description language emphasizes urgency —
a pattern Part XIX's ADVERSARIAL REVIEW COVERAGE countermetric (§19.6) is specifically positioned to
surface, because a coverage gap concentrated in urgency-framed candidates is a materially different, more
actionable finding than a coverage gap distributed randomly across the ledger.

FINAL LAW — PART XVII. Five gaming vectors threaten B1's scoring surface — manufactured fan-out,
provenance laundering, selective baseline choice, metric reflexivity, and false urgency framing — and
every wide-reach
recommendation earns an adversarial re-check, separate from its original scoring, before it is trusted
with an Owner's scarce attention; a ranking system that never adversarially reviews its own top
recommendations is trusting the very incentive structure it created.

PART XVIII — EDGE CASES: AMBIGUOUS TARGETS, CYCLIC STRENGTHENS CHAINS, AND RETIRED RECIPIENTS
===============================================================================================

18.1 Why edge cases earn their own Part rather than footnotes scattered across Parts III-XIV. The
mechanisms specified so far assume a reasonably well-behaved relation graph: a clear target system, an
acyclic strengthens chain, and recipients that remain addressable (per F1 Part IX) for the life of a
leverage record. Real ensemble states, per F1 Part V's own invariants, are not always this clean —
Invariant Two's unresolved duplicates and Invariant Five's stale properties are named there precisely
because messiness is expected, not exceptional. This Part specifies B1's behavior at three concrete
edge cases the mechanisms above do not fully resolve on their own.

18.2 Edge case one: the ambiguous target system. A candidate intervention sometimes has no single,
clean target — a fix that touches a shared component used identically by three systems with no
principled way to say which of the three "owns" the direct modification (a COMPONENT, per F1 Part II
§2.3, contributed to by multiple subsystems). The scanner (Part VII) cannot begin its bounded traversal
without a starting node. B1's resolution, NORMATIVE: where a candidate's target resolves to a shared
component rather than a single system, the scanner treats EACH of the component's parent systems as a
co-target and unions their traversals, but the resulting predicted-recipient set explicitly excludes
the co-target systems THEMSELVES from being double-counted as recipients of each other's traversal —
the union is over REACHED systems, not over starting points, which prevents an ambiguous-target
candidate from inflating its own multiplier merely by having multiple legitimate entry points into the
same graph region.

18.3 Edge case two: the cyclic strengthens chain. IAS-F1 Part VI does not forbid a strengthens edge
from eventually looping back — System A strengthens System B, which strengthens System C, which,
through some genuinely independent mechanism, strengthens System A again. A naive unbounded traversal
(already excluded by Part VII §7.2's depth bound) would, in a cyclic graph, never terminate; the depth
bound alone prevents non-termination, but it does not, by itself, prevent a SHORT cycle from being
double-counted within the bound — System A appearing in its own predicted-recipient set at depth two
or three if the cycle is short enough. B1's resolution, NORMATIVE: the scanner's traversal EXCLUDES the
original target system from its own predicted-recipient set regardless of how many hops it takes to
cycle back to it, because a system cannot be said to be "amplified" by an intervention applied to
itself in the sense Part I §1.2 defines leverage — self-amplification, if real, is simply the direct,
Tier Zero effect of the intervention (Part X §10.2), already captured by the candidate's basic function
and never counted as multiplier fan-out.

18.4 Edge case three: the retired recipient. A predicted or confirmed recipient system, per F1's own
ecology and extinction mechanisms (referenced, not rebuilt), may be RETIRED between the time a
leverage record is written and the time it is evaluated or acted upon — the system that was strengthened
no longer exists in the estate. B1's resolution, NORMATIVE: a retired recipient's edge is not deleted
from the leverage record's history (Part V §5.6's append-preferred discipline applies here too) but is
marked RETIRED-POST-MEASUREMENT, and it is EXCLUDED from the candidate's active multiplier count going
forward — a candidate whose multiplier was four at measurement time but whose one recipient later
retires is re-presented, on any future ranking cycle, as multiplier three, with the retired edge's
history preserved as a footnote rather than silently dropped, because the historical fact that the
intervention DID amplify that system while it existed remains a true, useful data point for future
analogical-historical measurement (Part IV §4.2) even after the system itself is gone.

18.5 Compound edge case: an ambiguous, cyclic, partially-retired candidate. Consider, illustratively, a
candidate whose target is a shared validation component (ambiguous target, §18.2) feeding three parent
systems, one of which sits in a short cycle back to a fourth system that itself strengthens one of the
three original parents (cyclic chain, §18.3), and where one of the original three parent systems is
retired mid-evaluation (§18.4). Applying the three resolutions in sequence — union the co-targets'
traversals excluding double-counts, exclude self-referential cycle closures, mark the retired parent
RETIRED-POST-MEASUREMENT and exclude it from the active count — resolves the compound case to a stable,
auditable predicted-recipient set without requiring a fourth, bespoke rule: the three edge-case
resolutions compose cleanly because each operates on a distinct part of the record (starting-point
union, cycle exclusion, recipient lifecycle) rather than overlapping in scope, which is itself evidence
the three resolutions were correctly scoped in the first place rather than accidentally sufficient.

18.6 The edge case B1 explicitly does NOT resolve on its own authority: contested target ownership. Where
two systems each claim final authority (F1 Part V Invariant Three) over what should be a single,
unambiguous target for a candidate, B1 does not adjudicate the authority dispute itself — that
adjudication belongs to whatever estate mechanism resolves Invariant Three violations (IAS-A2's
arbitration, per the Compounding Graph), and B1's leverage record for such a candidate is marked
UNRESOLVED-AUTHORITY, excluded from Part IX's ranked recommendation set entirely until the underlying
authority question is settled, per the SAME structural-before-relational invariant ordering F1 Part V
§5.7 establishes for its own worked example — a leverage claim built on a target whose very identity is
contested is not yet a claim B1 can responsibly rank.

18.7 Why these edge-case resolutions are conservative by design, and what that costs. Every resolution
in this Part (§18.2-18.4, §18.6) errs toward EXCLUDING ambiguous signal from the confident count rather
than including it optimistically — the ambiguous-target union excludes double-counts rather than
risking inflation, the cyclic exclusion drops the self-loop rather than counting it generously, the
retired-recipient rule shrinks the active count rather than preserving a now-fictional edge, and the
contested-authority case excludes the whole candidate from ranking rather than guessing at a resolution.
This conservatism is DERIVED directly from Part XVI-XVII's failure-mode catalogue: every one of these
edge cases, resolved generously instead of conservatively, would open a new, subtler gaming vector or
false-leverage pathway, and the cost of conservatism — some genuine leverage undercounted at the margin
— is judged, NORMATIVE, to be cheaper than the institutional cost of a leverage claim that later proves
to have counted a phantom edge.

18.8 A fourth, narrower edge case: temporarily unreachable versus permanently retired. §18.4's
retired-recipient rule assumes the estate has a clean, timestamped RETIREMENT event to key off of, but
a recipient system can also become TEMPORARILY unreachable — a graph partition during a mid-migration
window, a system whose F1 property record is mid-reconciliation and therefore transiently absent from
the traversal, or a component whose parent systems are being re-organized without the component itself
being retired. Collapsing this case into §18.4's RETIRED-POST-MEASUREMENT treatment would be wrong in
the opposite direction from that section's own conservatism: it would permanently shrink a candidate's
active multiplier count over a recipient that is about to become reachable again, discarding real,
soon-to-be-relevant leverage rather than merely deferring its confirmation. B1's resolution, NORMATIVE:
a predicted or confirmed recipient that becomes unreachable is first marked TEMPORARILY-UNREACHABLE,
not RETIRED, and remains in the candidate's active predicted-recipient set with its measurement mode
downgraded one evidentiary step (an OBSERVED edge to a temporarily-unreachable recipient is treated as
DERIVED until reachability is confirmed again) rather than removed; only after the unreachable state
persists past a stated grace period — mirroring §3.7's onboarding-grace-period logic on the opposite
side of a system's lifecycle — does the record transition to RETIRED-POST-MEASUREMENT under §18.4's
own rule. This distinction matters operationally because the two cases call for opposite remediation:
a TEMPORARILY-UNREACHABLE recipient calls for patience and a later re-check, while a genuinely RETIRED
one calls for accepting a permanently lower active multiplier and moving on. Worked illustration: a
recipient system undergoing a scheduled two-week migration disappears from the graph's traversal
results; the candidate's leverage record marks that edge TEMPORARILY-UNREACHABLE and downgrades its
mode rather than excluding it from the active count outright; when the migration completes and the
system re-appears with its strengthens edges intact, the edge reverts to its prior mode with no
information lost — a materially different, and less costly, outcome than if §18.4's permanent-retirement
treatment had been applied prematurely to a system that was never actually gone.

FINAL LAW — PART XVIII. Ambiguous targets are unioned without double-counting, cyclic chains exclude
self-reference, retired recipients shrink the active count while preserving history, and contested
authority excludes the candidate from ranking until resolved elsewhere — every edge-case resolution
in this dataset errs toward undercounting, because an undercounted leverage point is a missed
opportunity, and an overcounted one is an institutional decision built on a graph that was never real.

PART XIX — METRICS AND ANTI-GOODHART COUNTERMETRICS
===============================================================================================

19.1 Why B1 needs countermetrics stated alongside its metrics, not after. A ranking system whose
metrics are published without their countermeasures invites exactly the metric-reflexivity pressure
Part XVII §17.5 names; this Part specifies B1's primary metrics paired, one for one, with the
countermetric that prevents each from being gamed in isolation, following the depth contract's own
requirement that metrics ship with anti-Goodhart countermetrics rather than as bare numbers.

19.2 Primary metric one: CONFIRMED MULTIPLIER RATE, and its countermetric. CONFIRMED MULTIPLIER RATE,
DEFINED, is the proportion of ADOPTED candidates (Part V §5.3) whose retrodictively-confirmed
multiplier (Part VIII) meets or exceeds their originally-presented tier (Part X) at adoption time — a
direct measure of whether B1's recommendations, once acted on, deliver what they promised. Optimized
in isolation, this metric rewards presenting ONLY very conservative, low-tier, easily-confirmed
candidates, since a Tier One candidate is far easier to confirm at its stated level than a Tier Three
one. Its countermetric, TIER DISTRIBUTION OF RECOMMENDATIONS, tracks the proportion of recommendations
made at each tier over time; a confirmed multiplier rate approaching 100% alongside a tier distribution
collapsed entirely into Tier Zero and One is a diagnostic signature of gaming, not of excellence — the
two metrics must be read together, never the first alone.

19.3 Primary metric two: TIME-TO-ADOPTION FOR TOP-TIER CANDIDATES, and its countermetric. This metric,
DEFINED, measures how long a candidate sits in UNDER-EVALUATION (Part V §5.3) before reaching ADOPTED,
for candidates in the confident recommendation tier — a proxy for whether B1's ranking is actually
accelerating institutional action on high-leverage opportunities or merely accumulating a well-scored
backlog nobody acts on. Optimized in isolation, this metric rewards recommending only candidates so
uncontroversial that adoption is trivially fast, again biasing toward Tier Zero/One safety. Its
countermetric, ADOPTION-TIER CORRELATION, checks whether time-to-adoption is INDEPENDENT of tier
(healthy — the estate adopts leverage at every tier at a comparable pace) or negatively correlated with
tier (unhealthy — high-tier candidates systematically stall, meaning the ranking surfaces them but the
estate's own adoption process cannot act on ambitious recommendations, a finding that would implicate
governance capacity rather than B1's ranking accuracy, and must be reported as such rather than
disguised as a B1 success by only ever recommending what adopts fast).

19.4 Primary metric three: POOL COVERAGE, and its countermetric. Introduced in Part XVI §16.5 as the
structural defense against the local-optimum trap, POOL COVERAGE, DEFINED, measures the proportion of
the F1 relation graph's systems that have contributed at least one candidate to the ledger within a
rolling window. Optimized in isolation (chasing coverage for its own sake), this metric rewards
generating trivial, low-value candidates from previously-uncovered systems merely to check a coverage
box, diluting the ledger with noise. Its countermetric, CONFIRMED-CANDIDATE DENSITY PER COVERED SYSTEM,
tracks not merely whether a system contributed a candidate but whether that system's candidates
achieve confirmation rates comparable to the estate average; a system newly "covered" only by
candidates that are uniformly refuted or stuck at HYPOTHESIZED is not genuinely covered in the sense
Part XVI intends, and the two metrics together distinguish real coverage expansion from coverage
theater.

19.5 Primary metric four: SYNERGY DETECTION RATE, and its countermetric. DEFINED as the proportion of
CONFIRMED pairing records (Part XI §11.6) among all pairing hypotheses entered into evaluation,
this metric measures the synergy detector's own hit rate. Optimized in isolation, it rewards proposing
only extremely conservative, near-certain pairings (structurally near-identical to already-confirmed
synergies) and avoiding genuinely novel pairing hypotheses that carry real uncertainty — precisely the
opposite of what Part XIII's recombination lab exists to explore. Its countermetric, NOVEL-PAIRING
PROPORTION, tracks what fraction of evaluated pairings involve a target combination with no prior
structural analogy in the confirmed-pairing history; a high synergy detection rate paired with a novel-
pairing proportion near zero is the diagnostic signature of a synergy detector that has stopped taking
genuine risks in favor of a comfortable, self-confirming pattern.

19.6 Primary metric five: FALSE-LEVERAGE INCIDENCE, and its countermetric. DEFINED as the rate at
which adversarial review (Part XVII §17.6) or post-adoption retrodiction (Part VIII) downgrades a
candidate's presented tier, this metric is itself already a countermetric against inflation, but it
carries its own reflexive risk: a system driven to minimize its OWN false-leverage incidence number
will, absent a check, simply stop presenting anything above Tier One, where inflation risk is lowest by
construction. Its countermetric, ADVERSARIAL REVIEW COVERAGE, tracks the proportion of Tier Two/Three
candidates that actually received a full adversarial pass (Part XVII §17.6) before presentation; a
false-leverage incidence of zero achieved by never presenting a high-tier candidate at all, or by
skipping adversarial review on the few that are presented, is worse than a modest, honestly-measured
false-leverage incidence on a healthy volume of reviewed high-tier candidates, and the two metrics
together make that distinction legible rather than hidden inside a single reassuring zero.

19.7 The meta-metric: metric-pair correlation drift over time. Because every metric in this Part is
deliberately paired with a countermetric specifically to prevent the metric from being optimized in
isolation, the single most important THING TO WATCH, NORMATIVE, is not any individual metric's trend
but whether the correlation between each metric and its countermetric DRIFTS in the direction that
would indicate the countermetric is starting to fail at its job — for instance, if confirmed multiplier
rate rises WHILE tier distribution simultaneously narrows toward Tier One, the two metrics are moving
in the gaming-consistent direction together, which is a stronger signal of a problem than either metric
alone crossing any fixed threshold, and Part XX's evaluation protocol treats this correlation-drift
check as a required, recurring audit rather than a one-time calibration.

19.8 Metric staleness: the property-honesty discipline applied to the metrics themselves. Every metric
in §19.2-19.6 is computed FROM the leverage and pairing ledgers, which means every metric inherits the
staleness risk F1 Part IV §4.3 assigns to any property computed from underlying data — a metric report
generated once and then quoted repeatedly across subsequent Owner conversations, without being
recomputed against the ledger's current state, decays into exactly the kind of stale claim masquerading
as current fact F1's Invariant Five (Part V §5.6) exists to forbid at the system-property level. This
Part extends that discipline explicitly to B1's own metrics, NORMATIVE: every reported metric value
carries its own last-computed timestamp, displayed alongside the value rather than only in an
underlying log, and a metric older than one full ranking cycle (Part IX §9.6) is marked STALE in its
own presentation rather than presented with the same visual and rhetorical confidence as a freshly
computed one. The specific institutional risk this guards against is a metric-reflexivity variant one
level removed from §19.7's meta-metric concern: not gaming the underlying ledger to produce a favorable
metric, but simply continuing to CITE an old, favorable metric snapshot past its useful life because
recomputing it might reveal an unfavorable drift — a passive, non-adversarial version of the same
distortion, arising from institutional inertia rather than intent, and one the countermetric-pairing
discipline of §19.2-19.6 does not by itself address, because staleness is a property of WHEN a metric
was computed, not of what it measures. A metrics dashboard, if this dataset's mechanisms are ever
translated to runtime per the Dataset-to-Runtime Translation section, should therefore treat "last
computed" as a first-class field on every displayed metric with the same seriousness Part V §5.2 treats
"last-reconciled" on every leverage record.

FINAL LAW — PART XIX. Every metric in this dataset ships bound to a countermetric that would fail if
the metric were gamed, and every metric carries its own staleness marker rather than being quoted past
its useful life — the meta-discipline governing all six surfaces is the same: read them together,
watch their correlation drift, distrust an old number wearing a fresh number's confidence, and treat a
suspiciously perfect metric with more suspicion, not less, because a ranking system that always looks
good on its own terms has usually stopped measuring anything real.

PART XX — EVALUATION PROTOCOL: HOW TO PROVE A LEVERAGE CLAIM, BENCHMARKS, AND RETRODICTION TESTS
===============================================================================================

20.1 Why an evaluation protocol is a distinct Part from the metrics that feed it. Part XIX specifies
WHAT is measured; this Part specifies the PROCEDURE by which a specific leverage claim, or the B1
system as a whole, earns the right to be believed — the difference between having a metric and having a
disciplined process for computing it honestly and checking it against reality, which is the operational
meaning of Article VI's "evaluation / how-to-prove" requirement applied to this dataset.

20.2 Single-candidate proof standard: what makes ONE leverage claim provable. A single leverage record
(Part V) is considered PROVEN, as distinct from merely PRESENTED, only when: its target and predicted-
recipient set survive an adversarial review (Part XVII §17.6) if the candidate is Tier Two or above; at
least one predicted edge has been upgraded to retrodictive confirmation (Part VIII) via a documented,
hierarchy-compliant baseline (Part VIII §8.3); and its final tier assignment respects §10.6's anti-
inflation rule given the confirmed evidence actually in hand. A candidate meeting all three conditions
is PROVEN at whatever tier its confirmed evidence supports; a candidate meeting none is, honestly,
still a HYPOTHESIS, however plausible its structural story, and B1's own internal vocabulary is
disciplined to never blur that distinction in how a candidate is reported.

20.3 System-wide proof standard: what makes the B1 SYSTEM, not just one candidate, provable. Beyond
any single claim, B1's own institutional value is provable only via RETRODICTION AGAINST HISTORY,
mirroring the retrodiction protocol F1 Part VIII §8.7 uses for its own level-test validation: taking a
sample of interventions the estate ADOPTED in the past — before B1 existed, or independent of B1's
recommendation — and checking whether B1's mechanism, applied retrospectively to the candidate pool as
it existed at that time, would have ranked the interventions that turned out to be genuinely high-
leverage (confirmed, with hindsight, to have amplified multiple systems) ABOVE the interventions that
turned out to be narrow, local fixes. A retrodiction sample where B1's ranking correlates positively
with hindsight-confirmed leverage is OBSERVED evidence the mechanism works; a sample where the
correlation is weak or absent is a finding this dataset requires be reported honestly rather than
excluded from the sample after the fact.

20.4 The retrodiction sample's construction, and the bias it must guard against. A retrodiction sample
constructed by the same team that built B1, drawn from interventions that team already remembers as
having been high-leverage, is contaminated by hindsight-selection bias — the sample would trivially
"prove" the mechanism because the cases were chosen for already knowing the answer. NORMATIVE: a valid
retrodiction sample is drawn from a FIXED, PRE-REGISTERED set of historical interventions — selected by
a criterion independent of their known outcome (for instance, every intervention adopted in a specific
prior time window, regardless of how it turned out) — before B1's retrospective ranking is computed
against them, exactly the pre-registration discipline that distinguishes a genuine test from a
post-hoc confirmation exercise.

20.5 Benchmark construction: the synthetic candidate suite. Because the estate's own history of
confirmed, multi-system leverage points is, per Part I §1.3's own admission, sparse (the estate's
strengthens-edge density is low by construction), a retrodiction sample alone is unlikely to be large
enough to support strong statistical claims about B1's ranking accuracy on its own. This dataset
therefore specifies, as a SEPARATE evaluation surface, a SYNTHETIC CANDIDATE SUITE — a fixed set of
illustrative, clearly-labeled synthetic interventions with STIPULATED ground-truth multipliers (some
genuinely high-leverage by construction, some deliberately narrow, some deliberately false-leverage-
shaped per Part XVI-XVII's catalogue, some deliberately ambiguous per Part XVIII's edge cases) against
which B1's mechanism can be run repeatedly as it evolves, checking whether each new version of the
detection engine, ranker, or synergy detector still correctly classifies the suite's known-answer cases
— an EXPERIMENTAL evaluation surface, explicitly marked as such, that complements rather than
substitutes for §20.3's real-history retrodiction.

20.6 Regression discipline: the suite as a floor, not a ceiling. NORMATIVE: no change to any B1
subengine (scanner, estimator, ranker, synergy detector, lab) may ship if it causes the synthetic
candidate suite's known-answer cases to be misclassified relative to their stipulated ground truth —
a false-leverage-shaped synthetic candidate that a prior engine version correctly demoted must still be
correctly demoted by a new version; a genuinely high-leverage synthetic candidate that a prior version
correctly surfaced must still be surfaced. This is the same regression-test discipline the estate's own
Python testing doctrine applies to code, imported here as a NORMATIVE evaluation requirement for a
ranking mechanism rather than for executable code, consistent with Article VIII's prohibition on
production code in this mission — the suite is specified as a fixed set of labeled cases and an
evaluation criterion, not as a test script.

20.7 What "SEALED" evaluation evidence looks like for this dataset specifically. Per Article XI, this
dataset is not SEALED merely because its Parts are content-complete; its strengthening evidence (the
Strengthening Matrix's own requirement: "a change scored by systems-amplified, ranked above a local
fix") must be checkable. The concrete evaluation artifact this Part requires before that evidence can
be claimed non-hypothetically: at least one worked, full-lifecycle trace (of the shape this dataset's
own Parts II, VII, VIII, XIII, and XIV already develop illustratively) that is not synthetic but drawn
from a real candidate in the estate's actual ledger, carried from PROPOSED through a genuine ranking
decision, with the ranking decision's correctness checkable against subsequent, independent
observation — until such a real trace exists, the Strengthening Matrix's evidence requirement for
IAS-B1 remains, honestly, UNRESOLVED, a status this dataset states plainly in its Gap Meta-Analysis
rather than asserting evidence it does not yet possess.

20.8 The held-out-suite discipline: preventing the synthetic benchmark from being overfit by its own
maintainers. Part XX §20.6 requires every B1 subengine change to pass the full synthetic candidate
suite before shipping; left unguarded, this creates its own reflexive risk, structurally identical to
Part XVII §17.5's metric-reflexivity concern but aimed at the evaluation surface rather than at the
ranking surface — an implementer facing a regression failure has an incentive to adjust the SUITE's own
stipulated ground truth to match whatever the new engine version produces, rather than fixing the
engine to match the suite's original, independently-reasoned ground truth. This inverts the entire
purpose of a regression benchmark, converting it from a check on the engine into a check the engine
always passes by construction. Countermeasure, NORMATIVE: the synthetic candidate suite (Part XX §20.5)
is partitioned into a VISIBLE subset, used during ordinary development and iteration, and a HELD-OUT
subset, whose specific cases and stipulated ground truth are not consulted during subengine development
and are run only at the final pre-ship gate, mirroring the standard train/held-out discipline the
estate's own testing doctrine already applies to code more broadly, imported here for an evaluation
suite rather than a test suite. A change that passes the visible subset but fails the held-out subset is
NOT shipped, and — critically — the held-out subset's ground truth is never revised to accommodate a
failing change; only the ENGINE is revised, and if a genuine, well-reasoned case exists for updating a
held-out case's stipulated ground truth (the original stipulation was itself wrong), that revision is a
separate, explicitly-logged decision, documented with its own rationale, never a silent adjustment made
in the same change that needed the adjustment to pass. This discipline is what keeps §20.6's regression
requirement a genuine constraint on B1's evolution rather than a formality that evolves in lockstep with
whatever the engine happens to do.

FINAL LAW — PART XX. A leverage claim is proven candidate by candidate through adversarial review plus
retrodictive confirmation; the B1 system as a whole is proven through pre-registered retrodiction
against real history and regression-tested against a held-out synthetic suite whose ground truth never
bends to match a failing engine — and until a real, independently-checkable full-lifecycle trace exists
in the estate's own ledger, this dataset's own strengthening claim remains a hypothesis honestly
labeled, not a fact asserted early.

PART XXI — INTEGRATION: IAS-B2, IAS-E1, AND THE ESTATE KNOWLEDGE BASE
===============================================================================================

21.1 Why integration is specified rather than assumed. The Compounding Graph's amplification spine
names B1 as a specific node — "IAS-E1 usage data → IAS-B1 Cognitive Multiplier (finds the highest-
leverage change) → IAS-B2 Insight Synthesis (cross-dataset insights feed new FD deposits / ACIS laws)"
— but a graph edge in that document is a claim about relationship, not a specification of the
interface the relationship runs over. This Part supplies that specification for B1's two named
neighbors and its relationship to the estate's broader knowledge base, as SPECIFICATION only, per
Article VIII.

21.2 The inbound edge from IAS-E1: usage data as a ranking input, not a candidate source. IAS-E1
(Observability Fabric) produces, per the Strengthening Matrix's own description of E1, cross-system
visibility on which systems are used, dead, or dark, plus flow and bottleneck data. B1's consumption of
this feed is narrow and specific, NOT a wholesale candidate-generation source (that role belongs to FD,
ACIS, hard_rules, the learners, and the lab, per Part I-II and Part XIII): E1's usage data feeds Part
IV §4.5's aggregation weighting specifically — a recipient system's CONSUMERS property (F1 Part IV),
which E1 is best positioned to keep freshly reconciled via its own observability mechanism, is exactly
the weighting factor B1's aggregation already requires, and rather than B1 building a second, parallel
consumer-counting mechanism (which would duplicate E1's own object per Article I), B1 CONSUMES E1's
reconciled consumer counts as an input, a FEEDS relation in F1 Part VI §6.2 terms, strengthening B1's
own measurement accuracy without either system rebuilding the other's telemetry.

21.3 The inbound edge from IAS-E1, second channel: dead-capability and dark-matter flags as negative
priors. E1's dead-capability and dark-matter findings (systems or capabilities that are unused or
invisible to the estate's own accounting) are a second, distinct input: a candidate whose target or
predicted-recipient set includes a system E1 has flagged as dead or dark is a candidate whose
multiplier should be discounted, NORMATIVE, because amplifying an already-dead system's function is not
genuine leverage in the sense Part I §1.2 defines it — the recipient must be a system whose improved
function has consumers to matter to. B1's aggregation (Part IV §4.5) therefore treats an E1 dead-
capability flag on any predicted recipient as an automatic downweight, distinct from and additional to
the consumer-count weighting of §21.2, closing a specific gap the earlier Parts left implicit: a wide
fan-out to systems nobody uses is not, by this dataset's own logic, a wide fan-out worth acting on.

21.4 The outbound edge to IAS-B2: leverage points as candidate insight-synthesis material. IAS-B2
(Cross-Dataset Insight Synthesis, Non-Duplication Ledger candidate 13, "insights present in no single
dataset") reads across multiple datasets to surface insights none holds alone — a DIFFERENT object
from B1's ranking of candidate interventions (Part II's boundary discipline, restated: B2's object is
the insight, B1's object is the ranked change). The outbound edge, per the Compounding Graph's spine, is
that B1's CONFIRMED leverage points (Part V's ADOPTED status, ideally with §20.2's PROVEN standard met)
become raw material B2 can synthesize across: a pattern of WHICH KINDS of interventions repeatedly turn
out to be high-leverage (shared-vocabulary fixes, guard-clause idioms, cross-hook event channels) is
itself a cross-dataset insight B2's own object is built to surface, but which B1 itself does not
attempt to generalize into a principle — that generalization, per Article II's ensemble-object
boundary, belongs to B2 and, downstream of B2, potentially to ACIS's law schema, not to B1, which
FEEDS the raw confirmed-leverage-point history forward rather than performing the synthesis itself.

21.5 Why B1 does not perform its own insight synthesis, even though it could. It would be technically
possible for B1 to add a "what patterns recur across my own confirmed leverage points" capability
internally; this Part explicitly declines that expansion, NORMATIVE, because doing so would duplicate
B2's distinct object (candidate 13's disposition: BUILD → IAS-B2) under B1's own roof, the exact
naming-only-novelty risk Article I exists to prevent, applied reflexively to this dataset's own design
restraint rather than only to external proposals. B1's discipline here is itself a worked instance of
minimal-sufficient-doctrine (Part XV) applied to B1's OWN scope: the smaller, correctly-bounded system
(rank candidates, hand confirmed history to B2) is preferred over the larger, scope-creeping one (rank
candidates AND synthesize patterns across them), because the smaller one is sufficient for B1's stated
object and the larger one would not add genuine new leverage-discovery capability, only redundant
insight-synthesis capability B2 already owns.

21.6 Integration with the estate knowledge base, broadly. Beyond the two named IAS neighbors, B1's
leverage ledger (Part V) and its confirmed, SEALED-CAPABILITY recombination pairings (Part XIV) are
themselves a knowledge asset the estate's broader knowledge infrastructure (the vault, the compound-
learnings pipeline, the session-lesson ledger referenced throughout this dataset's worked examples)
should be able to consume — not as a NEW knowledge store duplicating the vault's own object, but as a
specific, typed VIEW over the vault's existing content, filtered and re-organized by leverage rather
than by chronology or by source. This is DERIVED as a natural consequence of Part II §2.4's dependency
direction: because B1 consumes FD deposits, ACIS claims, and learner outputs as raw material without
re-storing them in a parallel structure, the leverage ledger's own records are lightweight index-and-
score layers over knowledge the vault already holds canonically, never a second copy of the underlying
content.

21.7 Worked integration trace: one candidate's full cross-system journey. An FD deposit (Part II
§2.6's guard-clause example) enters B1's ledger. E1's consumer-count data (§21.2) weights its predicted
recipients during aggregation; none of its recipients carry an E1 dead-capability flag (§21.3), so no
downweight applies. The candidate is scanned, estimated, ranked, adopted, and confirmed (Parts VII-X,
XX). Once ADOPTED and confirmed, its record — along with several structurally similar confirmed
leverage points accumulated over prior cycles, all sharing the "shared guard-clause idiom" shape — is
made available to IAS-B2, which (outside this dataset's own scope, per §21.4-21.5) may or may not
synthesize a cross-dataset insight from the pattern; that synthesis, if it occurs, is B2's finding, not
B1's, and this dataset's own record of the trace stops at the boundary of the FEEDS edge, exactly where
Article II's ensemble-object discipline says it should.

21.8 What B1 explicitly does not integrate with, and how staleness in an integration input is handled.
Two boundaries worth naming precisely rather than leaving implicit. First, B1 does NOT integrate
directly with IAS-A2's arbitration mechanism (the Compounding Graph's contested-duplicate resolver):
where Part XVIII §18.6 marks a candidate UNRESOLVED-AUTHORITY, B1's ledger simply excludes it from
ranking and waits — it does not open a channel to A2 requesting priority resolution, does not escalate,
and does not track A2's own queue. This is a deliberate FEEDS-only boundary, NORMATIVE: A2 resolves
authority disputes for reasons entirely independent of any single candidate's leverage score, and a
leverage-ranking system that could nudge an authority arbitration's priority by how highly it ranks
the blocked candidate would be exercising exactly the kind of authority Invariant Three (F1 Part V)
reserves elsewhere — B1 waits for A2's own resolution timeline like any other consumer, with no
privileged channel. Second, staleness in the E1 integration itself (§21.2-21.3) is handled explicitly
rather than assumed away: E1's consumer-count and dead-capability feeds carry their own LAST-RECONCILED
timestamp, and where that timestamp exceeds the estate's staleness threshold (the same threshold F1
Part IV §4.3 applies to any property record), B1's aggregation (Part IV §4.5) does not silently use the
stale value as if it were current — it downgrades the affected weighting factor's own confidence,
which propagates into the candidate's overall confidence interval exactly as an UNRESOLVED-PATH scanner
flag does (Part VII §7.5), rather than either blocking the candidate outright or pretending the input
is fresher than it is. Worked illustration: E1's observability fabric experiences a two-day outage;
during that window, any candidate whose aggregation would normally consult E1's consumer-count data
instead uses the LAST known reconciled value with an explicit staleness discount applied to its
resulting confidence interval, and Part IX's ranker surfaces that discount alongside the score rather
than silently treating a two-day-old count as equivalent to a fresh one — the same "absence of fresh
evidence is not evidence of absence" discipline §3.7's cold-start handling and Part VII §7.5's
contested-path handling both apply, here at the cross-dataset integration boundary rather than at the
single-dataset graph-traversal boundary.

FINAL LAW — PART XXI. B1 consumes E1's usage and dead-capability data as ranking inputs, feeds its
confirmed leverage history forward to B2 as raw material for cross-dataset synthesis, and treats the
estate's own vault as the canonical knowledge store it indexes rather than duplicates — every
integration edge in this Part is a FEEDS or STRENGTHENS relation, never a re-implementation of a
neighbor's distinct object.

PART XXII — EVOLUTION, OPEN QUESTIONS, AND THE GOVERNANCE BOUNDARY
===============================================================================================

22.1 How B1 itself is expected to evolve, under its own discipline. Per Part VI §6.7 and Part XI §11.6,
B1 applies its own ranking and synergy discipline reflexively to its own subengines; this Part specifies
the EXPECTED shape of that evolution over time, not as a roadmap (out of scope for a dataset under
Article VIII's no-production-code constraint) but as a set of DERIVED trajectories the mechanism's own
logic implies. The scanner's depth bound (Part VII §7.2) is expected to be tuned as retrodictive
evidence (Part VIII, Part XX §20.3's retrodiction sample) accumulates on whether depth-two chained
edges are, in practice, confirmed at a rate that justifies their current discount weighting or whether
the weighting itself needs recalibration — an EXPERIMENTAL question this dataset does not resolve, only
frames as the specific, checkable form future recalibration should take.

22.2 Open question one: does the local-optimum trap have a structural, rather than merely a labeling,
mitigation. Part XVI §16.6 commits B1 to honest labeling (the pool coverage statement) as the
achievable standard given §16.4's acknowledged sampling-gap limitation. Left UNRESOLVED: whether a
future version of the detection engine could reduce the sampling gap structurally — for instance, by
periodically forcing the scanner to traverse from RANDOMLY selected, rather than only candidate-
triggered, starting systems, trading some efficiency for genuine coverage breadth. This dataset states
the question rather than answering it, because answering it would require exactly the kind of
retrodictive evidence (does forced-random scanning actually surface leverage the candidate-triggered
scanner misses) that does not yet exist.

22.3 Open question two: the F1 self-reference boundary, inherited and sharpened. F1 Part VII §7.6 left
its own second-order classification CONTESTED. B1's Tier Three definition (Part X §10.5) noted its own
proximity to that same boundary — a Federation-Wide Lever operating on a shared substrate is, by the
level test, very close to being an ensemble-level object in its own right rather than merely a
candidate B1 ranks. UNRESOLVED: at what point, if ever, does a repeatedly-confirmed Tier Three lever
stop being "a candidate B1 evaluates" and start being "a standing capability that should itself be
formalized as part of F1's own ontology or invariant set" — a question this dataset defers to
whatever estate-level governance process would own such a formalization decision, consistent with
Part XIV §14.4's authority boundary, rather than resolving it unilaterally here.

22.4 Open question three: cross-project federation of the leverage ledger. This dataset's worked
examples and mechanisms are specified at the scope of a single project's estate; Article II's ensemble
boundary (inherited from F1 Part III §3.4) scopes B1 to the estate it governs. Whether leverage points
confirmed in ONE project's ensemble transfer, even partially, to a structurally similar system in a
DIFFERENT project's ensemble — a cross-project federation analogous to IAS-D2's own cross-project
immune-fingerprint federation — is named here as a CONTESTED possibility: plausible by analogy to D2's
own confirmed mechanism, but untested for B1's specific object, and explicitly not claimed as a current
B1 capability.

22.5 Open question four: the estimator's cost ceiling as the practical bottleneck on B1's own growth.
Part VIII §8.5 named estimator invocation as a scarce resource; as the estate's candidate pool grows
(Part XIX's pool-coverage metric, if healthy, implies growth), the estimator's cost discipline becomes
the binding constraint on how much of B1's own predicted leverage can ever be upgraded to retrodictive
confirmation (Part IV §4.3) rather than left at prospective-structural status indefinitely. This is
named as a HYPOTHESIZED future bottleneck, not yet OBSERVED at the estate's current scale, but flagged
because Part XX §20.7's SEALED-evidence requirement depends on real confirmed traces existing, and a
bottleneck on confirmation capacity is a direct threat to this dataset's own path to non-hypothetical
sealing.

22.6 Open question five: whether the tier vocabulary itself should be periodically re-calibrated against
the estate's own growth. Part X fixed four tiers with illustrative multiplier ranges (Zero at one, One
at two-to-three, Two at four-to-six, Three at a majority-spanning reach) calibrated against the estate's
CURRENT strengthens-edge density (F1 Part VI §6.10's observed sparsity). As the estate grows — more
systems, more relations, a denser graph — the same absolute multiplier that qualifies as Tier Two today
may, in a much denser future graph, represent a proportionally smaller and less distinguishing reach,
while a multiplier that would be extraordinary today could become merely ordinary. UNRESOLVED: whether
tier boundaries should be defined in ABSOLUTE terms (as this dataset currently does, for auditability
and stability of the shared vocabulary, per Part X §10.1's own reasoning) or RE-CALIBRATED periodically
against the estate's measured edge density, trading some vocabulary stability for continued
discriminative power as the ensemble scales. This dataset commits to the absolute definition for now,
NORMATIVE, precisely because a shifting tier boundary would undermine the plain-language legibility Part
X §10.1 built the tier system to provide — "Tier Two" should mean the same thing in a conversation held
a year apart — but flags the tension explicitly rather than pretending the absolute choice is permanently
costless as the estate's own density changes.

22.7 What this dataset would need to observe before any of its open questions could be closed. Each of
the five UNRESOLVED questions above (§22.2-22.6) shares a common shape: none can be answered by further
reasoning from this dataset's existing Parts alone, because each depends on evidence — retrodiction
results, cross-project trial outcomes, estimator cost trends at scale, tier-density correlation over
time — that does not yet exist and cannot be manufactured by argument. This is stated as a deliberate
closing discipline rather than an oversight: a dataset that resolved every open question by assertion,

DECISION LOG
===============================================================================================

DL-1. Decision: anchor "leverage" to the strengthens-edge subgraph specifically (Part III §3.2), not to
the higher-cardinality feeds subgraph or to an unweighted union of all six F1 relation types.
Alternatives considered: (a) count all outbound edges of any type as fan-out; (b) define a new,
B1-specific relation type layered on top of F1's six. Why this: F1 Part VI §6.10 itself flags feeds as
the highest-cardinality, least decision-critical relation — anchoring there would drown leverage signal
in structural noise, while a new relation type would fracture the shared ontology F1 exists to unify.
Trade-offs: strengthens-anchoring makes B1 dependent on the estate's strengthens edges being reasonably
well-curated, which is itself uneven across the estate. Reversal conditions: if retrodiction (Part XX
§20.3) shows strengthens-anchored predictions systematically underperform a feeds-inclusive model,
revisit with feeds admitted at a steep, separately-calibrated discount rather than excluded outright.

DL-2. Decision: split detection into two collectors at two speeds (scanner continuous/cheap, estimator
invoked/expensive) rather than one unified, always-deep pipeline (Part VI §6.2, §6.5). Alternatives
considered: (a) run full retrodictive measurement on every new candidate immediately; (b) run only
structural scanning and never retrodict at all. Why this: (a) is computationally and institutionally
unaffordable at estate scale (Part VIII §8.5); (b) never earns the confidence Part IV's evidentiary
ladder requires for top-tier claims. Trade-offs: the two-speed split means most candidates sit at
prospective-structural confidence for a materially longer time than a hypothetical always-deep pipeline
would produce. Reversal conditions: if estimator cost falls dramatically (a future tooling change), the
gate at Part VI §6.5 should be revisited toward running the estimator more eagerly.

DL-3. Decision: gate low-confidence candidates OUT of the ranker's confident tier BEFORE scoring, rather
than scoring everything on one combined scale and letting confidence act only as a tiebreaker (Part IX
§9.4). Alternatives considered: (a) a single combined score = reach × depth-of-effect × confidence,
ranked directly; (b) confidence as a post-hoc filter applied only to the top N results. Why this: Part
IV §4.6's worked M-vs-N comparison shows a combined-score approach can invert the correct recommendation
when a wide, unconfirmed candidate multiplies against a narrow, proven one. Trade-offs: gating before
scoring means some genuinely high-value but under-evidenced candidates are invisible in the confident
tier until evidence catches up, deferring institutional attention. Reversal conditions: if Part XIX's
metric-pair correlation drift (§19.7) shows the gate itself is being gamed by cheaply manufacturing thin
confirmation just to clear the gate, tighten the gate's own evidentiary bar rather than relaxing it.

DL-4. Decision: require the synergy detector's pairing records to carry a HARD gate (non-empty joint-only
recipient set) rather than a soft confidence-weighted synergy score that could still surface a weak or
empty-overlap pair (Part XI §11.3). Alternatives considered: (a) score every candidate pair on a
continuous synergy scale and let the ranker filter; (b) require human review of every pairing before
ledger entry. Why this: Part XII's adversarial catalogue (shared-cause illusion, narrative-plausible
non-synergy) demonstrates that plausible-looking pairs routinely fail to show a genuine joint-only
effect, and a soft score would let these narrative-plausible non-synergies accumulate in the ledger as
noise. Trade-offs: the hard gate means some pairs with a real but very small joint effect may fail to
clear the non-empty threshold if measurement precision is coarse. Reversal conditions: if precision
improves enough to reliably detect small joint effects, consider a minimum-magnitude threshold instead
of a strict non-empty gate.

DL-5. Decision: fold minimal-sufficient-doctrine in as a TIEBREAKER only, applied strictly after leverage
scoring, never as a pre-filter on candidate generation (Part XV §15.3). Alternatives considered: (a)
apply a size cap to candidate generation itself, refusing to even scan large interventions; (b) weight
size directly into the two-axis leverage score. Why this: DERIVED from Article VI's own depth-without-
padding discipline applied reflexively — filtering by size before leverage is known would systematically
bias the candidate pool toward small, easy-to-conceive interventions and starve the recombination lab's
search space. Trade-offs: as a pure tiebreaker, minimal-sufficient-doctrine never influences which
candidates make it into the ledger in the first place, only which of several equally-scored candidates
is preferred — a narrower influence than a size-aware scoring model would have. Reversal conditions:
none identified within this dataset's own scope; a change here would require F1's own compression-family
disposition to be revisited first.

DL-6. Decision: require every wide-reach (Tier Two/Three) recommendation to pass a SEPARATE adversarial-
review pass (Part XVII §17.6) rather than folding adversarial checks into the ordinary scanner/estimator
pipeline. Alternatives considered: (a) build the four gaming-vector checks directly into the scanner and
estimator so every candidate is adversarially checked by default; (b) rely on Owner review alone as the
final adversarial check. Why this: folding adversarial checks into the default pipeline for every
candidate (including Tier Zero/One) would be a disproportionate cost for the risk level (Part XVII
§17.1's concentration observation — gaming risk concentrates at the wide-reach tiers); relying on Owner
review alone offers no structural guarantee the specific gaming vectors are checked. Trade-offs: Tier
One candidates receive no adversarial pass at all under this design, meaning smaller-scale gaming at
that tier could go unchecked longer. Reversal conditions: if Part XIX's false-leverage-incidence metric
shows a meaningful rate of Tier One inflation, extend adversarial review down to Tier One.

DL-7. Decision: keep FD's sealing step entirely innocent of ranking, and require B1 to consume sealed
deposits as a downstream, pool-wide ranker rather than extending FD itself with a per-deposit multiplier
tag computed at seal-time (Part II §2.7). Alternatives considered: (a) tag each deposit with a rough
leverage estimate at the moment FD seals it, sorted on later with no separate ranking engine; (b) have
FD periodically re-open and re-score its own back-catalogue of deposits on a fixed schedule. Why this:
§2.7's own argument — a multiplier is only meaningful relative to the FULL current candidate pool and
the CURRENT state of F1's relation graph, neither of which is visible from inside a single session's
sealing step, and a seal-time tag would either freeze stale or force FD into a continuously-running
ranking role it was never built to hold, duplicating B1's object under FD's roof. Trade-offs: keeping
FD innocent of ranking means every deposit sits unranked, however briefly, between sealing and B1's
next ingestion cycle (Part VI §6.1) — a real but bounded latency cost, never a correctness cost.
Reversal conditions: none identified within this dataset's own scope; revisiting this decision would
require re-litigating the FD/B1 boundary Part II exists to foreclose, not merely re-tuning a parameter.

What gap this dataset closes. Before IAS-B1, the estate had many mechanisms that RECORD a single true
thing (FD's deposits, ACIS's promoted claims, hard_rules' sealed entries, every learner's pattern
capture) and zero mechanisms that SEARCH ACROSS the accumulated pool of recorded things for the specific
subset whose adoption would improve multiple other systems at once. The gap was not a recording gap —
the estate over-provisions recorders, as the Non-Duplication Ledger's nine-candidate Family B cluster
attests — it was a RANKING-BY-CROSS-SYSTEM-EFFECT gap, unaddressable by any single resource-level
recorder because the ranking requires reading the F1 relation graph across systems, an ensemble-level
operation none of the resource-level recorders can perform from inside their own scope.

Systems strengthened, with the mechanism each time. FD is strengthened because its accumulating deposit
volume, once large enough that manual triage exceeds human attention (Part II §2.5), now has a
downstream consumer that surfaces which deposits matter most, without FD needing to build that
ranking function itself. ACIS is strengthened because its promoted claims become candidate leverage
points B1 can rank alongside every other producer's output, giving ACIS's evidence-ladder work a
concrete downstream use beyond the ladder's own promotion criteria. hard_rules and rule_compiler are
strengthened because a shared-vocabulary or shared-schema leverage point (Part III §3.6's worked
example) that touches the digest compiler's own stability is exactly the kind of candidate B1's engine
is built to surface and rank, giving the digest compiler's maintainers a prioritized signal rather than
an undifferentiated stream of proposed changes. Every learner in the estate is strengthened because B1
gives learner output a second life as ranked, cross-system-evidenced institutional priority rather than
a single-session artifact that competes for attention purely on recency.

Duplications avoided. This dataset absorbed five ledger candidates (11, 12, 15, 19, 20) that, examined
independently, would each have built a parallel measurement or generation mechanism duplicating parts
of FD, hard_rules, or evolution_engine's single-dataset recombine operation under a new name — the
Non-Duplication Ledger's own overlap column names this risk explicitly for each. By architecting the
synergy detector and recombination lab as SUBENGINES of one detection engine (Part VI §6.2) sharing one
ledger schema (Part V), rather than as five freestanding systems, this dataset avoided building five
separate, only-partially-compatible ranking and recording mechanisms where one, disciplined system
suffices.

New capabilities genuinely introduced. The two-axis (reach, depth-of-effect) scoring model with
confidence-interval gating (Part IX) did not exist anywhere in the estate before this dataset. The
tier vocabulary (Part X) gives the estate a shared, plain-language way to talk about leverage magnitude
that no prior mechanism supplied. The joint-only-recipient-set formalization of synergy (Part XI §11.2)
gives the estate a falsifiable definition of "two capabilities beating their sum" where previously the
claim, when made informally, had no measurement standard at all. The controlled-trial, reversibility-
gated recombination lifecycle (Parts XIII-XIV) gives the estate a governed path from a novel structural
hypothesis to a permanent ontology addition that did not exist as a named, staged process before.

Open risks, stated honestly. The local-optimum-sold-as-global trap (Part XVI) is mitigated by honest
labeling, not eliminated — this dataset's own analysis (§16.4) concludes the sampling gap cannot be
fully closed by B1's current mechanism set, an acknowledged, load-bearing limitation rather than a
solved problem. The estimator's cost ceiling (Part XXII §22.5) is a HYPOTHESIZED future bottleneck on
how much of the ledger can ever reach retrodictive confirmation, not yet observed at current scale but
structurally inevitable as the candidate pool grows. Cross-project federation of confirmed leverage
points (Part XXII §22.4) remains CONTESTED and unimplemented. And, most fundamentally, per Part XX

UNRESOLVED pending a real, independently-checkable full-lifecycle trace in the estate's actual ledger;
every worked example in Parts I-XXII is illustrative and synthetic, and the dataset says so rather than
implying otherwise.

===============================================================================================
ANTI-PATTERNS
===============================================================================================

AP-1. SKIP-THE-EVALUATION-TIER (Part V §5.3). The most likely implementation error: a candidate moves
directly from PROPOSED to ADOPTED, bypassing UNDER-EVALUATION, because a candidate looks obviously
good, or because deep-work time pressure makes the estimator's cost (Part VIII §8.5) feel like an
avoidable step. Why it happens: the ranker's advisory-only authority (Part IX §9.1) creates no
technical barrier to an Owner or process simply acting on a PROPOSED record's plausible-looking
prospective-structural fan-out without waiting for confirmation. Prevention: treat the UNDER-EVALUATION
transition as a hard status-machine requirement enforced wherever leverage records are consumed for
adoption decisions, not merely as a NORMATIVE suggestion in this text — any adoption workflow reading
the ledger should refuse to surface a PROPOSED-status record as adoption-ready.

AP-2. COLLAPSING THE TWO-AXIS SCORE INTO ONE NUMBER (Part IX §9.3). The second most likely error: an
implementer, for dashboard simplicity, multiplies or sums reach and depth-of-effect into a single
"leverage score" for display, discarding the deliberate separation Part IX establishes. Why it happens:
a single sortable number is easier to build a UI around than a two-dimensional comparison, and the
motivation is almost always benign (usability) rather than an attempt to game anything. Prevention:
if a single display number is genuinely required for triage, compute it as a LABELED derivative (e.g.,
"reach-weighted score, see full record for depth-of-effect") that always links back to the two-axis
record, never as the record's primary or only representation — Part IV §4.6's M-vs-N worked example is
the concrete cautionary case to re-check against before shipping any single-number simplification.

AP-3. TREATING PROSPECTIVE-STRUCTURAL MULTIPLIER AS EQUIVALENT TO RETRODICTIVE MULTIPLIER (Part IV
§4.2-4.3, Part X §10.6). The third most likely error, and the one this dataset's own failure-mode
catalogue (Part XVI) is most directly aimed at: a candidate's scanner-output multiplier (Part VII) is
presented, reported, or acted upon as though it were confirmed, simply because the scanner's traversal
is deterministic and easy to trust as "the graph says so." Why it happens: prospective-structural
measurement is DERIVED from OBSERVED graph structure (Part IV §4.2), which feels stronger than it is —
the graph structure is real, but the PREDICTED EFFECT of acting on it is not yet tested. Prevention:
every leverage record's presentation format should carry its measurement mode as prominently as its
score (Part V §5.2's schema already requires this at the data level); any downstream consumer,
dashboard, or report that drops the measurement-mode field when displaying a multiplier has reintroduced
exactly the false-leverage risk Part XVI names, regardless of how carefully the underlying ledger itself
was built.
