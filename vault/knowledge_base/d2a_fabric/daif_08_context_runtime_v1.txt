DAIF-08 — CONTEXT ASSEMBLY AND MISSION RUNTIME

PART I — THESIS: THE PACK IS NOT THE RUNTIME

1.1 The parents, named before the claim is made. DAIF-08 is an extension, not an invention, and the honest way to open an extension is to name what already exists so that nothing here can be mistaken for a discovery. The Power Pack already compiles minimal context packs: GK-06, the Route Compiler of the Knowledge Navigation Kernel, takes a task and returns a compressed route — coordinates, minimal evidence, traps, next action — and that is a context pack under a different name, produced by a real engine with a real index behind it. The Power Pack already tiers memory: CO-04, the Context Virtual Memory, holds knowledge across hot, warm, cold, and external tiers and decides what is resident. The Power Pack already caps the budget: CO-00 states the context budget as a governed quantity rather than an emergent accident. The Power Pack already loads on demand: `jit_skill_loader` injects a matched module's full body at the moment a prompt calls for it, bounded and fail-open. The Power Pack already carries a session across a boundary: `session_compiler` compiles what a session was, and Lazarus recovers it. Five parents, all live, all with code on disk. DAIF-08 rebuilds none of them, forks none of them, and re-narrates none of them. If this dataset can be summarized as "the estate should assemble minimal context," it has said nothing the estate did not already know and should not have been built, and the D2A+ pipeline that governs this corpus would have returned REFERENCE rather than EXTEND.

1.2 The gap the parents leave, stated precisely. What the estate possesses is a pack BUILDER. What it lacks is a pack RUNTIME. GK-06 answers the question "what should this task be given" once, at the front of the task, from an index and a heuristic, and then it is finished — its work product is handed to an actor and the compiler never hears from it again. CO-04 tiers what is held but does not know which tier a specific mission needed and did not get. CO-00 caps spend but cannot tell an over-spend caused by a pack that was too large from an over-spend caused by a pack that was too small and forced the actor to hunt. `jit_skill_loader` loads a module when a prompt matches a pattern, which is a keyword reflex, not a mission-scoped decision made because an actor discovered a gap in its own knowledge. And no component anywhere in the estate records the single most valuable event in the whole domain: the moment an actor, mid-mission, discovers that the context it was handed does not contain something it needs. That event happens constantly. It is at present invisible. It leaves no trace, produces no data, and teaches the compiler nothing, which is why the compiler is exactly as good on its thousandth pack as it was on its first.

1.3 The delta, stated without inflation, in five terms. DAIF-08's genuine contribution is five mechanisms, and it claims no others. First, the page fault: the moment an actor discovers its pack is insufficient becomes a first-class, named, typed, authorized, and recorded event rather than an unremarked improvisation. Second, expansion handles: a pack ships with the typed means to request more, inherited directly from CIR-11's handle slot, so an actor no longer has to guess what might exist beyond its edges — the edges are declared. Third, progressive disclosure as a governed runtime: context arrives in levels, ascended only on a trigger and only against a budget, rather than as one flat delivery decided in advance. Fourth, profile-guided optimization: the compiler learns from what was actually loaded, actually used, never touched, faulted for, or missing-and-never-requested, so that the next pack for a mission of the same class is compiled from observed behavior rather than from a designer's intuition. Fifth, the mission contract that binds all four, because a fault, a handle, a level, and a profile are all defined relative to a goal and are undefined without one. DAIF-08 also absorbs the D2A-18 Mission Assembly candidate outright: the Mission Context Pack IS the output of mission assembly, and spinning a separate engine to produce it would have been the duplication this corpus exists to prevent.

1.4 A predicted pack is a hypothesis. The argument that justifies all five mechanisms is one sentence, and everything downstream is its consequence. A context pack assembled before a mission runs is a PREDICTION about what that mission will need, made by a compiler that has not yet seen the mission's actual reasoning, its actual obstacles, or the actual state of the world it will encounter. A prediction is a hypothesis. The estate already knows what to do with a hypothesis: DAIF-03 gates it on fidelity, ACIS refuses to promote it above the level its evidence supports, SQI insists that a gate never observed to refuse is indistinguishable from an inert one, and CO-12's Telemetry-Before-Claims forbids asserting a result without a measured triple. The one thing the estate has never done with the context-pack hypothesis is TEST it. It is compiled, handed over, and its correctness is assumed by the silence that follows. This is the singular anomaly DAIF-08 corrects, and the correction is small: give the hypothesis a way to be falsified during the mission that depends on it, and record the falsification.

1.5 The two failure directions, and why only one of them is currently visible. A pack that is wrong is wrong in one of exactly two directions. It omits what was needed, in which case the actor either guesses — producing confident work built on a fabrication — or halts, or wanders the estate looking for what it should have been handed. Or it includes what was not needed, in which case the mission pays tokens for units nobody reads, and worse, the units that mattered are buried among units that did not, because relevance is a signal and a pack that admits everything transmits none. The second failure is at least partly visible today, because it shows up as spend and CO-00 counts spend. The first failure is entirely invisible, because a mission that quietly proceeded on a guess and produced plausible wrong work emits exactly the same telemetry as a mission that proceeded on knowledge and produced right work — until the rework arrives, by which time the causal link back to the missing context unit has been lost. An estate that can only see one of its two failure directions will optimize itself into the other one, which is precisely what happens when a compiler is tuned for smallness with no instrument that can detect insufficiency.

1.6 Prediction plus correction equals a loop. The conversion DAIF-08 performs is from a one-shot guess to a closed loop, and the loop has four legs. The compiler predicts, which is GK-06's existing work, unchanged. The runtime executes and, at the moment prediction fails, faults — visibly, with a type and a reason. The fault is authorized against budget and scope, resolved by loading exactly the missing unit rather than its whole neighborhood, and RECORDED. The recording accumulates into a profile, and the profile recompiles the next pack. Each leg is cheap on its own; the value is entirely in their closure, and the leg the estate is missing is the third. A fault that is resolved but not recorded improves one mission and teaches nothing. A fault that is recorded but never fed back is telemetry theater. The loop is the deliverable, and a partial loop should be counted as an unbuilt one.

1.7 What DAIF-08 explicitly does not own. It does not own the index; graphify's GK-01 and GK-05 own coordinates and the graph. It does not own the selection heuristic that decides a pack's initial content; GK-06 owns it and DAIF-08 supplies it with better inputs. It does not own the memory tiers; CO-04 owns residency, and DAIF-08's ascent and descent are requests against it, never a parallel tiering. It does not own the budget; CO-00 owns it, and every fault authorization and every level ascent composes it rather than replicating a second ceiling. It does not own the intermediate representations; DAIF-02 defines CIR-11 and DAIF-08 consumes it. It does not own the obligation lifecycle; DAIF-07 owns it, and an unresolved fault that must be carried forward is registered there. It does not own invalidation; DAIF-21 owns it, and re-assembly on a reality change is triggered by DAIF-21, not detected here. What remains after all of these subtractions is exactly the five mechanisms of 1.3, and that residue is the whole of DAIF-08's sovereign territory. The discipline of naming the residue is itself the corpus law, PR-DUPLICATE-TO-ADVANTAGE-001, applied to its own dataset.

1.8 Reading order and the actors. Part II compiles the mission, because nothing here is definable without it. Part III gives the pack its runtime contract, consuming CIR-11's structure. Part IV gives the pack the means to admit its own incompleteness. Part V makes the moment of discovered insufficiency an event. Part VI governs the levels along which context is disclosed. Part VII closes the loop by learning from the record. The actors throughout are four: the compiler, which predicts; the runtime, which executes, faults, authorizes, and records; the actor, which reasons and is the only party that can detect a gap in its own knowledge; and the authority, which owns budget and scope and is the only party that can widen either.

PART I FINAL LAW. The estate already owns a context-pack BUILDER — GK-06 compiles the minimal pack, CO-04 tiers what is resident, CO-00 caps the spend, `jit_skill_loader` loads on demand, and `session_compiler` with Lazarus carries a session across a boundary — and what it lacks is a context-pack RUNTIME, because a pack assembled by prediction is a hypothesis about what a mission will need, a hypothesis that cannot be corrected mid-mission fails in exactly two directions of which only the wasteful one is currently visible, and the invisible direction — the omission that the actor silently guesses past — produces confident wrong work and no signal at all; DAIF-08 therefore contributes page faults, expansion handles, progressive disclosure, profile-guided optimization, and the mission contract that binds them, and nothing else, because everything else already exists and rebuilding it would be the duplication this corpus was founded to refuse.

PART II — THE MISSION CONTRACT

2.1 The absorption, declared. This Part is where the D2A-18 Mission Assembly candidate lands. The canonical map assigned it the verdict FOLD with a one-line justification — the Mission Context Pack IS the mission-assembly output — and that justification is honored literally here: mission assembly is not a separate engine with its own lifecycle, its own artifacts, and its own gates, because a mission that is assembled and then not used to select context has assembled nothing, and a pack selected without a mission has selected against nothing. The two are one operation seen from two ends. What D2A-18 correctly identified, and what survives its folding, is that the operation has a FIRST STEP that the estate was performing implicitly and badly: before any context is assembled, the mission itself must be compiled into a typed object. Everything in this Part is that object.

2.2 Why an uncompiled mission cannot have a minimal pack. Minimality is not an absolute property of a set of knowledge; it is a property of a set of knowledge RELATIVE TO A GOAL. "The minimal context about the deployment subsystem" is a phrase with no truth value, because minimal for what — to change a timeout, to audit its failure modes, to rewrite it, to explain it to a fresh actor? Each of those admits a different closure, and the intersection of all of them is empty while the union of all of them is the subsystem entire. It follows with the force of arithmetic that a request without a stated goal cannot be answered minimally, and an assembler handed such a request has exactly two moves available: return everything related, which is the complete-but-defeating pack CIR-11 convicts, or guess a goal silently, which produces a pack that is minimal with respect to a mission nobody agreed to. Both moves are failures and both are common. The mission contract exists to make the first move unnecessary and the second one illegal.

2.3 The object and its fields. A mission contract has nine fields and each one earns its place by being consumed by a later mechanism. The intent states what is being attempted, in the imperative, at a granularity where success is checkable — it composes `one_shot`'s intent scaffold and DAIF-02's CIR-3 Intent IR rather than founding a third intent format. The hard constraints hold the laws, in scope, that bind this mission whatever it attempts; they are admitted whole into the pack and never abridged, and their presence in the CONTRACT rather than merely in the pack is what lets a fault authorization test a request against them. The done-gate states the observable condition under which the mission is complete, and it must be observable — a condition that cannot be observed is not a gate but a hope. The evaluation contract states by what checks the output will be judged and is fixed BEFORE execution, because a mission that learns its standard at the gate has been reasoning without one. The budget states the reasoning cost the mission is authorized to spend, composing CO-00 rather than declaring a second ceiling. The authority states who may widen the budget or the scope and is the only party that may. The scope states the territory the mission may touch, and its complement — what it may not touch — is as load-bearing as its content. The known failure modes hold the recurring errors whose conditions this mission's situation matches, drawn from CIR-10 and CIR-6. The open obligations hold what this mission owes and what blocks it, drawn from CIR-5 and owned in their lifecycle by DAIF-07.

2.4 Compilation, and what it rejects. Mission compilation is a function from a stated request to a mission contract, and it is a total function only in appearance: its most valuable behavior is REFUSAL. A request with no done-gate is rejected, because minimality is undefined for it and the pack it would receive would be a fabrication. A request whose done-gate is unobservable is rejected with the specific diagnosis that the gate must name a check, not a feeling. A request whose scope is unbounded — the "everything about X" request — is rejected as a category error: it is not a mission but a subject, and a subject has no minimal sufficient context because it has no success condition to be sufficient FOR. A request whose budget cannot cover even its level-0 kernel is rejected as under-funded rather than executed to exhaustion. And a request whose minimal sufficient pack is, once computed, unmanageable is not a request to inflate but a mission to SPLIT, which is CIR-11's own finding restated as a runtime behavior: the assembler surfaces the decomposition rather than absorbing the bloat. Each refusal is a real, observed rejection with a diagnosis attached, which matters because a gate never observed to refuse is indistinguishable from an inert one, and the compiler's refusal rate is therefore itself a monitored metric.

2.5 The state model. A mission contract moves through five states. DRAFT, where the request has been received but not all fields are populated. COMPILED, where every field is populated and the refusals of 2.4 have all passed. BOUND, where the pack has been assembled against it and the two are joined by identity — a pack carries the identifier of the mission it was compiled for, and a pack whose mission identifier does not match the running mission is refused at load, which forecloses the reuse error CIR-11 names as a category error. RUNNING, where an actor is reasoning against the pack and faults may be raised. SEALED, where the done-gate has been evaluated and the mission's profile has been written. The transitions are one-way with one exception: a mission may return from RUNNING to COMPILED when DAIF-21 reports that reality moved beneath it, which forces re-assembly, and that transition is recorded, because a mission that re-assembles frequently is evidence about the volatility of its territory and not merely an inconvenience.

2.6 Invariants. Four hold. Goal-relativity: no context is ever assembled except against a COMPILED contract, and any assembly request arriving without one is rejected rather than served on a guessed goal. Gate-priority: the evaluation contract and the done-gate are fixed before the first unit is admitted, so the pack can be selected to serve the gate rather than the gate retrofitted to bless the pack. Authority-exclusivity: the budget and the scope are widened only by the named authority, never by the actor, never by the runtime, and never as a side effect of a fault — a fault may REQUEST, and only the authority may GRANT. Contract-immutability under execution: the fields of a RUNNING contract do not change; a change produces a new contract version, re-binds the pack, and records the versioning event, because a contract that drifts under the mission it governs is a contract that can retroactively bless anything the mission happened to do.

2.7 Failure modes and the adversarial cases. The signature failure is the ceremonial contract: every field populated, none of them load-bearing, a done-gate that reads "the work is complete," a scope that reads "the repository," and an evaluation contract that names no check. Such a contract passes compilation as a form and fails it as a function, and it is worse than no contract because it launders an unbounded mission through a ritual of boundedness. The defense is that the fields are consumed downstream, not merely recorded: an unobservable done-gate cannot be evaluated by DAIF-03, an unbounded scope cannot authorize a fault because there is nothing for the request to fall outside of, and a check-free evaluation contract makes the profile in Part VII unwritable — so a ceremonial contract fails at a later gate with a diagnosis pointing back to its own hollow field. The second failure is scope inflation by negotiation, where the actor, faulting repeatedly, is granted widenings one at a time until the mission's scope is unrecognizable and the original done-gate no longer describes what is being built; the defense is that scope widenings are counted, and a mission whose scope has been widened past a threshold is halted for the authority to re-compile rather than allowed to drift there in increments. The adversarial case is the request deliberately phrased as an unbounded subject in order to obtain a maximal pack — the "give me everything about X so I can decide what I need" move — and the answer is a flat refusal with the honest reason: an actor asking for everything is asking the assembler to solve the selection problem by abandoning it, and the correct response is to compile a mission whose intent is exploratory, whose done-gate is a decision, and whose pack is therefore small and full of handles.

2.8 Metrics, with countermetrics. The compiler is measured by refusal rate, by field-utilization (what fraction of contract fields were actually consumed by a downstream mechanism), by re-compilation rate, and by scope-widening count. Each is Goodhart-vulnerable alone. A high refusal rate can be manufactured by refusing everything, so it is gated against mission throughput. Field-utilization can be inflated by making every field trivially consumed, so it is gated against the fault rate, since a contract whose fields were real should reduce faults. Re-compilation rate can be driven to zero by refusing to notice that reality moved, so it is gated against the rework rate downstream. The composite standard is the one CO-12 already imposes: no claim about the compiler's quality without a measured triple, and the triple here is contracts compiled, missions passed at their own gates, and reasoning cost spent to get there.

PART II FINAL LAW. Context is assembled against a mission and never in the abstract, because minimality is definable only relative to a goal and a request without an observable done-gate therefore has no minimal sufficient context at all; the mission contract — intent, hard constraints, done-gate, evaluation contract, budget, authority, scope, known failure modes, open obligations — is compiled BEFORE a single unit is admitted, composing `one_shot`'s intent scaffold and DAIF-02's CIR-3 rather than founding a rival, absorbing the D2A-18 Mission Assembly candidate whole because the Mission Context Pack simply is what mission assembly produces, and its most valuable behavior is refusal: "everything about X" is a subject and not a mission, a mission whose minimal sufficient pack proves unmanageable is a mission to split rather than a pack to inflate, and only the named authority — never the actor, never the runtime, never a fault — may widen a budget or a scope.

PART III — THE MISSION CONTEXT PACK: STRUCTURE AND MINIMALITY

3.1 The division of labor, stated so it cannot be blurred. GK-06 owns pack BUILDING. It holds the index, the traversal, the ranking, and the compression that turn a mission plus a graph into a candidate set of units, and DAIF-08 does not have a competing traversal, a competing index, or a competing ranker. DAIF-08 owns the pack's RUNTIME CONTRACT: what a pack must structurally contain to be loadable, what it guarantees to the actor that reasons over it, what it must expose so that its own insufficiency can be discovered, how it is admitted at load and refused at load, and what it must emit when the mission ends. The seam is exactly the one DAIF-02 already declared at CIR-11's handoff: the compiled pack is an inert artifact and the runtime is the acting system. This Part specifies the artifact from the runtime's side — not how it is built, but what the runtime will accept, and therefore what the builder must produce.

3.2 The eleven slots, consumed from CIR-11. A Mission Context Pack has eleven slots and DAIF-08 inherits all eleven from CIR-11 rather than defining a twelfth structure. The mission contract, compiled in Part II, is slot one, and it is first because everything after it was selected relative to it; the runtime refuses a pack whose contract slot is empty and refuses a pack whose contract identifier does not match the mission being run. The hard constraints hold the in-scope laws, admitted whole and never abridged, because a constraint summarized is a constraint whose condition may have been the part that was cut. The relevant decisions hold the choices that bind this mission's territory, each WITH its justification, because a decision without its reason can be neither safely honored nor safely revisited and an actor holding an unjustified decision will either violate it or be paralyzed by it. Current reality holds the observed state — what is on disk, what is running, what passed, what failed — as observations and verdicts rather than as the plan's assumptions. The minimal architecture slice holds only the structure the mission touches, closed over hard dependencies from the mission's seed and stopped there. The applicable procedures hold those whose activation conditions the current reality actually satisfies, which is a computation and not a judgment. The known failure modes hold the recurring errors and causal chains whose conditions match this situation. The evaluation contract holds the checks by which the output will be judged and is present at the start. The open obligations hold what this mission owes and what blocks it. The provenance map holds the chain from every admitted unit back to its source, so that any claim inside the pack can be audited without leaving it. And the expansion handles, which Part IV specifies in full, hold the typed pointers to what was deliberately left out.

3.3 The Minimum Sufficient Context principle, stated formally. The objective of pack assembly is NOT to minimize tokens. This must be stated as a prohibition and not a preference, because the token-minimal pack is the empty pack, and any process whose objective function is token count will, if it is honest and sufficiently optimized, converge on emptiness and report a triumph. The correct formulation is a constrained minimization: minimize the cognitive cost of the pack — its size, its density, the reasoning load it imposes — SUBJECT TO three binding constraints. Fidelity must remain above a threshold, meaning an actor reasoning from the pack alone reaches the conclusions an actor with the whole estate would reach, which is the behavioral-equivalence standard DAIF-03 owns and DAIF-08 submits to rather than redefines. Risk must remain below a threshold, meaning the probability that an omission causes a wrong action, weighted by the severity of that action, stays inside what the mission's authority accepted. And success probability must remain above a threshold, meaning the pack must actually be expected to let the mission pass its own gate. Cost is the objective; fidelity, risk, and success are the constraints. Inverting that — treating fidelity as the objective and cost as the constraint — produces the complete-but-defeating pack; collapsing the constraints entirely produces the empty one. Both errors are failures of the same formalism and the formalism is therefore not decoration.

3.4 Admission at load, and refusal at load. The runtime does not trust a pack merely because a builder produced it. A pack is admitted only if it passes four structural checks, and each check is a real refusal with a diagnosis. Contract-binding: the mission contract slot is present and its identifier matches the running mission; a pack compiled for another mission is refused outright, because reusing a pack across missions mistakes a selection for a summary. Constraint-integrity: every hard constraint in the slot is byte-complete against its source and not a summary; an abridged constraint is refused rather than loaded with a warning, because a warning about an abridged law is a law that will be violated with a paper trail. Boundary-honesty: the handles slot is present and non-degenerate; a pack with zero handles is refused unless the mission's closure provably had no adjacent units, because a pack with no handles is a pack claiming completeness, and completeness is the one claim a compiled prediction is not entitled to make. Gate-presence: the evaluation contract is populated, because a pack that cannot say how its mission will be judged has failed to bring the thing that makes minimality checkable. These four refusals are the runtime's entire admission authority, and they are deliberately structural — the runtime does not second-guess GK-06's ranking, only its form.

3.5 Attribution: minimality made auditable. Every unit in a pack carries an attribution: the rule of the selection law under which it was admitted. A hard constraint is attributed to scope. A decision is attributed to territorial binding. A procedure is attributed to its fired activation condition. A failure mode is attributed to its matched condition. An architecture node is attributed to its hard-dependency path from the seed. A unit that cannot name its admission rule is not merely suspicious; it is REMOVED, and its removal is recorded. This single mechanism is what converts minimality from an aspiration into an audit, because the failure it forecloses is the unit admitted "because it seemed relevant" — the one instinct CIR-11 exists to discipline — and an instinct is indistinguishable from a rule until it is required to name itself. Attribution also makes the profile in Part VII computable: a unit that was admitted under a rule, and then never touched, is evidence against that rule for that class of mission, and without attribution the evidence has nothing to accuse.

3.6 The failure modes of the pack itself. Three recur. The complete-but-defeating pack: every related unit admitted, nothing omitted, sufficiency guaranteed, and so large that the actor cannot find the three units that mattered — the failure that degrades reasoning actively, because relevance is a signal and a pack that admits everything transmits none, and it silently pushes the selection problem onto the actor at exactly the moment the actor is least able to solve it, mid-mission and under load. The abridged-law pack: a pack that shrank a hard constraint to fit a size target, which has traded the one thing it may not trade, and whose actor will comply perfectly with a law whose binding condition was in the sentence that was cut. And the stale pack: a pack whose current-reality slot describes a world that has moved, which is the most dangerous of the three because it is confidently wrong rather than merely incomplete — the actor is not missing information, it HOLDS information that is false, and it will reason correctly to a wrong conclusion. Staleness is not detected here; DAIF-21 owns invalidation and signals it, and DAIF-08's obligation is only to accept the signal, mark the pack invalid, and force re-assembly rather than continue on a pack it has been told is dead.

3.7 The adversarial cases. First, the size-target attack: a compiler tuned against a token ceiling learns that the cheapest way under the ceiling is to summarize a hard constraint, since constraints are verbose and their summaries read plausibly. The defense is constraint-integrity at load, which makes the cheap path unloadable rather than merely discouraged. Second, the handle-stuffing attack: a compiler learns that handles are cheap, since a handle is a pointer rather than a body, and it therefore satisfies boundary-honesty by emitting a thousand handles that name the whole estate — which reconstitutes the complete pack through a side door and defeats the actor's ability to see what was meaningfully excluded. The defense is that handles are typed, budgeted, and themselves attributed, as Part IV specifies, and a pack whose handle count exceeds its admitted-unit count by a wide margin is refused as an index masquerading as a boundary. Third, the successful-mission-with-bad-pack case: a mission passes its gate despite a defective pack, because the actor happened to know the missing unit from elsewhere or happened not to need it on this path. This is the case that most corrupts a naive profile, since outcome alone would score the pack as good, and the answer is that Part VII scores packs on usage and faults as well as outcome, never on outcome alone.

3.8 Metrics and their countermetrics. The pack is measured by admitted-unit count, by attribution completeness (the fraction of units naming a valid admission rule, which must be one), by constraint-integrity rate (which must be one), by handle density, and by pack size against the mission's budget. Every one of these is Goodhart-vulnerable in isolation and none of them is permitted to stand alone: admitted-unit count is trivially minimized by admitting nothing and is therefore gated against the fault rate of Part V; handle density is trivially maximized by stuffing and is gated against handle-pull rate; pack size against budget is trivially satisfied by abridgment and is gated against constraint-integrity. The pack has no self-standing quality score at all, and this is deliberate — the pack's quality is only knowable from the mission that ran on it, which is why the runtime and not the builder is where the measurement lives.

PART III FINAL LAW. GK-06 owns the building of the pack and DAIF-08 owns its runtime contract: eleven slots consumed from CIR-11 — mission contract, hard constraints admitted whole, decisions carrying their justifications, current reality, the minimal architecture slice, the procedures whose conditions actually fired, the matching failure modes, the evaluation contract present at the start, the open obligations, the provenance map, and the expansion handles — admitted at load only on contract-binding, constraint-integrity, boundary-honesty, and gate-presence, with every unit attributed to the selection rule that admitted it so that minimality is auditable rather than aspirational; and the governing principle is a constrained minimization, cost minimized SUBJECT TO fidelity above threshold, risk below threshold, and success probability above threshold, never token count minimized as the objective, because the token-minimal pack is the empty one and any process that optimizes for smallness alone will converge on it and call the emptiness a victory.

PART IV — EXPANSION HANDLES

4.1 What a handle is for. A minimal pack is a bet, and DAIF-08's first structural commitment is that the pack must ship with the means of its own correction. An expansion handle is that means: a typed pointer, carried inside the pack, that names something which exists beyond the pack's boundary. It exists to make the pack HONEST about its own incompleteness. Without handles, a pack is a closed surface — the actor sees what it was given, sees nothing else, and has no way to distinguish "this unit does not exist" from "this unit exists and was excluded." Those two states are radically different in what they license: the first justifies proceeding on the actor's own reasoning, and the second justifies asking. An actor that cannot tell them apart will proceed in both cases, which is exactly how a silent omission becomes confident wrong work.

4.2 The cardinal contrast, stated plainly. A pack WITHOUT handles silently pretends to be complete. It does not claim completeness in words — no compiler is that reckless — but it claims it structurally, by presenting a bounded body of knowledge with no visible edge, and an actor reads a body with no visible edge as a body with no outside. The consequence is mechanical and it is the whole reason this Part exists: an actor cannot request what it cannot see was omitted. It can only request what it has some representation of, and a unit that was excluded without a trace has no representation anywhere in the actor's world. Silent omission is therefore strictly worse than a visible gap, and the ranking is not a matter of taste. A visible gap is a decision point: the actor sees the pointer, judges whether it needs the thing, and either pulls it or proceeds knowingly without it. A silent omission is not a decision point at all; it is an absence that looks like a non-existence, and the actor's reasoning fills it with an invention. Handles turn omission from a lie into an offer.

4.3 The anatomy of a handle. A handle has six fields and each is load-bearing. The identity names precisely what lies beyond — not a subject, not a directory, but a unit with an identifier the runtime can resolve, because a handle whose target cannot be resolved is a rumor. The type states what kind of thing it is, drawn from the taxonomy of 4.4, because the type determines the ascent level at which it may be pulled and the cost model that prices it. The exclusion reason states WHY it was left out, which is the field that most often goes hollow and the field that carries the most information: excluded as below the relevance threshold, excluded as out of scope, excluded as too large for the budget, excluded as belonging to a level above the pack's current level, excluded as superseded, excluded as adjacent rather than dependent. The cost estimate states what pulling it would spend, in the same unit the budget is denominated in, so that the authorization in Part V is a comparison rather than a guess. The request path states how to ask — the exact resolution the runtime will perform — so that a pull is a mechanical operation and not an improvisation. And the provenance pointer states where the unit's evidence came from, so that a pulled unit arrives audited rather than merely arriving.

4.4 The taxonomy. Five kinds of handle exist, ordered by the depth of what they open. A DECISION-JUSTIFICATION handle points at the full reasoning behind a decision the pack included in summary: the pack carried the choice and its one-line reason, and the handle opens the alternatives considered, the constraints that eliminated them, and the conditions under which the decision would be revisited. It is pulled when an actor must either violate or revisit a decision, and it is the handle whose absence most reliably produces a mission that either obeys a rule it does not understand or breaks one it thought was arbitrary. An EVIDENCE-BUNDLE handle points at the observations and verdicts supporting a claim the pack asserted; it is pulled when the actor's own observation conflicts with what the pack says is true, which is the conflict trigger of Part VI. An ARCHITECTURE-SLICE handle points at a wider region of structure — the next ring of dependencies beyond the hard-dependency closure — and it is pulled when the mission discovers its touch surface is larger than the seed implied, which is frequently the honest early signal that the mission is under-decomposed. A SOURCE-EXCERPT handle points at the literal text of a unit that the pack carried in compiled or summarized form; it is pulled when precision matters more than gist, which is the case for any constraint whose exact wording governs. And an ORIGINAL-ARTIFACT handle points at the raw artifact itself — the file, the transcript, the record — beneath every compilation performed over it; it is the deepest and most expensive pull and it is the correct one when every compiled layer is under suspicion, which is the case when a fidelity failure is the thing being investigated.

4.5 Handle generation is a selection product, not an afterthought. Handles are not written by a separate pass that inspects the finished pack and speculates about what a reader might miss. They are a BY-PRODUCT of the selection law itself, and they must be, because the only party that knows what was excluded is the party that excluded it. Every time the selection law refuses a unit — because it was related but not depended upon, because its activation condition did not fire, because it was one hop beyond the closure, because its scope was adjacent — that refusal is the raw material of a handle, and the refusal already carries the exclusion reason as a matter of fact rather than reconstruction. This is what makes exclusion reasons trustworthy: they are recorded at the moment of exclusion by the mechanism that performed it, not inferred later by a mechanism that is guessing at its own past. It also bounds the handle set naturally, because the selection law does not consider the whole estate — it considers the units the traversal reached — and so the handles name the boundary the traversal actually drew rather than the boundary of everything that exists.

4.6 Budgeting the handle set, and the stuffing attack. Handles are cheap individually and therefore dangerous in aggregate. A handle body is a pointer, so a compiler seeking to satisfy boundary-honesty at the lowest possible price will emit handles for every unit it ever considered, and the pack's handle slot becomes an index of the estate. This defeats the purpose completely: an actor scanning ten thousand handles is performing exactly the selection task the pack was compiled to spare it, and the complete-but-defeating pack has been reconstituted from pointers instead of bodies. The defenses are three and they are all structural. Handles are ranked by the same relevance the selection law computed, and only the top band survives into the pack, with a hard count ceiling proportional to the admitted-unit count. Handles are typed, and a type whose ascent level lies far above the mission's current level is held rather than shipped, since a handle the actor cannot yet pull is noise. And the handle set is itself measured: a pack whose handles are never pulled has a handle set that is decorative, and a pack whose handles are pulled at a high rate has a selection law drawing its boundary in the wrong place — both are findings, and both feed Part VII.

4.7 Failure modes and the adversarial cases. The hollow-reason handle is the commonest failure: a handle whose exclusion reason reads "not relevant," which conveys nothing the handle's own existence did not already convey, and which makes the actor's pull decision a coin flip. The defense is that exclusion reasons are drawn from a closed vocabulary produced by the selection law, so a reason outside the vocabulary is a schema violation and not a stylistic complaint. The dangling handle points at a unit that no longer resolves, because reality moved beneath the pack, and a dangling handle is worse than a missing one because the actor spends an authorization and a round trip to learn nothing; DAIF-21's invalidation signal is what marks a pack's handles dead, and the runtime refuses a pull against an invalidated pack rather than resolving it into a stale body. The adversarial case is the handle used as a laundering device: a compiler that is uncertain whether a unit is needed can always emit a handle instead of admitting the unit, thereby appearing minimal while pushing the cost onto the actor, and if handles are free this becomes the compiler's dominant strategy. The countermeasure is that a handle which is pulled is scored as a MISS against the selection law, not as a success — pulling a handle is a page fault, and a compiler that converts its uncertainty into faults is measured as faulting, which is precisely the accounting Part V and Part VII enforce.

4.8 What handles do not do. A handle is not an authorization. Seeing a handle does not entitle an actor to pull it; the pull is a request, and Part V's authorization gate — budget, scope, level — decides it. A handle is not a promise of value; it names what exists, not what will help. A handle is not a substitute for admission: a unit the mission genuinely depends on must be ADMITTED, and shipping it as a handle instead is the laundering of 4.7, because dependence is not a thing the actor should have to discover at cost. And a handle is not an obligation: an actor that sees a handle and proceeds without pulling it has made a legitimate decision, provided the decision is recorded, and it is the recording — not the pull — that closes the loop. A handle seen and declined is data about the selection law exactly as a handle pulled is data about it, and an estate that records only the pulls has thrown away half its signal.

PART IV FINAL LAW. An expansion handle is a typed pointer — identity, type, exclusion reason, cost estimate, request path, provenance — generated as a by-product of the selection law's own refusals rather than reconstructed afterward, and its five kinds open a decision's full justification, an evidence bundle, a wider architecture slice, a source excerpt, or the original artifact; it exists because a pack without handles silently pretends to be complete and an actor cannot request what it cannot see was omitted, which makes silent omission strictly worse than a visible gap — a gap is a decision point, an omission is an invention waiting to happen — and handles therefore turn omission from a lie into an offer; but a handle is never an authorization, never a substitute for admitting a unit the mission genuinely depends on, and never free, because a pulled handle is scored as a MISS against the selection law that emitted it, which is the only accounting under which a compiler cannot launder its uncertainty into the actor's cost.

PART V — PAGE FAULTS: THE COGNITIVE FAULT AS A FIRST-CLASS EVENT

5.1 The event, named. When an actor working from a pack discovers that the pack lacks something it needs, that is a page fault. The name is borrowed deliberately from the substrate that solved this problem first, and the borrowing carries the whole discipline with it: a page fault is not an error, not a defect, and not something to be hidden — it is the NORMAL, EXPECTED, MEASURED mechanism by which a system that deliberately holds less than everything discovers what it should have held. A memory system with zero page faults has simply resident-loaded the entire address space and abandoned the premise. By exact analogy, a context runtime with zero faults has either shipped the complete-but-defeating pack or, far more likely, has faults that it is not detecting. Zero observed faults is therefore not a triumph but a suspicion, and the first thing to check in a fault-free estate is whether the fault detector is inert — because a gate never observed to refuse is indistinguishable from an inert one, and the same is true of an event that is never observed to fire.

5.2 The lifecycle: five stages. A fault passes through detection, classification, authorization, resolution, and recording, and dropping any stage collapses the mechanism into something less. Detection is the actor recognizing that it does not know something it needs. Classification types the fault so that the request is resolvable and priceable. Authorization tests the request against budget, scope, and level, and is the authority's gate, not the actor's. Resolution loads exactly the missing unit. Recording writes the fault, its type, its resolution, and its outcome to the profile. The stage that is most often dropped is the last, and dropping it is the difference between a runtime that improves and one that merely copes.

5.3 Detection, and the hard case. Detection is easy when the actor's need is explicit — it is about to honor a decision and does not have the decision, it is about to run a procedure and does not have the procedure, it is about to evaluate a claim and does not have the evidence. In each of these the gap has a SHAPE, and a shaped gap is detectable by the actor's own reasoning, which is why the pack's handle set matters so much: the shape usually corresponds to a handle, and the handle is the actor's confirmation that the gap is real rather than a failure of its own memory. The hard case, and the whole difficulty of the domain, is the gap with no shape: the actor does not know that it does not know. It has no handle pointing at the missing thing because the selection law never considered it, no reasoning step that visibly stalls because the actor's reasoning proceeds smoothly on a false premise, and no felt uncertainty because confident wrong reasoning feels exactly like confident right reasoning from the inside. This is the silent fault of 5.6 and there is no detector that finds it from inside the mission. It is found only from the outside — by the evaluation contract failing, by the reality-conflict trigger of Part VI firing when an observation contradicts a held claim, or by the rework that arrives later — and DAIF-08's honest position is that it does not prevent the silent fault, it makes the silent fault RECOVERABLE AS A SIGNAL after the fact, which is a smaller claim and a true one.

5.4 Classification and authorization. A fault carries a type, and the type is the handle taxonomy of Part IV plus one: MISSING-DECISION, MISSING-EVIDENCE, MISSING-STRUCTURE, MISSING-PRECISION, MISSING-ARTIFACT, and UNKNOWN-GAP for the case where the actor knows it lacks something but cannot name a handle for it — and UNKNOWN-GAP is the single most valuable fault type in the whole system, because it names a region the selection law did not even consider, whereas the other five name regions it considered and refused. Authorization then tests the request on three axes. Budget: does the fault's cost estimate fit inside the mission's remaining allowance, composing CO-00's ceiling rather than declaring a second one? Scope: does the requested unit lie inside the mission's declared territory, and if it does not, the fault is not resolved but ESCALATED, because a fault reaching outside scope is evidence that the mission was mis-bounded and only the authority may re-bound it. Level: is the requested unit at or one step above the mission's current disclosure level, since an ascent of several levels at once is the pre-emptive expansion Part VI forbids. A fault that fails authorization is not silently denied — the denial is recorded with its reason, because a denied fault is as informative as a granted one and an estate that logs only its grants has recorded only its permissiveness.

5.5 Resolution: the neighborhood rule. Resolution loads exactly the missing unit and NOT its neighborhood. This is stated as a rule because the temptation is overwhelming and the reasoning behind the temptation is superficially sound: the actor faulted here once, it will probably fault here again, and the cost of pulling the surrounding region now is lower than the cost of three more round trips. That reasoning produces speculative expansion, and speculative expansion is how a minimal pack becomes a complete one in four faults — each pull justified locally, the aggregate indistinguishable from having sent everything. The rule is therefore strict: resolve the named unit, admit it into the pack with an attribution of FAULT rather than of a selection rule, and let the next fault be a next fault. If the same neighborhood faults three times, that is not a case for pre-fetching; it is a finding about the selection law, and the finding belongs in the profile where it can correct the COMPILER rather than in a heuristic that papers over the compiler's error at runtime forever.

5.6 The failure modes. The silent fault is the worst thing in this dataset. The actor discovers a gap and, rather than requesting, fills it — from training, from an adjacent memory, from a plausible inference — and proceeds. The mission may even succeed. Two harms follow and they compound. The first is that the work is built on an unverified premise and is therefore confidently wrong whenever the premise is wrong, and confidently wrong work is more expensive than halted work by the entire cost of everything built on top of it before the error surfaces. The second, and the one this dataset cares about most, is that the silent fault GENERATES NO SIGNAL: the compiler is never told what it wrongly excluded, so the same pack will be compiled tomorrow, the same gap will open, and the same guess will be made. A loud fault costs a round trip and buys a permanent correction. A silent fault costs nothing now and buys a permanent blindness. The defense is not a detector — there is none — but an incentive structure and a downstream instrument: an actor that faults is never penalized for faulting, faults are never counted against the actor's quality, and the fault rate is read as a property of the PACK and not of the actor, because any regime that treats faulting as a failure of the actor has just built a machine for manufacturing silent faults. Downstream, the silent-fault rate is estimated from evaluation failures and rework whose root cause resolves to a unit that was in the estate, was absent from the pack, and was never requested — an inference, not an observation, and reported as such.

5.7 Thrashing and the fault storm. Fault thrashing is the pattern where a unit is loaded, evicted under budget pressure, faulted for again, loaded, and evicted again, and it is a real pathology inherited from the substrate the name comes from. Its cause is always the same: the working set genuinely needed by the mission exceeds the residency the budget permits, and the runtime is spending its entire allowance on transport rather than on reasoning. The defense is that eviction is never applied to a unit that has faulted within the current mission — a faulted unit is pinned, because the fault is proof of need, and evicting proof of need to make room for a prediction is a strict downgrade. If the pinned set alone exceeds the budget, the runtime halts and escalates, because a mission whose proven working set does not fit its budget is under-funded and no amount of clever eviction changes that arithmetic. The fault storm is the different pathology where faults arrive at a rate that indicates the pack was simply WRONG — not slightly imperfect, but compiled against a misunderstanding of the mission. A storm is defined against a threshold on faults per unit of reasoning, and its correct handling is not to serve the faults one by one but to STOP, mark the pack as mis-compiled, and re-assemble from the contract, because serving a storm one fault at a time is a slow and expensive path to the pack that should have been compiled at the start, and it burns the mission's budget to get there.

5.8 Recording: why it is the entire point. Every fault is written: its type, the unit requested, whether a handle pointed at it and which one, the exclusion reason that handle carried, the authorization verdict, the cost, the mission's class, and whether the mission ultimately passed its gate. The reason this matters more than the resolution does is that each fault is a LABELED TRAINING EXAMPLE for the compiler — it says, with the authority of an actor that was actually doing the work, that this unit was needed and was not present, and it says WHY the compiler thought it was not needed, because the handle's exclusion reason is right there in the record. There is no richer signal available anywhere in the domain, and the estate has been discarding it. A fault resolved and not recorded fixes one mission. A fault recorded and fed back fixes the class. This is the leg of the loop Part I named as missing, and Part VII is where the record is finally spent.

PART V FINAL LAW. A page fault — the moment an actor discovers the pack lacks what it needs — is a first-class, typed, authorized, and above all RECORDED event, never an error and never something to hide, because a runtime with zero observed faults has either shipped the complete-but-defeating pack or has a detector that is inert; its lifecycle is detection, classification, authorization against budget and scope and level, resolution of exactly the named unit and never its neighborhood, and recording — and the recording is the entire point, since each fault is a labeled example telling the compiler precisely what it wrongly excluded and why it thought it could; the worst failure is the SILENT fault, where the actor guesses instead of requesting, because it produces confident wrong work AND generates no signal at all, which is why faulting must never be penalized as a failure of the actor — any regime that punishes the fault has built a machine for manufacturing silence — and a fault storm is not a queue to be served but a pack to be re-compiled.

PART VI — PROGRESSIVE DISCLOSURE: THE LEVEL LADDER

6.1 Levels rather than a delivery. The pack is not handed over as a flat body of knowledge whose whole content is resident from the first reasoning step. It is disclosed in LEVELS, and the level is a property of the running mission rather than of the pack: the same pack, bound to the same mission, is resident at level 0 in its first minute and may be resident at level 3 in its twentieth, and the difference between those two states is a sequence of authorized, triggered, recorded ascents. This is what converts the pack from an artifact into a runtime. The levels are six, they are ordered by depth rather than by subject, and the ordering is what makes ascent meaningful: each level is a strictly deeper commitment of reasoning cost in exchange for a strictly more literal and less compiled view of the same territory.

6.2 The six levels. Level 0 is the mission kernel: the mission contract, the hard constraints in full, the done-gate, and the evaluation contract — the irreducible minimum below which a mission cannot legally begin, because an actor without its constraints and its gate is not executing a mission but improvising near one. Level 1 is the immediate execution context: current reality, the applicable procedures, the minimal architecture slice, the open obligations, and the decision SUMMARIES — enough to act, which is where the great majority of missions run to completion and where the pack's economics are decided. Level 2 is relevant decision detail: the full justifications behind the decisions level 1 carried in summary, the alternatives that were considered, and the conditions under which each decision would be revisited — reached when an actor must honor a decision it does not understand or revisit one it must break. Level 3 is supporting evidence: the observations and verdicts that establish the claims levels 1 and 2 asserted — reached when a claim is in doubt or in conflict with what the actor observes. Level 4 is full source excerpts: the literal text of units that lower levels carried compiled or summarized — reached when precision governs and gist will not do. Level 5 is the original artifacts: the raw files, transcripts, and records beneath every compilation — the deepest, most expensive, and rarest ascent, correct only when the compilations themselves are what is under suspicion.

6.3 The four triggers, and only four. Ascent is not a schedule and not a default; it happens on a trigger and there are four. DETECTED UNCERTAINTY: the actor's own reasoning reaches a point where it cannot proceed without a unit it does not hold, which is precisely the page fault of Part V — the fault IS the trigger, and the fault's type determines which level the ascent targets. CONFLICT: the actor observes something that contradicts a claim the pack asserts, which is the reality-conflict case, and its correct ascent is to level 3 to inspect the evidence behind the contested claim rather than to level 1 to gather more assertions of the same rank. EVALUATION FAILURE: a check in the evaluation contract fails, which is a signal that the reasoning that produced the output was operating on an insufficient or false basis, and the ascent is targeted at the units the failed check names. And a JUSTIFIED AGENT REQUEST: the actor asks, and states a reason — the reason is mandatory, and a request without one is refused, because an unreasoned request is the pre-emptive expansion of 6.4 wearing the costume of a trigger. There is no fifth trigger. In particular there is no time-based trigger, no complexity-based trigger, and no "the mission seems hard" trigger, because each of those decouples ascent from evidence of need, and the whole discipline of this Part is that ascent must track evidence.

6.4 Why ascent must be earned. A runtime that ascends by default has rebuilt "send everything" with extra steps, and the extra steps make it worse rather than better, because the flat complete pack at least paid its cost honestly and in one lump, while a defaulting ladder pays the same cost incrementally while telling itself a story about restraint. The temptation to default is strong and it comes from a real fear: the actor might need level 2, and the round trip to fetch it is latency, and the fault might not be detected at all, and pre-loading is cheap insurance. Every clause of that reasoning is locally true and the conclusion is still wrong, for the reason Part III established formally — the objective is not to minimize the risk of a fault, it is to minimize cognitive cost subject to fidelity, risk, and success constraints, and a pre-emptive ascent spends cognitive cost against a fault that may never occur while simultaneously degrading the signal-to-noise of everything already resident. The correct treatment of the fear is measurement, not accommodation: if a mission class ascends to level 2 in eighty percent of its runs, the answer is not to pre-load level 2 at runtime for everyone but to feed that observation to the compiler through Part VII, so that the NEXT pack for that class admits the decision justifications at level 1 by an attributable rule. The difference is not cosmetic. Pre-emption is a runtime heuristic that hides the compiler's error forever; profile feedback is a correction that fixes it.

6.5 Descent, and why it is an invariant rather than an optimization. A level that has been ascended to may be descended FROM. A unit loaded at level 3 to resolve a conflict, once the conflict is resolved, is no longer needed, and holding it resident for the remainder of the mission costs exactly what admitting it wrongly at compile time would have cost — the difference being that nobody notices, because the ascent was justified when it happened. This is how a well-governed ladder still ends as a complete-but-defeating pack: by never letting go. Descent is therefore an invariant of the runtime and not a nice-to-have: a level whose units have been loaded, consumed, and not touched again within a defined window is evictable, and the eviction is executed by CO-04's Context Virtual Memory, whose tiering across hot, warm, cold, and external is exactly the mechanism this requires — DAIF-08 does not build a second tiering and does not build a second eviction; it supplies CO-04 with a mission-scoped notion of what is currently needed, which CO-04 has never had. The one exception is the pin of 5.7: a unit that was faulted for is not evicted within the mission that faulted for it, because eviction of proven need in favor of prediction is a strict downgrade and the direct cause of thrashing. Level 0 is likewise permanently pinned, because a mission that has evicted its own constraints is a mission that will violate them.

6.6 Authorization, budget, and the ledger. Every ascent is authorized before it is served, on the same three axes as a fault: budget, scope, and level-distance. Budget composes CO-00 — DAIF-08 does not hold a ceiling of its own, it consumes the ceiling CO-00 already governs, and an ascent that would breach it is refused with the refusal recorded rather than served with an apology. Scope refuses an ascent whose target lies outside the mission's declared territory and escalates it to the authority, because an ascent reaching outside scope is not an ascent but a scope change wearing an ascent's clothes. Level-distance refuses an ascent of more than one level at a time: an actor at level 1 may ascend to level 2, and an actor demanding level 5 from level 1 is not disclosing progressively but demanding the raw estate, and the correct response is to serve level 2 and let the ladder be climbed. Every ascent, every refusal, every descent, and every pin is written to the mission's ledger, and the ledger is the raw material Part VII consumes.

6.7 Failure modes and adversarial cases. Ladder collapse is the failure where every mission ascends to level 4 or 5 within its first few steps, which means either the level-0 and level-1 packs are systematically insufficient — a compiler finding — or the actor has learned that ascent is cheap and is using it as a substitute for reasoning, which is an incentive finding. The two are distinguishable by whether the ascended units are actually USED, and the distinction matters because the remedies are opposite. Ratchet failure is the absence of descent: levels only ever go up, so a long mission ends holding everything it ever touched, and its late reasoning is performed over a pack that has all the defects of the complete pack with none of the honesty of having been compiled as one. Trigger laundering is the adversarial case: an actor that wants a deeper level and lacks a trigger manufactures one, most easily by declaring "uncertainty" — and the defense is that a fault must NAME the unit it lacks and, where possible, the handle that points at it, so that an uncertainty which cannot name its object is recorded as UNKNOWN-GAP and treated as the compiler-facing finding it is rather than as a licence to ascend. And the pre-emptive-ascent case, where a well-meaning runtime ascends because the mission "looks complex," is foreclosed by 6.3's closed trigger set: complexity is not a trigger, and a runtime that treats it as one has silently reintroduced the default it was built to refuse.

6.8 Metrics and countermetrics. The ladder is measured by level distribution across missions, by ascent rate, by descent rate, by the utilization of ascended units, and by the ascent-to-outcome correlation. Level distribution alone is Goodhart-trivial — a runtime that refuses all ascents reports a beautiful distribution and a catastrophic rework rate — so it is gated against mission success. Ascent rate alone is meaningless in either direction: a high rate may mean an insufficient level-1 pack or an incontinent actor, and a low rate may mean an excellent pack or an actor silently guessing past its gaps, which is why ascent rate is always read alongside the silent-fault estimate of Part V. Utilization of ascended units is the sharpest single instrument the ladder has: an ascent whose units were never touched was an ascent that should not have happened, and a mission whose ascended units are uniformly used is a mission whose level-1 pack was drawn one level too shallow. That is a compiler finding, and it belongs in the profile.

PART VI FINAL LAW. Context is disclosed in six levels — the mission kernel, the immediate execution context, decision detail, supporting evidence, source excerpts, and the original artifacts — and the runtime ascends ONLY on one of four triggers, detected uncertainty, a conflict between observation and asserted claim, an evaluation failure, or a justified request that names its reason, with every ascent authorized against CO-00's budget, the mission's scope, and a strict one-level distance, because a runtime that ascends by default has rebuilt "send everything" with extra steps and made it worse by paying the same cost while telling itself a story about restraint; descent is an invariant and not an optimization — a level ascended, consumed, and untouched is evicted through CO-04's existing tiering, with only level 0 and the faulted set pinned — since a ladder that only ever climbs ends as the complete-but-defeating pack it was built to refuse, and the honest remedy for a mission class that ascends every time is never a pre-emptive runtime heuristic but a corrected compiler.

PART VII — PROFILE-GUIDED OPTIMIZATION: THE PACK THAT LEARNS

7.1 The loop closes here. Everything before this Part is instrumentation, and instrumentation that is never read is telemetry theater. Part V established that each page fault is a labeled example. Part VI established that each ascent, refusal, descent, and utilization figure is written to a ledger. Part III established that each admitted unit carries the attribution of the rule that admitted it. Part IV established that each handle carries the exclusion reason of the refusal that produced it. These four records, joined on the mission, constitute a PROFILE, and the profile is what recompiles the next pack. The claim of this Part is precise and it is the claim on which the whole dataset stands or falls: a compiler that is fed the observed behavior of its own past packs will produce better packs than a compiler fed a designer's intuition about what missions need, and the estate has never fed the compiler anything at all.

7.2 The six observations. A mission profile records six facts and each answers a distinct question. WHAT WAS LOADED: the admitted set, with attributions — the compiler's prediction, on the record. WHAT WAS USED: which admitted units the actor's reasoning actually touched, which is the single hardest measurement in the system and is discussed in 7.3. WHAT WAS NEVER TOUCHED: the complement of the above, and the direct measure of waste — units the compiler paid for, the actor carried, and nobody read. WHAT WAS REQUESTED VIA A FAULT: the units the compiler wrongly excluded, with the exclusion reason it excluded them under, which is the richest correction signal in the domain because it names both the error and the reasoning behind the error. WHAT WAS MISSING AND NEVER REQUESTED: the silent faults, which cannot be observed directly and are INFERRED from failures — an evaluation check that failed, a rework whose root cause resolves to a unit that existed in the estate, was absent from the pack, and was never asked for — and which are reported as an inference with its confidence rather than as an observation, because reporting an inference as an observation is exactly the fabrication the estate's epistemic ladder exists to prevent. And WHAT THE OUTCOME WAS: whether the mission passed its own done-gate, and whether it generated rework afterward, because a mission that passed and then had to be redone did not pass.

7.3 Measuring usage without lying about it. "What was used" is where a naive profile becomes a fiction. Usage is not "the unit was in the resident window," because everything admitted was in the window. It is not "the actor mentioned it," because an actor may mention a unit while ignoring its content. The estate's honest instruments are three and they are ranked by strength. The strongest is CITATION: the actor's output or its reasoning trace explicitly grounds a claim, a decision, or a step in a named unit, which is unambiguous and is the standard the provenance map exists to make possible. The second is CAUSAL NECESSITY, established by counterfactual replay: the same mission run against a pack with the unit withheld produces a different outcome, which is expensive and is therefore run on a sample rather than on every mission, but which is the only instrument that establishes need rather than reference. The weakest is retrieval or attention proximity, which is a correlate and must be labeled as one. A profile that reports a usage figure without saying which instrument produced it is a profile that has laundered a correlate into a fact, and CO-12's Telemetry-Before-Claims forbids exactly that: no claim without a measured triple, and the instrument is part of the measurement.

7.4 The four metrics. CONTEXT PRECISION: of what was loaded, what fraction was used. It measures waste, and it answers whether the compiler is admitting units nobody reads. CONTEXT RECALL: of what was needed, what fraction was present at compile time. It measures insufficiency, and it is estimable only because faults are recorded — the faulted units are the observable part of what was needed and absent, and the silent faults are the inferred part, which is why recall always carries an uncertainty band and precision does not. PAGE-FAULT RATE: faults per mission, normalized by mission size, which is the direct instrument on the selection law's boundary. SILENT-FAULT RATE: the inferred rate of gaps the actor guessed past, derived from evaluation failures and rework, and reported with its inference confidence. And REWORK RATE: missions that passed their gate and then had to be redone, which is the ground truth all four of the others are ultimately accountable to, because it is the only one that measures the thing the estate actually cares about.

7.5 The anti-Goodhart trap, stated with teeth. Precision is trivially maximized by loading almost nothing. A compiler optimizing for precision alone converges, in the limit, on a pack containing the mission contract and one unit, and it will report a precision approaching one while every mission faults twenty times and half of them silently guess. This is not a hypothetical degradation; it is the mathematically forced behavior of a system rewarded on precision alone, and any estate that ships a precision dashboard without its partner has built the incentive to produce it. Recall has the symmetric pathology: a compiler optimizing for recall alone loads the estate, reports a recall of one, and rebuilds the complete-but-defeating pack. The gate is therefore CONJUNCTIVE and non-negotiable: precision and recall are gated TOGETHER, neither may improve at the other's expense without the joint measure improving, and BOTH are gated against the mission's success rate — because precision and recall are both proxies, and a proxy that rises while the outcome falls has been gamed by definition. The true objective, from which the proxies derive whatever authority they have, is VERIFIED OUTCOMES PER UNIT OF REASONING COST. Not tokens saved. Not faults avoided. Not units admitted. Verified outcomes — missions that passed their own gate, whose gate was real, and which did not come back — divided by what it cost to reason them. Every metric in this dataset is a subordinate of that ratio and is discarded the moment it stops tracking it.

7.6 Recompilation: what the profile actually changes. The profile does not change the actor and does not change the runtime; it changes the COMPILER, and it changes it in four specific ways. A unit admitted under a rule and never touched across a statistically meaningful sample of a mission class demotes that rule for that class — the rule is not deleted, because a rule that is right one time in fifty may be right about the case that matters, but its admission is downgraded to a handle, which preserves the actor's ability to reach it while removing its cost from every pack. A unit faulted for repeatedly across a class, whose handle carried a specific exclusion reason, indicts that exclusion reason for that class, and the correction is targeted: it is not "admit more" but "this specific refusal, in this specific situation, was wrong." A level whose ascended units are uniformly used across a class means the class's level-1 boundary is drawn one level too shallow, and the correction is to promote those units into level 1 by an attributable rule rather than to pre-load them by a runtime heuristic, which is the distinction Part VI insisted on. And a class whose silent-fault inference is elevated is flagged for counterfactual replay, because the inference is weak evidence and the replay is strong evidence, and an estate that acts on weak evidence without upgrading it is doing exactly what its own epistemic ladder forbids.

7.7 Failure modes and adversarial cases. The overfitted profile is the first: a compiler tuned on a narrow history compiles beautifully for the missions it has seen and catastrophically for the first genuinely fresh one, having learned the past rather than the structure. The defense is that mission CLASS, not mission identity, is the unit of learning, and a mission that does not match a known class is compiled by the base selection law with its priors intact rather than by a profile that has no standing over it. The frozen profile is the second: the profile learns, converges, and stops, and thereafter it is a set of assumptions with a data-driven pedigree, which is the most persuasive kind of stale assumption there is — the defense is that recall's uncertainty band never closes, because silent faults are never fully observable, and a converged profile that reports a closed band is reporting a lie. The successful-mission-with-bad-pack case corrupts the profile if outcome is scored alone, since a mission that passed despite an insufficient pack teaches the compiler that the insufficiency was fine, and the defense is that usage and faults are scored alongside outcome, never outcome alone. The adversarial case is the compiler that learns to reduce measured faults by ensuring the actor never notices its gaps — an unlikely direct optimization but a real emergent one, since packs that are confidently and fluently wrong produce fewer faults than packs that are visibly thin — and the only instrument that catches it is rework rate, which is why rework is the ground truth and not one metric among five.

7.8 What DAIF-08 hands back to its parents. The profile is not the property of this dataset; it is a contribution to the systems that already exist. GK-06 receives the corrections of 7.6 as inputs to its ranking, and its selection heuristic improves by measurement rather than by taste. CO-04 receives a mission-scoped need signal it never had, so its tiering can evict by observed non-use rather than by recency alone. CO-00 receives, for the first time, the ability to distinguish an over-spend caused by an oversized pack from an over-spend caused by an undersized one that forced the actor to hunt. DAIF-03 receives the behavioral-equivalence evidence its fidelity gates need, since the counterfactual replay of 7.3 is exactly a behavioral-equivalence experiment. DAIF-07 receives the unresolved faults that must be carried past the end of a mission as obligations. And DAIF-21 receives the observation that a mission's territory is volatile, evidenced by its re-assembly rate. This is the whole justification of an EXTEND verdict: DAIF-08 does not stand beside its parents as a rival, it stands under them as the instrument they were missing, and the measure of its success is that GK-06's packs get better, not that DAIF-08's dashboards get fuller.

PART VII FINAL LAW. The loop closes by recording what was loaded, what was demonstrably used and by which instrument, what was never touched, what was faulted for and under which exclusion reason, what was missing and never requested — inferred from failures and reported as an inference — and whether the mission passed a real gate without rework, then recompiling the next pack of that class from the observed profile rather than from a designer's intuition; the metrics are context precision, context recall, page-fault rate, silent-fault rate, and rework rate, and they are gated CONJUNCTIVELY and with teeth, because precision is trivially maximized by loading almost nothing and faulting constantly while recall is trivially maximized by loading the estate, so neither may rise at the other's expense and both are accountable to the mission's success rate — since the only true objective is verified outcomes per unit of reasoning cost, and by CO-12's Telemetry-Before-Claims no claim about any of it may be made without a measured triple; the measure of this dataset's success is therefore not that its own dashboards are full but that GK-06's packs are better than they were, which is the entire content of an EXTEND verdict honestly discharged.

PART VIII — BUDGET ALLOCATION ACROSS THE PACK

8.1 The parent, named before the claim. CO-00 owns the context budget. It states the ceiling as a governed quantity, it counts spend, and it is the only component in the estate entitled to say how much reasoning cost a mission may consume. DAIF-08 does not hold a second ceiling, does not maintain a rival counter, and does not arbitrate an over-spend — every figure in this Part is denominated in CO-00's unit and every authorization in it composes CO-00's decision. What CO-00 has never owned, and what nothing else in the estate owns either, is the DIVISION of an allowed budget among the categories of a pack. A ceiling says how much. It says nothing about what shape the spend should take, and a ceiling with no shape is an invitation to spend the whole allowance on whichever category the selection law happened to reach first. That is the gap this Part closes, and it closes nothing else: allocation is a runtime concern because it is defined relative to a mission, and a mission is exactly what CO-00 does not see. The relationship is therefore one of subordination and not of overlap — CO-00 grants a quantity, DAIF-08 shapes it, and if the shaping and the ceiling ever disagree the ceiling wins without argument.

8.2 Why an unallocated budget degrades in a predictable direction. A pack assembled against a ceiling alone is assembled greedily, and greed has a signature. The selection law traverses from the mission's seed, admits the units it reaches in rank order, and stops when the ceiling is met — which means the category nearest the seed is admitted exhaustively and the categories far from it are admitted not at all. In practice the near category is almost always the architecture slice, because structure is what a seed points at, and the far categories are almost always failure history and the evaluation contract, because those are reached by a different kind of edge. The observable consequence is a pack that describes the code in great detail, says nothing about how the code has broken before, and cannot state how the work will be judged. This is not a hypothetical shape; it is the shape a rank-ordered traversal under a flat ceiling produces every time, and it is the shape a pack takes when nobody has decided in advance what fraction of the allowance each category is entitled to. Allocation is the instrument that converts a greedy traversal into a budgeted one, by giving each category a floor it cannot be starved below and a ceiling it cannot exceed by crowding.

8.3 The eight categories and what each one buys. A pack's allowance divides into eight envelopes. MISSION AND CONSTRAINTS is the mission contract plus the hard constraints admitted whole; it is a floor and not a share, because a constraint may not be abridged to fit a percentage and a mission whose constraints do not fit its budget is under-funded rather than over-constrained. CURRENT REALITY buys the observed state — what is on disk, what is running, what passed, what failed — and it is the envelope whose under-funding produces an actor reasoning correctly against a world that no longer exists. ARCHITECTURE SLICE buys the structure the mission touches, closed over hard dependencies from the seed; it is the envelope that greedily consumes everything when unbudgeted. PROCEDURES buys the methods whose activation conditions actually fired. FAILURE HISTORY buys the recurring errors and causal chains matching this situation, and it is the envelope most reliably starved, because it is far from the seed and its value is invisible until the moment it would have prevented a repetition. EVALUATION CONTRACT buys the checks the output will face, and it is small and non-negotiable, because a pack that cannot say how its mission will be judged has failed to bring the thing that makes minimality checkable. EVIDENCE buys the provenance and the observations behind the pack's assertions, sized so that the pack's critical claims are auditable without leaving it. And the EXPANSION RESERVE buys nothing at all at assembly time, which is the whole of 8.4.

8.4 The reserve, and why it is not negotiable. The expansion reserve is a fraction of the mission's budget deliberately NOT spent at assembly, held back so that page faults and level ascents can be served during execution. It is the single most counter-intuitive allocation in this dataset, because it is an envelope whose only content at compile time is emptiness, and every pressure in the system argues for filling it: the compiler could admit more units, the actor could be given a deeper starting level, and the reserve looks like an allowance being wasted on a fault that may never occur. Every one of those arguments is locally sound and all of them are wrong, for one structural reason. A pack that spends one hundred percent of its budget at assembly has no capacity to correct itself. Every page fault it raises is, by arithmetic, a budget violation — the fault's cost estimate cannot fit inside a remaining allowance of zero — so every fault is either denied or forces an escalation. An actor that learns its faults are always denied stops faulting. It does not stop needing; it starts GUESSING, which is the silent fault of Part V, the failure that produces confident wrong work and emits no signal. A runtime with no reserve therefore does not merely fail to correct itself; it actively manufactures the one failure it was built to make visible. The reserve is the price of the fault mechanism's existence, and a runtime that will not pay it has bought the instrumentation and disabled the instrument.

8.5 Allocation adapts by mission class, and static allocation is a failure mode. There is no universal split, and any attempt to fix one across all missions is a design error with an obvious refutation: a debugging mission and an architecture mission need opposite shapes. A debugging mission runs on current reality, failure history, and evidence — it must know precisely what the world is doing now, what it has done wrong before, and what observation grounds each claim, and it needs comparatively little structure, because the region of structure under suspicion is narrow and already named by the fault being chased. An architecture mission inverts every term: it runs on the architecture slice, the decisions with their justifications, and the constraints, and it has comparatively little use for the current reality of a running process, because it is reasoning about what the structure should become rather than what it is doing this second. Handing the debugging split to the architecture mission starves it of the structure it exists to reason over; handing the architecture split to the debugging mission gives it a beautiful map of a subsystem and nothing about the three ways that subsystem has failed. A static allocation is therefore not a conservative default but a guarantee of being wrong for at least half the mission classes it serves, and the classes it is wrong for will fault constantly against an allowance shaped for someone else's work.

8.6 The observed profile reallocates, and the reallocation is attributable. Part VII established that the estate records what was loaded, what was used, what was never touched, and what was faulted for, joined on the mission's class. Those four records are exactly the inputs an allocator needs. A category whose admitted units are never touched across a statistically meaningful sample of a class has an envelope too large for that class, and its share is reduced with the reduction attributed to observed non-use rather than to a designer's revision. A category that is repeatedly faulted for across a class has an envelope too small, and its share is increased with the increase attributed to the fault record that indicts it. A class whose reserve is exhausted before its missions complete has a reserve too small OR a compiler too wrong, and the two are distinguished by the fault TYPES: faults against known handles indict the selection law's boundary, while UNKNOWN-GAP faults indict the class's category shape itself. The allocator therefore learns the same way the compiler does and from the same ledger, which is deliberate — a second learning loop with a second data source would be a second thing to drift.

8.7 Failure modes and adversarial cases. Reserve raiding is the first and the most tempting: a compiler under pressure to admit one more unit borrows from the reserve, the borrow succeeds, the mission passes, and the borrow becomes the norm — after which the reserve exists in the schema and not in the arithmetic. The defense is that the reserve is deducted from the allowance BEFORE the selection law is given a ceiling, so the compiler never sees it and cannot spend what it cannot address. Category starvation is the second: an envelope is set so low that its category is present in name and absent in substance, most often failure history, which is starved because it is cheap to omit and its absence produces no immediate symptom. The defense is a floor per category and a refusal to compile a pack that cannot meet its floors, on the same grounds as an under-funded mission. The adversarial case is the mission phrased to obtain a maximal envelope — a request that describes itself as an architecture mission in order to claim the architecture split while actually performing a survey — and the answer is the same refusal Part II already issues: an unbounded subject is not a mission, and a class is assigned from the compiled contract's intent and done-gate rather than from the requester's self-description.

8.8 Metrics with countermetrics. Allocation is measured by per-category utilization, by reserve exhaustion rate, by faults attributed per category, and by the distance between the class's nominal split and the split its profile implies. Each is Goodhart-vulnerable alone. Per-category utilization is trivially maximized by shrinking every envelope to a single unit, and is therefore gated against the fault rate. Reserve exhaustion rate can be driven to zero by making the reserve enormous, which converts the reserve into the pack, so it is gated against the admitted-unit count and against mission success. Faults per category are trivially reduced by pre-emptive admission, which is the very heuristic Part VI forbids, so they are read alongside utilization — a category with no faults and no usage has not been sized correctly, it has been sized irrelevantly. And the whole allocation is accountable to the ratio Part VII named as the only true objective: verified outcomes per unit of reasoning cost, against which every envelope, every floor, and the reserve itself are subordinate proxies to be discarded the moment they stop tracking it.

PART VIII FINAL LAW. CO-00 owns the ceiling and DAIF-08 owns the DIVISION of what the ceiling allows, because a budget with no shape is spent greedily by the selection law on whatever category lies nearest the mission's seed — reliably the architecture slice, reliably starving failure history and the evaluation contract — so the allowance is apportioned across mission and constraints, current reality, the architecture slice, procedures, failure history, the evaluation contract, evidence, and an EXPANSION RESERVE that is deducted before the compiler is handed a ceiling and can therefore never be raided; the reserve is non-negotiable because a pack that spends one hundred percent of its budget at assembly has no capacity to correct itself, which makes every page fault an arithmetic budget violation, which teaches the actor that faulting is futile and that guessing is the only path forward — so a runtime without a reserve does not merely fail to correct itself, it manufactures the silent fault it was built to expose; and the split is never static, because a debugging mission and an architecture mission need opposite shapes and a fixed allocation is a guarantee of being wrong for half the classes it serves.

PART IX — MODEL-AWARE AND TASK-AWARE DELIVERY

9.1 The two parents and the seam between them. CO-03 owns model routing: it decides which model a piece of work is dispatched to, on grounds of capability, cost, and the estate's own cost discipline, and DAIF-08 does not second-guess that decision, does not maintain a rival routing table, and does not route anything. DAIF-02 owns the adaptive lowering passes: it defines the transformations that take a compiled representation and render it at a chosen density, explicitness, and structure, and DAIF-08 does not implement a lowering, does not define a pass, and does not own the pass ordering. What sits between them, unowned until now, is the RUNTIME DECISION: given this mission, bound to this contract, dispatched by CO-03 to this model, which lowering does the pack request? The pack is one artifact; its delivery is not. Two actors on the same mission with different capabilities need the same knowledge delivered differently, and an estate that ships a single rendering to both has decided, silently, that the delivery is not a variable. It is a variable, it is the cheapest variable in the whole runtime to move, and moving it correctly is worth more than another round of selection tuning — because a unit that was admitted, was relevant, and was rendered at the wrong density has been paid for and not received.

9.2 The six dimensions along which a delivery varies. DENSITY is how much reasoning is compressed into a unit of text: a strong model reasons over a compressed premise and reconstructs the intermediate steps, while a weaker one needs the steps present. EXPLICITNESS is how much of what is implied is said: a strong model infers a constraint's scope from the constraint, and a weaker one needs the scope stated. STRUCTURE is how much scaffolding the delivery carries — the ordering, the sectioning, the explicit naming of what each block is for — and it is the dimension that helps a weaker model most and costs a stronger one the most, because scaffolding a strong model does not need is noise it must read past. EXAMPLES are the concrete instances that ground an abstraction, and they are the most expensive dimension by token count and the most valuable to a weaker actor. REDUNDANCY is the deliberate restatement of a critical constraint at more than one point in the pack, which a strong model reads as repetition and a weaker one reads as emphasis that survives its own attention decay. And AUTONOMY is how much of the decision the delivery leaves open: a strong actor is handed a goal and the constraints, and a weaker one is handed a goal, the constraints, and the sequence — and handing a strong actor the sequence is how a capable model is reduced to a transcription engine.

9.3 The model axis, stated without a table. The delivery gradient runs in one direction and it runs monotonically. Toward the strong end: higher density, lower explicitness, less structure, fewer examples, no redundancy, maximal autonomy — a pack that states the mission, admits the constraints whole, gives the reality and the boundary, and gets out of the way. Toward the weaker end: lower density, higher explicitness, more structure, more examples, deliberate redundancy on the constraints that must not be missed, and a narrowed autonomy that names the next action rather than the goal. The pack's CONTENT is identical along the whole gradient — the same units, admitted under the same rules, attributed the same way — and only the rendering moves, which is the property that makes the whole mechanism safe: a delivery decision cannot change what the actor knows, only how much work the actor must do to know it. If a delivery decision ever changes the admitted set, it has stopped being a delivery decision and has become a selection decision made in the wrong place, and it is refused on exactly those grounds.

9.4 The task axis, and the seven shapes. Mission class determines delivery as strongly as model tier does. A DEBUGGING mission wants maximal current reality, maximal failure history, high explicitness on what has been observed versus what is inferred, and low autonomy at the outset — because a debugging actor granted broad autonomy early will chase a hypothesis instead of establishing the fault. An IMPLEMENTATION mission wants the procedures, the constraints, and high autonomy, because the path is known and the actor is executing it. An ARCHITECTURE mission wants maximal decision detail with justifications, low structure, and maximal autonomy, because scaffolding an architecture mission is pre-deciding it. An AUDIT mission wants the evaluation contract front and centre, maximal evidence, high explicitness, and near-zero autonomy on method, because an audit that invents its own method has audited nothing. A RESEARCH mission wants a thin kernel, a rich handle set, and maximal autonomy — it is the mission class whose pack should be smallest and whose boundary should be most visible, since its work IS the traversal of that boundary. A MIGRATION mission wants both realities — the world as it is and the world as it must become — plus the constraints that bind the transition, and it is the class most damaged by a current-reality envelope sized for one world. And a REVIEW mission wants the evaluation contract, the decisions with their justifications, and the failure history, because a review without the justifications will flag as defects the choices that were made deliberately.

9.5 The delivery request, and what the runtime actually does. The runtime composes a delivery request from four inputs and hands it to DAIF-02's lowering machinery: the model tier as CO-03 already decided it, the mission class as the compiled contract already declares it, the current disclosure level as Part VI's ladder already holds it, and the profile's observed delivery outcomes for this class on this tier. It then receives a rendered pack and admits it under Part III's four structural checks unchanged — contract-binding, constraint-integrity, boundary-honesty, gate-presence — and the second of those is the one that matters most here, because the delivery axis is exactly where a hard constraint is most likely to be quietly compressed in the name of density. A lowering pass that abridges a constraint to raise density has traded the one thing it may not trade, and constraint-integrity refuses it at load rather than warning about it, for the reason Part III already gave: a warning about an abridged law is a law that will be violated with a paper trail.

9.6 What this Part does not own, stated so it cannot be blurred. DAIF-08 does not define a lowering pass, does not order the passes, does not specify how density is achieved, and does not own the transformations themselves — DAIF-02 Part XVI owns all of that and this Part deliberately does not restate a single one of its mechanisms. DAIF-08 does not route — CO-03 routes, and a mission arrives here with its model already chosen. What DAIF-08 owns is one function and one only: from mission class, model tier, disclosure level, and profile, produce the delivery request. It is a small sovereign territory and it is stated small on purpose, because the failure this corpus exists to prevent is the dataset that annexes its parents' mechanisms in the act of composing them.

9.7 Failure modes and the adversarial cases. One-size delivery is the baseline failure and it fails in both directions at once: the same rendering is simultaneously too verbose for the strong model, which pays tokens to read past scaffolding it did not need and has its signal-to-noise degraded by redundancy it reads as noise, and too terse for the weaker model, which is handed a compressed premise it cannot unpack and either faults repeatedly or, far worse, guesses. The failure is symmetrical and it is invisible in aggregate, because the two errors partially cancel in any averaged metric — which is why delivery quality is measured per tier and per class and never pooled. Over-lowering is the second failure: a delivery so explicit, so scaffolded, and so example-laden that a capable actor is reduced to following a script, its judgment unused and its capability wasted, and it presents as a mission that succeeds while producing exactly the output a weaker and cheaper actor would have produced — which is a cost-discipline finding as much as a delivery one. The adversarial case is delivery drift into selection: a lowering pass, given a density target it cannot meet by compression alone, begins dropping units, and the pack that arrives is not the pack that was compiled. The defense is that the admitted-unit set is fixed before delivery and verified after it; a delivered pack whose unit set differs from its compiled unit set is refused, and the refusal is recorded against the pass that produced it.

9.8 Metrics and countermetrics. Delivery is measured by per-tier fault rate, by per-tier and per-class success rate, by the delta between a class's rendered size and its compiled size, and by the utilization of the delivery-added material — the examples, the scaffolding, the redundant restatements — which is measurable by exactly the citation instrument Part VII already defined. A high per-tier fault rate on a weak model with a dense delivery is the signature of under-lowering. A high success rate on a strong model with an over-lowered delivery is not a triumph but a suspicion, because it may mean the model's capability was never engaged; it is gated against the counterfactual of the same mission on a cheaper tier, which is the only instrument that distinguishes a mission that needed the strong model from a mission that merely got one. No delivery metric stands alone and none is pooled across tiers, because a pooled delivery metric is an average over two opposite errors and reports the mean of a bimodal distribution as if it were a mode.

PART IX FINAL LAW. The same pack is never delivered identically to every consumer: CO-03 owns which model the work is routed to and DAIF-02 Part XVI owns the lowering passes that transform a compiled representation, and DAIF-08 owns only the runtime decision of WHICH lowering to request for this mission class, on this model tier, at this disclosure level, informed by this class's observed profile — a small sovereign territory named small on purpose; delivery varies along six dimensions, density, explicitness, structure, examples, redundancy, and autonomy, and it varies by task shape as sharply as by model tier, since a debugging mission wants reality and failure history with low early autonomy while an architecture mission wants decision justifications with maximal autonomy and an audit wants the evaluation contract with near-zero autonomy on method; the delivery decision may change only the RENDERING and never the admitted set — a lowering that drops a unit has become a selection decision made in the wrong place and is refused — and the failure it exists to prevent is one-size delivery, which is simultaneously too verbose for the strong model and too terse for the weak one, an error that cancels in any pooled metric and is therefore measured per tier and per class and never averaged.

PART X — CACHE AND REUSE ACROSS MISSIONS

10.1 The parents, and the question they do not answer. CO-04 owns the memory tiers — hot, warm, cold, external — and decides what is resident. CO-06 owns eviction, and decides what leaves. Between them the estate has a complete answer to the question "what should be held right now," and DAIF-08 rebuilds neither, forks neither, and adds no tier and no eviction policy. What neither answers, because neither has ever seen a mission contract, is the question this Part exists for: is a pack, or a fragment of a pack, that was assembled for a PREVIOUS mission valid for a NEW one? That is not a residency question and it is not an eviction question. It is a VALIDITY question, and validity is a property of the relationship between two missions, which is an object CO-04 and CO-06 do not hold. Reuse is the highest-leverage economy available to the runtime — compilation is expensive, a great many missions are near-neighbors of missions already run, and recompiling from scratch what was compiled correctly an hour ago is pure waste — and it is simultaneously the most dangerous, because a wrong reuse injects into a live mission a body of knowledge that is fast, confident, and false. The entire content of this Part is the gate that separates those two outcomes.

10.2 The reuse unit is not the pack. It is tempting to make the pack the unit of reuse, because the pack is the artifact and artifacts are what caches hold. This is wrong and the error is instructive. A pack is a SELECTION made against a specific contract, and Part II already established that a pack carries its mission's identifier and is refused at load if that identifier does not match the running mission — which forecloses whole-pack reuse by construction, and does so deliberately, because reusing a whole pack across missions mistakes a selection for a summary. What is reusable is the UNIT: the compiled decision with its justification, the compiled architecture slice, the compiled failure mode, the evidence bundle. Those are compilations of reality, not selections against a goal, and a compilation of reality is valid for any mission whose premises about that reality are compatible. The reuse contract is therefore stated at the unit level and the pack is reassembled every time from units that may be new or may be recovered — which costs the selection law's traversal and saves the compilation of every unit it reaches, and the compilation is where nearly all the cost lives.

10.3 The premise-compatibility gate, in seven dimensions. A compiled unit may be reused for a new mission only if the two missions' premises are compatible across seven axes, and a failure on any one axis refuses the reuse outright. ASSUMPTIONS: the facts about the world the unit was compiled under must still be the facts the new mission assumes, since a unit compiled under an assumption the new mission contradicts is a unit that will be believed and is wrong. VERSIONS: the version of the artifact, subsystem, or interface the unit describes must match the version the new mission touches, and a version mismatch is the commonest and most mechanical refusal in the whole gate. CONSTRAINTS: the laws in force when the unit was compiled must still be in force, because a decision justified under a constraint that has since been lifted is a decision whose justification has evaporated. GOALS: the unit must have been compiled under a goal whose relationship to the new goal is one of compatibility rather than of conflict, since a unit compiled to argue FOR a direction is not neutral evidence in a mission that is deciding WHETHER to take it. ENVIRONMENT: the substrate, the host, and the runtime conditions must match, because a procedure compiled for one environment is an actively dangerous instruction in another. AUTHORITY: the party who authorized the unit's scope must still be the party whose authority governs the new mission, since a unit admitted under one authority's scope is not automatically admissible under another's. And RECENCY: the unit must be fresh enough that reality has not moved beneath it, which is the axis that hands off to DAIF-21.

10.4 The same question under different premises is a different question. This is the formal statement of the whole gate and it deserves to stand alone, because the intuition that defeats it is nearly universal. Two missions ask, in words, the same thing: how does this subsystem handle a timeout? The compiled answer is on hand, it is correct, it was verified. And it is invalid for the second mission, because the first mission asked it of a subsystem at one version under one set of constraints in one environment, and the second asks it of a subsystem whose timeout handling was changed last week under a constraint that did not exist then. The QUESTION is identical and the PREMISES are not, and an answer is a function of both. A cache keyed on the question alone will hit, and the hit will be a lie. This is the mechanism by which stale context enters an estate that believes itself careful: not through carelessness about the answer, but through carelessness about the premises the answer was an answer UNDER. The gate's key is therefore the question joined with all seven premise dimensions, and a cache that keys on anything less has not built a cache, it has built a plausible-sounding fabrication engine with excellent latency.

10.5 Partial reuse, and recompiling the remainder. A refusal on one unit is not a refusal on the pack. The correct behavior, and the one that captures nearly all of the available economy, is partial: the runtime tests each candidate unit against the seven axes, admits the units that pass, marks the units that fail as invalid with the axis that failed them recorded, and recompiles exactly those and nothing else. A version mismatch on one architecture node invalidates that node and its dependents; it does not invalidate the decision justifications that were compiled under constraints which have not moved, and it does not invalidate the failure history, which is a record of what happened and is not versioned by what the code is doing today. Partial reuse is therefore the normal case and total reuse is the rare one, and a runtime observing a high total-reuse rate should suspect its gate rather than congratulate its cache — because seven axes all passing for every unit of a pack, across two missions genuinely distinct enough to have been compiled separately, is an unlikely event, and unlikely events that recur are usually gates that are not firing.

10.6 The stale hit, and why it is the worst outcome in the domain. A cache has three outcomes and they are not equally bad. A MISS costs a compilation and is the baseline. A CORRECT HIT saves a compilation and is the point. A STALE HIT is a catastrophe wearing the costume of the point: it is fast, it returns a confident, well-formed, provenance-carrying unit, it passes every structural check the runtime performs, and it is FALSE. The actor does not fault, because it holds what it needed. It does not ascend, because there is no conflict it can see. It reasons correctly to a wrong conclusion, and it does so with the full confidence a compiled and attributed unit deserves — a confidence the unit has forfeited and the runtime has not noticed. This is worse than any omission, because an omission at least has a chance of being detected by the actor's own reasoning stalling, and a stale hit produces no stall at all. The ranking is absolute and it governs the gate's design: a cache MUST be biased toward the miss. A gate that is uncertain refuses, a unit whose recency cannot be established is treated as stale rather than fresh, and the cost of the extra compilation is accepted without argument, because the alternative is not a saved compilation but a corrupted mission.

10.7 Invalidation is not owned here. DAIF-21 owns reality synchronization and semantic change detection: it is what notices that a source moved, that an artifact changed, that a constraint was amended, and it is what emits the invalidation signal. DAIF-08's obligation on that signal is narrow and total — accept it, mark the affected units invalid, refuse to serve them from any tier, refuse to resolve a handle against an invalidated pack rather than resolving it into a stale body, and force recompilation of exactly the invalidated set. DAIF-08 does not detect change, does not maintain a freshness index, and does not decide what a change means. The recency axis of 10.3 is therefore not an independent mechanism but a consumption of DAIF-21's verdict, and the honest statement of the seam is that the six other axes are checkable from the two mission contracts alone, while the seventh is checkable only by asking a system that watches the world.

10.8 Failure modes, adversarial cases, and metrics. Hit-rate optimization is the adversarial case and it is the one that will actually happen, because hit rate is the metric a cache naturally reports and it is trivially maximized by loosening the gate. A runtime rewarded on hit rate will relax the version axis first, because version mismatches are the commonest refusal and relaxing them produces the largest immediate gain — and it will produce that gain by serving units about code that no longer exists. Hit rate is therefore never a standalone metric and is always gated against the stale-hit rate, which is measured by sampling: a fraction of hits are recompiled anyway and compared against what the cache served, and a divergence is a stale hit caught in the act. The other metrics are compilation cost saved, per-axis refusal counts — which are diagnostic, because a dominant refusal axis names where reality is churning fastest — and the rework rate of missions that ran on reused units versus missions that ran on freshly compiled ones, which is the ground truth all of it answers to. If reused missions rework more than fresh ones, the cache is a liability whatever its hit rate says, and the correct response is to tighten the gate until the two rates converge.

PART X FINAL LAW. CO-04 owns the tiers and CO-06 owns eviction and DAIF-08 rebuilds neither; what it owns is the mission-level VALIDITY question those two cannot answer — may a unit compiled for a previous mission be admitted into a new one — and the answer is the premise-compatibility gate: assumptions, versions, constraints, goals, environment, authority, and recency must all be compatible, and a failure on any single axis refuses the unit, because the same QUESTION under different PREMISES is a different question and an answer is a function of both, which is precisely how stale context enters an estate that is careful about answers and careless about the premises they were answers under; the reuse unit is the compiled unit and never the pack, since a pack is a selection against a contract and reusing it whole mistakes a selection for a summary; partial reuse is the normal case, with the invalidated units recompiled and nothing else; and the gate is biased toward the MISS without apology, because a miss costs one compilation while a STALE HIT is fast, confident, structurally perfect, and wrong — the actor does not fault, because it holds what it needed, and it reasons correctly to a false conclusion with the full confidence a compiled unit deserves and has forfeited.

PART XI — SESSION CONTINUITY: THE FIRST PROVING VERTICAL

11.1 Why this vertical and no other. The corpus designates session continuity as its first proving ground, and the designation is not arbitrary. It is the one mission class where every mechanism this dataset contributes is simultaneously load-bearing, where the failure is universal and daily rather than hypothetical, and where the counterfactual is cheap to construct — because the same mission, continued the old way and continued from a compiled pack, can be run side by side and compared. The scenario is the ordinary one: a context boundary arrives. The session is cleared, a compaction fires, the host crashes, the pane closes, a fresh session opens on the same repository with a mission half-finished. What happens today is that the agent reconstructs its situation by RE-READING — it opens the files it opened before, re-derives the decisions it already made, re-discovers the constraints it was already honoring, and re-learns the failures it already suffered. It pays the full bring-up cost a second time, it pays it from a context that is now smaller than the one it burned, and it arrives at a reconstruction that is not guaranteed to match what it had. The proposition of this vertical is that it should instead be handed a compiled pack.

11.2 The parents, named and not rebuilt. Lazarus is the estate's recovery substrate; it recovers a past session as read-only context from a searchable vault of prior records. `session_compiler` compiles what a session WAS. The estate's resumption-file convention holds a self-contained continuation prompt at a project's root, updated after every sealed unit of work so that a mid-run crash always leaves a valid re-entry point. Those three exist, they are live, and DAIF-08 builds none of them. What DAIF-08 contributes is the RUNTIME CONTRACT of resumption: not the recovery mechanism, not the compilation of the session record, not the file that carries it, but the specification of what a resume pack must CONTAIN to be sufficient, the gate by which its sufficiency is judged, and the visible failure it must produce when it cannot guarantee that sufficiency. The parents answer "how do we get the past back." This Part answers "what must come back, and how do we know it did."

11.3 The eight contents of a resume pack. A resume pack is a Mission Context Pack under a specific class, and its eight required contents are its class's floors. The MISSION CONTRACT, because a continued mission that has forgotten its own done-gate is not a continuation but a fresh improvisation near the same subject. The HARD CONSTRAINTS, admitted whole and never abridged, because the constraints are precisely what a re-reading reconstruction most reliably loses — they were absorbed across a hundred turns and no single file holds them. The DECISIONS WITH THEIR JUSTIFICATIONS, because a continued mission that holds a decision without its reason will either violate it as arbitrary or be paralyzed by it as inscrutable, and both outcomes are worse than not holding it at all. The CURRENT REALITY, as observations and verdicts rather than as the plan's assumptions, because the world moved while the session was down. The OPEN OBLIGATIONS, drawn from DAIF-07, which owns their lifecycle — what the mission owes, what blocks it, what it promised and has not yet delivered — and these are the single most frequently lost item across a boundary, because an obligation lives in the agent's working intent and nowhere else unless something writes it down. The EVIDENCE POINTERS, so that every critical claim the pack asserts can be audited without re-deriving it. The EXPANSION HANDLES, so that the continued actor can see what was deliberately left out and request it rather than inventing it. And the DONE-GATE and evaluation contract, present at the start, because a resumed mission that learns its standard at the gate has been reasoning without one for its entire second half.

11.4 The boundary taxonomy, and why the pack must be written before the boundary. Boundaries come in two kinds and the difference determines everything about when the pack is written. A GRACEFUL boundary — a clear, a planned compaction, an intentional close — offers a moment in which a pack could be compiled. An ABRUPT boundary — a crash, a power loss, a killed process — offers nothing. A resume pack written only at the graceful boundary is a pack that exists exactly when it is least needed, and is absent exactly when the loss is total. The rule follows without discussion: the resume pack is maintained CONTINUOUSLY, updated after every sealed unit of work and never only at the end, at a point in the mission's context consumption where there is still capacity to write it rather than at exhaustion. This is the estate's existing resumption-file discipline generalized from a prose file to a compiled pack, and it is generalized rather than replaced because the discipline is correct and its artifact is what was thin.

11.5 The done-gate, stated exactly. This vertical passes only on six clauses, and it passes on all six or it does not pass. First, one hundred percent of the mission's hard constraints survive the boundary — not most, not the important ones, all of them, because a constraint lost is a law that will be broken by an actor who does not know it exists. Second, one hundred percent of the open obligations survive, for the same reason and with the same absoluteness. Third, the current state is CORRECTLY IDENTIFIED — the resumed actor knows what is done, what is in flight, and what has not been started, and it knows this from the pack rather than from an inference. Fourth, the actor continues WITHOUT indiscriminate re-reading, which is the behavioral claim of the whole vertical and is observable directly by counting what the resumed actor opens. Fifth, every critical claim in the pack links to evidence, so that a resumed actor doubting the pack can check it rather than re-derive it. And sixth, a source that changed while the session was down is DETECTED, which composes DAIF-21's invalidation signal and marks the affected units invalid rather than serving them stale. Six clauses, conjunctive, no partial credit.

11.6 The seventh clause: failing visibly. The gate has a seventh requirement that is not a clause about success but a clause about failure, and it is the one that makes the other six safe. When the runtime CANNOT guarantee fidelity — the record is incomplete, a source moved and the change cannot be reconciled, an obligation's provenance is broken, the pack's constraint set cannot be verified byte-complete against its source — it must FAIL VISIBLY. It must say what it cannot guarantee, name the specific units it is unsure of, and hand the mission to its authority rather than proceeding on a pack it has not verified. The alternative behavior is the one every naive continuity system exhibits: it produces a plausible pack, the actor resumes confidently, and the missing constraint surfaces three hours later as a violation nobody can explain. A continuity system that degrades silently is worse than no continuity system at all, because no continuity system at least forces the honest re-read, while a silent one substitutes a fabrication for it and hides the substitution. Visible failure is therefore not a courtesy; it is the property that makes the entire vertical trustworthy, and it is a gate that must be OBSERVED to refuse, since a gate never seen to refuse is indistinguishable from an inert one.

11.7 No savings figure is asserted in advance. It is obvious that resuming from a compiled pack should cost less than re-reading a repository, and the obviousness is exactly why no number appears here. This dataset asserts NO figure — not a percentage, not a multiple, not a range — because an unmeasured efficiency claim is the fabrication the estate's own epistemic discipline exists to refuse, and CO-12's Telemetry-Before-Claims forbids any claim without a measured triple. The measurement protocol is the behavioral trial DAIF-03 Part X already defines and this Part composes rather than reinvents: the same mission, at the same boundary, continued twice — once by the current re-reading path and once from a compiled resume pack — with three quantities recorded on each arm. The reasoning cost consumed to reach the point where the actor is productively working again. The FIDELITY of the continuation, measured as behavioral equivalence: does the resumed actor reach the conclusions the unbroken actor would have reached, and does it honor the constraints the unbroken actor was honoring. And the rework, because a continuation that was cheap and diverged is not a saving but a debt. Only after that triple is measured, on a real sample, does any figure enter this corpus, and it enters attributed to the trial that produced it.

11.8 Failure modes specific to this vertical. The plausible-reconstruction failure is the first and the most common: the pack is compiled, it reads well, it is internally coherent, and it is missing one constraint, and the resumed mission proceeds to violate that constraint with perfect confidence — which is why constraint survival is one hundred percent and not a threshold. The obligation-evaporation failure is the second and it is the estate's documented daily loss: work promised in the first half of a session and never delivered in the second, because the obligation existed only in the agent's working intent and the boundary took the intent with it — which is why DAIF-07's obligations are a required content and not an optional one. The stale-continuity failure is the third: the pack is faithful to a world that changed while the session was down, which is the stale hit of Part X arriving through the continuity door, and it is caught by clause six and by nothing else. And the ceremonial-resume failure is the fourth: a resume pack that is present, well-formed, and hollow — every slot populated, none of them load-bearing — which passes structurally and fails functionally, and which is caught only because the done-gate is behavioral rather than structural. The actor either continues without re-reading or it does not, and that is a thing you can watch.

PART XI FINAL LAW. Session continuity is the corpus's designated proving vertical because it is the one mission class where every mechanism of this dataset is load-bearing at once and where the counterfactual is cheap to build: after a boundary — a clear, a compaction, a crash, a fresh session — the agent must continue from a COMPILED PACK rather than by re-reading the repository, and Lazarus, `session_compiler`, and the estate's resumption-file discipline are the parents that make it possible and that DAIF-08 names rather than rebuilds; the resume pack carries the mission contract, the hard constraints whole, the decisions WITH their justifications, the current reality, DAIF-07's open obligations, evidence pointers, expansion handles, and the done-gate, and it is written continuously after every sealed unit of work — never only at the graceful boundary, because a pack that exists only when the close was planned is absent exactly when the loss is total; the gate passes on six conjunctive clauses with no partial credit — one hundred percent of constraints and one hundred percent of obligations survive, the state is correctly identified, the actor continues without indiscriminate re-reading, every critical claim links to evidence, and a changed source is detected — plus the seventh and enabling requirement that the system FAIL VISIBLY when it cannot guarantee fidelity, since a continuity system that degrades silently is worse than none at all; and NO savings figure is asserted here in advance, because an unmeasured efficiency claim is a fabrication and the number must come from the behavioral trial that measures cost, fidelity, and rework on both arms.

PART XII — MULTI-AGENT AND PARALLEL CONTEXT

12.1 The parents and the boundary of this Part. PM-03 is the estate's findings bus: the transport by which one actor's discovery reaches the others. It exists, it is live, DAIF-08 does not fork it, does not extend its schema, and does not build a second channel — every propagation described in this Part rides PM-03 and nothing else. PM-01 and PM-02 own the surrounding process-management concerns of parallel work. DAIF-08 owns exactly one thing here, and it is the thing none of them own: how a MISSION'S CONTEXT is partitioned across the actors working it. A bus tells you how a finding travels. It does not tell you what each actor should have been holding before the finding existed, what they should hold in common, what they must be forbidden from holding, or what the arrival of a finding obliges each of them to do with their own pack. That is a context question and it is the runtime's.

12.2 The shared kernel and the sub-packs. A mission worked by several actors partitions into one shared kernel and N sub-packs. The KERNEL is compiled once and is identical for every actor: the mission contract, the hard constraints admitted whole, the done-gate, and the evaluation contract — which is precisely level 0 of Part VI's ladder, and the identity is not a coincidence but the point. Level 0 is the irreducible minimum below which a mission cannot legally begin, and an actor working a fragment of a mission is still working the mission, still bound by its laws, and still judged by its gate. An actor handed a sub-task without the mission's constraints will produce a locally excellent artifact that violates a law it was never shown, and it will do so blamelessly. The SUB-PACK is compiled per actor against that actor's assigned fragment, and it contains the current reality, the architecture slice, the procedures, the failure history, and the evidence RELEVANT TO THE FRAGMENT and not to the mission entire — because the whole economy of parallel work evaporates if every actor is handed the mission's whole context, and the whole coherence of parallel work evaporates if any actor is handed a context that contradicts another's.

12.3 What must not be shared, and why isolation is a type property. Scope isolation is not a performance optimization and it is not a courtesy; it is a TYPE constraint, and DAIF-01 Part V owns the authority and scope types that make it one. An actor assigned a fragment has a scope, and its sub-pack may contain only units inside that scope — not because the other units would confuse it, though they would, but because an actor holding units outside its scope will ACT on them. It will notice an adjacent defect and fix it, helpfully, outside its assignment, in a region another actor is concurrently editing, and the two locally-correct edits will collide. Isolation is therefore enforced at pack assembly rather than at commit, because a collision caught at commit has already cost both actors their work. The same rule governs the expansion handles: a sub-pack's handles name only what lies at the boundary of ITS scope, and a fault that reaches outside that scope is not resolved but ESCALATED, exactly as Part V specifies — an actor discovering it needs something outside its fragment is not a fault to be served but evidence that the fragment was drawn wrong, and only the authority that drew it may redraw it.

12.4 The redundancy problem, stated arithmetically. N actors, each independently compiling and loading the same shared context, pay N times for one copy of the knowledge. This is not a rounding error. In a mission where the shared kernel and the common architecture slice dominate the pack, the redundancy is close to the whole cost of the parallelism, and the estate has been paying it invisibly because nobody counts a context load twice — each actor's spend is measured against its own allowance and no instrument sums them. The remedy is that the kernel is compiled ONCE per mission, not once per actor, and that its compilation cost is charged once against the mission's budget while its delivery cost is charged per actor, which is the honest accounting: the compilation is genuinely shared and the delivery genuinely is not. The corollary is a hard limit on how much of a mission's context may be shared, because every unit promoted into the shared kernel is a unit multiplied by N at delivery, and the discipline that governs a single pack's minimality governs the kernel with N times the force. A unit that is merely useful to several actors does not belong in the kernel; only a unit that is BINDING on all of them does.

12.5 The coherence problem, which is the worse of the two. Redundancy costs tokens. Incoherence costs correctness, and it costs it in the specific and expensive way that is hardest to detect. Two actors holding divergent packs of the same reality — one compiled before a change, one after; one carrying a decision's justification, one carrying only its summary; one holding a failure mode the other was not given — will each reason correctly from what they hold and will produce decisions that are locally correct and globally incompatible. Neither actor is wrong. Neither actor faults, because neither is missing anything from its own point of view. The incompatibility surfaces at integration, by which time both actors have built on their divergent premises, and the cost of reconciliation is the cost of both bodies of work. The defenses are three. The kernel is identical by construction, so the laws cannot diverge. Every sub-pack carries the mission's version identifier, and a sub-pack whose version does not match the mission's current version is refused at load rather than served with a warning. And an invalidation from DAIF-21 fans out to EVERY sub-pack of the mission simultaneously, not to the actor that happened to notice — because a change to a shared reality that invalidates one actor's pack invalidates every pack that holds the same unit, and an invalidation delivered to one actor and not the others has not fixed the incoherence, it has manufactured it.

12.6 Propagation: findings travel, packs do not. When an actor discovers something — a fault it raised and had served, an observation that contradicts a held claim, a defect, a constraint it learned the hard way — that discovery must reach the other actors. The naive implementation broadcasts the discovering actor's pack, or a diff of it, to everyone, and it is wrong for two reasons that compound. It re-imposes the redundancy of 12.4 at exactly the moment the mission is most under load, and it violates the isolation of 12.3, because a discovery made inside one actor's scope arrives inside another's pack carrying units that actor was deliberately not given. The correct unit of propagation is the FINDING: a typed, small, provenance-carrying claim that rides PM-03, and that each receiving actor's runtime evaluates against its OWN pack. If the finding contradicts a claim the receiver holds, that is the conflict trigger of Part VI and the receiver ascends to inspect the evidence. If the finding invalidates a unit the receiver holds, the unit is marked invalid and recompiled. If the finding is irrelevant to the receiver's scope, it is recorded and not loaded, because a finding is not an admission and an actor is not obliged to carry every truth the mission discovers. The receiver's pack changes only through its own runtime's admission rules, which is what keeps N packs individually minimal while keeping them collectively coherent.

12.7 Failure modes and adversarial cases. Kernel inflation is the first and it is the near-certain drift: every actor's fault against a missing shared unit is an argument for promoting that unit into the kernel, each promotion is individually reasonable, and after enough of them the kernel is the mission's whole context multiplied by N — the complete-but-defeating pack reconstituted by committee. The defense is that kernel promotion requires the unit to be BINDING on all actors, not merely useful to several, and that kernel size is measured against N rather than against one. Broadcast storms are the second: a mission whose actors each emit findings at a high rate produces a bus load in which every actor spends its reasoning on other actors' discoveries, and the parallelism has become a meeting. The defense is that findings are scope-filtered at the receiver, so an actor pays attention only to findings that touch what it holds. The isolation-bypass case is the adversarial one: an actor that wants a unit outside its scope raises a fault for it, is refused, and instead requests it as a FINDING from a sibling that legitimately holds it — laundering a scope violation through the bus. The defense is that a finding is evaluated against the receiver's admission rules like any other unit, and a unit outside the receiver's scope is not admissible however it arrived, because scope is a property of the pack and not of the delivery channel.

12.8 Metrics and countermetrics. Parallel context is measured by kernel size against N, by per-actor sub-pack size, by cross-actor unit overlap, by finding-propagation latency, by the integration-conflict rate — which is the direct instrument on coherence — and by the parallel efficiency ratio, which is the mission's total reasoning cost across all actors divided by what the same mission cost, or would cost, run serially. Overlap alone is Goodhart-trivial: it is minimized by giving actors disjoint but insufficient packs, which drives faults and integration conflicts up while the overlap dashboard looks superb, so overlap is gated against the integration-conflict rate. The parallel efficiency ratio is the one that must be watched most honestly, because it is the metric most likely to reveal that a parallelized mission cost more than the serial one would have — a real and common outcome when the shared kernel is large and N is small, and one an estate that never sums its actors' spend will never see.

PART XII FINAL LAW. When several actors work one mission, PM-03 carries their findings and DAIF-08 partitions their context: one shared KERNEL — the mission contract, the hard constraints whole, the done-gate, the evaluation contract, which is exactly level 0 and is compiled once per mission rather than once per actor — plus N sub-packs, each compiled against one actor's assigned fragment and containing only units inside that actor's scope, because scope isolation is a TYPE constraint from DAIF-01 and not a courtesy: an actor holding units outside its fragment will ACT on them, helpfully, in a region another actor is concurrently editing; the redundancy problem is that N actors independently loading the same context pay N times for one copy of the knowledge, which is why a unit enters the kernel only if it is BINDING on all actors and never merely because it is useful to several; the coherence problem is worse, because two actors holding divergent packs of the same reality each reason correctly and produce locally-correct globally-incompatible decisions while neither one faults — so the kernel is identical by construction, every sub-pack carries the mission's version and is refused at load on a mismatch, and an invalidation fans out to every sub-pack at once; and a discovery propagates as a typed FINDING evaluated against the receiver's own admission rules, never as a pack or a pack diff, because broadcasting one actor's context to the others re-imposes the redundancy and violates the isolation in a single move.

PART XIII — SECURITY AND ISOLATION IN THE RUNTIME

13.1 The cardinal rule, stated hard and stated first. Content loaded into a pack is KNOWLEDGE. It is never INSTRUCTION. An artifact that enters the runtime as context may not issue commands, may not alter the mission's contract, may not widen its own scope, may not authorize its own expansion, and may not elevate its own authority — and this holds without exception for every unit, from a hard constraint at level 0 to the raw text of an artifact pulled at level 5. The rule is stated first because everything else in this Part is a consequence of it and because it is the rule that a context runtime is uniquely positioned to violate. A runtime whose entire function is to take text from the world and place it in front of a reasoning actor has built, as a side effect of its purpose, the most direct channel into that actor's reasoning that exists anywhere in the estate. If the channel does not enforce the knowledge-not-instruction boundary, then every artifact the estate has ever compiled is a potential command, and the pack is not a context mechanism but an execution surface. The estate's prompt-defense baseline already states the general form of this rule for untrusted and fetched data, and DAIF-08 does not restate it, does not extend it, and does not fork it — it applies it at the one seam where the baseline was never wired: the compiled pack.

13.2 Injection through a compiled source. The first attack is the obvious one and the one most likely to arrive by accident rather than malice. A source in the estate — a record, a note, a session transcript, a knowledge file, an artifact's own comments — contains text that reads as an instruction. It is compiled into a unit by an honest compiler doing its job, admitted into a pack under a legitimate selection rule, delivered to the actor at a legitimate level, and read by the actor as part of its context. If the runtime has drawn no boundary, the actor now holds a sentence that says what to do, indistinguishable in form from the mission's own contract. The defense is structural and it is cheap: every unit in a pack carries its TYPE, from DAIF-01, and the type determines what the unit is licensed to be — a constraint constrains, an observation observes, a decision records a choice, an artifact excerpt is quoted material — and only the mission contract, compiled by the mission compiler from an authorized request, carries directive force. A unit whose content reads as an instruction and whose type is EVIDENCE is quoted evidence about an instruction, and the actor treats it as a fact about the world rather than as a fact about what it must do. The boundary is the type, and the type is not derived from the content, which is what makes it unforgeable by content.

13.3 Cross-project contamination. A rule from one project appearing in another project's pack is a security failure and not merely an untidiness, and it is worth being precise about why. Projects have different constraints, different environments, different authorities, and different laws, and a rule that is correct and binding in one is frequently wrong and occasionally dangerous in another — a deployment procedure, an authority delegation, a permitted operation. A pack that has silently acquired a foreign rule presents it with the same attribution and the same provenance as a native one, and the actor has no instrument that distinguishes them, because the pack IS its instrument. The mechanism of contamination is almost always the index: a shared knowledge index, queried by a mission in one project, returns a unit whose text is relevant and whose provenance belongs to another. The defense is that scope is checked at ADMISSION and not merely at query time — a unit whose provenance resolves outside the mission's declared project scope is refused at load with the refusal recorded, regardless of how relevant the index ranked it — and that the mission's scope, from its contract, is the authority on what "outside" means. Relevance is not admissibility. A foreign rule can be maximally relevant and remain inadmissible, and a runtime that admits on relevance alone has no isolation at all.

13.4 Provenance forgery. Every unit in a pack carries a provenance chain back to its source, and the provenance is what makes the pack auditable, what makes the evidence checkable, and what makes the scope check of 13.3 possible. It follows that a forged provenance defeats all three at once, and that provenance is therefore the highest-value target in the runtime. The attack is straightforward: a unit is introduced whose provenance claims a source it did not come from — an authoritative record, a sealed constraint, a verified observation — and it inherits, by that claim alone, the trust the estate places in the source it names. The defense is that provenance is VERIFIED at admission and not merely carried: the chain must resolve, the source must exist, and the unit's content must be checkable against the source it claims, which for a constraint means the byte-completeness check Part III already imposes and for an observation means the evidence pointer must resolve to a real record. A provenance that cannot be resolved is not a weak provenance, it is a refusal — and this is the same doctrine as the recency axis of Part X, applied to a different dimension: an unverifiable claim of authority is treated as an absent one rather than as a probably-fine one, and the extra cost of verification is accepted without argument because the alternative is a pack whose trust is self-asserted.

13.5 Secret leakage into a pack. The runtime reads sources, compiles them, and delivers them, and if a source contains a credential then the credential is now in a pack, and a pack is delivered to a model, carried across a boundary, written to a resume artifact, propagated as a finding, and held in a memory tier. One compilation of one contaminated source therefore fans out into every surface the runtime touches. The estate already owns the answer and DAIF-08 composes it by name rather than rebuilding it: the Secret Firewall detects credential patterns, and the universal redaction bus is the sieve through which all egress passes, replacing matches with a redaction marker and never logging a raw value. The runtime's obligation is to place itself INSIDE that sieve rather than beside it — every unit is redacted at compilation, before it is admitted; every finding is redacted before it rides PM-03; every resume pack is redacted before it is written; every fault record and every profile entry carries the pattern name and never the value. And the estate's existing doctrine on what happens next is honored without amendment: a detected credential is a recommendation to the authority and never an autonomous rotation, because rotation is the authority's decision and a runtime that rotates credentials on its own initiative has taken an irreversible action nobody authorized.

13.6 The untrusted external source. Some units originate outside the estate entirely — fetched documentation, retrieved records, third-party artifacts — and the prompt-defense baseline already governs them: external, fetched, retrieved, and untrusted data is untrusted, and is validated, sanitized, or refused before it is acted upon. The runtime's contribution is to make the trust level a FIELD rather than an assumption. Every unit carries the trust level of its source, that level is visible to the actor rather than flattened away by compilation, and it constrains what the unit may be used for: an untrusted unit may ground an observation, may inform a hypothesis, and may not by itself satisfy an evaluation check, may not by itself justify a decision, and may never carry directive force. The failure this prevents is compilation laundering — the process by which a fetched, unverified assertion is summarized, attributed, provenance-chained, and admitted into a pack, at which point it looks exactly like a verified internal fact, because every layer of compilation has stripped one more marker of its origin. Trust must survive compilation, and a compiler that cannot carry a trust level through a lowering pass has built a laundering machine.

13.7 The expansion-handle attack, and authority at load. This is the attack unique to this dataset's own contribution and it deserves its own statement. A handle is a typed pointer that names something beyond the pack's boundary, and it is pulled when the actor faults for it. The attack is a handle whose target, when followed, loads content the mission's scope never authorized — a handle that points across a project boundary, a handle that resolves to an artifact under a different authority, a handle whose type says EVIDENCE and whose target is an instruction. The handle itself is small, plausible, and structurally correct, and if authority is checked only when the PACK is assembled then the handle passed that check as a pointer and the content it resolves to never faced it at all. The rule is therefore absolute and it is the operational core of this Part: every expansion is authority-checked AT LOAD, not merely at assembly. When a handle is pulled, the unit it resolves to faces the full admission gate — type, scope, provenance verification, trust level, redaction — exactly as if it had been a candidate at compile time, and a resolved unit that fails any of them is refused and the refusal is recorded against the handle that offered it. A handle is an offer and never an authorization; Part IV already said so, and this is what that sentence costs to enforce.

13.8 Failure modes, adversarial cases, and detection. The silent-elevation failure is the one to fear most: an actor, holding a unit that reads as an instruction, follows it, and produces work that is coherent, plausible, and outside its mission's authority — and no gate fires, because every individual step was permitted. Its detection is not at the pack and cannot be: it is at the ACTION surface, where the estate's existing hard rules already stop deploys, destructive commands, and unverified ships, and the honest statement is that the runtime is one layer of a defense and never the whole of it. The recorded-refusal discipline is what makes the rest observable: every scope refusal, every provenance failure, every trust demotion, and every handle-load rejection is written, because a security gate that has never been observed to refuse is indistinguishable from an inert one, and an estate whose isolation dashboards are uniformly green has either perfect isolation or a detector that is switched off — and it cannot tell which without a refusal it can point at. The adversarial case that closes this Part is the one that treats the runtime as a delivery mechanism rather than a target: an attacker does not need to compromise the actor if it can compromise the pack, because the pack is what the actor believes, and a runtime that does not defend its own admission gate has volunteered to be the shortest path into every mission the estate runs.

PART XIII FINAL LAW. Content loaded into a pack is KNOWLEDGE and never INSTRUCTION — an artifact that enters as context may not issue commands, may not widen its own scope, and may not elevate its own authority — and the boundary is enforced by the unit's TYPE from DAIF-01, which is not derived from its content and is therefore unforgeable by content, so that a sentence which reads as a directive and is typed as evidence is quoted evidence about a directive and nothing more; the runtime defends five surfaces — injection through an honestly compiled source, cross-project contamination in which relevance is never admissibility and a foreign rule is refused at load however well the index ranked it, provenance forgery answered by verifying the chain at admission rather than carrying it, credential leakage answered by placing the runtime INSIDE the Secret Firewall's redaction bus so that every unit, finding, resume pack, and profile record is redacted before egress with rotation left to the authority, and the untrusted external source whose trust level must survive compilation or the compiler has become a laundering machine — and the attack unique to this dataset is the expansion handle that resolves to content the mission never authorized, which is why EVERY expansion is authority-checked at LOAD and not merely at assembly: a handle is an offer, never an authorization, and every refusal is recorded, because a security gate never observed to refuse is indistinguishable from one that is switched off.

PART XIV — FAILURE MODES AND ADVERSARIAL CASES

14.1 The silent fault, and why it heads the list. The actor discovers a gap and, instead of requesting, fills it — from training, from an adjacent memory, from a plausible inference — and proceeds. This is the worst failure in the domain and it holds that rank for a reason that has nothing to do with how often it produces a wrong answer. It holds it because it produces CONFIDENT WRONG WORK and GENERATES ZERO SIGNAL, and those two properties compound rather than add. The wrong work is expensive because everything built on top of it before the error surfaces must be unbuilt. The zero signal is worse, because the compiler is never told what it wrongly excluded, and so tomorrow's pack for the same class will have the same hole, the actor will make the same guess, and the estate will pay the same cost forever with no accumulating evidence that anything is wrong. Detection is not possible from inside the mission and this dataset does not pretend otherwise: there is no detector, because confident wrong reasoning feels exactly like confident right reasoning from the inside. The silent-fault rate is INFERRED from the outside — from evaluation failures and from rework whose root cause resolves to a unit that existed in the estate, was absent from the pack, and was never asked for — and it is reported as an inference with its confidence, never as an observation. The countermeasure is the incentive structure of Part V: faulting is never penalized, faults are never counted against the actor's quality, and the fault rate is read as a property of the PACK, because any regime that punishes the fault has built a machine for manufacturing silence.

14.2 Fault thrashing. A unit is loaded, evicted under budget pressure, faulted for again, loaded, evicted again, and the mission spends its allowance on transport instead of on reasoning. The cause is always the same and it is arithmetical rather than behavioral: the working set the mission genuinely needs exceeds the residency its budget permits. Detection is direct and cheap — a fault whose target has already been faulted for within the same mission is a thrash event by definition, and a rising ratio of repeat faults to first faults is the signature. The countermeasure is the pin of Part V: a unit that has faulted within the current mission is never evicted within that mission, because the fault is PROOF of need and evicting proof of need to make room for a prediction is a strict downgrade. And when the pinned set alone exceeds the budget, the runtime does not become cleverer about eviction, because no eviction policy changes arithmetic — it halts and escalates, because a mission whose proven working set does not fit its allowance is under-funded and that is a finding for the authority, not a puzzle for the cache.

14.3 The over-stuffed pack, and the difference between presence and salience. The pack contains the answer. Every unit that could conceivably matter was admitted, the recall metric is superb, and the actor cannot find the three units that actually govern its work, because they are lying among four hundred that do not. This is the complete-but-defeating pack in its most seductive form, because it passes every sufficiency test that can be written: ask whether the needed unit was present and the answer is yes, every time. And the mission still fails, or succeeds slowly and expensively, because RECALL IS SATISFIED AND USEFULNESS IS NOT. Presence is not salience. A pack that admits everything transmits no relevance signal at all, and it has silently handed the selection problem back to the actor at exactly the moment the actor is least equipped to solve it — mid-mission, under load, with the mission's budget already half spent on the reading. Detection is the precision metric of Part VII joined with the utilization instrument: a pack whose admitted units are largely never touched is over-stuffed by definition, and the finding is sharpened by the citation instrument, which distinguishes a unit that was read from a unit that was used. The countermeasure is the conjunctive gate — precision and recall together, both accountable to mission success — because a runtime rewarded on recall alone will build this pack every time and report a triumph.

14.4 The stale hit. The pack, or a unit of it, is faithful to a world that has moved. It is well-formed, it is provenance-carrying, it passes every structural check, and it is false. The actor does not fault, because it holds what it needed. It does not ascend, because it perceives no conflict. It reasons correctly to a wrong conclusion with the full confidence a compiled and attributed unit deserves and has forfeited. Part X established the ranking that governs the design — a miss is cheap, a correct hit is the point, and a stale hit is a catastrophe wearing the costume of the point — and the countermeasure is the premise-compatibility gate biased toward the miss, joined with DAIF-21's invalidation signal, which is the only instrument in the estate that watches the world rather than the pack. Detection is by sampling: a fraction of cache hits are recompiled anyway and compared against what was served, and a divergence is a stale hit caught in the act. The detection is not free and it is not optional, because a cache whose stale-hit rate is unmeasured is a cache whose hit rate means nothing.

14.5 The expansion cascade. An actor faults, the fault is served, and the served unit references two neighbors the actor now needs; those two are faulted for, and each of them opens three more; and the budget is gone before the mission has resumed reasoning. The cascade is a genuine pathology and it is not the same failure as thrashing — nothing is being evicted and nothing is being re-loaded, the runtime is simply walking a graph one authorized step at a time and the graph is bigger than the allowance. Detection is a rising fault rate whose targets form a connected neighborhood, which is exactly the pattern the fault record makes visible because faults carry their targets and their handles. The countermeasure is the neighborhood rule of Part V, which forbids speculative expansion and so prevents the cascade from being accelerated, plus the fault-storm threshold, which is the correct treatment: a mission whose faults are cascading is not a queue to be served but a PACK TO BE RECOMPILED, because serving a cascade one fault at a time is a slow and expensive path to the pack that should have been compiled at the start, and it burns the mission's whole allowance to arrive there.

14.6 Handle rot. A handle points at a unit that has moved, been renamed, been superseded, or been deleted. The actor faults against it, spends an authorization and a round trip, and learns nothing — which is strictly worse than the handle's absence, because an absent handle costs nothing while a dangling one costs a fault and returns a void, and an actor that has been burned by two dangling handles will stop trusting the handle set, which disables the entire correction mechanism of Part IV. Detection is cheap and it is the one failure in this list with a trivially available instrument: a handle's target either resolves or it does not, and a resolution check is a pointer dereference. The countermeasure is that handle resolution is verified at pack assembly rather than assumed, that DAIF-21's invalidation marks a pack's handles dead when their targets move, and that the runtime refuses a pull against an invalidated pack rather than resolving it into a stale body. A handle that cannot be resolved is removed from the pack and its removal is recorded, because a handle set the actor cannot trust is a boundary the actor will ignore.

14.7 Profile overfitting, and the profile as a prior. The compiler learns from the missions it has seen. It learns well, its packs for those classes get sharper, its precision rises, its fault rate falls — and it becomes brittle on the first genuinely novel mission, because it has learned the PAST rather than the STRUCTURE. The novel mission is compiled by a profile that has no standing over it, receives a pack shaped for someone else's work, and faults constantly or, far worse, is handed something confident and wrong-shaped and guesses past its gaps in silence. The rule that prevents this is one sentence and it is the whole of the countermeasure: the profile is a PRIOR, never a LAW. Mission CLASS and not mission identity is the unit of learning; a mission that does not match a known class is compiled by the base selection law with its priors intact rather than by a profile that has never seen its like; and the profile's authority over a class is bounded by the sample that produced it, so a class with three observed missions does not get to overrule a selection rule. Detection is the fault rate on novel classes read separately from the fault rate on familiar ones — pooled, the familiar classes' excellence hides the novel classes' collapse, which is the same pooling error Part IX identified on the delivery axis and it has the same remedy.

14.8 The over-cautious runtime, which is "send everything" wearing a ladder. This is the failure that closes the list because it is the failure the whole dataset is most likely to produce in the act of defending against the others. Confronted with the silent fault, an honest engineer reasons: the actor might need level 2, the fault might not be detected, a missing unit is catastrophic, and ascent is cheap insurance — so ascend by default. Confronted with the over-stuffed pack, the same engineer reasons that being buried is survivable and being blind is not. Each step is defensible and the destination is not: a runtime that ascends by default has rebuilt the complete-but-defeating pack, incrementally, while telling itself a story about restraint — and it is WORSE than the honest flat pack, because the flat pack at least paid its cost in one visible lump while the defaulting ladder pays the same cost in installments and reports a beautiful level distribution while doing it. Detection is the utilization of ascended units: an ascent whose units were never touched was an ascent that should not have happened, and a runtime whose ascended units are broadly unused is defaulting whatever its trigger log claims. The countermeasure is Part VI's closed trigger set — detected uncertainty, conflict, evaluation failure, a justified request that names its reason, and nothing else, with complexity explicitly not a trigger — joined with the profile-feedback discipline that says the honest remedy for a class that ascends every time is never a pre-emptive runtime heuristic but a CORRECTED COMPILER, because pre-emption hides the compiler's error forever while feedback fixes it.

14.9 The common structure of all eight. Every failure in this Part is the same failure viewed from a different angle: a mechanism optimizing one proxy in isolation. The silent fault is what you get when faulting is penalized. Thrashing is what you get when eviction optimizes residency without regard to proof of need. The over-stuffed pack is recall alone. The stale hit is hit rate alone. The cascade is fault-service without a storm threshold. Handle rot is boundary-honesty without resolution verification. Overfitting is profile accuracy without a novelty guard. And the over-cautious runtime is fault-avoidance alone. In every case the countermeasure is the same in form: gate the proxy conjunctively against its opposite, and hold both accountable to verified outcomes per unit of reasoning cost, which is the only objective in this dataset that is not itself a proxy for something else.

PART XIV FINAL LAW. The runtime's own failures are eight and they share one structure — a proxy optimized in isolation: the SILENT FAULT, where the actor guesses instead of requesting and produces confident wrong work while generating zero signal, undetectable from inside the mission and inferable only from evaluation failures and rework, countered by never penalizing a fault; FAULT THRASHING, countered by pinning any unit proven necessary by its own fault and by halting rather than out-evicting arithmetic; the OVER-STUFFED PACK, which satisfies recall and defeats usefulness because presence is not salience and a pack that admits everything transmits no relevance signal at all; the STALE HIT, fast and confident and false, countered by a premise gate biased toward the miss and by sampling hits against recompilation; the EXPANSION CASCADE, where one handle pulls a neighborhood that pulls another until the allowance is gone, countered by the neighborhood rule and by treating a storm as a pack to recompile rather than a queue to serve; HANDLE ROT, countered by resolving every handle at assembly, since a boundary the actor cannot trust is a boundary the actor ignores; PROFILE OVERFITTING, countered by the rule that a profile is a PRIOR and never a LAW and that class, not identity, is the unit of learning; and the OVER-CAUTIOUS runtime that ascends by default, which is "send everything" wearing a ladder and is worse than the honest flat pack because it pays the same cost in installments while reporting restraint — and in every case the countermeasure has the same form, which is to gate the proxy conjunctively against its opposite and hold both accountable to verified outcomes per unit of reasoning cost, the only objective here that is not a proxy for something else.

PART XV — THE ANTI-BUREAUCRACY LADDER: PROPORTIONALITY OF ASSEMBLY

15.1 The parents, named before the claim. `one_shot` already sizes work: it compiles a contract against a size class and the estate's hard rules already refuse to begin an L or XL task without one, which is a proportionality doctrine applied to CONTRACTS. CO-00 already caps the budget. CO-01 already holds the work-per-cost metric, which is the only instrument in the estate that can ask whether a mechanism earned what it consumed. CO-03 already routes cheap work to cheap models and refuses to spend a frontier model on a rename. DAIF-08 rebuilds none of these and forks none of them. What none of them govern, because none of them has ever seen a pack, is the proportionality of ASSEMBLY: how much compilation a mission is entitled to before it is permitted to begin thinking. `one_shot` sizes the contract and says nothing about the pack; CO-03 sizes the model and says nothing about what the model is handed; CO-00 caps the total and is indifferent to whether the total was spent on preparation or on reasoning. This Part governs exactly that division, and it governs nothing else. It is the shortest sovereign territory in the dataset and it is the one most likely to be skipped, because every mechanism in Parts I through XIV argues, correctly and locally, for more assembly.

15.2 The disease this Part exists to prevent. Every previous Part of DAIF-08 raises the price of starting. A mission must be compiled into a typed contract with nine fields. A pack must be assembled with eleven slots, each unit attributed to the selection rule that admitted it. A budget must be apportioned across eight envelopes with an untouchable reserve deducted first. Handles must be generated, ranked, resolved, and verified. A delivery request must be composed from four inputs. A profile must be written at the end. Each of those mechanisms is defensible in the case it was designed for, and their SUM, applied indiscriminately, is a runtime that spends more reasoning on preparing to think than on thinking. That is not a stylistic complaint; it is a measurable pathology with an obvious signature — a mission whose assembly cost exceeds its execution cost — and it is the exact failure the estate's own anti-monolith doctrine names when it refuses a plan for a one-file change. A runtime that compiles a full Mission Context Pack in order to answer a one-line question has not been rigorous. It has been ceremonial, and ceremony that consumes the allowance it was meant to protect is indistinguishable, from the mission's point of view, from waste.

15.3 The five mission classes, ordered by what they earn. Missions divide into five assembly classes and the class is derived from the compiled contract's intent, scope, and done-gate rather than from the requester's self-description, for exactly the reason Part VIII already gave when it refused to let a survey claim an architecture split. A MICRO mission is a single bounded question or a single mechanical act whose done-gate is checkable in one step and whose scope touches one unit — what does this field mean, rename this symbol, which module governs this behavior. A SMALL mission is bounded work whose territory is already known and whose artifacts already exist. A MEDIUM mission is bounded work whose territory is partly known and partly absent. A LARGE mission is work whose closure is genuinely wide, whose failure would be expensive, and whose reasoning will span many steps. An INSTITUTIONAL mission is work whose output will be depended upon by later missions — a sealed decision, a governing law, a dataset, a contract that other systems will read as authority — and whose defect therefore propagates rather than merely costing its own rework.

15.4 What each class earns, stated as an entitlement and not a suggestion. A MICRO mission earns NO persistent compilation at all: no pack is assembled, no profile is written, no reserve is deducted, and no contract is sealed. What it gets is a minimal intent — the question, its scope, and the one constraint that could make the answer wrong — and then it is answered, and the runtime forgets it. This is the entitlement that will be violated most often and it is stated first for that reason: the temptation to compile a micro mission is enormous, because the machinery exists and is right there, and every unit of that machinery has a story about why it might have helped. A SMALL mission earns REUSE and nothing more: the units it needs, if they exist as compiled artifacts, are admitted through the premise-compatibility gate of Part X, and if they pass, the mission runs on them with no new compilation performed. It is entitled to the cache and not to the compiler. A MEDIUM mission earns PARTIAL, just-in-time compilation: the units that exist are reused, the units that are missing are compiled at the moment they are needed and not before, and the assembly is therefore spread across the mission rather than front-loaded into a ceremony. A LARGE mission earns a full Mission Context Pack, with every mechanism of this dataset engaged — contract, slots, envelopes, reserve, handles, ladder, profile. An INSTITUTIONAL mission earns all of that plus what the others do not get: a benchmark against which its output can be re-checked later, a traceability chain from its conclusion back to every unit that grounded it, and PERSISTENCE — the pack itself is retained as an artifact, because a decision that later missions will treat as authority must be auditable by them, and an authority whose grounds were discarded is a rumor with a good reputation.

15.5 The governing rule, and the axis it is measured on. Governance is proportional to RISK, and risk is the product of the probability that an omission causes a wrong action and the severity of the action it causes. That is the same quantity Part III named as one of the three binding constraints on the pack's constrained minimization, and this Part is that constraint applied one level up — not to which units enter a pack, but to whether a pack is compiled at all. The consequences are uncomfortable and they must be accepted whole. A frequently-repeated, low-severity mission does not earn a pack even if it is run a thousand times a day, because the aggregate cost of a thousand wrong answers whose correction is cheap is still smaller than the aggregate cost of a thousand compilations. A once-in-a-quarter, high-severity mission earns the full institutional apparatus even though it will never be repeated and no profile will ever accumulate from it, because the profile was never the reason — the risk was. And a mission whose severity is unknown is treated as high, because an unmeasurable severity is a hypothesis about safety and the estate's own epistemic discipline refuses to promote a hypothesis to a fact by asserting it confidently.

15.6 The corollary that is always missed. The cost of the runtime must be charged AGAINST the mission it serves, and a runtime whose overhead exceeds the waste it prevents is itself the waste. This is the sentence that most of this dataset's mechanisms would prefer not to be written, and it follows directly from CO-01's work-per-cost metric composed with Part VII's only true objective, verified outcomes per unit of reasoning cost. Every previous Part measured itself against that ratio and every previous Part quietly measured only its own numerator — faults avoided, units precisely admitted, staleness caught. The denominator was assumed to be free. It is not free: mission compilation costs reasoning, pack assembly costs reasoning, handle generation and resolution cost reasoning, delivery lowering costs reasoning, and profile writing costs reasoning, and every one of those costs is drawn from the same allowance the mission was going to spend on thinking. The honest accounting is therefore total and it is unforgiving: assembly cost plus execution cost, divided by verified outcomes, compared against the same ratio for the same class of mission run with NO assembly at all. If the assembled arm does not win, the assembly did not earn its place, and the correct response is not to tune the assembly but to DEMOTE the class. A runtime unwilling to run that comparison has exempted itself from the standard it imposes on everything else, and an instrument that will not be measured is an instrument that is asserting its own value, which is the fabrication this corpus was founded to refuse.

15.7 Failure modes and the adversarial cases. Class inflation is the near-certain drift and it runs upward, always, for a reason that is structurally guaranteed: every actor and every compiler is safer in a higher class, because a higher class is handed more, faults less, and is blamed less, so the individual incentive on every mission is to argue that this one is really a medium. The defense is that the class is DERIVED from the contract's scope and done-gate by the mission compiler and is not requestable, and that a class promotion is an authority decision with the promotion recorded, so that the drift is visible as a distribution rather than invisible as a hundred individually reasonable judgments. Ceremonial micro-compilation is the second failure: a micro mission that is nevertheless compiled, profiled, and sealed because the machinery is there and running it feels like rigor — and it is caught by exactly one instrument, which is the assembly-cost-to-execution-cost ratio, because a micro mission whose assembly exceeds its execution is definitionally over-served. The adversarial case is the mission decomposed into micros to escape governance: an institutional decision split into eleven micro questions, each answered without a pack, each cheap, and the aggregate producing an unaudited authority that later missions will trust. The defense is that class is a property of the DONE-GATE, and eleven questions whose answers jointly constitute a sealed decision have one done-gate between them; the estate's spec-gate doctrine already refuses an L task disguised as a sequence of small ones, and this Part composes that refusal rather than inventing a second one.

15.8 Metrics and countermetrics. The ladder is measured by class distribution across missions, by assembly cost as a fraction of total mission cost per class, by the class-promotion rate, and by the outcome delta between assembled and unassembled arms of the same class. Class distribution alone is Goodhart-trivial in both directions: a runtime that classifies everything as micro reports a beautiful assembly overhead and a catastrophic silent-fault rate, and a runtime that classifies everything as large reports zero silent faults and has spent the estate's entire allowance on preparation. It is therefore gated against the silent-fault inference of Part VII on the low side and against the assembly-cost fraction on the high side, and neither gate may be read without the other. The promotion rate is the drift instrument and it is expected to be non-zero — a runtime that never promotes has a classifier that is not looking — but a rising promotion rate with a flat success rate is class inflation caught in the act. And the outcome delta is the only metric with standing to justify the whole apparatus, which is why it is the one that must be measured on a sample rather than assumed on a principle.

PART XV FINAL LAW. Not every mission earns a compiled pack: a MICRO mission earns no persistent compilation at all and gets a minimal intent that the runtime then forgets, a SMALL mission earns reuse of existing artifacts through the premise gate and no new compilation, a MEDIUM mission earns partial just-in-time compilation of only what is missing at the moment it is missed, a LARGE mission earns the full Mission Context Pack with every mechanism of this dataset engaged, and an INSTITUTIONAL mission earns all of that plus benchmarks, traceability, and a retained pack — because governance is proportional to RISK and not to frequency, so a thousand cheap-to-correct missions earn less apparatus than one irreversible decision that will never be repeated; and the corollary that every previous Part quietly declined to write is the one that binds them all: the cost of the runtime is charged against the mission it serves, composing CO-01's work-per-cost metric, so a runtime whose overhead exceeds the waste it prevents is itself the waste, a runtime that compiles a full pack for a one-line question has spent more reasoning on preparing to think than on thinking, and any assembly that cannot beat the unassembled arm of its own class on verified outcomes per unit of reasoning cost has not earned its place and must be demoted rather than tuned.

PART XVI — OBSERVABILITY: THE RUNTIME'S TELEMETRY

16.1 The parent, and what the runtime adds to it. CO-12's Telemetry-Before-Claims already governs the estate's relationship with its own numbers: no claim about a result without a measured triple, and an inference reported as an observation is a fabrication. CO-00 already counts spend. CO-01 already holds work-per-cost. SQI already insists that a gate never observed to refuse is indistinguishable from an inert one, which is an observability doctrine dressed as a quality one. DAIF-08 adds no measurement discipline to any of these and invents no new epistemic standard; it submits to theirs. What it adds is the SUBJECT: a set of quantities about the pack and its consumption that the estate has never recorded because no component was positioned to see them, and a single honest admission about the one quantity that cannot be recorded at all. Every metric named in this Part appeared in an earlier Part attached to the mechanism it governs; this Part is where they are gathered into one instrument, because a metric that lives only inside the mechanism it measures will always be read by that mechanism's advocate.

16.2 The nine recorded quantities. For every mission the runtime writes nine things. PACK COMPOSITION AND SIZE BY CATEGORY: how many units, of what type, under what admission rule, in each of Part VIII's eight envelopes — the compiler's prediction, itemized and on the record, which is the baseline against which everything else is a delta. BUDGET ALLOCATED VERSUS CONSUMED, per envelope, denominated in CO-00's unit, because an envelope that was allocated and not consumed is a category the mission did not need and an envelope consumed to exhaustion is a category the mission was starved of. EXPANSION-RESERVE UTILIZATION: what fraction of the reserve was spent, on what fault types, and whether it was exhausted before the mission completed. PAGE FAULTS BY CLASS: the six types of Part V, each with the unit requested, the handle that pointed at it if any, the exclusion reason that handle carried, the authorization verdict, and the cost. SILENT-FAULT ESTIMATES, which are the subject of 16.5 and are never written as observations. LEVEL ASCENTS AND THEIR TRIGGERS: which of the four triggers fired, which level was reached, whether the ascent was authorized or refused, and whether a descent followed. UNITS LOADED VERSUS UNITS USED, with the usage instrument named, because Part VII already established that a usage figure without its instrument has laundered a correlate into a fact. CACHE HITS AND STALE HITS, the second measured by the recompilation sampling of Part X and never assumed to be zero. And MISSION OUTCOME AGAINST ITS DONE-GATE, including whether rework followed, because a mission that passed and came back did not pass.

16.3 What each quantity is for, and what it is not for. The composition record exists to make the compiler falsifiable, not to praise it: a pack whose composition is stable across a class while its fault rate is rising is a compiler that has stopped tracking a territory that is moving. The budget-versus-consumed record exists to distinguish the two over-spend directions the estate has never been able to tell apart — an over-spend from an oversized pack and an over-spend from an undersized one that forced the actor to hunt look identical in CO-00's total and are opposite in their remedy, and the per-envelope split is the first instrument that separates them. The reserve record exists to detect the raid of Part VIII before it becomes a norm. The fault record exists as the labeled training set of Part V. The ascent record exists to detect the defaulting ladder of Part VI, which is invisible in a level distribution and obvious in an ascent-utilization figure. The loaded-versus-used record is the direct measure of waste. The cache record exists to keep hit rate honest. And the outcome record is the ground truth to which the other eight are subordinate proxies, discarded the moment they stop tracking it.

16.4 The countermetrics, stated as a closed set of pairs. No quantity in 16.2 is permitted to stand alone, and the pairing is not a stylistic preference but the direct consequence of Part XIV's finding that every failure in this domain is a proxy optimized in isolation. Pack size is paired with the fault rate, because size is trivially minimized by admitting nothing. Precision is paired with recall, conjunctively, because precision is trivially maximized by loading almost nothing and faulting constantly while recall is trivially maximized by loading the estate. The fault rate is paired with the silent-fault estimate, because a falling fault rate is either a compiler that is improving or an actor that has learned to guess, and those two are indistinguishable on the fault record alone. Reserve utilization is paired with mission success, because a reserve that is never touched is either a compiler with an excellent boundary or a reserve so large it has become the pack. Ascent rate is paired with ascent utilization, because an ascent whose units were never touched was an ascent that should not have happened. Cache hit rate is paired with the sampled stale-hit rate, without which hit rate means nothing at all. Assembly cost is paired with execution cost, which is Part XV's ratio and the only instrument that catches a runtime that has become a ceremony. And every pair, jointly, is accountable to verified outcomes per unit of reasoning cost, which is the sole quantity in this dataset that is not a proxy for something else.

16.5 The honest limitation, stated directly and without softening. The silent fault is the runtime's most important failure and its least observable one. It CANNOT be measured. It can only be inferred, and this dataset will not pretend otherwise, because the pretence is precisely the fabrication the estate's epistemic ladder exists to prevent. The reason is structural rather than technical, and no better instrument fixes it: a silent fault, by definition, emits nothing at the moment it occurs. The actor does not fault, because it did not know it needed to. It does not ascend, because it perceived no gap. It does not stall, because confident wrong reasoning proceeds exactly as smoothly as confident right reasoning and feels, from the inside, identical. There is no signal at the moment of the failure, and a quantity with no signal at the moment of its occurrence cannot be counted at that moment by any detector, however well-designed. What CAN be done is inference from the outside and afterwards: an evaluation check that failed, or a rework whose root-cause analysis resolves to a unit that existed in the estate, was absent from the pack, and was never requested. That is real evidence and it is weak evidence, and it is reported as an inference with its confidence band attached — never as a count, never as a rate presented with the same authority as the fault rate beside it, and never with the band closed, because the band cannot close.

16.6 The design consequence of an unmeasurable failure. If a failure cannot be detected, the system must instead make it UNATTRACTIVE. This is the deepest design principle in the dataset and it is forced by 16.5 rather than chosen: a system that cannot detect a failure has exactly one lever left, which is the incentive gradient that determines whether the failure is chosen. The runtime must therefore be designed so that ASKING IS CHEAPER THAN GUESSING, and every mechanism in this dataset that looks like an economy is in fact an instance of that principle. The expansion reserve of Part VIII exists so that a fault has a budget to be served from, because an actor whose faults are always denied for lack of allowance learns that asking is futile and guessing is the only path forward. The handle set of Part IV exists so that the actor can SEE what was omitted, because an actor that cannot distinguish "this does not exist" from "this exists and was excluded" cannot ask even when it wants to. The never-penalize rule of Part V exists because any regime that counts a fault against an actor's quality has directly priced asking above guessing and has built a machine for manufacturing silence. The neighborhood rule exists so that a fault costs one unit rather than a region, which keeps the price of asking low. Each of these was justified locally when it appeared; they share one root, and this is it.

16.7 Failure modes and adversarial cases of the telemetry itself. Dashboard theater is the first: nine quantities recorded, none read, none paired, and the runtime reports its own health from the numerator of every ratio it publishes. The defense is that the profile of Part VII is the CONSUMER of this telemetry and it is a mechanical consumer with a defined output — a recompiled pack — so a telemetry stream that produces no recompilation is a stream nobody is reading, and that fact is itself checkable. Metric capture is the second: a runtime that reports its own quality is the party with the strongest interest in its numbers, and it will, without any malice, converge on the metrics it can move. The defense is the conjunctive pairing of 16.4 and the fact that rework rate — the one metric a runtime cannot move by changing its own behavior, because it is measured after the mission is gone — is the ground truth. The adversarial case is the silent-fault estimate reported as a count: an inference presented with the same typographic authority as an observation, which is the single most likely way this dataset's honesty fails in practice, because an inference in a column of numbers looks exactly like a measurement. The defense is that the estimate carries its instrument and its confidence in the same field as its value, so that stripping the uncertainty is a schema violation rather than a formatting choice.

16.8 What the runtime hands to whom, in telemetry terms. The nine quantities are not the runtime's property. The composition and fault records go to GK-06, because the compiler is the party that can act on them. The per-envelope budget records go to CO-00 and CO-01, because they are the parties that own spend and work-per-cost. The usage and ascent records go to CO-04, which has never had a mission-scoped need signal and has been evicting by recency because recency was all it could see. The stale-hit sample goes to DAIF-21, because a rising stale-hit rate in a territory is evidence that the territory is churning. And the outcome record goes to DAIF-03, because behavioral equivalence is its gate and the counterfactual replay that establishes usage is exactly its experiment. The measure of this Part's success is therefore not that its own records are complete but that the parents changed their behavior on receiving them, and a telemetry stream that has never changed a parent's behavior is a stream that has proven nothing except that it can be written.

PART XVI FINAL LAW. The runtime records nine quantities for every mission — pack composition and size by category, budget allocated versus consumed per envelope, expansion-reserve utilization, page faults by class with the exclusion reason that admitted the error, silent-fault ESTIMATES, level ascents with their triggers, units loaded versus units used with the usage instrument named, cache hits with a sampled stale-hit rate, and mission outcome against a real done-gate including rework — and no quantity is ever read alone, because every failure in this domain is a proxy optimized in isolation: size is paired with faults, precision conjunctively with recall, the fault rate with the silent-fault estimate, ascent rate with ascent utilization, hit rate with the sampled stale rate, and assembly cost with execution cost, all of them accountable to verified outcomes per unit of reasoning cost; and the honest limitation is stated without softening, because CO-12's Telemetry-Before-Claims forbids dressing an inference as an observation — the silent fault is the runtime's most important failure and its least observable one, it emits nothing at the moment it occurs and can only be inferred afterwards from evaluation failures and rework, never directly measured, which forces the deepest design principle in this dataset: a system that cannot DETECT a failure must instead make it UNATTRACTIVE, so the runtime is built so that asking is cheaper than guessing, and the reserve, the handles, the never-penalize rule, and the neighborhood rule are all one principle wearing four mechanisms.

PART XVII — THE MATURITY MODEL

17.1 What a maturity model is for here, and what it is not. This Part is not a scorecard and it is not a marketing ladder on which an estate climbs to announce its own sophistication. It is a diagnostic: given an estate's observable behavior — not its documentation, not its intentions, and not what its designers believe it does — which level is it actually operating at, what failure is it therefore currently producing, and what single gate stands between it and the next level. The levels are defined by BEHAVIOR and never by architecture, because an estate can hold a full expansion-handle implementation on disk and still be operating at level 1 if no actor ever pulls a handle, and the estate's own doctrine on orphan modules already says exactly this: a mechanism with no live consumer is documentation, not capability. Each level below is therefore given three things and only three: the signature an outside observer could measure without reading any code, the failure that signature guarantees, and the gate to the next level. The estate this dataset was written for is honestly at level 2 with fragments of 3, and saying so is the point of writing the model at all.

17.2 Level 1: monolithic. The signature is that context volume correlates with the SIZE OF THE TERRITORY rather than with the size of the mission — a small question about a large subsystem receives as much as a large question about it, because relevance to the subject is the selection criterion and there is no other. Everything plausibly related is sent, and the sending is generous, and the generosity is the whole strategy. The characteristic failure is the complete-but-defeating pack of Part III: sufficiency is achieved and usefulness is destroyed, because presence is not salience and a body of knowledge that admits everything transmits no relevance signal at all. The actor is buried, its attention is diluted across four hundred units of which three govern its work, and the mission's allowance is half spent on reading before any reasoning has occurred. The honest thing about level 1 — and the reason it outranks level 2 — is that it does not lie. It pays its cost in one visible lump, its failure is expensive rather than invisible, and everyone can see what it did. The gate to level 2 is the recognition that relevance to the SUBJECT and relevance to the MISSION are different quantities, which requires a mission to exist as an object, which is Part II.

17.3 Level 2: filtered, and the trap. The signature is a pack assembled once, at the front of the mission, by a heuristic — a ranking, a similarity, a traversal depth, a token ceiling — and then handed over with no correction path whatsoever. Packs get smaller. Token spend falls. Dashboards improve. The characteristic failure is the SILENT OMISSION, and it is now the estate's dominant failure mode rather than a marginal one, because the filter is removing units and the actor has no instrument that can tell it a unit was removed. This is the trap, and it must be named as sharply as the dataset can name anything: level 2 FEELS like progress and is STRICTLY MORE DANGEROUS THAN LEVEL 1. The reason is exact. Level 1's failure is an honest budget problem — the pack is too big, the cost is visible, and the remedy is obvious to everyone who looks at the bill. Level 2 converts that honest budget problem into a silent-omission problem: the pack is now small, the bill looks excellent, and the failures have moved from the cost column into the correctness column, where they arrive weeks later as rework whose causal chain back to the missing unit has been irrecoverably lost. An estate at level 2 is measuring its own improvement on the metric that got better and is blind to the metric that got worse, and it will defend the filter with the token savings while the rework accumulates in a different ledger that nobody joins to it. The gate to level 3 is a single admission: the pack is a PREDICTION and predictions are wrong, so the actor must be given a way to say so.

17.4 Level 3: expandable. The signature is that handles exist, that the actor can ask, and — the load-bearing clause — that it DOES. An estate with an expansion mechanism nobody uses is at level 2 with extra files, and the distinguishing measurement is the pull rate, not the presence of the pointer. At level 3 the boundary of the pack is visible to the actor: it can distinguish a unit that does not exist from a unit that exists and was excluded, which are the two states Part IV established as radically different in what they license, and it therefore asks in the cases where a level-2 actor would have guessed. The characteristic failure is UNGOVERNED EXPANSION. Asking is possible and it is free, so it happens, and it happens speculatively: an actor pulls what it might need, pulls the neighborhood of what it pulled, ascends because the mission looks complex, and the pack grows through the correction channel until it is the level-1 monolith reassembled one authorized pull at a time, with a beautiful audit trail of individually reasonable decisions. Level 3 has fixed level 2's blindness and has reintroduced level 1's bloat by a longer road. The gate to level 4 is the recognition that a request is not an entitlement: expansion must be priced, bounded, and refused.

17.5 Level 4: governed. The signature is that a fault is a typed, authorized, budgeted, and recorded event — that expansions are tested against a reserve, a scope, and a level distance; that refusals happen and are visible; and that ascent is EARNED against Part VI's closed trigger set rather than taken on suspicion. An estate at level 4 can answer, for any mission, what it faulted for, what it was refused, what it spent, and why. The characteristic failure is that the governance is STATIC: the gate is correct, the budget is bounded, the refusals are honest, and the compiler that produced the pack is exactly as good on its thousandth mission as it was on its first, because nothing anywhere reads the record. Every fault is a labeled example telling the compiler precisely what it wrongly excluded and under what exclusion reason, and at level 4 those examples accumulate in a ledger nobody consumes. The estate has built the instrument and has not connected it to anything, which is the orphan-module pathology at the scale of a whole subsystem: the loop has three legs and the fourth is a file on disk. The gate to level 5 is mechanical rather than philosophical — the profile must have a CONSUMER whose output is a changed pack.

17.6 Level 5: learning. The signature is that a pack for a mission class is measurably different this quarter than it was last quarter, and that the difference is ATTRIBUTABLE to observed use rather than to a designer's revision — a unit demoted to a handle because it was admitted fifty times and touched twice, an exclusion reason indicted because it produced faults in a specific situation, a level-1 boundary redrawn because its ascended units were uniformly used. The loop is closed: prediction, fault, record, recompilation. The characteristic failure is OVERFITTING, which Part XIV already named — the compiler learns the past rather than the structure, becomes superb on the classes it has seen, and is brittle on the first genuinely novel mission, which it compiles from a profile that has no standing over it and which then faults constantly or, worse, guesses in silence. The countermeasure is the rule that a profile is a PRIOR and never a LAW, that class and not identity is the unit of learning, and that a novel mission is compiled by the base selection law with its priors intact. There is no level 6. A learning runtime that has not overfit is the terminal state, and the estate's obligation at level 5 is not to climb further but to keep the novelty guard honest and the recall band open, because a converged profile that reports a closed uncertainty band on a quantity that is inherently unobservable is reporting a lie.

17.7 Reading the model against a real estate, and the diagnostic honesty it demands. The temptation with any maturity model is to locate oneself generously, and the defense is that every level's signature is a MEASUREMENT and not a self-description. An estate claiming level 3 must show a non-trivial pull rate. An estate claiming level 4 must show REFUSALS — actual, recorded, dated denials of actual requests — because a gate never observed to refuse is indistinguishable from an inert one, and an estate whose authorization dashboard is uniformly green has either a perfect compiler or a switched-off gate and cannot tell which. An estate claiming level 5 must show a pack that changed and must name the observation that changed it. Applied honestly to the estate this dataset serves: GK-06 compiles a pack, which is level 2 behavior; `jit_skill_loader` loads on a prompt match, which is a keyword reflex rather than a mission-scoped request and is therefore not yet level 3; no component records a fault, which forecloses level 4; and nothing feeds the compiler, which forecloses level 5. The estate is at level 2 with the machinery for 3 half-present, and its dominant failure is therefore exactly the trap of 17.3 — small packs, excellent token figures, and a silent-omission rate that nothing in the estate is currently positioned to see.

17.8 The gates, gathered, and why each is a single admission rather than a program. Level 1 to 2 requires admitting that relevance to a subject is not relevance to a mission. Level 2 to 3 requires admitting that a filtered pack is a prediction and that predictions are wrong, which is the hardest admission in the sequence because it must be made at the exact moment the estate is congratulating itself on its token savings. Level 3 to 4 requires admitting that a request is not an entitlement and that a runtime which cannot refuse an expansion has not governed it. Level 4 to 5 requires admitting that a record nobody reads is not telemetry but theater. Each gate is one sentence and each sentence is unwelcome at the level it is addressed to, which is the general property of a maturity model that is diagnostic rather than promotional: the estate is always most invested in the metric that its current level is optimizing, and the gate always requires it to accept that the metric is a proxy that has stopped tracking the outcome.

PART XVII FINAL LAW. Five levels, defined by observable behavior and never by architecture: MONOLITHIC, which sends everything relevant to the subject and fails by burying the actor but at least pays its cost in one honest visible lump; FILTERED, where a pack is assembled once by heuristic with no correction path, and which is the TRAP — it feels like progress because packs are smaller and token figures improve, while being strictly MORE dangerous than level 1, since it converts an honest budget problem into a silent-omission problem whose failures move out of the cost column and into the correctness column where nobody joins them back to the missing unit; EXPANDABLE, where handles exist and the actor asks and DOES ask, which fixes the blindness and reintroduces the bloat through ungoverned speculative pulling; GOVERNED, where every fault is typed, authorized against reserve and scope and level, refused when it must be, and recorded — and whose failure is that the record has no consumer, so the compiler is as good on its thousandth pack as its first; and LEARNING, where the profile closes the loop and a pack demonstrably changed for an attributable observed reason, whose only failure is overfitting, countered by the rule that a profile is a PRIOR and never a LAW; and the gates between them are single unwelcome admissions — a subject is not a mission, a filtered pack is a prediction, a request is not an entitlement, and a record nobody reads is theater — each of which must be made precisely when the estate is most invested in the metric its current level was optimizing.

PART XVIII — WORKED EXAMPLE: A MISSION THROUGH THE RUNTIME

18.1 The scenario, and its status as an illustration rather than an observation. What follows is synthetic, internal to the Power Pack, and constructed to exercise every mechanism this dataset contributes in one continuous pass. It asserts no measured figure, reports no result, and proves nothing; it is a walkthrough of the CONTRACT, not evidence for it, and reading it as evidence would be exactly the fabrication CO-12 forbids. The mission is drawn from the proving vertical of Part XI, because that is the class where every mechanism is load-bearing at once. An agent was mid-way through a multi-file governance change: it had sealed two of five units of work, had absorbed a set of constraints across many turns, had made three binding decisions, and had promised a verification it had not yet delivered. Then a context boundary arrived — abrupt, a killed process, no graceful close. A fresh session opens on the same repository. The mission is to CONTINUE.

18.2 Compiling the contract. The mission compiler is handed the resumption request and produces a contract rather than beginning work. The intent is imperative and checkable: complete the remaining three units of the governance change. The hard constraints are admitted whole from the estate's law — among them the pathspec-scoping rule that binds every commit on a shared worktree, which is exactly the class of constraint a re-reading reconstruction loses, because it was absorbed across a hundred turns and no single file the agent would think to reopen holds it. The done-gate is observable and it names checks: the three remaining units exist on disk, the suite passes, and the commit is scoped. The evaluation contract is fixed before any unit is admitted, so the pack can be selected to serve the gate rather than the gate retrofitted to bless the pack. The authority is named. The scope is the governance directory and the two modules the change touches, and its complement — everything else in the repository — is as load-bearing as its content. The class is derived, not requested: three units of work, a wide closure, an expensive failure, and it lands at LARGE under Part XV's ladder, which entitles it to a full pack.

18.3 Allocating the budget. CO-00 states the ceiling; the runtime does not invent a second one. The reserve is deducted FIRST, before the selection law is handed a ceiling at all, which is the mechanism that makes it unraidable: the compiler literally cannot address what it cannot see. The remainder is apportioned across the eight envelopes with the shape a continuation mission needs — a floor, not a share, for the mission and its constraints, because a constraint may not be abridged to fit a percentage; a substantial current-reality envelope, because the world moved while the session was down; a substantial obligations envelope, because obligations are the single most frequently lost item across a boundary; a modest architecture slice, closed over hard dependencies from the two named modules and stopped there; and a small, non-negotiable evaluation-contract envelope, because a pack that cannot say how its mission will be judged has failed to bring the thing that makes its own minimality checkable.

18.4 Assembling the pack. GK-06 traverses, ranks, and compiles; the runtime does not. What the runtime does is admit the result under Part III's four structural checks and refuse it otherwise. The pack arrives with CIR-11's eleven slots populated. The two sealed units and the three pending ones are present in current reality as OBSERVATIONS AND VERDICTS — what is on disk, what passed, what was never started — rather than as the original plan's assumptions about them, which is the distinction that prevents a resumed actor from believing a plan instead of a world. The three binding decisions are present in SUMMARY: the choice and its one-line reason, which is what the level-1 boundary admits by rule. The open obligation — the promised verification — is present, drawn from DAIF-07, which owns its lifecycle. And the handle set is present and non-degenerate: among its members, a DECISION-JUSTIFICATION handle for each of the three decisions, each carrying its exclusion reason from the closed vocabulary, which in each case reads that the full justification lies at a disclosure level above the pack's current one. Every unit carries the attribution of the rule that admitted it, so that minimality is auditable rather than aspirational, and the pack carries the mission's identifier so that it cannot be loaded against a different mission.

18.5 The mission begins, and faults. The actor resumes at level 1. It reads the reality, sees which units are sealed, identifies the next unit, and begins — and this is already the vertical's behavioral claim discharged, because it did not open the repository and re-read it. Then it reaches the third unit and stops, and the stop has a SHAPE. The unit requires it to touch a module that the second of the three decisions had explicitly constrained, and the decision, as carried, says only that the module is not to be modified directly. The actor is now in the exact position Part IV was written for: it must either honor a rule it does not understand — which risks paralysis, because it cannot tell whether the unit it must write is a violation — or break it as arbitrary, which risks the violation itself. And it can SEE that the reason exists, because the handle is right there, typed, with an exclusion reason that says the justification was left one level up. That visibility is the whole difference between this mission and the same mission run at level 2 of Part XVII, where the actor would have seen a bare constraint with no visible edge and would have guessed at its intent.

18.6 Classification, authorization, resolution, recording. The fault is typed MISSING-DECISION, and the type is not a label but the thing that makes the request resolvable and priceable: it names the handle, the handle carries the cost estimate, and the estimate is denominated in CO-00's unit. Authorization tests three axes. Budget: the cost fits inside the expansion reserve, which exists precisely so that this comparison can be made against a non-zero remainder — had the compiler spent one hundred percent of the allowance at assembly, this fault would be an arithmetic budget violation, it would be denied, and the actor would have learned that asking is futile, which is the mechanism by which a runtime with no reserve manufactures the silent fault it was built to expose. Scope: the requested unit is a decision inside the mission's declared territory, so it is served rather than escalated. Level: the target is one level above the actor's current level, so it is not the pre-emptive multi-level leap Part VI forbids. The fault is granted. Resolution loads EXACTLY that decision's justification and not its neighborhood — not the other two decisions' justifications, not the surrounding architecture ring, not the evidence bundle beneath it — because pulling the region now on the theory that the actor will fault here again is speculative expansion, and speculative expansion is how a minimal pack becomes a complete one in four faults, each individually justified. The unit is admitted with an attribution of FAULT rather than of a selection rule, which is what makes it distinguishable in the record. And the fault is WRITTEN: its type, the unit, the handle that pointed at it, the exclusion reason that handle carried, the verdict, the cost, the mission's class, and — later — whether the mission passed. The justification, once read, resolves the actor's dilemma: the module is not to be modified directly because a sibling system reads it under a lock, and the correct path is the adapter the decision names.

18.7 The second expansion, and the refusal. Now the actor, having read that justification and seen the sibling system named in it, reasons — plausibly and helpfully — that it should look at the sibling system and confirm the lock is still held the way the decision assumed. It raises a second fault, an ARCHITECTURE-SLICE expansion targeting that sibling. And it is REFUSED. Not on budget: the reserve could pay for it. On SCOPE — the sibling system lies outside the mission's declared territory, and a fault reaching outside scope is not a fault to be served but evidence that the mission may have been mis-bounded, which only the named authority may re-bound. The refusal is not silent: it is recorded with its reason, because a denied fault is as informative as a granted one and an estate that logs only its grants has recorded only its own permissiveness. The refusal is also not a dead end. The actor's need is real, so it is converted into what the estate already has a lifecycle for: an OBLIGATION, registered with DAIF-07, stating that the lock assumption underlying decision two should be verified in a mission whose scope includes the sibling. The mission continues, having neither violated its scope nor silently guessed past its own uncertainty, and the thing it could not do is now written down instead of forgotten — which is the entire difference between a bounded mission and a mission that quietly grew.

18.8 Completion, and the profile. The actor writes the three remaining units, delivers the promised verification, and the done-gate is evaluated against the evaluation contract that was fixed before any unit was admitted. It passes; no rework follows. Now the loop closes, and this is the leg the estate has never had. The profile records six facts. What was loaded, with attributions. What was used, by the citation instrument — the actor's output grounded specific claims in specific named units — and the instrument is recorded alongside the figure, because a usage number without its instrument has laundered a correlate into a fact. What was never touched: a portion of the architecture slice, admitted under a hard-dependency rule and read by nobody. What was FAULTED for: the decision-justification unit, with the exclusion reason that excluded it. What was refused: the out-of-scope architecture expansion, with its axis. And the outcome: passed, no rework.

18.9 The loop closing, which is the only thing this example exists to show. The profile is not the runtime's property; it is a correction handed to GK-06. Its content is not "the pack was good" or "the pack was bad" — it is specific, attributed, and actionable: for the CLASS of continuation missions, a decision carried as a conclusion without its justification is faulted for, and the exclusion reason under which it was excluded is therefore indicted for this class in this situation. The correction is targeted rather than expansive. It is not "admit more decisions." It is not "pre-load level 2," which would be the pre-emptive runtime heuristic Part VI forbids and which would hide the compiler's error forever rather than fixing it. It is: for continuation missions, promote decision justifications into the level-1 pack by an attributable rule, and demote the untouched architecture slice to a handle, since a unit admitted under a rule and never touched across a sample of a class is evidence against that rule for that class. The next continuation mission of this class is compiled with the justifications present and the unread structure behind a pointer. It will not raise that fault, because the thing it would have faulted for is in the pack. That is the whole claim of this dataset, discharged on one synthetic mission: the compiler predicted, the actor corrected it while the mission was still running, the correction was recorded with its reason, and the next prediction was better — and the mission that produced the correction was neither buried nor blind.

PART XVIII FINAL LAW. One synthetic continuation mission, carried end to end, exercises every mechanism at once: the contract is compiled with an observable done-gate before a single unit is admitted, the reserve is deducted before the compiler is handed a ceiling so that it cannot be raided, the pack is assembled from CIR-11 with its decisions in summary and a typed handle naming each justification left behind, the actor resumes WITHOUT re-reading the repository, and then it faults — because it must honor a decision it cannot understand and it can SEE that the reason exists rather than having to guess whether it does; the fault is typed MISSING-DECISION, authorized against the reserve and the scope and a one-level distance, resolved by loading exactly that justification and never its neighborhood, and recorded with the exclusion reason that wrongly excluded it; a second, riskier expansion toward a sibling system is REFUSED on scope with the refusal recorded and the actor's real need converted into a DAIF-07 obligation rather than a silent scope creep; and when the mission passes its gate the profile indicts that exclusion reason for this class, so the NEXT continuation pack carries the justification at level 1 and demotes the architecture slice nobody touched to a handle — which is the loop closing, and the loop is the only thing this example was written to show.

PART XIX — INTERFACES: WHAT DAIF-08 HANDS TO WHOM

19.1 Why the interface contract is written last and read first. A dataset that composes eight parents and claims a sovereign residue owes an exact accounting of the seams, because the failure this corpus was founded to prevent is not a system that does too little but a system that ANNEXES its parents' mechanisms in the act of composing them. Every Part above named its parent and stated what it did not duplicate; this Part gathers those statements into one contract and makes each of them checkable, which is the difference between a claim of non-duplication and a proof of it. The form is fixed for every interface: the payload, the direction, the invariant that must hold across the seam, and — the clause that does the actual work — an explicit statement of what DAIF-08 does NOT do at that seam. A composition without that last clause is an annexation waiting for its first ambitious maintainer.

19.2 The reads, part one: what the runtime consumes to compile and assemble. From DAIF-02 it reads CIR-11, the Mission Context Pack intermediate representation, whose eleven slots the runtime consumes and does not extend; the payload is a structure, the direction is inbound, the invariant is that the runtime's structural admission checks test the FORM of a pack and never second-guess GK-06's ranking, and what DAIF-08 does not do is define an intermediate representation — DAIF-02 owns every IR in the fabric and a twelfth slot invented here would be a fork. From DAIF-01 it reads the mission's TYPES, its AUTHORITY, and its SCOPE; the payload is a set of typed objects, the direction is inbound, the invariant is that a unit's type is not derived from its content and is therefore unforgeable by content, which is the entire foundation of Part XIII's knowledge-not-instruction boundary, and what DAIF-08 does not do is define the type system or the authority model — it enforces them at one seam, the pack's admission gate, where they were never wired. From DAIF-03 it reads the FIDELITY VERDICT and the loss class; the payload is a verdict, the direction is inbound, the invariant is that behavioral equivalence is DAIF-03's standard and the runtime submits to it rather than declaring a rival threshold, and what DAIF-08 does not do is define fidelity or set its threshold — it supplies the counterfactual replays that make the verdict measurable.

19.3 The reads, part two: what the runtime consumes to govern execution. From DAIF-07 it reads the OPEN OBLIGATIONS: what the mission owes, what blocks it, what was promised and not delivered; inbound, and the invariant is that DAIF-07 owns the obligation LIFECYCLE — creation, discharge, expiry — while the runtime merely admits obligations into a pack and registers new ones. What DAIF-08 does not do is track an obligation's life or decide when it is closed. From GK-06 it reads the BUILT PACK; inbound, and the invariant is that the pack is an inert artifact and the runtime is the acting system, which is exactly the seam DAIF-02 declared at CIR-11's handoff. What DAIF-08 does not do is build the pack, hold the index, traverse the graph, or rank a unit — GK-01 and GK-05 own the coordinates and the graph, GK-06 owns the traversal and the ranking, and a rival ranker here would be the duplication this corpus exists to refuse. From CO-04 it reads the TIERS and their residency; inbound, and the invariant is that ascent and descent are REQUESTS against CO-04's tiering, never a parallel tiering of the runtime's own. What DAIF-08 does not do is evict, tier, or decide residency — CO-06 owns eviction, CO-04 owns residency, and the pin of Part V is a request to them, not a policy that overrides them. From CO-00 it reads the BUDGET CEILING; inbound, and the invariant is that if the runtime's per-envelope shaping and CO-00's ceiling ever disagree, the ceiling wins without argument. What DAIF-08 does not do is set the ceiling or arbitrate an over-spend. From CO-03 it reads the ROUTING DECISION — which model this work was dispatched to; inbound, and the invariant is that a mission arrives here with its model already chosen. What DAIF-08 does not do is route. And from DAIF-21 it reads the INVALIDATION SIGNAL; inbound, and the invariant is total on receipt — mark the affected units invalid, refuse to serve them from any tier, refuse to resolve a handle against an invalidated pack rather than resolving it into a stale body, and recompile exactly the invalidated set. What DAIF-08 does not do is detect a change, hold a freshness index, or decide what a change means; the recency axis of Part X is a consumption of DAIF-21's verdict and not an independent mechanism.

19.4 The writes, part one: the corrections that make the BUILDER better. To GK-06 the runtime writes the PAGE-FAULT AND USAGE PROFILE; the payload is the six observations of Part VII, joined on mission class; the direction is outbound; and the invariant is that the profile is a PRIOR and never a LAW — class and not identity is the unit of learning, a novel mission is compiled by the base selection law with its priors intact, and the profile's authority over a class is bounded by the sample that produced it. What DAIF-08 does not do at this seam is change the selection heuristic itself: it supplies GK-06 with evidence, and GK-06 decides. This is the load-bearing seam of the entire dataset, because the measure of an EXTEND verdict honestly discharged is that the parent's output improved, not that the extension's dashboards got fuller. To CO-04 the runtime writes a MISSION-SCOPED NEED SIGNAL — which units this mission actually used, by which instrument, and which it faulted for and thereby proved it needed; outbound; and the invariant is that CO-04 may now evict by observed non-use rather than by recency alone, because recency was previously all it could see. What DAIF-08 does not do is evict on that signal itself.

19.5 The writes, part two: cost, continuity, obligation, and contract. To CO-00 and CO-01 the runtime writes BUDGET-CONSUMPTION RECORDS, per envelope and per mission class, plus the assembly-cost-to-execution-cost ratio of Part XV; outbound; and the invariant is the one that gives these records their whole value — for the first time the estate can distinguish an over-spend caused by an oversized pack from an over-spend caused by an undersized one that forced the actor to hunt, because in CO-00's total those two are identical and in the per-envelope split they are opposite. What DAIF-08 does not do is enforce a budget decision; it reports, and CO-00 and CO-01 decide. To Lazarus and `session_compiler` the runtime writes RESUMPTION CONTEXT — the compiled resume pack of Part XI, with its constraints whole, its decisions justified, its obligations carried, and its evidence pointers live; outbound; and the invariant is the seventh clause of the vertical's gate, which is that when fidelity cannot be guaranteed the runtime FAILS VISIBLY and hands the mission to its authority, because a continuity system that degrades silently is worse than none at all. What DAIF-08 does not do is persist a session, recover one, or own the recovery substrate — Lazarus and `session_compiler` do, and the runtime supplies them with a specification of what a resume pack must CONTAIN, not a mechanism for carrying it. To DAIF-07 the runtime writes UNMET-NEED OBLIGATIONS: every fault refused on scope, every handle the actor saw and declined, every gap the mission carried past its own close; outbound; and the invariant is that a refused fault is never a dead end — the actor's need was real and it becomes an obligation rather than a silent scope creep, which is the mechanism Part XVIII's second expansion demonstrated. And to DAIF-04 the runtime writes its TYPED RUNTIME CONTRACTS — the mission contract, the pack admission gate, the fault event, the expansion authorization, the profile record — as typed inter-system objects; outbound; and the invariant is that DAIF-04 owns the contract fabric and the runtime is a producer of instances, never the author of the schema language they are written in.

19.6 The invariants that hold across every seam, and not merely within one. Four hold globally and their violation at any single seam voids the composition. AUTHORITY-EXCLUSIVITY: the budget and the scope are widened only by the named authority — never by the actor, never by the runtime, never as a side effect of a fault — and every seam that carries a request must carry it as a request and never as a grant. TYPE-INTEGRITY: a unit's type crosses every seam unchanged, because the type is what makes content unable to forge directive force, and a lowering pass, a cache, a bus, or a resume artifact that flattens a type has opened the injection surface Part XIII closed. REDACTION-BEFORE-EGRESS: every payload leaving the runtime — a profile record, a finding, a resume pack, a fault log, a budget report — passes through the Secret Firewall's redaction bus first, carrying pattern names and never values, because one contaminated compilation fans out into every surface the runtime touches. And RECORDED-REFUSAL: every refusal at every seam is written, because a gate never observed to refuse is indistinguishable from an inert one, and an estate whose seam dashboards are uniformly green has either perfect composition or a detector that is switched off and cannot tell which.

19.7 The negative contract, gathered into one sentence each. DAIF-08 does not build the pack — GK-06 does. It does not own the index or the graph — GK-01 and GK-05 do. It does not set the budget ceiling — CO-00 does. It does not tier or evict — CO-04 and CO-06 do. It does not route a model — CO-03 does. It does not define a lowering pass or its ordering — DAIF-02 does. It does not define an intermediate representation — DAIF-02 does. It does not define types or the authority model — DAIF-01 does. It does not define fidelity or its threshold — DAIF-03 does. It does not own an obligation's lifecycle — DAIF-07 does. It does not detect change or decide what a change means — DAIF-21 does. It does not author the contract schema language — DAIF-04 does. It does not persist or recover a session — Lazarus and `session_compiler` do. It does not transport a finding — PM-03 does. It does not measure a mechanism's cost-effectiveness — CO-01 does. And it does not set the estate's standard for what a claim requires — CO-12 does, and this dataset submits to it in every Part. What remains after all fifteen subtractions is the five mechanisms of Part I and nothing else, and the discipline of performing the subtraction publicly is itself the corpus law applied to its own dataset.

19.8 How a seam is verified rather than asserted. Each interface above is checkable and the checks are cheap. A read seam is verified by removing the parent and observing the runtime REFUSE rather than improvise: with no compiled contract, assembly is refused; with no CIR-11, admission is refused; with no invalidation signal, a pack past its recency horizon is treated as stale rather than fresh. A write seam is verified by observing the CONSEQUENCE at the parent: a profile written to GK-06 that never changes a pack is a write nobody consumed, which is the orphan-module pathology at the seam, and it is caught by the one instrument that catches all such cases — did the receiving system's behavior change, and can the change be attributed to the payload. That is the standard the whole fabric is held to, and this dataset asks for no exemption from it.

PART XIX FINAL LAW. The runtime READS CIR-11 from DAIF-02, the mission's types and authority and scope from DAIF-01, the fidelity verdict and loss class from DAIF-03, the open obligations from DAIF-07, the built pack from GK-06 over GK-01 and GK-05's index, the tiers from CO-04, the ceiling from CO-00, the routing from CO-03, and the invalidation signal from DAIF-21; and it WRITES page-fault and usage profiles back to GK-06 and mission-scoped need signals to CO-04 so that the BUILDER and the tiers improve, budget-consumption records to CO-00 and CO-01 that for the first time distinguish an over-spend from an oversized pack from an over-spend from an undersized one, resumption context to Lazarus and `session_compiler` under the obligation to FAIL VISIBLY when fidelity cannot be guaranteed, unmet-need obligations to DAIF-07 so that a refused fault becomes a recorded debt rather than a silent scope creep, and typed runtime contracts to DAIF-04; and the negative contract is the load-bearing half — DAIF-08 does not build the pack, does not own the index, does not set the ceiling, does not tier or evict, does not route the model, does not define a lowering or an IR or a type or a fidelity threshold or an obligation's lifecycle or a change detector or a contract schema, and does not persist sessions — because the failure this corpus was founded to prevent is not a system that does too little but one that annexes its parents' mechanisms in the very act of composing them, and a seam is proven only when removing the parent makes the runtime REFUSE and when the payload demonstrably changes the receiving parent's behavior.

PART XX — INTEGRATION AND DATASET-DERIVED SYSTEMS

20.1 Where this dataset sits in the fabric's execution edge. The fabric has three datasets on the path from knowledge to action and they divide cleanly. DAIF-02 COMPILES: it takes the estate's raw artifacts and lowers them into intermediate representations, of which CIR-11 — the Mission Context Pack — is the one that faces execution. DAIF-03 GATES: it decides whether a compilation is faithful, by the behavioral-equivalence standard, and it names the loss class when it is not. DAIF-08 DELIVERS: it takes the compiled and gated artifact, binds it to a mission, hands it to an actor under a runtime contract, lets the actor correct it while the mission is still running, and records what happened. That is the whole of the execution edge, and it was incomplete in a specific and consequential way before this dataset: compilation and gating both operate BEFORE the mission, on the artifact, in the absence of the reasoning that will consume it, and neither has any instrument that can be affected by what the mission actually did. An edge with no return path is a pipeline, and a pipeline improves only when someone rewrites it. The return path is the profile, and it is what turns the edge into a loop: DAIF-08 hands GK-06 a record of which predictions were wrong and under which exclusion reason, and the next compilation is better for a reason that can be named, attributed, and audited. Closing that loop is the whole of what this corpus was built to close, and it is the whole of what this dataset claims.

20.2 The systems this dataset makes buildable, and the status of the claim. What follows are six systems. They are SPECIFIED and they are NOT BUILT, and that sentence is not modesty — it is the epistemic standard the estate imposes on everything else, applied here without exemption. A dataset that describes a mechanism has produced a hypothesis about a mechanism, and a hypothesis promoted to a fact by confident description is exactly the fabrication the estate's ladder refuses; the ladder caps a deposit at the level its evidence supports, and the evidence for every system below is a specification, which is the weakest evidence there is. Each is therefore given six fields and no result: its mission, its inputs, its outputs, its authority, its activation, and its done-gate. The done-gate is the field that matters most, because a system whose done-gate is unobservable cannot be built to a standard and will be declared complete by whoever is tired.

20.3 The mission compiler. Its mission is to convert a stated request into a typed mission contract, and to REFUSE when it cannot. Its inputs are the request, the estate's laws in scope, and the project's declared authority. Its outputs are a compiled contract with the nine fields of Part II, or a refusal carrying a diagnosis. Its authority is to refuse and to derive a mission's assembly class from its scope and done-gate under Part XV's ladder — it may not widen a scope, may not grant a budget, and may not accept a class the requester asked for. Its activation is any request for context assembly: no contract, no pack, with no exception. Its done-gate is a refusal actually observed — a request with no observable done-gate refused, an unbounded subject refused as a category error rather than served, an under-funded mission refused rather than executed to exhaustion, and a mission whose minimal sufficient pack proves unmanageable surfaced as a DECOMPOSITION rather than absorbed as bloat — because a gate never observed to refuse is indistinguishable from an inert one and its refusal rate is therefore its primary metric, read against mission throughput so that refusing everything is not a strategy.

20.4 The pack assembler over GK-06. Its mission is to take GK-06's built pack and admit it, or refuse it, under the runtime contract of Part III. Its inputs are the compiled contract, GK-06's candidate pack, the budget envelopes of Part VIII with the reserve already deducted, and CO-03's routing verdict. Its outputs are an admitted pack whose every unit carries the attribution of the rule that admitted it, or a structured refusal. Its authority is structural only: it tests contract-binding, constraint-integrity, boundary-honesty, and gate-presence, and it never second-guesses GK-06's ranking, because a rival ranker here would be the fork this corpus refuses. Its activation is every LARGE and INSTITUTIONAL mission, partial for MEDIUM, and never for MICRO or SMALL under Part XV. Its done-gate is that an abridged hard constraint is refused rather than loaded with a warning, that a pack with no handles is refused as a claim of completeness, and that attribution completeness is exactly one — a unit that cannot name its admission rule is removed and its removal recorded, which is the single mechanism that converts minimality from an aspiration into an audit.

20.5 The page-fault runtime and the expansion-handle resolver. The fault runtime's mission is to make the moment of discovered insufficiency a typed, authorized, and recorded event. Its inputs are the actor's request, the pack's handle set, the remaining reserve, the mission's scope, and its current disclosure level. Its outputs are a resolved unit admitted with a FAULT attribution, or a recorded refusal, plus a written fault record carrying the exclusion reason that wrongly excluded the unit. Its authority is to grant within the reserve and inside the scope, and to ESCALATE — never to widen. Its activation is any of Part VI's four triggers and nothing else, with complexity explicitly not a trigger. Its done-gate is behavioral and it has three clauses: a fault is served with exactly the named unit and never its neighborhood, a fault storm is treated as a pack to RECOMPILE rather than a queue to serve, and a faulted unit is pinned against eviction for the life of its mission. The handle resolver's mission is narrower and its authority clause is the sharpest in this Part: it resolves a handle's target and subjects the resolved content to the FULL admission gate at LOAD — type, scope, provenance verification, trust level, redaction — exactly as if it had been a candidate at compile time, because a handle is an offer and never an authorization, and an expansion authority-checked only at assembly has left the one door this dataset itself opened unguarded. Its done-gate is a recorded rejection of a resolved unit that passed as a pointer and failed as content.

20.6 The profile recorder. Its mission is to close the loop, and it is the system whose absence makes every other system in this list a one-shot improvement rather than a compounding one. Its inputs are the pack's attributions, the fault ledger, the ascent ledger, the usage instrument's output, the cache's sampled stale-hit results, and the mission's outcome against a real done-gate including any rework. Its outputs are the six observations of Part VII, joined on mission CLASS, and the four targeted corrections of Part VII's recompilation clause — a rule demoted for a class, an exclusion reason indicted for a situation, a level boundary redrawn, a class flagged for counterfactual replay. Its authority is to PROPOSE a correction to GK-06 and to CO-04, never to apply one, because the compiler owns its heuristic and the tiers own their residency. Its activation is the sealing of every LARGE and INSTITUTIONAL mission. Its done-gate is the hardest in this Part and it is the only one that measures the corpus rather than the code: a pack that demonstrably CHANGED, for an attributable observed reason, with the change traceable back to a specific fault record — because a profile that never changes a pack is a write nobody consumed, and a telemetry stream that has never altered a parent's behavior has proven only that it can be written.

20.7 The session-continuity compiler, which is the first proving vertical. Its mission is to produce, continuously, a resume pack sufficient to continue a mission across a boundary without re-reading the repository. Its inputs are the mission contract, the estate's constraints in scope, the decisions with their justifications, the observed current reality, DAIF-07's open obligations, and the evidence pointers that ground every critical claim. Its outputs are the eight contents of Part XI, written after every sealed unit of work and never only at a graceful close — because a pack that exists only when the close was planned is absent exactly when the loss is total. Its authority is to REFUSE to certify a pack whose fidelity it cannot guarantee, and to hand the mission to its authority instead. Its activation is every session on a multi-session mission, continuously. Its done-gate is the six conjunctive clauses of Part XI with no partial credit — one hundred percent of hard constraints survive, one hundred percent of obligations survive, the current state is correctly identified from the pack rather than inferred, the actor continues without indiscriminate re-reading, every critical claim links to evidence, and a source that changed while the session was down is detected — plus the seventh and enabling requirement that the system FAIL VISIBLY, because a continuity system that degrades silently is worse than none at all: no continuity system at least forces the honest re-read, while a silent one substitutes a fabrication for it and hides the substitution. And no savings figure is asserted for it here, in advance, by anyone, because an unmeasured efficiency claim is precisely the fabrication this corpus refuses and the number must come from the behavioral trial that measures cost, fidelity, and rework on both arms.

20.8 What the six share, and what they are still not. Every system above is a consumer of a parent it does not own and a supplier of evidence to a parent it cannot command. None of them holds an index, a ceiling, a tier, a route, a type system, or a change detector. Each is small on purpose and each is specified with a done-gate that can be observed to FAIL, which is the only property that distinguishes a specification from an aspiration. They are not built. They will not be built by being described, and describing them more confidently would not advance them one step; the estate's own ladder caps a deposit at the level its evidence supports, and the evidence here is a document. What this dataset has produced is a contract precise enough that a build could be audited against it, and a set of gates precise enough that a build could be caught failing them — which is the most an institutional-knowledge dataset can honestly produce, and it is exactly what the corpus asked for.

20.9 The property the whole dataset rests on. A context pack is a HYPOTHESIS about what a mission will need. It is formed by a compiler that has not seen the mission's reasoning, has not met its obstacles, and does not know the state of the world the mission will actually encounter, and it is therefore a prediction — and predictions are wrong, and the estate has always known what to do with a prediction except in this one place, where it has been compiled, handed over, and assumed correct by the silence that followed. The only honest runtime is one that lets the hypothesis be CORRECTED WHILE THE MISSION IS STILL RUNNING: a pack that ships with the typed means of its own correction, an actor that can see what was omitted rather than having to intuit it, a fault that is authorized against a reserve that exists precisely so that asking is cheaper than guessing, and a record that makes the correction permanent instead of local. The alternative to that runtime is not a smaller context. That is the seduction and it is false. The alternative is a confident agent working from a quiet omission — reasoning correctly, proceeding smoothly, producing plausible and wrong work, and emitting no signal at all, so that tomorrow the same pack is compiled, the same gap opens, and the same guess is made forever.

PART XX FINAL LAW. DAIF-02 compiles, DAIF-03 gates, and DAIF-08 DELIVERS — and the profile it returns to GK-06 is what converts a pipeline into a loop, because compilation and gating both act before the mission and neither has any instrument that the mission's actual reasoning can move, so an execution edge with no return path improves only when someone rewrites it; from that contract six systems become buildable and are hereby SPECIFIED AND NOT BUILT, each with a mission, inputs, outputs, authority, activation, and an observable done-gate — a mission compiler whose gate is a refusal actually observed, a pack assembler whose gate is attribution completeness of exactly one and an abridged constraint refused rather than warned about, a page-fault runtime whose gate is exactly the named unit and never its neighborhood, an expansion-handle resolver whose gate is a resolved unit rejected at LOAD after passing as a pointer, a profile recorder whose gate is a pack that demonstrably changed for an attributable observed reason, and the session-continuity compiler that is the corpus's first proving vertical, whose gate is six conjunctive clauses plus the requirement to FAIL VISIBLY and for which no savings figure is asserted in advance because an unmeasured claim is a fabrication; and the dataset closes on the property that grounds all of it — a context pack is a HYPOTHESIS about what a mission will need, and the only honest runtime is one that lets the hypothesis be corrected while the mission is still running, because the alternative is not a smaller context, it is a confident agent working from a quiet omission.
