PART I — MISSION AND BOUNDARY: WHAT THIS DATASET COMPILES, AND WHAT IT HANDS DOWNSTREAM INSTEAD OF DECIDING

1.1 Dataset #01's Part V named fourteen engines and gave each one sentence of constitutional charter. The first of those fourteen, the Crawl Intent Compiler, was charged with converting an underspecified request — a sentence an Owner or an upstream system types without knowing or caring how Crawl OS internally organizes itself — into a Crawl Mission Contract: a single, internally consistent, fully populated record that every other engine in the family can execute against without needing to re-interpret the original request itself. This dataset is that charter's operational elaboration. It does for the Crawl Intent Compiler what Dataset #10 already did for the Evidence Integrity Fabric: it takes a one-sentence engine description and asks, in exhaustive and checkable detail, how does this actually work, what does it produce, and what happens when the input is ambiguous, contradictory, or simply too thin to compile.

1.2 The central artifact this dataset owns is the Crawl Mission Contract itself: a sixteen-field schema, drawn directly from the founding resource document's own enumeration, that every subsequent Crawl OS engine treats as its input contract rather than re-deriving intent from the raw request each engine happens to receive. Objective, entities, expected sources, depth, breadth, freshness, required evidence, permitted domains, authorized authentication, maximum cost, maximum time, stop conditions, output schema, confidence-level requirement, update policy, and preservation obligation — sixteen fields, each with its own compilation logic, each capable of failing to populate cleanly, and each is elaborated in its own Part later in this dataset (Part III lists the schema in full; Parts IV through XIV each take one field, or a closely related pair of fields, and work through how it is actually filled).

1.3 A distinction this dataset holds throughout, and states plainly here so it never needs restating with equal force later: to compile a field is not to decide the substance that field governs. The permitted-domains field, once populated, becomes an input the future Dataset #16 Authorization engine evaluates — this dataset does not itself decide whether a domain is authorized, only records what was requested or inferred. The maximum-cost field is a declared ceiling this dataset hands to CO-01's already-sealed budget machinery — this dataset does not compute cost, meter spend, or own any of the general resource-accounting vocabulary CO-01 already owns. The confidence-level-requirement field states a threshold a mission needs met — this dataset does not compute confidence, and does not reimplement DRK-03's CONNECT sufficiency scale to do so. Compilation is population, not adjudication. Every Part from here forward that touches a field bordering another engine's territory repeats this exact distinction in its own worked terms, because the single most common way a compiler like this one drifts out of its own boundary is by quietly starting to decide what it was only ever supposed to record.

1.4 This dataset's own STOP #1 finding, recorded in `wiggly-sparking-dolphin.md` and restated in `DATASET_02_CONTRACT.md`'s non-ownership section, is worth stating here in the dataset's own voice rather than only in the planning artifact that discovered it: an earlier automated duplicate-detection pass flagged this dataset's proposed scope as a possible fold into DRK-03, on the strength of the single shared word "evidence" appearing in this schema's required-evidence field name. On inspection the match was empty — DRK-03 owns confidence computation over evidence already gathered; this dataset owns declaring, before a single byte of evidence exists, how much of it a mission will need. A field name sharing a word with another system's vocabulary is not the same claim as owning that vocabulary, and this dataset's own Part VIII returns to exactly this distinction when it works through the required-evidence field in detail.

1.5 The six example requests the founding document itself uses to motivate this engine's existence — read this repository, research this market, extract every function, compare these competitors, watch for when it changes, build a dataset from every source — are not decorative. Each names a different combination of which of the sixteen fields dominate the compilation effort and which nearly fill themselves in by default, and Part XVII returns to all six as named templates, each with its own worked field-population pattern. A reader who wants the fastest route to understanding what this dataset actually does, before working through the field-by-field detail in Parts IV through XIV, can read Part XVII's six templates first and treat the intervening Parts as the mechanics each template invokes.

1.6 Forward references: this Part's claim that compilation is population, never adjudication, is the discipline Part IX tests against the permitted-domains field, Part X tests against the cost/time-ceiling fields, and Part XIII tests against the confidence-level-requirement field — three fields most likely to tempt a future implementation into quietly owning territory that belongs to Dataset #16, CO-01, and DRK-03 respectively. Part XV's compilation state machine is the mechanical backbone the rest of this dataset assumes; a reader confused about when a field gets filled relative to when the whole contract becomes dispatchable should read Part XV before continuing past Part III's schema listing. Part XXV closes this dataset exactly as Dataset #10 closed itself: a done-criteria restatement, a citation audit, a forward-reference spot-check, and this dataset's own contamination audit performed honestly against its own text, including this Part's own words, which is itself in scope for that audit the moment Part XXV runs it.

1.7 One further boundary, stated once and assumed throughout: this dataset compiles a single mission's contract. It does not own coordinating multiple concurrent missions, scheduling how many run at once, or arbitrating resource contention between them — Dataset #01 Part IV §4.8 already places general agent-orchestration and parallel-session scheduling outside Crawl OS's territory entirely, in systems sealed elsewhere in the estate, and this dataset inherits that same boundary rather than re-drawing it. A Mission Contract, once compiled, is one self-contained unit of intent; what happens when several such units exist at once is governed by machinery this dataset does not own and does not attempt to describe.

1.8 The founding document's own field list, quoted in the original Spanish and reproduced in translation in Part III, uses the word "evidencias" for the required-evidence field — the same surface-level lexical overlap with DRK-03's territory that produced the STOP #1 false positive described in 1.4. This dataset's own prose, from here through Part XXV, uses "required evidence" or "evidentiary burden" throughout rather than any phrasing that could be misread as claiming ownership of evidence computation itself, precisely so that a future automated scan of this dataset's own text does not reproduce the same false-positive pattern that a scan of the schema's field names already once produced.

1.9 Downstream execution state, stated with the same honesty Dataset #10 insisted on: `modules/crawl_os/mission_compiler.py` does not exist yet. No code compiles a Mission Contract today. This dataset is prose specification, discoverable via graphify per the Consumer Declarations file this front matter cites, and its downstream execution status is PLANNED, not REACHABLE, in the exact vocabulary `modules/liveness/reachability.py` already uses elsewhere in this estate for precisely this situation — a gap declared honestly rather than hidden behind confident-sounding prose about a compiler that does not yet run.

1.10 With the boundary drawn and the schema named, the remainder of this dataset proceeds in the order a real compilation pass would: Part II describes the pipeline shape before any individual field is discussed, Part III lists the full schema as a single reference point, Parts IV through XIV work through the fields (or closely related field pairs) one at a time, Part XV describes the state machine that ties field-population to contract lifecycle, Parts XVI through XXII cover the compiler's cross-cutting disciplines (ambiguity, templates, revision, rejection, disambiguation, handoff, auditing), Part XXIII names this dataset's own failure taxonomy, Part XXIV covers evolution, and Part XXV closes.

PART I FINAL LAW

The Crawl Intent Compiler converts an underspecified request into a fully populated, internally consistent Crawl Mission Contract, and every field it populates is a declared input handed to the engine that actually owns the decision that field bears on — this dataset compiles intent; it does not adjudicate cost, confidence, authorization, or routing, and any future elaboration of this engine that begins deciding those questions instead of merely declaring their inputs has left this dataset's own boundary.


PART II — FROM VAGUE REQUEST TO STRUCTURED INTENT: THE COMPILATION PIPELINE

2.1 Before any of the sixteen Mission Contract fields can be populated, a raw request has to pass through three stages this dataset treats as strictly ordered: parsing, normalization, and field assignment. Parsing takes the raw sentence — "read this repository," "investigate this market," whatever the Owner or an upstream caller actually typed — and identifies its grammatical shape: a verb naming the requested action, a direct object or set of objects naming what the action targets, and any qualifying clauses narrowing scope, timeframe, or depth. Parsing does not yet decide what any Mission Contract field will contain; it only establishes which words in the request are doing which job, the same way a compiler's lexer identifies tokens before a parser assembles them into a syntax tree.

2.2 Normalization takes the parsed shape and resolves it against what the estate already knows. A request naming "this repository" without a path is normalized against the working context the request arrived in — a session's current directory, a prior mission's target, an explicit path supplied alongside the request. A request naming a company by an informal or ambiguous name is normalized against site memory's existing knowledge of that entity, when Dataset #01 Part V §5.11's Crawl Memory and Institutional Learning engine already holds a resolved identity for it. Normalization is where a genuinely ambiguous request either resolves cleanly or is flagged for the escalation discipline Part XVI covers in full; this Part does not resolve ambiguity, it only identifies where normalization succeeded cleanly and where it did not.

2.3 Field assignment is the final stage: a normalized intent record, now carrying a resolved verb, resolved target entities, and whatever qualifying detail parsing captured, is mapped onto the sixteen Mission Contract fields. This is not a single mechanical lookup table — a request's verb alone can determine several fields at once. "Watch for when it changes" immediately implies a freshness field oriented toward recurring re-acquisition rather than a single pass, a stop-condition field oriented toward "never, until explicitly cancelled" rather than a fixed completion criterion, and an update-policy field this dataset's own Part VII works through as a distinct case. "Extract every function" implies a depth/breadth combination oriented toward exhaustiveness within a single target rather than broad discovery across many targets, and an output-schema field pre-populated toward a structured, code-shaped result rather than free prose. Part XVII's six templates are exactly this observation formalized: each canonical request verb pre-populates a distinct cluster of fields, and field assignment for a request matching one of those six templates begins from the template's defaults rather than from an empty schema.

2.4 A request that does not cleanly match one of the six templates still passes through the same three stages, just without a template's head start. The pipeline does not require template membership to function; templates are an optimization this dataset documents because the founding resource document's own six examples motivate them so directly, not a requirement the compiler enforces on every request that arrives.

2.5 Each of the three stages produces its own intermediate record, and each intermediate record is retained rather than discarded once the next stage consumes it — the raw request, the parsed shape, the normalized intent record, and the final compiled contract together form the compilation audit trail Part XXII covers as its own subject. A compiled Mission Contract that cannot be traced back through these three stages to the raw request that produced it has failed a discipline this dataset holds as tightly as Dataset #10 holds an Evidence Object's provenance chain: every field populated in the final contract must be traceable to the stage that populated it, whether that stage was direct extraction from the parsed request, a normalization-stage resolution against existing estate knowledge, or an explicit default applied at field-assignment time.

2.6 The distinction between these three population sources — extracted, resolved, and defaulted — matters enough to name explicitly here, because Part XVI's ambiguity-handling discipline and Part XIX's rejection taxonomy both depend on it. A field is extracted when the raw request states it plainly enough that parsing alone determines the value with no interpretation required. A field is resolved when normalization determines the value by consulting existing estate knowledge — site memory, working context, a prior mission's contract — rather than reading it directly off the request's own words. A field is defaulted when neither extraction nor resolution produces a confident value and the compiler falls back to a documented default rather than escalating for clarification. Every populated field in a compiled contract carries one of these three provenance tags, and Part XXII's auditing discipline treats a field with no provenance tag at all as equivalent to Part I §1.3's forbidden silent fabrication — a field must know, and record, how it came to hold the value it holds.

2.7 A worked example clarifies the three stages together. Consider the request "compárame estos competidores" — compare these competitors — arriving with a working context naming three prior missions, each having already produced a compiled contract against one of three named companies. Parsing identifies the verb "compare," the qualifying phrase "these competitors," and no further explicit qualifying detail. Normalization resolves "these competitors" against the working context's three prior missions, producing three resolved entity references rather than requiring the compiler to guess which companies the Owner means. Field assignment then populates the objective field with a comparison objective spanning all three resolved entities, the entities field with the three resolved references themselves (each carrying a resolved provenance tag), the required-evidence field with a default appropriate to comparison missions (Part VIII works through this default in detail), and so on through the remaining fields, several by template default per Part XVII's comparison-mission template, several by extraction or resolution where the request or the working context supplied enough to avoid a default.

2.8 The pipeline does not run once per request and then stop. A mission, once dispatched, can generate follow-up requests the same conversational context produces — a refinement, a scope narrowing, an explicit correction. Part XVIII's revision-and-amendment discipline covers how a follow-up request re-enters this same three-stage pipeline against an already-compiled contract rather than starting from an empty one, and how the pipeline distinguishes a genuine amendment to an existing mission from a request that, despite superficial continuity with the conversation, actually names a new mission entirely.

2.9 This Part's own boundary, stated plainly: parsing, normalization, and field assignment together produce a compiled contract's candidate field values. They do not themselves validate that the resulting contract is internally consistent, they do not themselves decide whether the contract is ready to dispatch, and they do not themselves handle the case where a field simply cannot be populated with any confidence at all. Those three concerns belong to Part XV's state machine (which governs validation and dispatch readiness) and Part XVI (which governs the escalation path when population genuinely fails). This Part's job ends the moment a candidate value, tagged with its provenance, exists for a field — what happens to that candidate value next is the rest of this dataset's subject.

2.10 One final observation on why this three-stage shape was chosen over a simpler single-pass extraction: a single pass conflates "the request didn't say" with "the request said something the compiler couldn't parse," and conflates both of those with "the compiler had no way to check whether its guess was right." Separating parsing from normalization from field assignment keeps those three distinct failure surfaces distinct, which is precisely what makes Part XIX's rejection taxonomy able to name specific, diagnosable failure classes rather than a single undifferentiated "could not compile" verdict that would tell a future implementer nothing about where the actual difficulty lay.

PART II FINAL LAW

A raw request becomes a compiled Mission Contract through three strictly ordered stages — parsing identifies grammatical shape, normalization resolves that shape against existing estate knowledge, and field assignment maps the result onto the sixteen-field schema — and every field's value carries a provenance tag recording which of the three stages produced it, because a compiler that cannot say how it arrived at a value has, in substance if not in name, fabricated it.


PART III — THE CRAWL MISSION CONTRACT SCHEMA IN FULL: SIXTEEN FIELDS DEFINED

3.1 This Part exists as a single reference point: the sixteen fields of the Crawl Mission Contract, named together in one place, each given a precise one-paragraph definition, before Parts IV through XIV take each field or closely related field pair in turn and work through its compilation logic in depth. A reader who only needs to know what the schema contains, without yet needing the mechanics of how each field gets populated, can stop after this Part; a reader who needs the mechanics continues into Part IV.

3.2 Objective. A structured statement of what the mission is meant to accomplish, distinct from the raw request's own wording — the objective field is the compiler's own restatement, in a controlled vocabulary the downstream engines can act on directly, of the intent Part II's pipeline extracted, resolved, or defaulted. Part IV covers objective extraction together with entity extraction, since the two nearly always resolve from the same portion of a raw request.

3.3 Entities. The specific real-world things the mission concerns — a repository, a company, a named individual, a market category, a document — each carrying, where normalization succeeded, a resolved reference into existing estate knowledge rather than a bare string copied from the request. Part IV covers entity extraction in full.

3.4 Expected sources. A prediction, not a commitment, of where the mission's evidence is likely to be found — named domains, document types, or source categories the compiler expects the downstream Frontier Engine and Acquisition Strategy Router to actually visit, understood from the first as a prediction subject to revision once real acquisition begins. Part V covers source prediction.

3.5 Depth. How far into a single source or source cluster the mission should pursue its objective — a shallow depth accepts a source's front-facing content; a deep depth pursues nested structure, linked detail pages, or exhaustive enumeration within that one source. Part VI covers depth together with breadth.

3.6 Breadth. How many distinct sources or source clusters the mission should pursue in parallel — a narrow breadth commits to one authoritative source; a wide breadth deliberately samples many sources to cross-check or aggregate. Part VI covers breadth in the same worked terms as depth, since the two are calibrated together far more often than independently.

3.7 Freshness. How current the acquired evidence needs to be, and whether staleness beyond some threshold invalidates it for the mission's purpose — a one-time historical research mission may tolerate evidence of any age; a price-monitoring mission may require evidence acquired within the last hour. Part VII covers freshness together with the update-policy field.

3.8 Required evidence. The mission's declared evidentiary burden — how much corroboration, and of what kind, the mission's objective demands before its output can be considered trustworthy — expressed as an input Dataset #10's Evidence Provenance and Integrity Fabric and DRK-03's CONNECT sufficiency vocabulary consume, never as a confidence computation this dataset performs itself. Part VIII covers required-evidence translation.

3.9 Permitted domains. The set of domains, or domain categories, the mission is authorized to acquire from — populated here as a declared field, evaluated for actual permission by the future Dataset #16 Authorization engine, per this dataset's own non-ownership boundary stated in Part I §1.3. Part IX covers permitted domains together with authorized authentication.

3.10 Authorized authentication. Which authenticated sessions, credentials, or access grants, if any, the mission is permitted to use in pursuit of its objective — again a declared field this dataset populates without itself deciding what authentication is actually authorized, that adjudication belonging to the same future Dataset #16. Part IX covers this field in full.

3.11 Maximum cost. A declared ceiling on the token, compute, or service cost the mission may consume — a boundary this dataset hands to CO-01's already-sealed cost-accounting machinery rather than computing or metering itself. Part X covers maximum cost together with maximum time.

3.12 Maximum time. A declared ceiling on the wall-clock duration the mission may run — the temporal counterpart to maximum cost, subject to the identical non-ownership boundary. Part X covers this field in full.

3.13 Stop conditions. The taxonomy of triggers that end a mission's active pursuit of its objective — target satisfied, a ceiling reached, a dead end encountered, a contradiction discovered, an explicit Owner abort — populated as an explicit field precisely so that no mission is ever dispatched with no defined way to end. Part XI covers stop conditions in full.

3.14 Output schema. The shape the mission's final deliverable is expected to take — a markdown report, a structured dataset, a comparison table, a change-alert notification — declared up front so the downstream Knowledge Distillation Fabric (Dataset #01 Part V §5.9's sibling engine) knows what it is being asked to eventually produce. Part XII covers output schema in full.

3.15 Confidence-level requirement. The minimum confidence threshold the mission's eventual output must meet before it can be delivered as final rather than provisional — a declared requirement DRK-03's CONNECT vocabulary evaluates against, never a confidence value this dataset computes. Part XIII covers this field in full.

3.16 Update policy. Whether, and how often, a mission re-runs itself against fresh acquisition once its initial pass completes — closely related to the freshness field but distinct from it, since freshness governs how current a single acquisition's evidence must be while update policy governs whether the mission recurs at all. Part VII covers update policy together with freshness.

3.17 Preservation obligation. Whether, and for how long, the mission's acquired evidence must be retained after the mission itself concludes — a field this dataset populates as a declared requirement that Dataset #10's already-sealed retention and freshness-window machinery (Part XII specifically) then operates against. Part XIV covers preservation obligation in full.

3.18 These sixteen fields are not independent of one another in practice, even though each is defined independently here for clarity. A mission's depth and breadth jointly constrain what expected sources are realistic to name; a mission's freshness requirement constrains what update policy makes sense; a mission's required-evidence field constrains what confidence-level requirement is achievable at all. Part XV's compilation state machine includes an explicit internal-consistency check precisely because these sixteen fields, populated independently by Parts IV through XIV's separate mechanics, must still cohere as a single contract before that contract is validated for dispatch.

3.19 A contract missing any one of these sixteen fields, whatever the reason, is not a valid Crawl Mission Contract — it is an intermediate compilation artifact still in progress. Part XV's state machine names the exact point at which a contract earns the status "validated," and full population of all sixteen fields, each carrying a provenance tag per Part II §2.6, is the first of that state machine's necessary conditions.

3.20 One organizing observation ties the sixteen fields together, worth stating explicitly before this Part hands off to the field-by-field treatments that follow: the schema separates naturally into fields this dataset actually decides (objective, entities, expected sources, depth, breadth, stop conditions, output schema — the mission's own shape) and fields this dataset only declares as inputs for another engine to adjudicate (permitted domains, authorized authentication, maximum cost, maximum time, required evidence, confidence-level requirement — boundaries a future or already-sealed sibling system owns). Freshness, update policy, and preservation obligation sit between the two groups, declared here but operated against by machinery — Dataset #10's already-sealed retention discipline chief among them — that this dataset does not itself run. A reader working through Parts IV to XIV in order will notice this grouping recur constantly: each Part opens by stating which of the two categories its field belongs to, because the compilation discipline required differs sharply between a field this dataset decides outright and a field this dataset merely records for someone else's decision.

PART III FINAL LAW

The Crawl Mission Contract is exactly sixteen fields — objective, entities, expected sources, depth, breadth, freshness, required evidence, permitted domains, authorized authentication, maximum cost, maximum time, stop conditions, output schema, confidence-level requirement, update policy, and preservation obligation — and a contract missing any one of them, for any reason, has not yet reached the status this dataset calls validated, regardless of how complete the remaining fifteen fields may be.


PART IV — OBJECTIVE AND ENTITY EXTRACTION

4.1 The objective and entities fields are treated together in this Part because, across the overwhelming majority of real requests, they resolve from the same clause of the same sentence — the verb supplies the objective's shape, and the verb's direct object or objects supply the entities. "Read this repository" yields an objective of comprehension against a single entity, the named repository. "Compare these competitors" yields an objective of comparison against multiple entities, the named competitors. Separating the two into distinct fields, rather than merging them into one, exists because downstream engines need to reason about them independently: the Acquisition Strategy Router (Dataset #01 Part V §5.2, elaborated in the future Dataset #03) routes primarily off objective shape, while the Frontier Engine and Extraction engines (§5.6 and §5.7 respectively) route primarily off which entities are actually named.

4.2 Objective extraction begins with identifying the request's governing verb — read, investigate, extract, compare, watch, build, or any of the many verbs a real request might use that do not map cleanly onto one of Part XVII's six canonical templates. A verb's identification is not merely lexical; "look into" and "investigate" carry the same objective shape despite sharing no surface vocabulary, and a compiler that only pattern-matches exact verb strings will misclassify a large fraction of real requests. This dataset does not prescribe a specific verb-classification mechanism — that is an implementation decision for the future `modules/crawl_os/mission_compiler.py` — but it does prescribe the discipline the mechanism must satisfy: the resulting objective field must be expressed in the controlled vocabulary Part III §3.2 establishes, never as a verbatim copy of whatever verb phrase the raw request happened to use, so that downstream engines can route consistently regardless of how a given Owner happened to phrase an equivalent request.

4.3 Entity extraction is a harder problem than objective extraction, and this dataset treats it as such. A raw request names entities with wildly varying specificity: a full URL leaves nothing to resolve; a company's informal or colloquial name requires resolution against site memory or, failing that, an entity-resolution pass this dataset does not itself perform but declares as a required step before the entities field can be marked resolved rather than merely extracted; a pronoun or demonstrative ("this," "these," "it") requires resolution against working context exactly as Part II §2.2 describes. The entities field's provenance tag, per Part II §2.6, must distinguish all three cases explicitly, because a downstream engine consuming an entities field populated by direct extraction can trust the reference at face value, while one consuming a field populated by contextual resolution should be prepared for the possibility that the resolution was itself uncertain.

4.4 A single request can name entities at more than one level simultaneously, and this dataset requires the entities field to preserve that structure rather than flattening it. "Compare these competitors" names three entities at the primary level — the three companies — but each company itself may carry sub-entities relevant to the comparison, such as specific product lines or specific named executives, if the raw request's qualifying clauses named them. The entities field is therefore not a flat list but a structured record capable of nesting, and Part XX's disambiguation discipline returns to this structure directly when it works through what happens when a nested entity itself resolves ambiguously.

4.5 Objective extraction interacts with entity extraction in one further way worth naming explicitly: a request's objective can itself be entity-dependent in a way that changes what "the objective" even means. "Extract every function" names an objective of exhaustive extraction, but what counts as "every function" depends entirely on which entity — which repository, which specific codebase boundary within a monorepo — the entities field resolves to. This dataset requires objective extraction to run after entity resolution has at least reached a provisional result, precisely because an objective compiled against an unresolved entity reference risks compiling an objective that later turns out to be incoherent once the entity actually resolves.

4.6 A worked example: the request "investiga este mercado" — investigate this market — arriving with no working context and no prior mission naming a specific market. Objective extraction identifies "investigate" as the governing verb, yielding a research-oriented objective shape per Part XVII's market-research template. Entity extraction attempts to resolve "this market" and finds no working context to resolve against — no prior mission, no explicit market name in the surrounding conversational turn. This is precisely the case Part XVI's ambiguity-handling discipline exists for: entity extraction cannot proceed to a resolved or even a confidently defaulted value, and the compiler must escalate for clarification rather than guessing at a market the Owner never actually named. The objective field, meanwhile, can still be tentatively populated — the shape of what's being asked is clear even though what it's being asked about is not — and Part XVI covers exactly this asymmetric case, where one field escalates while a related field remains populatable.

4.7 Objective extraction must also detect compound objectives — a single request naming what are, on inspection, two or more distinct missions rather than one. "Read this repository and tell me if it duplicates any existing Power Pack module" names both a comprehension objective and a comparison objective against the estate's own existing module set, and this dataset requires the compiler to recognize this compound shape rather than silently collapsing it into a single objective that loses half the request's actual intent. Part XIX's rejection taxonomy does not classify a compound objective as a rejection — a compound request is compilable, just not into a single contract — but Part XV's state machine notes that a compound objective produces multiple contracts from a single compilation pass, each independently validated and dispatched, sharing only their common originating request in the compilation audit trail Part XXII covers.

4.8 Entity extraction's relationship to Dataset #01 Part V §5.11's Crawl Memory and Institutional Learning engine deserves explicit statement: this dataset consumes site memory's existing entity knowledge as an input to resolution, exactly as Part II §2.2 already establishes, but does not itself own or populate site memory. A newly resolved entity, once a mission concludes, may become a candidate for site memory's own institutional-learning process to retain — but that retention decision belongs entirely to the engine Part V §5.11 charters, and this dataset's entity-extraction mechanics stop at consuming what site memory already holds, never at deciding what site memory should hold going forward.

4.9 A failure mode specific to entity extraction, distinguished here from Part XXIII's full failure taxonomy because it is common enough to deserve a dedicated warning in its own right: over-confident entity resolution, where a compiler resolves an ambiguous reference to the wrong real-world entity because a plausible match existed in working context even though the raw request's own wording, examined carefully, pointed elsewhere. A request following a mission about one company that then asks to "compare it to its biggest rival" risks resolving "it" correctly but resolving "its biggest rival" against a stale or incorrect assumption about which company that actually is, if the compiler does not re-verify the resolution against current context rather than reusing a cached assumption from earlier in the conversation. This dataset requires entity resolution to re-verify against current working context on every compilation pass, never to reuse a prior pass's resolution as an unexamined given.

4.10 With objective and entities established, the compiled contract's remaining fourteen fields depend on these two more than on each other — depth and breadth (Part VI) calibrate against the objective's shape and the entity count; expected sources (Part V) predicts against the resolved entities specifically; required evidence (Part VIII) calibrates against objective shape. This Part's output, a resolved or explicitly escalated objective-and-entities pair, is therefore the load-bearing first real compilation step every other field-population Part in this dataset assumes has already run.

PART IV FINAL LAW

Objective and entity extraction resolve together from the same clause of a raw request far more often than not, but they are compiled as two distinct, independently provenance-tagged fields because downstream engines route on each independently — and a compiler that resolves an entity reference without re-verifying it against current working context on every pass, rather than trusting a prior pass's resolution, has reintroduced exactly the kind of silent, unexamined assumption this dataset's provenance discipline exists to make impossible.


PART V — SOURCE PREDICTION: NAMING EXPECTED SOURCES BEFORE A SINGLE FETCH OCCURS

5.1 The expected-sources field is unlike every other field in the Mission Contract schema in one specific respect: it is, by its own nature, a prediction the compiler expects to be at least partially wrong. Every other field states something the mission requires, permits, or bounds; expected sources states something the compiler guesses, before any acquisition has actually happened, about where the mission's evidence is likely to be found. This Part works through why the field exists despite that uncertainty, how it gets populated, and how downstream engines are expected to treat a prediction as exactly that rather than as a commitment.

5.2 The field exists because a genuinely blind Frontier Engine — one given an objective and entities with zero guidance about where to start looking — is measurably less efficient than one given even a rough prediction to begin from. Dataset #01 Part V §5.6 charters the Frontier Engine's large-scale crawl management without specifying how a crawl's initial seed set gets chosen; this dataset's expected-sources field is exactly that initial seed prediction, handed downstream as a starting point the Frontier Engine is free to expand, correct, or abandon entirely once real acquisition begins to reveal what is actually there.

5.3 Source prediction draws on three inputs, in a rough priority order this dataset states without treating the order as rigid: an explicitly named source in the raw request itself (the strongest signal, requiring no prediction at all, only extraction); site memory's record of where a previously resolved entity's evidence has been found before, when Dataset #01 Part V §5.11's engine holds such a record; and a general prediction based on entity type and objective shape when neither of the first two inputs applies — a company entity under a research objective predicts toward the company's own domain, major review or reference sites, and recent news coverage, in roughly that order, absent any more specific signal.

5.4 A prediction populated by the third input — general entity-type-and-objective-shape inference — carries the weakest provenance tag of the three, and this dataset requires that weakness to be recorded explicitly rather than presented with the same confidence as an explicitly named source. Part VIII's required-evidence translation and Part XIII's confidence-level requirement both interact with this weakness: a mission whose expected sources are all weakly predicted, rather than strongly named or resolved from memory, may reasonably carry a higher required-evidence burden precisely because the starting prediction is less trustworthy and the mission's eventual confidence will depend more heavily on what the Frontier Engine actually discovers than on what this Part predicted in advance.

5.5 Source prediction must also account for domain diversity when the objective shape calls for it. A comparison objective across multiple entities predicts sources for each entity independently rather than assuming all entities share a common source type — three competitors may have wildly different web presences, one dominant in traditional media coverage, one dominant in a specific industry trade publication, one with almost no independent coverage at all and requiring direct acquisition from its own site as the only realistic source. A prediction that assumes uniform source shape across a comparison mission's entities has failed to actually predict anything useful and has instead produced a template response dressed up as a prediction.

5.6 The field's relationship to the permitted-domains field (Part IX) deserves explicit note: expected sources predicts where evidence probably is; permitted domains declares where the mission is allowed to look. The two fields are populated independently and can diverge — a prediction might reasonably include a source the mission's permitted-domains field then excludes, if the raw request or an inherited authorization boundary restricts acquisition to a narrower domain set than an unconstrained prediction would suggest. This dataset requires the compiler to surface such a divergence explicitly in the compilation audit trail rather than silently narrowing the prediction to match the permission boundary, because a narrowed prediction that looks native to the mission rather than constrained by policy would obscure a genuine limitation the downstream Frontier Engine, and ultimately the Owner, should be aware exists.

5.7 A worked example: the request "extrae todas las funciones" — extract every function — resolved in Part IV's own worked terms against an entity that is a specific repository already named by path. Source prediction here is nearly trivial: the entity itself, the named repository, is the entire predicted source set, since an exhaustive-extraction objective against a codebase entity has no meaningful "expected sources" question beyond the codebase itself. This is worth naming because it illustrates that source prediction is not always a hard problem — for entity types with an obviously bounded, self-contained source (a single repository, a single document, a single named page), the field populates almost mechanically from the entity itself, and the harder cases this Part has spent most of its length on (companies, markets, loosely bounded topics) are the ones that actually exercise the prediction machinery in depth.

5.8 Source prediction interacts with the depth and breadth fields (Part VI) in a specific way worth flagging here even though Part VI covers depth and breadth in full: a wide-breadth mission predicts a correspondingly larger expected-sources set, and a narrow-breadth mission predicts a smaller one, but the causal direction runs both ways in practice — sometimes breadth is set first (the raw request explicitly calls for comparing many sources) and sources are predicted to match, and sometimes source prediction reveals that far more plausible sources exist than the request's other signals suggested, prompting the compiler to widen breadth beyond what would otherwise have been the default. Part VI's own treatment returns to this interaction from the breadth side.

5.9 The expected-sources field, once populated, is not revisited by this dataset's own compilation pipeline after the contract is dispatched — Part II §2.8 already establishes that revision belongs to Part XVIII's amendment discipline, triggered by an explicit follow-up request, not by the Frontier Engine's own discovery that the prediction was wrong. This is a deliberate boundary: the Frontier Engine is expected to expand and correct its own working source set continuously as acquisition proceeds, per Dataset #01 Part V §5.6's own charter, without needing to route back through this dataset's compilation pipeline every time reality diverges from an initial prediction. The expected-sources field is a starting point handed off once, not a field this dataset keeps synchronized with the Frontier Engine's own evolving understanding.

5.10 A failure mode specific to source prediction, worth naming here rather than deferring entirely to Part XXIII: false-confidence prediction, where a compiler populates the expected-sources field with a provenance tag suggesting strong extraction or memory-based resolution when the actual basis was general entity-type inference. This matters more than it might first appear, because a downstream engine or an Owner reviewing a compiled contract who sees a confidently-tagged source prediction has no way to know, without re-deriving the compilation trail, that the prediction was actually the weakest of the three input types Part 5.3 names. This dataset requires provenance tags on the expected-sources field to distinguish all three input types explicitly, with no collapsing of memory-resolved and inference-predicted sources into a single undifferentiated "predicted" tag.

5.11 Source prediction's relationship to authorization deserves one further note distinct from 5.6's divergence-recording discipline: a predicted source that later, once evaluated by the future Dataset #16 Authorization engine, turns out to be entirely unauthorized does not retroactively invalidate the compiled contract's other fields. Consider a mission whose objective concerns a general market landscape, and whose source prediction reasonably includes a product page on an external e-commerce competitor's site as a plausible landscape data point — a permitted research target under Dataset #01 Part IV §4.9's own boundary, since studying what an external site publishes about its own market position is squarely within Crawl OS's acquisition territory even though the site itself operates in a domain this family's own governance forbids Crawl OS's datasets from acquiring domain expertise about. If that specific source is later found unauthorized for this particular mission's stated authentication scope, the mission's remaining predicted sources and the rest of the compiled contract stand independently; Part XIX's rejection taxonomy reserves full-contract rejection for cases where no viable source remains at all, not for the ordinary case of one predicted source among several turning out to be inaccessible.

PART V FINAL LAW

Expected sources is a prediction, never a commitment, populated from explicit naming, memory-based resolution, or general entity-type inference in roughly that order of strength, and its provenance tag must distinguish which of the three produced it — because a downstream engine or an Owner who cannot tell a confidently named source from a weakly inferred guess has been handed a false sense of certainty about a field this dataset's own charter requires to be understood, from the moment it is populated, as subject to revision the instant real acquisition begins.


PART VI — DEPTH AND BREADTH CALIBRATION

6.1 Depth and breadth are calibrated together, in this Part and in practice, because they trade against each other under the same cost and time ceilings Part X declares as separate fields: a fixed budget spent going deep into few sources leaves less available for going wide across many, and vice versa, and a compiler that sets one without reference to the other risks producing a contract whose depth and breadth combination is simply not achievable within its own declared cost and time ceilings — precisely the internal-consistency failure Part III §3.19 flags as blocking a contract from reaching validated status.

6.2 Depth calibration begins from the objective field's shape. A comprehension objective ("read this repository") defaults toward depth sufficient to cover the entity's full structure once, without repeated passes. An exhaustive-extraction objective ("extract every function") defaults toward maximum depth within the entity's boundary, since the objective's own definition of success requires nothing less than complete internal coverage. A monitoring objective ("watch for when it changes") defaults toward shallow depth on each individual pass, since the objective's value comes from repeated shallow checks over time rather than from any single pass going deep — Part VII's update-policy field is what actually delivers the repetition; depth here governs only how far a single pass goes.

6.3 Breadth calibration begins from the entities field's cardinality and the objective's shape together. A single-entity objective defaults toward narrow breadth, since there is only one entity to pursue. A comparison objective across multiple named entities defaults toward breadth matching the entity count, with each entity's source set (per Part V's prediction) pursued at comparable depth so the comparison is not skewed by unevenly thorough acquisition across the entities being compared. A market-research objective, where entities may be loosely bounded or only partially named at compilation time, defaults toward a breadth wider than the explicitly named entity count, deliberately sampling beyond what was named in order to surface entities the request's own wording did not anticipate.

6.4 The interaction between depth, breadth, and the required-evidence field (Part VIII) deserves note here even though Part VIII covers required evidence in full: a mission with a high required-evidence burden generally needs either greater depth (more corroborating detail from fewer sources) or greater breadth (independent corroboration across more sources), and this dataset does not prescribe which trade a given mission should make — that calibration depends on the specific objective and entity types involved, and is properly a compiler-implementation decision within the bounds this Part establishes, not a fixed rule this dataset could state once and have hold universally.

6.5 A worked example clarifies the calibration in combination. The request "constrúyeme un dataset con todas las fuentes" — build a dataset from every source — resolved against a topic entity rather than a single named organization, defaults under Part XVII's dataset-construction template toward both maximum depth (exhaustiveness within each source) and maximum breadth (as many sources as the cost and time ceilings permit) simultaneously, since the objective's own definition of success is comprehensive coverage along both axes at once. This is the one canonical template where depth and breadth do not trade against each other by the compiler's own default logic — both are pushed as high as the declared ceilings allow — and Part X's cost/time-ceiling fields become the binding constraint on the mission's actual scope far more directly than for any of the other five templates.

6.6 Depth and breadth calibration must also account for a mission's stated or inferred urgency, distinct from its freshness requirement (Part VII). A request carrying explicit time pressure — "necesito esto para mañana," I need this by tomorrow — constrains the achievable depth/breadth combination independent of what the objective's own shape would otherwise default toward, and this dataset requires urgency signals, when present in the raw request, to be captured during Part II's parsing stage and carried forward as an input to depth/breadth calibration rather than discarded once the objective and entities fields are populated.

6.7 A failure mode specific to this Part, distinguished here because it recurs often enough in the founding resource document's own worked concerns to deserve explicit naming: breadth inflation without corresponding depth, where a compiler defaults toward wide breadth because a market-research or comparison objective superficially calls for it, without checking whether the required-evidence field's burden can actually be satisfied by shallow passes across many sources rather than requiring deeper engagement with fewer, more authoritative ones. A wide-shallow mission can produce a large volume of acquired content while satisfying almost none of a demanding evidentiary burden, and this dataset requires depth/breadth calibration to check against the required-evidence field's already-populated value (Part VIII populates before Part VI's calibration finalizes, per the dependency this Part VI §6.4 already notes) rather than calibrating breadth in isolation from what the mission actually needs to prove.

6.8 Depth and breadth, once calibrated, are expressed in the compiled contract as qualitative bands rather than precise numeric targets — shallow/moderate/exhaustive for depth, narrow/moderate/wide for breadth — deliberately leaving the exact numeric translation (how many pages deep, how many sources wide) to the downstream Acquisition Strategy Router and Frontier Engine, which are positioned to translate a qualitative band into a concrete acquisition plan against real, current information about what a given source actually contains, information this dataset's own compilation pipeline does not have access to before acquisition begins.

6.9 This qualitative-band choice is itself a boundary worth stating explicitly: this dataset does not own the numeric translation of depth and breadth into concrete crawl parameters — page counts, link-following limits, source-visit counts — because that translation depends on real-time information about source structure this dataset's compilation pipeline has no access to. Numeric translation belongs to the future Dataset #03 Acquisition Strategy Router and Dataset #06 Large-Scale Crawl Frontier Engineering, consuming the qualitative band this dataset compiles as their own input rather than a number this dataset would have had to guess at blind.

6.10 Depth and breadth calibration closes this dataset's first block of field-population Parts — objective and entities (Part IV), expected sources (Part V), and now depth and breadth (Part VI) together form what this dataset treats as the mission's core shape, populated first because every remaining field, from freshness through preservation obligation, calibrates against this core shape rather than the reverse. Part VII, taking up freshness and update policy next, is the first of the remaining Parts to depend explicitly on this core shape already being settled.

6.11 A closing note on how depth and breadth are re-examined, distinct from how they are first set: this dataset requires the compiler to re-run the calibration in 6.1 through 6.4 once, silently, immediately after every other field in Part III's schema has received its own first-pass value, rather than treating depth and breadth as finished the moment Part VI's own logic first produces a band. The reason is ordering: 6.4 already notes that calibration depends on the required-evidence field, which Part VIII populates after Part VI's own position in this dataset's Part sequence, and a depth/breadth value calibrated before required evidence exists is calibrated against an assumption rather than the mission's actual evidentiary burden. This second, silent pass is not a revision in Part XVIII's sense — no follow-up request triggered it, and it happens within a single compilation pass rather than across two — it is simply the point at which a value provisionally set early in the pipeline is confirmed or adjusted once every field it actually depends on has itself been populated. A compiled contract whose depth and breadth were never re-examined this way carries values calibrated against an incomplete picture of the mission, even though nothing about the compilation process itself failed outright.

PART VI FINAL LAW

Depth and breadth are calibrated together against the objective's shape, the entities field's cardinality, and the required-evidence field's burden, expressed as qualitative bands rather than precise numeric targets because this dataset's compilation pipeline has no access to the real-time source information that numeric translation actually requires — and a compiler that inflates breadth to match an objective's superficial shape without checking whether the resulting wide-shallow acquisition can satisfy the mission's actual evidentiary burden has calibrated against the wrong signal.


PART VII — FRESHNESS AND THE UPDATE POLICY FIELD

7.1 Freshness and update policy are companion fields, both governing time, and both easy to conflate on first encounter — this Part exists to draw the distinction sharply and keep it that way through every later reference either field receives. Freshness answers a question about a single acquisition: how current must the evidence be for this pass to count as satisfying the mission. Update policy answers a question about the mission as a whole: does this mission run once, or does it recur, and on what schedule. A mission can have a strict freshness requirement and no update policy at all — a one-time price check that must reflect the current moment but never repeats. A mission can equally have a loose freshness requirement and an aggressive update policy — a slow-moving regulatory-filing monitor that tolerates evidence a few days old but checks continuously over a long horizon. The two fields are populated independently, even though, per Part III §3.20's grouping, both sit in the "declared but operated against by another engine's machinery" category rather than the "this dataset decides outright" category.

7.2 Freshness compiles from three signals, considered together rather than in strict priority order the way Part V's source-prediction inputs were: an explicit temporal qualifier in the raw request ("recent," "as of today," "historical is fine"); the objective's own shape, which carries an implicit freshness default even absent an explicit qualifier (a monitoring objective implies strict freshness on every individual pass; a historical-research objective implies freshness is nearly irrelevant); and the entity's own volatility, when site memory or general entity-type knowledge suggests how quickly a given kind of entity's public information tends to change. A static reference document entity implies loose freshness by its own nature; a live pricing page implies strict freshness by its own nature, independent of what the raw request's wording happened to say.

7.3 A freshness value, once compiled, is expressed as a maximum tolerable age rather than a vague qualitative label — "evidence acquired within the last hour," "evidence acquired within the last thirty days," "no freshness constraint" — because Dataset #10's already-sealed retention and freshness-window machinery (Part XII specifically) operates on exactly this kind of bounded threshold, checking a stored Evidence Object's acquisition timestamp against the mission's declared tolerance rather than against a fuzzy qualitative description this dataset would otherwise have to translate downstream anyway. Compiling directly to the threshold Dataset #10's machinery actually consumes avoids a translation step that would itself be a place for silent drift between what this dataset intended and what Dataset #10 actually enforces.

7.4 Update policy compiles into one of a small number of recurrence shapes: single-pass (the default, and the correct value for any mission whose objective does not itself imply recurrence), scheduled recurrence at a stated interval, and continuous/event-driven recurrence triggered by an external signal rather than a fixed clock — a change-detection mission that re-checks the moment site memory's own change-intelligence machinery (the future Dataset #11) signals a change rather than on any fixed schedule. This dataset compiles the update-policy field with whichever shape the request implies, but does not itself implement the scheduling, triggering, or change-detection mechanics that make scheduled or event-driven recurrence actually happen — that execution belongs to systems outside this dataset's own ownership, most directly the future Dataset #11 Temporal Web and Change Intelligence dataset, with this field serving as the declared instruction that future engine executes against once it exists.

7.5 The interaction between freshness and update policy deserves one worked example, since the two fields' independence is easier to see in a concrete case than in the abstract. Consider "vigila cuándo cambia" — watch for when it changes — applied to a company's pricing page. Update policy compiles to continuous/event-driven recurrence: the mission does not run on a fixed schedule but re-checks whenever a change is detected. Freshness, on each individual re-check, compiles to strict — the whole point of a change-monitoring mission is that each check reflects the current state of the page, not a cached or delayed view of it. Now consider the same request applied to a slow-moving regulatory filing that updates perhaps twice a year: update policy still compiles to continuous/event-driven recurrence, since the objective's own shape is unchanged, but freshness on each check can compile more loosely, since a filing that changes twice a year does not need the same sub-hour freshness a live pricing page needs — checking daily rather than by the minute is amply fresh for an entity of this volatility, and Part 7.2's third signal, entity volatility, is precisely what produces this looser value even though the objective and update-policy shape are identical to the pricing-page case.

7.6 A failure mode specific to this pairing, worth naming directly: conflating "recurs often" with "must be current," which produces a compiled contract demanding sub-hour freshness on every pass of a mission whose entity simply does not change that quickly, wasting acquisition effort re-checking a source that has not meaningfully changed since the prior pass. This dataset requires freshness and update policy to be calibrated from entity volatility independently of each other's values, precisely so that a high-frequency update policy does not automatically inflate the freshness requirement beyond what the entity itself actually warrants.

7.7 Freshness and update policy both interact with the maximum-cost and maximum-time fields (Part X) in an asymmetric way worth flagging here: a single-pass mission's freshness requirement affects only that one pass's cost, but a recurring mission's update policy multiplies whatever per-pass cost the mission incurs by however many recurrences the mission's own stop conditions (Part XI) eventually allow before terminating. This dataset requires a compiler populating update policy toward scheduled or continuous recurrence to also verify, as part of Part XV's internal-consistency check, that the mission's declared cost and time ceilings can plausibly accommodate the implied number of recurrences — a continuous-recurrence mission compiled against a cost ceiling sized for a single pass is internally inconsistent the moment the second recurrence would exceed it.

7.8 A boundary worth restating from Part I §1.3 in this specific field's own terms: this dataset's freshness and update-policy fields declare what a mission needs; they do not implement the scheduling daemon, the change-detection trigger, or the re-acquisition dispatch that actually makes a recurring mission recur. A compiled contract with update policy set to continuous recurrence is, at the moment of compilation, a declaration awaiting an execution substrate this dataset does not build — precisely the same PLANNED-not-REACHABLE honesty Part I §1.9 already applies to this dataset's own downstream module, extended here to the specific claim that a compiled recurring mission "runs" at all before that substrate exists.

7.9 Freshness's relationship to Dataset #10's Part XII is worth one further precise statement, since the two fields are easy to conflate given how closely they interact. Dataset #10 Part XII's freshness window governs how long an already-acquired, already-stored Evidence Object remains usable before it is considered stale and requires re-acquisition — an operational mechanism acting on evidence that already exists. This dataset's freshness field governs a different moment entirely: the threshold a mission declares before any acquisition has happened, which Dataset #10's machinery then consumes as the tolerance it checks stored evidence against. The two are connected — this dataset's declared value becomes an input to Dataset #10's operational check — but they are not the same mechanism operating at the same moment, and a reader conflating "declaring a freshness requirement" with "enforcing a freshness window" has collapsed a distinction this dataset and Dataset #10 both depend on staying separate.

7.10 One remaining case deserves explicit treatment: a request naming no temporal qualifier at all, against an entity of unknown or unresolved volatility, under an objective shape (such as general comprehension) that carries no strong implicit default either. This is the genuine no-signal case, and this dataset's default for it is a moderate freshness tolerance — evidence acquired within the mission's own compilation-to-dispatch window counts as fresh enough, neither demanding sub-hour currency nor tolerating arbitrarily stale evidence — combined with single-pass update policy, the safer default absent any signal suggesting the mission should recur. This default, like every default this dataset applies, carries a defaulted provenance tag per Part II §2.6, distinguishing it plainly from a freshness value the request or entity volatility actually determined.

PART VII FINAL LAW

Freshness governs how current a single acquisition's evidence must be; update policy governs whether the mission recurs at all — two fields populated independently from the request's explicit qualifiers, the objective's implicit default, and the entity's own volatility, and a compiler that inflates one to match the other, rather than calibrating each from its own actual signal, has produced a contract that either wastes acquisition effort chasing unnecessary currency or accepts staleness the mission's own objective never actually tolerated.


PART VIII — REQUIRED-EVIDENCE TRANSLATION: DECLARING THE MISSION'S EVIDENTIARY BURDEN

8.1 The required-evidence field is this dataset's most carefully bounded field, precisely because it sits closest to territory DRK-03 already owns, and because Part I §1.4 already recorded the STOP #1 false positive this exact field name once produced. This Part exists to state, in full and exhaustive detail, what this dataset actually does with this field — translate an objective and its stakes into a declared evidentiary burden — and what it explicitly does not do, which is compute, score, or adjudicate whether any given piece of acquired evidence satisfies that burden.

8.2 Required evidence compiles from the objective's own stakes, a dimension distinct from the objective's shape (which Part IV already covers) or its depth/breadth (Part VI). A mission whose output will inform a low-stakes, easily-reversible decision — a casual comparison of two products for personal interest — compiles toward a light evidentiary burden: a single corroborating source may suffice. A mission whose output will inform a high-stakes, hard-to-reverse decision — a comparison feeding a real commitment of resources, or a research pass whose conclusion the Owner intends to act on directly — compiles toward a heavier burden: multiple independent corroborating sources, cross-checked rather than accepted from a single origin.

8.3 Stakes, in the absence of an explicit signal in the raw request, are inferred from the objective's own shape per a rough correspondence this dataset states without treating it as rigid: exhaustive-extraction objectives (Part XVII's code-extraction template) generally carry lower evidentiary stakes per individual data point, since the objective's own completeness check (did the extraction actually cover everything) substitutes for corroboration in a way a research or comparison objective's more interpretive conclusions cannot. Comparison and research objectives generally carry higher evidentiary stakes, since their conclusions involve judgment calls a single source's framing could bias.

8.4 The compiled required-evidence field takes the shape of a declared burden level — light, moderate, heavy — accompanied by any explicit corroboration requirement the raw request stated directly ("verify this against at least two independent sources," "I only need a rough sense, don't verify exhaustively"). This dataset does not compile required evidence into a numeric confidence threshold or a specific DRK-03 CONNECT sufficiency value; that translation, from a declared burden level into the actual sufficiency computation DRK-03's vocabulary performs, belongs to DRK-03 itself, consuming this field as one of its own inputs rather than receiving a pre-computed value this dataset would have had no legitimate basis for computing.

8.5 This Part's boundary against DRK-03, restated with the specificity Part I §1.4 promised: DRK-03's CONNECT vocabulary computes confidence and sufficiency from actual acquired evidence — how many corroborating sources exist, how independent they are, how directly they support the claim in question. This dataset's required-evidence field states, before any evidence has been acquired, what level of eventual corroboration the mission's stakes warrant. The former is a computation over real acquired content; the latter is a declared requirement stated in advance. A compiler that begins computing confidence scores in populating this field, rather than declaring a burden level, has crossed directly into DRK-03's owned territory, and this dataset requires that crossing be treated as a rejection-worthy implementation error per Part XXIII's own failure taxonomy, not a permissible shortcut.

8.6 Required evidence interacts with the confidence-level-requirement field (Part XIII) in a way distinct enough to warrant separating the two fields rather than merging them, even though both ultimately feed DRK-03's evaluation: required evidence declares how much corroboration the mission needs before its output is trustworthy at all; confidence-level requirement declares the minimum computed confidence score the eventual output must clear before it can be delivered as final. A mission can have a heavy required-evidence burden and a moderate confidence-level requirement — demanding thorough corroboration but accepting a moderately confident synthesis of what that corroboration actually shows — or the reverse, a light evidentiary burden with a strict confidence-level requirement, appropriate to a narrow factual question where a single authoritative source, if genuinely authoritative, can support high confidence without needing independent corroboration at all.

8.7 A worked example: the request "compárame estos competidores," already used as a worked example in Part II §2.7, compiles a required-evidence burden of moderate-to-heavy per the correspondence in 8.3, since a competitive comparison's conclusions carry real interpretive weight even absent an explicit stakes signal in the raw request. The compiled field states this burden level and adds an explicit corroboration note derived from the comparison objective's own shape: each competitor's key comparison points should draw from at least two independent sources where the entity's expected-sources prediction (Part V) makes that achievable, rather than accepting a single source's framing of each competitor uncontested.

8.8 A failure mode specific to this field, distinguished here because the STOP #1 history makes it worth flagging explicitly: burden inflation as a hedge, where a compiler defaults every mission toward a heavy evidentiary burden regardless of actual stakes, on the theory that more corroboration is always safer. This is a failure, not a caution, because a heavy burden compiled against a genuinely low-stakes mission inflates the mission's cost and time consumption (per Part X's ceilings) without any corresponding benefit to the Owner, who receives a slower, more expensive result for a question that never warranted the extra corroboration. This dataset requires the burden level to track actual inferred stakes, never to default upward as an unexamined hedge.

8.9 The required-evidence field's relationship to Part XI's stop conditions deserves one explicit note: a mission's stop conditions can include "sufficient corroboration reached," which is only a meaningful stop condition once the required-evidence field has declared what "sufficient" means for this specific mission. A stop condition of this kind, without a populated required-evidence field to check against, is not actually checkable, and Part XI's own treatment of stop conditions returns to this dependency directly.

8.10 One final distinction, easy to lose given how much of this Part has drawn boundaries against what this field does not do: the required-evidence field is not optional or secondary simply because this dataset does not compute against it directly. It is one of the sixteen fields Part III §3.19 requires populated before a contract reaches validated status, and a contract compiled with every other field populated but required evidence left blank is, per that same rule, not a valid Crawl Mission Contract regardless of how complete the rest of the schema is — declaring the evidentiary burden is squarely this dataset's own job, even though computing against that declaration is not.

8.11 One further case worth closing this Part with: a mission whose objective changes mid-compilation because entity resolution (Part IV) or source prediction (Part V) revealed information the original request did not anticipate — a "research this market" request that, once entities resolve, turns out to concern a market with unusually poor public documentation. This dataset requires the required-evidence field to be populated after, not before, entity resolution and source prediction have reached at least a provisional result, precisely so that a genuinely thin information environment can lower the achievable evidentiary burden honestly rather than compiling a heavy burden the mission's own predicted sources have no realistic way to satisfy. A required-evidence field compiled in ignorance of what sources are actually predicted to exist is not a conservative choice, it is a burden set against a fiction, and Part XV's internal-consistency check treats a required-evidence value the mission's own expected-sources field cannot plausibly support as exactly the kind of incoherence that blocks a contract from reaching validated status.

PART VIII FINAL LAW

Required evidence declares the mission's evidentiary burden — how much corroboration its stakes warrant — as an input DRK-03's CONNECT vocabulary consumes, never as a confidence computation this dataset performs itself; a compiler that begins scoring evidence rather than declaring a burden level has left this dataset's boundary, and one that inflates every mission's burden as an unexamined hedge against risk has failed the field's actual purpose just as surely as one that leaves it unpopulated.


PART IX — PERMITTED DOMAINS AND AUTHORIZED AUTHENTICATION: POPULATING, NOT DECIDING

9.1 These two fields sit closest to the future Dataset #16 Authorization, Compliance and Safety engine's own territory, and this Part's discipline mirrors Part VIII's discipline against DRK-03 exactly: this dataset populates the fields, states plainly what basis each population had, and stops there, leaving actual permission adjudication to the engine chartered specifically to make that call — an engine that, per `wiggly-sparking-dolphin.md`'s build order, was deliberately sequenced early (position four of eighteen) precisely so it could bound the acquisition engines from close to their first Part rather than auditing them long after the fact.

9.2 Permitted domains compiles from an explicit list when the raw request names one directly ("only look at their official site, nothing else"), from the entity's own resolved identity when Part IV's entity extraction produced a specific organization or repository whose obvious primary domain the compiler can name directly, and from a broader default scope — "any domain reasonably related to the resolved entities" — when neither a request-level nor an entity-level explicit signal narrows it. This third, broadest case is exactly the case Dataset #16's future adjudication logic exists to narrow at evaluation time; this dataset's job in that case is only to record that no narrower explicit signal existed, not to guess at what a narrower list should contain.

9.3 Authorized authentication compiles almost entirely from explicit signal, because a compiler has no legitimate basis for inferring that a mission is authorized to use any specific credential or authenticated session absent the raw request, the working context, or an explicit Owner-level authorization record actually saying so. Absent such a signal, this field compiles to its safest default: no authenticated access, unauthenticated-only acquisition. This default is not a light one lifted lazily to avoid the harder question — Dataset #01 Part IX's Anti-Detection-Is-Not-Authorization Doctrine already establishes, at the constitutional level this dataset inherits without needing to restate, that acquiring access without genuine authorization is out of bounds regardless of technical feasibility, and this field's conservative default is this dataset's own operational expression of that constitutional floor.

9.4 A worked example distinguishing the two fields' compilation logic: the request "léete este repositorio," read this repository, resolved against a private repository the Owner has explicit access to within the current session's own authenticated context. Permitted domains compiles narrowly and confidently — the repository's own host, nothing broader, since the objective gives no reason to range beyond it. Authorized authentication compiles from the session's own existing authenticated context, explicitly rather than defaulted, since the working context itself supplies the authorization signal directly: the Owner is already authenticated to the resource being requested, and the compiler records that fact as the field's provenance rather than treating it as an assumption.

9.5 Contrast that with the same verb applied to a repository the Owner names but has no evident access to within current working context — "read this repository" naming a private repository with no session-level authentication signal present. Permitted domains still compiles the same way, narrowly to the repository's own host. Authorized authentication compiles to its conservative default, no authenticated access, and this dataset requires the compiler to flag this specific combination — a mission whose entity plausibly requires authentication the mission has not been given — as a candidate for Part XVI's ambiguity-escalation discipline rather than silently dispatching a mission that will predictably fail once the future Dataset #16 or the acquisition engines themselves discover the access gap downstream.

9.6 Both fields interact with the expected-sources field (Part V) in a way worth naming: a predicted source outside the compiled permitted-domains scope is exactly the divergence Part V §5.6 already requires the compiler to surface explicitly rather than silently narrow away. This Part adds the reverse case: a compiled permitted-domains scope broader than anything the expected-sources prediction actually names is not itself a problem requiring escalation — a broad permission with a narrow actual prediction simply means the mission is authorized to range further than it currently expects to need to, which is a safe asymmetry, unlike the reverse case where prediction outruns permission.

9.7 A failure mode specific to this Part, distinct from Part VIII's burden-inflation-as-hedge failure but structurally similar: authorization inflation, where a compiler defaults authorized authentication toward broader access than the working context actually supports, reasoning that broader access makes mission dispatch smoother. This is precisely the failure this field's entire design exists to prevent, and this dataset treats it as more severe than most compilation failures — an authorization field populated beyond what the working context genuinely supports is not merely inefficient the way an inflated evidentiary burden is; it risks dispatching a mission that attempts access it was never actually granted, the exact failure Dataset #01 Part IX's doctrine exists to foreclose.

9.8 A further distinction worth stating: this dataset's compiled authorized-authentication field names which existing authenticated context, if any, the mission may use — it does not itself perform authentication, request new credentials, or negotiate access on the mission's behalf. Those actions, when they are legitimate at all, belong to the acquisition engines themselves (most directly Dataset #01 Part V §5.3's Browser Interaction and Session Governance engine, elaborated in the future Dataset #05) operating within whatever authorization the future Dataset #16 has actually cleared, never to this dataset's own compilation pipeline reaching out to establish access proactively.

9.9 Permitted domains, unlike authorized authentication, can reasonably default toward a broader rather than maximally narrow scope in the ordinary no-explicit-signal case, because domain permission and authentication permission carry different risk profiles: acquiring publicly available content from a domain the compiler did not explicitly narrow to is a materially different risk than acquiring content behind an authentication boundary the mission was never actually granted. This dataset's asymmetric defaults — broad-but-declared for domains, narrow-and-conservative for authentication — reflect that different risk profile rather than an inconsistency in how the two closely related fields are treated.

9.10 With permitted domains and authorized authentication compiled, this dataset has now populated every field bordering the future Dataset #16's territory. Part X, taking up cost and time ceilings next, moves to the fields bordering CO-01's territory, governed by the identical discipline restated in that Part's own terms: declare the boundary, do not adjudicate within it.

9.11 A closing note on how these two fields are recorded once populated, distinct from how they are used: both fields carry their own provenance tag per Part II §2.6, and for permitted domains and authorized authentication specifically, this dataset requires the tag to name not just extracted, resolved, or defaulted, but which of the three inputs 9.2 and 9.3 describe actually produced the value — an explicit request-level list, a resolved entity's own primary domain, working context's existing authenticated session, or the conservative no-signal default. This finer-grained provenance matters more for these two fields than for most others in the schema, because the future Dataset #16's own adjudication is expected to weight an explicitly requested domain differently than a broadly inferred default, and a provenance tag collapsed down to a bare "resolved" or "defaulted" label would strip exactly the distinction that future engine's own evaluation logic needs to make its adjudication meaningfully.

PART IX FINAL LAW

Permitted domains and authorized authentication are populated, never adjudicated — this dataset declares what domain scope and what existing authenticated access a mission's own signals actually support, defaulting broad-but-declared for domains and narrow-and-conservative for authentication given their different risk profiles, and leaves the actual permission decision to the future Dataset #16 Authorization engine; a compiler that defaults authentication toward broader access than working context genuinely supports has not taken a helpful shortcut, it has dispatched a mission toward exactly the unauthorized-access failure this family's constitution already forbids.


PART X — COST AND TIME CEILINGS: DECLARED INPUTS, NOT A COSTING ENGINE

10.1 Maximum cost and maximum time are this dataset's most tightly bounded fields against a sibling system's territory, because CO-01's Operating Economics dataset already owns the estate's general cost/token/resource-accounting vocabulary in full, and Dataset #01's own composition-points list forbids any Crawl OS dataset from reintroducing that vocabulary rather than composing with it. This Part's discipline is therefore the strictest in this dataset: these two fields are declared ceilings, full stop, with no costing logic, no per-engine cost estimation, and no budget-optimization reasoning performed anywhere in this dataset's own text.

10.2 Maximum cost compiles from an explicit ceiling in the raw request when one exists ("no gastes más de X en esto"), from the estate's own general default operating ceiling — a value this dataset references from CO-01 rather than restating, since restating it here would be exactly the duplication this dataset's non-ownership boundary forbids — when the request states no explicit figure, and from an objective-shape-informed adjustment to that default when the objective's own scale clearly warrants a departure from the general default (an exhaustive-extraction mission against a large monorepo plausibly warrants a higher ceiling than the general default contemplates; a narrow single-fact lookup plausibly warrants a lower one). This dataset states the adjustment logic exists without stating the adjustment's actual numeric values, since those values belong to CO-01's own accounting, not to this dataset's declarations about them.

10.3 Maximum time compiles by the identical logic, substituting wall-clock duration for cost: explicit request-level signal first, CO-01's general default second, objective-shape adjustment third. The two fields are compiled independently rather than derived from one another, because cost and time do not move in lockstep — a mission bottlenecked on browser-rendering latency (Dataset #01 Part V §5.3's territory) can consume significant time at modest token cost, while a mission requiring extensive language-model reasoning over a small number of pages can consume significant cost in comparatively little wall-clock time.

10.4 The single discipline this Part holds most strictly, restated from Part I §1.3 in this field's own specific terms: this dataset's compilation logic never estimates the actual cost or time a specific mission will consume once dispatched. It declares a ceiling — a boundary the mission must not exceed — and hands that ceiling to CO-01's own already-sealed accounting and metering machinery, which tracks actual consumption against the declared ceiling as the mission runs. A compiler that begins estimating "this mission will probably cost approximately Y" rather than declaring "this mission may not exceed Y" has crossed from declaring a boundary into performing exactly the cost-forecasting function CO-01 already owns.

10.5 A worked example distinguishing declaration from estimation: the request "constrúyeme un dataset con todas las fuentes," already used as Part VI §6.5's worked example for its exhaustive depth/breadth combination, compiles a maximum-cost and maximum-time field toward the higher end of CO-01's own default range, given the objective's own scale — this dataset states that the objective's exhaustive shape warrants a higher-than-default ceiling, and defers to CO-01's own accounting for what that higher ceiling's actual numeric value should be, rather than this dataset computing a specific number from first principles.

10.6 Cost and time ceilings interact with Part VI's depth/breadth calibration in the reverse direction from how Part VI §6.1 already described the interaction: Part VI treats cost and time as constraints depth/breadth must fit within; this Part treats a mission's inferred scale, partly a function of the depth/breadth values Part VI already calibrated, as one input to how far above CO-01's general default the ceiling fields should be adjusted. This is not circular, because the two Parts run within the same compilation pass in a fixed order — Part VI's initial calibration happens first from objective shape alone, this Part's ceiling adjustment happens next informed by that initial calibration, and Part VI §6.11's own second silent pass then re-examines depth and breadth once every field, including these now-adjusted ceilings, has a value, closing the loop without ever requiring either Part to depend on a value the other has not yet produced.

10.7 A failure mode specific to this Part, distinguished from Part VIII's and Part IX's inflation failures because the direction of harm runs the opposite way: ceiling deflation as a false economy, where a compiler defaults every mission toward the lowest plausible ceiling regardless of objective scale, on the theory that a lower ceiling is always safer. This produces a different, quieter harm than inflation does: a mission compiled with an under-scaled ceiling is likely to hit Part XI's cost-or-time-exhausted stop condition before its objective is actually satisfied, delivering an incomplete result the Owner did not actually ask for, having asked instead for the mission's stated objective to be completed. This dataset requires ceiling calibration to track genuine objective scale in both directions, neither inflating as an unexamined hedge nor deflating as a false economy.

10.8 One further boundary worth stating explicitly, closing this Part's discipline: this dataset's ceiling fields do not themselves enforce anything. Enforcement — halting a mission when it approaches or reaches its declared ceiling — is CO-01's own already-sealed machinery acting on the declared value this dataset compiled, the identical relationship Part VII §7.3 already describes between this dataset's freshness field and Dataset #10's freshness-window enforcement. A declared ceiling this dataset compiles and a ceiling actually being enforced during a live mission are two different moments performed by two different systems, connected only by the value this dataset hands downstream.

10.9 Maximum cost and maximum time, once compiled, feed directly into Part XI's stop-condition taxonomy as one of several named termination triggers — "ceiling reached" is a stop condition only because these two fields exist to be reached in the first place, exactly the same dependency Part VIII §8.9 already noted between the required-evidence field and a "sufficient corroboration reached" stop condition. Part XI takes up the full taxonomy next.

10.10 With cost and time ceilings compiled, this dataset has now populated every field bordering CO-01's owned territory with the same discipline it already applied to DRK-03's and the future Dataset #16's territory: declare the value, cite the boundary, and defer computation, adjudication, or enforcement entirely to the system that actually owns it.

10.11 A last distinction worth drawing explicitly, since it is easy to lose amid this Part's repeated emphasis on what these fields do not do: declining to estimate is not the same as declining to reason about scale. This dataset's compilation logic is fully entitled to reason qualitatively about whether a mission's objective, depth, and breadth suggest a ceiling toward the higher or lower end of CO-01's own default range — 10.2 and 10.3 already require exactly that reasoning — the boundary this Part holds is narrower and more specific than "never think about cost at all": it forbids only the specific act of producing a numeric cost or time figure from first principles, the one computation CO-01's own accounting machinery is positioned to perform correctly and this dataset's compilation pipeline is not, lacking as it does any visibility into actual token pricing, actual per-engine latency, or actual current load on the estate's own execution substrate.

PART X FINAL LAW

Maximum cost and maximum time are declared ceilings this dataset hands to CO-01's already-sealed accounting and enforcement machinery, never values this dataset estimates, forecasts, or enforces itself — a compiler that begins predicting actual consumption rather than declaring a boundary has duplicated CO-01's owned vocabulary, and one that deflates every ceiling as a reflexive false economy has produced missions that stop short of their own stated objective for no reason the objective itself ever demanded.


PART XI — STOP CONDITIONS: THE TERMINATION TAXONOMY

11.1 Every Mission Contract this dataset compiles carries at least one stop condition, and this Part exists to name the full taxonomy of triggers a stop condition can be drawn from, precisely so that Part XV's state machine can enforce, as one of its own validation checks, that no contract reaches validated status with an empty or incoherent stop-conditions field — the specific invariant this dataset's own contract file already states plainly: a contract dispatched with no defined way to end is an unbounded-mission risk this dataset exists to prevent, not merely to discourage.

11.2 Target satisfied. The mission's objective, as compiled in Part IV, has been achieved to the standard the required-evidence field (Part VIII) and confidence-level-requirement field (Part XIII) jointly establish. This is the ordinary, successful termination case, and every compiled contract carries it as a stop condition regardless of what other conditions also apply — a mission with no path to a successful completion state is not a mission this dataset should compile at all, since Part XIX's rejection taxonomy exists precisely to catch requests where no viable target-satisfied path exists.

11.3 Ceiling reached. Either the maximum-cost or maximum-time field (Part X) has been reached before the target-satisfied condition triggered. This stop condition exists on every compiled contract as a safety bound, independent of whether the mission's objective shape makes reaching a ceiling likely — even a narrow, cheap, fast mission carries this stop condition, because an unexpected acquisition difficulty (a source proving far harder to extract from than predicted, an authentication wall requiring extensive negotiation) can push even a modest mission toward its ceiling, and the mission must have a defined behavior for that case regardless of how unlikely it seemed at compilation time.

11.4 Dead end encountered. Every predicted source (Part V) has been exhausted without producing evidence sufficient to satisfy the required-evidence field, and no further plausible source exists to pursue. This differs from ceiling reached in an important way: a dead end can trigger well before either ceiling is reached, and represents the mission's own conclusion that continued pursuit is unlikely to succeed rather than an external resource limit forcing termination. A compiled contract's dead-end stop condition includes, where the objective shape warrants it, an explicit definition of what "exhausted" means for this specific mission — a single-entity comprehension mission's predicted sources are exhausted once the entity itself has been fully covered; a broad market-research mission's predicted sources may never be fully exhausted in any strict sense, and for that objective shape this dataset compiles the dead-end condition against a marginal-return threshold instead — continued acquisition producing negligible new corroboration relative to what has already been found.

11.5 Contradiction discovered. Acquired evidence directly and irreconcilably contradicts other acquired evidence bearing on the same claim, in a way the mission's own objective has no mechanism to resolve. This stop condition does not apply to every mission — an exhaustive-extraction objective against source code has essentially no room for this kind of contradiction to arise — but applies centrally to comparison and research objectives, where two sources disagreeing about a material fact is a real and common outcome. This dataset compiles this stop condition not as an instruction to silently pick one contradicting source over the other, but as an instruction to halt and surface the contradiction explicitly as part of the mission's eventual output, consistent with Dataset #01 Part VIII's prohibition against Markdown-as-proof — a mission that quietly resolved a genuine contradiction without surfacing it would be presenting confident output over a claim the evidence itself does not actually support cleanly.

11.6 Explicit Owner abort. The Owner, or an authorized upstream caller, explicitly cancels the mission before any other stop condition would have triggered. Every compiled contract carries this stop condition implicitly, without requiring it to be separately populated, because Dataset #01 Part IV §4.8 already places the general authority to halt an in-progress mission outside Crawl OS's own territory — Crawl OS's missions are themselves scheduled and, by direct extension, cancellable by the broader estate systems that schedule them, and this dataset's stop-conditions field records the mission-internal triggers, not the estate-external cancellation authority that always sits above any of them.

11.7 A worked example combining several of these: the request "vigila cuándo cambia," already used in Part VII §7.5 as a freshness/update-policy worked example, compiles stop conditions that look different from a single-pass mission's. Target-satisfied does not apply in the ordinary sense, since a continuous-recurrence mission per Part VII's update-policy compilation has no single completion state — instead, this dataset compiles a recurring mission's target-satisfied condition as "never, by design," relying instead on ceiling-reached (a recurring mission's cumulative cost across many passes eventually reaches its ceiling even if each individual pass is cheap) and explicit Owner abort as the mission's actual termination paths, since a monitoring mission's own nature is to continue until something external ends it.

11.8 This Part's taxonomy interacts with Part XIX's rejection taxonomy in a specific, bounded way worth distinguishing clearly: a stop condition governs how an already-dispatched, already-running mission ends; a rejection (Part XIX) governs a request the compiler determined could never be compiled into a viable contract at all. A request compiled successfully, dispatched, and then immediately hitting its dead-end stop condition on the very first acquisition attempt is not a rejection — it is a mission that ran, per its own compiled stop conditions, and concluded without satisfying its target. These are different outcomes recorded differently in the compilation audit trail (Part XXII), and this dataset requires the distinction to be preserved rather than treated as functionally equivalent simply because both, from the Owner's perspective, might look like "it didn't work."

11.9 A failure mode specific to this Part: an incomplete stop-condition set, where a compiler populates only target-satisfied and ceiling-reached, omitting dead-end and contradiction-discovered even when the objective shape clearly warrants them. A comparison or research mission compiled without a contradiction-discovered stop condition risks running to its cost or time ceiling attempting to force a resolution to a genuine disagreement between sources, rather than halting to surface that disagreement honestly the moment it is found — wasting the mission's remaining budget on a problem no further acquisition can actually solve.

11.10 With stop conditions taxonomized, this dataset has now completed the block of fields most directly governing a mission's own internal dynamics — freshness and update policy (Part VII), required evidence (Part VIII), permitted domains and authorized authentication (Part IX), cost and time ceilings (Part X), and now stop conditions together form the mission's full operating envelope. Part XII, taking up output schema next, turns from how the mission runs to what the mission is expected to eventually produce.

11.11 One further note on how multiple stop conditions coexist on a single compiled contract, since nothing in 11.2 through 11.6 states how a mission behaves once more than one condition becomes simultaneously eligible to trigger. This dataset requires the compiler to record stop conditions as an unordered set rather than a prioritized list — whichever condition the dispatched mission actually reaches first, in real execution, is the one that governs that mission's termination, and no compiled contract privileges one populated condition over another in advance of execution actually determining which one fires. The one exception, restated from 11.6, is explicit Owner abort, which this dataset treats as always available and always capable of superseding whatever other condition the mission was approaching, precisely because that authority sits above the mission's own compiled contract entirely rather than being one more condition compiled alongside the rest.

PART XI FINAL LAW

A compiled Mission Contract's stop conditions are drawn from a fixed taxonomy — target satisfied, ceiling reached, dead end encountered, contradiction discovered, and the ever-present explicit Owner abort — and every contract carries at least target-satisfied or its recurring-mission equivalent alongside ceiling-reached as a safety bound, because a mission dispatched with no defined way to end is not a bounded unit of work this dataset can honestly call compiled, regardless of how complete its other fifteen fields may be.


PART XII — OUTPUT SCHEMA DECLARATION

12.1 The output-schema field is the first field in this dataset's sequence that looks forward past the mission's own execution to the deliverable the mission's requester will actually receive, and it exists because Dataset #01 Part V §5.9's Knowledge Distillation Fabric needs to know, before acquisition even begins, what shape its own eventual work should take rather than inferring it after the fact from whatever content happened to accumulate.

12.2 Output schema compiles primarily from the objective field's own shape, per a correspondence closely related to but distinct from the depth/breadth correspondence Part VI already established: a comprehension objective defaults toward a narrative report; an exhaustive-extraction objective defaults toward a structured, machine-readable enumeration rather than prose; a comparison objective defaults toward a structured comparison table or equivalent side-by-side structure; a monitoring objective defaults toward a change-alert notification format rather than either a report or an enumeration, since its deliverable is not a single document but a recurring signal; a dataset-construction objective defaults toward the specific structured dataset format the request's own wording implies.

12.3 An explicit output-format signal in the raw request always overrides the objective-shape default — "dame un resumen en dos párrafos," give me a two-paragraph summary, compiles an output schema far narrower than the comprehension objective's own default narrative-report shape would otherwise suggest, and this dataset requires an explicit signal of this kind to be captured during Part II's parsing stage, exactly as Part VI §6.6 already requires for urgency signals, since an output-schema override is easy to lose if it is not captured at the same early stage where the rest of the request's explicit qualifying detail is captured.

12.4 The compiled output-schema field is expressed structurally — the deliverable's shape and major sections, not its literal final content — since this dataset compiles before any acquisition has happened and has no legitimate basis for anticipating a report's actual substance. A comparison-objective mission's output schema, for instance, names the dimensions the comparison will be structured around (the entities being compared, the attributes compared across them) without asserting what the comparison will actually conclude, exactly the same population-not-adjudication discipline Part I §1.3 already establishes for every other field bordering a decision this dataset does not itself make.

12.5 Output schema interacts with the required-evidence field (Part VIII) in one specific way worth naming: a heavier evidentiary burden generally implies an output schema that surfaces its own corroboration explicitly — citations, source attribution, an explicit confidence caveat where the confidence-level requirement (Part XIII) was not fully met — while a lighter evidentiary burden's output schema can present its conclusion more directly without the same apparatus. This dataset does not mandate a fixed citation format, since that belongs to the future Dataset #01 Part V §5.9 Knowledge Distillation Fabric's own implementation, but it does require the compiled output-schema field to note whether corroboration-surfacing apparatus is expected at all, given the mission's own required-evidence burden.

12.6 A worked example: the request "extrae todas las funciones," already used repeatedly across this dataset as a worked example for its exhaustive-extraction shape, compiles an output schema of a structured enumeration — one entry per function, carrying whatever attributes (name, location, signature) the entity type and downstream consumer plausibly need — explicitly distinct from a narrative report walking through the codebase in prose. This dataset states the schema's structural shape without prescribing the exact serialization format (JSON, a structured markdown table, a database record) that belongs to the downstream Knowledge Distillation Fabric's own implementation choices.

12.7 A failure mode specific to this field: schema mismatch with objective, where a compiler defaults toward a narrative report regardless of the objective's actual shape, on the theory that prose is a safe universal default. This is a failure because an exhaustive-extraction mission delivered as narrative prose, rather than a structured enumeration, forces the requester to re-extract the very structure the mission was supposed to have already produced — the deliverable technically exists but fails the objective's own actual purpose. This dataset requires output-schema compilation to track the objective's real shape, never to default toward prose as a one-size-fits-all safety net.

12.8 Output schema also interacts with the preservation-obligation field (Part XIV) in a way worth flagging before that Part's own full treatment: a mission whose output schema is a recurring change-alert (per the monitoring-objective default in 12.2) implies a preservation question distinct from a single-report mission's — does each individual alert need to be preserved as its own record, or only the current state. This dataset does not resolve that question itself; it notes the interaction exists so Part XIV's own compilation logic accounts for it rather than treating every mission's preservation needs as uniform regardless of output schema.

12.9 A boundary worth restating in this field's own terms: this dataset's output-schema field declares the deliverable's expected shape; it does not itself assemble, format, or produce that deliverable. Production belongs entirely to the downstream Knowledge Distillation Fabric, consuming this field as one of its own inputs alongside the acquired evidence itself, exactly the same declaration-not-execution boundary every prior field-population Part in this dataset has already drawn against its own respective downstream consumer.

12.10 With output schema compiled, this dataset turns in Part XIII to the confidence-level-requirement field — the last of the fields bordering DRK-03's territory, and the field most directly determining whether a mission's eventual output, once produced against this Part's declared schema, can actually be delivered as final rather than provisional.

12.11 One closing case worth naming: a mission whose compound-objective status, per Part IV §4.7, produces multiple independent contracts from a single originating request. Each resulting contract compiles its own independent output-schema field, matched to its own independent objective, rather than the two contracts inheriting a shared schema derived from the original request's overall shape. A "read this repository and tell me if it duplicates any existing Power Pack module" request, already used as Part IV §4.7's own worked example, produces one contract with a narrative-report output schema for the comprehension half and a second contract with a structured-comparison output schema for the duplication-check half — two schemas, not one, precisely because Part XV §15.5 already establishes that compound-objective contracts proceed through FIELD-ASSIGNING as fully independent instances rather than a single instance sharing derived values.

12.12 Output schema, like every other field this dataset compiles, is subject to Part XVIII's revision discipline once a mission is already dispatched — a mid-mission request to change how the eventual deliverable should be shaped ("actually, just give me a table instead of a report") is a legitimate amendment, not a new mission, provided the underlying objective and entities remain unchanged. This dataset requires such a revision to update only the output-schema field and to leave every other already-populated field untouched, since a formatting preference expressed mid-mission carries no implication about the mission's actual objective, evidentiary needs, or any of the other fifteen fields this dataset compiles independently of how the result will eventually be presented.

PART XII FINAL LAW

Output schema declares the deliverable's expected structural shape, compiled from the objective's own shape and any explicit request-level override, without asserting or anticipating the deliverable's actual content — a compiler that defaults toward a uniform narrative-report shape regardless of the objective's real needs has produced a technically-complete deliverable that fails the mission's own actual purpose, the same failure this dataset's every other field-population discipline exists to prevent in its own specific terms.


PART XIII — THE CONFIDENCE-LEVEL REQUIREMENT FIELD

13.1 This field closes this dataset's DRK-03-bordering block, begun in Part VIII, and its own boundary is stated with the same precision Part VIII already established: the confidence-level-requirement field declares a minimum threshold a mission's eventual output must clear before delivery as final; it does not compute the confidence value that gets checked against that threshold, a computation DRK-03's CONNECT vocabulary owns entirely.

13.2 Confidence-level requirement compiles primarily from the same stakes assessment Part VIII §8.2 and §8.3 already establish for required evidence, since the two fields, though independent per Part VIII §8.6, draw from a common underlying signal: how consequential the mission's output is to whatever decision or action it will inform. A low-stakes mission's confidence-level requirement compiles low — a moderately confident answer is good enough when little rides on it. A high-stakes mission's confidence-level requirement compiles high — the output should not be delivered as final unless DRK-03's own computation clears a substantially higher bar.

13.3 The field's value is expressed as a qualitative tier — low, moderate, high, near-certain — rather than a specific numeric threshold, for the identical reason Part VI §6.8 already gives for expressing depth and breadth as qualitative bands: this dataset's compilation pipeline has no visibility into DRK-03's own internal numeric scale, and translating a qualitative stakes assessment into a specific number belongs to DRK-03's own evaluation logic, consuming this dataset's qualitative tier as its input rather than receiving a number this dataset would have had no legitimate basis for producing.

13.4 A mission whose eventual output fails to clear its compiled confidence-level requirement is not, by that fact alone, a failed mission in the sense Part XI's stop conditions describe. This dataset requires such an output to be delivered as explicitly provisional rather than withheld entirely or silently delivered as though it had cleared the bar — the Owner or upstream caller receiving a provisional output, clearly marked as such, retains the ability to decide whether a lower-confidence answer is still useful to them, a decision this dataset's own compilation pipeline has no basis for making on their behalf. Delivering a provisional output as though it were final would violate the same honesty discipline Dataset #01 Part VIII's prohibition against Markdown-as-proof already establishes at the constitutional level.

13.5 A worked example: the request "compárame estos competidores," used throughout this dataset as its recurring comparison-objective worked example, compiles a confidence-level requirement of moderate-to-high, tracking the same stakes assessment that already produced its moderate-to-heavy required-evidence burden in Part VIII §8.7. If DRK-03's eventual computation over the mission's actually-acquired evidence produces only a low-confidence result — perhaps because the predicted sources (Part V) proved thinner than expected — the compiled contract's confidence-level requirement is what determines that the resulting comparison must be delivered as provisional, explicitly flagged as resting on weaker corroboration than the mission's own stated stakes would ideally warrant, rather than presented with unearned certainty.

13.6 Confidence-level requirement interacts with the output-schema field (Part XII) in a specific, concrete way: a compiled output schema for any mission whose confidence-level requirement is anything above the lowest tier must include, per Part 12.5's already-stated corroboration-surfacing apparatus, an explicit mechanism for marking the output provisional if DRK-03's actual computation falls short. An output schema that has no way to represent "this did not clear its own confidence bar" is incompletely compiled against a mission whose confidence-level requirement this Part has populated at anything above the lowest tier, and this dataset treats that combination as an internal-consistency failure Part XV's state machine must catch.

13.7 A failure mode specific to this field, worth naming given how closely it parallels Part VIII §8.8's burden-inflation failure: confidence-tier inflation, where a compiler defaults every mission toward a high or near-certain confidence-level requirement regardless of actual stakes. Beyond the same cost and time consequences Part VIII already names for evidentiary-burden inflation (a higher confidence-level requirement generally demands more corroboration to satisfy, feeding back into required evidence and hence into depth/breadth per Part VI §6.4), this specific inflation carries an additional cost: a mission compiled with an unrealistically high confidence-level requirement against a genuinely thin information environment is disproportionately likely to be delivered as provisional regardless of how well the acquisition actually went, since no realistic amount of corroboration against a thin source set will clear an artificially inflated bar.

13.8 This dataset requires the confidence-level-requirement field, like required evidence in Part VIII §8.11, to be compiled with awareness of what the expected-sources field (Part V) actually predicts is achievable, rather than compiled in isolation from it. A mission whose predicted sources are uniformly weak (per Part V §5.4's weakest-provenance case) should not, absent an explicit Owner-stated requirement overriding this default, compile toward a near-certain confidence-level requirement the mission's own predicted source quality gives no realistic path to satisfying.

13.9 A boundary restated one final time in this field's own specific terms, closing this dataset's full DRK-03-bordering discipline begun in Part VIII: required evidence and confidence-level requirement together are this dataset's complete interface to DRK-03's CONNECT vocabulary — two declared inputs, populated from the mission's own inferred stakes and predicted source quality, handed downstream for DRK-03's own computation to evaluate actual acquired evidence against. No future elaboration of this dataset may add a third field, or expand either of these two, into anything resembling an independent confidence computation; DRK-03 owns that computation in full, now and in every future revision of this schema.

13.10 With confidence-level requirement compiled, this dataset turns in Part XIV to the preservation-obligation field — the last of the sixteen individual field-population Parts, and the field most directly interfacing with Dataset #10's already-sealed retention machinery.

13.11 A last worked distinction, closing this dataset's DRK-03-bordering block in full: consider two missions with identical required-evidence burdens (Part VIII) but different confidence-level requirements — one accepting a moderate tier, one demanding near-certain. Both missions may acquire the identical set of corroborating sources, since required evidence governs how much corroboration is gathered, not how confidently DRK-03's own computation ultimately reads it. The near-certain mission is simply more likely to fall short of its own bar and be delivered provisional, per 13.4, even against evidence a moderate-tier mission would have accepted outright as final. This is the clearest illustration this dataset offers of why the two fields, though drawing on a common stakes signal per 13.2, must remain independently compiled rather than collapsed into one: they answer different questions about the same underlying evidence.

13.12 A revision note mirroring Part 12.12's for output schema: an Owner mid-mission stating "actually, I need to be more sure about this before you finish" is a legitimate amendment to the confidence-level-requirement field alone, raising the tier without necessarily changing required evidence, entities, or any other already-populated field. This dataset requires such a revision to be checked, per Part XV §15.8's re-entry discipline, against whatever acquisition has already occurred — a mission already well past its original tier's corroboration needs may satisfy a raised tier with little or no additional acquisition, while one raised early still has the opportunity to adjust its remaining acquisition effort accordingly rather than discovering the shortfall only at delivery.

PART XIII FINAL LAW

Confidence-level requirement declares the minimum threshold a mission's output must clear before delivery as final, compiled from the same stakes assessment that governs required evidence and expressed as a qualitative tier DRK-03's own computation evaluates against — an output that fails to clear its own compiled requirement is delivered as explicitly provisional, never silently as final, because presenting an uncertain conclusion with unearned certainty is the exact failure this dataset's every honesty discipline, inherited from Dataset #01's own constitutional floor, exists to prevent.


PART XIV — THE PRESERVATION OBLIGATION FIELD

14.1 Preservation obligation is the last of the sixteen fields Part III enumerated, and it closes this dataset's field-population sequence by declaring whether, and for how long, a mission's acquired evidence must survive after the mission itself concludes — a question distinct from freshness (Part VII, which governs how current evidence must be during the mission) and distinct from the required-evidence field (Part VIII, which governs how much corroboration the mission needs while it runs).

14.2 Preservation obligation compiles from three considerations, weighed together rather than in the strict priority order Part V's source-prediction inputs used: an explicit retention instruction in the raw request ("guarda todo esto para referencia futura," keep all of this for future reference, or conversely an explicit signal that the mission's evidence is disposable once the deliverable is produced); the objective's own shape, since some objective types carry an implicit retention need independent of explicit instruction (a dispute-relevant research mission implies a stronger default preservation need than a casual comparison mission); and the mission's own confidence-level requirement (Part XIII), since a mission whose output was delivered as provisional per Part 13.4 carries a stronger case for preserving its underlying evidence, so that a future re-evaluation, once stronger corroboration becomes available, can be checked against what was actually found rather than requiring a full re-acquisition from nothing.

14.3 The compiled preservation-obligation field takes the shape of a declared retention category — this dataset does not invent its own retention taxonomy from scratch but instead maps directly onto the three retention categories Dataset #10 Part XII's already-sealed freshness-and-retention machinery already defines, since inventing a parallel taxonomy here would duplicate territory this dataset does not own. This field states which of Dataset #10's existing categories a given mission's evidence belongs to; it does not define what those categories mean or how they are enforced, both of which remain entirely Dataset #10's own territory.

14.4 A worked example: the request "vigila cuándo cambia," used throughout this dataset as its recurring monitoring-objective worked example, compiles a preservation obligation stronger than a single-pass mission's default, since a change-monitoring mission's own value depends on being able to compare a current acquisition against what was previously found — without preserved prior evidence, the mission's own change-detection logic (belonging to the future Dataset #11) would have nothing to compare the current state against. This dataset compiles this mission's preservation obligation toward Dataset #10's retention category appropriate to ongoing comparison needs, deferring to Dataset #10's own definition of what that category actually entails operationally.

14.5 Preservation obligation's relationship to the output-schema field (Part XII), flagged in advance at Part 12.8, is resolved here: a recurring monitoring mission's preservation obligation covers each individual acquisition pass, not only the current state, precisely because the change-detection comparison Part 14.4 describes requires a prior state to compare against — a preservation obligation covering only "the most recent state" would make change detection impossible on the very next pass, since nothing would remain to compare the new state to.

14.6 A failure mode specific to this field: preservation obligation defaulted too low, where a compiler compiles every mission toward the lightest retention category regardless of objective shape or confidence-level outcome, on the theory that minimal retention is always the safer, more conservative default. This inverts the actual risk this field exists to manage — Part 14.2's third consideration exists precisely because a provisional-confidence output with no preserved underlying evidence cannot later be strengthened without a full re-acquisition, an outcome considerably more costly than the modest storage cost of preserving evidence that later analysis might not end up needing. This dataset requires preservation-obligation compilation to weigh this asymmetry explicitly rather than defaulting reflexively toward minimal retention.

14.7 The reverse failure deserves equal naming: preservation obligation inflated across every mission regardless of actual need, treating maximal retention as a costless safety margin. Dataset #10 Part XII's own retention machinery already establishes that retention is not costless — stored evidence consumes the object store's own capacity and remains subject to Dataset #10's own tamper-evidence and access-control obligations for as long as it is retained — and this dataset's own preservation-obligation field, like every other field this Part's companions have already covered, must track genuine mission need rather than defaulting upward as an unexamined hedge, the identical discipline Part VIII §8.8 and Part IX §9.7 already establish for their own respective fields.

14.8 A boundary restated in this field's own specific terms, closing this dataset's engagement with Dataset #10's territory: this dataset's preservation-obligation field declares which of Dataset #10's existing retention categories a mission's evidence belongs to; it does not implement retention, enforce expiry, manage storage capacity, or perform any of the operational mechanics Dataset #10 Part XII already owns in full. A future elaboration of this field that begins defining new retention categories, rather than mapping onto Dataset #10's existing ones, has duplicated territory this dataset explicitly does not own.

14.9 With preservation obligation compiled, all sixteen of Part III's fields have now received their own dedicated compilation treatment across Parts IV through XIV. Part XV, taking up the compilation state machine next, describes how these sixteen independently-populated fields come together into a single validated, dispatchable contract — the mechanical backbone Part I §1.6 already flagged as the discipline a reader confused about compilation ordering should read before continuing further.

14.10 One closing observation ties this Part back to the dataset's opening boundary in Part I §1.3: of the sixteen fields this dataset owns, preservation obligation is the one whose downstream consumer — Dataset #10 — is, unlike the future Dataset #16 or CO-01, an already-sealed, fully operational sibling dataset. This asymmetry is worth naming because it means this field's compilation logic can be, and is, checked against real, already-written machinery rather than only against a future engine's anticipated needs, and any future revision to this field's compilation logic should verify its continued fit against Dataset #10's actual Part XII text rather than against an assumed or remembered description of it.

14.11 With this Part closing all sixteen of Part III's fields, one summary observation is worth stating before Part XV turns to the state machine that assembles them: across Parts IV through XIV, the recurring failure pattern this dataset has named again and again — inflation as an unexamined hedge, deflation as a false economy, silent fabrication in place of honest defaulting or escalation — is the same underlying failure recurring in each field's own specific vocabulary. A compiler that internalizes only one field's version of this discipline while treating the others as unrelated has not actually learned this dataset's central lesson; the sixteen fields are different in content but identical in the discipline each one demands.

14.12 A revision note completing the pattern Parts 12.12 and 13.12 already establish for their own fields: preservation obligation can also be amended mid-mission or even after a mission concludes, when an Owner realizes evidence should have been retained longer than the mission's original compilation anticipated. Unlike output schema or confidence-level requirement, this amendment interacts directly with Dataset #10's own already-sealed retention machinery rather than only with this dataset's compilation pipeline — a preservation-obligation upgrade issued after the original retention window has already lapsed cannot recover evidence Dataset #10's own expiry logic has already discarded, and this dataset requires that limitation to be surfaced honestly to whoever requests the amendment rather than accepted as though the upgrade were always retroactively effective.

PART XIV FINAL LAW

Preservation obligation declares which of Dataset #10's already-sealed retention categories a mission's evidence belongs to, compiled from explicit request signal, objective shape, and the mission's own confidence-level outcome — never a retention taxonomy this dataset invents itself, and never defaulted reflexively toward either minimal retention, which forecloses future strengthening of a provisional result, or maximal retention, which incurs a real and unnecessary cost Dataset #10's own machinery must then carry indefinitely.


PART XV — THE COMPILATION STATE MACHINE

15.1 This Part is the mechanical backbone every prior field-population Part has referenced but not fully described: the sequence of states a Mission Contract passes through from the moment a raw request arrives to the moment a fully populated, internally consistent contract is handed off to a downstream engine. Every field-specific mechanic Parts IV through XIV describe operates within this state machine's own structure, and this Part exists to make that structure explicit and complete in one place.

15.2 The states, in strict order: RECEIVED (the raw request has arrived, nothing yet parsed), PARSING (Part II §2.1's grammatical-shape identification is underway), NORMALIZING (Part II §2.2's resolution against estate knowledge is underway), FIELD-ASSIGNING (Part II §2.3's mapping onto the sixteen-field schema is underway, encompassing every field-specific mechanic Parts IV through XIV describe), INTERNALLY-CHECKING (the internal-consistency check Part III §3.18 and §3.19 require, plus Part VI §6.11's second silent depth/breadth pass, plus every other cross-field consistency check named throughout Parts IV through XIV — Part VII §7.7's ceiling-versus-recurrence check, Part XI §11.1's stop-condition-completeness check, Part XIII §13.6's confidence-tier-versus-output-schema check, and any others a given compiled contract's specific field values happen to trigger), and finally exactly one of two terminal states: VALIDATED (every field populated, every consistency check passed, the contract is ready for Part XXI's handoff) or REJECTED (Part XIX's rejection taxonomy applies, and no viable contract will be produced from this compilation attempt).

15.3 A contract does not skip states, and does not regress to an earlier state once it has advanced past it within a single compilation attempt — this is the same no-skip discipline Dataset #10 Part II already established for the Evidence Object's own lifecycle state machine, adopted here for the identical reason: a state machine that permits skipping or silent regression cannot be audited after the fact, since the compilation audit trail (Part XXII) depends on every contract's state history being a complete, honest record of what actually happened during compilation.

15.4 One state transition deserves its own explicit note, since Part XVI's ambiguity-handling discipline depends on it: FIELD-ASSIGNING can transition not only forward to INTERNALLY-CHECKING but sideways to an ESCALATION-PENDING sub-state when a field cannot be populated with any confidence at all, exactly the case Part IV §4.6 already worked through for the "investiga este mercado" example. ESCALATION-PENDING is not itself a terminal state — it awaits the clarification Part XVI's discipline requests, and once that clarification arrives, the contract resumes FIELD-ASSIGNING from wherever it paused rather than restarting the entire pipeline from RECEIVED.

15.5 Compound objectives, per Part IV §4.7, produce a specific branching behavior at the FIELD-ASSIGNING state: rather than one contract proceeding linearly through the remaining states, the state machine forks into multiple independent contract instances, each proceeding through FIELD-ASSIGNING, INTERNALLY-CHECKING, and its own terminal state independently, sharing only their common RECEIVED, PARSING, and NORMALIZING history in the compilation audit trail. A compound request's two resulting contracts can reach different terminal states — one VALIDATED, one REJECTED — without either outcome affecting the other, since Part XIX's rejection taxonomy applies per-contract, not per-originating-request.

15.6 The INTERNALLY-CHECKING state's own completeness deserves emphasis: this dataset requires every cross-field consistency check named anywhere in Parts III through XIV to run at this state, not merely the ones a given implementation happens to remember. This is worth stating explicitly because the risk this dataset's own contract file names under non-ownership — silently drifting away from a discipline stated once in one Part's prose without a mechanism forcing it to actually run — applies to this dataset's own internal-consistency checks just as much as to the boundary disciplines Parts VIII through X hold against DRK-03, the future Dataset #16, and CO-01. A future implementation of `modules/crawl_os/mission_compiler.py` that runs only some of these named checks, rather than all of them, has not implemented Part XV's state machine faithfully regardless of how much of the rest of this dataset it otherwise honors.

15.7 A worked trace, drawing together several prior worked examples: "compárame estos competidores" arrives (RECEIVED), its verb and qualifying phrase are identified (PARSING), "these competitors" resolves against three prior missions in working context (NORMALIZING), all sixteen fields populate per Parts IV through XIV's own mechanics including Part VI §6.11's later silent recalibration (FIELD-ASSIGNING), the compiled contract passes every named consistency check with no incoherence detected (INTERNALLY-CHECKING), and the contract reaches VALIDATED, ready for Part XXI's handoff. Contrast "investiga este mercado" with no resolvable working context: RECEIVED, PARSING succeeds, NORMALIZING fails to resolve the market entity, FIELD-ASSIGNING proceeds on every other field but the entities field enters ESCALATION-PENDING, and the contract's overall state remains suspended until Part XVI's clarification request is answered.

15.8 This Part's own boundary: the state machine governs how a single contract moves through compilation; it does not govern what happens to a contract after it reaches VALIDATED and is dispatched to a downstream engine, nor does it govern Part XVIII's revision-and-amendment process, which operates on an already-VALIDATED-and-dispatched contract rather than one still moving through this Part's own states. A revised contract, per Part XVIII's own discipline, re-enters this state machine at whichever state the revision actually requires re-running — a minor field correction might re-enter at FIELD-ASSIGNING for just the corrected field, while a genuine scope change might re-enter as far back as PARSING.

15.9 A failure mode specific to this Part: state-skipping under time pressure, where an implementation, in the name of efficiency, collapses INTERNALLY-CHECKING into FIELD-ASSIGNING, checking each field's consistency as it is populated rather than running the full cross-field check once every field exists. This is a failure specifically because several of the named consistency checks — Part VI §6.11's second depth/breadth pass chief among them — depend on every other field already having a value, and a check run before that precondition is met is not the check this dataset actually specifies, regardless of how similar its name might be.

15.10 With the state machine fully described, this dataset turns in Part XVI to the ambiguity-handling discipline this Part's own ESCALATION-PENDING sub-state has referenced but not yet fully specified — the mechanics of when and how a compiler actually escalates, rather than merely the state-machine slot escalation occupies.

15.11 A last note on why this dataset specifies a state machine at all rather than simply describing compilation as a loose sequence of activities: a named, ordered set of states is what makes the compilation audit trail (Part XXII) an actual audit trail rather than a narrative reconstruction assembled after the fact from whatever logging happened to exist. A future implementation that tracks state transitions explicitly, per this Part's own five-plus-two-terminal enumeration, produces an audit trail an external reviewer can verify mechanically; one that only logs prose descriptions of what happened produces a trail that requires trusting the implementation's own account of itself — exactly the trust this dataset's provenance-tagging discipline (Part II §2.6) exists to make unnecessary everywhere else in this schema, and which this Part's own state machine extends to the compilation process itself.

PART XV FINAL LAW

A Mission Contract moves through a fixed, no-skip, no-silent-regression sequence of states — RECEIVED, PARSING, NORMALIZING, FIELD-ASSIGNING, INTERNALLY-CHECKING, and exactly one of VALIDATED or REJECTED, with an ESCALATION-PENDING sub-state available from FIELD-ASSIGNING when a field cannot be populated at all — and every cross-field consistency check this dataset names anywhere in its own text must run at INTERNALLY-CHECKING in full, because a state machine that runs only the checks an implementation happens to remember is not, regardless of its name, the discipline this Part actually specifies.


PART XVI — AMBIGUITY AND UNDERSPECIFICATION HANDLING

16.1 Every prior Part in this dataset has referenced escalation as the response to a field that cannot be populated with confidence, without yet specifying what escalation actually means, when it is warranted, and how it differs from the ordinary defaulting this dataset's other fields rely on constantly. This Part closes that gap.

16.2 The distinction this Part exists to draw precisely: defaulting (Part II §2.6's third population source) applies a documented fallback value when no better signal exists, and proceeds with compilation using that value. Escalation, by contrast, applies when no documented default exists that would not risk compiling a contract against a materially wrong assumption — the "investiga este mercado" case Part IV §4.6 already worked through, where no default market identity exists that would not simply be a guess dressed up as a value. The dividing line is not "how confident is the compiler," a continuous and subjective measure this dataset does not rely on, but a discrete, checkable question this dataset requires every field-population mechanic to ask: does a documented default exist for this field, given the objective and entity context available, or does none exist. Parts IV through XIV each specify, for their own field, what the documented defaults actually are; a case falling outside every documented default for a given field is, by this Part's own definition, an escalation case rather than a defaulting case.

16.3 Escalation, once triggered, produces a specific clarifying question directed at whoever issued the raw request — never a silent guess, and never an indefinite stall. The question names the specific field that could not be populated and the specific reason (Part 16.2's discrete test having failed), rather than a generic "I need more information" that would force the requester to guess what the compiler actually needs. "Investiga este mercado" escalates with a question naming the entities field specifically and the specific gap: no market identity was named or resolvable from working context, and the compiler needs that named before compilation can proceed.

16.4 Not every field is equally likely to escalate, and this dataset states the pattern explicitly rather than leaving it implicit: entities (Part IV) escalates most often, since entity resolution has the least reliable fallback of any field this dataset compiles — a wrong entity resolution compiles a contract that pursues entirely the wrong real-world target, a failure mode too costly to risk defaulting through. Objective (Part IV) escalates rarely, since a request specific enough to name entities at all is usually specific enough to at least roughly imply an objective shape. Permitted domains and authorized authentication (Part IX) escalate in the specific narrow case Part IX §9.5 already names — an entity plausibly requiring authentication the mission was not given — rather than in the ordinary case, which defaults per Part IX's own stated logic without needing escalation at all.

16.5 A single compilation pass can produce more than one simultaneous escalation, and this dataset requires the compiler to surface every field genuinely requiring escalation together in a single clarifying exchange rather than escalating one field, receiving an answer, discovering a second field also needs escalation, and repeating the cycle — a pattern that would needlessly extend a compilation pass across many back-and-forth exchanges when the underlying gaps were all knowable at once. Part XV's ESCALATION-PENDING sub-state accommodates this directly: the state holds until every simultaneously-identified escalation has been answered, not merely the first one raised.

16.6 A failure mode specific to this Part, and arguably the single most consequential failure mode named across this entire dataset given how directly it interacts with Part I §1.3's central boundary: escalation avoidance through silent inference, where a compiler, under pressure to produce a fully populated contract without interrupting the requester, applies a plausible-sounding value to a field that actually fails Part 16.2's discrete default-exists test. This is not merely aggressive defaulting — it is a violation of the provenance discipline Part II §2.6 establishes, since a value produced this way carries no honest defaulted or resolved tag, only a fabricated one dressed in either label. This dataset treats this failure with the same severity Part I §1.3 already assigns to any field this dataset silently decides rather than declares, because the two failures are, at bottom, the same failure: a compiler exceeding what it actually knows and presenting the result as though it did not.

16.7 The reverse failure deserves equal naming: escalation overuse, where a compiler escalates fields that do, on the discrete test in 16.2, actually have a documented default available — treating every field the raw request did not state explicitly as an escalation case rather than checking whether Parts IV through XIV's own documented defaults actually cover it. This failure defeats the entire purpose of this dataset's careful, field-by-field default specification, forcing the requester to answer clarifying questions this dataset's own compilation logic was supposed to resolve without their involvement.

16.8 Escalation's relationship to Part XIX's rejection taxonomy deserves one explicit distinction: an escalation that receives no answer, or receives an answer that still does not resolve the underlying gap, eventually becomes a rejection — Part XIX's own taxonomy names this specific path directly. An escalation that receives a clear, resolving answer returns the contract to FIELD-ASSIGNING per Part XV §15.4 and proceeds normally. The two outcomes are distinguished in the compilation audit trail (Part XXII) by whether an escalation was ever raised at all, giving a future auditor visibility into which compiled contracts required clarification and which compiled cleanly on the first pass.

16.9 A worked example distinguishing escalation from a near-miss defaulting case: contrast "investiga este mercado" (genuine escalation, no documented entity default exists) with "investiga la competencia," investigate the competition, arriving immediately after a mission whose entities field named a single specific company. Here, "the competition" resolves against working context — the prior mission's named company implies a comparison-adjacent research objective against that company's own competitors, a resolvable case under Part II §2.2's normalization logic even though the raw request itself named no specific competitor. This is defaulting through resolution, not escalation, precisely because a documented resolution path (working context's immediately prior entity) exists and applies.

16.10 With ambiguity and underspecification handling specified, this dataset closes its second major block of Parts — the compilation mechanics proper, spanning Part II's pipeline through this Part's escalation discipline. Part XVII, taking up the six canonical mission templates next, returns to the concrete worked-example territory this Part and its predecessors have drawn on throughout, now organized as reusable defaults rather than scattered illustrations.

16.11 A closing note connecting this Part back to Part I's opening boundary one final time: escalation is the mechanism by which this dataset stays honest about the limits of what a compiler can determine from a request, working context, and estate knowledge alone. A compiler that never escalates, across a large enough volume of genuinely underspecified requests, is not a more capable compiler than this dataset describes — it is one that has quietly abandoned the discrete test in 16.2 in favor of the silent inference 16.6 names as this dataset's single most consequential failure mode. Escalation's existence in this schema is not a limitation to be minimized away in some future, more sophisticated implementation; it is the honest edge of what compilation from text alone can responsibly determine, and it remains that edge regardless of how good a future implementation's parsing or resolution logic becomes.

PART XVI FINAL LAW

Escalation triggers when a field fails a discrete, checkable test — no documented default exists for it given the available objective and entity context — never from a continuous, subjective confidence judgment, and a compiler that silently infers a plausible value for a field genuinely failing that test has fabricated a field exactly as surely as one that skipped population entirely, while a compiler that escalates fields Parts IV through XIV already provide documented defaults for has defeated this dataset's entire purpose of resolving what it honestly can without needlessly interrupting the requester.


PART XVII — MISSION TEMPLATES FOR THE SIX CANONICAL REQUEST SHAPES

17.1 This dataset has referenced six canonical request shapes throughout — the founding resource document's own six worked examples — as recurring illustrations for nearly every field this dataset compiles. This Part gathers them into one place, not as new material but as an organized synthesis of defaults already established piecemeal across Parts IV through XVI, so a reader who wants the fast path through this dataset's mechanics can start here and consult the individual field Parts only where a specific template's default needs justification.

17.2 Comprehension. "Léete este repositorio" — read this repository. Objective: understand structure and content of a single, usually well-bounded entity. Entities: typically one, often explicitly named or resolvable from working context. Depth: sufficient to cover the entity's full structure once (Part VI §6.2). Breadth: narrow, single-entity (Part VI §6.3). Required evidence: light-to-moderate, since a comprehension mission's own completeness check substitutes for corroboration much as an exhaustive-extraction mission's does (Part VIII §8.3). Output schema: narrative report (Part XII §12.2). Stop conditions: target-satisfied dominant, dead-end unlikely against a well-bounded entity.

17.3 Research. "Investiga este mercado" — investigate this market. Objective: build an informed understanding of a loosely bounded subject, often with multiple sub-entities not fully named in advance. Entities: frequently underspecified, the template this dataset's own Part IV §4.6 and Part XVI §16.3 use as the canonical escalation worked example when no working context resolves the subject. Depth: moderate, since exhaustiveness within any single source rarely serves a broad research objective as well as coverage across sources does. Breadth: wide, deliberately sampling beyond any explicitly named starting point (Part VI §6.3). Required evidence and confidence-level requirement: moderate-to-heavy, given the interpretive weight research conclusions typically carry (Part VIII §8.3, Part XIII §13.2). Output schema: narrative report, often with an explicit corroboration-surfacing apparatus per Part XII §12.5's required-evidence interaction.

17.4 Exhaustive extraction. "Extrae todas las funciones" — extract every function. Objective: complete, structured enumeration within a well-bounded entity. Entities: typically one, precisely bounded (a repository, a specific document). Depth: maximum within the entity's own boundary (Part VI §6.2). Breadth: narrow, matching the single-entity default. Required evidence: light per data point, since completeness substitutes for corroboration (Part VIII §8.3). Output schema: structured enumeration, never narrative prose (Part XII §12.6, and the specific schema-mismatch failure Part XII §12.7 names against defaulting toward prose here). Stop conditions: target-satisfied defined precisely as full coverage of the bounded entity, dead-end essentially inapplicable given the entity's own bounded and enumerable nature.

17.5 Comparison. "Compárame estos competidores" — compare these competitors. Objective: structured comparison across multiple named or resolvable entities. Entities: multiple, ideally resolved from working context or explicit naming rather than defaulted, given entity resolution's own high escalation priority (Part XVI §16.4). Depth and breadth: matched across all entities for comparison fairness (Part VI §6.3). Required evidence and confidence-level requirement: moderate-to-heavy, this dataset's own most frequently reused worked example throughout Parts II, VIII, and XIII specifically because comparison's interpretive weight recurs across nearly every field this dataset compiles. Output schema: structured comparison table (Part XII §12.2). Stop conditions: contradiction-discovered specifically relevant here, per Part XI §11.5's own comparison-objective framing.

17.6 Monitoring. "Vigila cuándo cambia" — watch for when it changes. Objective: detect and report change against a single entity over an open-ended horizon. Entities: typically one. Depth: shallow per individual pass (Part VI §6.2). Update policy: continuous/event-driven recurrence, this dataset's defining case for that recurrence shape (Part VII §7.4). Freshness: strict on each individual check, calibrated against entity volatility rather than uniformly (Part VII §7.5's own worked contrast between a live pricing page and a slow-moving filing). Output schema: recurring change-alert, not a single report (Part XII §12.2). Stop conditions: target-satisfied reframed as "never, by design," relying on ceiling-reached and explicit Owner abort instead (Part XI §11.7). Preservation obligation: elevated, since change detection requires comparing against preserved prior state (Part XIV §14.4).

17.7 Dataset construction. "Constrúyeme un dataset con todas las fuentes" — build a dataset from every source. Objective: comprehensive coverage along both depth and breadth simultaneously, this dataset's one canonical template where the two do not trade against each other by default (Part VI §6.5). Entities: often a topic rather than a single named organization, implying wider breadth defaults than the comprehension or exhaustive-extraction templates. Cost and time ceilings: pushed toward the higher end of CO-01's default range given the objective's own scale (Part X §10.5). Output schema: the specific structured dataset format the request's own wording implies (Part XII §12.2). This template is the one most likely to approach its own cost or time ceiling as the binding constraint on actual achieved scope, per Part VI §6.5's own observation.

17.8 These six templates are defaults, not rigid categories, and this dataset restates the boundary Part II §2.4 already establishes: a request matching none of the six cleanly still compiles through the same three-stage pipeline (Part II) without a template's head start, and a request superficially resembling one template can still diverge from its defaults wherever an explicit signal in the raw request overrides them — 12.3's explicit-output-format override is one instance of a pattern that applies identically across every field a template would otherwise default.

17.9 A request can also blend more than one template without being a genuinely compound objective in Part IV §4.7's sense. "Investiga este mercado y compárame a los tres líderes," research this market and compare the three leaders, is a single objective whose shape borrows research's wide-breadth entity discovery for an initial phase and comparison's matched-depth structure for a subsequent phase, rather than two independent objectives warranting Part XV §15.5's contract-forking treatment. This dataset does not attempt to name every possible blend as its own seventh, eighth, or ninth template — the six canonical shapes exist because the founding resource document named exactly these six, not because this dataset claims they exhaust every possible request shape, and a blended case compiles from the underlying field-by-field mechanics Parts IV through XVI already specify rather than from a template this dataset has not separately named.

17.10 With all six templates gathered, this dataset turns in Part XVIII to mission revision and amendment — the discipline governing what happens when a compiled, dispatched contract, regardless of which template or blend produced it, needs to change mid-flight.

17.11 One further use for this Part's gathered synthesis, beyond a fast-path reading aid: template membership is itself a useful diagnostic during Part XV's INTERNALLY-CHECKING state. A compiled contract whose fields diverge sharply from every one of the six templates' defaults, without any explicit request-level signal accounting for the divergence, is not necessarily wrong — Part 17.8 already establishes that a request matching no template compiles just as faithfully as one that does — but it is worth a compiler flagging internally as a candidate for closer review, since an unexplained divergence from every known default pattern is at least as likely to indicate a compilation error somewhere in Parts IV through XVI's own mechanics as it is to indicate a genuinely novel request shape this dataset's six templates simply never anticipated. This dataset does not require such a flag to block VALIDATED status the way a genuine internal-consistency failure does; it is an advisory signal for whoever reviews the compilation audit trail, not a gate.

PART XVII FINAL LAW

The six canonical request shapes — comprehension, research, exhaustive extraction, comparison, monitoring, and dataset construction — are reusable defaults synthesizing the field-by-field mechanics Parts IV through XVI already establish, never a rigid taxonomy a request must fit to be compilable; every default any template supplies yields to an explicit signal in the raw request, and a request matching no template, or blending several, compiles through this dataset's own general pipeline exactly as faithfully as one matching a template precisely.


PART XVIII — MISSION REVISION AND AMENDMENT

18.1 This Part fulfills the promise Part II §2.8 made at this dataset's own outset: a mission, once dispatched, is not immune to change, and this Part specifies exactly how a follow-up request re-enters this dataset's compilation machinery against an already-compiled contract rather than starting fresh.

18.2 The threshold question every revision request must answer, before any field is touched: does this follow-up request amend the existing mission's contract, or does it name a genuinely new mission that happens to arrive in the same conversational context. This dataset draws the line at objective and entities: a follow-up request that leaves both unchanged, adjusting only one or more of the remaining fourteen fields, is an amendment. A follow-up request that changes either the objective's own shape or the entities it concerns is a new mission, compiled from RECEIVED per Part XV's state machine, regardless of how much surrounding conversational continuity it shares with the prior mission.

18.3 Amendments compile through a scoped re-entry into Part XV's state machine, entering at whichever state the specific amendment actually requires re-running rather than restarting from RECEIVED. Parts 12.12, 13.12, and 14.11 already worked through three concrete cases — an output-schema change, a confidence-level-requirement change, a preservation-obligation change — each re-entering at FIELD-ASSIGNING for the single affected field, then proceeding through INTERNALLY-CHECKING to confirm the amended value does not introduce an inconsistency with the mission's other, unchanged fields.

18.4 A deeper amendment — one touching depth, breadth, required evidence, or any field with wide cross-field dependencies per Part III §3.18 — cannot be scoped as narrowly, since Part VI §6.11's second silent recalibration pass and the various cross-field checks Part XV §15.2 and §15.6 name were computed against the pre-amendment field values. This dataset requires such an amendment to re-enter at FIELD-ASSIGNING broadly enough to re-run every consistency check touching the amended field, not merely to overwrite the one field's value and leave stale, pre-amendment consistency conclusions standing.

18.5 A worked example distinguishing scoped from broad re-entry: consider a comparison mission already dispatched against three named competitors, whose Owner then states "actually, add a fourth competitor to the comparison." This changes the entities field's cardinality, which Part VI §6.3 already establishes as a direct input to breadth calibration — the amendment cannot be scoped to the entities field alone, since breadth (and everything breadth in turn feeds, including expected sources per Part V §5.8 and the cost/time ceilings per Part X §10.6) must be recalibrated against the new four-entity count. This dataset classifies this as an amendment, not a new mission, since the objective (comparison) is unchanged, but it is a broad re-entry, not a scoped one, precisely because entities sits near the top of this dataset's own dependency chain rather than at a leaf field like output schema.

18.6 A boundary worth stating plainly: an amendment never re-enters at a state earlier than the point genuinely required by the specific fields it touches, even for a broad re-entry. The four-competitor amendment in 18.5 re-enters at FIELD-ASSIGNING, not at PARSING or NORMALIZING, because the follow-up request's own parsing and normalization (identifying "add a fourth competitor" as an entities-field addition, resolving which fourth competitor is meant) are themselves new parsing and normalization work belonging to the follow-up request's own PARSING and NORMALIZING pass, distinct from re-running the original request's already-settled parse. This dataset therefore treats every amendment as its own miniature pass through PARSING and NORMALIZING for the follow-up request's own content, followed by a FIELD-ASSIGNING re-entry scoped or broad per 18.3 and 18.4's distinction, applied against the original contract's other, unamended fields.

18.7 Amendments interact with the compilation audit trail (Part XXII) directly: every amendment is recorded as a distinct entry referencing the contract's prior state, never as a silent overwrite that erases what the field previously held. This mirrors Dataset #10 Part X's own change-history discipline for the Evidence Object almost exactly, and this dataset adopts the same never-delete principle for the identical reason — an auditor examining a mission's history should be able to see not only its current compiled state but the sequence of amendments that produced it.

18.8 A failure mode specific to this Part: amendment misclassification, where a follow-up request that actually changes the objective or entities is compiled as a scoped field amendment rather than recognized as a new mission. This produces a contract whose objective and entities fields no longer accurately describe what the mission is actually pursuing, while every other field retains stale values calibrated against the original, now-superseded objective — a comparison mission's required-evidence field, calibrated against three competitors, silently governing a request that has, in substance, become a single-entity deep-dive on one of them. This dataset requires the threshold question in 18.2 to be checked explicitly on every follow-up request, never assumed answered by conversational continuity alone.

18.9 The reverse failure deserves equal naming: treating every follow-up request as a new mission regardless of how narrowly it amends the existing one, discarding an already-partially-executed mission's progress and cost expenditure to recompile from scratch over what was, in substance, a single-field correction. This dataset requires the threshold question to be answered honestly in both directions — neither collapsing a genuine new mission into a stale amendment, nor inflating a genuine amendment into a wasteful full recompile.

18.10 With revision and amendment specified, this dataset turns in Part XIX to the rejection taxonomy Part XVIII's own threshold question and Part XVI's escalation discipline have both already referenced without fully specifying — the cases where no viable contract, amended or fresh, results from a compilation attempt at all.

18.11 A cumulative-amendment case deserves explicit treatment before this Part closes: a mission amended several times in sequence, each individual amendment passing 18.2's threshold question cleanly, but whose accumulated changes, taken together, have drifted the contract far enough from its original compilation that the mission it now describes bears little resemblance to the one originally dispatched. This dataset does not treat any single amendment in such a sequence as a misclassification per 18.8 — each one, examined on its own terms against the contract state immediately preceding it, genuinely left objective and entities unchanged. But this dataset requires the compilation audit trail's own amendment count, maintained per Part XXII's own tracking discipline, to be checked against a threshold at each new amendment request: past a certain accumulated volume of change, even a technically-qualifying amendment warrants an explicit prompt asking whether the mission, considered as a whole against its own original compilation, would be better served by a fresh compilation than by a further incremental patch. This dataset does not fix that threshold numerically, leaving it to a future implementation's own calibration, but it requires the check to exist rather than allowing amendment drift to accumulate indefinitely uncounted.

18.12 A revision request can also, in principle, attempt to change the mission's downstream engine handoff (Part XXI) after that handoff has already occurred — asking, in effect, to redirect an already-dispatched mission to a different acquisition strategy mid-flight. This dataset treats this case as out of its own scope entirely, not as a Part XVIII amendment: Part XXI §21.3 already establishes that this dataset's own responsibility for a given mission concludes the instant handoff occurs, and a request to alter routing after that point belongs to whatever downstream system actually received the handoff, never to a re-entry into this dataset's own compilation pipeline.

PART XVIII FINAL LAW

A follow-up request amends an existing mission's contract only when its objective and entities remain unchanged from the original compilation; any change to either constitutes a new mission compiled from RECEIVED, and a genuine amendment re-enters Part XV's state machine at whichever state — scoped to a single leaf field or broad enough to re-run every dependent consistency check — the specific amended field's own dependency chain actually requires, recorded in the compilation audit trail as a new entry against the contract's prior state, never as a silent overwrite.


PART XIX — THE COMPILER'S OWN REJECTION TAXONOMY

19.1 This Part specifies REJECTED, the second of Part XV's two terminal states, closing a gap nearly every prior Part in this dataset has referenced without fully specifying: the cases where a compilation attempt produces no viable Mission Contract at all, distinct from Part XI's stop conditions, which govern how an already-VALIDATED, already-dispatched mission ends.

19.2 No viable objective. The raw request, after parsing and normalization, states no identifiable action a Crawl OS mission could pursue — a request that is not actually a crawl-intent request at all, misrouted to this compiler by whatever upstream system dispatches requests to it. This dataset does not treat this as an error in the strict sense; it is a correct rejection of a request this dataset was never the right destination for in the first place.

19.3 Unresolvable entity with no escalation path. Part XVI §16.2's escalation mechanism was triggered, a clarifying question was raised, and no answer resolved the gap — either because no answer arrived at all, or because the answer received still failed the same discrete default-exists test that triggered the original escalation. This is the rejection path Part XVI §16.8 already names directly.

19.4 Internally incoherent request. The raw request states directly contradictory requirements that no compilation choice can reconcile — an explicit maximum-cost ceiling stated alongside an explicit depth/breadth combination Part VI's own calibration logic cannot plausibly satisfy within that ceiling, with the raw request itself, not merely a default, insisting on both. This differs from the internal-consistency failures Part XV's INTERNALLY-CHECKING state catches for defaulted or resolved field values — this rejection class applies specifically when the contradiction originates in the request's own explicit, non-negotiable statements, leaving the compiler no coherent value to compile at all.

19.5 No viable source. Part V's source prediction, even at its weakest entity-type-inference tier, produces no plausible source at all for the mission's resolved entities — an entity so obscure or so narrowly private that no source prediction mechanism this dataset specifies can name even a tentative starting point. This rejection is rare, since Part V §5.7 already establishes that even a bounded, self-contained entity (a named repository, a specific document) predicts its own source nearly mechanically; this class exists for the genuine residual case where even that mechanical fallback fails.

19.6 Authorization foreclosed at compilation time. Distinct from the ordinary Part IX §9.5 case (a mission dispatched with a flagged access gap for the future Dataset #16 to evaluate), this rejection applies when the raw request itself explicitly names an acquisition target Dataset #01 Part IX's Anti-Detection-Is-Not-Authorization Doctrine already forecloses outright — a request explicitly asking to circumvent an access control rather than merely omitting an authentication signal this dataset would otherwise default conservatively. This dataset requires this class to be distinguished sharply from the Part IX §9.5 flagging case: 9.5 dispatches a mission with an honestly flagged gap for a downstream engine to adjudicate; this class rejects outright, at compilation time, because the request's own stated intent already violates a constitutional floor no downstream adjudication could legitimately overturn.

19.7 A worked example: "extrae todas las funciones de este repositorio privado al que no tengo acceso," extract every function from this private repository I don't have access to, compiles entities and objective cleanly but rejects under 19.6's authorization-foreclosed class, since the request's own explicit statement of lacking access, combined with an explicit request to acquire from it anyway, states outright what Part IX §9.5's ordinary flagging case only ever infers from an access gap's mere absence of signal.

19.8 Every rejection carries a recorded reason drawn from this Part's own five-class taxonomy, recorded in the compilation audit trail (Part XXII) alongside whatever partial compilation state existed before the rejection fired — this dataset requires a rejection to be as fully auditable after the fact as a successful VALIDATED contract, since an unexplained rejection is as much an opacity failure as a silently fabricated field would be.

19.9 A failure mode specific to this Part: rejection as an escape hatch for genuine ambiguity, where a compiler rejects a request that Part XVI's escalation discipline should instead have handled through a clarifying question. This dataset requires rejection to be reserved for the five classes 19.2 through 19.6 actually name, never used as a shortcut past the more effortful but more honest escalation path Part XVI already specifies for every case where a clarifying question, rather than an outright rejection, is the correct response to genuine underspecification.

19.10 With the rejection taxonomy specified, this dataset turns in Part XX to multi-source and multi-entity disambiguation — a discipline distinct from outright rejection, governing the cases where a request compiles successfully but resolves to more than one plausible interpretation the compiler must actively choose between rather than either defaulting past or escalating over.

19.11 A closing distinction worth drawing between rejection and the two other negative-outcome mechanisms this dataset specifies elsewhere, since a reader working through this dataset in sequence has now encountered all three and may benefit from seeing them contrasted directly. Escalation (Part XVI) pauses compilation to ask a clarifying question, with the expectation that an answer will let compilation resume; it is this dataset's default response to genuine underspecification. Rejection (this Part) ends compilation entirely, with no expectation of resumption, reserved for the five classes named in 19.2 through 19.6, several of which — 19.4's internal incoherence, 19.6's foreclosed authorization — could not be resolved by any clarifying answer even in principle, since the problem lies in what the request itself explicitly demands rather than in what it left unstated. Part XI's stop conditions, by contrast, apply only after a contract has already reached VALIDATED and been dispatched, governing how a running mission ends rather than whether a contract compiles at all. A reader who finds a specific case ambiguous between these three mechanisms should apply this test in order: if the gap could plausibly be resolved by an answer to a clarifying question, it is an escalation case; if the request's own explicit content forecloses any coherent contract regardless of any answer, it is a rejection case; if the question only arises after a contract already exists and has been dispatched, it belongs to Part XI entirely and is not this Part's concern.

19.12 One further boundary worth naming explicitly, given how often this dataset's failure-mode discussions elsewhere name defaulting and escalation together: no rejection class in this Part's taxonomy exists to catch a field that merely lacks an explicit signal and has no documented default. That case is squarely Part XVI's escalation territory, per 19.3's own restatement of Part XVI §16.8's boundary, and this dataset requires an implementation to exhaust the escalation path fully — raising the clarifying question, awaiting or failing to receive a resolving answer — before a compilation attempt may transition to REJECTED under 19.3's class specifically. Rejection under any of the other four classes, by contrast, requires no prior escalation attempt at all, since none of those four classes describe a gap any clarifying question could resolve.

PART XIX FINAL LAW

A compilation attempt rejects into exactly one of five named classes — no viable objective, an unresolvable entity with no escalation path left, an internally incoherent explicit request, no viable source even at the weakest prediction tier, or authorization explicitly foreclosed by this family's own constitutional floor — and never as a substitute for the more effortful, more honest escalation path Part XVI already specifies for genuine underspecification that a clarifying question, not an outright refusal, is the correct response to.


PART XX — MULTI-SOURCE AND MULTI-ENTITY DISAMBIGUATION

20.1 Part IV §4.4 already noted that the entities field can carry nested structure — sub-entities within a primary entity — and flagged that a nested entity resolving ambiguously was this Part's own subject. This Part specifies exactly how that resolution works, distinct from both Part XVI's escalation discipline (which applies when no resolution is possible at all) and Part XIX's rejection taxonomy (which applies when no viable entity exists whatsoever).

20.2 Disambiguation applies when an entity reference resolves to more than one plausible real-world candidate, none of which the available signals rule out entirely — distinct from the pure escalation case (no candidate resolves at all) and the pure single-resolution case (exactly one candidate resolves cleanly). A company name shared by multiple distinct real organizations, referenced with no further qualifying detail, is this Part's canonical case: several plausible candidates exist, and nothing in the raw request or working context rules any of them out on its own.

20.3 This dataset's disambiguation discipline ranks candidates by three signals, considered together rather than strictly in sequence: recency in working context (an entity referenced in a recent prior mission outranks one with no recent context at all), specificity of match (a candidate matching additional qualifying detail the raw request supplied — an industry, a location, a rough size — outranks one matching only the bare name), and site memory's own prior resolution history (a candidate this working context or site memory has resolved this exact ambiguous reference to before outranks one with no such history, though this signal alone is never treated as dispositive, since a prior resolution could itself have been a different disambiguation choice made in a different context that does not necessarily apply here).

20.4 When ranking produces a clear leading candidate — one candidate outranking every other on at least two of the three signals — this dataset resolves to that candidate and records the resolution as a specific provenance sub-tag, distinct from Part II §2.6's ordinary resolved tag, marking it explicitly as a disambiguated resolution among multiple plausible candidates rather than a clean single-candidate resolution. This distinction matters because a disambiguated resolution carries a real, if usually small, risk of having chosen the wrong candidate, a risk a clean single-candidate resolution does not carry, and a downstream consumer or auditor benefits from being able to see which resolutions carried that risk.

20.5 When ranking produces no clear leading candidate — multiple candidates tied or nearly tied across all three signals — this dataset escalates per Part XVI's own discipline, treating the tied-ambiguity case as failing Part XVI §16.2's discrete default-exists test exactly as a fully unresolvable reference would, even though, unlike the fully unresolvable case, multiple named candidates do exist. The escalating clarifying question, in this specific case, names the tied candidates explicitly rather than asking a generic "which entity did you mean," giving the requester the fastest possible path to resolving the tie.

20.6 A worked example: "investígame la empresa Atlas," research the company Atlas, arriving with no working context and no further qualifying detail, where site memory holds prior resolution history for two distinct organizations both informally referred to as Atlas — one a logistics company, one a fintech startup. Neither candidate is ruled out by the raw request's own wording, and with no recency signal favoring either and no additional specificity in the request itself, this dataset's ranking produces a tie, escalating with a clarifying question naming both candidates directly.

20.7 Contrast that with the same request arriving in a working context where the immediately prior mission concerned the logistics company Atlas by its full formal name. Recency now favors the logistics candidate decisively, producing a clear leading candidate under 20.4's ranking, and this dataset resolves without escalation, recording the disambiguated-resolution provenance sub-tag to flag, for any future auditor, that the resolution rested on recency rather than the request's own explicit specificity.

20.8 Multi-source disambiguation operates on the same underlying mechanism applied to Part V's expected-sources field rather than to entities: when source prediction identifies multiple plausible primary sources for the same entity with no signal favoring one over the others, this dataset applies 20.3's ranking logic to sources rather than entities, and reserves escalation for the genuine no-signal tie case exactly as 20.5 already specifies for entities.

20.9 A boundary worth stating explicitly: this dataset's disambiguation ranking is a compilation-time judgment call, made honestly and recorded transparently via the provenance sub-tag, never a claim of certainty about which candidate is actually correct. A disambiguated resolution that later, once acquisition begins, turns out to have chosen the wrong candidate is not, by that fact alone, evidence that this Part's ranking logic failed — Part XVIII's amendment discipline exists precisely to correct a mission whose entity resolution turns out, once real acquisition surfaces the mismatch, to have chosen wrong, treated as an entities-field amendment per Part XVIII §18.2's own threshold question.

20.10 With disambiguation specified, this dataset turns in Part XXI to the mission-to-engine handoff interface — the mechanics of what happens to a VALIDATED contract once Part XV's state machine has produced one, closing the gap between this dataset's own compilation discipline and the downstream engines Parts III through XX have referenced throughout without yet specifying how the actual handoff occurs.

20.11 A case worth naming that this Part's earlier treatment has not yet addressed directly: disambiguation among candidates that are not fully independent of one another but stand in a parent-subsidiary or similar corporate relationship. A name resolving ambiguously between a parent organization and one of its named subsidiaries is not the same disambiguation problem as two wholly unrelated organizations sharing a name, since the raw request's own objective often determines which level of the relationship is actually meant even where the entity name alone does not. This dataset requires the objective field, already populated per Part IV ahead of any entity disambiguation per Part 4.5's own ordering requirement, to inform ranking in this specific case: an exhaustive-extraction objective against a codebase plausibly implies the specific subsidiary or product entity that actually owns the codebase in question, while a broad market-research objective plausibly implies the parent organization's own wider footprint, even when the bare entity name alone would rank the two candidates as a tie under 20.3's three signals in isolation.

20.12 A closing note on how disambiguation interacts with Part XX's own sibling disciplines one final time: a disambiguated resolution, once recorded, is treated by every later Part in this dataset exactly as any other resolved-tier provenance would be, save for the specific sub-tag 20.4 requires. Part VIII's required-evidence compilation, Part IX's permitted-domains compilation, and every other field-population mechanic downstream of entity resolution proceeds against the disambiguated candidate precisely as it would against an unambiguously resolved one — this Part's discipline governs only how the entities field itself arrives at its value, never how the rest of this dataset's sixteen fields subsequently treat that value once it exists.

PART XX FINAL LAW

An entity or source reference resolving to multiple plausible candidates ranks them by working-context recency, request-level specificity, and site memory's prior resolution history; a clear leading candidate resolves with a disambiguated-resolution provenance sub-tag flagging the residual risk a clean single-candidate resolution does not carry, while a genuine tie escalates per Part XVI's discipline naming the tied candidates directly — and a disambiguated resolution later found wrong is corrected through Part XVIII's amendment discipline, never treated as proof this Part's ranking logic itself failed.


PART XXI — THE MISSION-TO-ENGINE HANDOFF INTERFACE

21.1 This Part specifies the one moment every other Part in this dataset has referenced but deferred: a contract reaching Part XV's VALIDATED state is handed off to a downstream engine, most directly the future Dataset #03 Acquisition Strategy Router, and this Part states exactly what that handoff consists of and, as importantly, what it does not.

21.2 The handoff payload is the compiled contract itself — all sixteen fields, each carrying its own provenance tag per Part II §2.6, plus the compilation audit trail (Part XXII) recording the contract's full state history from RECEIVED through VALIDATED, including any escalations (Part XVI), disambiguations (Part XX), or amendments (Part XVIII) that occurred along the way. This dataset hands off the complete record, never a stripped-down summary, because a downstream engine encountering unexpected acquisition difficulty may need to consult the contract's own provenance detail — was this field extracted, resolved, defaulted, or disambiguated — to decide how much latitude it has to interpret the field flexibly versus how strictly it must honor an explicitly stated requirement.

21.3 The handoff is a one-time transfer of a complete, validated artifact — this dataset does not maintain an ongoing synchronization channel with the downstream engine once handoff occurs, the identical boundary Part V §5.9 already establishes for the expected-sources field specifically, generalized here to the full contract. A downstream engine's own discovery that reality diverges from the compiled contract's predictions (a predicted source proving unreachable, an estimated depth proving insufficient) is the downstream engine's own problem to manage within its own charter, or, where the divergence is severe enough to warrant it, a trigger for Part XVIII's amendment discipline initiated by whatever process is monitoring the mission's progress — never a channel this dataset's own compilation pipeline keeps open to the dispatched mission.

21.4 This dataset does not itself select which downstream engine receives a given VALIDATED contract. That selection — routing a research-objective contract toward the engines a research mission actually needs, a comprehension-objective contract toward a different combination — belongs entirely to the future Dataset #03 Acquisition Strategy Router's own charter, consuming the compiled contract's objective and other fields as its own routing input. This dataset's handoff interface is uniform across every objective shape; the downstream routing decision that interface feeds is not this dataset's concern.

21.5 A boundary worth restating in this Part's own specific terms, closing this dataset's engagement with every downstream engine its sixteen fields have referenced throughout: this dataset's responsibility ends at producing a complete, internally consistent, honestly provenance-tagged VALIDATED contract and handing it off. It does not verify that the downstream engine actually received the handoff, does not track the mission's subsequent execution, and does not itself produce or format the mission's eventual deliverable — every one of those responsibilities belongs to systems this dataset has named throughout as PLANNED or already-sealed siblings, never to this dataset's own compilation pipeline reaching beyond the boundary Part I §1.3 drew at this dataset's very outset.

21.6 The handoff interface's own failure surface is narrow precisely because the interface itself is narrow: a VALIDATED contract that fails to actually reach its intended downstream engine (a dispatch failure, distinct from anything this dataset's own compilation logic could have caused) is not a compilation failure in any sense Part XXIII's taxonomy names — it is an execution-substrate failure occurring after this dataset's own responsibility has already concluded, and this dataset does not attempt to specify recovery behavior for a failure occurring entirely outside its own charter.

21.7 A worked closing trace, drawing together this dataset's own recurring comparison-objective example one final time: "compárame estos competidores" compiles through Parts II through XX's mechanics to a VALIDATED contract naming three resolved entities, a comparison objective, matched depth/breadth, a moderate-to-heavy required-evidence burden, a moderate-to-high confidence-level requirement, a structured-comparison output schema, and every other field this dataset's sixteen-field schema requires. This Part's own handoff interface transfers that complete contract, with its full provenance and state history, to whatever downstream routing the future Dataset #03 determines — the exact point at which this dataset's own responsibility for this specific mission concludes.

21.8 This Part closes the third of this dataset's four major blocks — the cross-cutting compilation disciplines spanning Part XV's state machine through this Part's handoff interface. Part XXII, taking up mission auditing and traceability next, specifies the compilation audit trail this Part has referenced throughout as part of the handoff payload but not yet fully described in its own right.

21.9 One final boundary, restated because it bears directly on this Part's own subject: the handoff interface this Part specifies is, per Part I §1.9's honesty discipline, currently PLANNED rather than REACHABLE — no downstream Dataset #03 exists yet to actually receive a handoff, and this Part's specification describes the interface a future `modules/crawl_os/mission_compiler.py` will expose once that downstream engine exists to consume it, not a transfer this dataset's own text can claim actually occurs today.

21.10 With the handoff interface specified, all four of this dataset's field-population, compilation-mechanics, and cross-cutting-discipline blocks are complete. Part XXII turns to auditing and traceability, Part XXIII to this dataset's own failure taxonomy, Part XXIV to evolution and amendment of this dataset itself, and Part XXV closes with this dataset's own done criteria and final law.

21.11 A worked contrast worth drawing before this Part closes, since it clarifies exactly where this dataset's own responsibility ends relative to a mission's full lifecycle: consider the same comparison mission from 21.7 encountering, once dispatched, a predicted source that proves entirely unreachable during actual acquisition. This dataset's own handoff interface already transferred that prediction honestly tagged per Part V §5.4's weakest-provenance case where applicable, and the downstream engine's own discovery that the prediction failed to pan out is exactly the kind of divergence 21.3 already names as belonging to the downstream engine's own charter, or, if severe enough, a trigger for a Part XVIII amendment initiated by whoever is monitoring the mission — never a defect in this dataset's own handoff, which discharged its full responsibility the moment it transferred an honestly provenance-tagged prediction rather than a falsely confident one.

21.12 This Part's own relationship to Part XVIII's amendment discipline deserves one final clarifying note: an amendment re-entering Part XV's state machine, per Part XVIII §18.3 and §18.4, produces a new VALIDATED state once its re-entry completes, and that new VALIDATED state triggers this Part's own handoff interface exactly as an original compilation's VALIDATED state does — an amended contract is hand off through the identical mechanism 21.2 already specifies, carrying its updated compilation audit trail forward rather than requiring any handoff logic distinct from what this Part already establishes for a first-time compilation.

PART XXI FINAL LAW

A VALIDATED contract is handed off to a downstream engine as a single, complete, one-time transfer — every field, its full provenance, and its complete compilation audit trail, never a stripped summary and never an ongoing synchronization channel — with routing selection belonging entirely to the future Dataset #03 Acquisition Strategy Router and this dataset's own responsibility concluding the instant that complete transfer occurs, currently PLANNED rather than REACHABLE until that downstream engine exists to actually receive it.


PART XXII — MISSION AUDITING AND TRACEABILITY

22.1 This Part specifies the compilation audit trail nearly every prior Part in this dataset has referenced without fully describing: the complete, honest record of a Mission Contract's own compilation history, from the raw request that triggered it through every state transition, escalation, disambiguation, and amendment along the way, handed off alongside the contract itself per Part XXI §21.2.

22.2 The audit trail's structure mirrors Part XV's state machine directly: one entry per state transition, each entry carrying a timestamp, the state entered, and whatever field-level changes that transition produced. A PARSING-to-NORMALIZING transition's entry records what the parsing stage identified; a FIELD-ASSIGNING entry records each field's populated value together with its provenance tag per Part II §2.6; an INTERNALLY-CHECKING entry records which consistency checks ran and their individual pass/fail results, not merely the aggregate outcome, so that a future auditor can see exactly which check, if any, a contract failed rather than only that it failed one.

22.3 Escalations (Part XVI), disambiguations (Part XX), and amendments (Part XVIII) each produce their own distinctly-typed audit entries, distinguishable from ordinary state transitions, precisely because Part XVI §16.8 and Part XVIII §18.7 both already require these events to be visible to a future auditor as their own category rather than collapsed into an undifferentiated compilation history. An auditor reviewing a contract's trail should be able to answer, at a glance, three questions this dataset treats as load-bearing: did this contract require any clarification during compilation, did any entity resolve ambiguously, and has this contract been amended since its original compilation — each answerable directly from the trail's own entry types without needing to reconstruct the answer from raw state-transition timestamps.

22.4 The audit trail is retained for as long as the mission's underlying evidence is retained per the mission's own preservation-obligation field (Part XIV), since a trail outliving its own evidence would document a mission an auditor could no longer verify against anything, and a trail expiring before its evidence would leave that evidence's provenance-chain integrity — the exact concern Dataset #10's own tamper-evidence discipline exists to protect — resting on record-keeping this dataset's own machinery had already discarded.

22.5 Auditing interacts with Part XVI's escalation discipline in one specific way worth naming: a pattern of frequent escalation on a specific field, visible only by examining many contracts' audit trails together rather than any single one, is a signal this dataset treats as valuable feedback for the future `modules/crawl_os/mission_compiler.py`'s own calibration — a field escalating unusually often may indicate its documented defaults (Parts IV through XIV) need revision, a case Part XXIV's evolution discipline covers directly. This dataset does not itself perform that aggregate analysis; it only ensures the audit trail's own structure makes such analysis possible for whatever system eventually performs it.

22.6 A boundary worth stating in this Part's own specific terms: the audit trail records what this dataset's own compilation pipeline actually did — which fields were populated by which mechanism, which checks ran and passed or failed, which escalations or amendments occurred — never what a downstream engine subsequently did with the VALIDATED contract once handed off. Auditing the mission's actual execution, once dispatched, belongs to whatever downstream systems govern that execution, most directly the future Dataset #13 Crawl Cortex Supervision and Autonomous QA dataset; this dataset's own audit trail ends, like its own responsibility per Part XXI §21.5, at the moment of handoff.

22.7 A worked example distinguishing what belongs in this dataset's own audit trail from what does not: the recurring comparison-mission worked example, once dispatched, later encounters a source proving unreachable during actual acquisition. This event is not recorded in this dataset's own audit trail, since it occurred entirely after handoff, within the downstream engine's own execution — unless that unreachability triggers a Part XVIII amendment, in which case the amendment itself, not the underlying acquisition event that motivated it, becomes a new entry in this dataset's own trail per 22.3's amendment-entry category.

22.8 A failure mode specific to this Part: an incomplete audit trail that records only the terminal outcome — VALIDATED or REJECTED — without the intermediate state history, escalations, or disambiguations that produced it. A trail reduced to only its terminal state cannot support Part XVI §16.8's distinction between a contract that compiled cleanly and one that required clarification, cannot support Part XX's disambiguated-resolution flag, and cannot support Part XVIII's amendment history — an implementation that persists only the final compiled contract, discarding the process that produced it, has technically produced a VALIDATED or REJECTED outcome but has not implemented this dataset's own auditing discipline at all.

22.9 A closing note on this Part's relationship to Dataset #10's own Part XIX Negative Evidence Record: a REJECTED contract's audit trail, per 22.8's own completeness requirement, constitutes exactly the kind of institutional-learning signal Dataset #10 Part XIX already establishes for failed acquisitions — a compilation attempt that never even reached dispatch is, in its own way, negative evidence about what this dataset's own compilation logic could not resolve, valuable to whatever future calibration or Part XXIV evolution this dataset undergoes, in the same spirit Dataset #10's own negative-evidence discipline already applies one layer downstream.

22.10 With auditing and traceability specified, this dataset turns in Part XXIII to its own failure taxonomy — the compiler-specific failure classes this Part's own audit trail exists to make discoverable after the fact, gathering together the individual failure modes Parts IV through XXI have each already named in their own specific field or discipline's terms.

22.11 A worked closing example tying this Part's own record-keeping discipline to a concrete case, since abstract description of an audit trail's structure is easier to internalize against a single walked-through trace: the "investiga este mercado" request from Part IV §4.6 produces an audit trail whose entries read, in order, RECEIVED (raw request logged verbatim), PARSING (verb "investigate" identified, objective shape provisionally tagged research), NORMALIZING (entity resolution attempted against working context, no candidate found), an escalation-type entry (clarifying question raised naming the entities field specifically, per Part XVI §16.3's own required specificity), a gap in the timeline while awaiting an answer, then either a resumed FIELD-ASSIGNING entry once an answer arrives or, absent one, eventually a REJECTED entry citing Part XIX §19.3's unresolvable-entity class. A future auditor examining only this trail, without any other context, can reconstruct the full compilation history and understand precisely why this particular mission took the path it took — the exact standard 22.3's own three load-bearing questions exist to guarantee is always answerable.

22.12 This Part closes with one further observation on why the audit trail is specified in this dataset at all, rather than left as an implementation detail a future `modules/crawl_os/mission_compiler.py` could reasonably invent on its own: every other Part in this dataset that names a failure mode — burden inflation, authorization inflation, escalation avoidance, amendment misclassification, and the rest Part XXIII gathers next — is only detectable after the fact if some record of the compilation process survives long enough to be examined. This Part is, in that specific sense, the precondition for Part XXIII's entire taxonomy being checkable against real compiled contracts rather than remaining a purely theoretical catalog of ways a compiler could go wrong.

PART XXII FINAL LAW

The compilation audit trail records this dataset's own complete compilation history — every state transition, every escalation, every disambiguation, every amendment, each distinctly typed and each carrying its own field-level detail — retained for as long as the mission's own evidence is retained, and ending at handoff exactly where this dataset's own responsibility ends; a trail reduced to only its terminal VALIDATED or REJECTED outcome, discarding the process that produced it, has not implemented this dataset's auditing discipline regardless of how accurate that terminal outcome happens to be.


PART XXIII — FAILURE TAXONOMY FOR THE COMPILER ITSELF

23.1 Nearly every prior Part in this dataset has named its own field- or discipline-specific failure mode as it went — burden inflation in Part VIII, authorization inflation in Part IX, ceiling deflation in Part X, schema mismatch in Part XII, confidence-tier inflation in Part XIII, amendment misclassification in Part XVIII, and more besides. This Part gathers all of them into a single named taxonomy, mirroring the structure Dataset #10 Part XXIII already established for its own family-wide failure catalog, so that a future implementation or auditor has one place to check a suspected failure against rather than needing to recall which of Parts IV through XXI happened to name it.

23.2 Field fabrication. Populating any field with a value carrying no honest extracted, resolved, defaulted, or disambiguated provenance tag — the single failure this dataset treats as most severe throughout, named directly in Part I §1.3's opening boundary, Part II §2.6's provenance discipline, and Part XVI §16.6's escalation-avoidance-through-silent-inference case specifically. Every other failure class in this taxonomy is, at bottom, a specific instance or consequence of this one.

23.3 Boundary crossing. Compiling a field in a way that performs another system's owned computation rather than declaring an input for that system to consume — Part VIII §8.5's confidence-scoring crossing into DRK-03's territory, Part X §10.4's cost-estimation crossing into CO-01's territory, Part IX's authorization-adjudication crossing into the future Dataset #16's territory. This dataset treats boundary crossing as severely as field fabrication, since a value produced this way, however internally consistent it appears, is not a value this dataset had any legitimate authority to compute.

23.4 Inflation as an unexamined hedge. Defaulting a field toward a more conservative-sounding or more demanding value than the mission's actual signals warrant, on the unexamined theory that more is always safer — named specifically for required evidence (Part VIII §8.8), authorized authentication (Part IX §9.7), confidence-level requirement (Part XIII §13.7), and preservation obligation (Part XIV §14.7). This dataset treats inflation as a genuine failure, not a caution, because it imposes real cost (Part X) and real delay on missions that never needed the inflated value in the first place.

23.5 Deflation as a false economy. The mirror failure to inflation, defaulting a field toward a less demanding value than actual signals warrant on the theory that minimal commitment is always safer — named specifically for cost and time ceilings (Part X §10.7) and preservation obligation (Part XIV §14.6). This dataset treats deflation as equally severe as inflation, since both directions produce a contract calibrated against a fiction rather than the mission's actual needs.

23.6 Escalation avoidance and escalation overuse. Part XVI §16.6 and §16.7 already name both directions of this failure pair directly: silently inferring a value for a field genuinely failing the discrete default-exists test, or conversely escalating a field that test actually clears. Listed here as its own numbered class both because Part XVI §16.6 names it this dataset's single most consequential failure mode and because it is, structurally, the escalation-specific instance of the boundary-crossing failure named in 23.3 — a compiler that infers past a genuine escalation case has crossed from its own legitimate compilation authority into a decision it had no basis for making.

23.7 Amendment misclassification. Part XVIII §18.8 and §18.9 already name both directions: compiling a genuine new mission as a scoped amendment, or conversely discarding a genuine amendment's progress by recompiling from scratch. Listed here to complete the taxonomy's coverage of every negative-outcome mechanism this dataset specifies, alongside 23.8 and 23.9 below.

23.8 Rejection misuse. Part XIX §19.9 already names this directly: rejecting a request that Part XVI's escalation discipline should have handled through a clarifying question instead, using rejection as a shortcut past the more effortful but more honest escalation path.

23.9 State-machine violation. Part XV §15.3's no-skip, no-silent-regression discipline violated — a contract advancing to VALIDATED without every named consistency check actually running (Part XV §15.6's own named risk), or a state transition recorded out of the sequence 15.2 specifies. This failure is distinct from a field-level failure in that it corrupts the compilation audit trail's own integrity (Part XXII §22.8) rather than any single field's value, making the resulting contract's own history unreliable even where every individual field happens to hold an honest value.

23.10 Incomplete audit trail. Part XXII §22.8's own named failure: persisting only a contract's terminal state while discarding the process that produced it, defeating the auditing discipline Part XXII exists to establish regardless of how accurate the terminal outcome itself is.

23.11 This taxonomy's own completeness deserves one explicit note, mirroring the honesty Dataset #10 Part XVI §16.10 already models for its own self-auditing limits: this dataset's nine named classes cover every failure mode this dataset's own text has identified while drafting itself, but a future implementation encountering a failure mode not named here has not thereby proven the failure is acceptable — it has found a gap in this taxonomy, and Part XXIV's evolution discipline is the correct channel for adding a tenth class, never a silent assumption that an unnamed failure is therefore a non-failure.

23.12 A closing observation on how these nine classes relate to one another: field fabrication (23.2) and boundary crossing (23.3) are the two foundational failures every other class in this taxonomy ultimately reduces to in some form — inflation and deflation are specific patterns of field fabrication applied to a defaulting decision; escalation avoidance is boundary crossing applied to the line between what a compiler may infer and what it must ask about; amendment misclassification and rejection misuse are boundary crossing applied to this dataset's own process-control mechanisms rather than to a field value. A future implementer who internalizes only 23.2 and 23.3 deeply, understanding why each of the other seven classes is a specific manifestation of one or the other, has understood this dataset's central discipline more completely than one who memorizes all nine as independent, unrelated rules.

23.13 This taxonomy's detection method deserves one final note distinguishing it from Dataset #10 Part XVI's own self-auditing mechanism, cited in 23.11: Dataset #10's fabric checks its own stored objects against external reference points (a stored hash, a known-good pattern) at any later time, independent of when the object was originally constructed. This dataset's own failures, by contrast, are detectable primarily through Part XXII's audit trail, examined either during Part XV's own INTERNALLY-CHECKING state (catching a violation before VALIDATED is ever reached) or after the fact by a human or automated reviewer working through a completed trail. A future implementation gains real practical value from running as many of this taxonomy's nine checks as possible during INTERNALLY-CHECKING itself, since a failure caught before handoff never reaches a downstream engine at all, while a failure caught only in post-hoc audit trail review has already been acted on by whatever consumed the flawed VALIDATED contract in the meantime.

PART XXIII FINAL LAW

This dataset's own failure taxonomy names nine classes — field fabrication, boundary crossing, inflation as an unexamined hedge, deflation as a false economy, escalation avoidance and overuse, amendment misclassification, rejection misuse, state-machine violation, and an incomplete audit trail — every one of them, on inspection, a specific manifestation of this dataset's two foundational failures: fabricating what should have been honestly declared, or deciding what belonged to another system's own owned authority.


PART XXIV — EVOLUTION AND AMENDMENT SPECIFIC TO THIS DATASET

24.1 This Part governs changes to this dataset's own text, distinct from Part XVIII's discipline governing changes to a compiled Mission Contract — the same distinction Dataset #10 Part XXIV already draws between amending an Evidence Object and amending the dataset that governs how Evidence Objects are constructed, adopted here for the identical reason.

24.2 This dataset's sixteen-field schema (Part III) is versioned as a whole, per the same schema-versioning discipline Dataset #10 Part XX already establishes for the Evidence Object schema, adapted here to this dataset's own artifact. A change adding a seventeenth field, removing one of the existing sixteen, or altering an existing field's fundamental meaning is a version-incrementing change; a change that only refines a field's compilation mechanics within Parts IV through XIV's existing text — a better default, a more precise escalation trigger — does not require a version increment, since the field's own contract with downstream consumers (Part III's own definitions) remains unchanged even as this dataset's internal guidance for populating it improves.

24.3 Amendment triggers this dataset names explicitly, mirroring Dataset #10 Part XXIV's own worked pattern: a new field genuinely needed by a downstream consumer this dataset did not originally anticipate; Part XXII §22.5's own aggregate-escalation-pattern signal indicating a documented default needs revision; a composition-boundary change in a sibling dataset this dataset cites (CO-01, DRK-03, Dataset #10, or the future Dataset #03/#16 once they seal) that changes what this dataset may or must declare; and Part XXIII's own taxonomy proving incomplete against a failure mode a future implementation actually encountered, per 23.11's own honest acknowledgment of that possibility.

24.4 A worked example of the composition-boundary trigger specifically: if the future Dataset #16 Authorization, Compliance and Safety dataset, once sealed, defines a permitted-domains adjudication interface requiring a field shape this dataset's own Part IX did not anticipate — a structured allow-list format rather than the loosely-described scope Part IX §9.2 currently specifies — this dataset requires Part IX's own text to be revised to match Dataset #16's actual sealed interface, never the reverse, since Part I §1.3's boundary already establishes that adjudication authority belongs entirely to Dataset #16, and this dataset's own declaration mechanics must conform to whatever interface the system actually holding that authority defines.

24.5 Coordinated review is required, per the identical discipline Dataset #10 Part XXIV already establishes, whenever a proposed change to this dataset would also require a corresponding change in a sibling dataset this dataset cites — a change to the required-evidence field's declared burden-level vocabulary (Part VIII §8.4) that DRK-03's own CONNECT computation logic would need to consume differently cannot be made to this dataset alone; it requires DRK-03's own maintainers, or whatever governance process oversees DRK-03, to confirm the corresponding consumption logic will actually accept the new vocabulary before this dataset's own text changes to describe it.

24.6 A two-trigger worked example, mirroring the pattern Dataset #10 Part XXIV already used for its own amendment illustration: consider Part XXII §22.5's aggregate-escalation signal revealing that the entities field escalates far more often than any other field even after Part XVI §16.4's already-stated expectation that entities would escalate "most often" among the sixteen — a magnitude beyond what this dataset's own text anticipated, suggesting Part IV's entity-resolution defaults themselves may need strengthening rather than merely accepting frequent escalation as expected behavior. If this pattern coincides with the future Dataset #06 Large-Scale Crawl Frontier Engineering dataset sealing with its own entity-resolution capability this dataset's Part IV could plausibly compose with rather than duplicate, both triggers point toward the same Part IV revision, and this dataset requires the revision to address both signals together in one coordinated amendment rather than two separate, potentially conflicting patches.

24.7 A boundary worth restating specifically for this Part: amendment to this dataset's own text is never performed to accommodate a specific difficult compilation case after the fact — a future implementer encountering one unusually hard request should apply this dataset's existing mechanics (escalation where warranted, defaulting where a documented default applies, rejection where one of Part XIX's five classes genuinely fits) rather than treating the difficulty as grounds for a targeted, ad hoc amendment to this dataset's own specification. Genuine amendment triggers, per 24.3, arise from patterns visible across many compilations or from a sibling system's own sealed interface changing, never from a single hard case this dataset's existing mechanics could, on reflection, actually handle.

24.8 With evolution and amendment specified, this dataset turns in its closing Part, Part XXV, to its own done criteria — the final restatement of what SEALED requires, the contamination and citation audits this dataset's own honesty discipline demands of itself, and this dataset's own final law.

24.9 A version-history discipline this dataset requires but has not yet stated explicitly: every version-incrementing change per 24.2 is recorded, at the point it is made, with the specific trigger from 24.3 that motivated it and the specific Parts actually revised — never a bare version-number bump with no accompanying record of why. This mirrors Dataset #10 Part X's change-history discipline for the Evidence Object one level up, applied here to this dataset's own text rather than to any artifact this dataset compiles: a future maintainer encountering version three of this schema should be able to trace, without needing to reconstruct it from git history or institutional memory, exactly which of 24.3's four trigger classes produced versions two and three respectively, and what each version actually changed.

24.10 A worked example of the taxonomy-gap trigger specifically, since 24.3's other three triggers each already received a worked illustration earlier in this Part while this fourth one has not: suppose a future implementation encounters a compiled contract whose entities field resolved correctly, whose required-evidence and confidence-level-requirement fields were both populated honestly per their own respective disciplines, but whose depth and breadth fields (Part VI) were calibrated against a stale snapshot of the required-evidence field taken before Part VIII §8.11's own late-resolution requirement had actually run — a genuine sequencing bug distinct from every failure Part XXIII's nine classes currently name, since it is neither fabrication, boundary crossing, inflation, deflation, an escalation failure, an amendment failure, a rejection failure, a state-machine violation in Part XV §15.3's sense, nor an incomplete audit trail. This dataset requires such a discovery to trigger a Part XXIII amendment adding a tenth named class — sequencing violation, say — rather than being folded silently into the nearest existing class or left undocumented as an one-off bug fix, precisely because 23.11 already commits this dataset to treating an unnamed failure as a taxonomy gap rather than proof of acceptability.

24.11 A closing boundary on this Part's own scope, distinguishing it one final time from Part XVIII: everything this Part specifies governs the text of this dataset itself — its sixteen-field schema, its documented defaults, its named failure classes, its own version history. Nothing in this Part governs any already-compiled Mission Contract, past or future; a schema version change does not retroactively alter a contract compiled under an earlier version, and this dataset requires such a contract to remain valid under the schema version it was actually compiled against, migrated forward only if Part XXIV's own future revision explicitly specifies a migration path, exactly as Dataset #10 Part XX already requires for its own Evidence Object schema versioning.

PART XXIV FINAL LAW

This dataset's own text amends only under a genuine trigger — a downstream field need this dataset did not anticipate, an aggregate escalation pattern revealing a default needs revision, a sibling system's sealed interface changing what this dataset may declare, or a taxonomy gap a real failure exposed — never to accommodate a single difficult compilation case this dataset's existing mechanics could, on reflection, already handle, and never without coordinated review whenever the change would also require a corresponding change in a system this dataset cites rather than owns.
