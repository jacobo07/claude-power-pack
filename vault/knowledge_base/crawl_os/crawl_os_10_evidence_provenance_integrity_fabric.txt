PART I — MISSION AND BOUNDARY: WHAT THIS DATASET ELABORATES, AND WHAT IT DOES NOT RESTATE

1.1 Dataset #01 named the Evidence Object as Crawl OS's constitutional unit of output and listed, in its Part VII, every field that object must carry before it may be treated as evidence rather than mere retrieved text. That list is a schema. A schema states what must exist. It does not state how the thing that must exist actually comes to exist, stays trustworthy across the interval between construction and consumption, survives contact with a second acquisition of the same source months later, or gets handed safely to whichever engine needs it next without leaking a field that should never have left the fabric that built it. This dataset is the operational elaboration of that gap: given the schema Dataset #01 already fixed as binding, how does the Evidence Integrity Fabric named in Dataset #01's Part V, §5.9, actually construct, verify, protect, store, retire, and hand off the object the schema describes.

1.2 The relationship between this dataset and Dataset #01 is inheritance, not parallel authority, and the distinction must be stated with the same sharpness Dataset #01 itself demanded of every composition boundary it drew. This dataset does not define a second Evidence Object schema, does not restate the required-fields list from Dataset #01's Part VII §7.2, and does not relitigate any ownership boundary Dataset #01 already settled — not the cost ladder, not the Reality Gates' pass criteria, not the DRK-03 confidence-vocabulary boundary, not the GK-04 graph boundary. Every one of those questions has exactly one binding answer, and that answer lives in Dataset #01. What this dataset adds is the machinery: the lifecycle a single Evidence Object moves through, the mechanics by which a provenance chain is actually assembled rather than merely required to exist, the taxonomy of anomaly signals the fabric watches for in practice, and the operational detail beneath compositions Dataset #01 only declared.

1.3 A dataset that violated this boundary — that quietly reproduced Dataset #01's field list under a slightly different framing, or that redefined confidence scoring in terms not already anchored to DRK-03's CONNECT vocabulary — would have committed exactly the failure Dataset #01's own Part XXV §25.5 named as an inheritance rule: a family member that restates rather than cites its dependency has broken the single-answer property that makes the family coherent. This dataset's own done-gate, stated fully in Part XXV below and in the dataset contract that preceded this Part, includes a citation check for exactly this reason — not as a formality, but because the failure mode it guards against is common enough among elaborating datasets to need a mechanical check rather than an author's good intentions.

1.4 The reason an elaborating dataset is needed at all, rather than simply expanding Dataset #01's own Part VII until it covers operational mechanics too, is the same reason Dataset #01's own fourteen engines were not collapsed into one Part: a schema and its operational mechanics are different kinds of knowledge, consumed by different kinds of reader at different moments. An engineer implementing the Evidence Integrity Fabric's construction logic needs the lifecycle state machine, the hashing procedure, the anomaly-detection taxonomy — material this dataset owns. A downstream engine deciding whether to trust an Evidence Object it received needs the schema and the invariants Dataset #01 already states. Conflating the two into one document would force every reader through material irrelevant to their actual question, and a corpus optimized for that convenience over precision has already made the tradeoff this entire family exists to refuse.

1.5 The Evidence Integrity Fabric's charter, restated from Dataset #01's Part V §5.9 as the fixed point this dataset elaborates around, is to construct, for every acquisition, the Evidence Object described in Part VII, and to refuse to pass an acquisition downstream as institutionally usable evidence until that object's required fields are populated with genuine, non-fabricated values, including an honest confidence score and any observed anomaly signals. Every Part that follows in this dataset is an elaboration of some clause inside that single sentence: "construct" becomes Part II's lifecycle and Part III's provenance mechanics; "genuine, non-fabricated values" becomes Part IV's hashing, Part V's anomaly taxonomy, and Part IX's tamper-evidence; "honest confidence score" becomes Part VI; "refuse to pass downstream until populated" becomes Part XI's package composition and Part XVII's gate interface.

1.6 A second charter clause, present in Dataset #01 but easy to under-weight, deserves explicit elevation here because this dataset is where its operational consequence actually lives: the no-secrets doctrine of Dataset #01's Part X applies to the Evidence Object as concretely as to any other artifact the estate produces, and an Evidence Object that faithfully records a session cookie, an authentication token, or a credential observed during acquisition, because faithfulness to what was observed is otherwise this dataset's highest value, has violated a doctrine with strictly higher priority than faithful recording. Part VII of this dataset exists precisely to resolve that apparent tension: the fabric records that a credential was present and relevant to the acquisition's success without ever persisting the credential's actual value, and the mechanics of doing that correctly, rather than merely being told to do it, is owned here.

1.7 This dataset's scope is bounded on its far side as sharply as on its near side. It does not own what happens to an Evidence Object once it is institutionalized and consumed by the Knowledge Distillation Fabric — that consumption, and the compact reusable artifacts it produces, belongs to the engine chartered for it in Dataset #01's Part V §5.10, and a future Crawl OS dataset elaborating that engine will cite this dataset's Part XI on package composition as its own upstream boundary, exactly as this dataset cites Dataset #01's Part VII as its own. Nor does this dataset own the Crawl Cortex Supervisor's evidence-quality-assurance loop, chartered in Part V §5.12, though Part XVI below describes the signals this fabric surfaces for that loop to consume — surfacing a signal is not the same as owning the loop that consumes it, and the distinction is load-bearing throughout this dataset exactly as it was throughout Dataset #01.

1.8 A worked framing closes this Part and previews the register the rest of this dataset holds throughout. Consider a single acquisition: a page fetched, converted, and handed to the fabric. Dataset #01's Part VII already answers what fields the resulting Evidence Object must carry. This dataset answers a different, later set of questions about that same object: in what order were its fields populated, and could construction have been interrupted partway through in a way that left a half-formed object masquerading as complete (Part II); how was its provenance chain actually assembled from the mission context rather than merely asserted to exist (Part III); what, mechanically, triggered its confidence score, and did that mechanism actually call out to DRK-03 rather than quietly reimplementing a parallel scale (Part VI); if the object is queried eleven months later by a mission that has never heard of the one that created it, how does it get found and trusted (Part XXII); and if two Evidence Objects for the same source, acquired eight months apart, disagree, which one does a downstream consumer believe and why (Part X, Part XIII). None of these questions has an answer in Dataset #01, because none of them is a question about what an Evidence Object must contain. All of them are questions about what actually happens to one, and that is this dataset's entire territory.

1.9 The relationship this dataset holds to the family's remaining acquisition-facing datasets — the Fetch Runtime Fabric's own eventual dataset, the Browser Interaction Fabric's, the Extraction and Rule Induction Fabric's — is upstream rather than composed: those engines produce the raw material an Evidence Object is constructed from, but they do not construct the object themselves, and a future dataset elaborating one of those engines that also attempted to define how its own output becomes an Evidence Object would have reached backward into this dataset's territory exactly as Dataset #01's Part V §5.16 forbade any engine from reaching into a neighboring stage's responsibility. This dataset owns construction from the moment raw acquisition output arrives at the fabric's boundary onward; it does not own producing that raw output in the first place.

1.10 A final scoping note addresses the dataset's own self-description honestly, in the register Dataset #01's Part XXV §25.11 modeled for its own closing Part. Every mechanism this dataset specifies — the lifecycle state machine, the hashing procedure, the query interface — is specified today as a binding design, not verified today as a running implementation, because the downstream module this dataset targets, `modules/crawl_os/evidence.py`, is PLANNED and does not yet exist, exactly as Dataset #01's own downstream module does not yet exist. This dataset's authority is the same authority Dataset #01 held over its own Parts: it binds whoever eventually implements the fabric to build the mechanics specified here, rather than designing new ones at that time, and it states the interim state honestly rather than claiming an operational status this repository's liveness gate has no mechanism to certify for a prose dataset.

PART I FINAL LAW

This dataset elaborates the Evidence Integrity Fabric's construction, verification, storage, and hand-off mechanics; it does not restate Dataset #01's Evidence Object schema, confidence-vocabulary boundary, or graph-composition boundary, each of which has exactly one binding answer that lives in Dataset #01 and is cited here, never reproduced. An elaborating dataset that quietly restates its dependency's territory has broken the single-answer property that makes this family coherent.


PART II — THE EVIDENCE OBJECT LIFECYCLE: CONSTRUCTION THROUGH INSTITUTIONALIZATION OR DISCARD

2.1 An Evidence Object is not created complete in a single instant. It moves through a small, fixed sequence of states between the moment raw acquisition output first arrives at the fabric's boundary and the moment it is either institutionalized, per Dataset #01's Part XIV, or discarded as insufficient, and this Part names that sequence precisely because a fabric that treats object construction as an undifferentiated blob of work happening "sometime after acquisition, before consumption" cannot answer the operational questions Part I raised: whether construction was interrupted, whether a partially populated object could be mistaken for a complete one, or at what point a fabricated field would actually have been inserted had the fabric's own discipline failed.

2.2 The first state is RAW-RECEIVED: the instant the fabric takes custody of an acquisition's raw output — fetched bytes, rendered DOM, extracted structured content — from whichever upstream engine produced it, before any Evidence Object field has been populated. An object in this state is not yet an Evidence Object at all in any sense a downstream consumer could observe; it is the fabric's own internal working state, and no engine outside the fabric may reference or depend on anything in this state, because nothing in it has yet been verified, hashed, or checked for anomaly.

2.3 The second state is FIELDS-POPULATING: the fabric is actively deriving each of Dataset #01's Part VII §7.2 required fields from the raw input and from context supplied by the mission — the canonical URL, the redirect chain, the acquisition timestamp, the raw and normalized hashes, the provenance chain's mission and frontier references. This state is explicitly interruptible in the fabric's own design: a crash, a timeout, or an upstream engine failure partway through population must never result in a partially populated object silently reaching a later state. The fabric's own transactional discipline, elaborated fully in Part IX below under tamper- and interruption-evidence, guarantees that an interrupted FIELDS-POPULATING object either resumes from a checkpoint or is discarded entirely; it is never promoted forward with silently absent fields.

2.4 The third state is SIGNAL-EXAMINED: every required field exists, and the fabric now runs the anomaly-detection taxonomy of Part V against the populated object, checking topical consistency, length against site-memory baseline, title-path consistency, and every other signal named there, recording whatever anomalies are actually observed rather than assuming their absence. An object that completes SIGNAL-EXAMINED with zero anomalies recorded has been checked and found clean; an object that never reached this state has not been checked at all, and the fabric's own internal bookkeeping must be able to distinguish those two conditions without ambiguity, because "no anomalies on record" and "never examined for anomalies" look identical to a careless downstream reader and must never look identical to the fabric itself.

2.5 The fourth state is SCORED: the signals recorded in SIGNAL-EXAMINED are handed, per Part VI's mechanics, to DRK-03's confidence-scoring logic, and the returned score is written into the object's confidence field. An object may complete SCORED with a low or even near-zero confidence value; SCORED describes that the scoring mechanism actually ran and actually derived its value from real signals, not that the resulting value was favorable. This distinction is Dataset #01's Part VII §7.6 restated as a lifecycle guarantee rather than a prose warning: structural completeness of the scoring step is orthogonal to the score it produces.

2.6 The fifth state is GATE-EVALUATED: the fully populated, signal-examined, scored object is submitted to the Production Reality Gates of Dataset #01's Part XIII, and the object records which gates it passed, which it failed, and — per Dataset #01's Part XIII §13.15 provision for gates that do not apply to a given acquisition type — which gates were marked not applicable and why. An object here may fail every gate and still be a well-formed GATE-EVALUATED object; failing gates is a recorded, honest outcome of evaluation, not evidence that evaluation itself malfunctioned.

2.7 From GATE-EVALUATED, the lifecycle branches into exactly two terminal states, and the branch point is where Part I's framing question — institutionalized or discarded — is actually decided. INSTITUTIONALIZATION-ELIGIBLE is reached when the object's gate results and the four criteria of Dataset #01's Part XIV are jointly satisfied; from there, the object proceeds to Part XI's package composition and, where applicable, Part XVIII's GK-04 edge construction. RETAINED-NON-INSTITUTIONAL is reached when the object is well-formed and honestly scored but does not meet Part XIV's institutionalization criteria; such an object is not deleted — Dataset #01's Part XIV never authorized deletion of a well-formed but non-institutionalized object — it is retained in evidence memory, queryable by the specific mission that produced it, but is not promoted into the estate's broader reusable-knowledge surface.

2.8 A third, narrower terminal state exists alongside the two above and must not be confused with either: CONSTRUCTION-ABORTED, reached when FIELDS-POPULATING is interrupted in a way Part IX's transactional discipline cannot resume, or when the raw input itself is judged, before any field population meaningfully begins, to be unusable for reasons unrelated to content quality — a transport-level corruption, an acquisition the Governance, Safety, and Authorization engine retroactively revokes mid-construction. A CONSTRUCTION-ABORTED record is deliberately minimal — it exists chiefly to prevent the same raw input from silently being re-attempted in a loop without anyone noticing the pattern — and it is explicitly not an Evidence Object in the sense Dataset #01's Part VII defines one; it is the fabric's own record that construction was attempted and did not complete, distinct in kind from RETAINED-NON-INSTITUTIONAL, which is a complete object that simply scored too low to institutionalize.

2.9 The lifecycle is deliberately linear with no state permitted to be skipped, and this linearity is itself a design decision worth defending explicitly: a fabric permitted to jump directly from RAW-RECEIVED to SCORED, bypassing SIGNAL-EXAMINED under time pressure, would produce exactly the fabricated-value failure Dataset #01's Part VII §7.9 names as indistinguishable in consequence from Part I's founding clean-Markdown failure. Every state transition in this lifecycle exists specifically to make that shortcut structurally unavailable rather than merely discouraged, mirroring the same design principle Dataset #01's Part VI applied to the cost ladder's escalation discipline.

2.10 A worked trace closes this Part. A mission acquires a documentation page. RAW-RECEIVED: the fabric takes custody of the fetched HTML. FIELDS-POPULATING: canonical URL, redirect chain (empty, no redirect occurred), timestamp, raw hash, normalized hash, provenance chain referencing the mission and frontier item, all populate without interruption. SIGNAL-EXAMINED: the content's length is checked against site memory's baseline for this documentation site and found consistent; no anomaly is recorded. SCORED: the signals are handed to DRK-03's logic, which returns a high confidence value given the clean signal profile. GATE-EVALUATED: Gates A through J are checked; all applicable gates pass. INSTITUTIONALIZATION-ELIGIBLE: Part XIV's four criteria are satisfied — the extracted API signatures are reusable, general to future missions against the same project, strongly provenanced, and not already redundant with an existing institutionalized record. The object proceeds to Part XI's package composition. At no point in this trace did the fabric assert a value it had not yet derived, and that discipline, more than any single field in the schema, is what this Part exists to make mechanically enforceable rather than merely stated.

PART II FINAL LAW

An Evidence Object moves through a fixed, linear lifecycle — RAW-RECEIVED, FIELDS-POPULATING, SIGNAL-EXAMINED, SCORED, GATE-EVALUATED, then exactly one of INSTITUTIONALIZATION-ELIGIBLE, RETAINED-NON-INSTITUTIONAL, or CONSTRUCTION-ABORTED — and no state may be skipped. An interrupted or partial object is never silently promoted forward; it resumes from a checkpoint or is recorded as aborted, and a well-formed but low-scoring object is retained honestly rather than deleted or disguised.


PART III — PROVENANCE CHAIN MECHANICS: CONSTRUCTING AN UNBROKEN LINEAGE

3.1 Dataset #01's Part VII §7.5 named the provenance chain field as the mechanism that makes Part III's obligation to explain any claim's provenance on demand mechanically possible, and named what the chain must be able to answer: which mission requested an acquisition, which frontier item it corresponds to, and which prior acquisition, if any, it supersedes. That is the field's contract. This Part specifies how the chain is actually assembled during FIELDS-POPULATING so that the contract is met by construction rather than by a later, separate reconciliation step that could itself fail silently.

3.2 The chain's assembly begins before the acquisition itself, at the moment the Resource Discovery and Frontier Engine hands a prioritized item to the Acquisition Strategy Router, per Dataset #01's Part V §5.3/§5.4. That hand-off already carries the two identifiers the provenance chain's first two elements require: the mission identifier, assigned once by the Crawl Intent Compiler when the Mission Contract is produced, and the frontier item identifier, assigned once by the Resource Discovery and Frontier Engine when the item is added to the frontier. The fabric does not generate either identifier itself; it receives both, attached to the acquisition context, and copies them into the object's provenance chain field verbatim during FIELDS-POPULATING. A fabric that generated its own identifier for either value, rather than propagating the one already assigned upstream, would have created a second numbering scheme that a downstream query following the chain outward would have no way to reconcile with the one the mission and frontier engines actually use.

3.3 The chain's third element, the superseded-acquisition reference, is populated differently, because unlike the mission and frontier identifiers it is not always present. During FIELDS-POPULATING, the fabric queries evidence memory for any prior Evidence Object whose canonical URL matches the current acquisition's canonical URL and whose mission differs from or equals the current one, per the cross-mission reuse discipline elaborated fully in Part XV. If exactly one such prior object exists, its identifier populates the supersedes field and Part X's change-history mechanics activate. If none exists, the field is explicitly recorded as empty rather than omitted, preserving the same absent-versus-never-checked distinction Part II §2.4 established for anomaly signals. If more than one plausible prior match exists — a case that should not arise under correct fabric operation but which Part XVI's self-auditing exists specifically to catch — the object is flagged for manual resolution rather than the fabric guessing which prior object is the correct one to supersede.

3.4 A distinct and easily conflated question the provenance chain does not answer, and must not be extended to answer, is which specific fragment of an Evidence Object's content actually supports a specific downstream claim — Dataset #01's Part VII §7.2 lists that as its own separate required field, populated not during FIELDS-POPULATING but later, at the moment the Knowledge Distillation Fabric actually draws on a specific fragment to construct a claim. The provenance chain answers where an object came from; the supporting-fragment field answers what, within an object, a specific later claim actually rested on. Conflating the two would collapse a lineage question and a citation question into one field that could not honestly answer either, exactly the failure Dataset #01's Part VII §7.4 warned against for the dual-hash fields.

3.5 The chain's construction must survive a case the founding resource document's own worked examples repeatedly surface: a mission that spans multiple sessions, paused and resumed, possibly across a fabric restart. Because the mission and frontier identifiers are assigned once, at mission and frontier-item creation respectively, and are propagated rather than regenerated, a resumed mission's acquisitions carry identical provenance-chain values to the ones a completed-in-one-session mission would have carried, and nothing about the fabric's own downtime between the pause and the resume is visible in, or relevant to, the chain itself. Session continuity is a Governance and orchestration concern; provenance-chain correctness does not depend on it.

3.6 A worked failure illustrates why this mechanics-level specification, rather than merely stating that a provenance chain must exist, is necessary. Consider a fabric implementation that populates the mission identifier from a global "current mission" variable read at the moment FIELDS-POPULATING executes, rather than from context explicitly attached to each individual acquisition. Under concurrent execution — two missions' acquisitions being processed by the fabric in an overlapping window, which the Crawl Cortex Supervisor's own architecture in Dataset #01's Part V §5.12 assumes is possible — that global variable can be overwritten by the second mission's context before the first mission's acquisition finishes populating, silently attaching the wrong mission identifier to the first acquisition's Evidence Object. The resulting object would be structurally complete, pass SIGNAL-EXAMINED and SCORED without any anomaly triggering, and carry a provenance chain that is fluently wrong rather than absent — a failure invisible to every check this dataset specifies elsewhere, because every other check examines whether a field is populated and internally consistent, not whether it is populated with the correct value from a source outside the object itself. The fabric's design must therefore attach mission and frontier context explicitly to each acquisition at the moment it enters RAW-RECEIVED, never read implicitly from shared, mutable state at the moment a field happens to populate.

3.7 The provenance chain's downstream resolution obligation, restated from Dataset #01's Part III §3.10, requires that a Crawl OS engine be able to answer a provenance query on demand without external reconstruction. This dataset's contribution to that obligation is the query interface specified fully in Part XXII, but the obligation's precondition is established here: a chain that was correctly assembled during construction is a chain that can be resolved later without reconstruction; a chain assembled sloppily, or reconstructed after the fact from log correlation because the field itself was left empty, is not the mechanism Dataset #01's Part III actually committed to, however similar the eventual query result might look to a downstream reader who cannot tell the two apart from outside.

3.8 The chain's relationship to Part XVIII's GK-04 edge construction deserves one clarifying sentence here, expanded fully in that later Part: the provenance chain is Crawl OS's own internal lineage record, complete in itself, and the GK-04 edge is a separate, compact pointer constructed from a subset of the chain's content once institutionalization is judged. The chain does not depend on GK-04's writer existing to be complete and correct; it is fully functional as Crawl OS's own record from the moment FIELDS-POPULATING finishes, exactly as Dataset #01's Part XVI §16.7 already established.

3.9 A final mechanical note addresses what happens when the frontier item a provenance chain references is itself later removed from an active frontier — for instance because the mission's contract was amended mid-execution to narrow its scope, per whatever future dataset governs mission compilation in more operational detail than Dataset #01's own Part III provides. The provenance chain's frontier-item reference is a historical fact about what was requested at acquisition time, not a live pointer into the frontier's current state, and it is never retroactively updated or nulled when the frontier itself changes shape later. An Evidence Object's provenance is a record of what happened, and a record that mutated to match a later, different state of the world it originally described would no longer be a record at all.

3.10 The chain's three elements — mission, frontier item, superseded-acquisition — are individually simple, and the temptation this Part exists to preempt is treating that simplicity as a reason to skip specifying assembly mechanics at all, on the theory that three fields populated from obvious sources need no further elaboration. The worked failure in 3.6 demonstrates why that theory is wrong: the fields are simple, but the discipline of populating them from explicitly attached context rather than ambient state, and of distinguishing an honestly empty supersedes field from one that was never checked, is exactly the kind of operational detail Part I named as this dataset's entire reason for existing separately from Dataset #01's schema-level Part VII.

PART III FINAL LAW

A provenance chain is assembled from context explicitly attached to each acquisition at RAW-RECEIVED — never from ambient or shared mutable state — and its three elements record, respectively, the requesting mission, the satisfied frontier item, and any prior acquisition it supersedes, the last of which is either populated from an actual evidence-memory match or explicitly recorded empty, never silently omitted. A chain assembled from the wrong source can be fluently, invisibly wrong in a way no other check in this dataset detects.


PART IV — THE DUAL-HASH DISCIPLINE IN PRACTICE: RAW INTEGRITY VERSUS NORMALIZED IDENTITY

4.1 Dataset #01's Part VII §7.4 established why two hashes, not one, are required: the raw hash proves what was actually received at the moment of acquisition, and the normalized hash identifies the content after Crawl OS's own canonical transformation, serving deduplication and change-detection purposes the raw hash cannot serve because two substantively identical acquisitions can differ in irrelevant raw-byte details. This Part specifies how each hash is actually computed during FIELDS-POPULATING, what algorithm and input boundary each one commits to, and the operational failure modes a fabric implementation must guard against to keep the two hashes' distinct purposes intact rather than collapsing back into the single, purpose-conflating hash Dataset #01 already rejected.

4.2 The raw hash's input boundary is precise and non-negotiable: it is computed over the exact byte sequence the fabric received from the upstream engine at RAW-RECEIVED, before any transformation, encoding normalization, or whitespace adjustment of any kind. If the acquisition involved a browser render, the raw hash covers the rendered DOM serialization the Browser Interaction Fabric handed off, not the initial HTML response that preceded rendering — the raw hash's job is to prove what the fabric actually took custody of, and for a rendered acquisition, that is the rendered output, not an earlier intermediate the fabric never directly received. A fabric that hashed the pre-render HTML for a rendered acquisition, on the theory that it is somehow more "raw," would produce a hash that proves the wrong claim: not what the fabric received, but what a different, earlier stage of the pipeline produced before handing off something else entirely.

4.3 The normalized hash's input boundary is equally precise but categorically different: it is computed over the content after the fabric's own canonical Markdown transformation, the same transformation Dataset #01's Part V §5.8 describes as unifying document and web content at the representation layer, and critically, after a fixed, documented set of normalization steps applied consistently across every acquisition — whitespace collapse, removal of acquisition-specific query parameters and session tokens detected in the content body itself, and canonicalization of any timestamp the page renders in a locale-dependent format. The specific normalization steps are not this Part's content to enumerate exhaustively, because that enumeration is exactly the kind of implementation detail a future contributor building `modules/crawl_os/evidence.py` will refine against real sites; what this Part fixes as binding is the requirement that the steps be fixed and documented once, applied identically to every acquisition, and never varied per-source in a way that would make two objects' normalized hashes incomparable without knowing which variant of normalization each one used.

4.4 The purpose the two hashes jointly serve — distinguishing "did the source send different bytes" from "did the substantive content change" — fails silently if the normalization step is too aggressive or too weak, and this Part names both failure directions explicitly because each is a real, distinct risk rather than a single generic "get normalization right" caution. Normalization too weak — for instance, failing to strip a session token embedded in a query parameter that appears in the page's own rendered links — produces false change signals: the same substantive content, acquired twice, yields two different normalized hashes because an irrelevant, acquisition-specific artifact leaked into the hash's input, and Part X's change-history mechanics will incorrectly record a change that never actually happened to the source's substance. Normalization too aggressive — for instance, stripping numeric content that happens to resemble a session token but is actually a meaningful data value on the page — produces the opposite failure: two genuinely different acquisitions collapse to the same normalized hash, and a real change in the source's substance goes undetected. Both failures are equally serious, and a fabric implementation tuning normalization against one risk without measuring its effect on the other has only moved the failure, not resolved it.

4.5 The relationship between the raw hash and Part IX's tamper-evidence mechanics deserves explicit statement here rather than deferral, because the two are easy to conflate and serve different purposes. The raw hash, computed once at FIELDS-POPULATING and stored in the object, is a fixed reference value. Part IX's tamper-evidence mechanism is the separate, later discipline of re-computing a hash over the object's currently stored content at query time and comparing it against that fixed reference, to detect whether the stored object has been altered since construction. The raw hash by itself proves nothing about tampering; it is the comparison, performed later, that does the tamper-detection work, and a fabric that computed the raw hash but never re-verified it against stored content at any later point would have built a field that looks like an integrity mechanism without actually functioning as one.

4.6 A worked example distinguishes the two hashes' practical consequences for a downstream consumer. A mission acquires a project's changelog page in January and again in July. The raw hashes differ, because the page's HTML includes a dynamically rendered "last visited" timestamp specific to each request — an entirely expected, acquisition-specific difference that says nothing about whether the changelog's actual content changed. The normalized hashes, computed after normalization strips exactly that kind of acquisition-specific noise, are compared: if they match, Part X's change-history mechanics record that the source was re-acquired but its substantive content is unchanged, and no new institutionalization event is triggered for content already institutionalized in January; if they differ, a genuine content change is recorded, and the July acquisition's Evidence Object supersedes the January one per Part III's mechanics, carrying forward a superseded-acquisition reference that lets any future consumer trace exactly when the changelog actually changed, as distinct from when it was merely re-visited.

4.7 The dual-hash discipline's failure mode, where it occurs in practice, is nearly always the collapse Dataset #01's Part VII §7.4 warned against in the abstract: an implementation under time pressure computing a single hash and populating both fields with the same value, treating the distinction as redundant because in the common case the two hashes happen to agree closely enough that no one notices the difference until a case like 4.6's changelog example actually arises and the collapsed implementation cannot distinguish a re-visit from a real change. This Part's mechanical specification — two distinct input boundaries, two distinct purposes, one comparison discipline that only functions correctly when the boundaries stay distinct — exists specifically to make that collapse visible in a code review of the fabric's implementation, rather than discoverable only months later when a downstream consumer receives a spurious change-history entry.

4.8 Both hash fields use the same underlying hash algorithm — the specific algorithm choice, again, is an implementation detail this dataset does not fix, beyond requiring that whatever algorithm is chosen is applied identically and consistently across every acquisition and every dataset in the Crawl OS family that constructs an Evidence Object, so that a hash comparison across two objects constructed months apart, possibly by different fabric versions, remains meaningful. A hash-algorithm change is itself a Part XX schema-versioning event, not a silent per-acquisition choice, for exactly the reason Dataset #01's Part VII §7.8 already established for the schema as a whole.

4.9 The normalized hash's role in cross-mission provenance, elaborated fully in Part XV, depends entirely on the consistency this Part establishes: two missions, acquiring the same source independently and unaware of each other, can only be recognized as having acquired substantively identical content if their normalized hashes were computed under the same fixed normalization rules. A normalization rule that varied by mission, or by which engine happened to construct the object, would silently break cross-mission deduplication in a way indistinguishable, from outside, to a fabric that never attempted deduplication at all — the mechanism would appear to exist, in the sense that a normalized hash field is populated, while never actually functioning, because no two objects' hashes could ever be meaningfully compared.

4.10 A final provision closes this Part and extends Dataset #01's Part VII §7.9 fabrication prohibition into hash-specific terms. A hash value is either computed from bytes the fabric actually possesses at the moment of computation, or the corresponding field is recorded as uncomputable and the specific reason is stated — a transport failure that left the fabric with a partial, unusable byte stream, for instance. A hash value copied forward from a prior acquisition of the same source, on the assumption that the content probably has not changed, is a fabricated field in every sense Dataset #01's Part VII already condemned, regardless of how often that assumption happens to be correct, because a hash's entire value as evidence depends on it having actually been computed from the bytes it claims to describe.

PART IV FINAL LAW

The raw hash proves what the fabric actually received; the normalized hash identifies substantive content after fixed, consistently applied canonical normalization. The two serve distinct purposes that collapse into uselessness the moment either is computed from the wrong input boundary, applied inconsistently across acquisitions, or copied forward rather than freshly derived from bytes the fabric actually possesses at computation time.


PART V — THE ANOMALY SIGNAL TAXONOMY: WHAT THE FABRIC ACTUALLY WATCHES FOR

5.1 Dataset #01's Part VII §7.3 named anomaly signals as the field in which the fabric records exactly the observations a downstream reader needs to independently judge whether to trust an acquisition, and Dataset #01's Part VIII §8.5 named several such observations in the course of specifying Gate B's content-verification discipline: topical consistency against the mission's objective, title-path consistency, length against a site's own baseline, and, where the mission's contract calls for it, visual consistency. This Part collects those scattered mentions into a single taxonomy, states each signal's precise detection mechanism, and adds signals Dataset #01 named the category for without naming the members.

5.2 The taxonomy is organized by what each signal actually detects, not by which pipeline stage produces it, because a downstream consumer reading an anomaly list cares what kind of doubt is being raised, not which internal component raised it. Four categories cover the signals this dataset specifies: consent and access anomalies, content-mismatch anomalies, structural anomalies, and provenance anomalies. Every signal named below belongs to exactly one category, and a signal that seems to straddle two — a case this Part addresses directly in 5.9 — is resolved by asking which category's downstream consequence is more severe, since severity, not detection mechanism, is what a downstream reader ultimately needs signaled.

5.3 Consent and access anomalies detect the specific failure Dataset #01's Part VIII named as its founding worked example: a consent wall, a cookie interstitial, a login gate, or a paywall preview rendered as a 200-status page. The detection mechanism combines two independent checks that must both run rather than either substituting for the other: a lexical check for known interstitial markers — phrases like "accept cookies," "sign in to continue," "subscribe to read more," maintained as a living pattern list rather than a fixed one, per Part XVI's self-auditing discipline — and a structural check for a page whose visible content-to-chrome ratio is anomalously low relative to the site's own baseline, catching interstitials whose specific wording the lexical list has not yet learned. Either check alone under-detects; the lexical check misses novel wording, and the structural check alone cannot distinguish a genuinely short page from an interstitial, which is why the two are combined rather than treated as alternatives.

5.4 Content-mismatch anomalies detect the case Dataset #01's Part VIII §8.9 worked through as its closing example: content that is genuine, well-formed, and not an interstitial, but that does not match what the mission actually sought. The detection mechanism is the length-against-baseline comparison Part VIII already specified — site memory's recorded typical length for this site and page type, compared against the current acquisition's actual length, with a deviation beyond a threshold tuned per page type rather than fixed globally, since a reference page's typical length and an article's typical length differ by an order of magnitude and a single global threshold would either over-trigger on one or under-trigger on the other. A second content-mismatch signal, distinct from length, is topical drift: the extracted content's dominant topic, assessed via the same classification the Extraction and Rule Induction Fabric already applies for page-type classification per Dataset #01's Part V §5.7, compared against the mission's stated objective, flagged when the two diverge beyond what the mission's own breadth tolerance permits.

5.5 Structural anomalies detect a page that is neither an interstitial nor topically mismatched but is nonetheless malformed in a way that should reduce confidence in what was extracted from it: a title inconsistent with the requested URL's path (Dataset #01's Part VIII §8.5's title-path consistency check), a redirect chain terminating on a domain unrelated to the one originally requested, a MIME type inconsistent with what the acquisition method expected to retrieve, or extracted structured content — a table, a list, a code block — that fails basic well-formedness for its own claimed structure, such as a table whose rows report inconsistent column counts. These signals share a common shape: each one is a consistency check between two properties of the same acquisition that should agree under normal conditions and whose disagreement, whatever its cause, is itself information a downstream reader needs.

5.6 Provenance anomalies detect a narrower but structurally important case: a signal generated not from the acquired content itself but from the acquisition's own metadata proving internally inconsistent — an acquisition timestamp that precedes the mission's own creation timestamp, a redirect chain whose length exceeds a sane bound suggesting a redirect loop the fetch layer only partially escaped, or a session profile reference that does not resolve to any session the Browser Interaction Fabric's session governance actually recognizes. These anomalies most often indicate a defect in the acquiring engine or the fabric's own construction logic rather than anything about the target site, and Part XVI's self-auditing discipline treats a provenance anomaly's recurrence rate as one of its primary health signals for exactly that reason: a rising rate of provenance anomalies points inward, at the fabric or its upstream engines, not outward at an increasingly hostile web.

5.7 Every anomaly signal, once detected, is recorded with three components, not merely a flag: the signal's category, a machine-checkable severity tier (informational, degrading, or disqualifying), and the specific observed value that triggered it — the actual length observed versus the baseline it deviated from, the actual title versus the actual URL path, not merely "title-path mismatch: true." The specific value is required because Part VI's confidence-scoring mechanics, and any future dispute-resolution review under Part XIII, need the underlying observation, not merely the boolean fact that some check failed, to make a defensible judgment about how much the anomaly should actually discount confidence.

5.8 The severity tiers carry distinct, fixed consequences for the SCORED state's downstream confidence value, and the mapping is deliberately coarse rather than a continuous weighting, because a continuous weighting invites exactly the kind of ad hoc tuning per-signal that would make two objects' confidence scores incomparable unless a reader also knew the exact weight schedule in effect when each was computed. An informational-tier anomaly is recorded but does not itself gate confidence below a high value if no other anomaly is present — a minor, common, and usually harmless deviation. A degrading-tier anomaly caps the achievable confidence value at a moderate ceiling regardless of how clean every other signal is. A disqualifying-tier anomaly — chiefly a confirmed consent/access anomaly or a confirmed content-mismatch anomaly at high deviation — forces the confidence value to the lowest band and, per Part XVII, is sufficient by itself to fail Gate B regardless of every other gate's result.

5.9 A signal that appears to straddle two categories is resolved, as 5.2 previewed, by its more severe classification: a page whose title is inconsistent with its URL path (structural, by 5.5's definition) and which also fails the topical-drift check (content-mismatch, by 5.4's definition) is recorded under both categories simultaneously rather than forced into one, and its severity tier is the higher of the two categories' independent tier assignments. An anomaly record is not required to belong to exactly one category; it is required to never be silently dropped because a fabric implementation's internal routing could only file it under one.

5.10 The taxonomy is explicitly not closed, and this Part's own governance under Part XX's schema-versioning discipline reflects that: a new signal — a category the fabric's own operational history reveals as common enough to warrant formal detection, surfaced by Part XVI's self-auditing or by a pattern the Crawl Cortex Supervisor's evidence-quality loop repeatedly flags manually before automated detection catches up — is added to this taxonomy through the same amendment process Dataset #01's Part XXIV specifies, never silently, and never by an individual engine deciding on its own that a new check is worth adding without that check's severity tier and detection mechanism being specified with the same rigor as the signals named here.

5.11 A worked example demonstrates the taxonomy's combined operation on a single acquisition. A mission targets a product page on an e-commerce competitor's site, for a Crawl OS mission whose own subject matter is unrelated to commerce — a legitimate research target regardless of the site's own business, since Part IV's contamination boundary in Dataset #01 governs this corpus's own prose content, not what external sites a mission may legitimately study. The page returns a 200 status. The lexical check finds no consent-wall markers. The structural content-to-chrome ratio is within normal bounds — no consent anomaly. The extracted content's length is within the site's baseline for a product page. But the title reads "Session Expired — Please Log In Again" while the URL path names a specific product, a clear title-path structural anomaly, disqualifying tier because it is corroborated by a second signal: the topical-drift check finds the extracted content's dominant topic is authentication rather than the product the mission sought. Two independent signals, one structural and one content-mismatch, corroborate the same underlying failure — an expired session silently redirected to a soft login page rendered with a 200 status — and the object's confidence is forced to the lowest band accordingly, exactly the kind of failure Dataset #01's Part VIII §8.6 named as independently disqualifying regardless of how clean the length signal alone appeared.

PART V FINAL LAW

The anomaly signal taxonomy has four categories — consent and access, content-mismatch, structural, and provenance — each with named detection mechanisms and a three-tier severity scale whose consequence for confidence is fixed and coarse rather than continuously tuned per signal. A signal is recorded with its category, severity, and the specific observed value that triggered it, never as a bare boolean, and the taxonomy grows only through the Part XXIV amendment process, never by silent per-engine addition.


PART VI — CONFIDENCE SCORING INTERNALS: FEEDING DRK-03's CONNECT WITHOUT DUPLICATING IT

6.1 Dataset #01's Part XVII drew the boundary precisely: DRK-03 owns the general confidence and sufficiency logic expressed in its CONNECT vocabulary, and Crawl OS's Reality Gates own the acquisition-specific signals that feed that logic as inputs, never a second, parallel confidence scale. This Part specifies the mechanics of that feeding — what, exactly, the Evidence Integrity Fabric hands to DRK-03 at the SCORED lifecycle state, in what shape, and what it receives back — because Dataset #01's own Part XVII §17.4 stated the composition as an architectural fact without specifying the call's actual interface, and an interface unspecified at this level of detail is not yet buildable.

6.2 The signal payload the fabric constructs for DRK-03 at SCORED is a structured summary of Part V's anomaly taxonomy applied to the specific object being scored: the full list of recorded anomalies with their categories, severities, and observed values, plus a small set of positive signals DRK-03's own CONNECT model requires as counterweight to anomaly presence — confirmation that the acquisition method matched the mission's cost-ladder-recorded expectation, confirmation that all Part VII required fields populated without an aborted interruption, and the raw and normalized hash values themselves, which DRK-03's own dispute-resolution consumers, per Part XIII below, may need to cross-reference independently of the confidence number that results from this call.

6.3 The fabric does not pre-aggregate the anomaly list into any kind of intermediate score before handing it to DRK-03. This is the single most important operational discipline this Part specifies, because a fabric that computed even a simple provisional number — for instance, a naive count of disqualifying-tier anomalies — before calling DRK-03 would have taken the first step toward exactly the parallel confidence scale Dataset #01's Part XVII forbade, regardless of whether that provisional number was ever exposed outside the fabric's own internals. DRK-03's CONNECT logic is the sole owner of the function from signals to a confidence number; the fabric's job stops at supplying honest, complete signals in the taxonomy's own terms.

6.4 The returned confidence value, together with whatever sufficiency classification CONNECT's own vocabulary attaches to it, is written into the Evidence Object's confidence field and, where CONNECT's model records one, an accompanying sufficiency-classification field distinct from Dataset #01's own Part XIX verdict ontology — the two are related but not identical, and this Part's closing provision, 6.10, states precisely how they compose rather than collapse into each other.

6.5 A specific integration hazard deserves explicit treatment because it recurs across every estate subsystem that composes with a shared vocabulary owned elsewhere: DRK-03's CONNECT model may itself evolve — a threshold recalibrated, a new sufficiency tier introduced — on a cadence independent of this dataset's own amendment schedule. The fabric's integration with CONNECT must therefore call DRK-03's current logic live at scoring time, never cache a snapshot of CONNECT's thresholds inside the fabric's own code, because a cached snapshot is, functionally, the parallel scale Dataset #01's Part XVII forbids, merely frozen at whatever moment the cache was taken rather than actively maintained as a second implementation.

6.6 A worked trace makes the interface concrete. An acquisition completes SIGNAL-EXAMINED with one degrading-tier structural anomaly (a redirect chain of length two, within sane bounds but non-empty) and no other anomalies. The fabric constructs the signal payload: one degrading-tier anomaly with its category and observed redirect-chain detail; confirmation the acquisition method matched the router's cost-ladder expectation; confirmation all required fields populated cleanly; the raw and normalized hashes. This payload is handed to DRK-03's CONNECT logic. CONNECT returns a moderate-band confidence value, reflecting a clean acquisition with one minor, expected deviation, plus a sufficiency classification stating the object meets a mission's default evidence bar but would not meet a mission whose contract demanded high confidence specifically. Both values populate the Evidence Object. At no point did the fabric itself decide what "one degrading anomaly, otherwise clean" is worth in confidence terms; that judgment is entirely DRK-03's.

6.7 The composition's failure mode, where it is most likely to occur in an actual implementation under delivery pressure, is not a dramatic reimplementation of CONNECT from scratch but a smaller, more tempting shortcut: a fabric that, observing that most acquisitions with zero anomalies receive a similar high confidence value from DRK-03, begins short-circuiting the call for the zero-anomaly case and simply writing a hardcoded high value directly, on the reasoning that the call is "obviously" going to return that value anyway. This is a duplication of CONNECT's logic in miniature, for exactly the subset of cases where duplication is easiest to justify and therefore easiest to let slip past review, and it breaks the moment CONNECT's own thresholds are recalibrated in a way the fabric's hardcoded shortcut never observes, silently drifting out of sync with the vocabulary it was supposed to be composing with rather than replacing.

6.8 The fabric's relationship to CONNECT's own sufficiency classification, distinct from the raw confidence number, matters specifically for Part II's GATE-EVALUATED and INSTITUTIONALIZATION-ELIGIBLE transitions: Dataset #01's Part XIV's institutionalization criteria reference confidence, but the specific threshold a given mission's contract sets for "sufficient to institutionalize" is itself expressed in CONNECT's sufficiency vocabulary, per Dataset #01's Part XVII §17.6's ninety-percent worked example, and the fabric's own lifecycle logic reads that classification directly from CONNECT's response rather than re-deriving a pass/fail judgment from the raw confidence number using threshold logic the fabric invented itself.

6.9 The signals the fabric supplies to CONNECT are themselves subject to the same fabrication prohibition Dataset #01's Part VII §7.9 established for every other field: a signal payload constructed from a plausible guess about what the anomaly list "probably" contains, rather than the actual recorded anomalies from SIGNAL-EXAMINED, would poison DRK-03's own judgment with fabricated input, and CONNECT's confidence number, however honestly computed on DRK-03's side, would be honest computation over dishonest input — a failure this Part treats with the same severity as fabricating the confidence field directly, because the practical consequence for a downstream reader is identical.

6.10 A closing provision distinguishes the CONNECT sufficiency classification from Dataset #01's Part XIX verdict ontology, since both are seven-or-fewer-term vocabularies attached to an Evidence Object and the two are easy to conflate. CONNECT's sufficiency classification answers a general estate question — is this evidence, in the abstract, strong enough for typical estate purposes — using a vocabulary shared across every domain that cites confidence requirements, not only Crawl OS. Part XIX's verdict ontology answers a Crawl-OS-specific question — what happened to this specific acquisition attempt, using terms like VOID and ESCALATED that have no meaning outside the acquisition context. An object carries both, populated by different mechanisms at different points in its lifecycle, and a future reader conflating them — treating a VERIFIED verdict as though it were a CONNECT sufficiency tier, or vice versa — has misread the object regardless of how plausible the conflation might seem from the two vocabularies' superficial similarity in shape.

PART VI FINAL LAW

The fabric supplies DRK-03's CONNECT logic with honest, complete anomaly signals and never pre-aggregates them into any intermediate score of its own, however small or however confident the fabric might be about the eventual result — that discipline, not merely avoiding a second named confidence scale, is what keeps the composition genuine rather than a duplication in miniature. CONNECT is called live at scoring time, never cached, and its sufficiency classification is read directly rather than re-derived from thresholds the fabric invented itself.


PART VII — THE REDACTION PASS: ENFORCING THE NO-SECRETS DOCTRINE AT THE OBJECT LEVEL

7.1 Dataset #01's Part X established, at the constitutional level, that no raw credential, token, or cookie may ever be persisted in the corpus, and Part I §1.6 of this dataset named the apparent tension that provision creates against this fabric's own highest value: an Evidence Object is supposed to faithfully record what was actually observed during acquisition, and a session that required a credential to succeed genuinely did involve that credential, so a naive reading of "faithful recording" would have the fabric persist exactly what Dataset #01's Part X forbids. This Part resolves that tension mechanically, by specifying the redaction pass every Evidence Object undergoes before any field reaches durable storage.

7.2 The redaction pass runs as a mandatory, non-bypassable step inside FIELDS-POPULATING, positioned after every field has been derived from raw acquisition context but before any field is written to durable storage — never as a separate, later cleanup step that could be skipped under time pressure, and never as a step an engine could disable per-mission regardless of how trusted that mission's own operator believes the storage destination to be. The pass examines every field's populated value, not only the fields where a credential is most likely to appear, because a credential can leak into an unexpected field — a redirect chain that happens to carry a token in a query parameter, an extracted content fragment that happens to quote a page's own visible API-key example — and a redaction pass scoped only to the "obvious" fields would miss exactly the leaks that matter most because they were unanticipated.

7.3 The pass's detection mechanism reuses, deliberately, the same credential-pattern detection this estate already maintains for exactly this purpose elsewhere in Claude Power Pack, rather than the fabric inventing a second, parallel credential detector specific to Crawl OS — an instance of the same non-duplication discipline Part VI applied to DRK-03's CONNECT logic, applied here to the estate's existing secret-detection capability instead. A field value matching a known credential pattern — an API key shape, a bearer token shape, a session cookie's characteristic structure — is redacted; the pass does not attempt semantic judgment about whether a specific matched value is "probably fine to keep," because that judgment is precisely the kind of case-by-case exception Dataset #01's Part X treats as unacceptable regardless of how low-risk a specific instance might appear.

7.4 Redaction replaces the matched value with a marked notice recording that a credential-shaped value was present at this field and was removed, never with a silent empty string that would leave a future reader unable to distinguish "no credential was ever present here" from "a credential was present and stripped." This distinction matters specifically for Part IX's tamper-evidence discipline: a field silently emptied during redaction, indistinguishable from a field that was simply never populated, would defeat any later integrity check attempting to confirm the object matches what construction actually produced, because the check would have no way to know redaction, rather than corruption, explains the field's current value.

7.5 The credential's presence, once redacted, is preserved as a fact even though its value is not: the anomaly-taxonomy signal this triggers — a provenance anomaly, per Part V §5.6's taxonomy, since a credential's presence is a fact about the acquisition's own metadata rather than about the target content — records that authenticated access was used and that a credential was present at a specific field, without recording what that credential's value was. A downstream reader who needs to know whether an acquisition required authentication, for instance to correctly interpret why a public-seeming page nonetheless needed a session, can answer that question fully from the redacted object; a downstream reader who needs the credential's actual value to reproduce the acquisition cannot, and Part XIV's replay discipline is specified around exactly that limitation rather than around an assumption that replay always has access to the original credential.

7.6 The redaction pass's scope extends explicitly to the raw hash discipline Part IV specified, and this extension resolves an apparent conflict between the two Parts worth naming directly: Part IV requires the raw hash to be computed over the exact bytes the fabric received, and a naive reading might conclude that redacting a field before hashing would make the raw hash no longer "raw." The resolution is that the raw hash is computed once, over the true raw bytes, before the redaction pass runs, and is retained as-is; redaction applies to the Evidence Object's own populated fields, which are derived from, not identical to, the raw bytes the hash describes. The raw hash's own value is never itself a field that could leak a credential in readable form — a hash is a fixed-size digest, not the original content — and so it requires no redaction of its own, though the raw bytes it was computed from are never separately persisted anywhere the redaction pass has not also reached, per Part XXI's storage-architecture discipline.

7.7 A specific edge case deserves direct treatment because the founding resource document's own crawling literature names it as a common real-world occurrence: a credential embedded not in a header or cookie but directly in extracted visible page content, such as a support-forum page quoting a leaked API key as part of a user's own troubleshooting post, entirely unrelated to any credential Crawl OS itself used to access the page. The redaction pass applies identically regardless of whether the credential belongs to Crawl OS's own session or to some third party the target content happens to mention, because Dataset #01's Part X's prohibition is stated in terms of what the corpus persists, not in terms of whose credential it is — a leaked third-party key persisted verbatim in an Evidence Object is exactly the kind of harm Part X exists to prevent, independent of whether Crawl OS itself was ever authenticated with it.

7.8 The pass's own correctness is itself subject to Part XVI's self-auditing discipline, specifically because a redaction pass that silently under-detects is a failure mode with no natural downstream signal — nothing else in this dataset's mechanics would notice a credential that slipped through, since every other check examines structural completeness and consistency, not the semantic content of what a field actually contains. Part XVI's self-auditing therefore includes a dedicated, periodic sweep of already-stored Evidence Objects specifically for credential-pattern matches that should have been caught at construction time, treating any positive hit as a Part XVI-severity finding requiring immediate remediation regardless of how old the affected object is.

7.9 A closing provision states the pass's relationship to Dataset #01's Part VII §7.9 fabrication prohibition precisely, because the two provisions could appear to conflict on first reading: redacting a field's value is not fabricating it, because the redaction marker is honest about what happened — it states plainly that a credential-shaped value was present and was removed, rather than presenting a fabricated substitute value as though it were the genuine one. A redaction marker and a fabricated field are opposites in exactly the dimension that matters: one is an honest statement about an intentional, policy-driven omission, and the other is a dishonest claim about an unobserved value.

PART VII FINAL LAW

The redaction pass runs mandatorily inside FIELDS-POPULATING, before any field reaches durable storage, examining every field rather than only the obvious ones, and replacing a matched credential-shaped value with an honest marker recording that removal occurred — never a silent empty field indistinguishable from one that was never populated. A credential's presence is preserved as a fact even when its value is not, and the pass applies identically regardless of whose credential the leaked value happens to be.


PART VIII — CHAIN OF CUSTODY: WHO MAY TOUCH AN EVIDENCE OBJECT, AND WHEN

8.1 An Evidence Object's trustworthiness depends not only on how it was constructed but on what has and has not happened to it since construction, and a fabric that specifies construction discipline in exhaustive detail while leaving post-construction access unspecified has built a strong front door on a building with no other walls. This Part names, precisely, which engines may read an Evidence Object, which may write to one, and under what lifecycle state each kind of access is permitted, closing the gap Parts II through VII left open by focusing on construction alone.

8.2 During RAW-RECEIVED and FIELDS-POPULATING, exactly one actor has write access to the object still being assembled: the Evidence Integrity Fabric's own construction logic. No other engine — not the router, not the extraction engine, not the Crawl Cortex Supervisor — may write to an object in either of these two states, and this exclusivity is what makes Part II's interruption-and-resume discipline coherent: if a second actor could write to a FIELDS-POPULATING object, the fabric's own checkpoint-and-resume logic could not distinguish its own resumed writes from a second actor's concurrent writes, and Part IX's tamper-evidence mechanics, specified around a single, identifiable writer, would have no way to attribute an unexpected mid-construction change to its actual source.

8.3 From SIGNAL-EXAMINED onward, through SCORED and GATE-EVALUATED, write access narrows further rather than widening: the fabric itself may still append derived fields — the confidence score at SCORED, gate results at GATE-EVALUATED — but no engine, including the fabric, may modify a field populated during an earlier state once that state has completed. A confidence score computed at SCORED does not get silently recomputed if a later engine disagrees with it; disagreement is handled through Part XIII's dispute-resolution mechanism, which produces a new, explicitly linked record rather than mutating the original. This append-only discipline from SIGNAL-EXAMINED onward is the chain-of-custody analogue to Part II's linear, no-skip lifecycle: just as a state may not be skipped, a completed state's fields may not be revised in place.

8.4 Read access, once an object reaches INSTITUTIONALIZATION-ELIGIBLE or RETAINED-NON-INSTITUTIONAL, is broader by design and governed by a different principle than write access: any Crawl OS engine, and per Part XXII any properly authenticated external estate consumer, may read a sealed Evidence Object's fields, because the entire purpose of constructing provenance-carrying evidence, per Dataset #01's Part I, is for it to be readable and trustworthy by exactly the widest audience that might need it, not narrowly gated to whichever engine happened to construct it. The asymmetry — narrow write access during construction, broad read access after sealing — mirrors the asymmetry Dataset #01's Part XX drew between its acquisition and evidence-judgment planes: the planes that produce a judgment are narrow and separated; the record of the judgment, once produced, is meant to be widely consulted.

8.5 A CONSTRUCTION-ABORTED record, per Part II §2.8, follows a narrower custody rule than a completed object: because it was never sealed through GATE-EVALUATED, it carries no guarantee that its partial fields are internally consistent, and read access to it is restricted to the fabric's own diagnostic tooling and Part XVI's self-auditing sweep — an aborted record exists to prevent silent re-attempt loops, not to serve as evidence any downstream engine should treat as informative about the target source, and broad read access to a fundamentally incomplete record would risk exactly the confusion Dataset #01's Part I named as its founding failure, applied here to the fabric's own internal bookkeeping rather than to external content.

8.6 The chain-of-custody discipline extends to a case not yet addressed elsewhere in this dataset: an Evidence Object's redacted fields, per Part VII, must never be un-redacted by any subsequent access, including privileged diagnostic access. There is no engine, role, or emergency procedure within this dataset's scope that restores a redacted value from the sealed object itself — Dataset #01's Part X and this dataset's Part VII treat redaction as a one-way transformation applied before durable storage, and a chain-of-custody rule permitting any actor to reverse it, however privileged, would undermine the entire mechanism by making its guarantee conditional on trusting every future holder of that privileged access rather than unconditional.

8.7 A worked scenario illustrates why the append-only discipline of 8.3 matters beyond the abstract principle. Suppose the Crawl Cortex Supervisor's evidence-quality-assurance loop, reviewing a batch of sealed objects, discovers that Part VI's DRK-03 call for a specific object appears to have used a stale, cached CONNECT threshold due to a since-fixed bug in the integration Part VI §6.5 warned against. The correct remediation is not to reach into the sealed object and silently overwrite its confidence field with a recomputed value — doing so would erase the historical fact that the object was, at the time it was sealed, scored under a defective integration, information a future audit of that defect's actual impact would need. The correct remediation is Part XIII's dispute-resolution mechanism: a new record is created, explicitly referencing the original object, carrying a corrected confidence value, and the original object's own historical record remains intact and queryable exactly as it was sealed.

8.8 This Part's custody rules apply identically regardless of which specific implementation of the fabric is running them — a future scaling of Crawl OS across multiple deployment instances, per whatever operational dataset eventually elaborates Dataset #01's Part XX deployment-plane separation in infrastructure terms, does not relax the single-writer-during-construction rule into "single writer per instance," because two instances writing to what they each believe is an exclusively-owned FIELDS-POPULATING object for the same acquisition would reproduce exactly the concurrency hazard Part III §3.6 already worked through for provenance-chain assembly, applied here to the object's custody as a whole rather than to one field within it.

8.9 A final provision names the custody discipline's relationship to Part XVIII's GK-04 composition directly: the typed edge GK-04's future writer constructs from an institutionalized object is a read-only derivative, constructed from data the fabric hands over per Part XVI §16.4's interface, and constructing that edge is not itself a write to the Evidence Object — the object's own custody rules are untouched by anything GK-04's graph subsequently does with the edge content it received. GK-04 owns what happens to the edge after it is written; it never gains write access to the Evidence Object that informed it.

8.10 A final distinction closes this Part between custody, as specified above, and authorization, governed entirely by Dataset #01's Part V §5.13 Governance, Safety, and Authorization engine. Custody answers which engine may touch an Evidence Object's storage representation and in what lifecycle state; authorization answers whether an acquisition was permitted to occur at all, a question settled before the object's construction even begins. An engine holding valid custody-level read access to a sealed object is not thereby granted any opinion on whether the underlying acquisition was authorized — that judgment belongs entirely upstream, and a custody rule that quietly absorbed authorization questions would have reached into Governance's own territory exactly as Dataset #01's Part IV forbids any engine from reaching into a boundary it does not own. The two disciplines compose without overlapping: Governance decides whether an acquisition may happen; custody decides who may subsequently touch the record of what happened.

PART VIII FINAL LAW

Write access to an Evidence Object narrows through its lifecycle rather than widening: exclusive to the fabric's own construction logic during RAW-RECEIVED and FIELDS-POPULATING, append-only for derived fields from SIGNAL-EXAMINED onward, and closed entirely once sealed — a completed field is never revised in place, and a redacted value is never restored by any actor under any circumstance. Read access, by contrast, widens once an object is sealed, because a record built to be trustworthy is a record meant to be consulted, not narrowly held by whichever engine constructed it.


PART IX — TAMPER-EVIDENCE AND INTEGRITY VERIFICATION

9.1 Part IV distinguished the raw hash's fixed reference value from the separate, later discipline of re-verifying stored content against it, and named that discipline tamper-evidence without specifying its mechanics. This Part specifies those mechanics: when re-verification runs, what it compares, what it does when a mismatch is found, and how the fabric distinguishes a genuine tamper event from the honest, expected changes — redaction, schema migration — that also alter a stored object's bytes without constituting tampering in any sense this Part treats as a violation.

9.2 Re-verification runs at two distinct triggers, not continuously and not only on demand, because a continuous check would waste resources the cost-ladder discipline Dataset #01's Part VI already treats as scarce, while a purely on-demand check would leave long-dormant objects unverified for arbitrarily long stretches. The first trigger is query-time: any read access under Part VIII's custody rules re-computes a hash over the object's currently stored raw-field content and compares it against the stored raw hash before returning the object to the requesting engine, catching a tamper event at the moment it would actually matter to a consumer. The second trigger is Part XVI's periodic self-auditing sweep, which re-verifies every object in evidence memory on a fixed cadence regardless of whether any query happened to touch it, catching a tamper event on an object nobody has queried recently, before it silently accumulates alongside others.

9.3 A hash mismatch at query-time re-verification does not immediately declare the object corrupted and refuse to return it; it first checks whether the object's own change-history or redaction metadata records an event that would legitimately explain a stored-content change since the raw hash was computed — a schema migration under Part XX, for instance, may legitimately transform stored field representations in ways the original raw hash predates. If a legitimate, recorded event explains the discrepancy, the object is returned with a note that its stored representation has been migrated since original construction, and no tamper flag is raised. If no such recorded event explains the mismatch, the object is flagged TAMPER-SUSPECTED, is not returned to the requesting engine in its current form, and is escalated to Part XVI's self-auditing discipline for investigation, exactly as any other integrity anomaly would be.

9.4 The distinction between a legitimately explained mismatch and a tamper-suspected one depends entirely on Part XX's schema-versioning discipline recording every migration event durably and queryably, because the re-verification check in 9.3 has no other source of truth to consult. A migration performed without a corresponding recorded event would be indistinguishable, from re-verification's perspective, from actual tampering, and would either falsely flag every migrated object as tamper-suspected — a false-positive storm that would train operators to distrust or ignore the tamper-evidence mechanism entirely — or, worse, would require re-verification to be weakened to tolerate unexplained mismatches, which would defeat the mechanism's entire purpose. This is why Part XX treats every migration as a mandatory recorded event rather than an optional courtesy.

9.5 A TAMPER-SUSPECTED flag is not, by itself, proof that malicious alteration occurred; it is proof that the object's currently stored content does not match what construction originally produced, for a reason the fabric's own recorded history cannot explain. The distinction matters because the actual cause could be a storage-layer bit-rot event, a backup-restore that reverted an object to a stale copy without regenerating its metadata, or a genuine unauthorized modification, and Part XVI's investigation, not the re-verification check itself, is responsible for narrowing among these possibilities. The re-verification check's job stops at detection; attribution is a separate, deliberately slower process that a fast, frequent check like query-time re-verification cannot and should not attempt to perform inline.

9.6 The normalized hash participates in tamper-evidence differently than the raw hash, and this Part states the distinction explicitly because Part IV's own worked example already demonstrated why the two hashes diverge under entirely legitimate conditions — a re-acquisition producing a different raw hash but an identical normalized hash. Tamper-evidence's re-verification check operates on the raw hash specifically, because the raw hash's fixed input boundary — the exact bytes received at RAW-RECEIVED — is what tampering, as this Part defines it, actually alters; the normalized hash's role belongs to Part X's change-history mechanics, detecting legitimate content evolution across re-acquisitions, not detecting alteration of a single, already-sealed object's stored representation.

9.7 A worked scenario distinguishes a genuine tamper event from the two legitimate-mismatch cases 9.3 and 9.6 already named. An Evidence Object, sealed six months ago, is queried by a mission researching the same source again. Re-verification computes the current stored raw-field hash and finds it does not match the stored raw hash. The object's change-history and schema-version metadata record no migration event in the intervening six months. The mismatch is therefore TAMPER-SUSPECTED. Part XVI's investigation examines the storage layer's own access logs, per Part XXI's storage architecture, and finds a single, isolated write to that object's storage location outside any recognized fabric process — not a redaction, not a migration, not a re-acquisition, an unexplained write. This is the concrete case tamper-evidence exists to surface: a change to sealed, supposedly immutable evidence that no legitimate mechanism in this dataset produced, discovered because the mechanism was checked, not because anyone happened to notice.

9.8 The consequence of a confirmed tamper finding, once Part XVI's investigation completes and cannot attribute the change to a legitimate cause, is that the affected object is marked TAMPER-CONFIRMED and is excluded from every downstream consumption path — it is not returned by Part XXII's query interface, it is not eligible for Part XI's package composition even if it was previously institutionalized, and any GK-04 edge already constructed from it, per Part XVIII, is flagged for the graph's own review rather than silently left pointing at now-untrusted evidence. A TAMPER-CONFIRMED object is retained, never deleted, precisely because the record of the tampering itself — when it was detected, what the mismatch was, what investigation found — is itself valuable institutional knowledge about a security or integrity incident, exactly the kind of negative-memory value Part XIX elaborates for failed acquisitions applied here to a failed custody event instead.

9.9 A necessary limitation of this Part's mechanism must be stated honestly rather than implied away: hash-based tamper-evidence detects that stored content has changed since a reference point; it cannot, by itself, detect an alteration that occurred before the raw hash was ever computed — a compromised upstream engine that handed the fabric already-corrupted raw bytes at RAW-RECEIVED would produce a raw hash that faithfully, correctly describes those corrupted bytes, and re-verification would find no mismatch at all, because nothing about the object's storage changed after construction. Guarding against that earlier-stage risk is Part V's anomaly taxonomy and Gate-level content verification's territory, not this Part's; tamper-evidence's honest scope is post-construction integrity, not pre-construction trustworthiness, and conflating the two would overstate what a hash comparison can actually prove.

9.10 This Part's relationship to Part VIII's custody rules closes the chapter this pair of Parts opened together: custody specifies who may act on an object and when; tamper-evidence specifies how an unauthorized or unexplained action, should custody's own preventive controls ever fail to stop one, is detected after the fact rather than never discovered at all. The two are complementary layers, not redundant ones — custody is prevention, tamper-evidence is detection — and a fabric relying on custody alone, without the independent detection layer this Part specifies, would have no way to discover a custody failure it did not anticipate in advance, which is precisely the kind of unanticipated failure any real access-control system must assume is eventually possible.

PART IX FINAL LAW

Tamper-evidence re-verifies a sealed object's raw hash at query-time and on a periodic sweep, distinguishing a legitimately recorded change — a redaction, a schema migration — from an unexplained mismatch, which is flagged TAMPER-SUSPECTED and escalated to investigation rather than either silently accepted or immediately assumed malicious. A confirmed tamper finding excludes the object from every downstream consumption path while retaining it as institutional knowledge about the incident itself, and the mechanism's honest limitation is that it detects post-construction alteration only, never a corruption already present in the raw bytes at the moment of construction.


PART X — THE CHANGE HISTORY FIELD: RE-ACQUISITION WITHOUT LOSING THE PRIOR RECORD

10.1 Dataset #01's Part VII §7.2 listed a change history among the Evidence Object's required fields without specifying its mechanics, and Part III §3.3 of this dataset specified how a new acquisition discovers a prior object to supersede, without specifying what happens to that prior object once superseded. This Part closes that gap: what the change-history field actually records, how a chain of successive acquisitions of the same source is represented, and the precise rule — already previewed but not yet stated as binding — that a superseded object is never deleted.

10.2 A change-history entry is created at the moment Part III's supersession check finds a prior match, and it is attached to the new, superseding object rather than retroactively inserted into the prior, now-superseded one — the prior object, per Part VIII's append-only custody discipline, cannot be modified after sealing, and a change-history mechanism requiring retroactive edits to a sealed object would violate that discipline on its very first use. The new object's change-history entry records the superseded object's identifier, the normalized-hash comparison result from Part IV §4.6 that determined whether the change is substantive, and the timestamp of the new acquisition — everything a downstream reader needs to walk the chain backward without needing to modify anything already sealed.

10.3 A chain of successive acquisitions of the same source over time forms a linked, one-directional structure: each object's change-history entry points to exactly the one object it immediately supersedes, never to an arbitrary earlier point in the chain, and never to more than one prior object. A downstream reader wanting the full history of a source across every acquisition walks the chain backward one link at a time, from the current object to its immediate predecessor, to that predecessor's own predecessor, and so on — a mechanism deliberately simple enough that its correctness does not depend on any single object's change-history entry containing more than the one fact it is responsible for.

10.4 The substantive-versus-cosmetic distinction Part IV §4.6 established using the normalized hash comparison is recorded explicitly in the change-history entry, not merely implied by the entry's existence, because a downstream reader — particularly the Knowledge Distillation Fabric deciding whether a re-acquisition warrants re-institutionalizing a claim already built on the prior object — needs to know which kind of change occurred without independently re-deriving the comparison. A change-history entry recording "supersedes object X, substantive change" triggers different downstream handling than one recording "supersedes object X, cosmetic re-acquisition only, content unchanged" — the former may warrant new institutionalization per Dataset #01's Part XIV; the latter typically does not, because the knowledge it would produce already exists.

10.5 A re-acquisition that finds its predecessor's content unchanged does not skip Evidence Object construction entirely, even though no new knowledge results from it — the new object is still constructed in full, through the same lifecycle Part II specifies, because the re-acquisition itself is a fact worth recording: it establishes that the source was checked on this date and found unchanged, which is precisely the kind of freshness evidence Part XII's retention and freshness-window mechanics depend on. A fabric that short-circuited construction for a cosmetic-only re-acquisition, reasoning that nothing new was learned, would lose the ability to answer "when was this source last confirmed current" with anything better than the date of its last substantive change, which is a materially weaker guarantee than the date of its last confirmed check.

10.6 The rule that a superseded object is never deleted is stated here as binding, not merely as an inference from Part VIII's append-only discipline, because the temptation to delete superseded objects — on the reasoning that the current object already represents the source's latest known state, and older objects are simply outdated clutter — recurs often enough in general data-management practice that this Part names it directly and rejects it. A superseded object remains queryable through Part XXII's interface for exactly the same reason Dataset #01's Part VII treats a provenance chain as permanent: a downstream claim built on the superseded object's content, at the time it was current, remains true about what was known at that time, and deleting the object that supported that claim would strand the claim's own provenance, breaking Dataset #01's Part III §3.10 obligation retroactively for every claim ever built on now-superseded evidence.

10.7 A specific interaction between change history and Part XIV's institutionalization criteria deserves direct statement: an object superseded by a substantive change does not automatically lose its own institutionalized status if it had already earned one — Dataset #01's Part XIV never authorized retroactive de-institutionalization on supersession alone, and the correct handling instead marks the superseded object's institutionalized knowledge as historical, distinct from current, while the superseding object's own institutionalization eligibility is evaluated independently on its own merits through the same lifecycle every other object passes through. A source's documentation being current as of March and superseded by an April re-acquisition does not make the March knowledge false; it makes it dated, and dated is a different, weaker claim than false, one this mechanism is specifically built to preserve rather than collapse into a binary current-or-discarded state.

10.8 The change-history chain's length is unbounded by this Part's own design, and a fabric implementation is not permitted to truncate an old chain by discarding its earliest links to save storage, because doing so would silently convert 10.6's never-delete rule into a delete-eventually rule under a different name. Storage-cost management for long chains is Part XXI's own architectural concern — cold storage tiers, compression, whatever mechanism that Part specifies — and is explicitly not resolved by shortening the chain's logical length, which would be a correctness compromise disguised as a storage optimization.

10.9 A worked trace demonstrates the full mechanism across three acquisitions of one source. January: a documentation page is acquired, constructed, sealed, with an empty supersedes field — no prior object exists. April: the same page is re-acquired; Part III's supersession check finds the January object; Part IV's normalized-hash comparison finds a match — no substantive change; the April object is constructed in full, its change-history entry records "supersedes January, cosmetic only." July: the page is re-acquired again; supersession finds the April object as the immediate predecessor, not the January one; the normalized hash now differs — a substantive change; the July object's change-history entry records "supersedes April, substantive." A reader querying the July object today, per Part XXII, can walk backward — July supersedes April, April supersedes January — and reconstruct the complete history of this source's acquisition, including the fact that the source was confirmed unchanged in April before actually changing by July, a fact that would be lost entirely under any mechanism that skipped construction for the cosmetic-only case.

10.10 This Part's closing provision states its relationship to Part XV's cross-mission provenance directly, because the two mechanisms operate on adjacent but distinct questions that a careless implementation could conflate: change history tracks a single source's evolution across time, regardless of which mission performed each acquisition; cross-mission provenance, elaborated fully in the next Part, tracks whether two different missions, acquiring the same source independently, should be treated as producing the same evidence rather than two unrelated objects. A re-acquisition by the same mission that originally acquired a source is a change-history event; an independent acquisition by a different mission of a source another mission has already acquired is, additionally, a cross-mission provenance event, and both mechanisms may fire for the same pair of objects when the two conditions coincide.

PART X FINAL LAW

A change-history entry is attached to the new, superseding object, never inserted retroactively into the sealed prior one, and records the predecessor's identifier, the substantive-versus-cosmetic determination, and the acquisition timestamp — forming a one-directional chain a downstream reader walks backward without needing any object in the chain to be modified. A superseded object is never deleted; it remains queryable, its institutionalized knowledge marked historical rather than false, because a claim built on it when it was current remains a true claim about what was known at that time.


PART XI — EVIDENCE PACKAGE COMPOSITION FOR THE KNOWLEDGE DISTILLATION FABRIC

11.1 Dataset #01's Part V §5.10 chartered the Knowledge Distillation Fabric to consume verified Evidence Packages, a term Dataset #01 used without specifying what distinguishes a package from a bare collection of Evidence Objects, or how the fabric this dataset elaborates actually constructs one. This Part specifies that construction, occurring at the moment an object or a related set of objects reaches INSTITUTIONALIZATION-ELIGIBLE per Part II §2.7, as the final operational step this dataset owns before responsibility passes to the downstream engine Dataset #01 already chartered.

11.2 An Evidence Package is not a new schema layered on top of the Evidence Object; it is a bounded, purpose-built bundle of one or more already-sealed Evidence Objects, plus a package-level manifest recording which mission requested the package, which specific claim or research question the bundled objects jointly support, and, for a multi-object package, how the objects relate to each other — corroborating the same fact independently, or covering complementary aspects of a broader claim no single object could support alone. The manifest is new content this Part specifies; the objects themselves carry no new fields beyond what Dataset #01's Part VII and this dataset's earlier Parts already established.

11.3 The single-object package is the common case and requires the least additional construction: one Evidence Object, having independently reached INSTITUTIONALIZATION-ELIGIBLE, is wrapped in a manifest naming the mission and the specific claim it supports, and handed to the Knowledge Distillation Fabric as a complete package. This is the shape the worked example in Part II §2.10 already traced through to its INSTITUTIONALIZATION-ELIGIBLE outcome, and this Part's contribution to that trace is the final step: the manifest wrapping that object before hand-off.

11.4 A multi-object package arises when a mission's evidence requirement, per Dataset #01's Part III, cannot be satisfied by any single acquisition — a claim about a project's current API surface that requires corroboration across the documentation page, the changelog, and the source repository's own README, each independently acquired, independently constructed as its own Evidence Object, independently reaching INSTITUTIONALIZATION-ELIGIBLE. The package construction step recognizes when a mission's frontier, per its own contract, named multiple items as jointly required for one claim, and bundles the resulting objects into a single package once all of them, not merely some, have independently reached eligibility — a package is never assembled from a mix of eligible and still-pending objects, because handing the Knowledge Distillation Fabric a partial bundle would misrepresent what evidence actually supports the claim it is about to distill.

11.5 The manifest's relationship-recording obligation, named in 11.2, is not decorative: a package bundling three objects that independently corroborate the same fact carries a materially different evidentiary weight than a package bundling three objects that each cover a different, non-overlapping aspect of a broader claim, and the Knowledge Distillation Fabric's own downstream confidence in the resulting knowledge artifact depends on knowing which situation it is looking at. The manifest states this explicitly — corroborating or complementary, per object pair — rather than leaving the Knowledge Distillation Fabric to infer the relationship from the objects' own content, an inference this Part's construction step is better positioned to make correctly, since it has direct access to the mission's own frontier structure that established why these particular objects were bundled together in the first place.

11.6 Package construction never institutionalizes anything by itself; institutionalization, per Dataset #01's Part XIV, was already decided independently for each bundled object before packaging began. This ordering — objects earn institutionalization eligibility individually, packaging happens only after, packaging itself makes no additional eligibility judgment — prevents a specific failure mode: a package that bundled a strong object with a weak one, and used the strong object's eligibility to carry the weak one past a bar it had not independently earned, would have let packaging quietly substitute for the gate discipline Part II's lifecycle exists to enforce object by object.

11.7 A package, once constructed and handed to the Knowledge Distillation Fabric, is retained in evidence memory as its own addressable record distinct from the individual objects it bundles, so that a future audit of a specific knowledge artifact the Distillation Fabric produced can trace back to the exact package — and, through it, the exact set of Evidence Objects — that informed it, without needing to reconstruct which objects happened to be eligible at the moment packaging occurred. This retained package record is itself immutable once constructed, following the same append-only-then-sealed discipline Part VIII established for individual objects.

11.8 The package's relationship to Part XVIII's GK-04 edge construction runs in parallel rather than in sequence: a package's construction and a GK-04 edge's construction are two independent consumers of the same institutionalization-eligible state, each triggered by it, neither depending on the other completing first. A package can be handed to the Knowledge Distillation Fabric before, after, or concurrently with a GK-04 edge being constructed for the same object, and no ordering guarantee between the two is either provided or required, because the two consumers serve entirely different downstream purposes — one produces reusable prose knowledge, the other produces a graph-queryable pointer — and coupling their timing would introduce a dependency this dataset's own composition boundaries, per Part I §1.7, explicitly forbid Crawl OS from imposing on either side.

11.9 A worked example distinguishes the corroborating and complementary relationship types concretely. A mission investigates whether an open-source project's license changed in the past year. Two objects are acquired: the project's current LICENSE file, and a changelog entry announcing a licensing update — these are complementary, not corroborating, because neither alone answers the mission's question; the LICENSE file shows the current state, and the changelog entry shows that a change occurred and roughly when, and only together do they support the claim "this project's license changed, and here is what it changed to." A separate mission, verifying the same license claim through two independent means — fetching the LICENSE file directly and also fetching a third-party package registry's own cached copy of the same file — produces two corroborating objects, because both independently claim to show the same current state, and their agreement, rather than their complementarity, is what strengthens the resulting claim.

11.10 This Part's own scope closes deliberately short of the Knowledge Distillation Fabric's own responsibility, restated one final time because the boundary is easy to blur at exactly this hand-off point: this dataset's obligation ends at handing over a well-formed, honestly manifested package of already-institutionalization-eligible objects; what the Distillation Fabric does with that package — how it actually produces compact, cited, reusable prose from the bundled evidence — belongs entirely to the engine Dataset #01's Part V §5.10 chartered for that purpose, and a future dataset elaborating that engine inherits this Part's package format as its own upstream input, exactly as this dataset inherited Dataset #01's Part VII as its own.

PART XI FINAL LAW

An Evidence Package bundles one or more already-institutionalization-eligible Evidence Objects with a manifest naming the requesting mission, the claim supported, and, for multi-object packages, whether the bundled objects corroborate or complement each other — a distinction the Knowledge Distillation Fabric needs and cannot reliably infer from the objects alone. Packaging never itself grants eligibility; it bundles what individual objects already earned, and a package's construction runs independently of, never blocking or blocked by, Part XVIII's GK-04 edge construction for the same underlying evidence.


PART XII — RETENTION, EXPIRY, AND THE FRESHNESS WINDOW

12.1 Dataset #01's Part VI §6.2 named the freshness window as a property of a mission's own contract, governing whether a cached prior acquisition satisfies the cheapest rung of the cost ladder without a new acquisition being necessary at all. This Part specifies the fabric-side mechanics that make a freshness-window check answerable: how an object's own age is determined, how retention differs from freshness, and what happens to an object once it ages past every freshness window any active or plausible future mission might apply to it.

12.2 Freshness and retention are distinct questions this Part deliberately does not conflate, though a careless reading of "how long does an object live" could easily merge them. Freshness asks whether a specific object, given its acquisition timestamp, is current enough to satisfy a specific mission's specific freshness requirement — a question answered fresh, per mission, at query time. Retention asks whether an object continues to exist in evidence memory at all, independent of whether any given mission would currently consider it fresh — a question answered once, by this Part's own retention policy, independent of any particular mission's requirements. An object can be simultaneously stale, by every plausible mission's freshness standard, and still retained, because Part X §10.6 already established that a superseded or dated object is never deleted merely for having aged.

12.3 The freshness check itself is mechanically simple, given the acquisition timestamp Part VII §7.2 of Dataset #01 already requires every object to carry: a mission's contract states a maximum acceptable age, and an object satisfies that mission's freshness requirement if and only if the current time minus the object's acquisition timestamp is less than or equal to that stated maximum. This Part's contribution beyond the arithmetic itself is specifying what "current time" means for an object with a change-history chain: freshness is always evaluated against the specific object being considered, using that object's own acquisition timestamp, never against the chain's earliest or latest member by default, because a mission querying evidence memory per Part XXII may legitimately want to know the freshness of a specific historical object in the chain, not only the current one.

12.4 Retention policy, by contrast, is governed by a small number of named categories rather than a single global rule, because different classes of object carry different institutional value independent of age. Institutionalized objects — those that reached INSTITUTIONALIZATION-ELIGIBLE and were packaged per Part XI — are retained indefinitely by default, since Dataset #01's Part XIV already established that institutionalization is a judgment about durable, reusable value, and a durable value judgment that expired on a timer would contradict its own premise. RETAINED-NON-INSTITUTIONAL objects, per Part II §2.7, are retained for a bounded period tied to the mission that produced them — long enough to remain queryable for the lifetime of that mission and any reasonably foreseeable follow-up, but not indefinitely, since an object that never earned institutional value and belongs to a long-closed mission carries diminishing justification for consuming storage indefinitely. CONSTRUCTION-ABORTED records, per Part II §2.8, are retained for the shortest bounded period of the three categories, sufficient only to serve their own stated purpose of preventing silent re-attempt loops.

12.5 The bounded retention periods named in 12.4 for the two non-institutionalized categories are not this Part's content to fix as specific numbers, for the same reason Part IV §4.3 declined to enumerate specific normalization steps: the actual figures are an implementation-tuning decision a future contributor building `modules/crawl_os/evidence.py` will calibrate against real storage economics and real query patterns, composing with CO-01's operating-economics territory per Dataset #01's Part XV rather than this dataset setting a cost-driven number itself, which would reach into CO-01's boundary exactly as Dataset #01's Part XV §15.11 already forbade for any Crawl OS dataset. What this Part fixes as binding is the category structure and the ordering of retention generosity — institutionalized longest, non-institutionalized-but-well-formed next, aborted shortest — not the specific durations within each category.

12.6 An object approaching the end of its retention period is not silently deleted without signal: Part XVI's self-auditing discipline surfaces a retention-expiry list on a periodic cadence, giving the Crawl Cortex Supervisor's institutional-learning loop, per Dataset #01's Part V §5.12, an opportunity to flag any object on that list whose retention should be extended — for instance, because a currently-active mission's own frontier still references it, even though the object itself never independently earned institutionalization. This review step exists specifically to prevent the mechanical retention-category rule from deleting an object that remains genuinely load-bearing for reasons the category rule alone could not know about.

12.7 Actual deletion, when it occurs after the review step in 12.6 finds no reason to extend retention, is itself a recorded event, not a silent disappearance: a tombstone record — the object's identifier, its category, its acquisition timestamp, and the reason its retention period lapsed — persists after the object's full content is removed, so that a future query for the deleted object's identifier, perhaps arriving through a stale reference in some other system, returns an honest "this object existed and was retired on this date for this reason" rather than a bare not-found response indistinguishable from an identifier that was never valid at all.

12.8 The freshness window's interaction with Part X's change-history chain deserves one clarifying sentence: a mission checking whether cached evidence satisfies its freshness requirement, per Dataset #01's Part VI's first cost-ladder rung, checks the most recent object in a source's change-history chain, not an arbitrary earlier link, because "is there fresh cached evidence" is inherently a question about the current state of knowledge, and only the chain's most recent member represents that current state — an earlier link's own freshness, evaluated per 12.3 against its own timestamp, answers a different, historical question that the cost-ladder's first-rung check does not ask.

12.9 A worked example ties retention and freshness together across a realistic timeline. An object institutionalizes in January. A mission in June checks whether it satisfies a thirty-day freshness requirement: five months have elapsed, the check fails, and the mission's contract requires fresh evidence, triggering a new acquisition per the cost ladder rather than reusing the stale one. That new June acquisition supersedes the January object per Part X's mechanics. The January object, despite failing every subsequent freshness check for any thirty-day-window mission from June onward, is retained indefinitely under 12.4's institutionalized-object rule, remains queryable, and its own historical institutionalized knowledge is marked dated rather than deleted, exactly as Part X §10.7 already established — freshness failure and retention expiry are entirely independent outcomes, and this example's object experiences the first without ever approaching the second.

12.10 This Part's closing distinction restates 12.2's opening one in operational terms, because the conflation it warns against is exactly the kind of error a fabric implementation under storage-cost pressure would be tempted to make: a proposal to delete objects once they fail freshness for every currently active mission's requirements has confused a freshness question, which is mission-relative and re-evaluated at every query, with a retention question, which is category-relative and evaluated once against this Part's own fixed policy. An object may be simultaneously stale for every mission alive today and correctly, deliberately retained, and a retention policy that collapsed the two questions into one would silently discard exactly the historical evidence Part X's change-history chain exists to preserve.

PART XII FINAL LAW

Freshness is a per-mission, per-query judgment about whether a specific object's age satisfies that mission's stated requirement; retention is a category-based, once-decided policy about whether an object continues to exist in evidence memory at all — the two are independent, and an object may be simultaneously stale by every active mission's standard and correctly retained under this Part's own policy. Deletion, when a retention period genuinely lapses after review, is never silent; a tombstone record survives to answer any future query honestly.


PART XIII — DISPUTE RESOLUTION: WHEN A DOWNSTREAM CONSUMER CHALLENGES AN EVIDENCE OBJECT

13.1 Part VIII §8.7 introduced dispute resolution as the correct remediation when a sealed object's confidence score is later discovered to have been computed under a defective integration, and named its shape only in outline: a new record, explicitly linked, rather than a mutation of the original. This Part specifies that mechanism fully, extending it beyond the single scoring-defect case to the general question of what happens when any downstream consumer — an engine, a mission, a human reviewer under Dataset #01's Part VI seventh rung — believes a sealed Evidence Object's recorded judgment is wrong.

13.2 A dispute is opened against a specific, sealed Evidence Object by a specific consumer, and it names a specific claim being challenged — a confidence score believed miscalibrated, an anomaly signal believed spurious, a gate result believed incorrectly evaluated, a provenance chain believed to reference the wrong mission. A dispute may not be opened against the object's existence or its raw and normalized hashes, because those fields describe what was mechanically observed or computed and are not the kind of judgment a dispute contests; a disagreement about whether a hash was computed correctly is a Part IX tamper-evidence or Part XVI self-auditing matter, not a dispute in this Part's sense.

13.3 Opening a dispute never modifies the original object, per Part VIII's append-only-then-sealed custody discipline restated once more here as the mechanism's foundational constraint. Instead, a dispute record is created, referencing the original object's identifier, the specific field or judgment being contested, the challenger's stated reasoning, and, where available, new evidence the challenger offers in support of the contest — for instance, a subsequent acquisition of the same source that produced a materially different result under conditions the challenger believes invalidate the original.

13.4 Dispute resolution itself is not performed by the challenger, nor by the original object's own constructing fabric instance acting alone, but by re-running the specific judgment being contested through the same mechanism that originally produced it, using whatever new evidence the dispute record supplies, and comparing the result against the original. If the confidence score is disputed, Part VI's DRK-03 integration is re-invoked with the original signal payload plus any new signals the dispute supplies; if a gate result is disputed, the specific Reality Gate in question, per Dataset #01's Part XIII, is re-evaluated. The resolution mechanism is never a fresh, independent judgment invented for dispute-handling specifically — reusing the same mechanism that produced the original judgment is what makes the comparison between original and resolved meaningful.

13.5 Three outcomes are possible once resolution completes, and each produces a distinct, honestly labeled result. UPHELD: the re-run judgment matches the original within an acceptable tolerance, and the dispute record closes noting the original stands, valuable specifically because it documents that a specific concern was raised and formally checked, even though nothing changed. REVISED: the re-run judgment differs materially from the original, and a new Evidence Object or amended field record is created — never overwriting the original — carrying the corrected judgment and explicitly linked, in both directions, to the disputed original and the dispute record that produced the correction. INCONCLUSIVE: the available evidence, including whatever the dispute supplied, is insufficient to confidently resolve the disagreement either way, and the dispute record closes noting this honestly rather than forcing a false confidence in either UPHELD or REVISED.

13.6 The REVISED outcome's relationship to Part X's change-history mechanics deserves explicit distinction, because both produce a new record linked to a prior one and the two could be conflated without this Part's clarification. Change history records a legitimate re-acquisition of the same source over time — the world changed, or the fabric simply checked again. A dispute-driven revision records a correction to a judgment about the same underlying acquisition — the world did not change, but the original judgment about it is now understood to have been wrong. The revised record's raw and normalized hashes remain identical to the original's, since no new acquisition occurred; only the disputed field's own judgment differs, which is precisely how a downstream reader distinguishes a Part X supersession from a Part XIII revision without needing separate documentation to explain which mechanism produced a given linked record.

13.7 A dispute's own legitimacy is itself subject to review before resolution proceeds, because an unbounded right to dispute any judgment, re-triggering the underlying scoring or gate mechanism on demand, would create a resource-consumption vector Dataset #01's Part VI cost-ladder discipline would treat as exactly the kind of unaccounted cost that discipline exists to prevent. A dispute must state specific, falsifiable reasoning — new evidence, a named defect in the original mechanism's execution, a specific inconsistency in the original record — and a dispute stating only general dissatisfaction with a low confidence score, absent any specific, checkable reason, is rejected before resolution consumes any further resources, with the rejection itself recorded so a pattern of frivolous disputes from a given source is visible to Part XVI's self-auditing.

13.8 The party authorized to open a dispute is deliberately broad rather than narrow: any Crawl OS engine that consumes a sealed object may dispute a judgment it relies on, and, per Dataset #01's Part VI's seventh rung, an authorized human reviewer may dispute a judgment on a mission's behalf where the mission's own contract calls for that level of scrutiny. This breadth mirrors Part VIII §8.4's read-access breadth deliberately: a mechanism for correcting an honest mistake should be at least as widely available as the record being corrected is widely readable, because a narrowly gated dispute mechanism would leave most of the object's actual readership with no path to challenge a judgment they have good reason to distrust.

13.9 A worked example demonstrates the full cycle. A mission consuming a sealed object notices that its recorded confidence is high despite an anomaly signal that, on the mission's own domain-specific reading, looks more serious than the object's severity tier reflects — the object records a length-baseline deviation as informational tier, but the mission's own knowledge of this specific site suggests the deviation should have been degrading tier. The mission opens a dispute naming the specific anomaly, its recorded tier, and the domain-specific reasoning for why that tier looks miscalibrated. Resolution re-examines the anomaly-detection mechanism's own tier-assignment logic against the recorded observed value from Part V §5.7, finds that the length-baseline threshold used at construction time was, in fact, tuned for a different page type than the one actually acquired — a genuine defect in signal classification, not a subjective disagreement. The dispute resolves REVISED: a corrected record reclassifies the anomaly as degrading tier, triggers a re-invocation of Part VI's DRK-03 call with the corrected signal, and the resulting lower confidence value populates the revised record, explicitly linked back to the original.

13.10 This Part's closing provision addresses what a REVISED outcome means for anything already built on the disputed original — a package already handed to the Knowledge Distillation Fabric per Part XI, or a GK-04 edge already constructed per Part XVIII. Neither is retroactively unwound automatically; the revision record exists, is discoverable by following the original object's link, and any future consumer of the original — including, critically, the Knowledge Distillation Fabric or GK-04's own graph traversal — encounters a explicit pointer to the revision rather than silence about the correction. Whether a specific downstream artifact built on the pre-revision judgment is itself re-examined is a decision for that artifact's own owning engine, not something this Part's dispute mechanism reaches into and mandates, exactly as Part VIII §8.9 already established that GK-04 owns what happens on its own side of the edge it constructed.

PART XIII FINAL LAW

A dispute names a specific, falsifiable challenge to a sealed object's judgment, is resolved by re-invoking the same mechanism that produced the original judgment with any new evidence supplied, and closes UPHELD, REVISED, or INCONCLUSIVE — never by silently modifying the original, which remains sealed and intact regardless of outcome. A REVISED outcome creates a new, explicitly linked record; it does not retroactively unwind anything already built on the original, which remains that downstream artifact's own owning engine's decision to re-examine or not.


PART XIV — THE REPLAY DISCIPLINE: RECONSTRUCTING WITHOUT RE-ACQUIRING

14.1 Part VII §7.5 named replay as a use this dataset specifies around a deliberate limitation: a redacted Evidence Object cannot supply the original credential a reproduction of its acquisition would need. This Part specifies replay's actual mechanics — what it means to reconstruct an acquisition's context from a sealed Evidence Object without performing a new acquisition, what it can and cannot honestly claim to prove, and how it differs from Part X's change-history-driven re-acquisition, which does perform a new acquisition and is a different mechanism entirely despite superficial similarity in purpose.

14.2 Replay answers a narrower question than re-acquisition: given a sealed Evidence Object, can the specific conditions of the original acquisition — the exact URL, the exact acquisition method, the exact engine and version, the exact session profile reference where applicable — be reconstructed precisely enough that a human reviewer, or a diagnostic tool, could manually repeat the acquisition and compare the new result against the sealed object's own recorded fields. Replay itself does not perform this repetition; it produces the reconstructed context that makes manual or diagnostic repetition possible, drawing entirely on fields Dataset #01's Part VII already requires the object to carry — the acquisition method, the engine and version, the canonical and requested URLs, the redirect chain.

14.3 The replay context's completeness is bounded by exactly the redaction Part VII of this dataset applies: where the original acquisition required a session profile whose underlying credential was redacted, the replay context names that a session profile was required and references its non-secret identifier, but cannot supply the credential itself, and any actual re-attempt using that replay context would need a currently valid credential obtained through whatever authorized process originally supplied one — replay reconstructs the acquisition's shape, not a currently exploitable means of repeating it, and a replay mechanism that somehow could restore the original credential would have violated Part VIII §8.6's absolute prohibition on ever un-redacting a sealed field.

14.4 Replay's primary consumer, per this dataset's own scope, is Part XIII's dispute-resolution mechanism and Part XVI's self-auditing discipline, both of which sometimes need to understand precisely what an original acquisition did in order to evaluate whether a disputed judgment or a suspected defect traces back to something about how the acquisition itself was performed, distinct from how the resulting object was scored or gated. A secondary consumer is a human reviewer under Dataset #01's Part VI's seventh rung, investigating an anomaly pattern that automated mechanisms have flagged but not resolved, who needs the reconstructed context to manually inspect what happened without waiting for, or requiring, a fresh acquisition to be performed first.

14.5 Replay explicitly does not claim that a manually repeated acquisition, performed using its reconstructed context, will produce identical results to the original — the target site may have changed, per Part X's own change-history discipline; the acquiring engine may have been updated since the original acquisition per Part XX's versioning; network conditions or geographic routing may differ. Replay's honest claim is narrower and mechanical: it faithfully reconstructs what the fabric's own records show the original acquisition did, not a guarantee about what a repetition would find. A replay mechanism that implied reproducibility beyond this narrow, honest claim would itself be a version of Dataset #01's Part I founding failure — a clean, complete-looking reconstruction implying more certainty than the underlying mechanism can actually deliver.

14.6 The distinction between replay and Part X's re-acquisition, previewed in 14.1, is stated fully here because the two are easy to conflate given that both, in some sense, concern "doing the acquisition again." Re-acquisition is a live action: a new fetch, a new Evidence Object, a new entry in the source's change-history chain, performed because a mission's own frontier called for fresh evidence. Replay is a diagnostic reconstruction: no new fetch occurs as part of replay itself, no new Evidence Object is created, and replay's output is context for a human or tool to act on, not an acquisition result in its own right. A dispute or an audit that needs to know "what would this source return today" needs re-acquisition; one that needs to know "what did this specific sealed object's acquisition actually do" needs replay, and the two questions, despite their surface similarity, are answered by entirely different mechanisms for exactly that reason.

14.7 Replay context is itself constructed on demand from a sealed object's fields rather than pre-computed and stored at construction time, because pre-computing it would mean maintaining a second, derived representation of the same information the object's own fields already carry, subject to its own staleness and consistency risks relative to the object it derives from — a duplication this Part avoids for the same reason Part VI avoided pre-aggregating anomaly signals before handing them to DRK-03. Any consumer needing replay context requests it fresh from the sealed object's fields at the moment it is needed, guaranteeing the context always reflects exactly what the object currently, immutably records.

14.8 A worked example demonstrates replay's actual use. Part XVI's self-auditing discipline flags a cluster of objects, all acquired through a specific engine version, all carrying an unusually high rate of provenance anomalies per Part V §5.6. Investigating this cluster does not require re-acquiring any of the flagged sources; it requires replay: reconstructing, from each flagged object's own fields, the exact engine version, acquisition method, and session context each acquisition used, to determine whether the anomaly pattern correlates with a specific engine version's own defect rather than with anything about the target sites themselves. The replay contexts, compared across the cluster, reveal that every flagged object used the same engine version, and this finding — reached without a single new acquisition — is what actually localizes the defect Part IX §9.5 already anticipated as one possible explanation for an integrity anomaly's underlying cause.

14.9 Replay's own output, once constructed for a given investigation, is not itself retained as a new durable record by default, distinguishing it from every other mechanism this dataset specifies that produces a lasting artifact — a dispute record, a change-history entry, a tombstone. Replay context is transient, reconstructed for the specific investigation that requested it and discarded once that investigation concludes, unless the investigating mechanism — Part XIII's dispute resolution, Part XVI's self-auditing — explicitly chooses to attach a summary of what the replay revealed to its own durable output, in which case that summary, not the raw replay context itself, becomes the lasting record.

14.10 A closing provision states replay's relationship to authorization directly, extending Part VIII §8.10's custody-versus-authorization distinction into this Part's own territory: reconstructing an acquisition's context for diagnostic purposes is not itself a new acquisition and therefore does not require Governance, Safety, and Authorization's re-approval the way an actual re-acquisition per Part X would — replay reads already-authorized, already-completed historical records, it does not act against any external target. Should a diagnostic investigation's findings lead someone to want to actually repeat the acquisition against the live source, that repetition is a new acquisition in every sense Dataset #01's Part V governs, subject to the same authorization discipline as any other, and replay's own reconstructed context, however detailed, confers no standing authorization of its own.

PART XIV FINAL LAW

Replay reconstructs an acquisition's context — method, engine version, URLs, redirect chain, non-secret session references — from a sealed object's own fields, for diagnostic use by dispute resolution, self-auditing, or human review, without performing any new acquisition and without claiming that a manual repetition would reproduce identical results. It is distinct from Part X's re-acquisition, which is a live, new action; replay is a transient reconstruction of history, discarded after use unless an investigating mechanism explicitly chooses to retain a summary of its findings.


PART XV — CROSS-MISSION PROVENANCE: EVIDENCE REUSED ACROSS MULTIPLE MISSIONS

15.1 Part III §3.3 referenced a cross-mission reuse discipline this Part now specifies fully: when two different missions, unaware of each other, independently acquire the same source, how does the fabric recognize that fact, and what does recognizing it actually change about how the resulting objects are constructed, scored, and made available to each mission. This is a distinct question from Part X's change-history mechanics, which track one source's evolution across successive acquisitions regardless of which mission performed each one; this Part tracks whether two acquisitions performed by different missions, close enough in time that neither is clearly superseding the other, should be treated as the same evidence rather than two independent, redundant objects.

15.2 Recognition begins identically to Part III's supersession check: during FIELDS-POPULATING, the fabric queries evidence memory for a prior object matching the current acquisition's canonical URL. Where Part III's mechanics resolve a single, unambiguous prior match as a supersession, this Part handles the case Part III explicitly deferred: a prior match whose acquisition timestamp is recent enough, relative to the source's own observed change frequency in site memory, that the new acquisition should not automatically be treated as superseding it — the two acquisitions may simply be two independent, near-simultaneous observations of the same, unchanged content, requested by two missions that have no knowledge of each other's activity.

15.3 The fabric's response to a recognized cross-mission match is governed by the normalized-hash comparison Part IV already specifies, applied here for a purpose distinct from Part X's change detection: if the new acquisition's normalized hash matches the near-simultaneous prior object's normalized hash, the new acquisition is not constructed as an entirely independent Evidence Object requiring its own full FIELDS-POPULATING derivation from scratch; instead, a lightweight cross-mission reference object is constructed, carrying the new mission's own provenance-chain elements per Part III's mechanics but pointing to the existing, already-verified object's confidence score, anomaly signals, and gate results rather than re-deriving them, since re-deriving identical judgments from identical content would waste exactly the resources Dataset #01's Part VI cost ladder exists to conserve.

15.4 The reference object's provenance chain, despite pointing to a shared underlying acquisition, remains fully honest about which mission it actually serves: it carries the new mission's own mission identifier and frontier-item reference, per Part III §3.2's mechanics, and explicitly records that its confidence, anomaly, and gate data are inherited from the referenced original object rather than independently derived for this specific mission's acquisition. A downstream reader consuming the reference object can distinguish, from this explicit inheritance marker, an object whose evidence was freshly derived for this exact mission from one reusing a near-simultaneous sibling mission's already-verified work — a distinction Part I §1.8's own worked framing named as a question this dataset must be able to answer honestly.

15.5 Cross-mission reuse is not automatic for every matching pair regardless of similarity; it requires the normalized-hash match Part IV specifies, computed exactly as it would be for Part X's change-history purpose, and a raw-content check confirming the underlying acquisition method and engine were compatible enough that the original object's anomaly and gate judgments genuinely transfer — an acquisition performed via a basic fetch and one performed via a full browser render of the same URL, even with matching normalized content, are not automatically treated as interchangeable for reuse purposes, because Part V's structural anomaly checks and Part XIII's Reality Gate evaluation may have examined properties — a rendered visual capture, for instance — that a basic-fetch acquisition never produced at all, and reuse across incompatible methods would silently misrepresent what was actually checked for the reusing mission's own acquisition.

15.6 A worked example demonstrates the mechanism operating correctly. Two missions, launched independently within the same hour by different operators researching unrelated questions, both include the same widely-referenced technical specification page in their respective frontiers. The first mission's acquisition completes fully: FIELDS-POPULATING, SIGNAL-EXAMINED, SCORED, GATE-EVALUATED, reaching INSTITUTIONALIZATION-ELIGIBLE with a clean confidence profile. The second mission's acquisition, beginning minutes later, finds the first mission's object during FIELDS-POPULATING's supersession check; the normalized hashes match; the acquisition methods are compatible. A lightweight reference object is constructed for the second mission, carrying its own provenance chain and an explicit inheritance marker pointing to the first mission's object, without re-running signal examination or a second DRK-03 scoring call. Both missions can now cite provenance-carrying evidence for the same specification page, one holding the original derivation, one holding an honestly marked reference to it, at a fraction of the acquisition cost a fully independent second acquisition would have required.

15.7 A reference object's relationship to Part XI's package composition follows the inheritance marker's own logic: a package built to support the second mission's claim may include the reference object, and the Knowledge Distillation Fabric consuming that package can, per the inheritance marker, trace back through the reference to the original object's own full evidentiary detail if the mission's contract requires that level of scrutiny, without the reference object itself needing to duplicate that detail. This mirrors Part XVIII's GK-04 edge construction, where a compact pointer stands in for a fuller record rather than replacing it — cross-mission reuse applies the same compact-pointer discipline within Crawl OS's own evidence memory rather than at the estate graph boundary.

15.8 The mechanism's failure mode, where it occurs, is a reference object outliving the freedom Part XV's own reuse logic assumed: if the original object referenced by a cross-mission reference is later disputed under Part XIII and REVISED, every reference object pointing to it inherits a stale judgment until each is itself re-examined. This Part's own binding requirement addresses that risk directly: a REVISED outcome under Part XIII must trigger a lookup of every extant cross-mission reference pointing to the disputed original, and each is flagged for re-evaluation rather than left silently pointing at a judgment now known to be superseded — a reference object's inheritance is a live pointer to the original's current state for exactly this reason, never a frozen copy taken at reference-construction time.

15.9 Cross-mission reuse's boundary against Part X's change-history mechanics is drawn by timing and mission-independence together, not by either alone: two acquisitions of the same source by the same mission, close together in time, are never candidates for cross-mission reuse regardless of matching hashes, because they are, definitionally, the same mission's own repeated interest in a source, handled entirely by Part X's supersession mechanics; cross-mission reuse activates specifically, and only, when the two acquiring missions are different and neither has superseded the other in any temporal sense Part III's own mechanics would recognize.

15.10 This Part's closing provision states honestly what cross-mission reuse does not attempt to solve: discovering that two missions are about to acquire the same source before either acquisition actually begins, which would require the Resource Discovery and Frontier Engine or the Acquisition Strategy Router, per Dataset #01's Part V §5.3/5.4, to coordinate across concurrently active missions' frontiers — a capability neither of those engines' charters currently include and which this dataset, owning evidence construction rather than frontier planning, has no authority to add on their behalf. This Part's mechanism operates entirely after the fact, recognizing redundancy once a second acquisition's FIELDS-POPULATING actually begins, not preventing the redundant acquisition attempt from being initiated in the first place — a genuine, honestly stated limitation rather than a capability this Part could claim without overreaching into a neighboring engine's own charter.

PART XV FINAL LAW

When two different missions' near-simultaneous acquisitions of the same source produce matching normalized content via compatible acquisition methods, the second is constructed as a lightweight reference object carrying its own honest provenance chain but inheriting the first's confidence, anomaly, and gate judgments through an explicit, live-pointing inheritance marker — never a frozen copy. A Part XIII revision of the original propagates to every reference pointing at it, and the mechanism honestly does not attempt to prevent redundant acquisition attempts before they begin, only to recognize and collapse the redundancy once it occurs.


PART XVI — THE FABRIC'S OWN FAILURE MODES AND SELF-AUDITING

16.1 Every Part before this one specified a mechanism and, in most cases, referenced "Part XVI's self-auditing discipline" as the backstop catching that mechanism's own possible failure — the anomaly taxonomy's living pattern list, the redaction pass's silent under-detection risk, tamper-evidence's periodic sweep, retention's expiry review, cross-mission reuse's revision propagation. This Part collects those scattered references into the single, coherent self-auditing discipline they all presupposed, because a fabric whose every mechanism depends on a backstop that was never itself specified has built a building whose foundation is a name, not a structure.

16.2 Self-auditing runs two categories of check on a fixed schedule, distinct in purpose though sharing the same underlying periodic-sweep mechanism. Integrity checks re-verify already-sealed objects against the guarantees this dataset makes about them — Part IX's tamper-evidence re-verification, Part VII §7.8's credential-pattern re-sweep, Part XII §12.6's retention-expiry review. Pattern checks examine trends across the fabric's own operational history that no single object's own construction could surface — a rising provenance-anomaly rate per Part V §5.6, a cluster of objects sharing a defect per Part XIV §14.8's worked example, a redaction pass whose detection rate has dropped in a way suggesting its underlying pattern list, per Part V §5.3, has gone stale against evolving interstitial wording.

16.3 The distinction between the two categories matters for how each is remediated. An integrity check's finding — a specific TAMPER-SUSPECTED object, a specific credential that slipped past redaction — is remediated per-object, following each mechanism's own specified consequence, and does not by itself imply anything is wrong with the fabric's general operation. A pattern check's finding implies something systemic: a rising anomaly rate, a defect cluster, a stale pattern list are findings about the fabric's own construction logic or its supporting reference data, not about any single object, and remediation is correspondingly systemic — a code fix, a pattern-list update, an engine-version rollback — rather than an object-by-object correction.

16.4 Self-auditing's own severity classification, referenced throughout this dataset as "Part XVI-severity," is a three-tier scale distinct from Part V's anomaly-signal severity tiers, and this Part states the distinction explicitly to prevent the same conflation Part VI §6.10 already guarded against for CONNECT and the verdict ontology. OBSERVATIONAL findings are logged for trend tracking and require no immediate action — a single provenance anomaly, isolated and not yet part of any rising pattern. ACTIONABLE findings require remediation on a bounded timeline but do not halt fabric operation — a stale pattern list, a single confirmed tamper finding. CRITICAL findings — a systemic redaction failure, a defect cluster affecting a substantial fraction of recent acquisitions — halt new object construction through the affected pathway until remediated, because continuing construction through a known-defective pathway would manufacture more objects requiring the same remediation, compounding the problem the finding just surfaced.

16.5 The self-auditing discipline's own trigger for a CRITICAL halt is deliberately conservative rather than permissive, because a fabric that continues constructing objects through a pathway known to be systemically defective is actively working against Dataset #01's Part VII §7.9 fabrication prohibition at scale — every object constructed after a CRITICAL finding through the affected pathway, before remediation, risks carrying the same defect, and Part I §1.10's own honesty-about-interim-state discipline extends here: the fabric must be capable of declaring itself temporarily unable to construct trustworthy objects through a specific pathway, rather than continuing to produce output that looks complete while carrying a known, unremediated defect.

16.6 Self-auditing's relationship to the Crawl Cortex Supervisor's own evidence-quality-assurance loop, chartered in Dataset #01's Part V §5.12, is composition rather than absorption, following the same discipline Part I §1.7 already established for this dataset's boundary with that loop: self-auditing is this Part's own mechanism, internal to the Evidence Integrity Fabric, producing findings in the categories and severities named above; the Supervisor's loop is a separate, estate-level oversight mechanism that consumes those findings as one of its own inputs, alongside findings from every other engine's own internal health-checking, and correlates them against patterns visible only at the estate level — a rising CRITICAL rate specifically in the Evidence Integrity Fabric, compared against stable rates elsewhere, is exactly the kind of cross-engine correlation the Supervisor's loop exists to notice and this Part's own self-auditing, scoped to one engine, structurally cannot.

16.7 The periodic sweep's own cadence is, like several other operational parameters this dataset has declined to fix as specific numbers, an implementation-tuning decision for `modules/crawl_os/evidence.py`, calibrated against the actual volume of Evidence Objects the fabric is constructing and the actual cost of running a full sweep — a decision composing with CO-01's operating-economics territory exactly as Part XII §12.5 already established for retention periods. What this Part fixes as binding is that a sweep runs on some fixed, non-zero cadence rather than only reactively when a specific dispute or investigation happens to trigger a targeted check, because a purely reactive posture would mean systemic defects — the pattern checks named in 16.2 — are never discovered except by accident, defeating the entire purpose of a dedicated self-auditing discipline.

16.8 A worked scenario ties the categories and severities together. The periodic sweep's pattern check finds that objects constructed in the past week show a provenance-anomaly rate three times the fabric's own historical baseline. This is, on its own, an ACTIONABLE finding — a real, measured deviation, not yet confirmed to be a single root cause. Investigation, using Part XIV's replay discipline exactly as its own worked example demonstrated, traces the anomalies to a single recently deployed engine-version update to one of the eleven adapters. The finding is re-classified CRITICAL, because the affected pathway — acquisitions routed through that specific engine version — is now known-defective and continuing to construct objects through it would compound the problem. Construction through that pathway halts; the Acquisition Strategy Router, per Dataset #01's Part V §5.4, is notified to route around the affected engine version until remediation completes, an interaction this Part's own charter does not itself own but which its CRITICAL-severity finding is what actually triggers.

16.9 Self-auditing's findings are themselves retained as durable records, following the same never-silently-discard discipline this dataset has applied to every other category of record — a superseded object, a tombstone, a dispute record. A finding's record, once remediation completes, is not deleted; it is marked remediated, with the remediation action and date recorded, so that a future audit of the fabric's own operational history can distinguish a pathway that has never had a CRITICAL finding from one that has had several, each successfully remediated — information relevant to judging the fabric's own long-run reliability that a system which discarded resolved findings would have silently lost.

16.10 A closing provision addresses self-auditing's own self-referential limit honestly, in the register this dataset has held throughout for stating what a mechanism cannot do alongside what it can: self-auditing can detect a defect in the fabric's construction logic, its pattern lists, or its integration with DRK-03 and the redaction detector, because each of those has an external reference point — a stored hash, a known-good pattern source, a directly observable rate — against which a deviation is measurable. It cannot, by the same mechanism, detect a defect in its own detection logic — a bug in the sweep's own pattern-check implementation that caused it to systematically under-report a specific class of finding would not be caught by the sweep checking itself, because the sweep has no external reference point for its own correctness in the way it has one for the objects it examines. That residual risk is not resolved within this dataset's own scope; it is the reason the Crawl Cortex Supervisor's estate-level correlation, per 16.6, and periodic human review under Dataset #01's Part VI seventh rung both remain necessary layers this Part's own mechanism does not and cannot replace.

PART XVI FINAL LAW

Self-auditing runs integrity checks (re-verifying specific sealed objects against this dataset's own guarantees) and pattern checks (surfacing systemic trends no single object's construction could reveal) on a fixed periodic cadence, classifying findings OBSERVATIONAL, ACTIONABLE, or CRITICAL — the last halting construction through a known-defective pathway rather than continuing to manufacture objects that would inherit the same defect. Findings are retained, never silently discarded, and the mechanism's own honest limit is that it cannot detect a defect in its own detection logic, a residual risk that estate-level Supervisor correlation and human review, not this Part's own mechanism, exist to cover.


PART XVII — ELABORATING THE INTERFACE WITH THE PRODUCTION REALITY GATES

17.1 Part II §2.6 named GATE-EVALUATED as the lifecycle state in which a fully constructed, scored Evidence Object is submitted to Dataset #01's Part XIII Reality Gates, and referenced this Part for the operational detail of what that submission actually consists of. This Part specifies that interface: exactly which of this dataset's own fields and mechanisms feed which of the ten gates, and — completing a loop Part I §1.7 already flagged as this dataset's own boundary — confirms that gate pass/fail logic itself remains entirely Dataset #01's own territory, never re-derived here.

17.2 The interface is a mapping, not a re-specification: each of the ten gates named in Dataset #01's Part XIII draws on specific fields this dataset's mechanics populate, and this Part states that mapping explicitly so a future implementer of `modules/crawl_os/evidence.py` knows precisely which internal state feeds which external evaluation, without needing to re-derive the mapping from reading both datasets' prose side by side. Gate A, Acquisition Reality, per Dataset #01's Part XIII §13.3, draws on this dataset's RAW-RECEIVED and FIELDS-POPULATING completion status — whether the object reached FIELDS-POPULATING at all, or was recorded CONSTRUCTION-ABORTED per Part II §2.8, is the fact Gate A actually evaluates.

17.3 Gate B, Content Reality, per Dataset #01's Part VIII, draws primarily on Part V's anomaly taxonomy: the consent-and-access and content-mismatch categories specifically, since Dataset #01's own Part VIII §8.5 named exactly the checks — topical consistency, title-path consistency, length-baseline comparison — that Part V of this dataset operationalized as detection mechanisms. A disqualifying-tier anomaly in either of those two categories, per Part V §5.8's severity mapping, is what actually causes Gate B to fail; Gate B's own pass/fail threshold logic, however, belongs entirely to Dataset #01's Part XIII, which this dataset's anomaly signals merely feed as input, in the same non-duplication spirit Part VI established for DRK-03's confidence logic.

17.4 Gate C, Authorization Reality, draws on a field this dataset's mechanics populate but do not themselves evaluate: the provenance chain's mission reference, per Part III, combined with whatever authorization record the Governance, Safety, and Authorization engine attached to that mission at the point of acquisition, per Dataset #01's Part V §5.13. This dataset's own fabric never independently judges whether an acquisition was authorized — Part VIII §8.10 already drew that boundary — and Gate C's evaluation reads the authorization record Governance itself produced, using this dataset's provenance chain only to locate which mission's authorization record is the relevant one.

17.5 Gate D, Extraction Fidelity, draws on Part V's structural anomaly category — a malformed table, an inconsistent MIME type — combined with whatever confidence the Extraction and Rule Induction Fabric itself attaches to its own output before handing it to this dataset's fabric for construction; Gate D evaluates the joint result, and this dataset's contribution is limited to the structural anomalies its own taxonomy can detect after the fact, not the extraction process's own internal confidence, which belongs to the upstream engine that performed it.

17.6 Gate E, Provenance Completeness, draws directly and entirely on Part III's provenance chain mechanics: whether the mission, frontier-item, and supersession fields populated per Part III §3.2 and §3.3's mechanics without falling into the ambiguous multiple-match case that Part III flags for manual resolution rather than resolving automatically. An object whose provenance chain assembly hit that ambiguous case and was flagged, per Part III §3.3, fails Gate E by construction, since an unresolved ambiguity is definitionally an incomplete provenance record regardless of how complete every other field is.

17.7 Gate F, Secret Hygiene, draws directly on Part VII's redaction pass: whether the pass ran to completion without error, and whether Part VII §7.8's periodic credential-pattern re-sweep has, as of the gate evaluation, found any hit against this specific object. Gate F is the one gate whose relevant input this dataset's own mechanism is most directly responsible for producing correctly, since a redaction failure is, per Part VII §7.9, treated with the same severity as any other fabrication — Gate F failing is this dataset's own failure surfacing through Dataset #01's own evaluation layer, not merely an external judgment about content the fabric had no hand in.

17.8 Gate G, Confidence Honesty, per Dataset #01's Part XIII §13.8, draws on exactly the mechanism Part VI of this dataset specifies: whether the object's confidence score actually derives from the DRK-03 CONNECT call fed with real, complete signals rather than a default, a hardcoded value, or a value copied forward from a different acquisition. Part VI §6.9's fabrication-input prohibition is what Gate G's own evaluation is checking for, and this dataset's own construction discipline — never pre-aggregating, never caching CONNECT's thresholds, never copying a prior confidence value forward — is what makes Gate G passable honestly rather than passable only by an object whose confidence was fabricated in a way that happens to look plausible.

17.9 Gate H, Freshness Sufficiency, draws directly on Part XII's freshness mechanics: the acquisition timestamp compared against the mission's stated maximum age, exactly as Part XII §12.3 specifies the check. Gate I, Coverage Sufficiency, draws on whether a mission's full evidence requirement — potentially spanning multiple frontier items per Part XI §11.4's multi-object package case — has been jointly satisfied, which this dataset's fabric can report accurately only once every relevant object in a mission's frontier has itself reached GATE-EVALUATED, making Gate I evaluable only after every contributing object's own gate evaluation completes, an ordering constraint worth naming explicitly since it distinguishes Gate I from every other gate, each of which evaluates a single object in isolation.

17.10 Gate J, Reusability Readiness, per Dataset #01's Part XIII, draws jointly on this dataset's Part XIV's institutionalization-eligibility criteria as already applied at Part II §2.7's branch point and on Part XI's package-manifest completeness — an object cannot be Gate J-ready if it has reached INSTITUTIONALIZATION-ELIGIBLE per this dataset's own lifecycle but has not yet been successfully wrapped in a well-formed package per Part XI's mechanics, since reusability, as Dataset #01's Part XIII treats it, means ready for actual downstream consumption, not merely eligible in principle.

17.11 A closing worked example ties the full mapping together for a single object. An acquisition completes RAW-RECEIVED and FIELDS-POPULATING without interruption (Gate A input satisfied). SIGNAL-EXAMINED records zero consent-access or content-mismatch anomalies (Gate B input clean). The provenance chain, per Part III, resolves the mission's authorization record as valid (Gate C). Extraction produced no structural anomalies (Gate D). The provenance chain's three elements all populated without ambiguity (Gate E). The redaction pass ran clean, no later re-sweep hit (Gate F). SCORED reflects a genuine DRK-03 call over real signals (Gate G). The acquisition timestamp is within the mission's stated freshness window (Gate H). This object is the mission's only frontier item, so coverage is trivially satisfied once this one object gates (Gate I). Institutionalization eligibility and package construction both complete (Gate J). All ten gates pass, and the object proceeds through Part XI's package composition to the Knowledge Distillation Fabric — a trace that demonstrates the interface as a whole rather than any single gate in isolation.

17.12 This Part's own scope closes with the same discipline every composition boundary in this dataset holds: gate pass/fail thresholds, gate ordering, and what a partial-pass profile means for a mission's overall completeness are entirely Dataset #01's Part XIII's own territory, unchanged and uncited-away by anything specified here. This Part's contribution is exclusively the mapping from this dataset's own fields and mechanisms to each gate's required input, so that the two datasets compose as a working interface rather than remaining two independently correct documents that happen never to specify how their own content actually connects.

PART XVII FINAL LAW

Each of Dataset #01's ten Reality Gates draws on specific fields and mechanisms this dataset's Parts II through XVI populate — the lifecycle's completion status, the anomaly taxonomy, the provenance chain, the redaction pass, the DRK-03-derived confidence, the freshness check, and institutionalization-and-packaging completeness — and this Part states that mapping explicitly. Gate pass/fail threshold logic itself remains entirely Dataset #01's own territory; this dataset supplies honest input, never a second evaluation layer duplicating what Dataset #01 already owns.


PART XVIII — THE GK-04 COMPOSITION IN OPERATIONAL DETAIL

18.1 Dataset #01's Part XVI declared the composition point with GK-04 at the architectural level: an institutionalized Evidence Object triggers a typed edge write, Crawl OS constructs the edge's content, GK-04 owns writing and storage. This Part elaborates the operational detail Dataset #01's own Part XVI §16.5 explicitly deferred to a later dataset — precisely what content this dataset's fabric constructs for that edge, drawn from which of its own fields, and at what exact moment in this dataset's own lifecycle the construction actually triggers.

18.2 The trigger moment is Part II §2.7's INSTITUTIONALIZATION-ELIGIBLE transition specifically, not any earlier lifecycle state and not the later moment Part XI's package composition completes — edge-content construction and package construction are, per Part XI §11.8, independent parallel consumers of the same eligibility state, and this Part's own trigger is stated identically to avoid any implication that one waits for the other. The instant an object's lifecycle reaches INSTITUTIONALIZATION-ELIGIBLE, both Part XI's package-construction step and this Part's edge-content-construction step become eligible to run, in either order or concurrently.

18.3 The edge content this dataset constructs, per Dataset #01's Part XVI §16.4's interface specification, consists of exactly four elements this dataset's own fields directly supply: the origin marker, fixed as crawl_os for every edge this fabric ever constructs; the Evidence Object's own identifier, letting GK-04's future writer reference the specific object rather than a category of objects; the estate node the evidence now supports, drawn from the mission's own Mission Contract per Dataset #01's Part III, since the contract is what originally stated what claim or estate artifact the acquisition was in service of; and a compact summary of the object's own confidence and gate-pass profile, drawn from Part VI's SCORED confidence value and Part XVII's ten-gate result set, giving a graph traversal enough information to judge the edge's own evidentiary strength without needing to fetch the full Evidence Object first.

18.4 The edge content deliberately excludes several fields this dataset's Evidence Object carries in full, and this exclusion is as binding as the inclusion list in 18.3: the raw and normalized content, the full anomaly signal list with observed values, and the complete provenance chain's frontier-item-level detail are not part of the edge content, because Dataset #01's Part XVI §16.7 already established that the edge is a compact pointer, not a replacement for the object's own full record — an edge carrying the object's entire content would collapse the very distinction that Part established, effectively duplicating the Evidence Object's own storage inside GK-04's graph rather than pointing to it.

18.5 This dataset's fabric constructs the edge content and hands it to GK-04's writer as a single, complete payload; it does not construct the edge in stages, and does not hand over a partial payload expecting GK-04's writer to fetch remaining fields itself from evidence memory. This discipline matters specifically because Dataset #01's Part XVI §16.4 already established that GK-04 owns writing, storage, and query surface, never reaching back into Crawl OS's own evidence memory to assemble content Crawl OS itself should have supplied — a staged or partial hand-off would quietly require GK-04's writer to perform work Dataset #01's own boundary assigned entirely to Crawl OS's side of the interface.

18.6 The interim state, restated from Dataset #01's Part XVI §16.5 and honestly reaffirmed here because this Part's own operational detail does not change the underlying fact: GK-04's writer does not yet exist, so this dataset's edge-content-construction step, however precisely specified here, has never actually executed against a real writer. What this Part adds to that honest interim state is that the construction step's own logic — the four-element payload, the exclusion list, the single-complete-payload discipline — is fully specified today and requires no further design decision once the writer exists; a future contributor implements against this Part's specification rather than inventing the payload shape at that time, exactly as Dataset #01's Part XVI §16.11 already committed to.

18.7 A specific failure mode this Part's operational detail exists to preempt, beyond what Dataset #01's own architectural-level Part XVI could anticipate without this level of specificity: an edge-content construction that draws the confidence and gate-pass summary named in 18.3 from a stale copy computed earlier in the object's lifecycle, rather than from the object's final, sealed SCORED and GATE-EVALUATED state. Because edge construction triggers at INSTITUTIONALIZATION-ELIGIBLE, which by Part II's lifecycle definition occurs strictly after both SCORED and GATE-EVALUATED complete, the edge content's confidence-and-gate summary is always drawn from the object's final, sealed values — there is no earlier point in the lifecycle from which a stale value could be drawn, and this ordering guarantee is itself a direct consequence of the lifecycle discipline Part II established, not an additional check this Part needs to separately enforce.

18.8 The edge content's estate-node reference, named in 18.3 as drawn from the Mission Contract, requires one further precision: where a mission's contract names a specific estate artifact — a dataset, a decision record — as the target the acquired evidence supports, the edge content references that artifact directly; where the mission's contract states a more general research objective without naming a specific pre-existing artifact, the edge content references the mission itself as the target, since the estate artifact the evidence ultimately supports may not exist yet, or may be the Knowledge Distillation Fabric's own future output rather than anything nameable at acquisition time. This dataset's fabric never invents an artifact reference the mission's own contract did not actually supply.

18.9 A worked example, extending Dataset #01's Part XVI §16.8's own worked case with the operational detail this Part adds, traces the full construction. A mission institutionalizes a project's documented rate limits. The object reaches INSTITUTIONALIZATION-ELIGIBLE. Edge-content construction triggers: origin marker crawl_os; the object's own identifier; the estate node, drawn from the mission's contract, naming the specific documentation dataset this rate-limit fact now supports; a compact summary noting high confidence and all ten gates passed. This payload, complete and single, is ready to hand to GK-04's writer the moment that writer exists — nothing about this payload's construction depends on anything GK-04's own future implementation will need to decide, because this Part has already specified the payload's full shape.

18.10 This Part's closing provision names its own relationship to Part XIII's dispute-resolution mechanism directly, extending 18.7's finality guarantee into the one case where a sealed object's judgment can still change after institutionalization: a REVISED outcome under Part XIII, per Part VIII §8.9 and Dataset #01's Part XVI §16.5's own honest interim-state framing, does not retroactively reconstruct or re-send edge content — once the writer exists, a REVISED finding for an already-institutionalized object triggers a new, distinct edge-flag interaction, not a silent overwrite of content already handed to GK-04's own storage layer, preserving the same append-only discipline this dataset has applied to every other durable artifact it produces.

PART XVIII FINAL LAW

Edge-content construction triggers at INSTITUTIONALIZATION-ELIGIBLE, drawing exactly four elements — origin marker, object identifier, estate-node reference from the Mission Contract, and a compact confidence-and-gate summary — as a single, complete payload, explicitly excluding the object's full content and detail, which the edge points to rather than duplicates. The payload's shape is fully specified today though GK-04's writer does not yet exist; a future implementer builds against this specification rather than inventing one, and a Part XIII revision of an already-institutionalized object triggers a new edge-flag interaction, never a silent overwrite.


PART XIX — THE NEGATIVE EVIDENCE RECORD: FAILURE AS INSTITUTIONAL KNOWLEDGE

19.1 Dataset #01's Part IV §4.11's negative memory, and its own Part V §5.11 charter for Crawl Memory and Change Intelligence, both name a failed or degraded acquisition's own record as institutional knowledge in its own right, distinct from and not lesser than a successful acquisition's record. This Part specifies how this dataset's own mechanics actually produce that negative-evidence record — what a RETAINED-NON-INSTITUTIONAL or CONSTRUCTION-ABORTED object, per Part II's lifecycle, contributes to the estate's institutional memory beyond simply existing as an unpromoted record sitting in evidence memory.

19.2 A negative evidence record is not a new object category alongside the ones Part II already named; it is a specific, structured extraction from an object that reached RETAINED-NON-INSTITUTIONAL, summarizing why institutionalization was not earned in terms the Crawl Memory and Change Intelligence engine's site memory can actually consume and reuse for future missions against the same or similar sources. Where Part II's lifecycle states that a well-formed but low-scoring object is retained rather than deleted, this Part specifies what makes that retained object actively useful rather than merely not-discarded.

19.3 The extraction draws directly on Part V's anomaly taxonomy: which categories and severities of anomaly were recorded, and, critically, whether the anomaly pattern is one Site Memory should now treat as a baseline expectation for this specific source going forward — a site whose acquisitions repeatedly trigger a specific structural anomaly, for instance, teaches Site Memory that this site's pages structurally resemble the anomaly's pattern under normal, non-failing conditions, refining the very baseline Part V §5.4's length-comparison check depends on, rather than the anomaly being treated as a one-off surprise on every subsequent visit.

19.4 A negative evidence record's most direct consumer, per Dataset #01's Part V §5.11's own charter, is the Acquisition Strategy Router's future routing decisions for the same source: a source whose negative evidence record shows a consistent consent-and-access anomaly at the cost ladder's second rung, per Dataset #01's Part VI, informs a future mission's router that this source's own observed baseline requires escalating past the basic-fetch rung by default for this specific site — not a blanket escalation policy this dataset has no authority to set, per Part VI §6.4's observed-reason discipline, but a site-specific, evidence-backed observation the router itself is chartered to weigh precisely because it is a real observed reason, not an untested prediction.

19.5 The negative evidence record's relationship to Part II's CONSTRUCTION-ABORTED category is narrower than its relationship to RETAINED-NON-INSTITUTIONAL, and this Part draws that distinction explicitly because the two source records carry different evidentiary weight. A CONSTRUCTION-ABORTED record's own minimal content, per Part II §2.8, supports only a narrow negative-evidence extraction — that construction was attempted and could not complete, and roughly why — useful chiefly for detecting the kind of re-attempt loop Part II §2.8 already named as the record's primary purpose. A RETAINED-NON-INSTITUTIONAL record's negative evidence is richer, because the object completed its full lifecycle through GATE-EVALUATED and carries a complete anomaly and confidence profile, supporting the kind of specific, site-baseline-refining extraction 19.3 describes.

19.6 This dataset's own boundary against owning Crawl Memory's actual storage and retrieval mechanics, per Part I §1.7's composition discipline, applies identically here: this Part specifies what negative evidence this dataset's own mechanics produce and hand off, not how Crawl Memory and Change Intelligence stores, indexes, or later retrieves that extraction for a future mission's router to consult. A future dataset elaborating that engine's own charter inherits this Part's negative-evidence extraction format as its own upstream input, in the same inheritance relationship this dataset holds toward Dataset #01's Part VII.

19.7 A specific discipline this Part enforces, extending Dataset #01's Part VII §7.9's fabrication prohibition into negative-evidence-specific terms: a negative evidence extraction must derive entirely from the anomaly signals and gate results the object's own lifecycle actually recorded, never from a plausible-sounding generalization an engine constructs after the fact about "sites like this one tend to." The extraction is object-specific and source-specific, built from what was actually observed about this particular source across its own change-history chain, and a fabric that began generalizing across unrelated sources to produce a broader negative-evidence pattern would have quietly drifted from evidence-backed observation into unverified prediction — exactly the distinction Dataset #01's Part VI §6.4 already drew for the router's own escalation discipline, applied here to the memory feeding that discipline instead.

19.8 A worked example demonstrates the mechanism's value concretely. A source is acquired three times across several months by three different missions, each time reaching RETAINED-NON-INSTITUTIONAL because the extracted content, while genuine, consistently falls short of the specific reusability criteria Dataset #01's Part XIV requires — the source's own content, it turns out, is structured in a way that never quite meets the generality criterion for this family's typical missions. Each acquisition's negative evidence record notes this same shortfall independently. A fourth mission's own Mission Contract, per Dataset #01's Part III, is compiled with this negative-evidence pattern available to consult, and its own evidence requirement is calibrated accordingly — either accepting a lower institutionalization bar explicitly, per its own contract, or excluding this source from its frontier entirely as a source that has, across three independent attempts, never produced institutionalizable evidence for this family's typical purposes. The fourth mission benefits from three prior missions' honest failures without needing to repeat any of the acquisitions that produced them.

19.9 The negative evidence record's own retention follows Part XII's category structure rather than a separate policy this Part invents: a RETAINED-NON-INSTITUTIONAL object's negative evidence retains for exactly as long as the object itself does, per Part XII §12.4's bounded, mission-tied retention period for that category, and does not persist independently past the underlying object's own retention expiry — a negative-evidence record that outlived the object it was extracted from would be an unattributed claim about a source with no traceable object to verify it against, violating the same provenance discipline Part III established for every other kind of evidence this dataset produces.

19.10 A closing distinction separates this Part's negative evidence record from Part XVI's self-auditing findings, since both are, in a loose sense, records of something not going as expected and could be conflated without this clarification. Self-auditing findings are about the fabric's own operation — is the fabric itself working correctly. Negative evidence records are about a target source's own observed behavior — is this source, external to the fabric entirely, one that tends to produce difficult or insufficient evidence. A source with a strong negative-evidence history is not a fabric defect; it may simply be a source poorly suited to this family's evidentiary needs, and treating a strong negative-evidence pattern as though it indicated a self-auditing-severity fabric problem would misattribute an external fact about the world to an internal fact about the mechanism observing it.

PART XIX FINAL LAW

A negative evidence record is extracted from a RETAINED-NON-INSTITUTIONAL or CONSTRUCTION-ABORTED object's own recorded anomaly and gate history, derived strictly from what was actually observed about that specific source, never generalized into an unverified prediction about similar sources. It refines Site Memory's own baseline and informs future routing decisions with a real, observed reason per Dataset #01's Part VI discipline, retains only as long as its underlying object does, and is distinct from Part XVI's self-auditing findings — one is a fact about a source external to the fabric, the other a fact about the fabric's own operation.


PART XX — SCHEMA VERSIONING AND MIGRATION IN PRACTICE

20.1 Dataset #01's Part VII §7.8 established that the Evidence Object schema is versioned and that a version change is a Part XXIV amendment event, owned centrally by the Evidence Integrity Fabric, never a silent per-engine field addition. This Part specifies the operational mechanics that discipline requires: how a version is actually recorded on an object, what a migration does to already-sealed objects, and how Part IX's tamper-evidence mechanism, which this Part's own migration-recording obligation directly serves, distinguishes a legitimate migration from an unexplained mismatch.

20.2 Every Evidence Object carries its own schema-version marker, populated at FIELDS-POPULATING from whatever version of this dataset's own specification was current at construction time, and this marker is itself an ordinary field subject to Part VIII's append-only-then-sealed custody discipline — it is set once, at construction, and is never itself revised in place even when the object's other fields are later migrated to a newer schema version, because the marker's own honest job is to record what version the object was originally constructed under, not what version its current stored representation happens to conform to after migration.

20.3 A migration event, triggered by a Part XXIV amendment that changes the schema — a new required field, a redefined field boundary such as Part IV §4.8's hash-algorithm-change example, a retired field — transforms already-sealed objects' stored representations to conform to the new schema, and this transformation is itself recorded as a distinct, durable event: which objects were migrated, from which version to which, and when, exactly the recorded event Part IX §9.3 and §9.4 already established as the mechanism tamper-evidence's re-verification check consults to distinguish a legitimate stored-content change from an unexplained one.

20.4 Migration is deliberately not automatic-and-immediate for every sealed object the instant a new schema version is adopted; it runs as its own governed process, because Part VIII's custody discipline treats a sealed object's completed fields as closed, and a migration that silently rewrote every sealed object's stored bytes the moment a new schema shipped would be indistinguishable, from any object's own perspective, from the kind of unauthorized bulk alteration Part IX's tamper-evidence exists to detect — the distinction between a legitimate migration and a mass tamper event is entirely the recorded-event discipline in 20.3, and a migration process that skipped recording would forfeit that distinction for every object it touched.

20.5 A specific migration failure mode this Part names directly: a new schema version that changes a field's meaning without changing its name — for instance, redefining what counts as a disqualifying-tier anomaly under Part V's taxonomy without renaming the severity field that carries the classification — would leave already-sealed objects carrying values that were correct under the old meaning but would be misread as though they used the new one, without any migration process necessarily touching those objects' stored bytes at all, since the field's name and storage format are unchanged. This Part requires that any schema change altering a field's meaning, even without altering its storage shape, be treated as a migration event requiring the same recorded-event discipline as a structural change, precisely because the risk of misinterpretation is identical regardless of whether the underlying bytes actually moved.

20.6 A migrated object's relationship to Part XX's own versioning history is queryable in both directions: given a currently stored object, its migration history — every version it has passed through, from original construction to its current representation — is discoverable, and given a specific historical schema version, every object originally constructed under that version, whether since migrated or not, is discoverable. This bidirectional discoverability matters specifically for Part XVI's self-auditing pattern checks, which sometimes need to isolate a finding to a specific schema version's own defect rather than a defect spanning every version, exactly as Part XVI §16.8's own worked example isolated a finding to a specific engine version using an analogous discoverability requirement.

20.7 Not every schema version needs to trigger a migration of every existing object; a schema addition that introduces a new, optional field with no meaningful default for historical acquisitions may instead simply record that objects constructed before the new version lack that field, honestly, rather than forcing a fabricated default value into every historical object merely to make the schema look uniform across versions — Dataset #01's Part VII §7.9's fabrication prohibition applies here with full force, and a migration that populated a new field with an invented, made-up value for objects that never actually had that information observed would be exactly the fabrication that provision condemns, regardless of the administrative convenience a uniform-looking schema would offer.

20.8 A worked example distinguishes a field-addition migration that requires no data change from one that does. Schema version two adds an optional visual-capture-consistency anomaly signal to Part V's taxonomy, applicable only to acquisitions where a visual capture was actually taken per Dataset #01's Part VIII §8.5's provision for missions whose contract calls for one — objects constructed under version one simply lack this signal, honestly, and no migration touches their stored bytes; a downstream reader querying a version-one object per Part XXII understands, from the version marker, that the absence of this specific signal reflects it not yet existing at construction time, not a check that ran and found nothing. Schema version three, by contrast, redefines the raw hash's algorithm per Part IV §4.8's own named example — every future acquisition uses the new algorithm, and existing sealed objects' raw hashes remain under the old algorithm permanently, since Part IV §4.10 already forbids recomputing a hash from anything other than the original bytes at the moment of computation, and those original bytes are not being re-examined merely to adopt a new algorithm; the version marker on each historical object is what lets a future comparison correctly interpret which algorithm produced which stored hash value.

20.9 This Part's own governance under Dataset #01's Part XXIV amendment process, restated as binding rather than merely inherited: a schema version change is proposed, reviewed, and adopted through the same three-step process Dataset #01's Part XXIV specifies for the family as a whole, never unilaterally adopted by a single implementation of `modules/crawl_os/evidence.py` deciding on its own that a field needs to change. This is the mechanical enforcement of Part I §1.3's own citation-not-restatement discipline extended into schema governance specifically: a schema is exactly as coherent as its single-authority governance, and a schema that could be silently altered by whichever implementation happened to touch it next would have lost that coherence regardless of how well every other Part in this dataset specified the schema's intended behavior.

20.10 A closing provision states this Part's relationship to Part I §1.2's own opening framing directly, closing a loop that began at this dataset's very first paragraph: Part I asked how the Evidence Object, once fixed as a schema by Dataset #01, actually survives the interval between construction and consumption across time. This Part's answer to the specific sub-question of schema evolution is that survival is achieved not by freezing the schema forever, which would eventually make the schema unable to represent new, legitimate needs, and not by silently drifting it, which would break every guarantee this dataset has built on the assumption of one coherent, currently-knowable schema, but by versioning it explicitly, migrating deliberately and honestly, and recording every change as durably as this dataset records everything else it considers institutional knowledge.

PART XX FINAL LAW

Every Evidence Object carries a schema-version marker set once at construction and never revised, and a migration — whether it changes a field's storage shape or only its meaning — is a recorded, governed event under Dataset #01's Part XXIV amendment process, never a silent per-implementation change. A new optional field with no meaningful historical default is recorded as honestly absent from older objects, never backfilled with a fabricated value merely to appear uniform, and every migration's recorded event is precisely what lets Part IX's tamper-evidence distinguish legitimate schema evolution from an unexplained, potentially malicious stored-content change.


PART XXI — STORAGE ARCHITECTURE: WHERE EVIDENCE OBJECTS ACTUALLY LIVE

21.1 Several earlier Parts referenced this Part's storage-architecture discipline without specifying it — Part VII §7.6's raw bytes never persisted anywhere the redaction pass has not reached, Part X §10.8's cold-storage tiers for long change-history chains, Part XVI §16.7's storage-layer access logs. This Part collects those references into a coherent architecture: how an Evidence Object is actually laid out in durable storage, what separation exists between different categories of stored data, and why that separation matters for every guarantee this dataset has made about custody, integrity, and honest retrieval.

21.2 The architecture separates three distinct storage regions, not as an implementation preference but as a structural requirement several earlier Parts already depend on. The object store holds sealed Evidence Objects' own field content, addressed by object identifier, subject to Part VIII's custody rules. The derived-index store holds structures built for query performance — indexes by mission, by source URL, by confidence band — that Part XXII's query interface consults, never the object store's own content directly, so that a query-performance optimization can be rebuilt or re-tuned without ever touching the object store's own authoritative content. The audit-log store holds the append-only record of every access and every self-auditing finding, per Part XVI, structurally separate from the objects it describes so that a compromise or corruption of one region does not automatically compromise the other's own independent record of what happened to the first.

21.3 This three-region separation is what makes Part IX's tamper-evidence mechanism meaningful rather than circular: if the object store's own access log lived inside the object store itself, a sufficiently capable unauthorized write could alter both the object's content and the log describing what happened to it in a single action, defeating the very mechanism meant to detect it. The audit-log store's structural independence — a different storage substrate, a different access path, ideally a different privilege boundary — is what lets Part XVI §16.8's worked example actually work: an unexplained write to the object store is discoverable precisely because the audit-log store's own record of writes is not subject to the same compromise.

21.4 The object store's own internal layout follows Part II's lifecycle categories rather than a single undifferentiated pool, because several mechanisms this dataset specifies need to operate over exactly one category without scanning the others — Part XII's retention-expiry review operates over RETAINED-NON-INSTITUTIONAL and CONSTRUCTION-ABORTED objects specifically, never needing to touch the indefinitely retained institutionalized category at all, and a storage layout that partitioned by lifecycle terminal state, rather than storing every object in one pool distinguished only by a status field, makes that review mechanically cheaper and less error-prone than a full-pool scan filtered after the fact.

21.5 Part X's change-history chains are stored as forward-linked records within the object store, each object's supersedes reference resolving to another object's own identifier rather than to a separately maintained chain-index structure, because Part X §10.8 already forbade truncating a chain's logical length for storage-cost reasons, and a chain represented as direct object-to-object links, rather than as a separately maintained index that could itself become stale or inconsistent with the objects it describes, has no such secondary structure whose own staleness could silently break the chain's own correctness.

21.6 Cold storage, named without specification in Part X §10.8, is this Part's own answer to that Part's deferred storage-cost concern: an object whose retention category, per Part XII, keeps it alive but whose access frequency has dropped to near zero — an old link in a long change-history chain, superseded many times over — may be moved to a cheaper, higher-latency storage tier without any change to its identifier, its custody rules, or its queryability under Part XXII; cold storage is a performance and cost optimization applied transparently beneath the object's own logical identity, never a different retention policy or a weakening of any guarantee this dataset has made about the object's content or accessibility.

21.7 The redaction pass's own relationship to storage, extending Part VII §7.6's brief mention into full architectural terms: the raw bytes an acquisition originally produced exist transiently, during FIELDS-POPULATING, in working memory or a similarly transient staging area distinct from all three durable regions named in 21.2, and are never themselves written to any durable store — not the object store, not a backup, not a cache — except insofar as the raw hash, a fixed-size digest rather than the bytes themselves, is what the object store durably retains. This is the storage-level mechanism that makes Part VII's redaction guarantee actually hold: a credential that was present in the raw bytes cannot later be recovered from durable storage by any means, including a storage-layer backup or snapshot, because the raw bytes themselves were never durably stored in the first place, only derived, already-redacted fields and a digest were.

21.8 A specific consequence of 21.7 deserves direct statement because it resolves an apparent tension with Part IX's tamper-evidence mechanism: re-verification, per Part IX §9.2, re-computes a hash over the object's currently stored raw-field content, not over the original transient raw bytes, which no longer exist anywhere durable by the time re-verification runs. This is consistent, not contradictory, because Part IX's own mechanism was always specified as comparing stored content against the stored reference hash, never as re-deriving the reference hash from bytes that Part VII's redaction discipline has already ensured are not retained — tamper-evidence detects a change to what is durably stored, which is exactly the surface an unauthorized alteration would actually touch.

21.9 The audit-log store's own retention follows neither Part XII's category structure nor any expiry policy this dataset otherwise specifies for objects, because an access log's evidentiary value, for exactly the security and integrity purposes Part IX and Part XVI depend on it for, does not diminish with age the way a RETAINED-NON-INSTITUTIONAL object's operational relevance might; the audit-log store is retained indefinitely by this Part's own binding policy, distinct from and not derived from any other retention rule this dataset has established, because a tamper investigation into an old object, per Part IX §9.7's worked scenario, needs an access log old enough to cover that object's own full lifetime, however long ago it was sealed.

21.10 A worked example ties the three-region architecture together for a single storage-layer event. A query, per Part XXII, requests a sealed object; the derived-index store resolves the query to the correct object identifier; the object store returns the object's own field content; Part IX's tamper-evidence check, running as part of this same read, re-computes a hash over the returned content and compares it against the stored reference; the entire read event — who requested it, when, what was returned, whether tamper-evidence passed — is appended to the audit-log store, structurally separate from both the object that was read and the index that resolved the query, exactly the separation 21.3 identified as what makes the audit log's own evidentiary value independent of anything that might compromise the object or index stores.

21.11 A closing distinction addresses replication, a concern this Part has not yet named directly: nothing in the three-region separation forbids each region from being itself replicated across multiple physical or geographic copies for durability, and Part IX's tamper-evidence mechanism composes cleanly with replication precisely because it re-verifies against the stored reference hash regardless of which replica actually served a given read — a replica that has silently diverged from its peers, whether through corruption or an out-of-band write reaching only one copy, is caught by the same re-verification check as a single-region deployment would apply, and the audit-log store's own record of which replica served which read is what would let a future investigation localize a divergence to one specific copy rather than treating the whole storage layer as suspect.

PART XXI FINAL LAW

Storage separates into three structurally independent regions — the object store, holding sealed content; the derived-index store, holding rebuildable query structures; and the audit-log store, holding an indefinitely retained, independently compromised record of every access and finding — so that a compromise of one region does not silently compromise the mechanisms depending on another to detect it. Raw acquisition bytes are never durably stored anywhere, only their hash and their already-redacted derived fields, which is the storage-level mechanism that makes Part VII's redaction guarantee actually hold rather than merely stating an intention.


PART XXII — ACCESS AND QUERY INTERFACE: HOW OTHER ENGINES CONSUME EVIDENCE OBJECTS

22.1 Part III §3.7 and Part VIII §8.4 both referenced this Part's query interface without specifying it: the mechanism by which any Crawl OS engine, or a properly authenticated external estate consumer, actually retrieves a sealed Evidence Object, resolves a provenance chain, or answers a freshness question. This Part specifies that interface — its query shapes, its authorization checks, and its relationship to every retrieval-adjacent mechanism this dataset has already specified in isolation.

22.2 The interface exposes four query shapes, each corresponding to a distinct question a downstream consumer has actually needed to ask across this dataset's own worked examples. Retrieve-by-identifier returns a specific sealed object given its own identifier, the shape Part VIII §8.4's broad read-access provision assumes. Resolve-provenance-chain, given an object's identifier, walks Part III's chain outward to the originating mission and frontier item, or Part X's change-history chain backward through successive supersessions, the shape Part III §3.7's downstream-resolution obligation requires. Query-by-source, given a canonical URL, returns the most recent object in that source's change-history chain, the shape Part XII §12.8's freshness-check mechanics depends on. Query-by-mission, given a mission identifier, returns every object that mission's provenance chain references, supporting a mission's own audit of everything it has acquired.

22.3 Every query, regardless of shape, passes through Part VIII's custody rules and Part IX's tamper-evidence re-verification before returning a result, per Part IX §9.2's query-time trigger already established — the interface does not offer a "fast path" bypassing tamper-evidence for performance reasons, because a query result that skipped re-verification would silently weaken the exact guarantee Part IX exists to provide at precisely the moment a downstream consumer is about to act on the result.

22.4 The interface's authorization check, distinct from and layered beneath the custody rules named in 22.3, verifies that the requesting engine or estate consumer holds standing to read sealed Evidence Objects at all — a check this dataset's own fabric does not itself define, per Part VIII §8.10's custody-versus-authorization boundary, but defers to whatever estate-wide authentication mechanism governs cross-engine access generally, consulted once per query rather than embedded as a parallel authorization scheme this dataset would otherwise be inventing in exactly the kind of boundary violation Part VIII already forbade.

22.5 A query result's own shape mirrors this dataset's own honesty discipline rather than always returning a fully populated object: a retrieve-by-identifier query against a TAMPER-CONFIRMED object, per Part IX §9.8, returns not the object's content but an explicit statement that the object exists, was sealed, and is now excluded from consumption due to a confirmed tamper finding — never a silent not-found response indistinguishable from a query against an identifier that was never valid, and never the tampered content itself presented as though nothing were wrong.

22.6 A resolve-provenance-chain query against a cross-mission reference object, per Part XV, returns both the reference object's own honest inheritance marker and, where the requesting consumer's own access level permits, the full chain through to the original object it references — the interface never silently resolves a reference to look identical to an independently derived object, because Part XV §15.4 already established that this distinction is something a downstream reader needs to be able to observe, and an interface that flattened it away in the name of a simpler response shape would defeat that Part's own design.

22.7 The interface's handling of a REVISED object under Part XIII deserves specific statement because it is the one case where a single identifier could, without this Part's own discipline, return two different answers depending on when the query runs. A retrieve-by-identifier query against the original, disputed object's identifier always returns the original, sealed, unmodified content, per Part XIII §13.3's own custody guarantee — it never silently redirects to the revised record. A separate, explicit field on the returned object states whether a REVISED record exists and, if so, that record's own identifier, letting the requesting consumer decide whether to follow the pointer to the correction, exactly mirroring the discipline Part XVIII §18.10 already established for how a revision propagates to a GK-04 edge without silently overwriting anything.

22.8 Query performance, drawing on Part XXI's derived-index store, is this Part's own concern to specify only insofar as the index's existence must never become a second source of truth diverging from the object store's own authoritative content — every query result is validated against the object store directly, per 22.3's tamper-evidence requirement, and the derived-index store's role is limited to resolving which object identifier a given query shape should retrieve, never supplying the returned content itself. An index that returned content directly, bypassing the object store and its tamper-evidence check, would reintroduce exactly the risk Part XXI §21.3 designed the three-region separation to prevent.

22.9 A worked example demonstrates the interface handling a realistic multi-shape request. The Knowledge Distillation Fabric, per Part XI's package hand-off, needs to verify a package's constituent objects remain trustworthy before producing a knowledge artifact from them — it issues a retrieve-by-identifier query for each object in the package, each passing through custody, tamper-evidence, and authorization checks; for one object, the response includes a REVISED-record pointer per 22.7, and the Distillation Fabric, per its own downstream discretion which this dataset does not govern, chooses to follow the pointer and use the revised judgment rather than the original before producing its artifact — a choice this Part's interface makes possible by surfacing the pointer honestly, without this dataset itself mandating which choice the consuming engine makes.

22.10 This Part's closing provision distinguishes the query interface from Part XIV's replay discipline, since both, in different senses, "return information about a sealed object" and could be conflated without this clarification. The query interface returns the object's own current, sealed state — its fields, its provenance chain, its revision pointer — exactly as this dataset's mechanics constructed and maintain it. Replay reconstructs a different kind of artifact entirely: not the object's state, but the acquisition context that could inform a manual, external repetition of the underlying acquisition. A consumer wanting to know what an object currently says uses the query interface; a consumer wanting to know how to attempt reproducing the acquisition that produced it uses replay, and the two are never interchangeable despite both drawing on the same underlying sealed object as their source.

22.11 A closing note distinguishes this interface's own scope from the Crawl Cortex Supervisor's own consumption of self-auditing findings under Part XVI §16.6: the four query shapes named in 22.2 serve engines and estate consumers asking about specific Evidence Objects, missions, or sources; the Supervisor's own estate-level correlation consumes aggregate self-auditing findings, not individual object queries, and does not route through this Part's interface at all, since Part XVI's findings are themselves durable audit-log-store records rather than Evidence Objects this interface's four shapes were built to serve. A future implementation conflating the two — routing Supervisor correlation through the object-query interface, or exposing individual objects through whatever channel carries aggregate findings — would blur a distinction this dataset has held throughout: an object answers what happened to one acquisition, a finding answers what the fabric's own operation looks like in aggregate, and the two are never the same query answered two ways.

PART XXII FINAL LAW

The query interface exposes four shapes — retrieve-by-identifier, resolve-provenance-chain, query-by-source, query-by-mission — each passing through custody, tamper-evidence, and estate-level authorization before returning a result, never offering a bypass for performance. A tamper-confirmed object returns an honest exclusion statement rather than silence or corrupted content; a revised object always returns its own original, sealed content plus an explicit pointer to any revision, never a silent redirect; and the interface is distinct from Part XIV's replay, which reconstructs acquisition context for external reproduction rather than returning the object's own current state.


PART XXIII — FAILURE TAXONOMY FOR THE INTEGRITY FABRIC ITSELF

23.1 The dataset contract established before Part I named this dataset's own failure taxonomy, distinct from Dataset #01's Part XXII family-wide catalog, as one of this dataset's binding deliverables. This Part fulfills that commitment, collecting and formally naming the specific ways this dataset's own mechanics — as opposed to the acquisition or router failures Dataset #01's own Part XXII already catalogs — can fail, so that a future implementer or auditor has a fixed vocabulary for exactly this dataset's own failure surface rather than needing to infer one from scattered mentions across twenty-two Parts.

23.2 Fabricated-field insertion, named in the dataset contract, occurs whenever any field this dataset's mechanics populate is set to a plausible-looking value not actually derived from what was observed — a hash copied forward per Part IV §4.10's own named prohibition, a confidence value hardcoded per Part VI §6.7's worked shortcut, a negative-evidence generalization per Part XIX §19.7's warning. Every one of these specific instances, scattered across their own Parts, is a member of this single named failure class, and this Part's contribution is naming the class once so a future self-auditing check, per Part XVI, can be built to detect the class generally rather than requiring a separate detector per Part.

23.3 Premature institutionalization occurs when an object is treated as INSTITUTIONALIZATION-ELIGIBLE, per Part II §2.7, without genuinely satisfying Dataset #01's Part XIV criteria as fed by this dataset's own gate-evaluation and confidence mechanics — a lifecycle-skip failure Part II §2.9 already named as structurally prevented by the linear, no-skip lifecycle design, making this failure class one this Part expects self-auditing to find primarily as evidence of a lifecycle-discipline defect in a specific implementation, not as an expected steady-state occurrence.

23.4 Redaction bypass occurs when a credential-shaped value reaches durable storage despite Part VII's mandatory pass, whether through the pass's own under-detection, per Part VII §7.8's dedicated re-sweep, or through a storage-architecture defect per Part XXI §21.7 that durably retained raw bytes the redaction pass never had the opportunity to examine. This failure class carries the highest severity this taxonomy names, because its consequence — a leaked credential — is irreversible in the sense that once persisted and potentially replicated across backups, no later remediation fully undoes the exposure, distinguishing it from every other failure class in this taxonomy, each of which is remediable through this dataset's own dispute, migration, or correction mechanisms.

23.5 Provenance-chain breakage occurs when an object's chain, per Part III, fails to resolve to exactly one mission and one frontier item — the ambiguous-match case Part III §3.3 already specifies is flagged rather than silently resolved, meaning this failure class, correctly implemented, should never actually reach a downstream consumer in unresolved form; its presence in this taxonomy names the failure Part III's own flagging mechanism exists to catch before it propagates, not a failure this dataset's design tolerates once detected.

23.6 Silent object overwrite occurs when a sealed field is modified in place rather than through Part XIII's dispute-driven revision mechanism, violating Part VIII's append-only-then-sealed custody discipline directly — this is the failure class Part IX's tamper-evidence mechanism is specifically built to detect, and its presence in this taxonomy names, in this dataset's own vocabulary, exactly the event Part IX §9.7's worked scenario walked through as a TAMPER-CONFIRMED finding's underlying cause.

23.7 Confidence-scale duplication occurs when the fabric's own construction logic computes a confidence-relevant judgment independently of DRK-03's CONNECT call rather than composing with it, per Part VI §6.7's own named worked shortcut — the class this dataset's Part I §1.2 named as the single most consequential boundary violation an elaborating dataset could commit, restated here in operational, detectable terms: any code path in `modules/crawl_os/evidence.py` that maps anomaly signals to a confidence-relevant output without a corresponding call to DRK-03's own logic is, by this taxonomy's definition, an instance of this failure class regardless of how the output is used downstream.

23.8 Orphaned GK-04 edge payload occurs when Part XVIII's edge-content construction completes but the resulting payload is never actually handed to GK-04's writer once that writer exists — a failure class this dataset can name today, per the dataset contract's own foresight, but cannot yet observe in practice, since the writer's absence means no payload has ever been in a position to become orphaned; this taxonomy entry exists specifically so that once the writer is built, an implementer already has the failure class named and ready to build a detection check against, rather than discovering the need for one only after orphaned payloads have already accumulated.

23.9 A worked example demonstrates the taxonomy's use as a diagnostic tool rather than a merely descriptive list. Part XVI's self-auditing sweep surfaces a pattern-check finding: a cluster of objects whose confidence values cluster suspiciously tightly around a single round number, inconsistent with the variation Part V's anomaly taxonomy would predict given those objects' actually recorded signal diversity. Consulting this Part's taxonomy, the finding is classified as a probable instance of confidence-scale duplication per 23.7, specifically the "obviously going to return that value anyway" shortcut Part VI §6.7 already named — the taxonomy's specific, named failure class is what lets the investigation move directly to checking the suspected code path for a missing DRK-03 call, rather than beginning an unfocused investigation across every mechanism this dataset specifies.

23.10 This Part's closing provision states its own relationship to Dataset #01's Part XXII family-wide catalog directly, since a future reader auditing a failure could reasonably wonder which taxonomy governs. Dataset #01's Part XXII catalogs failures visible at the family level — Clean-Markdown, Router-Favorite, Plane-Collapse — many of which could, in principle, manifest through this dataset's own mechanics as one specific instantiation among several possible ones across the family. This Part's taxonomy is strictly narrower and more operationally specific: every entry here is a failure that can only occur inside this dataset's own construction, custody, or composition mechanics, and a failure that could equally well occur in the Fetch Runtime Fabric or the Browser Interaction Fabric belongs to Dataset #01's own catalog, not this Part's, exactly as this dataset's own ownership boundary, established in its contract, has held throughout.

23.11 A closing note addresses severity ordering across this taxonomy's remaining six classes, since 23.4 already established redaction bypass as the single highest-severity entry: premature institutionalization and confidence-scale duplication rank next, because both propagate a defect into downstream artifacts — a package per Part XI, an edge payload per Part XVIII — that other engines may already have consumed before Part XVI's self-auditing catches the underlying cause, requiring Part XIII's dispute mechanism to unwind the propagation after the fact. Provenance-chain breakage and silent object overwrite rank below those two, because Part III's own flagging and Part IX's own tamper-evidence respectively are designed to catch each before it reaches a downstream consumer at all, making their typical real-world instances near-misses caught by design rather than defects that actually propagated. Orphaned GK-04 edge payload ranks lowest of the seven, not because the underlying consequence is minor but because, per 23.8, the class cannot yet manifest at all while the writer remains unbuilt — its ranking reflects current inapplicability, not a judgment that it would stay least severe once the writer exists.

PART XXIII FINAL LAW

This dataset's own failure taxonomy — fabricated-field insertion, premature institutionalization, redaction bypass, provenance-chain breakage, silent object overwrite, confidence-scale duplication, and orphaned GK-04 edge payload — names failures specific to this dataset's own construction, custody, and composition mechanics, distinct from Dataset #01's family-wide catalog. Redaction bypass carries this taxonomy's highest severity because it is the one failure class no later remediation can fully undo; every other class is remediable through this dataset's own dispute, migration, or correction mechanisms once detected.
