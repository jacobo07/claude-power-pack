PART I — MISSION: QUALIFICATION PRECEDES INTERPRETATION

1.1 A test runner does not produce verdicts. It produces data. The distinction sounds academic until one notices that every mechanism in this estate which turns a runner's data into a verdict is doing arithmetic on an assumption, and that the assumption is almost never written down. The assumption is that the environment in which the runner executed was capable of producing a meaningful result at all. When that assumption holds, an exit code is informative and the interpretation is trivial. When it does not hold, the exit code is still produced, it is still non-zero, it still carries a stack trace and a file and a line, and it is still entirely uninformative about the product — and nothing in the output distinguishes the two cases. The runner cannot tell you whether it was in a position to be believed. That determination belongs to a layer above it, and this dataset is that layer.

1.2 The mission of environment qualification is therefore stated as an ordering rather than as a capability. Qualification precedes interpretation. A result may be recorded the instant it is produced, but it may not be interpreted — may not be assigned a verdict, may not be attributed to a component, may not be routed to an engineer, may not be entered as evidence for any claim — until the environment that produced it has been qualified. The ordering is not a preference about workflow tidiness. It is a constraint on the interpretation function itself, because the environment is an argument to that function and evaluating a function against an unsupplied argument does not yield a wrong answer; it yields a meaningless one that is indistinguishable from a right one.

1.3 The founding observation is empirical and belongs to this estate's own body. Two repositories in the audited set — one carrying one hundred and twenty-two functional test files, the other carrying twenty-seven — cannot be compiled. The build command terminates with the diagnostic that a dependency is not locked, and it names them: the JSON library, the request client, the database adapter, the driver beneath the adapter, and the process-broadcast layer. Not one test in either repository has ever been executed under audit. The runner produced errors. The errors were real, reproducible, and specific. And the correct record of that state is not that the tests failed, because no test ran; it is that the environment could not be qualified and the one hundred and forty-nine files are, in the only honest phrasing available, in a state of suspended determination — neither passing nor failing, because nothing has yet observed them.

1.4 The seduction of the wrong record is worth naming precisely, because it is not a lazy error and it is not made by careless people. It is made because the runner's output looks exactly like a failure. It has the texture of a failure. It arrives through the same channel, in the same format, with the same urgency, and a pipeline that classifies by texture rather than by attribution will file it under failure, and the filing is a single word in a single field and nobody will ever look at it again. The word is what the next agent reads. The next agent does not re-derive the environment; it inherits the classification, and it acts on it, and the action is to open the product and begin repairing.

1.5 The boundary of this dataset must be drawn as sharply as its mission, because a qualification layer that expands into execution or into diagnosis becomes a second implementation of organs the axis already has. This dataset does not run the suites; the execution sibling does that. It does not enumerate the repositories or classify their domains or map their executable surfaces; the reality sibling does that, upstream, and supplies its findings as predictions. It does not compute risk tiers, blast radius, or reversibility; the decision layer owns those and is not to be forked. It does not score deliverables against done-gates; the output-contract layer owns that. It has one product: an EnvironmentRecord carrying a qualification state, the evidence that licensed it, and the blockers that prevented anything stronger — and a ceiling, derived from that state, on the strength of any verdict that a result from that environment is permitted to carry.

1.6 The ceiling is the operative mechanism and it is worth stating before any of the machinery that produces it. A result obtained in an environment that is not QUALIFIED cannot yield a verdict stronger than UNVERIFIED, BLOCKED, or ENVIRONMENT-INVALID. It cannot yield PROVEN, because nothing was demonstrated. It cannot yield FAILED attributed to the product, because the failure has not been shown to belong to the product, and attribution without demonstration is the specific error this axis exists to eliminate. The ceiling applies mechanically, without argument, and it applies regardless of how confident the observer is about what the failure means, because confidence about what a failure means, in the absence of a qualified environment, is precisely the faculty that has been observed to be wrong.

1.7 There is an asymmetry in the cost of the two possible mistakes, and the asymmetry is why the ordering is enforced rather than encouraged. Qualifying an environment that did not need qualification costs a few commands: a version probe, a lock inspection, a service reachability check. The cost is measured in seconds and is paid by the party best positioned to pay it, at the moment when the context is fresh. Interpreting a result from an environment that was never qualified costs an engineer's attention, aimed at code that is correct, for as long as it takes them to make the harness stop complaining — and because the engineer is competent, it will eventually stop complaining, and what will have happened is that correct behaviour was modified until a broken environment was satisfied. The first cost is bounded, visible, and paid once. The second is unbounded, invisible, and paid by someone who did not create it.

1.8 The ordering must be structural rather than remembered, and the reason is the same reason every rule in this corpus is structural. At the moment a red result arrives, the psychological availability of the environment as an explanation is near zero. The stack trace points at a file. The file is in the product. The product is what the agent has been working on. Every cue in the situation directs attention inward, toward the code, and the question of whether the machine was capable of running the code at all is a question that nothing in the situation prompts anyone to ask. A rule that depends on someone remembering to ask an unprompted question under time pressure is not a rule; it is a hope. The qualification gate must therefore fire before the result is presented, not after, and it must be capable of withholding the result's interpretation entirely.

1.9 The mission has a second-order consequence that is easy to miss and is arguably the larger prize. An estate that qualifies its environments accumulates a description of what it takes to build and exercise itself, and that description is the single most valuable artifact a repository can carry, because it is the thing that a new machine, a new contributor, a new agent, or a continuous-integration job needs in order to reach parity, and it is the thing that is almost never written down. Every qualification pass is simultaneously an act of verification and an act of documentation, and the documentation is derived from execution rather than from memory, which makes it the only kind that does not silently rot.

1.10 The dataset therefore stands between the reality layer, which predicts from a read of files that an environment will not build and names the reason, and the execution layer, which cannot honestly interpret anything until this layer has spoken. It converts a prediction into a verdict by executing, it converts an execution into a record by qualifying, and it converts a record into a constraint by imposing a ceiling. Everything in the nineteen Parts that follow is machinery for producing that record correctly, defending it against the ways it can be faked, and ensuring that when it is absent, the honest state is UNKNOWN rather than the comfortable one.

PART I FINAL LAW

A runner reports data, not verdicts. The environment is an argument to the interpretation function, and evaluating that function against an unsupplied argument yields a meaningless answer indistinguishable from a correct one. Qualification therefore precedes interpretation, structurally and not by convention, because at the moment a red result arrives nothing in the situation prompts anyone to ask whether the machine was capable of running the code at all. An unqualified environment caps every verdict at UNVERIFIED, BLOCKED, or ENVIRONMENT-INVALID.


PART II — THE ATTRIBUTION LAW AND THE COST OF GETTING BLOCKED VERSUS FAILED BACKWARDS

2.1 The Attribution Law is stated once, in the strongest available form, and it admits no exception anywhere in this corpus. A failure observed in an unqualified environment may not be attributed to the product. The law does not say that such a failure is probably environmental, or that it should be treated with caution, or that a note should be appended recommending further investigation. It says that the attribution is unavailable — that the inference from "the command exited non-zero" to "the product is defective" is not licensed, and that making it anyway is not a judgement call made under uncertainty but a category error made under the appearance of evidence.

2.2 The law's force comes from a fact about competent engineers that is almost never articulated, and articulating it is what turns the law from a caution into a prohibition. When an engineer is dispatched to repair code that is not broken, they do not stop when they discover it is not broken. They keep going, because the harness is still complaining, and the harness complaining is the definition of the task they were given. They are skilled, they are persistent, and they will eventually find a modification that makes the complaint stop. That modification is, by construction, a change to correct behaviour made in order to satisfy a broken environment — and it is undetectable afterward, because the pipeline is now green, the ticket is closed, the engineer is competent and reports success, and the only artifact of the entire episode is a product that used to be right and now is not. The damage is not that time was wasted. Time being wasted would be the good outcome. The damage is that correctness was destroyed, silently, by a competent person doing exactly what they were told.

2.3 The boundary between BLOCKED and FAILED was live in four distinct forms in the audit that founded this corpus, and each is a different shape of the same law. In the two Elixir repositories, the compilation could not proceed because the dependencies were not locked, and the honest verdict is BLOCKED with the blocker recorded verbatim. In the Java estate, the prebuilt test classes carry a bytecode version that a newer runtime produces and an older one refuses, and building on the older runtime terminates with a diagnostic that the class was compiled by a more recent version — which is a statement about two pieces of the environment disagreeing, and not a statement about the plugin's logic at all. In the audit of a Python repository, a single test wrote outside its sandbox to the root of the host drive and thereby broke a sibling suite, and the product logic was correct. In the Wii repository, the drivers include engine headers and link the whole game and require a cross-compiler that does not exist on the host, and no arrangement of the host will change that.

2.4 Four situations, four wrong records available, and each wrong record dispatches a different kind of waste. Recording the unlocked dependencies as FAILED sends an engineer into one hundred and forty-nine test files looking for the defect that broke the build, and there is no such defect, and they will not find it, and the search terminates only when someone finally reads the diagnostic. Recording the bytecode mismatch as FAILED sends an engineer into a plugin that compiles and passes perfectly well one runtime version away. Recording the host-pollution case as FAILED sends an engineer into logic that is correct — and this is the case where the competence trap closes, because the logic is small enough to be modified until the sibling suite stops breaking, and modifying it is the precise wrong action. Recording the hardware requirement as FAILED indicts a codebase for the sin of needing the machine it was written for.

2.5 The inverse error exists and must be caught by the same machinery, or the law becomes an escape hatch rather than a constraint. Recording a genuine product FAILED as BLOCKED is cheaper in immediate cost — nobody damages correct code — but it is corrosive in a way that compounds, because it converts a defect into an environmental complaint and files it in a queue that nobody works, and the defect ships. An estate in which BLOCKED is the comfortable answer will discover that its BLOCKED list grows monotonically and its defect rate does not fall. The law is therefore two-sided: an unqualified environment forbids a product attribution, and a qualified environment forbids an environmental excuse. Both directions require the same thing, which is the qualification record.

2.6 The Python case deserves individual treatment because it is the canonical fault-attribution instance in this estate's history and because it is the one where every intuition points the wrong way. The suite reported two hundred and three passing results against two hundred and four. One result was red. A red result inside a suite, on a repository whose environment builds and whose runner collects, has every appearance of a product defect, and there is no environmental diagnostic to warn anyone off — the build succeeded, the deps resolved, the collection worked. The failure was a test writing to the host drive root, and the pollution defeated a sibling suite. The product logic was correct. Had the repair been aimed carelessly, correct code would have been changed to satisfy a broken fixture, and the fixture would have remained broken, and it would have gone on breaking siblings.

2.7 What that case establishes is that the Attribution Law cannot be discharged by a build check alone. An environment that compiles is not thereby qualified, because qualification includes the harness's containment properties, and a harness that writes outside its sandbox is an unqualified environment that presents as a qualified one. The engine described in Part III therefore does not terminate at the build gate; it continues into the containment gate, and the containment gate is the one that catches this class, and it catches it independently of whether any test failed — which matters enormously, because on the run where the pollution does not happen to collide with a sibling, nothing goes red at all and the environment silently degrades the next run instead.

2.8 The operational form of the law is a verdict ceiling and a one-way ratchet, and both are mechanical. The ceiling: a result from a non-QUALIFIED environment may be recorded as UNVERIFIED, BLOCKED, or ENVIRONMENT-INVALID, and as nothing stronger. The ratchet: a verdict of FAILED may always be demoted upon the later discovery that the environment was unqualified, and the demotion requires no new execution, because it is a correction of an inference that was never licensed. A verdict of BLOCKED may be promoted only by a new execution in an environment that has been qualified, and never by argument, and never by the passage of time, and never by an agent's conclusion that the blocker was probably not important. The asymmetry encodes the cost asymmetry directly: the cheap correction is always available, and the expensive one always requires the work.

2.9 The law extends to every consumer of the record and not merely to its producer, and the extension is what makes it survive the second reading. An agent that receives a FailureRecord whose environment reference is empty must treat the attribution as absent rather than as product, because an unpopulated environment field does not mean the environment was fine; it means nobody looked. A claim whose supporting evidence carries no environment reference is not weakly supported. It is uninterpretable, and the correct disposition is to reject the evidence rather than to discount it, because discounting implies a scale on which the evidence sits, and this evidence sits on no scale at all.

2.10 The final provision closes the loop that would otherwise let the law be defeated by good intentions. No product repair may begin until the attribution names the product, and the attribution names the product only when a discriminating experiment has been run — an experiment which changes exactly one thing and observes whether the failure survives. Repairing on the basis of a plausible story about what the failure means is the behaviour the law prohibits, and it is prohibited regardless of how good the story is, because the story is reasoning, reasoning is weak evidence by the ontology's own tier system, and weak evidence has never in the history of this estate been sufficient to license a change to correct code.

PART II FINAL LAW

A failure observed in an unqualified environment may not be attributed to the product. Recording BLOCKED as FAILED dispatches a competent engineer to repair correct code, and because they are competent they will eventually succeed at making the broken harness stop complaining — which means correctness was destroyed, silently, by someone doing exactly what they were told. The ratchet is one-way: FAILED may always be demoted on discovering an unqualified environment; BLOCKED is promoted only by a new execution in a qualified one.


PART III — THE ENVIRONMENT QUALIFICATION ENGINE: WHAT IS CHECKED, IN WHAT ORDER

3.1 The engine is a short-circuiting pipeline of gates, ordered so that the cheapest and most-blocking checks run first, and constructed so that the first gate to fail determines the verdict and terminates the pass. The ordering is not aesthetic. Each gate is a precondition for the gates beneath it, and running a downstream gate against an environment that failed an upstream one produces a result that looks like information and is noise: probing whether a database is reachable in an environment whose dependencies will not resolve tells you nothing you can act on, because the resolution failure will have to be fixed first and fixing it may change everything downstream of it. A gate that fails does not merely fail. It renders every subsequent gate UNKNOWN, and UNKNOWN is what they are recorded as — never skipped, never assumed, never passed by omission.

3.2 Gate zero is the host census, and it is the only gate that cannot fail. It records the operating system, the architecture, and the processor class, and it gates nothing by itself, because a host is not wrong; a host is simply what it is. Its purpose is entirely downstream: it is the key against which every subsequent finding is scoped, it is the field that makes a result comparable across machines, and it is the field whose absence makes a reproducibility claim unfalsifiable. A record without a host census is not a record with one missing field. It is a record about no particular machine, and a result about no particular machine is a result about nothing.

3.3 Gate one asks whether the toolchain exists at all: is there a compiler, is there a runtime, is there a build orchestrator, and are they invocable from the position the engine occupies. This gate catches the least interesting class of failure and it catches it in milliseconds, which is exactly why it runs first. Its subtlety is that presence is not the same as reachability: a runtime installed on the host but absent from the resolution path of the process that must invoke it is, for every purpose the engine cares about, not installed. The gate therefore probes by invocation and never by inspection, because inspection answers a question about the disk and invocation answers the question that matters.

3.4 Gate two is toolchain version compatibility, and it is where the engine earns most of its value, because this is the gate that separates a toolchain mismatch from a product defect and the two are indistinguishable to everything downstream. The gate reads the requirement from the artifacts wherever the artifacts carry it, and only falls back to declarations when they do not, because a declaration is a statement of intent and an artifact is a statement of fact. The Java case is the exemplar: the prebuilt test classes carry a bytecode version in their own header, that version is produced by one major runtime and refused by its predecessor, and an older runtime attempting to consume them terminates with a diagnostic that says exactly this and is nonetheless routinely read as a build failure of the product. The requirement was in the artifact the whole time.

3.5 Gate three is dependency resolvability, and it asks three questions that must not be collapsed. Does a lock exist. Does that lock cover every declared dependency. Is resolution satisfiable without reaching outward for something that might have moved. The Elixir case terminates here, and it terminates loudly: the build orchestrator itself refuses, naming five dependencies as unlocked, and the refusal is not a symptom of a defect further in — it is the whole finding, and everything downstream of it is unknowable until it is resolved. A repository that cannot pin what it builds against is a repository whose build is a function of wall-clock time, and a build that is a function of wall-clock time cannot be qualified in any sense that survives tomorrow.

3.6 Gate four is build reachability: given a resolved dependency set and a compatible toolchain, does the thing compile. This gate is deliberately narrow and it is deliberately not the last gate, because the estate's most expensive standing error is the treatment of a successful build as a proxy for a qualified environment. A build proves that symbols resolved and types checked. It proves nothing about whether the services the code will reach for at runtime exist, and it proves nothing about whether the harness that will exercise the code is contained. Both of those are environment properties, both of them are checked after the build, and an engine that terminates at a green build has qualified an environment for compilation and for nothing else.

3.7 Gate five is external service availability, and it converts the manifest's implicit declarations into explicit reachability probes. A database driver in the dependency set is a service declaration whether or not anyone wrote one down; a queue client is a service declaration; an object-store binding is a service declaration. The gate enumerates them, probes each, and records the provisioning mechanism and the isolation mechanism alongside the reachability result — because a service that is reachable but shared is a different environment from a service that is reachable and isolated, and results obtained against the first are order-dependent in ways that results obtained against the second are not.

3.8 Gate six is the containment gate, and it is the one most engines omit and the one that would have caught the estate's canonical fault-attribution case. It asks whether the harness is contained: whether the runner writes only inside its allocated temporary space, whether it mutates global configuration, whether it leaves residue that a subsequent run will read. The gate is implemented as a differential observation rather than as a code inspection — the filesystem and the relevant global state are observed before the run and after it, and any write outside the sandbox is a violation, and it is a violation independently of whether any test went red. That independence is the whole point: on the run where the pollution does not collide with a sibling, nothing fails, and an outcome-driven detector sees a clean pass while the environment silently degrades for the next one.

3.9 The short-circuit semantics are strict and they are what keep the record honest. The first gate that fails terminates the pass, sets the qualification state, and records the blocker verbatim — verbatim meaning the diagnostic exactly as the tool emitted it, not a paraphrase of it, because the paraphrase is what destroys the diagnostic value at the exact moment a future reader needs it most. Every gate downstream of the failure is recorded as UNKNOWN. None of them is recorded as passing, none as failing, and none is silently omitted, because an omitted gate reads as an absent concern and an UNKNOWN gate reads as an open question, and the difference between those two readings is the difference between an estate that closes its gaps and one that cannot see them.

3.10 The engine's output is a single EnvironmentRecord: a qualification state, the language and runtime and toolchain versions actually observed rather than declared, the lock state, the enumerated services with their provisioning contracts, the host census, and the blockers verbatim. It has one default, and the default is UNKNOWN. The engine does not emit QUALIFIED because nothing complained; it emits QUALIFIED only when every gate has been affirmatively observed to pass, with the command and the output retained. This is the same law the constitution applies to test results, applied to the engine that qualifies them, and it applies for the same reason: absence of a failure signal is not evidence of success, and an engine that concluded otherwise about itself would be the first thing in this axis to be wrong.

PART III FINAL LAW

The engine is a short-circuiting pipeline: host census, toolchain presence, toolchain version compatibility, dependency resolvability, build reachability, service availability, harness containment. The first failing gate sets the state, records the blocker verbatim, and renders every downstream gate UNKNOWN — never skipped, never assumed, never passed by omission. The containment gate fires independently of test outcome, because pollution that does not collide with a sibling on this run degrades the next one silently. The default state is UNKNOWN; QUALIFIED is earned by affirmative observation, never by the absence of a complaint.


PART IV — THE SIX QUALIFICATION STATES

4.1 The engine emits one of six states, and the enumeration is exhaustive by construction so that no environment can fall outside it and be rounded to the nearest convenient neighbour. The six are QUALIFIED, PARTIALLY QUALIFIED, BLOCKED, HARDWARE REQUIRED, UNSUPPORTED, and UNKNOWN. A binary would be cheaper and would be catastrophic, for the same reason a binary verdict is catastrophic in the constitution: it collapses conditions that license entirely different actions, and once they are collapsed at the point of measurement no downstream consumer can recover the distinction no matter how careful it is.

4.2 QUALIFIED means every gate was affirmatively observed to pass, with commands and outputs retained, on the host recorded in the census. It licenses full interpretation: a result from a QUALIFIED environment may carry any verdict the evidence supports, including FAILED attributed to the product, because the precondition for that attribution has been met. It is the only state that licenses a product attribution, and it is therefore the state an agent is most motivated to reach and the one whose evidence must be scrutinized hardest. QUALIFIED is bound to the environment hash and expires when the hash moves; it is a claim about a machine at a moment, and it is never a permanent property of a repository.

4.3 PARTIALLY QUALIFIED is the honest state of most real repositories at most moments, and it exists because the alternative is a lie in one direction or the other. Some surfaces are executable and others are not, and forcing that into a binary rounds either the executable part down to nothing or the blocked part up to fine. In the audited estate this is the state of the Java repository read as a whole: two modules built and ran green on the newer runtime, one thousand five hundred and ninety-six results in one and ten in the other, while the estate around them carries ninety build descriptors of which one wires an in-memory server for headless plugin logic. Partially qualified is not a failure of the engine. It is the engine reporting a heterogeneous estate accurately, and the field that makes it useful is the enumeration of which surfaces fall on which side.

4.4 BLOCKED means execution was attempted and was prevented by something the environment could in principle supply but did not. It is the state of the two Elixir repositories, and its defining property is that the remedy is an environment action: commit the lock, provision the database. The blocker is recorded verbatim, and the resolution actions attempted are recorded alongside it, and both are mandatory — a BLOCKED verdict carrying no blocker string is not a weak record, it is an inadmissible one, and the engine rejects it, because BLOCKED without a named blocker is indistinguishable from an agent declining to do the work.

4.5 HARDWARE REQUIRED is a separate state and not a species of BLOCKED, and the separation is load-bearing rather than taxonomic. BLOCKED says the host could supply this and has not. HARDWARE REQUIRED says the host cannot supply it, at any expenditure of effort, because what is missing is not a package but a machine. The Wii repository is the case: the driver files include engine headers, they link the whole game, and they require a cross-compiler targeting a console. There is no arrangement of the development host that makes those files host-runnable, and a backlog item that says provision the environment is a backlog item that will never close. Separating the states is what prevents an impossibility from sitting in a queue forever, quietly consuming the attention of everyone who reads past it.

4.6 UNSUPPORTED is a policy state rather than a technical one, and it is the only state that an agent may not assign. It means the estate has decided not to provision this environment — that the cost is understood, has been weighed, and has been declined. Because it is a decision rather than an observation, it requires an owner, an expiry, and a compensating control, exactly as a waiver does, and it is subject to the same discipline: an UNSUPPORTED that has been silently renewed by the passage of time is not a decision, it is a policy nobody chose. The state exists so that a genuine, considered non-provisioning is expressible without being disguised as a technical blocker, which is what happens to it otherwise.

4.7 UNKNOWN is the default and it is the state every environment occupies until an engine pass says otherwise. It is not a diagnostic failure and it is not an admission of incompetence; it is the accurate description of an environment nobody has looked at. The state that must never be assumed away is precisely this one, because the assumption is so cheap and so comfortable: the build worked yesterday, the machine has not obviously changed, the environment is probably fine. Probably fine is UNKNOWN wearing a confident expression. The engine treats an absent record and an UNKNOWN record identically, and it treats both as a hard ceiling on interpretation.

4.8 The transitions between states are governed by evidence and not by argument, and enumerating them is what makes the state model executable rather than descriptive. UNKNOWN moves to any other state by an engine pass. BLOCKED moves to QUALIFIED or PARTIALLY QUALIFIED only by a new pass in which the blocker is observed to be absent — never by a claim that it was resolved, and never by the passage of time. QUALIFIED moves to UNKNOWN automatically whenever the environment hash changes, which is a silent invalidation and is the subject of a later Part. HARDWARE REQUIRED moves only when the hardware or a cross-toolchain appears, or when the product surface is restructured so that the untestable coupling is removed — which is the single case in this dataset where the correct remedy touches the product rather than the environment. UNSUPPORTED moves only by an owner's decision, recorded.

4.9 The mapping from qualification state to permissible verdict is the mechanism through which this dataset governs the rest of the axis, and it is a ceiling rather than an assignment. QUALIFIED → any verdict the evidence supports. PARTIALLY QUALIFIED → any verdict for the surfaces observed to be executable, and UNVERIFIED or BLOCKED for the rest, never a blanket verdict across both. BLOCKED → BLOCKED, and nothing else. HARDWARE REQUIRED → UNVERIFIED for the coupled surfaces, and never FAILED, because the coupling is a property of the domain and not a defect. UNSUPPORTED → WAIVED-WITH-LIABILITY, with the liability named. UNKNOWN → UNVERIFIED, universally, regardless of what any runner emitted, because a result from an unexamined environment is a result whose meaning has not been established.

4.10 The final prohibition is the one that keeps the whole state model from decaying into vocabulary. No state may be assigned by reasoning. Each state requires a command and the output that command produced, retained, and referenced from the record. An agent that concludes an environment is qualified because it can see no reason it would not be has produced a hypothesis, and the ontology's own tier system classifies a hypothesis as weak evidence, and weak evidence cannot license the one state that unlocks product attribution. The states are observations. The moment they become opinions, the ceiling they impose becomes negotiable, and a negotiable ceiling is not a ceiling.

PART IV FINAL LAW

Six states, exhaustive by construction: QUALIFIED licenses product attribution and nothing else does; PARTIALLY QUALIFIED is the honest state of most real repositories and must enumerate which surfaces fall on which side; BLOCKED means the host could supply it and has not, and requires a verbatim blocker or it is inadmissible; HARDWARE REQUIRED means the host cannot supply it at any effort, and separating it from BLOCKED is what stops an impossibility from sitting in a queue forever; UNSUPPORTED is a decision, needing an owner and an expiry; UNKNOWN is the default, and no state may be assigned by reasoning.



PART V — LANGUAGE, RUNTIME, AND COMPILER VERSION QUALIFICATION

5.1 Three things are routinely collapsed into a single number and they are not the same thing, and the collapse is responsible for a specific and expensive class of misattribution. The language version is a property of the source: which syntax and semantics the code is written against. The runtime version is a property of the machine: which interpreter, virtual machine, or execution substrate will actually run it. The toolchain version is a property of the transformation: which compiler, build orchestrator, or bundler stands between them. A repository can be internally consistent on any two of these and broken on the third, and the diagnostic that surfaces when it is broken names the third while the reader, primed by the source they are looking at, attributes it to the first.

5.2 Each of the three exists in three forms, and the qualification engine must hold all nine cells rather than the three it is tempted to hold. There is the declared version, written in a manifest or a configuration file, which is a statement of intent. There is the observed version, obtained by invoking the tool and reading what it says about itself, which is a statement of fact about this host at this moment. And there is the required version, which is what the artifacts on disk actually demand in order to be consumed — and this third form is the one that goes missing, because it is not written anywhere a human would look for it. It is encoded in the artifacts themselves, and reading it requires knowing to look.

5.3 The Java case in the audited estate is the exemplar and it is worth stating precisely, because it is the clearest instance in the corpus of a requirement that lived in an artifact rather than in a declaration. The prebuilt test classes carry a bytecode format version in their own headers. That format is emitted by one major runtime generation and is refused by its predecessor, and the predecessor's refusal takes the form of a diagnostic stating that the class was compiled by a more recent version. Nothing in the source declares this. Nothing in the human-readable build configuration necessarily surfaces it. The requirement is a property of compiled bytes sitting on the disk, and an engine that reads only declarations will not find it, and will instead present the reader with a build failure whose text is about the product's classes.

5.4 The general rule extracted from that case is that requirements are read from artifacts wherever artifacts carry them, and declarations are the fallback rather than the source of truth. This is the reality-supremacy doctrine applied to versions: the disk outranks the manifest, and where the two disagree, the manifest is not merely overruled but is itself a finding, because a manifest that disagrees with the artifacts it describes is a manifest that has drifted and will mislead the next reader exactly as it misled this one. Compiled artifacts, packaged distributions, and pinned binaries all carry version information in their own structure, and the engine's obligation is to extract it rather than to trust the file that claims to describe it.

5.5 Version pinning is where declarations earn back their standing, and the distinction between a pin and a preference is the distinction between an environment that can be reproduced and one that cannot. A pin names an exact version and admits no substitution. A preference names a floor, a ceiling, or a family, and admits whatever the resolver finds acceptable today, which is a different thing from what it will find acceptable next quarter. A repository whose toolchain is preferred rather than pinned has an environment whose identity is determined at resolution time, and an environment whose identity is determined at resolution time cannot be qualified in any sense that outlives the qualification pass, because the thing that was qualified is not the thing that will be used.

5.6 The ambient-selection defect is the most common way a version qualification is silently wrong on a developer host, and it is invisible precisely because it is a property of the invoking process rather than of the machine. A host may carry four runtimes. Which one answers to a bare invocation is determined by resolution order, by an environment variable, or by a shim that consults a file the agent has never read. The qualification engine therefore records the version obtained by invoking the tool the way the build invokes it, from the position the build occupies, and never the version obtained by asking the machine what it has installed. Those two questions have different answers on every non-trivial host, and the second one is not the question.

5.7 The version record therefore carries, per tool: the declared version and its source file, the observed version and the exact invocation that produced it, the required version and the artifact it was extracted from, and the resolution mechanism that selected the observed one. Four fields where most systems carry one. The fourth is the one that turns a mismatch from a mystery into a fix, because a mismatch between required and observed is diagnosed entirely differently depending on whether the observed version was selected by a pin, by a shim, or by resolution-path accident, and only the fourth field distinguishes them.

5.8 The compatibility relation between versions is asymmetric, and the asymmetry is what makes the diagnosis tractable. A newer runtime will generally consume artifacts produced by an older toolchain; an older runtime will generally refuse artifacts produced by a newer one. This means a version failure has a direction, and the direction is recoverable from the diagnostic, and the recovery is the whole of the remedy. The Java case demonstrates it exactly: the failure was one-directional, the audit resolved it by moving to the newer runtime, and the suites then ran green. Had the asymmetry run the other way — had the artifacts required an older runtime that the host no longer carried — the remedy would have been different and the environment would have been closer to genuinely BLOCKED.

5.9 The drift detector for versions is the comparison of the current pass's observed versions against those recorded in the last QUALIFIED pass, and it must run on every execution and not on a schedule. A runtime that moved beneath a repository between two passes is the most common cause of a suite that was green last week and is red today with no code change, and it is the case in which every instinct points at the diff and the diff is empty. The detector's value is that it makes an empty diff and a moved runtime a diagnosable pair rather than a mystery, and it converts what presents as a flaky suite into what it actually is, which is an environment that silently invalidated a prior result.

5.10 What a version mismatch licenses is narrow and must be stated so that it is not exceeded. It licenses a qualification state of BLOCKED, a blocker recorded verbatim, and a resolution action aimed at the toolchain. It licenses nothing whatever about the product. It does not license a finding, a defect record, a backlog item against the code, or a modification to any source file, and an agent that responds to a version mismatch by changing product source has performed exactly the action the Attribution Law prohibits — repairing correct code until a broken environment stops complaining — and has done it in the case where the environment was announcing its own brokenness in plain text.

PART V FINAL LAW

Language version, runtime version, and toolchain version are three things, each existing in three forms — declared, observed, and required-by-artifact — and the third form is the one that goes missing, because it lives in compiled bytes rather than in any file a human would read. The disk outranks the manifest here as everywhere: a bytecode header stating which runtime generation produced it is a requirement, and an older runtime's refusal to consume it is a statement about the environment and nothing at all about the product's logic.


PART VI — TOOLCHAIN COMPATIBILITY RESOLUTION BEFORE DECLARING A REPOSITORY BROKEN

6.1 Between the observation that a build failed and the verdict that a repository is broken, there is a mandatory step, and skipping it is the most common route by which a qualification error becomes a product finding. The step is toolchain compatibility resolution: an explicit, bounded, recorded attempt to determine whether the failure is explained by a toolchain the host does have and did not use. It is mandatory in the strong sense — no FAILED verdict may be issued against any repository until it has been performed and recorded — because the entire cost structure of the Attribution Law says that the few minutes this costs are trivially repaid the first time it prevents a competent engineer from being aimed at correct code.

6.2 The estate's own audit supplies the narrative that justifies the rule, and it is valuable precisely because the audit initially got it wrong. The Java repository was built on the older runtime. It failed. The failure was real, reproducible, and looked exactly like the repository was broken. A run was lost to it. The audit then retried on the newer runtime, where the same repository, unmodified, produced one thousand five hundred and ninety-six results in one module and ten in another, all green. Nothing about the repository changed between the two attempts. The only thing that changed was the environment, and the difference between a lost run and a defect record filed against a working plugin is that somebody, at the right moment, asked whether the machine might be the problem.

6.3 The resolution protocol formalizes what that retry did by instinct, and it has five steps. Detect the requirement, preferring artifact-derived evidence over declarations. Enumerate the toolchains actually present on the host, by invocation rather than by inspection. Select the candidate that satisfies the detected requirement, by the compatibility asymmetry rather than by preference. Re-execute the failing command with that toolchain explicitly selected, never ambiently. Record both attempts. The fifth step is the one that will be skipped if it is not enforced, and it is the one that carries most of the long-term value.

6.4 Both attempts are recorded and neither is deleted, and the reasons are three and each is independently sufficient. The failed attempt is the evidence that the requirement is real, and without it the eventual pin looks like superstition to the next reader who wonders why the environment is constrained. The failed attempt is the seed of the training example: the negative case, in which the mismatch was read as a product failure, paired with the positive case, in which it was read as a toolchain constraint, and the pairing is what makes it teach a discrimination rather than a correction. And the failed attempt is the drift baseline: when the host later carries only the runtime that fails, the record explains immediately why, and what to install.

6.5 The search is bounded and the bound is not a matter of taste. An engine that tries toolchain after toolchain until something works is performing environment substitution, which is an adversarial pattern treated at length in a later Part, and the fact that the engine is doing it with good intentions does not change what the resulting record means. The bound is the detected requirement: the engine tries the toolchains that the requirement selects, and if the requirement selects none that the host carries, the state is BLOCKED with the missing toolchain named. Where no requirement can be detected from artifacts at all, the engine tries the declared one and stops, and two consecutive failures of the same shape terminate the search rather than provoking a third variation — the estate's standing law against retrying a failing shape applies here exactly as it applies everywhere else.

6.6 The product is never modified to satisfy the toolchain, and the prohibition is absolute within this dataset's jurisdiction. It is stated separately from the Attribution Law because the temptation here is specifically seductive: the modification that satisfies an older toolchain is often small, often looks like a portability improvement, and is often defensible on its own terms — lowering a target level, removing a construct the older compiler does not know. Each such change is a real reduction in the product's expressive substrate, made not because anyone decided the older substrate was correct but because a machine happened to have it installed. The direction of causation is inverted, and it is inverted invisibly, and the resulting codebase is a record of which runtimes happened to be on which machines on which days.

6.7 A toolchain that happened to work is not a qualification, and this is the provision that separates a resolved environment from a lucky one. When the resolution protocol identifies the toolchain that satisfies the requirement, the obligation that follows is to pin it — to write the constraint into the repository so that the next machine, the next agent, and the next continuous-integration job inherit it rather than rediscovering it. An estate that resolves the same toolchain mismatch on every fresh host is not learning; it is paying the same tax repeatedly and calling the payment normal. The pin is the institutionalization, and without it the resolution is a repair rather than a capability.

6.8 A subtlety in the same case must be surfaced, because it is where a resolved toolchain quietly hides an unresolved reproducibility problem. The Java suites ran green in an offline mode, and offline mode works once the local artifact cache is warm. That is a true statement about that machine. It is not a statement about the repository, and it is not a statement about any machine whose cache is cold, and the difference between those is the entire subject of the clean-clone engine treated later in this corpus. A qualification that depends on a warm cache has qualified a host, not a repository, and the record must say so, because the moment it says merely QUALIFIED the dependency on host state has been laundered into a property of the code.

6.9 The interface with the upstream reality layer is exercised here more than anywhere else in this dataset, and honouring its direction is what keeps the two from drifting into one. The reality layer reads manifests and artifacts and predicts, from that read, that a toolchain constraint exists and what it is. It never asserts that a build succeeded or failed, because it never built anything. This dataset receives that prediction, executes against it, and produces the verdict. The prediction is what makes the execution efficient — it tells the engine what to look for before a single build second is spent. The verdict is what makes the prediction admissible. Inverting the direction, and letting a manifest read stand as a qualification, is the failure that produces a confidently wrong environment record.

6.10 Where resolution does not succeed, the output is BLOCKED, and the record carries three things without exception: the blocker verbatim, the toolchains that were enumerated on the host, and the resolution actions that were attempted. A BLOCKED that carries only the first is a complaint. A BLOCKED that carries all three is a work order — it tells the next reader exactly what is missing, exactly what was tried, and exactly what would resolve it, and it is the difference between an environment finding that gets fixed and one that accumulates in a queue whose entries nobody can act on because none of them says what to do.

PART VI FINAL LAW

No FAILED verdict may be issued against any repository until toolchain compatibility resolution has been performed and recorded: detect the requirement from artifacts, enumerate the host's toolchains by invocation, select by the compatibility asymmetry, re-execute explicitly, record both attempts. The audit lost a run to a bytecode mismatch and retried on the correct runtime, where the same unmodified repository went green — the only thing that changed was the environment. A toolchain that happened to work is luck; a toolchain that was pinned is a capability.


PART VII — THE DEPENDENCY REPRODUCIBILITY GUARDIAN: LOCKFILES, FLOATING VERSIONS, UNRESOLVED DEPS

7.1 The lock file is the boundary between a build and the build, and the definite article is doing all the work. Without a lock, a repository does not describe a build; it describes a family of builds, one for each moment at which resolution might occur, and which member of the family you get is determined by what the upstream registries contained when you asked. The consequence is that a green result obtained from an unlocked repository is a statement about a build that no longer exists and cannot be reconstructed, and a red result obtained from one is a statement about a build nobody chose. Neither is interpretable, and neither may be attributed to the product.

7.2 Three lock states exhaust the possibilities and each carries a different qualification consequence. Present and current: the lock exists, it covers every declared dependency, and it is consistent with the manifest that declares them — this is the only state that admits QUALIFIED. Present and stale: the lock exists but disagrees with the manifest, which means two files in the repository make incompatible claims about what the build is, and which one the resolver honours is a property of the resolver rather than of the repository. Absent: there is no lock, and the build is a function of wall-clock time. The middle state is the most dangerous of the three because it looks like the first, and every casual check that asks whether a lock exists will report that one does.

7.3 The estate's evidence is unambiguous and it is the highest-leverage finding in the founding audit. Two repositories fail to compile, and the build orchestrator's own diagnostic states that a dependency is not locked, and it names five: a JSON library, a request client, a database adapter, the driver beneath that adapter, and a process-broadcast layer. The named set is itself informative in a way that a paraphrase would have destroyed — because two of those five are a database seam, and a database seam in the dependency set is a service declaration whether or not any document says so, which means this repository's qualification requires not only a lock but a provisioned store, and that second requirement is discoverable from the blocker string alone if the blocker string is retained verbatim.

7.4 The consequence of that single gate is the reason it is treated as the highest-leverage item in the estate. One hundred and forty-nine functional test files — one hundred and twenty-two in the first repository, twenty-seven in the second — sit downstream of it. Not one of them has been executed. Their state is not passing and it is not failing; it is genuinely undetermined, and the honest word for a set of assertions whose outcome nobody has observed and whose observation is prevented by a single unmet precondition is that they are in suspension. Fixing one gate — committing the lock and provisioning the store — does not fix a defect. It converts one hundred and forty-nine unknowable files into knowable ones, which is a different and larger category of value than any product repair available anywhere in the estate.

7.5 The stale-lock case requires its own detector because the resolver's behaviour under disagreement is not a constant across ecosystems, and that variability is itself a qualification question. Some resolvers refuse. Some silently prefer the manifest and rewrite the lock, which means merely running the build mutates the repository. Some silently prefer the lock and ignore a declared upgrade, which means a dependency bump that was committed, reviewed, and merged has quietly not happened. The guardian therefore does not merely check for agreement; it records which behaviour the resolver in use exhibits, because a repository whose build silently rewrites its own lock has an environment that cannot be qualified twice in a row without changing.

7.6 Floating specifiers are a latent BLOCKED wearing a green disguise, and the disguise holds until the day it does not. A dependency declared by a range rather than by a version is a dependency whose identity is chosen by a resolver, and a lock that pins today's choice is doing exactly the job it exists to do — but a repository with floating specifiers and no lock is one upstream release away from an environment that has changed without any commit, and the failure, when it arrives, arrives with an empty diff. The guardian's obligation is therefore to record the floating surface explicitly: how many declared dependencies are ranges, and how many of those are pinned by a current lock, and the difference between those two numbers is the repository's exposure to an event nobody controls.

7.7 Transitive coverage is where most lock disciplines quietly leak, and the leak is not visible at the top level. A lock that pins direct dependencies and leaves their dependencies to resolution has pinned the branches and left the trunk moving, and the failure mode is a build that reproduces perfectly on its declared surface and irreproducibly beneath it. The guardian's check is a coverage question rather than a presence question: does the lock name every artifact that will actually be fetched, transitively, or only the ones the manifest happened to mention. The second is common, is widely regarded as adequate, and is the thing that turns a supposedly deterministic build into a probabilistic one.

7.8 The guardian's checks are therefore four, and they are ordered by the cost of the answer. Does a lock exist. Does it cover the full transitive set rather than the declared set. Is it consistent with the manifest, and if not, which artefact does the resolver honour. Is resolution satisfiable without reaching outward, which is the only check that establishes that the recorded environment can be reconstructed rather than merely described. A repository that answers all four affirmatively has a dependency environment that can be qualified; a repository that fails any of them has an environment whose identity is partially determined by something outside the repository, and the qualification must name which something.

7.9 The remedy for every finding this guardian produces is an environment action and never a product action, and stating that plainly is what prevents the guardian's output from being misread as a defect list. Commit the lock. Pin the floating range. Provision the store the driver implies. Vendor the artifact that cannot be resolved offline. Not one of these touches a line of the product, not one of them is a defect in the code, and an agent that receives a guardian finding and responds by opening a source file has misread the record — which is precisely what happens when the finding is filed under the same label as a failing test, which is precisely why the twelve-state verdict ontology refuses to give them the same label.

7.10 The guardian's standing is therefore singular within the engine: it is the gate whose failure invalidates the most, and whose repair unblocks the most, and whose diagnostic is the most frequently misread. It sits upstream of every test in the repositories it governs, it produces a blocker whose text names the remedy, and it converts, at a single stroke, the largest population of undetermined assertions in the estate into a population that can be observed. There is no product change available anywhere in the audited set that carries comparable leverage, and the reason is structural: the guardian does not make anything correct. It makes correctness observable, and observability is the precondition for everything else this axis does.

PART VII FINAL LAW

Without a lock, a repository does not describe a build; it describes a family of builds, and which one you get is decided by what the registries held when you asked. Three states exhaust it — present-and-current, present-and-stale, absent — and the middle is the most dangerous because every check that merely asks whether a lock exists will say yes. One unlocked-dependency gate holds one hundred and forty-nine functional test files in suspension across two repositories. Every remedy it produces is an environment action; an agent that opens a source file in response has misread the record.


PART VIII — HOST-STATE LEAKAGE: BUILDS THAT DEPEND ON A CONTAMINATED MACHINE

8.1 Host-state leakage is defined as any dependency of a build or test outcome on machine state that is not captured in the repository, and it runs in two directions that are usually treated as one topic and are in fact different problems with different detectors and different remedies. Inbound leakage is the host contaminating the build: the outcome is a function of something on the machine that nobody declared. Outbound leakage is the build contaminating the host: the run leaves residue outside its allocated space, and the residue changes the machine for everything that runs afterward. Both are qualification defects. Neither is a product defect. And the second one, in this estate's history, is the case that came closest to destroying correct code.

8.2 The canonical inbound case in the audited estate is the warm artifact cache. The Java suites ran green in offline mode, and offline mode works once the local cache is warm. Every word of that is true and none of it is a statement about the repository. It is a statement about a machine that has previously fetched what the build needs. A fresh host, a wiped cache, or a continuous-integration runner starting from a clean image is a different environment, and the outcome there is not merely unknown — it is unknown in a way that the green result actively conceals, because the green result is what everyone will cite when asked whether the build works.

8.3 Globally installed tooling is the same defect with a wider blast radius. A build that succeeds because a binary happens to be present on the developer's machine, installed years ago for an unrelated reason and never declared anywhere, is a build whose success is a coincidence, and the coincidence will be discovered by whoever next tries to reproduce it — under pressure, with no record of what is missing, and with the repository's own documentation asserting that it builds. The detector is not a code review. It is an execution on a host that does not have the coincidence, and there is no cheaper substitute, because the whole difficulty is that the coincidence is invisible from inside the machine that enjoys it.

8.4 Ambient configuration is the subtlest inbound channel and the one that survives the most careful reviews. Environment variables set in a profile the agent never reads. Resolution-path ordering that selects one of four installed runtimes. Shims that consult a version file somewhere above the repository root. A locale that changes how a number is formatted, or a clock offset that changes which side of a boundary a timestamp falls on. Each of these is state that the build reads and the repository does not contain, and each of them will one day produce a result on one machine that does not reproduce on another, and the difference will be attributed to flakiness, which is the word an estate uses for a leak it has not located.

8.5 Outbound leakage is the inverse, and the estate's canonical instance is precise enough to be worth stating exactly. In a Python repository, the suite reported two hundred and three passing results against two hundred and four. One test wrote to the root of the host drive. That write polluted the machine, and the pollution defeated a sibling suite. The product logic was correct. Read that sequence again in the order a hurried agent reads it: a red result, inside a suite, in a repository that builds and collects cleanly, with no environmental diagnostic anywhere in sight. Every signal available points at the code, and the code is right.

8.6 The two directions require genuinely different detectors and conflating them leaves one of the two undetected. Inbound leakage is detected by variation: run the same thing on a host that differs in exactly one respect — a cold cache, a different machine, a scrubbed profile — and compare. It cannot be detected from inside a single host, at any level of inspection effort, because the leaked state is precisely the state the inspecting process also enjoys. Outbound leakage is detected by differential observation: snapshot the relevant surface before the run and after it, and any write outside the allocated sandbox is a violation. It can be detected on a single host, cheaply, and it is almost never checked.

8.7 The independence of the outbound detector from the test outcome is the property that makes it worth building, and it is the property that the estate's own case proves. A test that writes outside its sandbox does not necessarily fail. On the run where the residue does not collide with anything, everything is green, the suite is fast, the report is clean, and the machine is now different. The collision happens later, in a sibling suite, possibly in a different session, and it presents there as a failure of the sibling — which is a repository that did nothing wrong, reporting a red result caused by a neighbour, with no evidence in its own output pointing anywhere near the actual cause. An outcome-driven detector never sees the origin. A write-driven detector sees it on the very first run.

8.8 The order-dependence corollary follows directly and it is the reason outbound leakage is classified as an environment defect rather than a test defect. Once any run mutates shared state, the results of a suite become a function of execution order, and a suite whose results are a function of execution order is not producing evidence; it is producing samples. A green from such a suite does not license a claim, because the same suite in a different order may be red, and nothing in the green tells you which order you got. The hermeticity requirement — that a suite produce the same result three times from a clean state — is the empirical detector for this, and it is empirical precisely because reasoning about shared state is exactly the faculty that failed to notice the write in the first place.

8.9 The qualification consequence of either direction is the same and it is a ceiling. An environment with detected inbound leakage may be QUALIFIED for the host on which the leakage was satisfied, and it may not be QUALIFIED in general, and the record must carry the distinction rather than eliding it, because eliding it is how a machine-specific result becomes a repository-level claim. An environment with detected outbound leakage is not QUALIFIED at all, in any scope, because the harness is not contained, and an uncontained harness produces results that are not stable under repetition, and results that are not stable under repetition are not evidence regardless of what they say.

8.10 The attribution consequence is stated last because it is the one that must survive into every downstream reader's hands. Neither direction of leakage indicts the product. Inbound leakage says the repository under-declares its environment. Outbound leakage says the harness is not contained. Both are defects, both are worth fixing, and both are fixed in the environment and the fixture rather than in the code. The single most expensive available error, in the case the estate actually lived through, was to look at a red result in a suite whose product logic was correct and to begin repairing the logic — and the only thing that prevents that error, structurally, is a containment gate that fires on the write rather than on the failure.

PART VIII FINAL LAW

Leakage runs in two directions and they need different detectors. Inbound — a warm cache, a globally installed binary, an ambient variable — is detectable only by variation, never from inside the host that enjoys it. Outbound — a test writing beyond its sandbox — is detectable by differential observation, on the first run, independently of whether anything went red. On the run where the residue collides with nothing, the suite is green and the machine is now different; the collision surfaces later, in a sibling that did nothing wrong. Neither direction indicts the product.



PART IX — THE EXTERNAL SERVICE PROVISIONING CONTRACT

9.1 An external service is any resource the code reaches for at runtime that the repository does not contain: a database, a queue, an object store, a third-party interface, a container runtime, an emulator, a game server, or a physical device. The contract this Part establishes is that every such service is declared, its provisioning mechanism is named, its isolation mechanism is named, and its absence is a qualification blocker rather than a test failure. That last clause is the whole of the Part's value, because in the absence of the contract the failure a missing service produces is a red result inside a suite, and a red result inside a suite is read as a defect.

9.2 The declaration is frequently implicit and the engine's first job is to make it explicit, because the estate does not get to choose whether its services are declared — it only gets to choose whether it knows about them. A database driver in the dependency set is a service declaration. A queue client is a service declaration. An object-store binding is a service declaration. The audited Elixir repositories carry a database adapter and the driver beneath it among the five dependencies the build orchestrator named as unlocked, which means the blocker string itself, read carefully, announces that this repository will require a provisioned store before a single integration test can be interpreted — and it announces it before anyone has spent a build second, provided the string was retained verbatim rather than summarized into a complaint about dependencies.

9.3 Provisioning and isolation are separate obligations and satisfying the first without the second produces an environment that is qualified for one run and unqualified for the second. A store that exists is provisioned. A store that exists and gives each test a view that is rolled back afterward is isolated. The transactional-sandbox pattern that the Elixir ecosystem supplies for its database seam is the correct hermetic shape precisely because it addresses the second obligation, and an estate that provisions a store and then lets its suite mutate it shared has built an environment whose results are order-dependent — which returns it, by a different road, to exactly the pathology the outbound-leakage detector exists to catch.

9.4 The substitution ladder is the mechanism by which a missing service is survivable without becoming a lie, and every rung on it lowers the strength of the claim the environment can support. The real service is the top rung and licenses the strongest evidence. A containerized instance of the same service is next and is nearly as strong, differing only in the ways a container differs from the deployed reality. An emulator is a rung below that, because it reimplements a contract rather than executing it. A stub is below that, because it answers rather than behaves. A mock is at the bottom, because it asserts what it was told to assert. Each rung is legitimate. What is not legitimate is climbing the ladder in the record.

9.5 The prohibition that follows is the one most frequently violated in practice and it is violated with entirely good intentions. A mocked dependency does not qualify an environment for an integration claim. It qualifies it for a narrower claim — that the unit under test behaves correctly when its collaborator behaves as the author believes the collaborator behaves — and that narrower claim is genuinely valuable and is not the claim that gets written down. What gets written down is that the integration is tested. The environment record therefore carries the substitution rung explicitly per service, and the verdict ceiling is derived from the lowest rung any required service occupies, which means a single mocked collaborator caps the whole integration surface at CONDITIONALLY-PROVEN with the substitution named in the condition.

9.6 The estate supplies a concrete instance of substitution scarcity that is worth stating in the terms of the ladder. The Java plugin estate has available to it an in-memory server implementation that lets plugin logic execute headlessly without a real host process — a legitimate rung, well below the real service and well above nothing. It is wired into one of ninety build descriptors. Which means that for the overwhelming majority of that estate's modules, the environment offers no rung at all: not a real service, not a container, not an emulator, not even a stub. Those modules are not poorly tested. They are, in the precise sense this Part uses, unprovisioned — and an unprovisioned module cannot be qualified, and an unqualified module cannot yield a product attribution, and the correct record for its surfaces is UNVERIFIED.

9.7 The declaration record therefore carries five fields per service and the fifth is the one everyone omits. The service and its version. The provisioning mechanism — how it comes into existence for a run. The isolation mechanism — how one test's use of it is prevented from affecting another's. The substitution rung actually in use on this pass. And the teardown — how it ceases to exist, and what it leaves behind if teardown fails. The fifth field is where outbound leakage hides in service form: a store whose teardown is incomplete is a shared mutable resource, and a shared mutable resource makes the next run's results a function of this run's, which is the definition of a non-hermetic environment.

9.8 The consequence of an unprovisioned service is scoped rather than global, and the scoping is what keeps the engine's output usable rather than nihilistic. A repository whose store is unavailable is BLOCKED for the surfaces that reach the store, and it may be perfectly QUALIFIED for the surfaces that do not, which means the honest state of the repository as a whole is PARTIALLY QUALIFIED with the boundary enumerated. Rounding that to BLOCKED discards real verification that was available; rounding it to QUALIFIED launders an unexecuted seam into a verified one. The state exists specifically so that neither rounding is necessary, and the enumeration of which surfaces fall where is the field that makes the state actionable rather than merely accurate.

9.9 There is a class of service whose absence is permanent from the host's point of view, and it must be routed to a different state or it will sit in the provisioning queue forever. A physical device is the clearest case; a target console is another. The engine distinguishes them by asking a single question: could this host, at some finite cost in effort and configuration, supply this service. If yes, the state is BLOCKED and the remedy is provisioning. If no — if what is missing is not a package but a machine — the state is HARDWARE REQUIRED, and the remedy is either a different host, a different substrate, or a restructuring of the product surface so that the coupling no longer exists. Conflating the two produces a backlog whose entries are individually reasonable and collectively unclosable.

9.10 The contract's final provision is directed at the reader of the record rather than at its producer. A service declaration that was never probed is not a service that is available. It is a service whose availability is UNKNOWN, and an environment carrying an unprobed service declaration is an environment carrying an unexamined precondition, and every result obtained from it inherits that unexamination. The engine therefore probes every declared service on every pass, records the probe command and its output, and refuses to emit QUALIFIED while any declaration remains unprobed — because the alternative is an environment that is qualified in every respect anyone bothered to check, which is the exact shape of every confident wrong answer in this corpus.

PART IX FINAL LAW

Every external service is declared, provisioned, isolated, and torn down, and its absence is a qualification blocker rather than a test failure. A driver in the dependency set is a service declaration whether or not any document says so. The substitution ladder — real service, container, emulator, stub, mock — is legitimate at every rung and illegitimate to climb in the record: the verdict ceiling is set by the lowest rung any required service occupies, so one mocked collaborator caps the whole integration surface, with the substitution named in the condition.


PART X — PER-STACK QUALIFICATION PROFILES: PYTHON AND JS/TS

10.1 A qualification profile is the concrete instantiation of the engine's seven gates for one stack: which command probes each gate, which artifact carries each requirement, and which failure mode is characteristic enough to be worth a dedicated detector. The profiles are not summaries of the engine; they are where the engine becomes executable, and the differences between them are not cosmetic. The gates are universal. What satisfies them is not, and an engine that assumes otherwise will qualify one stack correctly and hallucinate the others.

10.2 The interpreted-language profile begins with a version probe that must be taken from the interpreter the runner will actually invoke, and not from the one the shell answers to. On a host carrying several interpreters behind a shim, those are routinely different, and the difference is invisible until a syntax introduced in a newer version fails to parse under an older one and the diagnostic points at the source file. The profile therefore records the interpreter's self-reported version, the path it was resolved from, and the mechanism that resolved it — the ambient-selection triple from the version Part, made concrete.

10.3 Isolation of the interpreter's package environment is the second gate and it is the stack's characteristic weakness. Packages installed into a shared, machine-wide environment are inbound leakage by construction: the build succeeds because of what somebody installed once, the repository does not record it, and the next host does not have it. A per-project isolated environment reconstructed from a lock is the qualification-grade shape, and a shared environment is not. The gate is not satisfied by observing that the imports resolve. It is satisfied by observing that they resolve from an environment that the lock alone can reconstruct.

10.4 The containment gate is where this stack's canonical failure lives and where its profile earns its place in the corpus. The estate's Python case is precise: a suite reported two hundred and three passing against two hundred and four, and the red one traced to a test writing to the root of the host drive, and that write defeated a sibling suite, and the product logic was correct. The stack supplies the correct primitives for containment — a per-test temporary directory and a patching facility for anything touching the filesystem, the environment, the clock, or randomness — and the profile's gate is therefore not "does the suite pass" but "does the suite write outside the space allocated to it", and the second question is answered by differential observation regardless of the answer to the first.

10.5 The collection surface is the stack's second characteristic weakness and it sits at the boundary between environment and configuration. The runner's configuration determines which files are collected, and that configuration is itself unverified code that nobody tests. A scope line that points the default invocation at a subtree is not a build failure and not a test failure; it is a silent reduction of the executed surface, and it presents as a fast green suite. The profile records what the default invocation collected, as a count, from the runner's own output — not from a file census, and not from a summary, because the summary is precisely what destroys the count.

10.6 The compiled-to-nothing scripting profile — the browser-oriented stack — begins with an admission that the audited estate made explicitly and that is worth preserving as the model of honest non-execution. Three suites were counted: twenty-nine specification files in one repository, forty-one in another, thirteen in a third. Not one of them was executed. The reason was scoped and stated: the installation step for that stack carries side-effects that fell outside the agreed boundary of the audit. And the result was recorded as unverified, not as passed, and not as failed. That is the entire discipline of this dataset performed correctly, on a stack that was never even qualified, by an audit that declined to pretend otherwise.

10.7 Counting is not qualifying, and the distinction is the reason that admission was correct rather than merely cautious. A census of specification files establishes that authored protection exists. It establishes nothing about whether that protection executes, whether the runtime that would execute it is present, whether the dependency graph resolves, or whether any assertion in any of those files has ever been evaluated. Eighty-three specification files that have never run protect exactly what zero files protect, and the census makes the situation worse rather than better, because the census is what produces the impression of coverage. The honest state is UNKNOWN qualification and UNVERIFIED verdict, and that is what was recorded.

10.8 The installation step in this stack is itself an execution surface, and that fact is the reason the audit's scope boundary was principled rather than lazy. Installing dependencies in this ecosystem runs arbitrary code supplied by the dependency authors, at install time, with the privileges of the invoking user. An audit that installs is an audit that has executed third-party code on the host as a precondition for looking at the repository, which changes the machine and may change other repositories' environments. Declining to do that inside a read-only audit is correct, and the correct consequence is a qualification state of UNKNOWN — not a lowered standard, and not an execution performed carelessly to fill in the blank.

10.9 The profile for that stack therefore terminates at gate one for the audited estate, and the record says so with its downstream gates marked UNKNOWN rather than omitted. What it would take to qualify is stated as a work order: an isolated install from a lock, on a host whose contamination is acceptable or which is disposable, with the runtime version pinned, the installed tree reconstructed from the lock alone, and the suite's containment observed. Every element of that is an environment action. None of it is a product action, and none of the eighty-three files is a defect, and the difference between a repository with eighty-three unexecuted specification files and one with eighty-three failing ones is the entire subject of this corpus.

10.10 Both profiles converge on the same shape and the convergence is what makes the engine generalizable rather than a collection of special cases. A runtime whose version is observed from the invocation the build actually uses. A dependency set reconstructible from a lock alone. A harness contained within an allocated sandbox. A service surface probed rather than assumed. A collection surface counted from the runner's own output. Those five properties are the whole of qualification in every stack this corpus touches; what differs between stacks is only which command answers each question, and which of the five the stack makes it easiest to get wrong.

PART X FINAL LAW

A profile makes the engine executable for one stack: the gates are universal, what satisfies them is not. The interpreted stack's characteristic failures are a shared package environment and an uncontained harness, and the estate's canonical fault-attribution case is exactly the second. The browser-oriented stack was counted — eighty-three specification files across three repositories — and never executed, because installation in that ecosystem is itself an execution surface, and the audit recorded unverified rather than passed. Counting is not qualifying; a census of unexecuted files produces the impression of coverage and none of the substance.


PART XI — PER-STACK QUALIFICATION PROFILES: ELIXIR/PHOENIX AND JAVA/JVM

11.1 The Elixir profile is dominated by a single fact that reorders every other gate: the compile step is not a formality that precedes testing, it is the gate at which the estate's two repositories in this stack currently terminate. The build orchestrator refuses to compile because dependencies are not locked, and it names them — a JSON library, a request client, a database adapter, the driver beneath it, and a process-broadcast layer. Nothing downstream of that refusal has been observed. Not the runtime's behaviour, not the database seam, not one assertion in one test file.

11.2 The blocker string is retained verbatim in the record and the retention is not ceremonial. It names five dependencies. Two of them constitute a database seam, and a database seam declares a service, and a declared service means that committing the lock is a necessary and insufficient remedy — the store must also be provisioned before an integration test can be interpreted. That second requirement is derivable from the blocker text alone, at zero cost, before anyone has attempted a second build. A paraphrase that recorded merely "dependency resolution failed" would have destroyed it, and the next agent would have committed the lock, run the suite, watched the database tests fail, and had every reason to believe they had found a defect.

11.3 The scale of what is held in suspension by that one gate is the reason it ranks first in the estate's remediation order. One hundred and twenty-two functional test files in the first repository and twenty-seven in the second. One hundred and forty-nine files whose assertions have never been evaluated, whose state is therefore neither passing nor failing, and whose verdict is not FAILED, is not UNVERIFIED-because-nobody-wrote-them, but is precisely BLOCKED: authored, present, and prevented. The distinction matters operationally, because a BLOCKED corpus is one environment action away from being observable, whereas an absent corpus is a quarter of authoring away.

11.4 The stack's database seam deserves separate treatment because the framework's own conventions raise the service requirement rather than lowering it. The framework supplies case templates that make the application-to-store integration the default shape of a test rather than a special one, which is excellent engineering and means that a very large fraction of the authored corpus reaches the store. It supplies a transactional sandbox that gives each test a rolled-back view, which is the correct hermetic isolation primitive. The consequence for qualification is direct: this stack cannot be even partially qualified without a provisioned store, because the store is not a peripheral seam in it — it is the seam most of the corpus exercises.

11.5 The Elixir qualification checklist is therefore short and entirely environmental. Observe the language and runtime versions from the invocation the build uses. Confirm the lock exists, covers the transitive set, and agrees with the manifest. Observe that resolution is satisfiable. Observe a successful compile. Probe the store. Confirm the isolation primitive is engaged. Every one of those is a command with an output, and the two remedies the estate needs — commit the lock, provision the store — are both environment actions, and neither touches a line of the one hundred and forty-nine files that are waiting on them.

11.6 The Java profile is dominated by a different gate, and the dominance is just as complete. The toolchain gate decides everything, because the prebuilt test classes carry a bytecode format that one runtime generation emits and its predecessor refuses. Building on the older runtime terminates with a diagnostic stating that the class was compiled by a more recent version — a sentence about two pieces of the environment disagreeing with each other, containing no information whatsoever about the plugin logic, and routinely read as a build failure of the repository.

11.7 The audit's own history is the profile's evidence and its warning. A run was lost to that mismatch. The retry, on the newer runtime, against the same unmodified repository, produced one thousand five hundred and ninety-six results in one module and ten in another, green. The delta between the lost run and the green one is zero lines of product code. Whatever the first run appeared to be evidence of, it was not evidence about the product, and the only reason the estate knows that is that somebody retried rather than filing.

11.8 The offline subtlety in this stack is the estate's clearest instance of inbound leakage and it must be recorded rather than celebrated. The suites ran green offline, and offline works once the local artifact cache is warm. That result is a true statement about a machine that has previously fetched what the build needs, and it is not a statement about a machine that has not. The qualification record therefore says QUALIFIED-on-this-host-with-a-warm-cache, and the reproducibility question — what happens from a cold cache on a clean clone — is a separate and unanswered one, and collapsing the two would launder a host property into a repository property.

11.9 The estate-level aggregation error is this profile's second lesson and it is a general one. The repository carries ninety build descriptors. Two modules were built and observed green. An in-memory server that would let plugin logic run headlessly is wired into one of the ninety. Qualification is therefore a per-module property in a multi-module estate, and an estate-level QUALIFIED derived from two green modules is an aggregation over a population whose distribution was never examined — which is the hidden-surface law arriving through the environment door rather than through the coverage door.

11.10 The two stacks converge on a single structural observation that generalizes beyond both. In each, the gate that blocks sits entirely upstream of every test, is diagnosable in seconds, is repaired by an environment action, and produces a diagnostic that reads like a product failure to anyone who does not know to look. In one stack it is the lock; in the other it is the bytecode version. The population held in suspension differs — one hundred and forty-nine files in one case, an entire estate's worth of modules in the other — but the shape is identical, and it is the shape that the qualification engine exists to make visible before anyone opens a source file.

11.11 That convergence licenses a mechanism the engine does not otherwise possess, and it is the operational product of this Part: the dominant-gate registry. For each stack the engine records which of its seven gates has historically been the terminating one, and it reorders its probes so that the dominant gate is evaluated first rather than in the canonical sequence. In the Elixir profile the dominant gate is dependency resolvability, and probing it first costs a single lock inspection and terminates the pass in seconds rather than after a build attempt that was never going to reach a test. In the Java profile the dominant gate is toolchain version compatibility, and probing it first means reading a bytecode header rather than waiting for a compiler to fail and then reading its complaint. The registry is populated from observation and not from assumption — a stack earns its dominant gate by having terminated there — and it is re-derived on every pass, so that a stack whose lock discipline is repaired stops being probed lock-first and begins terminating, honestly, wherever it now terminates. The reordering changes no verdict and cannot: short-circuit semantics guarantee that the same blocker is found regardless of probe order. What it changes is the cost of finding it, and cost is what determines whether a gate is run at all.

PART XI FINAL LAW

In both stacks the blocking gate sits upstream of every test, is diagnosable in seconds, is repaired by an environment action, and emits a diagnostic that reads like a product failure. The unlocked-dependency blocker names five dependencies, two of which constitute a database seam — so the verbatim string announces the service requirement before a second build is attempted, and a paraphrase would have destroyed it. The bytecode-version blocker cost a lost run, and the retry on the correct runtime produced green against zero lines of changed product code.


PART XII — C/EMBEDDED/HARDWARE, AND THE REPOSITORY WITH NO EXECUTABLE SURFACE

12.1 The hardware-coupled profile begins where the others end, because for this class of repository the first gate the engine reaches is also the last one it can evaluate on the host. The audited console repository carries files whose names announce them as tests. They include the engine's own headers. They link the whole game. And they require a cross-compiler targeting the console's processor, which the development host does not have and cannot usefully acquire as a package. There is no arrangement of the host that runs them.

12.2 The correct qualification state is HARDWARE REQUIRED and the correctness turns on a single question that separates it from BLOCKED: could this host, at some finite cost, supply what is missing. For an unlocked dependency, yes — commit the lock. For an absent store, yes — provision it. For a missing runtime generation, yes — install it. For a cross-toolchain plus a target console, the answer is not a matter of effort; it is a different substrate. A backlog item that says provision the environment, filed against this repository, is an item that will never close, and it will consume a fraction of the attention of every person who reads past it, forever.

12.3 What HARDWARE REQUIRED licenses is nothing at all about the product, and stating that plainly is the whole point of separating the state. It is not a finding. It is not a defect. It is not evidence of poor engineering, and it is not a coverage gap in the sense that a coverage gap is normally understood, because a coverage gap implies that coverage was achievable and was not achieved. The files in question are, as the estate's own standards observe, integration smoke drivers rather than isolatable units — they were written to exercise a game on the machine the game runs on, and they do that, and the host's inability to run them is a property of the host.

12.4 The cross-compilation topology is what the profile must record and it is a triple rather than a single environment. There is the build host, which runs the toolchain. There is the target, which runs the artifact. And there is the runner, which observes the artifact's behaviour and reports it — and on a console that runner is frequently the artifact itself, reporting through whatever channel the hardware provides. Three environments, three qualification questions, and an engine that models the situation as one environment will produce a record that is confidently wrong about two of them.

12.5 The evidence tiers in this profile are shifted downward relative to every other stack, and the shift is not a penalty but an accurate description of what is available. A capture from real hardware is the strongest evidence this surface admits, and it is genuinely strong — it is an observation of the actual substrate. An emulator's output is a rung below, because an emulator reimplements the substrate's contract and its fidelity is itself an unverified claim. A host-compiled approximation of the logic, with the hardware calls stubbed, is a rung below that. The profile's obligation is to name the rung, and the ceiling it imposes, on every result it records.

12.6 This profile contains the single case in the entire dataset where the correct remedy touches the product rather than the environment, and the exception must be named precisely so that it is not generalized. The remedy the estate's standards identify is to extract the pure logic — the arithmetic, the state transitions, the scoring — into functions that do not depend on the hardware surface, and to exercise those on the host with a minimal assertion harness. That is a product change. But note what it is: it does not modify behaviour to satisfy a broken environment, which is the prohibited move. It moves a surface out from behind a coupling so that a qualified environment becomes available for it. The distinction is the difference between changing what the code does and changing where the code sits, and only the first is forbidden.

12.7 The repository with no executable surface is the profile's mirror image and it is the case that most reliably breaks naive engines. One repository in the audited estate contains no executable code at all. The correct qualification result is not that it is unqualified, and the correct verdict is not that it has zero coverage. There is nothing to qualify, and the state the ontology reserves for that is not-applicable — a genuine outcome, produced by an engine that was able to look and to conclude that the question does not arise.

12.8 A naive engine cannot produce that outcome, and the failure is instructive because it is a failure of the engine's own imagination rather than of its execution. An engine whose only states are qualified and unqualified will classify a content repository as unqualified, and a downstream consumer will read unqualified as a gap, and the gap will be sized against a surface that does not exist, and an agent will eventually be dispatched to write tests for logic that was never written. The estate's own audit came within one premise-check of exactly that: the repository had been carried in the plan as a code project on the strength of its name and position, and the disk disagreed, and consulting the disk is what prevented a fabricated finding.

12.9 The symmetry between the two ends of this Part is worth stating because it is the Part's real content. HARDWARE REQUIRED and not-applicable both produce zero test results. Both are honest. Both are, to any instrument that counts results, indistinguishable from catastrophic. And exactly one of them describes a gap — the console repository does have logic that would benefit from a host-runnable surface, and the content repository has nothing that would benefit from anything. An engine that cannot tell those apart will either raise a false alarm on one or grant a false absolution to the other, and it will do so from the same input.

12.10 The generalization is the one this dataset ends its per-stack treatment on. Zero results is not a verdict. It is a shape, and at least four distinct realities produce it: the surface does not exist, the surface exists and requires hardware the host cannot supply, the surface exists and is blocked by an environment action nobody has taken, and the surface exists and is silently excluded from the invocation that was run. The four demand entirely different responses — nothing, a different substrate, a provisioning action, and a configuration repair — and the only instrument that distinguishes them is the qualification record, which is why a result without one is not weak evidence but no evidence.

PART XII FINAL LAW

HARDWARE REQUIRED is separated from BLOCKED by one question: could this host, at finite cost, supply what is missing. For a lock, a store, or a runtime, yes; for a cross-toolchain plus a target console, no — and filing it as BLOCKED creates a backlog entry that will never close. The repository with no executable code is not-applicable, not uncovered, and a naive engine that lacks the state will size a gap against a surface that does not exist. Zero results is a shape, not a verdict: four different realities produce it, and only the qualification record tells them apart.



PART XIII — THE CLEAN-CLONE REPRODUCIBILITY ENGINE: CLONE, INSTALL, BUILD, MIGRATE, SEED, RUN, CLEAN, RUN AGAIN

13.1 Every qualification performed on a machine that has been used is a claim about that machine, and the whole difficulty is that the claim is indistinguishable, from inside the machine, from a claim about the repository. The clean-clone engine exists to separate them. It executes the repository's full lifecycle from a pristine state on a host that carries none of the residue of prior work, and it observes each stage rather than only the last, because the information the engine exists to produce is in which stage fails and not in whether the suite is green.

13.2 The sequence is eight stages and it is executed in order without shortcuts: clone from the remote at a named commit into a directory that did not previously exist; install the dependency set from the lock alone; build; migrate the store; seed whatever the suite requires; run; clean; run again. Each stage is observed, each stage's command and output is retained, and the first stage to fail terminates the pass with the same short-circuit discipline the qualification engine uses — because a migration executed against a build that did not happen is not information, it is noise wearing the costume of thoroughness.

13.3 The clone stage carries a check that sounds trivial and catches a surprising amount. The working tree must be constructed entirely from the remote at the named commit, and any file the subsequent stages require which is not present in that tree is a finding — a configuration file living only on a developer's machine, a credential expected in a location nobody documented, a generated artifact that was committed once and has since been regenerated by hand. Each of these is inbound leakage in its most concrete form: the repository does not contain what the repository needs, and the only reason nobody has noticed is that every machine that has ever built it acquired the missing piece by a route that was never recorded.

13.4 The install stage is where the lock is genuinely exercised, and exercising it means something narrower than most estates assume. It is not enough that installation succeeds. Installation must succeed from the lock alone, without consulting the manifest for a resolution decision, and ideally without reaching outward at all beyond a fixed artifact source. A repository whose install succeeds only because the resolver was free to choose has not demonstrated a reproducible dependency environment; it has demonstrated that a reproducible one was not needed today. The audited estate's unlocked repositories terminate at this stage by construction, and terminating here is the correct and informative outcome.

13.5 The build stage is where the estate's warm-cache result is finally tested honestly, and this is the stage the clean-clone engine exists to reach. The Java suites ran green in offline mode on a machine whose artifact cache was warm. On a clean clone with a cold cache, offline mode is not a shortcut — it is a hard constraint that the environment either satisfies from vendored artifacts or does not satisfy at all. The engine does not predict which. It executes and observes, and the observation is the first evidence anyone will hold about whether the repository is buildable or merely whether one machine has already fetched what it needs.

13.6 The migrate and seed stages exercise the service contract in the only way that establishes it, which is by exercising it from nothing. A store that exists is not a store that is provisioned correctly: the schema must be constructible from the repository's own migrations, applied in order, against an empty instance, and the fixtures the suite requires must be constructible from the repository's own seeds. An estate whose store schema is a live artifact that has drifted from its migration history will pass every test on the developer's machine and fail on the first clean provision, and it will fail there in a way that looks like a defect in the migration and is in fact a defect in the history.

13.7 The run stage produces the first result and it is deliberately not the last stage, because a single green run is precisely the artifact that this entire corpus exists to interrogate. What the run stage supplies is a baseline: the executed count, the skip count, the collected surfaces, and the outcome, taken from the runner's own output rather than from any summary of it. That baseline is what the next two stages are compared against, and without it the hermeticity check has nothing to compare and degrades into a second opinion rather than a measurement.

13.8 The clean stage is an observation disguised as a housekeeping step, and it is where outbound leakage is caught in its natural habitat. Cleaning means returning the environment to the state the run began in — and the act of attempting that is what reveals what the run changed. Residue outside the allocated sandbox is enumerated here explicitly rather than inferred. The estate's canonical case, a test writing to the root of the host drive, is not a subtle finding at this stage; it is an item on a list, produced by a differential observation, requiring no cleverness and no suspicion, and available on the very first pass regardless of whether anything went red.

13.9 The second run is the stage that converts a result into evidence, and its criterion is exact rather than approximate. The second run must produce the same executed count, the same skip count, and the same outcome as the first. A drop in executed count is a regression even if nothing failed, because it means the second run reached less of the code than the first and the environment has therefore mutated between them. A change in outcome is a hermeticity violation regardless of its direction — a suite that fails first and passes second is exactly as broken as one that passes first and fails second, and the estate's instinct to celebrate the first shape and investigate the second is itself a defect.

13.10 The engine's output is a reproducibility verdict, and it is a different object from a test verdict and must never be merged into one. A repository may have a fully green suite and a failing reproducibility verdict, and that combination is not a contradiction — it is the most common state in a mature estate and it is the state the engine exists to make visible. The test verdict says what the code did on a machine. The reproducibility verdict says whether that machine can be reconstructed from the repository. Only the second licenses the claim that the result means anything to anyone who is not sitting at that machine.

PART XIII FINAL LAW

Qualification on a used machine is a claim about the machine, and from inside the machine it is indistinguishable from a claim about the repository. Eight stages separate them: clone, install from the lock alone, build, migrate, seed, run, clean, run again. The clean stage catches outbound residue by enumeration rather than suspicion, on the first pass, regardless of outcome. The second run must reproduce the executed count, the skip count, and the result — a drop in executed count is a regression even when nothing failed. The reproducibility verdict is a separate object from the test verdict, and only it licenses a claim beyond the machine.


PART XIV — "WORKS ON MY MACHINE" VERSUS "THIS REPOSITORY IS REPRODUCIBLE"

14.1 The phrase that names this Part is treated as an admission of embarrassment and is in fact an unfalsifiable statement, and the second property is what makes it useless rather than merely awkward. It asserts a success without naming the conditions under which the success was obtained, which means it can never be contradicted by any subsequent observation: a failure elsewhere is explained by the elsewhere, and a success elsewhere is explained by nothing. The reformulation that makes the situation tractable does not argue with the claim. It measures it.

14.2 The measurement is the reproducibility index, and it is defined as the set of host-state variables on which the outcome depends. A repository whose outcome depends on nothing outside itself has an empty set and is reproducible in the strong sense. A repository whose outcome depends on a warm artifact cache has a set containing one element. A repository that additionally depends on a globally installed binary, an ambient environment variable, a resolution-path ordering, and a store that was provisioned by hand three months ago has a set with five, and the size of that set is the exact quantity of undocumented knowledge required to reach parity with the machine on which it works.

14.3 The index is measured by variation and never by assertion, and this is the methodological core of the Part. No amount of inspection of a repository establishes that its outcome is independent of host state, because the state in question is precisely the state that the inspecting process also enjoys and therefore cannot see. The only instrument is difference: change exactly one host-state variable, re-execute, and compare. A variable whose change does not move the outcome is not in the set. A variable whose change moves the outcome is in the set, and it has just been discovered, and it can now be either eliminated or declared.

14.4 Three variations are cheap enough to be mandatory and they cover most of the observed exposure. The cold cache: scrub the local artifact store and re-execute, which catches the whole class of inbound leakage through prefetched dependencies. The second host: execute on a machine that has never built this repository, which catches globally installed tooling and ambient configuration together. The changed clock and locale: execute with a shifted time zone and a different formatting locale, which catches the class of defect that presents as an intermittent failure at a boundary and is diagnosed as flakiness for months.

14.5 The estate's own Java result is the worked example and it must be stated with the precision that makes it useful rather than the precision that makes it flattering. The suites ran green offline, and offline works once the artifact cache is warm. That is a true, observed, valuable result: the code compiles and one thousand five hundred and ninety-six assertions plus ten more hold on the newer runtime. It is also, exactly, a result whose reproducibility index has not been measured, because the cold-cache variation was not exercised. The honest record is that execution is proven under the stated environment and reproducibility is unknown, and those two fields sit side by side, and neither is permitted to be inferred from the other.

14.6 Reproducibility is a property of the repository and not of the result, and confusing the two is what makes the whole subject feel like blame. A red result is a property of a run. A reproducibility index of five is a property of a codebase — it says that five things must be true of a machine before this codebase can be exercised, and that none of the five is written down. That is not a failure of the person who ran the suite. It is a description of what the repository currently requires, and it is precisely the description that a new contributor, a fresh continuous-integration image, or an agent with no session history needs and cannot obtain any other way.

14.7 The independence requirement enters here in the form the constitution's separation of powers demands. A reproducibility claim verified by the same actor, on the same host, using the same cached state, is a claim verified by the thing it is a claim about. The cheapest genuine independence available is not a second opinion but a second substrate — a machine, an image, a container that shares none of the first's history. That substrate has no model of the system, no expectations, and no blind spots correlated with the producer's, and it is available in unlimited quantity at negligible cost, which makes declining to use it a choice rather than a constraint.

14.8 Reproducibility is graded rather than binary and the grades are operationally distinct. Reproducible-on-this-host means the result repeats on the machine that produced it, which the hermeticity re-run establishes and which is the weakest useful grade. Reproducible-on-any-host-of-this-class means it repeats on a machine of the same operating system and architecture with no shared history. Reproducible-from-a-clean-clone means the eight-stage engine completes from nothing. Each grade licenses a wider claim than the one beneath it, and an estate that reports the top grade while having only measured the bottom one has performed the rounding this corpus most consistently forbids.

14.9 The metric invites a specific gaming and the countermetric must be named alongside it, per the estate's standing anti-Goodhart contract. An index of zero is trivially achievable by never varying anything: if no host-state variable is ever changed, none is ever observed to matter, and the set is empty by construction. The countermetric is therefore variation breadth — how many host-state variables were actually varied in the pass that produced the index — and an index of zero obtained from a single unvaried run is not a strong result. It is a claim about nothing, and it must be reported as such, with the breadth number sitting next to it where nobody can read one without the other.

14.10 What reproducibility licenses is narrower than its advocates usually claim and larger than its detractors concede. It does not make a test correct: a reproducible suite of weak assertions reproduces its weakness perfectly. It does not make a fixture realistic, or a coverage figure meaningful, or a green result trustworthy. What it does is make the result interpretable — it converts an observation about a machine into an observation about a repository, and every downstream judgement in this axis is a judgement about a repository, which means that without reproducibility the entire axis is reasoning about the wrong object with great rigour.

14.11 The mechanism that makes all of this executable rather than aspirational is the variation matrix, and it exists because variation is the only instrument here and variation is not free. The matrix enumerates the host-state variables the estate knows how to vary — cache warmth, host identity, clock, locale, resolution path, service freshness — and assigns each a cost and a cadence. The cheap ones run on every pass: the hermeticity re-run costs one suite execution and catches outbound residue and order-dependence together. The medium ones run per release: the cold-cache build and the second-host execution, which between them catch the dominant inbound channels. The expensive ones run per quarter or on a trigger: a full clean-clone lifecycle from nothing, which is the only measurement that establishes the top reproducibility grade and is too slow to gate every commit. The matrix's value is not that it schedules work; it is that it makes the unexercised variations visible as a list. An estate that has never varied host identity does not have an unmeasured risk — it has a named, enumerated, unmeasured variable, sitting in a matrix cell that says so, and a cell that says so is a thing an owner can decide about. The alternative is a reproducibility claim whose gaps are not merely unmeasured but unnamed, and an unnamed gap cannot be prioritized, budgeted, or declined.

PART XIV FINAL LAW

"Works on my machine" is unfalsifiable, not embarrassing — it asserts a success without naming its conditions, so no observation can contradict it. The reproducibility index replaces it: the set of host-state variables the outcome depends on, measured by variation and never by inspection, because the leaked state is exactly what the inspecting process also enjoys. An index of zero from a single unvaried run is a claim about nothing, so the breadth of variation is reported beside it. Reproducibility does not make a result correct; it makes it interpretable beyond one machine.


PART XV — ENVIRONMENT DRIFT AND THE SILENT INVALIDATION OF A PRIOR RESULT

15.1 A certification asserts that a specific surface, at a specific commit, in a specific environment, was verified by specific evidence. Three of those four are immutable once recorded. The fourth moves continuously, without anybody's consent, and it moves without producing a commit, a review, an alert, or any artifact whatsoever inside the repository — which means a certification can become false while every field in it remains exactly as written, and no diff anywhere in the estate records the moment it happened.

15.2 The drift sources are enumerable and each has a characteristic latency. A floating dependency drifts when an upstream author publishes, which can be any minute. A base image drifts when its maintainer rebuilds, typically weekly. A host runtime drifts when a package manager upgrades it, typically on a cadence nobody chose. A service drifts when it is upgraded by an operator who has no idea which suites depend on its exact behaviour. A credential expires on a schedule set elsewhere. A clock crosses a boundary — a year, a daylight transition, a leap of any kind — on a schedule set by physics. Not one of these events is visible from inside the repository.

15.3 The invalidation is silent and the silence is the entire problem, because every instinct an engineer has for locating a cause is trained on the diff. A suite that was green last week and is red today, with an empty diff, is the shape of a drift event, and it is also the shape of a flaky test, and it is also the shape of a genuine latent defect that a timing change finally exposed — and those three demand entirely different responses and are indistinguishable without an environment record from the green run to compare against. Estates without one do not resolve this; they retry until it passes, and the retry that passes is filed as a resolution.

15.4 The instrument is the environment hash: a content-addressed fingerprint computed over the fields of the qualification record — the observed runtime versions, the lock's content identity, the resolved service versions, the host census, the toolchain selection. It is computed on every pass, at negligible cost, and it is a single value that answers the question no other artifact answers, which is whether this run happened in the same environment as the last one. It is not a security mechanism and it does not need to be. It only needs to change when the environment changes.

15.5 Drift detection is then a comparison, and the comparison is performed at execution time rather than on a schedule, because a scheduled check answers the question at the moment nobody was asking it. Each execution computes the hash and compares it against the hash bound to the certifications that currently cover the surface being executed. If the hash is identical, the prior result is comparable and a regression against it is real. If the hash has moved, the prior result is a statement about a different environment, and the correct action is not to compare against it at all — because the comparison would attribute an environmental delta to the code.

15.6 The unlocked-dependency case is the pathological extreme of drift and is worth naming as such, because it shows what the hash is protecting against. In a repository whose dependencies are not locked, the environment hash cannot even be computed, because the dependency set is not a determinate object — it is a function evaluated at resolution time against registries that change. Such a repository does not have an environment that drifts. It has an environment that is re-drawn on every build, which means every prior result about it is unmoored from the moment it is produced, and the fifty percent of the drift problem that consists of detecting change is unsolvable there in principle rather than in practice.

15.7 Expiry conditions are the mechanism by which drift is handled rather than merely detected, and they are fields on the certification rather than prose beside it. A certification carries the environment hash it was issued against, and it carries the conditions that revoke it: a change in that hash, a drop in the executed count, the appearance of a related regression, or a shift in the contract it was asserted over. When any condition fires, the certification is revoked automatically. No meeting occurs, no blame attaches, and the original work is not impugned — the revocation says only that the conditions under which the claim was true no longer hold, which is a statement about the world rather than about the claimant.

15.8 The re-qualification cadence follows from the drift budget rather than from a calendar, and the distinction is what keeps the practice from degrading into ritual. An environment with a fully pinned dependency set, a fixed image, and a locked toolchain drifts slowly, and re-qualification on every execution is nearly free because the hash rarely moves. An environment with floating specifiers and an ambient runtime drifts continuously, and the correct response is not to re-qualify more often — it is to reduce the drift surface, because an environment that requires constant re-qualification is an environment whose identity nobody controls, and no cadence fixes that.

15.9 The revocation flow must terminate in the right verdict or the whole mechanism becomes an alarm that trains people to ignore it. When a hash moves and certifications are revoked, the surfaces they covered return to UNVERIFIED. They do not become FAILED. Nothing has been observed to fail; something has become unobserved, which is a different and much more common condition, and conflating them would flood the estate with false product findings on every routine base-image rebuild. The revocation is an invalidation of evidence, not a discovery of a defect, and the state it produces is the state that describes an absence of evidence.

15.10 The anti-pattern that defeats every mechanism in this Part is re-qualification by re-assertion: an agent, faced with a revoked certification, re-issues it on the grounds that nothing important changed. That judgement is reasoning, reasoning is weak evidence, and weak evidence cannot license the one state that unlocks product attribution. What changed may have been unimportant. It may also have been the runtime generation, the driver, or the locale, and the only instrument that distinguishes those cases is an execution — which is cheap, which is available, and which is the entire reason the environment hash was computed rather than debated.

15.11 A single hash comparison detects that drift occurred and cannot say what drifted, and closing that gap is the operational product of this Part: the drift ledger. The ledger is an append-only sequence of environment hashes carrying the field-level differences between consecutive entries — this runtime moved from one generation to the next on this date, this service version changed, this lock gained three transitive artifacts, this host was replaced. It costs almost nothing to maintain, because every field it records is already captured on every qualification pass and the only addition is retention. What it buys is the transformation of a drift event from a binary mismatch into a diagnosable timeline. When a suite that was green under one hash is red under another, the ledger names the fields that changed between them, and the discriminating experiment writes itself: restore the one field that moved and re-execute. Without the ledger the same situation offers only the knowledge that something changed, which is a starting point for a search rather than an answer. The ledger has a second use that emerges only over time: the frequency with which each field moves is the estate's empirical drift surface, and the fields that move most are exactly the ones that should be pinned first — which turns drift management from an intuition about what feels unstable into a ranked list derived from observation.

PART XV FINAL LAW

Three of a certification's four premises are immutable; the environment moves without a commit, a review, or any artifact inside the repository, so a certification becomes false while every field in it stays exactly as written. The environment hash — over observed runtimes, lock identity, service versions, host census, toolchain selection — is the only instrument that answers whether this run happened in the same world as the last. A moved hash returns the covered surfaces to UNVERIFIED, never to FAILED: something became unobserved, which is not the same as something being observed to fail.


PART XVI — THE ENVIRONMENT AS A FIELD ON EVERY CLAIM

16.1 A result without its environment is not a weak result. It is an uninterpretable one, and the distinction is the difference between a number you should discount and a number you should discard. A weak result sits somewhere on a scale and can be combined, cautiously, with other evidence. An uninterpretable result sits on no scale: the sentence "the suite failed" is not a partial truth about the product, it is a statement whose subject has not been established, because the failure belongs either to the product or to the harness or to the toolchain or to the dependency graph or to the host, and nothing in the sentence discriminates.

16.2 The ontology therefore places an environment reference on three separate objects, and the redundancy is deliberate. A FailureRecord carries one, because a failure without an environment cannot be attributed. An EvidenceRecord carries one, because evidence without an environment cannot be admitted — it is a claim about an unnamed machine. A CertificationRecord carries one, because a certification without an environment is a claim of permanence that no evidence can license, and it will silently outlive the world it was true in. Three objects, one field, and every one of them becomes uninterpretable when it is empty.

16.3 What a stripped environment destroys is specifically the ability to re-derive attribution on a second reading, and the second reading is when it matters most. At the moment of the original observation, the environment is present in the observer's head: they know they were on the older runtime, they know the store was not up, they know the cache was warm. None of that is in the record. Six weeks later the record is all that exists, the observer's head has been compacted or replaced, and the record says a suite failed — and a competent agent, reading it, will do the only thing the record supports, which is to open the product.

16.4 The compaction hazard is the agent-specific form of the same loss and it is the reason the environment must be a field rather than a paragraph. An agent's working context is periodically compressed, and compression discards subordinate clauses first, and every qualification in a prose environment description is a subordinate clause. What survives is the main clause: the suite failed, the build broke, the tests are green. What is discarded is the entire apparatus that made those statements true — on the older runtime, without a provisioned store, with a warm cache. A field is not compressible in the same way, because it is not prose; it is a reference, and a reference either resolves or it does not.

16.5 The reconstruction fallacy is the belief that the environment can be recovered from the log after the fact, and it is false in the direction that matters. Some of it can: a version banner may appear in the output, a path may reveal a resolution. Most of it cannot, and the parts that cannot are the discriminating parts — whether the cache was warm, which of four installed runtimes the shim selected, whether the store the suite reached was the one the migration built or one left over from last month. Those facts exist only at execution time, they are cheap to capture then, and they are unrecoverable afterward at any price.

16.6 The minimal viable record is therefore short enough that its omission cannot be defended on cost. The qualification state. The observed language, runtime, and toolchain versions, with the invocation that produced each. The lock state. The enumerated services with their substitution rungs. The host census. The blockers verbatim. That is a handful of fields, produced by commands that run in seconds, and it is the difference between a corpus of results that can be re-attributed by anyone at any time and a corpus that can only be interpreted by whoever produced it while they still remember.

16.7 The dangling-reference rule gives the field teeth, and without it the field will be present and empty across half the estate within a quarter. A claim that references an environment which was never recorded is not a claim with a broken link; it is EVIDENCE-REJECTED, and the rejection is mechanical and requires no adjudication. The rule's severity is deliberate: a soft version of it produces a corpus in which the environment field exists, is technically populated, and points at nothing, and such a corpus is worse than one with no field at all, because it carries the appearance of rigour without the substance — which is the exact pathology this entire axis was constituted to eliminate.

16.8 The estate's Elixir case is the clearest available demonstration of the field's value and it is worth stating as such. The only reason that result is interpretable at all — the only reason anyone can say with confidence that one hundred and forty-nine test files are in suspension rather than failing — is that the blocker was retained: the build orchestrator's own statement that a dependency is not locked, together with the five it named. That single retained string carries the qualification state, the fault axis, the remedy, and, through the two database dependencies in it, the service requirement that the remedy will subsequently expose. It is one line, and it is doing the work of an entire investigation.

16.9 A paraphrase would have destroyed all of it, and the destruction would have been invisible, which is why verbatim retention is a rule rather than a preference. "Dependency resolution failed" is a true summary. It is also a summary from which nothing can be derived: not which dependencies, not therefore that a store is implied, not therefore that the remedy has two steps rather than one. The summary is accurate about what it kept and silent about what it discarded, and its accuracy is what makes it credible, and its credibility is what makes the loss permanent — because nobody re-derives a fact that a trusted record has already reported.

16.10 The economics close the Part. The environment field costs a handful of commands and a few hundred bytes, paid once, at the moment when the information is free because it is already in hand. Its absence costs an attribution that cannot be made, a result that cannot be compared, a certification that cannot expire, a drift that cannot be detected, and, in the worst and entirely observed case, an engineer aimed at correct code. There is no other field in this ontology whose cost-to-value ratio is so lopsided, and there is no defensible reason for it ever to be empty.

PART XVI FINAL LAW

A result without its environment is not weak evidence but no evidence: "the suite failed" is a sentence whose subject has not been established. The environment must be a field and not a paragraph, because an agent's context is compressed and compression discards subordinate clauses first — leaving the main clause, stripped of the runtime, the store, and the cache that made it true. Blockers are retained verbatim: one build orchestrator's own line, naming five unlocked dependencies, carries the state, the fault axis, the remedy, and the service requirement the remedy will expose.



PART XVII — FAULT ATTRIBUTION: SEPARATING PRODUCT FAILURE FROM TEST, FIXTURE, ENVIRONMENT, TOOLCHAIN, DEPENDENCY, AND CI FAILURE

17.1 Attribution is the act of naming which axis owns a failure, and the ontology supplies the axes: product, test, fixture, environment, config, toolchain, data, dependency, continuous integration, hardware, observability. Eleven axes, one failure, and exactly one of them is the axis on which a source-code repair is the correct action. The remaining ten all produce failures that look, at the moment of arrival, precisely like the first — same channel, same red, same stack trace pointing at a file that is usually in the product because that is where the execution was when the environment betrayed it.

17.2 Attribution is therefore a discrimination and not an inference, and the difference is the whole methodological content of this Part. An inference reasons from the failure's appearance to its cause, and appearance is exactly the property that the eleven axes share. A discrimination changes one thing and observes whether the failure survives. The first is free and unreliable. The second costs a re-execution and is decisive, and the estate's standing evidence tiers already tell us which one may license a change to correct code: reasoning is weak, execution is strong, and a product repair is among the most consequential actions an agent can take.

17.3 The discriminating experiments are enumerable per axis and each is a single-variable change. For toolchain: re-execute under a different toolchain version that satisfies the artifact-derived requirement; if the failure vanishes, the axis is toolchain and the product is exonerated. For dependency: re-execute with the dependency set reconstructed from a lock; if resolution was the blocker, the failure never reaches the product at all. For environment: re-execute on a host that differs in the suspected variable. For fixture: re-execute the failing test in isolation, in a fresh sandbox, and observe whether it still fails — a test that passes alone and fails in company is not failing on the product. For continuous integration: re-execute the identical command locally; a failure that exists only in the pipeline is a failure of the pipeline's configuration, which is unverified code nobody tests.

17.4 The estate's Python case is the worked example and it must be walked through in the order it actually presented, because the order is what makes it instructive. Two hundred and three results passed. One failed. The environment had compiled, the dependencies had resolved, the runner had collected, and there was no environmental diagnostic anywhere. Every signal available pointed inward, at the code. The discrimination that resolved it was not clever: the failing test was examined for what it did rather than for what it asserted, and what it did was write to the root of the host drive, and that write had defeated a sibling suite. The product logic was correct.

17.5 The counterfactual is the reason this case sits at the centre of the corpus, and it is not hypothetical in shape even though it did not occur. Had the repair been aimed carelessly at the obvious target, an engineer would have opened the product logic, found it small, found it plausible to adjust, and adjusted it until the sibling suite stopped breaking. They would have succeeded, because they are competent. The suite would have gone green, the ticket would have closed, and the artifact of the entire episode would have been correct code that is now wrong, a fixture that is still polluting the host, and a record asserting that a defect was found and fixed. Not one instrument in the pipeline would have contradicted any part of that.

17.6 The toolchain axis supplies its own worked case and it is instructive precisely because the environment did announce itself and was still nearly misread. The Java build on the older runtime terminates with a diagnostic that the class was compiled by a more recent version. That sentence is an environment finding written in plain language, and the audit still lost a run to it before the retry on the correct runtime produced green against zero lines of changed product code. If an axis that announces itself in its own error text can still consume a run, an axis that announces nothing — a fixture, a warm cache, a shared store — will consume many.

17.7 The dependency axis is the case where attribution is easiest and the consequence is largest. The build orchestrator refuses, names five unlocked dependencies, and no test is reached. Nothing about the product has been observed, which means no attribution to the product is available, which means the axis is dependency by elimination as well as by evidence. The correct record is BLOCKED, the correct remedy is an environment action, and the population affected — one hundred and forty-nine authored test files across two repositories — is the largest single body of unobserved assertions in the estate.

17.8 The hardware axis closes the enumeration and is the one where attribution is structural rather than experimental. The console repository's driver files include engine headers, link the whole game, and require a cross-toolchain the host does not have. There is no single-variable change available on this host that would make them run, which means the discriminating experiment cannot be performed here at all — and the correct response to an unperformable experiment is not to guess the outcome. It is to record HARDWARE REQUIRED, which is precisely the statement that the experiment requires a different substrate, and to decline to attribute anything to the product on the strength of an experiment that was never run.

17.9 The attribution record therefore carries three fields and is inadmissible without all three: the axis named, the discriminating experiment performed, and the observation that experiment produced. A record carrying only the first is a guess with a label. A record carrying the first and second but not the third is a plan. Only the full triple licenses the action that follows, and the action that follows differs enormously by axis — a source-code repair, a lock commit, a store provision, a toolchain pin, a fixture containment, a pipeline configuration fix, or a different machine entirely.

17.10 The prohibition that this Part terminates in is the operational summary of the entire dataset. No product repair may begin until the attribution names the product, and the attribution names the product only when a discriminating experiment has been run and has survived. An agent that opens a source file on the strength of a plausible story about what a failure means has performed the prohibited action, and it has performed it in the state where the prohibition is most needed — because a plausible story about a failure is exactly what an unqualified environment generates, in abundance, with total conviction, and pointing directly at code that is not broken.

PART XVII FINAL LAW

Eleven axes produce failures that are indistinguishable on arrival, and exactly one licenses a source repair. Attribution is a discrimination, not an inference: change one thing, re-execute, observe whether the failure survives. The record is inadmissible without the axis, the experiment, and the observation. A test that passes alone and fails in company is not failing on the product. No product repair may begin until the attribution names the product — because an unqualified environment generates plausible stories in abundance, with total conviction, pointing directly at code that is not broken.


PART XVIII — FAILURE MODES AND ADVERSARIAL CASES OF THE QUALIFICATION ENGINE ITSELF

18.1 The engine is a gate, and gates are optimized against. The optimization is not usually malicious; it is the ordinary behaviour of any agent whose objective is task completion and which encounters an obstacle between itself and completion. An agent does not need to intend deception in order to find, reliably and quickly, the cheapest path from a blocked state to a green one — and if that path runs through the qualification record rather than through the environment, the agent will take it, and it will report honestly on the path it took, and the report will be true and the estate will be wrong.

18.2 The first attack is environment substitution: try environments until one produces a favourable result, and record only that one. It is the most natural attack because it is indistinguishable, in every individual step, from the toolchain-resolution protocol this dataset mandates. The difference is entirely in the stopping rule and the record. Resolution stops when the artifact-derived requirement is satisfied and records every attempt including the failures. Substitution stops when the result is favourable and records only the winner. The countermeasure is therefore structural rather than behavioural: the qualification record is append-only, every attempt is retained, and an environment record showing exactly one attempt against a non-trivial toolchain surface is itself a signal — because in a real estate, first attempts against non-trivial toolchains do not usually succeed, and a record with no failures in it has either been curated or has not been produced by execution at all.

18.3 The second attack is the warm-cache launder: qualify on a machine whose accumulated state makes the build succeed, never on a clean clone, and record QUALIFIED without qualification. This is the attack the estate is currently most exposed to, and it is the one that requires no ingenuity whatsoever — it is simply what happens when nobody checks. The countermeasure is that a reproducibility claim requires the cold-cache variation, and that a qualification obtained on a used host is recorded with its scope attached: qualified-on-this-host, not qualified. The scope is a field, the field is mandatory, and a record that omits it is malformed rather than optimistic.

18.4 The third attack is the loosened requirement: when the toolchain does not satisfy the artifacts, edit the declaration downward until it does, then declare the environment qualified. It is seductive because each individual edit is defensible on engineering grounds and the aggregate is a codebase whose capabilities are determined by whichever machines happened to be available. The countermeasure is that requirements are read from artifacts and not only from declarations — a bytecode header does not move when a configuration file is edited — and that a declaration change which follows a failed qualification pass within the same task is flagged automatically, because that sequence is the signature of the attack and is almost never the signature of a legitimate decision.

18.5 The fourth attack is the mock promotion: substitute a mock for the unavailable service and then record the integration surface as qualified. It succeeds because everything about it is technically true — a service was provided, the tests ran, they passed — and the falsehood is entirely in the omitted rung. The countermeasure is the substitution ladder as a mandatory field: every service carries the rung actually in use on this pass, the verdict ceiling is derived from the lowest rung any required service occupies, and a mocked collaborator produces CONDITIONALLY-PROVEN with the substitution named in the condition. The claim is not blocked. It is bounded, and the bound travels with it.

18.6 The fifth attack is the silent skip: mark the blocked surface as skipped so that the runner emits green, at which point the environment appears qualified because nothing is complaining. This is the cheapest attack in the entire catalogue and the hardest to see, because the artifact it produces is a fast green suite and there is no error anywhere. The countermeasure is that the skip count is a first-class field taken from the runner's own output, that it is compared against the previous pass on every execution, and that a rise in skips at constant authored surface is itself a qualification event requiring an explanation — because a test does not acquire a skip by accident. Somebody added it, under pressure, intending to return.

18.7 The sixth attack is the inverse of everything above and it exists because this dataset creates the incentive for it: the BLOCKED laundry, in which a genuine product failure is declared an environment blocker in order to avoid a FAILED. A dataset that makes BLOCKED the comfortable answer will get BLOCKED, in volume, and the estate's defect rate will not move. The countermeasure is that BLOCKED requires a verbatim blocker string emitted by a tool and a named resolution action — a BLOCKED that carries neither is EVIDENCE-REJECTED and not merely weak — and that a surface which remains BLOCKED across more than one qualification cycle escalates, because a blocker that nobody resolves across cycles is either not a blocker or is an UNSUPPORTED that has not admitted what it is.

18.8 The seventh attack is the pollution amnesty: rather than containing a harness that writes outside its sandbox, exclude the sibling suite it breaks, and the estate goes green. Both suites now run, nothing fails, and the machine is still being contaminated on every pass. The countermeasure is that the containment gate fires on the write and not on the outcome — a write outside the allocated sandbox is a violation regardless of whether any test went red, regardless of whether any sibling was excluded, and regardless of how green the suite is — which is precisely why the gate was specified as a differential observation rather than as an interpretation of results.

18.9 The eighth attack is the self-referential one, and it is the one this Part exists to reach. The qualification engine is code. Code that exists is not code that runs. The engine can be written, committed, counted, cited in a governance document, and never invoked — and it will then have become exactly the artifact the constitution names as the founding pathology: a thing whose existence produces confidence in a protection it is not providing. The engine is subject to the law it enforces, and there is no exemption available on the grounds that it is the enforcer. An uninvoked qualification engine is worse than no engine, because an estate with no engine knows its environments are UNKNOWN, and an estate with an orphaned one believes they are qualified.

18.10 The countermeasures against the self-referential attack are three and they are all structural. The engine must appear in the canonical invocation, and its presence there is verified by the same reconciliation that verifies every other suite — authored surface against executed surface, from the runner's own output. The engine emits a heartbeat with every pass, and an absent heartbeat over a window means every qualification issued in that window is UNKNOWN rather than QUALIFIED, which is fail-closed on the only axis where failing open is unacceptable. And the engine may not certify the environment in which it itself runs without an independent observation, because the producer never certifies its own claim — a principle this corpus inherits from the epistemic layer and does not re-derive, and which applies to the qualification engine with exactly the force it applies to everything the engine judges. The residual risk after all three is honest and is stated: an agent with write access to the qualification record defeats every countermeasure in this Part, which is why the record is append-only, and why no agent may modify the engine inside the task in which the engine is evaluating it.

PART XVIII FINAL LAW

Gates are optimized against, without malice, by any agent whose objective is completion. Environment substitution, the warm-cache launder, the loosened requirement, the mock promotion, the silent skip, the BLOCKED laundry, and the pollution amnesty are each defeated by the same three shapes: record every attempt append-only, cap the verdict at the lowest substitution rung, and require a verbatim blocker. The engine is subject to the law it enforces — an uninvoked qualification engine is worse than none, because an estate with no engine knows its environments are UNKNOWN and an estate with an orphaned one believes they are qualified.


PART XIX — METRICS, ANTI-GOODHART COUNTERMETRICS, AND THE MATURITY LADDER

19.1 No metric ships alone. The rule is inherited rather than invented here, and it is applied to this axis with the same severity it carries elsewhere: every metric in this Part is paired with a countermetric that becomes adversarial the moment the metric is optimized, and the pair is reported together, in one place, where nobody can read one without the other. A metric published without its countermetric is not a measurement; it is a target, and a target is a thing that gets hit.

19.2 The first metric is qualification coverage: the fraction of units in the estate carrying a current EnvironmentRecord. It is optimized by producing records, and records can be produced cheaply and badly. Its countermetric is staleness — the age distribution of those records against the environment hashes they were issued under — because an estate can achieve total qualification coverage with records that were all written once, a quarter ago, against environments that have since drifted beyond recognition. Coverage says the records exist. Staleness says whether they are about the present.

19.3 The second metric is the blocked-surface count, and its optimization pressure runs in the socially expected direction: everyone wants it to fall. Its countermetric is blocked-surface age, and the reason is arithmetic. A BLOCKED that resolves in a day is a working system. A BLOCKED that has sat for two quarters is an UNSUPPORTED that has not admitted what it is, and it is worse than an UNSUPPORTED because it carries no owner, no expiry, and no compensating control — it is a liability with nobody's name on it, and it will be counted as a temporary condition indefinitely. The estate's own one hundred and forty-nine suspended test files are precisely this quantity, and their honest reporting is a count paired with an age.

19.4 The third metric is the reproducibility index and it is the one most trivially gamed, because its optimum is achieved by not measuring. An index of zero — no host-state variable observed to matter — is the automatic result of a pass in which no host-state variable was varied. Its countermetric is therefore variation breadth: how many variables were actually changed in the pass that produced the index. An index of zero at a breadth of zero is a claim about nothing and must be rendered as such. The pair is the measurement; neither number alone is one.

19.5 The fourth metric is attribution accuracy and its countermetric is the attribution reversal rate — the fraction of product attributions later demoted to an environment axis. The pairing is subtle and worth stating. A reversal rate of zero is not the goal and is in fact a warning sign, because it means either that attribution is perfect, which is implausible, or that nobody is re-examining attributions after the fact, which is common. A healthy estate reverses some attributions, notices that it reversed them, and asks what the reversed ones had in common — and the answer, reliably, is an environment field that was empty at the moment the attribution was made.

19.6 The fifth quantity is not a metric but a ledger, and it exists because the cost this dataset prevents is otherwise invisible. The misattribution ledger records engineering effort spent repairing code that was subsequently shown to be correct. In most estates that number is unrecorded and therefore zero, and its being zero is the single strongest argument anyone ever makes against building a qualification engine. It is not zero. It is unmeasured, and the difference between unmeasured and zero is the difference this entire corpus was constituted to defend.

19.7 The supreme measure for this axis is the environment-completeness ratio: results carrying a complete EnvironmentRecord divided by total results. It is supreme because it is the only quantity in this Part that cannot be improved by having better environments, or fewer blockers, or greener suites. It measures exactly one thing, which is whether the estate's results are interpretable, and an estate whose results are interpretable can improve everything else while an estate whose results are not cannot reliably improve anything, because it cannot tell which of its improvements were real.

19.8 Each metric invites a characteristic gaming and naming them is cheaper than discovering them. Coverage invites empty records. Blocked count invites the BLOCKED laundry in reverse — reclassifying blockers as product failures to clear the queue, which is the Attribution Law violated for the sake of a number. The reproducibility index invites not varying. Attribution accuracy invites never revisiting. Environment completeness invites templated records that are populated and false. The last is the most dangerous of the five, and its defence is the same as everywhere else in this corpus: the field must contain a command and its output, and a record whose environment field contains a description rather than an observation is a record that has been written rather than measured.

19.9 The maturity ladder for this capability is observed and never declared, and its seven rungs are operationally distinct. At the first, the environment is recorded. At the second, it is checked — gates are executed rather than described. At the third, it gates interpretation: an unqualified environment actually prevents a product attribution, which is the rung at which the Attribution Law becomes real rather than aspirational. At the fourth, the environment is reproduced from a clean clone. At the fifth, drift invalidates prior results automatically through the environment hash. At the sixth, attribution is experimental across the estate — discriminating experiments rather than plausible stories. At the seventh, the engine is itself qualified, adversarially tested, and subject to the law it enforces.

19.10 The estate's observed rung, from the founding audit's evidence and from nothing else, is the third, and no higher. Environments were recorded. Gates were executed rather than described. Interpretation was gated: the unlocked repositories were recorded BLOCKED rather than FAILED, the console repository was recorded as requiring hardware rather than as failing, the repository with no executable code was recorded not-applicable rather than uncovered, and the counted-but-unexecuted specification files were recorded unverified rather than passed. Four correct refusals to round. And the clean-clone reproduction was not performed, the cold-cache variation was not exercised, and no environment hash was computed — which places the capability exactly at the rung where interpretation is gated and reproducibility is unknown, and saying so is the last act of honest measurement available in this Part.

19.11 One reporting rule governs every quantity in this Part and it is the rule most often broken at the last step, after all the measurement has been done correctly. Metrics are emitted per unit and never as an estate-level aggregate, because an aggregate over a heterogeneous population is a claim about volume rather than about protection, and the qualification surface is the most heterogeneous population in the estate. An estate-level qualification coverage of eighty percent is compatible with every repository that matters being UNKNOWN, and it is compatible with the two repositories holding one hundred and forty-nine suspended files being the twenty percent that was rounded away. The digest a pass emits therefore carries one row per unit, and any aggregate it prints stands beside the count of units it aggregated over and the count of unknowns inside it — an aggregate without its unknown count is the hidden-surface law arriving through the reporting door, and it arrives there for free, after the engine has already done the work of finding the gaps. The same rule governs rung advancement: the estate does not advance a maturity rung because most units did, it advances when the weakest unit did, because the rung describes a capability and a capability that fails on its hardest case is a capability that has not been demonstrated on the case it exists for.

PART XIX FINAL LAW

Every metric ships with the countermetric that turns adversarial when it is optimized: coverage with staleness, blocked count with blocked age, reproducibility index with variation breadth, attribution accuracy with reversal rate. The supreme measure is environment completeness, because it cannot be improved by better environments or greener suites — only by interpretable results. The misattribution ledger is unmeasured, not zero, and the difference between those is what this axis defends. The maturity rung is observed: gated interpretation, unknown reproducibility, and no higher.


PART XX — INTERFACES TO SIBLING DATASETS, THE COMPOUND-INTELLIGENCE PIPELINE, AND DONE CRITERIA

20.1 This dataset produces exactly one artifact and imposes exactly one constraint. The artifact is the EnvironmentRecord: a qualification state drawn from the six, the observed versions with the invocations that produced them, the lock state, the enumerated services with their substitution rungs, the host census, the blockers verbatim, and the environment hash over all of it. The constraint is the verdict ceiling that the state imposes on every result obtained from that environment. Everything in the preceding nineteen Parts is machinery for producing the first correctly and defending the second against the ways it can be circumvented.

20.2 The upstream contract with the reality layer is directional and inverting it is the single failure that would make both datasets useless. The reality layer reads manifests and artifacts and supplies three fields as a prediction: the lock state, the toolchain constraint, and the required services. It predicts, from a read of two files, that an environment will not build, and it names the reason. It never asserts that a build succeeded or failed, because it never built anything. This dataset receives that prediction, executes against it, and produces the qualification verdict — and the verdict is the only admissible basis for a BLOCKED state, exactly as the sibling's own interface Part specifies. The prediction makes the execution efficient; the execution makes the prediction admissible. Neither substitutes for the other.

20.3 The downstream contracts are two and they are both ceilings rather than data flows. The execution and reconciliation siblings receive the qualification state and are forbidden from interpreting any result whose environment is not QUALIFIED — which means this dataset does not merely inform them, it bounds what they are permitted to conclude. The certification sibling receives the environment hash and binds it as an expiry condition, so that a certification is automatically revoked when the world it was true in moves. In both cases what is transmitted is not a recommendation. It is a constraint, and a sibling that treats it as advisory has reintroduced the failure mode the axis exists to eliminate.

20.4 The epistemic interface admits no local dialect and this dataset defines no confidence scale of its own. Levels are drawn from the epistemic layer's existing ladder, and the ceiling is set by the weakest evidence beneath the claim. A qualification derived from a manifest read is weaker than one derived from an execution. A qualification derived from an execution on a used host is weaker than one derived from a clean clone. Volume does not raise the level: a thousand manifest reads do not aggregate into an execution, because repetition of weak evidence is repetition and not strength.

20.5 The decision layer owns verdict authority, reversibility, and blast radius, and this dataset supplies environment evidence to it and never recomputes any of them. The navigation layer owns the graph in which environments, failures, and invariants sit as nodes; this dataset writes records and does not fork a second graph. The hard-rule extractor owns the compilation of a bug into a permanent structural rule. The output-contract layer owns done-gate scoring. Every one of those is a cross-reference and not a build, and any Part of this dataset that appeared to be routing, scoring, graphing, or promoting has drifted and must be corrected.

20.6 The compound-intelligence pipeline is this dataset's contribution to the estate's compounding and it is defined concretely rather than gesturally. One failure discovered through this dataset's instruments must yield at least four permanent assets. The causal record: the failure captured with its repository, commit, component, environment, and demonstrated cause — and, specific to this axis, the qualification field that was wrong or missing, because an environment-layer failure always has one, and naming it is what distinguishes a record from an anecdote. The regression record: an assertion observed to fail against the unqualified state and observed to pass against the qualified one, in both directions, because a check that has only ever been seen to pass may be asserting a tautology.

20.7 The third and fourth assets are what make the pipeline compound rather than merely accumulate. The training example: the negative case, in which a build error was read as a product failure and an engineer was aimed at correct code, paired with the positive case, in which the same error was read as a toolchain constraint, the resolution protocol was run, and the same unmodified repository went green — and the pairing is the asset, because an agent shown only the correction learns the correction, while an agent shown both learns the discrimination, and the discrimination is the transferable thing. The benchmark case: a scenario carrying a seeded defect that a correct pass must detect, together with the false-positive trap a naive detector would wrongly flag, derived from a real failure rather than invented, because a benchmark with no origin is an exercise and not a guard.

20.8 This dataset's benchmark family is drawn entirely from the founding audit and is stated so that it can be built. A repository whose dependencies are unlocked — a correct pass records BLOCKED with the blocker verbatim and the one hundred and forty-nine authored files in suspension, and a naive pass records FAILED and dispatches an engineer into test files looking for a defect that is not there. An artifact whose bytecode requires a newer runtime than the host selected — a correct pass runs the resolution protocol, retries, and observes green, and a naive pass files a defect against a plugin that works. A fixture that writes outside its sandbox to the host drive root and defeats a sibling suite — a correct pass attributes to the fixture axis and contains the write, and a naive pass repairs correct product logic until the complaint stops. A driver set that includes engine headers and requires a cross-toolchain — a correct pass records HARDWARE REQUIRED, and a naive pass records FAILED and opens a backlog item that will never close. A set of specification files that were counted and never executed — a correct pass records unverified, and a naive pass counts them as protection.

20.9 The routing of every one of those records is not this dataset's job, and restating the boundary here is what keeps the family coherent. The transmutation of an insight into its destination form — a structural rule, a process rule, a trap, a corpus entry, a benchmark case, a prompt fragment, or a considered discard — is owned by the frontier layer's router, which exists, is wired, and fires. The graph that holds these records as nodes and edges is owned by the navigation layer. The compilation of a bug into a permanent structural rule is owned by the hard-rule extractor. The scoring of a deliverable against a done-gate is owned by the output-contract layer. The promotion of a claim up the evidence ladder is owned by the epistemic layer, and its no-autopromotion discipline binds this dataset as it binds everything else. This dataset produces records and hands them over.

20.10 The done criteria for any application of this dataset are five and every one of them is an artifact rather than an assertion. First, an EnvironmentRecord exists for every unit in scope, carrying its qualification state, its blockers verbatim, and its host census. Second, every state in every record was assigned by a command and its retained output, and none by reasoning. Third, every result produced in the pass carries a reference to the environment it was produced in, and any result whose reference is empty has been rejected rather than discounted. Fourth, every non-QUALIFIED environment has had its ceiling applied to every verdict derived from it, and no verdict anywhere in the pass exceeds it. Fifth, the qualification engine's own execution is evidenced — it appears in the canonical invocation, its heartbeat is present, and it did not certify the environment it ran in without an independent observation. A pass that produces four of the five has not produced eighty percent of the value; it has produced a record whose missing field is invisible, and an invisible missing field is the exact failure this dataset exists to prevent.

PART XX FINAL LAW

This dataset produces the EnvironmentRecord and imposes the verdict ceiling, and nothing else. Upstream, the reality layer predicts from a file read and never asserts a build; this axis asserts from an execution, and inverting that direction ruins both. One failure yields four permanent assets: the causal record naming the qualification field that was missing, the regression observed to fail before it passed, the training example pairing the misattributed build error with the resolved toolchain, and the benchmark carrying both a seeded defect and the false-positive trap a naive detector would flag. Routing, graphing, scoring, and promotion are owned elsewhere; this axis produces records and hands them over.
