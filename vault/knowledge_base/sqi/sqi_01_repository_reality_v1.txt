PART I — MISSION: THE DISK OUTRANKS EVERY DESCRIPTION OF THE DISK

1.1 This dataset governs the first act of any verification pass, and the first act is not running a test. It is establishing what exists. Every subsequent judgement — which suites matter, which surfaces are unprotected, which environment must be qualified, which verdict is honest — is a function of an inventory, and an inventory that is wrong is not a slightly degraded input. It is a poisoned one, because every downstream conclusion inherits its error while acquiring the appearance of rigour from the machinery applied on top of it. A meticulous audit of a repository that is not the repository on disk produces findings that are internally consistent, professionally formatted, traceable to a method, and entirely fictional. The rigour does not correct the premise. It launders it.

1.2 The constitution establishes that observed reality on disk sits at the top of the authority hierarchy, above reproducible evidence, above verification, above contracts, above behaviour, above test output, above implementation claims, above agent reasoning, above human expectation. This dataset is the operational consequence of that placement. It answers the question the hierarchy leaves open: what does it actually mean to consult the disk, in what order, with what instruments, and how does one know when the consultation is complete. A hierarchy that names the disk as supreme and supplies no procedure for reading it has moved the problem rather than solved it, because in the absence of a procedure the agent will consult whatever description of the disk is nearest to hand, which is usually the plan.

1.3 The distinction being drawn is between a repository and every artifact that purports to describe one. A repository is a set of files with contents. A description of a repository is a claim about that set, and the set of available claims is large and mutually reinforcing: the name of the directory, the position of the project in the estate, a line in a task list, a paragraph in a prior session's handoff, an entry in a project index, a summary written by an agent three contexts ago, and the confident memory of the human who created it. Every one of those is a hypothesis about the disk. None of them is the disk. The failure this dataset exists to kill is the substitution of any of them for the disk, and the substitution is attractive precisely because the descriptions are cheap and the disk is not.

1.4 The asymmetry of cost is the entire mechanism. Reading a name takes no time. Reading a plan takes seconds. Walking a tree, identifying build descriptors, resolving a dependency manifest, discovering a runner configuration, and enumerating the executable entry points takes minutes, and minutes are what an agent under pressure is optimizing away. The result is not a decision to trust the description. It is the absence of a decision: the description is consulted because it arrived first and cost nothing, and the disk is never consulted at all, and no moment ever occurs at which anyone chose to skip it. Governance that depends on remembering to look is not governance, and so the reality scan is not a recommendation in this corpus. It is the precondition of every claim that follows it.

1.5 In the audit that founded this corpus, the substitution occurred and was caught, and the record of it is the reason this Part is first. A repository in the estate was carried in the plan as a project of a particular executable kind on the strength of its name and its neighbourhood. The plan was not careless; the inference was reasonable, and a competent human would have drawn it. The disk contradicted it flatly: there was no source of that kind anywhere in the tree, no build descriptor of that kind, and no executable surface of any kind at all. The correct classification was a content and knowledge repository whose product is documents, not programs. Had the plan been trusted for one more step, the audit would have generated a finding about missing tests for logic that does not exist.

1.6 That counterfactual deserves to be traced to its end rather than left as an abstraction, because its cost is not the fabricated finding. The fabricated finding would have been filed. It would have entered a tracker with a severity and an owner. It would have been assigned. An engineer would have opened the repository intending to write the missing test, would have found nothing to test, and would have concluded — reasonably — that they had misunderstood the finding, and would have asked, and the ambiguity would have been resolved by someone reasoning from the same plan that produced the error. The finding would have survived one full round of human scrutiny, because every participant was reasoning from the description and the description was self-consistent. A fabricated finding is not a wasted hour. It is a durable false belief with an audit trail attached.

1.7 The mission of this dataset is therefore stated as a supremacy rule with a procedure behind it. Before any claim about a repository is made, formed, recorded, or acted upon, the repository is scanned, and the scan's output — not the plan, not the name, not the memory, not the prior summary — is the substrate on which every subsequent operation runs. Where the scan and the description disagree, the scan wins immediately, the disagreement is itself recorded as a finding, and the description is treated as compromised until someone determines why it was wrong. A description that was wrong once about a repository is not corrected by the correction; it is evidence that the process which produces descriptions has a defect, and that defect is more valuable than the individual error.

1.8 The scan's authority is bounded, and the bound must be stated with the same force as the authority, because an unbounded scanner becomes an oracle and an oracle is not auditable. The scan establishes what exists and what kind of thing it is. It does not establish that the code is correct, that the tests are adequate, that the architecture is sound, or that the risks are covered. Those are the subjects of sibling datasets and of the pipeline that consumes this one. The scan produces an inventory and a classification and an executable-surface map, each with an explicit unknown state where the disk did not answer, and it produces nothing else. Its value lies in being narrow, cheap, and impossible to argue with, and each of those properties is lost the moment it begins to opine.

1.9 The relationship to the existing premise-verification layer must be named here so that this Part is not mistaken for a second implementation of it. This estate already possesses a standing rule that forbids writing code against an assumed file path, function, class, or signature, and a verifier that enforces it by returning the real public interface when a premise fails. That layer operates at the granularity of a symbol: does this function exist, with this shape. This dataset operates at the granularity of a repository: does this project exist, of this kind, with these surfaces. The two are the same discipline at two altitudes, they compose, and the correct behaviour when a claim about a symbol is in play is to invoke the verifier rather than to reproduce it here.

1.10 The final framing is the one that makes the discipline cheap enough to actually perform. Reading the disk is not an act of suspicion toward the person who wrote the description. It is an acknowledgement that descriptions decay, that repositories change without their descriptions changing, that a name chosen at inception encodes an intention rather than an outcome, and that the gap between the two widens monotonically with time and with nobody's fault. The plan was right when it was written. The disk is right now. Verification is the business of now, and the instrument that reports on now is the only instrument this axis is permitted to trust.

PART I FINAL LAW

The disk outranks every description of the disk — the name, the plan, the index, the prior summary, and the confident memory — because each of those is a hypothesis and only one of them is an observation. A meticulous audit of a repository that is not the repository on disk produces findings that are consistent, traceable, and fictional; the rigour applied on top does not correct the poisoned premise, it launders it. Scan before claiming. Where scan and description disagree, the scan wins, and the disagreement is itself a finding.


PART II — THE PREMISE-DISPROOF DISCIPLINE

2.1 A premise is a claim that is being used rather than examined, and the danger of a premise is precisely proportional to how obviously true it seems. Nobody verifies that the sky is above them. In a software estate, the premises that go unexamined are the ones that are load-bearing for the plan and were therefore never questioned by the person who wrote it: that this repository contains the language its name implies, that this suite is the suite that runs, that this test file is collected, that this scanner is invoked, that this dependency is resolved. Each of those has been observed to be false in this estate. None of them was ever doubted before it was checked, and the checking took, in each case, under a minute.

2.2 The discipline in this Part is therefore not scepticism, which is a disposition and cannot be scheduled, but disproof, which is a procedure and can be. Disproof means that before a plan is executed, its factual premises are enumerated explicitly, converted into observations that the disk can answer, and checked. The enumeration is the hard part and the part that is skipped, because a plan does not announce its premises; it embeds them in its verbs. A plan that says to audit the test coverage of a plugin project has asserted that a plugin project exists, that it has source, that it has tests, and that it has a runner. Four premises, none stated, all checkable, and the plan reads as a single instruction.

2.3 The founding instance is the one recorded in the audit, and it is worth stating in its precise sequence because the sequence is the lesson. The plan named a repository and assigned it a domain. The domain implied a language. The language implied a build descriptor. The build descriptor implied a test framework. The framework implied a suite. And the plan's finding — that this repository had no tests for its logic — was derived four inferential steps away from a premise that no one had checked. The scan checked it in one step: there was no source of the implied language anywhere in the tree, and no build descriptor of the implied kind, and therefore no suite, and therefore no logic to be untested. Four inferences collapsed at once, because they had all been standing on the same foundation.

2.4 The finding that would have been written is a specimen worth preserving. It would have been a critical-severity gap: production logic in a plugin project with zero unit-test references. That is a real class of defect, it is the first entry in the sealed failure taxonomy, it has genuine instances elsewhere in this same estate, and it would therefore have looked exactly like the true findings sitting beside it in the report. This is the property that makes fabricated findings so expensive: they are not implausible. They are drawn from the correct catalogue, they use the correct vocabulary, and they are indistinguishable from real findings by every check except the one that consults the disk.

2.5 A second, structurally opposite instance was recorded in the same audit, and citing only the first would teach half the lesson. The plan carried a claim that a plugin repository in the estate had essentially no tests at all. The disk disproved that claim in the other direction: the repository carried hundreds of test files and, on a corrected toolchain, produced a large body of passing assertions. The premise was not merely unverified; it was false, and it was false in a way that flattered the auditor, because a repository with no tests is a simple and satisfying finding. The corrected statement was subtler and more useful: the tests exist, they pass, and they are concentrated — the great majority of the files sit inside a single component, while the surface that governs the movement of value between actors carries no test reference at all.

2.6 The two instances together establish the shape of the discipline. Premise disproof is not a bias toward pessimism, in which every claim is assumed absent until proven present. It is symmetric. It disproved an assertion of presence in the first case and an assertion of absence in the second, and in both cases the correction was more valuable than the original claim, because in both cases the corrected statement was the one that could be acted upon. An estate in which premise disproof only ever finds things missing has not implemented disproof; it has implemented a prior, and a prior that is never contradicted by the data is not a method, it is a habit.

2.7 The operational form is a small number of questions asked in a fixed order, and the fixity matters because an ordered procedure can be audited for having been performed while a disposition cannot. What kind of thing does this plan assert this repository is, and what would have to be on disk for that to be true. What executable surface does this plan assume, and where would it live. What runner does this plan assume, and what file would configure it. What suite does this plan assume, and what would collect it. Each question is answered by a search that costs seconds. A plan whose premises survive all four is a plan grounded in reality; a plan that fails any of them is a plan that must be rewritten before it is executed, not corrected while it is executing.

2.8 The correction protocol is the part that is most often botched, and botching it converts a caught error into a hidden one. When a premise is disproved, the correct action is not to quietly adjust the plan and continue. It is to record the disproof as an artifact in its own right, with the premise stated as it was held, the observation that destroyed it, and the inferences that were standing on it. The record matters because the disproved premise is almost never local. It came from somewhere: an index, a naming convention, a prior summary, a classification made once and inherited ever since. Fixing the plan repairs one execution. Recording the disproof exposes the source, and the source is what will produce the next error.

2.9 The discipline extends to the scanner's own output, and this is where most implementations stop short. A scan result is itself a claim, produced by an instrument, subject to that instrument's blind spots. A scanner that detects a language by the presence of a file extension will report a language for a repository containing a single vendored example. A scanner that detects a framework by a dependency name will report a framework that is declared and unused. A scanner that detects tests by a filename convention will count a fixture-generation helper as a test. The scan's claims are stronger than the plan's claims because they are grounded in observation, and they are still claims, and the honest posture is to record them with their basis attached so that a downstream reader can see what the instrument actually saw.

2.10 The closing instruction is a single habit that costs almost nothing and is the highest-yield practice in this dataset. Before executing any plan that names a repository, a component, a suite, or a surface, ask what would have to be on disk for this plan to make sense, and then look. The looking is the whole of it. It does not require judgement, it does not require expertise, it cannot be argued with, and it is the only step in the entire pipeline that is capable of destroying a fabricated finding before it acquires an owner, a severity, and a life of its own.

PART II FINAL LAW

A plan does not announce its premises; it embeds them in its verbs, and a single unchecked premise can carry four inferences that collapse together. The audit's founding instance: a repository classified from its name was assumed to be an executable project of a particular kind, and the disk contained no source, no build descriptor, and no executable surface at all — a content and knowledge repository. The disproof prevented a critical finding that would have been plausible, correctly catalogued, and entirely fictional. Disproof is symmetric: it must be as willing to find protection present as absent.


PART III — THE REALITY SCANNER: WHAT IS INSPECTED, AND IN WHAT ORDER

3.1 The Reality Scanner is the instrument that converts a directory into an inventory, and its design is governed by one constraint that determines everything else: it must be cheap enough that skipping it is never rational. A scanner that takes an hour will be skipped under pressure, and a scanner that is skipped under pressure is a scanner that is absent exactly when the estate is least careful. Every design decision below is subordinate to that constraint. The scan is read-only, it is bounded, it terminates, it never executes repository code, and it produces a partial result rather than failing when a region of the tree resists inspection. Its output is an inventory with explicit holes, and the holes are first-class.

3.2 The read-only property is not a nicety and is not merely about safety. A scanner that executes repository code to discover its shape has confused the inventory phase with the execution phase, and it has done so in the most dangerous possible order, because it is executing code in an environment that has not yet been qualified — the qualification being a downstream step that depends on the inventory the scanner has not yet produced. Executing a build to learn what the build produces inverts the dependency. Worse, it grants arbitrary repository code the right to run before anything is known about it, which is the same category of error as trusting a description, with a larger blast radius. The scanner reads. It does not invoke.

3.3 The ordering of the scan is derived from information density rather than from convenience, and the principle is that each stage should maximally constrain the stages after it. The first stage is the top-level shape: what is at the root, how deep does the tree go, how many directories carry the signature of an independent project. The second is the manifest layer: build descriptors, dependency manifests, lock files, and project metadata, because a manifest is the single densest artifact in any repository — it names the language, the framework, the dependencies, the entry points, and frequently the runner, in one file. The third is the source layer. The fourth is the test and runner layer. The fifth is the automation layer. The sixth is the executable-surface layer, which is the subject of a later Part.

3.4 The manifest-first ordering is the scanner's most consequential design choice and deserves its justification. A repository can be classified from its source files by extension counting, and doing so is fast and is wrong often enough to matter. Extension counting cannot distinguish a language that the project is written in from a language that appears in a vendored dependency, a generated artifact, an example directory, or a single utility script. A manifest is a declaration of intent by the project's own authors, in the project's own words, and it is machine-readable. When a manifest and an extension census disagree, the disagreement is itself a finding — it usually means vendored code, generated code, or a subproject the top-level manifest does not know about — and the finding is more valuable than either signal alone.

3.5 The absence of a manifest is a signal, not a gap, and this is the point at which a naive scanner produces its worst error. A tree with no build descriptor of any kind, no dependency manifest, no lock file, and no runner configuration is not a project whose manifest the scanner failed to find. It is, with high probability, a project that has no executable surface — a documentation set, a knowledge base, a specification corpus, a collection of prompts. The founding instance in this estate was exactly this: the scan found no build descriptor because there was none to find, and the correct conclusion was not that the scan had failed but that the repository was not the kind of thing the plan believed it to be. A scanner that treats manifest-absence as an error rather than as evidence will report a failure where it should report a classification.

3.6 The lock-file inspection is a distinct stage from the manifest inspection, and separating them is what allows the scanner to predict a blocked environment before anyone attempts to qualify one. A manifest declares dependencies. A lock file records the resolution of those declarations to specific versions. A manifest present with a lock file absent, or a lock file present and inconsistent with its manifest, is the signature of an environment that will not build, and the scanner can emit that prediction from a read of two files without ever invoking a compiler. In the audit, two repositories in the estate carried Elixir manifests declaring the usual dependency set and could not compile at all, the toolchain refusing on the ground that the dependencies were not locked. The scanner's read of the manifest and lock state predicts that outcome before a single build second is spent.

3.7 The runner-configuration stage is where the scanner earns most of its value, because it is the stage at which the difference between authored protection and executed protection first becomes visible. The scanner locates the configuration that determines what the default test invocation collects, and it records that configuration verbatim rather than in summary, because the load-bearing detail is almost always a single line. In the audit, one repository's configuration scoped test collection to a single subtree, and the effect of that one line was that hundreds of functional tests living in sibling directories were never collected by the default runner. They passed when invoked directly. They were invisible to the invocation that anybody actually ran. The scanner does not judge that; it records the scope, and the reconciliation engine downstream computes the consequence.

3.8 The test-file census is deliberately separated from the collection census, and the separation is the whole point of the exercise. The scanner counts what exists — files matching the runner's discovery convention, wherever they live in the tree — and records it as the authored surface. It separately records what the configured invocation would collect. Those two numbers are different, they are almost always different, and the gap between them is the single most diagnostic quantity the scanner produces. In this estate's own repository, a census of test files returned dozens more than the canonical invocation collects; the majority of the authored protection lives outside the command that anybody runs. That gap is not discovered by running the suite, because the suite reports truthfully on what it ran and cannot report on what it never saw.

3.9 The automation stage inspects what invokes what, and it exists because a control that is never invoked is not a control. The scanner reads the continuous-integration configuration, the scheduled-task definitions, the hook registrations, and the scripts that a human would plausibly run, and it builds a map from each configured job to the commands it executes. The map's purpose is to answer one question for every tool the repository declares: is there any job, anywhere, that invokes it. In the audit, a security scanner was found declared and configured in a repository and invoked by nothing. It was present. It was inert. Its presence had been read, by everyone who read the manifest, as protection.

3.10 The scanner's output contract is an inventory in which every field is either an observation with its basis attached or an explicit unknown, and there is no third kind of field. The prohibition on the third kind — the inferred field, the reasonable default, the filled-in blank — is what distinguishes an inventory from a description, and it is the property that makes the whole apparatus worth building. A field that says the language is unknown is useful, because it directs attention. A field that says the language is probably the one implied by the directory name is worse than useless, because it is a description wearing an inventory's clothes, and every downstream consumer will treat it as an observation.

PART III FINAL LAW

The scanner reads and never invokes: executing repository code to learn its shape inverts the dependency, because qualification depends on the inventory the scan has not yet produced. Manifest before source, because a manifest is a declaration by the project's own authors while an extension census cannot separate a project's language from its vendored one. Manifest-absence is evidence of a non-executable repository, not a scan failure. Every field is an observation with its basis attached or an explicit unknown; the inferred field is a description wearing an inventory's clothes.


PART IV — LANGUAGE, FRAMEWORK, AND DEPENDENCY-MANAGER DETECTION

4.1 Detection is the stage most often mistaken for the whole of classification, and the mistake is understandable because detection produces a satisfying answer quickly. The repository is a Python project. The repository is an Elixir project. The answer is crisp, it is usually right, and it is not what the verification axis needs, because knowing the language tells you almost nothing about which tests are meaningful. What the axis needs from detection is narrower and more useful: the runner that this language's ecosystem would use, the collection convention that runner obeys, the manifest that governs its dependencies, the lock artifact that determines whether it can build, and the toolchain version constraint that determines whether it can build here. Detection is not an answer. It is the acquisition of five parameters.

4.2 The primary signal is always the manifest, and each ecosystem's manifest is read for the same five parameters regardless of its syntax. A dependency manifest names the declared dependencies, and the declared dependency set is the strongest available evidence about the frameworks in play, because a framework that is not declared is not in use no matter how many files appear to reference it, and a framework that is declared is at minimum an intention. A lock artifact records resolution. A tool-configuration section, which most modern manifests carry, names the runner and frequently constrains what it collects. A version constraint, on the language or the toolchain, is what determines whether this host can execute anything at all.

4.3 The toolchain-version parameter is the one that a detector built by a developer who works in a single ecosystem will omit, and its omission produces the most confusing failure class in the entire pipeline, because the symptom is a compilation error in code that is not broken. In the audit, a plugin repository's prebuilt test classes had been compiled against a newer language version than the one the audit host offered by default, and the build refused with a message about being compiled by a more recent version. The natural reading of that message is that something is wrong with the code. Nothing was wrong with the code. The correct action was to pin the toolchain to the version the artifacts required, at which point the suite compiled and produced a large body of passing assertions on the first attempt.

4.4 That instance generalizes into a rule that the detector must encode rather than leave to the operator's judgement. A build failure that occurs before any test is collected is, by default, an environment finding and not a product finding, and it must be routed to the blocked verdict rather than to the failed one. The routing is not a courtesy to the product; it is the attribution law operating at the earliest possible point in the pipeline. An engineer dispatched to repair a product because its build failed in an unqualified environment will succeed in changing the product until the environment stops complaining, and the code that emerges from that process is worse than the code that entered it, and every test in the suite will be green.

4.5 The lock-state parameter is what allows the detector to predict a blocked environment rather than discovering it. A manifest without a resolved lock artifact means that the set of code that will actually execute is undetermined, and an undetermined dependency set means that no result obtained from it can be attributed to anything, because a rerun may resolve differently. In the audit, two repositories in the estate carried a full complement of test files — well over a hundred between them — and could not compile, the toolchain refusing on the ground that dependencies were not locked. The correct record for those repositories is not a low score. It is that their entire test surface is unverifiable in this environment, and that the highest-value action available anywhere in the estate is to restore their build, because doing so converts two repositories from unknown to knowable in a single move.

4.6 A declared dependency that is never resolved and a declared dependency that is resolved but never exercised are different findings and must not be collapsed. In the same audit, a repository declared a benchmarking library, which is evidence of an intention to measure performance, and the dependency set was never resolved, so no benchmark could run. The declaration is a signal about what the authors believed they needed. Its non-resolution is a signal about what the environment can currently deliver. Recording only one of the two produces a report that either credits the repository with protection it does not have or denies it an intention it plainly held, and both errors distort the priority ordering that the estate will act on.

4.7 The framework parameter is read for exactly one purpose in this axis, and stating the purpose narrowly is what keeps the detector from becoming an architecture analyser. The purpose is to determine what the ecosystem's default isolation mechanism is, because the isolation mechanism determines what a hermetic test looks like in this repository and therefore what the standard is. A web framework in one ecosystem supplies a transactional sandbox that gives every test a rolled-back database transaction, which makes the integration test the default rather than an afterthought. A plugin ecosystem supplies an in-memory server implementation that lets plugin logic run headless. A hardware-adjacent ecosystem supplies nothing, and the absence is itself the finding. Detection's job is to name the mechanism that applies; the standards dataset owns the mechanism's content.

4.8 The multi-language repository is the ordinary case rather than the exception, and a detector that returns a single language is not simplifying, it is discarding. In this estate a plugin repository written in one language carries dozens of front-end specification files in another; a service repository carries a back-end in one language and an interface in a second; a game repository carries a hardware-coupled core in one language and a data pipeline in another. In each case the second language has its own runner, its own collection convention, its own manifest, and its own verdict, and a detector that reports only the dominant language will produce a report that is silent about an entire executed surface. The output is therefore a set of language contexts, each with its five parameters, not a label.

4.9 The counted-but-not-executed case must be recorded explicitly and must never be silently omitted. In the audit, front-end specification files were counted across three repositories and were deliberately not executed, because executing them required installing dependencies with side effects that the audit's own scope forbade. The honest record is that those suites exist, that their file count is known, and that their result is unverified — which is a different statement from passing and a different statement from failing, and the twelve-state ontology exists precisely so that this distinction survives into the report. A detector that omits the uncounted surface produces a report whose silence about it is indistinguishable from an assertion that it does not exist.

4.10 The detector's own falsifiability is the last requirement and the one that keeps it honest over time. Every parameter it emits carries the artifact it was derived from — this manifest, at this path, at this line — so that a downstream reader can check it in one step and so that a change to the repository invalidates the parameter rather than silently outdating it. A detector that emits conclusions without provenance is asking to be trusted, and trust is precisely what this axis is not permitted to grant to any instrument, including its own.

PART IV FINAL LAW

Detection is not a label; it is the acquisition of five parameters per language context — runner, collection convention, manifest, lock state, and toolchain constraint — and a repository has as many contexts as it has languages, never one. A build that fails before a single test is collected is an environment finding routed to BLOCKED, never a product finding routed to FAILED: an engineer dispatched to repair correct code will succeed in changing it until the broken environment stops complaining, and every test will be green. Every emitted parameter carries the artifact and line it came from.


PART V — RUNNER AND CI CONFIGURATION DISCOVERY

5.1 The runner configuration is the most under-inspected artifact in any repository and the one with the highest ratio of consequence to size. It is short, it is written once, it is almost never reviewed, it is frequently authored under pressure by someone solving a local problem, and it determines — completely and silently — which fraction of the authored protection the estate actually receives. A repository's test files are read by many people. Its runner configuration is read by nobody after the day it was written. This asymmetry is the reason the largest defects found in the founding audit were configuration defects rather than code defects, and it is the reason this Part exists as a stage in its own right rather than as a footnote to detection.

5.2 The configuration is inspected for four properties and no others, and the narrowness is deliberate. What is the discovery root, meaning the directory or directories from which the runner begins its search. What is the discovery pattern, meaning the filename and symbol conventions that make a file eligible for collection. What are the exclusions, meaning every mechanism by which an eligible file is nonetheless not collected. And what is the invocation, meaning the exact command that a human, a hook, or a continuous-integration job actually issues. The four are separable, each can be wrong independently of the others, and the failure produced by each is different enough that collapsing them into a single notion of coverage destroys the diagnosis.

5.3 The discovery root is where the estate's most expensive observed defect lives. In the audit, one repository's configuration scoped the runner's search to a single subtree of the project. The subtree was large — several thousand tests — and the suite was green, and the green was true. Two sibling directories held functional tests, and a bare invocation of the runner never collected either of them, because the scope line had answered the question of where to look before the question of what exists was ever asked. Those tests passed when invoked explicitly, and the audit invoked them explicitly and observed them pass. They were, in the estate's default configuration, protecting nothing at all, and they were one configuration line away from protecting a great deal.

5.4 That defect has a property that makes it uniquely resistant to ordinary review, and naming the property is the point of the example. The runner's own report cannot describe it. A report says how many tests were collected and how many passed, and both numbers are correct, and there is no field in any runner's output in which the tests it never saw could appear. The absence is not in the report. It is in the complement of the report, and nothing inside the pipeline is positioned to observe a complement. Only a reconciliation between an independently produced census of the authored surface and the runner's own account of what it collected can surface it, which is why the scanner counts test files separately from the runner and why the two counts are compared rather than one of them being trusted.

5.5 The exclusion property is a family rather than a single mechanism, and the family must be enumerated because each member hides in a different place. A file may be eligible and excluded by a scope line. It may be collected and discarded by a marker filter. It may be collected and skipped by a decorator on the test itself, in which case the runner's output reports it — in a count that is printed and that nobody reads. It may be collected and never reached because its module failed to import and the failure was tolerated. It may be reached and disabled by a conditional that evaluates false in this environment and would evaluate true elsewhere. Each of these produces a green result. Only one of them appears in the runner's summary line, and it appears as a number beside the passes, where it is read as noise.

5.6 The skip count deserves its own treatment because it is the only visible exclusion and is therefore the one that discipline can cheaply capture. A suite reporting a large number of passes and a handful of skips is read, universally, as the number of passes. The skips are the interesting quantity. A test does not acquire a skip decorator by accident; someone put it there, deliberately, under pressure, intending to return, and the intention was sincere and the return did not happen. The scanner therefore records the skip inventory as a standing artifact with a count and an identity, not as a number in a summary, because a liability with a name attached is revisited and a number in a summary is not.

5.7 The invocation property is where the runner configuration meets the automation layer, and the join is the point at which most estates lose track of reality entirely. A repository may have a configuration that would collect everything and an automation job that invokes the runner with an argument that overrides it. It may have several suite roots and a job that reaches one. It may have a documented command in its readme that differs from the command its automation runs, in which case a human verifying locally and the pipeline verifying remotely are checking different things and neither knows it. The scanner therefore does not ask what the configuration permits. It asks what is actually invoked, by what, and it records the answer per invoker.

5.8 The uninvoked tool is the automation layer's characteristic defect, and it is the purest possible instance of the law that existence does not imply connection. In the audit, a repository declared and configured a static security analyser. The declaration was in the manifest. The configuration was present. No job in the repository invoked it, ever. Every reader of that manifest — human or agent — would have concluded that the repository had a security analysis stage, and the conclusion would have been drawn from genuine evidence, correctly read. The tool was present and it was inert, and the presence was worse than absence, because an absent tool prompts someone to add one while an inert tool prompts nobody to do anything at all.

5.9 The scanner's output for this stage is therefore not a coverage number but a bipartite map: every declared quality instrument on one side, every invoking job on the other, and an edge wherever an invocation exists. The instruments with no incoming edge are the finding. This map is small, it is cheap to build from a read of the manifest and the job definitions, it requires no execution, and it answers in one glance the question that no report generated from inside the pipeline can answer, which is which of our controls are actually connected to anything. An instrument with no edge is recorded as unverified with a stated reason, never as absent and never as present.

5.10 A closing warning about the seductiveness of the continuous-integration signal, because it is the signal most likely to be over-trusted by the agent consuming this dataset. A green pipeline is test output plus scheduling. It inherits every weakness of test output, and it adds one of its own: its configuration is itself unverified code that nobody tests, and it is in that configuration that every defect described in this Part concentrates. A green pipeline is not a higher authority than a local run. It is the same authority with a longer feedback loop and a larger surface in which an exclusion can hide, and the estate's own history is that the exclusions hid there and were found by reading the configuration rather than by trusting the badge.

PART V FINAL LAW

The runner configuration is short, unread after the day it was written, and decides which fraction of authored protection the estate actually receives. Four properties, each independently wrong: discovery root, discovery pattern, exclusions, invocation. A scope line that orphans hundreds of passing tests cannot appear in any runner's report, because the absence lives in the report's complement and nothing inside the pipeline observes a complement. Build the bipartite map of declared instruments to invoking jobs; an instrument with no incoming edge is present, inert, and worse than absent.


PART VI — TOPOLOGY: MONOREPO, MULTI-REPO, HYBRID, AND THE REPO WITH NO EXECUTABLE SURFACE AT ALL

6.1 Topology is the shape of the estate's executable units and their relationship to its version-control units, and it is treated as a separate concern from classification because the two answer different questions. Classification asks what kind of thing a project is. Topology asks how many projects there are, where their boundaries fall, and whether the boundaries the repository asserts are the boundaries that actually govern execution. Getting topology wrong does not produce a wrong finding about a project; it produces findings about the wrong number of projects, which is a category of error that no amount of care applied within a project can correct.

6.2 The single-project repository is the case every tool assumes and the case that is rarest in a mature estate. Its properties are that one manifest governs the whole tree, one runner configuration governs all collection, one invocation reaches everything, and a coverage number computed at the root means what a reader would assume it means. When these properties hold, the entire remainder of this dataset simplifies, and the temptation is to assume they hold because the analysis is so much easier when they do. The scanner's obligation is to verify them rather than assume them, and the verification is a single question: is there more than one manifest in this tree, and if so, does the root manifest know about the others.

6.3 The monorepo is the case where multiple executable units share one version-control boundary, and its characteristic failure is the aggregation error. A coverage figure, a test count, or a pass rate computed across a monorepo is an average over units that have nothing to do with each other, and an average conceals its minimum. In this estate a plugin monorepo carries dozens of independent build descriptors, and its test population is not distributed across them: the great majority of its test files sit inside a single component. The aggregate is impressive and the aggregate is a claim about volume. The distribution is the claim about protection, and the distribution says that most of the components in that tree carry almost nothing.

6.4 That instance also supplies the monorepo's second characteristic failure, which is the unshared standard. The in-memory server implementation that makes headless plugin testing possible in that ecosystem was found wired into exactly one of the tree's many build descriptors. The mechanism existed, it worked, it was proven in the component that used it, and it had not propagated to any sibling. This is not a coverage gap; it is an adoption gap, and the two require different remedies. A coverage gap is closed by writing tests. An adoption gap is closed by making the mechanism available and making its absence visible, and it will not be closed by writing tests in the component that already has them, which is precisely where the pressure to write more tests will be felt.

6.5 The multi-repo estate is the inverse topology, in which one logical system spans several version-control boundaries, and its characteristic failure is the orphaned seam. A contract that crosses a repository boundary has no home: neither side owns the test that would validate it, each side stubs the other in its own suite, both suites are green, and the seam is exercised for the first time in production. This is the integration-without-contract class of the sealed taxonomy, and it is structurally guaranteed by the topology rather than caused by carelessness. The scanner's job is to record cross-repository dependencies as edges, because a seam that nobody can see is a seam that nobody will assign.

6.6 The hybrid is the ordinary case and it is the case that defeats naive tooling. A repository with a service in one language, an interface in a second, a data pipeline in a third, and a set of automation scripts in a fourth is not a project with a primary language and some accessories. It is four projects that happen to share a directory, each with its own manifest, runner, collection convention, and verdict, and a report that assigns the whole tree one classification and one verdict has produced a number that is true of nothing. The scanner therefore emits one record per executable unit and refuses to roll them up, and the refusal is not a limitation of the scanner but a property of the reality it is reporting on.

6.7 The unit boundary is established by manifests rather than by directories, and this rule resolves nearly every ambiguity in practice. A directory containing a manifest that declares its own dependencies and its own runner is a unit. A directory without one is part of its nearest ancestor unit. The rule is mechanical, it is cheap, it is derived from the project's own declarations rather than from an outsider's intuition about structure, and it produces a partition rather than a hierarchy of guesses. Where it produces an answer that surprises the operator — a directory that looked like a subcomponent turning out to be an independent unit with its own dependency set — the surprise is the finding.

6.8 The repository with no executable surface at all is the topology that this dataset insists on treating as first-class, and the insistence is grounded in the audit's founding correction. Such a repository has no manifest, no build descriptor, no runner configuration, and no source of any executable kind. It is a content and knowledge repository: its product is documents. The naive tool reports zero tests and zero coverage and flags it as the estate's worst offender, and the flag is not merely wrong but inverted, because the repository is in a state of complete correctness with respect to a standard that does not apply to it. The correct verdict is that the surface genuinely does not exist, and that verdict is one of the twelve for exactly this reason.

6.9 The consequences of getting that case wrong compound in a specific way that is worth tracing. A repository wrongly assigned an executable classification acquires a permanent deficit against every executable metric. It appears at the top of every remediation queue that sorts by coverage. It attracts attention proportional to a gap that does not exist. And each cycle of attention produces a report that reconfirms the misclassification, because the reports are generated by the same tooling that made the original error, and the tooling has no organ for asking whether the classification was right. A misclassified repository does not merely produce one wrong finding. It produces a self-reinforcing stream of them.

6.10 Topology is also temporal, and the scanner must record when it looked. A repository acquires a second manifest the day someone adds a service to it. An estate acquires a new seam the day someone splits a module out. A monorepo becomes a hybrid without any single commit that could be called the moment of transition. The topology recorded by a scan is a statement about a commit, and it expires, and the expiry is not a defect in the scan but a property of estates. The scan's output therefore carries its commit, and a topology claim cited without one is a claim about a repository that may no longer exist.

PART VI FINAL LAW

The unit boundary is the manifest, not the directory. A monorepo's aggregate is a claim about volume; only its distribution is a claim about protection — and in this estate the distribution put the great majority of a plugin tree's tests inside a single component, with the headless-testing mechanism wired into one build descriptor of many. The repository with no executable surface is not the estate's worst offender; it is correct with respect to a standard that does not apply, its verdict is that the surface does not exist, and misclassifying it produces not one wrong finding but a self-reinforcing stream of them.


PART VII — THE DOMAIN CLASSIFICATION ENGINE: THE SIXTEEN DOMAINS

7.1 Classification assigns each executable unit a domain, and the domain is not a descriptive convenience. It is the parameter that selects which failure modes are plausible, which tests are meaningful, which isolation mechanism is available, which surfaces carry risk, and which standard the unit will be measured against. A verification axis that treats all repositories as instances of a single kind will apply one standard uniformly, and a uniform standard is necessarily calibrated for the median case, which means it will be simultaneously too strict for the repositories that do not need it and far too lax for the ones that do. Classification is what makes a standard proportional, and proportionality is what makes it obeyed.

7.2 The engine recognizes sixteen domains, and the number is not aspirational — it is the closure of what the founding audit encountered plus the cases the audit's own scope boundaries forced it to name. The domains are: hosted service, back-end service, front-end application, library, command-line tool, infrastructure, plugin, game, firmware, hardware-coupled system, data pipeline, content and knowledge vault, dataset, prompt system, agent system, and hybrid monorepo. Each is defined by what it produces and by how it is executed, never by the language it is written in, because language is a detection parameter and domain is a risk parameter, and conflating them is how a hardware-coupled system written in a general-purpose language acquires the standard of a library.

7.3 The classification signal set is deliberately small and deliberately grounded in artifacts rather than in reading code. The presence and kind of an entry point. The presence of a route table, a command registry, an event subscription, or a scheduled-job definition. The presence of a deployment descriptor. The presence of a hardware abstraction header or a device-specific toolchain requirement. The presence of a package-publication descriptor. The ratio of document files to source files. The presence of a model-invocation dependency. Each of these is a read, none of them requires interpretation, and the classification is the conjunction of what was found, with an explicit unknown where the signals conflict.

7.4 Conflicting signals are common and must never be resolved by picking the strongest. A unit with a route table and a package-publication descriptor is not a hosted service that happens to publish; it is a hosted service and a library, and both standards apply, because the published interface has consumers who are not the service and who will break in ways the service's own suite cannot observe. A unit with a command registry and a deployment descriptor is a tool and an operational surface. The engine's output is therefore a set of domains per unit rather than a label, and the practice of forcing a single label is the mechanism by which one of the two applicable standards silently disappears.

7.5 The hybrid monorepo is a domain in its own right rather than an admission that classification failed, and treating it as a domain is what prevents its characteristic pathology. Its pathology is that a report about it is an average, and the average is dominated by whichever sub-unit is largest, and the largest sub-unit is rarely the riskiest. The domain's standard is therefore a structural one rather than a coverage one: it demands that the tree be partitioned, that each partition carry its own classification and its own verdict, and that no aggregate figure be emitted at the root at all. The absence of the root number is the point. A number that cannot be computed honestly is better absent than present.

7.6 The content-and-knowledge-vault domain exists because the founding audit needed it and because omitting it is what produces the estate's most confidently wrong finding. Its defining property is that it has no executable surface: its product is documents, and its failure modes are real but are not the failure modes of code. Its content can be internally contradictory. It can be stale with respect to the systems it describes. It can assert an interface that no longer exists. Those are genuine defects and they are verifiable, and none of them is discovered by a test runner. The domain's standard is consistency and currency, not coverage, and the verdict for its coverage is that the surface does not exist.

7.7 The dataset domain and the prompt-system domain are adjacent to the content vault and are separated from it because their failure modes differ. A dataset's product is structured data consumed by something downstream, and its defects are schema drift, silent truncation, encoding damage, and duplicate or contradictory records — all verifiable, all by validators rather than by unit tests. A prompt system's product is instructions consumed by a model, and its defects are contradiction between instructions, silent override by a higher-priority context, drift between a canonical file and the live copy that actually loads, and a rule that is present in the repository and absent from the running system. That last defect has been observed in this estate more than once, and it is the executable-governance law wearing a prompt system's clothes.

7.8 The agent-system domain is the one this estate is itself an instance of, and self-classification is not a conflict of interest but an obligation. An agent system's product is behaviour under instruction, its surfaces are hooks and dispatch points and tool permissions, and its characteristic failure is the built-but-unwired control: a gate that exists in the repository, is correct, is tested, and is never reached by any event, or whose refusal is discarded by the layer that aggregates hook outputs. Its standard therefore includes a gate-liveness obligation that no other domain requires — every gate must have been observed to refuse something — and the obligation exists because in this domain, uniquely, the artifact that enforces the standard is itself part of the system under test.

7.9 The remaining domains are separated by execution model rather than by subject matter, and the separation is what makes the standards portable. Infrastructure executes by being applied to an environment and its failures are environmental. A plugin executes inside a host it does not control and its failures are host-coupled. A game executes under a frame budget and its failures are temporal. Firmware and hardware-coupled systems execute on a substrate that a host cannot simulate, and their characteristic finding is that the host cannot run their suite at all, which is a blocked verdict and not a failing one. A data pipeline executes over data whose shape it does not control, and its failures are data-shaped.

7.10 The engine's contract is that classification is derived, recorded with its evidence, and revisable, and that no classification is ever inherited from a prior run without re-derivation. A repository changes kind. A vault acquires a script. A library acquires a service. A classification cached from a previous scan and reused because re-deriving it is expensive is a description, and this entire dataset exists to establish that descriptions are subordinate to disks. The re-derivation is cheap by construction — it is a small set of file reads — and the cheapness is what makes the prohibition on caching enforceable rather than aspirational.

PART VII FINAL LAW

Domain is a risk parameter, not a language label, and it selects which failure modes are plausible, which isolation mechanism exists, and which standard applies. Sixteen domains, and a unit may hold several: forcing a single label is how one of two applicable standards silently disappears. The content vault, the dataset, and the prompt system are executable-surface-free and their standard is consistency and currency, not coverage. The agent system carries a unique obligation — every gate observed to refuse something — because there, uniquely, the enforcing artifact is part of the system under test.


PART VIII — WHY CLASSIFICATION DETERMINES WHICH TESTS ARE MEANINGFUL AND WHICH ARE ARTIFICIAL

8.1 A test is meaningful when its failure would have told someone something they needed to know, and artificial when it would not. That is the whole of the criterion, and it is entirely relative to the domain, which is why classification is upstream of every judgement about test quality. A test that pins the exact byte layout of a serialized structure is indispensable in firmware, where a downstream consumer decodes it on hardware that cannot be patched, and is a maintenance liability in a hosted service, where the structure is an internal detail and the test will fail on every legitimate refactor while catching nothing. The same test, written with identical care, is protective in one domain and corrosive in another.

8.2 The corrosive case deserves the emphasis, because a verification axis that only ever demands more tests will be experienced as an adversary and will eventually be routed around. An artificial test is not neutral. It consumes authoring effort, it consumes execution time, it fails on changes that are correct, and each of those false failures trains the team to treat red as noise. A suite with a substantial artificial fraction produces exactly the behaviour it was built to prevent: engineers who look at a failure and reflexively assume the test is wrong. The damage is not that the artificial test protects nothing. It is that it degrades the signal value of every genuine test standing beside it.

8.3 The domain determines meaning through three parameters, and naming them makes the judgement mechanical rather than aesthetic. The first is the blast radius of a defect: who is harmed, how quickly, and how reversibly. The second is the observability of a defect in production: whether a wrong result announces itself or accumulates silently. The third is the patchability of the deployed artifact: whether a defect discovered after release can be corrected in minutes, in a release cycle, or never. A test's meaning is high where blast radius is high, observability is low, and patchability is poor, and it is low where the inverse holds — and those three parameters are properties of the domain, not of the code.

8.4 A hosted service scores low on patchability difficulty and high on observability: a defect can be corrected and redeployed quickly, and a wrong result frequently surfaces as an error rate that somebody is watching. That combination makes its meaningful tests the ones that guard against the failures which do not surface — silent data corruption, a permission that is broader than intended, a computation that is wrong in a way that produces a plausible number. Its artificial tests are the ones that pin internal structure. The estate's own hosted services carry large suites, and the useful question about them is never how many tests there are but whether the silent classes are among them.

8.5 A plugin inverts nearly every parameter and therefore inverts the standard. It executes inside a host it does not control, its defects are frequently only observable by a user in a live environment, and its most abusable surface is whatever moves value between actors. In the founding audit, a plugin tree carried a very large body of passing assertions, and the component governing the movement of value between actors carried no test reference anywhere in the entire tree. The suite was green, the count was large, and the count was concentrated in a component whose defects are visible and reversible, while the component whose defects are neither carried nothing. That is not a coverage failure. It is a meaning failure: effort went where testing was easy rather than where it mattered.

8.6 A hardware-coupled system inverts the patchability parameter to its extreme and thereby changes what a test even is. Its artifact is executed on a substrate the host cannot simulate, its defect cannot be patched after the fact in any ordinary sense, and its dominant contract is temporal — a frame budget, an interrupt latency, a bus timing. In this estate a hardware-coupled title's test files are compiled against the graphics abstraction headers of the target device, which means they link the entire engine and cannot run on a host at all. The naive reading is that the repository has tests which are failing to run. The correct reading is that it has integration smoke drivers rather than isolatable units, that the host cannot execute them, and that the verdict is blocked rather than failing.

8.7 That case yields the domain's actual standard, and the standard is a structural instruction rather than a coverage target. The meaningful move is to extract the logic that has no dependency on the hardware abstraction — the arithmetic, the state transitions, the scoring, the input mapping — into units that a host can execute with a minimal assertion harness, and to leave the temporal contracts to be measured on the device by the profiler the project already contains. Demanding host-runnable coverage of the graphics path in this domain is demanding an artificial test, and demanding a frame-budget assertion is demanding the single most meaningful one. Same repository, same effort budget, opposite outcomes, and the difference is entirely a function of having classified it correctly.

8.8 The content vault and its neighbours make the point in its purest form, because for them the entire category of unit test is artificial. There is no unit. A demand for coverage against a repository whose product is documents is not a strict standard misapplied; it is a category error, and the finding it generates is a fabrication in the precise sense established earlier in this dataset. What is meaningful in that domain is a validator: does every internal reference resolve, does every claimed interface still exist in the system it describes, is any assertion in this corpus contradicted by another. Those checks are cheap, they are executable, they catch real defects, and no test runner will ever produce them.

8.9 The generalization is that every domain has a meaningful check and an artificial one, and that the artificial check is always the one the tooling makes easiest. Coverage tooling is easy, so coverage becomes the demand. Unit-test scaffolding is easy, so unit tests become the demand. Neither tool knows anything about blast radius, observability, or patchability, and neither can distinguish a suite that guards the surface that matters from a suite of identical size that guards the surface that was convenient. Classification is the only mechanism in the pipeline that carries that information, which is why it is upstream of measurement and why a measurement taken without it is a number about effort rather than about protection.

8.10 The operational consequence is a question asked once per unit, before any test is written or demanded: in this domain, what defect would harm someone, arrive silently, and be expensive to correct — and is there anything asserting against it. That question takes a minute, it is answerable from the classification and the surface map, and it reorders the entire remediation queue. In this estate it reordered it decisively: the top of the queue was not the repository with the fewest tests, but the environments that could not build at all, the configuration line that orphaned hundreds of passing tests, the test that polluted its host, and the unprotected component in the repository with the largest green number.

PART VIII FINAL LAW

A test is meaningful when its failure would have told someone something they needed to know, and that is entirely a function of domain: blast radius, observability of a defect in production, and patchability of the deployed artifact. An artificial test is not neutral — it fails on correct changes and trains the team to read red as noise, degrading every genuine test beside it. The tooling always makes the artificial check easiest, which is why coverage becomes the demand; classification is the only organ in the pipeline that knows whether a suite guards the surface that matters or the surface that was convenient.


PART IX — DOMAIN PROFILES: SAAS, BACKEND, FRONTEND, LIBRARY, CLI

9.1 A domain profile is a compact operational contract with five fields, and the fields are fixed so that profiles can be compared. What the unit produces. What its executable surfaces are. What isolation mechanism its ecosystem supplies, which determines what a hermetic test looks like here. Which failure classes of the sealed taxonomy are structurally likely, as opposed to merely possible. And what the honest verdict is when the standard cannot be met in the current environment. A profile is not a description of good practice; it is a lookup table that the scanner consults to decide what it is entitled to expect on disk and what its silence about a surface actually means.

9.2 The hosted-service profile produces a running system that many parties depend on simultaneously and that is redeployed frequently. Its surfaces are routes, background jobs, queue consumers, scheduled tasks, and migrations, and each of those is a separate ingress with its own authorization posture. Its isolation mechanism is usually strong and usually supplied by the framework: a transactional sandbox that gives each test a rolled-back database transaction makes the integration test the default rather than an afterthought, and a repository in this domain that is not using its framework's sandbox has declined a mechanism it already owns. Its structurally likely failure classes are the integration seam without a contract and the production behaviour with no covering monitor, because its unit boundaries are numerous and its seams are where the coupling lives.

9.3 The hosted-service profile's most important field is the honest verdict, because this domain supplies the estate's clearest instance of a blocked one. In the founding audit, two service repositories in this estate could not be compiled at all: the toolchain refused on the ground that their dependencies were not locked. Between them they carry well over a hundred test files. The correct record is not a poor score and not a failing suite. It is that their entire test surface is unverifiable in this environment, that the fault lies with the environment rather than the product, and that restoring the build — resolving and committing the lock artifact, provisioning the database the suites require — is the single highest-return action available anywhere in the estate, because it converts two whole repositories from unknown to knowable in one move.

9.4 The back-end-service profile differs from the hosted service in exactly one parameter, and the parameter matters. It has no interactive surface of its own; it is consumed by other programs rather than by people, which means its contract is machine-readable and therefore precisely testable, and it also means that a breaking change to it is discovered by a consumer rather than by a user. Its surfaces are its interface definitions, its event subscriptions, and its persistence layer. Its dominant risk is contract drift: a field renamed, a type widened, an optional made required, each of which is invisible inside its own suite and catastrophic across the seam. The meaningful test here is the contract test — an assertion written from the consumer's perspective and executed against the producer.

9.5 The front-end-application profile is the domain where this estate's honest record is thinnest, and stating that plainly is more useful than pretending otherwise. Its surfaces are components, routes, and the state transitions between them, and its isolation mechanism is a simulated document environment rather than a real browser, which means the mechanism itself is a substitution whose fidelity is a question rather than an assumption. In the founding audit, front-end specification files were counted across three repositories and were deliberately not executed, because doing so required installing dependencies with side effects that the audit's scope forbade. The honest record is that those suites exist, their file counts are known, and their result is unverified — a state that is neither passing nor failing, and that the twelve-state ontology exists to preserve.

9.6 That non-execution is worth dwelling on because it is a model of how a scope boundary should be recorded. The audit did not estimate. It did not assume the suites would pass because they were maintained. It did not omit them, which would have made its silence about them indistinguishable from an assertion that they did not exist. It counted them, recorded the reason they were not run, and assigned the verdict that the evidence licensed. A scope boundary honestly recorded is a finding with an owner; a scope boundary silently observed is a hole in a report that its readers will never know about.

9.7 The library profile is defined by a property no other domain has: its consumers are unknown and unreachable. A hosted service can be corrected in minutes and its users receive the correction whether they want it or not. A library's defect propagates into artifacts that other people build, at times of their choosing, and a breaking change cannot be recalled. That inverts the entire test calculus. The meaningful tests in a library are the ones that pin the public interface exactly — signatures, return shapes, error types, edge behaviour at boundaries — because the interface is the product and everything behind it is an implementation detail. Its structurally likely failure class is the ignored edge case, because a library's inputs are supplied by strangers who did not read its assumptions.

9.8 The library profile also carries the domain's characteristic misallocation, which is worth naming because it is the mirror image of the plugin's. Library authors, being close to their own code, test the internals thoroughly and the boundary casually, and the internals are precisely the part they are entitled to change. The resulting suite is large, green, and brittle: it fails on every legitimate refactor and it says nothing about the contract that consumers actually depend on. The scanner's diagnostic here is a ratio rather than a count, comparing assertions that exercise the published interface against assertions that reach past it, and a library whose ratio is inverted has a suite that will be deleted the first time someone tries to improve the internals.

9.9 The command-line-tool profile is the one most often assumed to be trivially testable and most often left untested for exactly that reason. Its surfaces are its argument parser, its subcommand registry, its exit codes, its filesystem effects, and its standard streams, and each of those is a contract that something downstream depends on. An exit code is an interface: a script somewhere branches on it. A stream format is an interface: something is parsing it. The meaningful tests invoke the tool at its real boundary and assert on what a caller can observe — exit status, emitted text, files created and their contents — and the artificial ones reach inside and test the functions the parser calls, which proves that the internals work while leaving the interface, which is the entire product, unasserted.

9.10 The command-line profile is also where the destructive surface concentrates, and that raises its risk class independently of its size. A tool that deletes, moves, overwrites, or publishes carries a defect whose blast radius is bounded only by the arguments it is given, and its most meaningful test is the deny test: the assertion that a dangerous operation refuses under the condition it is supposed to refuse under. A tool tested only on its permitted paths has proven that it works and has proven nothing about whether it stops, and the stopping is what the estate is relying on. This is the security-without-a-contract class of the taxonomy in its most literal form, and in the command-line domain it is the default state rather than an exception.

PART IX FINAL LAW

A profile is a lookup table with five fields — product, surfaces, isolation mechanism, structurally likely failure classes, honest verdict when the standard cannot be met — and it tells the scanner what its own silence about a surface means. A service whose dependencies will not resolve is BLOCKED, and restoring its build converts a whole repository from unknown to knowable in one move. A library's consumers are unknown and unreachable, so the interface is the product and internal tests are the misallocation. A tool's exit code is an interface, and its most meaningful assertion is that a destructive path refuses.


PART X — DOMAIN PROFILES: INFRASTRUCTURE, PLUGIN, GAME, FIRMWARE, HARDWARE

10.1 The domains in this Part share a property that separates them sharply from those in the last: their execution substrate is not the host that authors them. Infrastructure executes by being applied to an environment. A plugin executes inside a host process it does not control. A game executes under a temporal budget that a development machine does not enforce. Firmware and hardware-coupled systems execute on silicon that a host cannot simulate. In every case the gap between the authoring substrate and the executing substrate is where the defects live, and in every case a suite that runs happily on the host has, by construction, not exercised the thing that will actually run.

10.2 The infrastructure profile produces a description of a desired environment state, and its surfaces are its resource declarations, its dependency ordering, its variable interpolation, and its destructive operations. Its characteristic failure is that it is correct in isolation and wrong on application, because the environment it will meet has drifted from the environment it was written against. Its meaningful checks are therefore a plan-versus-apply comparison against a real target and an explicit assertion about what the change destroys, and its artificial ones are schema validations of the declaration file, which prove that the document parses and say nothing about what applying it will do. Its honest verdict when no target environment is available is blocked, never passing.

10.3 The plugin profile is the estate's richest source of evidence and its clearest instance of concentrated protection. A plugin's surfaces are its command registrations, its event handlers, its scheduled tasks, its configuration loading, and whatever shared state it mutates in the host. Its isolation mechanism is an in-memory implementation of the host, which allows plugin logic to run headless without a live server, and in this estate that mechanism was found wired into exactly one build descriptor out of many — the mechanism works, it is proven, and it has not propagated. Its structurally likely failure classes are the unprotected critical unit and the missing deny test, and in the audit both landed at the same coordinate.

10.4 That coordinate is worth naming precisely because it is the sharpest single finding in the founding audit. The plugin tree produced a very large body of passing assertions on a corrected toolchain, and a search for any test reference to the component that governs the movement of value between actors returned nothing anywhere in the tree. Its two tests in that module guard a route guard and a legacy colour-parser. The surface that a hostile user would attack first, whose defects are irreversible in a live world, and whose failures harm every participant, is the surface with no assertion at all. The number was large. The number was concentrated. The concentration was invisible in the aggregate and was the entire finding.

10.5 The plugin profile also carries a configuration failure class that the domain makes uniquely lethal. Plugin configuration is authored by operators rather than developers, it is loaded at startup, and a malformed value frequently disables the plugin silently rather than crashing it, which means an entire command surface can vanish from a live host with no error that anyone reads. The meaningful test is a loader matrix — valid input parsed, malformed input rejected with a clear error, degenerate input producing a safe default rather than a crash — and the artificial one is a test of the parser's internals with fixtures the operator would never produce. This is the input-validation class of the taxonomy, and in this domain a single bad key in a configuration file can take out a plugin's whole command set.

10.6 The game profile is defined by a contract that no other domain in this list carries: correctness is insufficient, because a correct result delivered late is a defect. Its surfaces are its input handling, its state machine, its render path, and its persistence, and its dominant contract is the frame budget. Its characteristic failure class is performance without a baseline: a change that halves the frame rate ships green, because nothing in the suite measures time, and the regression is discovered by a player. Its meaningful checks are a benchmark with a threshold that fails the build, and its artificial ones are unit tests of render internals that assert on structure rather than on duration.

10.7 The firmware and hardware-coupled profiles are treated together because their determinative property is identical: the host cannot execute the artifact. In this estate a hardware-coupled title's test drivers include the target's graphics abstraction headers directly, which means they link the entire engine and require the device toolchain to build at all. The naive reading of that situation is a repository whose tests are failing. The correct reading is a repository whose tests are integration smoke drivers rather than isolatable units, whose host cannot run them, and whose verdict is blocked. The audit recorded exactly that, and it recorded that it had not stood up the device toolchain, which is the scope boundary that licenses the verdict.

10.8 The remedy that profile prescribes is structural rather than quantitative, and it is the clearest example in this dataset of classification changing what the right work is. The move is to extract the logic that carries no dependency on the hardware abstraction — arithmetic, state transitions, scoring, input mapping — into units a host can execute under a minimal assertion harness, and to leave the temporal contracts to be measured on the device by the profiler the project already contains. This is not a demand for more tests. It is a demand for a boundary, and once the boundary exists the tests are cheap. Demanding host coverage of the graphics path in this domain is demanding an artificial test; demanding a frame-budget assertion is demanding the most meaningful one available.

10.9 The hardware profile adds a final field that no software domain requires: evidence class. In every other domain, a runner's output is admissible evidence of behaviour. Here it is not, because the runner ran on a substrate that is not the substrate. The only strong evidence about a hardware-coupled surface is a capture from the real device — a log pulled from it, a frame timing measured on it, a screen captured off it — and a host-side simulation is medium evidence at best, admissible for logic and inadmissible for timing. The profile therefore caps the strength of any claim about the device made from host evidence alone, and the cap is not a matter of caution; it is the limiting-evidence law applied to a substrate boundary.

10.10 The common instruction across all five profiles is the one that makes them tractable. Identify the boundary between the authoring substrate and the executing substrate, and record every claim's position relative to it. A claim made on the authoring side about the authoring side is ordinary. A claim made on the authoring side about the executing side is a hypothesis, and it must be labelled one, and the labelling is what stops an estate from believing it has verified a device it has never touched.

PART X FINAL LAW

These domains execute somewhere their authors do not, and the gap between authoring substrate and executing substrate is where their defects live. A plugin's headless-testing mechanism can exist, work, and be wired into one build descriptor of many — a mechanism gap, not a coverage gap, and writing more tests where tests already are will not close it. Hardware-coupled drivers that include the device's graphics headers are not failing tests; they are unrunnable on the host, and the verdict is BLOCKED. Host evidence about a device caps at medium and is inadmissible for timing.


PART XI — DOMAIN PROFILES: DATA PIPELINE, CONTENT VAULT, DATASET, PROMPT SYSTEM, AGENT, HYBRID MONOREPO

11.1 The domains in this Part are the ones that standard testing tooling was not built for, and the consequence is uniform: applied naively, the tooling reports each of them as catastrophically under-tested, and in every case the report is a category error rather than a strict standard. What these domains share is that their product is not a program in the sense the tooling assumes — it is a transformation, a corpus, a structure, an instruction set, a behaviour, or a federation — and each has real, verifiable failure modes that no test runner will ever produce. The profiles below exist to name what is actually checkable, because the alternative is an estate that either fabricates findings against them or exempts them entirely.

11.2 The data-pipeline profile produces a transformation over data whose shape it does not control, and this single property determines everything else. Its surfaces are its ingestion adapters, its validators, its transformation stages, and its sinks. Its characteristic failure class is input without validation coverage: a parser that accepts what it should reject, or an adapter-selection chain that returns nothing on no match and merely logs a warning, so the no-adapter path is exercised only incidentally. Its meaningful checks are a validation matrix — valid input parsed, malformed input rejected with a clear error, degenerate input producing a safe default rather than a crash — and a schema assertion at every stage boundary. Its artificial checks are unit tests over transformation internals with fixtures that production never supplies.

11.3 The data-pipeline profile also supplies this estate's textbook hermeticity failure, and the instance is worth stating exactly because the diagnosis is what makes it instructive. In the audit, a repository in this domain produced a suite that was almost entirely green with a single failure, and the failure traced to a test that wrote outside its allocated temporary space to a location at the host's drive root. The pollution defeated a sibling suite. The product logic was correct throughout. An undisciplined reading would attribute the red to the code and dispatch someone to repair it; the correct attribution is that the environment was contaminated by the harness, that the verdict for the product is environment-invalid rather than failed, and that the remedy is to point the offending test at its temporary directory.

11.4 The content-and-knowledge-vault profile produces documents, has no executable surface, and is the domain whose misclassification founded this dataset. Its failure modes are real and are entirely verifiable: an internal reference that does not resolve, a described interface that no longer exists in the system it describes, an assertion contradicted by another assertion in the same corpus, and staleness with respect to a commit. Its meaningful checks are validators over those properties. Its verdict for coverage is that the surface genuinely does not exist, and that verdict is one of the twelve for exactly this case. A repository in this domain reported as the estate's worst offender by a coverage tool has been measured against a standard that does not apply to it, and the resulting number is a fabrication with a method attached.

11.5 The dataset profile is separated from the vault because its product is consumed by a program rather than by a reader, which makes its contract machine-checkable and its defects silent. Its surfaces are its schema, its record set, and whatever loader consumes it. Its characteristic failures are schema drift, silent truncation, encoding damage, and duplicate or contradictory records — none of which announces itself, all of which propagate into whatever the dataset feeds. Its meaningful checks are structural: every record validates against the declared schema, the record count matches a recorded baseline, no identifier appears twice, and the encoding round-trips. Its artificial check is a spot-read of a few records by a human, which is exactly the check most often performed.

11.6 The prompt-system profile produces instructions consumed by a model, and its failure modes are governance failures rather than logic failures. Its surfaces are the instruction files themselves, their load order, and their priority relationships. Its characteristic failures are contradiction between two instructions that never appear on the same page, silent override of a rule by a higher-priority context, and — the one this estate has observed repeatedly — divergence between a canonical file held in version control and the live copy that the running system actually loads. That last is the executable-governance law in its purest form: the rule exists in the repository, it is correct, it is committed, and it is not in the running system, and every reader of the repository believes it is enforced.

11.7 The prompt system's meaningful checks follow directly from those failures and are all cheap. Compare every canonical instruction file against its live counterpart and assert byte equality, because a divergence is undetectable by reading either one alone. Assert that every declared rule is reachable in the load order. Search for contradiction between rules that share a subject. And, most importantly, verify that a rule which claims to block something has been observed to block something, because an instruction that has never refused anything is in the same epistemic position as a regression test that has never been seen to fail.

11.8 The agent-system profile is the one this estate instantiates, and its profile is the strictest in the set precisely because the artifact that enforces the standard is part of the system under test. Its surfaces are its hooks, its dispatch points, its tool permissions, and its gates. Its characteristic failure is the built-but-unwired control: a gate that exists, is correct, is tested in isolation, and is either never reached by any event or whose refusal is discarded by the layer that aggregates hook outputs. Both have been observed here. Its meaningful check is therefore a liveness assertion per gate — construct the condition the gate exists to catch, present it, and observe the refusal — and it is a five-minute test that no amount of unit coverage substitutes for.

11.9 The hybrid-monorepo profile is a domain rather than a failure of classification, and its standard is structural. It demands that the tree be partitioned at its manifests, that each partition carry its own classification, its own surface map, and its own verdict, and that no aggregate figure be emitted at the root. The absence of the root number is the profile's central provision and its most counterintuitive one. A number that cannot be computed honestly is better absent than present, because an aggregate over unrelated units is dominated by whichever unit is largest, the largest unit is rarely the riskiest, and the resulting figure is a claim about volume that every reader will interpret as a claim about protection.

11.10 The instruction common to all six profiles is that the absence of a runner is not the absence of a check. Each of these domains has defects that are real, that harm someone, and that are detectable by a validator that takes an afternoon to write. The reason they go undetected is not that they are hard; it is that the tooling the estate already owns does not produce them, and an estate measures what its tooling emits. Classification is what breaks that loop, because it names the check that should exist before anyone asks what the existing tools happen to report.

PART XI FINAL LAW

These domains have real, verifiable defects that no test runner will ever produce, and applied naively the tooling reports each as catastrophically under-tested — a category error, not a strict standard. A data pipeline's test that writes outside its sandbox to the host drive root defeats a sibling suite while the product logic is correct: the verdict is ENVIRONMENT-INVALID, never FAILED. A prompt system's canonical file can diverge from the live copy that actually loads, so the rule is committed, correct, and absent from the running system. The absence of a runner is not the absence of a check.


PART XII — THE EXECUTABLE SURFACE MAPPER: THE INVENTORY AND ITS CONTRACT

12.1 The surface map is the artifact on which every subsequent verification judgement depends, and it answers a question that no coverage tool asks: what can actually be invoked in this repository, by whom, from outside. Coverage measures which lines a suite executed. The surface map measures which entry points production exposes. The two are orthogonal, and the gap between them is where every interesting defect in the founding audit was found — a component with high line coverage in the aggregate whose most abusable entry point had no assertion, a suite whose lines were all executed by an invocation nobody ever ran. Lines are what a test touched. Surfaces are what a user can reach.

12.2 A surface is defined precisely, and the precision is what makes the inventory mechanical rather than a matter of taste. A surface is any point at which control enters the unit from outside the unit's own code, together with the contract that governs that entry. The definition has three consequences. It excludes internal functions, however important, because they are reached only through a surface and are protected by whatever protects it. It includes every entry point regardless of whether anyone remembers it exists, which is the entire reason for building the map. And it binds a contract to each entry, because an entry point without a stated contract cannot be verified against anything.

12.3 The inventory's contract has six fields per surface and no optional ones, and every field is either an observation with its basis or an explicit unknown. The identity of the surface, expressed as the path and symbol where it is registered. Its kind, drawn from the enumerated taxonomy of the next two Parts. Its ingress, meaning who or what can invoke it — an anonymous caller, an authenticated one, a privileged one, the scheduler, another component. Its effect class, meaning what it changes: nothing, local state, shared state, persisted state, external state, or destructive. Its contract, meaning what it promises. And its protection, meaning the tests, monitors, and gates bound to it, which may legitimately be empty and which must never be silently empty.

12.4 The effect-class field is the one that converts the map from an inventory into a risk instrument, and it is the field a naive mapper omits. A surface that computes and returns is not the same kind of object as a surface that writes to a shared store, and neither is the same as one that deletes. The effect class is derivable from a read — what does this entry point call, transitively, and does any of it reach a mutation primitive — and it is the parameter that determines which risk tier the surface sits in and therefore what evidence its verification requires. A map without effect classes tells you what exists; a map with them tells you what matters.

12.5 The ingress field carries the security dimension and is the one that makes the missing deny test discoverable. A surface reachable only by a privileged caller, whose tests all exercise the privileged path, has been verified for the case that was never in doubt. The interesting assertion is the negative one: that an unprivileged caller is refused. The map makes this checkable by construction, because a surface whose ingress includes an unprivileged caller and whose protection set contains no assertion of refusal is a structural finding that requires no judgement to identify. It is a join between two fields, and it is the security-without-a-contract class of the taxonomy reduced to a query.

12.6 The map is built by registration rather than by inference wherever registration exists, and this is the mapper's central design rule. Nearly every framework in every domain requires that an entry point be registered — a route table, a command manifest, an event subscription list, a job schedule, a hook declaration, a plugin descriptor. Registration is a declaration by the code itself, it is machine-readable, and it is complete by construction because an unregistered entry point does not execute. Inferring surfaces from naming conventions or from call graphs is a fallback for the cases where no registry exists, and it is a fallback rather than a method because it produces both false positives and, worse, silent false negatives.

12.7 The completeness claim is the mapper's most dangerous output and must be stated with its basis or not at all. A map that claims to be complete is claiming that no entry point exists outside it, and that is a claim about a complement, which is exactly the class of claim that this dataset spends its length warning against. The honest form is bounded: this map enumerates every surface derivable from the registries at these paths, and it does not cover entry points registered dynamically at runtime, reached through reflection, or exposed by a dependency rather than by this code. Those exclusions are listed, they are the map's scope field, and a map without a scope field is asserting total coverage of an unbounded space.

12.8 The map is per-unit and never per-repository, and this follows directly from the topology rules. A monorepo has as many maps as it has manifests. A hybrid has one per language context. Merging them produces a document that is true of nothing, because the surfaces of a service and the surfaces of a front-end application are not comparable objects and any figure computed across them is an average over incommensurable things. The maps may be indexed together — an estate wants to ask which surfaces anywhere touch value, or which anywhere are reachable by an anonymous caller — but indexing is not merging, and the index preserves the unit boundary in every record.

12.9 The map's relationship to the knowledge graph must be stated so that this Part is not read as a licence to build a second one. The surfaces, their contracts, their protections, and the failures bound to them are nodes and edges, and this estate already owns a coordinate graph that holds exactly such things and that is populated and queried. The correct behaviour is to contribute the surface node type and its edges to that graph, not to stand up a rival structure alongside it. Two graphs over one estate is not redundancy; it is divergence, and the divergence is discovered when one of them is stale and nobody can say which.

12.10 The map's expiry is its final contract field and the one that keeps it honest. A surface map is a statement about a commit. A route added tomorrow is not in it. A command registered by a dependency upgrade is not in it. The map therefore carries the commit it was derived from, and any claim citing it inherits that commit, and a claim about a surface map without one is a claim about a repository that may no longer exist. The expiry is not a weakness of the instrument. It is the property that prevents an inventory from silently becoming a description, which is the transformation this entire dataset exists to prevent.

PART XII FINAL LAW

Coverage measures which lines a suite executed; the surface map measures which entry points production exposes, and the gap between them is where the interesting defects live. Six fields, none optional: identity, kind, ingress, effect class, contract, protection — and protection may legitimately be empty but never silently empty. Build from registries, because an unregistered entry point does not execute; infer only as a fallback, because inference produces silent false negatives. A map claiming completeness is claiming something about a complement: state the scope or make no claim.


PART XIII — SURFACES: COMMANDS, HTTP ROUTES, EVENT HANDLERS, SCHEDULED JOBS, QUEUES

13.1 The surface taxonomy exists because each kind of entry point has a different registry, a different ingress model, a different set of failure modes, and a different reason it gets forgotten. A mapper that treats all entry points as equivalent will find the ones that are easy to find and will miss precisely the ones that nobody remembers, and the ones nobody remembers are the ones with no protection, because protection is written by people who are thinking about the surface. The five kinds in this Part share the property that they are invoked by something outside the process's own call stack; the seven in the next share the property that they are invoked by the machinery of the system itself.

13.2 The command surface is the entry point invoked by an actor typing or dispatching an instruction, and it is found in a registry: a command manifest, a subcommand table, a plugin descriptor's command block. Its ingress is the most varied of any surface, ranging from an anonymous participant to a privileged operator, and its contract is usually stated informally in a help string that nothing verifies. Its characteristic failure is that its permitted paths are tested and its refusals are not. A command that mutates shared state and carries no assertion that an unprivileged caller is refused has an ingress field and an empty protection field in exactly the position that matters, and the map surfaces that join without anyone having to think about it.

13.3 The command surface in a plugin context carries an additional property that raises its risk class independently of anything in its code. It executes in a live, shared, multi-participant environment where its effects are frequently irreversible: value moved between actors, world state altered, a participant's holdings changed. There is no rollback for a live world. In the founding audit, the component in that estate's plugin tree that governs the movement of value between actors carried no test reference anywhere in the entire tree, while the tree as a whole produced a very large body of passing assertions. Its two module-level tests guard a route guard and a legacy colour parser. The command surface with the highest effect class had the emptiest protection field.

13.4 The route surface is the entry point invoked over a network by a party the process has never met, and it is the most thoroughly registry-backed surface of all: every web framework maintains a route table, and the table is complete by construction because an unregistered route is unreachable. Its ingress model is the richest — anonymous, authenticated, authorized, internal — and its contract is usually the most explicit, because a route has a request shape and a response shape and a status code, all of which are machine-checkable. Its characteristic failure is not that it is untested but that it is tested only for the shapes the author anticipated, which is the ignored-edge-case class arriving from a party who did not read the author's assumptions.

13.5 The route surface's second failure mode is the one that the surface map is uniquely positioned to catch, and it is a mismatch between the ingress the route declares and the ingress its tests exercise. A route whose middleware asserts authorization, and whose entire test set constructs an authorized request, has verified that the route works and has verified nothing about whether the middleware fires. The middleware is not the route's implementation detail; it is the route's contract with everyone who is not supposed to reach it. Every route whose ingress is anything other than anonymous therefore requires an assertion that an unauthorized request is refused, and a route without one is recorded as unverified for its authorization contract regardless of how many tests it has.

13.6 The event-handler surface is invoked by something happening rather than by someone asking, and this is what makes it the most frequently forgotten surface in every domain that has one. Nobody calls it. It does not appear in a route table that anyone reads or in a help output that anyone runs. It sits in a subscription list, it fires when the host emits, and its author is often the only person who ever knew it existed. Its contract is the shape of the event, which is owned by the emitter rather than by the handler, and its characteristic failure is that the emitter's shape changed and the handler continued to run, silently doing nothing or silently doing the wrong thing, because a handler that receives an unexpected shape rarely crashes.

13.7 The event handler also carries the domain's worst observability profile, and observability is what determines how long a defect lives. A route that breaks produces an error rate. A command that breaks produces a complaint. A handler that breaks produces silence, because nobody is watching a subscription, and the only symptom is that something downstream gradually stops being true. The meaningful assertion for a handler is therefore a contract test against the emitter's real event shape, executed as a seam rather than as a unit, and the artificial one is a unit test that constructs the event object by hand — which asserts that the handler processes the shape the author believes the emitter sends, which is precisely the belief in question.

13.8 The scheduled-job surface is invoked by a clock, and it inherits every weakness of the event handler while adding two of its own. The first is that its trigger is time, which means its failure mode is often that it did not run at all, and a job that did not run produces no output and therefore no signal — the absence of its effect is indistinguishable from the absence of any work to do. The second is that it is the surface most likely to hold destructive logic: pruning, expiry, rotation, cleanup. A destructive operation with a clock trigger and no monitor is the highest-risk surface in most repositories, and it is almost never the surface anyone thinks to test, because it is invisible in normal operation by design.

13.9 The queue-consumer surface is invoked by a message, and its distinguishing property is that its contract has three parts rather than one: the message shape, the delivery semantics, and the failure semantics. A consumer that is correct for a well-formed message may be catastrophic under redelivery, because the operation it performs is not idempotent and the queue guarantees at-least-once. This is the failure the unit test structurally cannot reach: the unit is correct, the assertion is strong, the fixture is realistic, and the defect lives entirely in the interaction between the handler and the delivery guarantee. The meaningful assertion is a redelivery test — deliver the same message twice, assert the state is the same — and its absence is a structural finding on every consumer whose effect class is anything but read-only.

13.10 The five surfaces in this Part are unified by a single diagnostic that the map makes trivial. For each, ask what invokes it, what it changes, and what asserts that it refuses to change what it should not. The three questions are three fields in the inventory, the join between them is a query rather than a judgement, and the surfaces whose protection field is empty at a high effect class are the estate's actual remediation queue — which, in the founding audit, bore no resemblance to the queue that a coverage tool would have produced.

PART XIII FINAL LAW

Each surface kind has its own registry, ingress model, and reason for being forgotten. A route tested only with authorized requests has verified the route and not the middleware, which is its contract with everyone not supposed to reach it. An event handler produces silence when it breaks, because nobody watches a subscription. A scheduled job's failure is often that it never ran, and its absence of effect is indistinguishable from an absence of work. A queue consumer's defect lives in the delivery guarantee, where no unit test can reach it: deliver twice, assert the state is unchanged.


PART XIV — SURFACES: HOOKS, PLUGINS, AGENTS, SCRIPTS, BUILD STAGES, MIGRATIONS, HARDWARE ENTRY POINTS

14.1 The surfaces in this Part are invoked by the machinery of the system rather than by an external actor, and that single property is why they are systematically under-protected. A surface that a user reaches is a surface someone thought about. A surface that the build reaches, or the deployment reaches, or the framework reaches, is infrastructure, and infrastructure is assumed to work because it is the thing that everything else runs on. The assumption is precisely inverted: these surfaces are the ones whose failure is most expensive, most silent, and least likely to be caught by anything, and in this estate they are where the highest-severity findings of the founding audit concentrated.

14.2 The hook surface is an entry point invoked by a host at a lifecycle moment — before a write, after a tool call, at session end — and it is the purest instance of the built-but-unwired failure. A hook exists as a file. It is correct. It has tests. And it is reachable only if it is registered with the host's dispatcher, and registration is a separate act in a separate file that nothing verifies. This estate has shipped gates that were never wired to an event, gates whose refusal was discarded by the layer that aggregated hook outputs, and gates whose canonical source in version control diverged from the live copy that actually loaded. All three produce a control that everyone believes is enforcing and that enforces nothing.

14.3 The hook's protection field therefore requires a field no other surface requires: liveness. A hook's tests may all pass while the hook is inert, and the passing tests are what make the inertness invisible. The only admissible evidence that a hook is live is that it has been observed to fire — and, for a hook whose purpose is to refuse, that it has been observed to refuse. Construct the condition it exists to catch, present it, and watch. This is a five-minute test, it must be performed at the moment the hook is built rather than at some later audit, and its absence leaves the hook in exactly the epistemic position of a regression test never seen to fail: it may be working, it may be inert, and from the outside, where everything is green, the two produce the same green.

14.4 The plugin surface, considered as a surface rather than as a domain, is the entry point at which a host loads foreign code and grants it authority. Its contract is the host's interface version, and its characteristic failure is that the interface changed and the plugin still loads, failing at the first call into the changed method rather than at load time. Its ingress is the host's own lifecycle, and its effect class is whatever the host permits, which is usually everything. The meaningful assertion is a load-and-enable test against the target host version, and its artificial counterpart is a unit test of the plugin's internal logic, which proves the logic and says nothing about whether the host will accept the artifact at all.

14.5 The agent surface is the entry point at which an autonomous process is dispatched with a task and a tool set, and it is the newest surface in the taxonomy and the one with the weakest conventions. Its contract is its permission set: what tools it may call, what paths it may write, what it may not do. Its characteristic failure is permission drift — an agent that acquires, through a configuration change nobody reviewed, the ability to modify the very rules that constrain it. Its meaningful assertion is a negative one, and it is the same negative the entire estate depends on: that an agent cannot amend the gate that is currently blocking it. The assertion must be executable, because a separation of powers that lives only in a document is enforced by nothing.

14.6 The script surface is the entry point that a human runs by hand, and it is the surface that no framework registers and therefore no mapper finds by registry. It lives in a tools directory, it has no manifest entry, it is invoked from a readme or from memory, and it frequently carries the highest effect class in the repository: it migrates, it deletes, it publishes, it rotates. It is the canonical example of the surface that inference must find because registration will not, and the fallback rule applies — enumerate executables by convention, accept the false positives, and record the enumeration's basis so a reader knows what it may have missed.

14.7 The build-stage surface is the entry point invoked by the build system, and it is the surface whose failure is most likely to be interpreted as a product defect. A build stage that fails produces a compiler error, and a compiler error looks like broken code. In the founding audit, a plugin tree's prebuilt test classes had been compiled against a newer language version than the audit host offered, and the build refused with a message about a more recent version; two service repositories refused to compile at all because their dependencies were unresolved and unlocked. In none of those cases was the product defective. The build stage is a surface, its contract is the toolchain it requires, and a violated toolchain contract is an environment finding routed to blocked.

14.8 The migration surface is the entry point that changes persisted state, and it is the only surface in the taxonomy whose defects are, in the general case, irreversible. Its contract has two halves, and estates routinely author one: the forward transformation is written, exercised against a small development store, and shipped, while the reverse transformation is either absent or is present and has never been executed. A rollback described in a document is a hypothesis with the word rollback written on it. The meaningful assertion is an executed round trip — apply, verify, reverse, verify — against a store whose shape resembles the real one, and the artificial one is a schema-parse check that proves the migration file is syntactically valid.

14.9 The hardware entry point is the surface at which software meets a substrate the host cannot simulate, and its treatment closes the taxonomy because it is the case where the surface map's evidence field does the most work. Its contract is a timing, a bus protocol, an interrupt latency, or a register layout, and no host-side test can produce admissible evidence about it. A capture from the device is strong evidence. A host simulation is medium evidence, admissible for logic and inadmissible for timing. In this estate a hardware-coupled title's drivers include the device's graphics abstraction headers, which means every one of its entry points is of this kind, and the honest verdict for all of them on a host is blocked.

14.10 The seven surfaces in this Part share one instruction, and it is the inverse of the instruction that governs the previous five. For an external surface, ask who can reach it. For an internal surface, ask what proves it is reached at all. The hook that never fires, the plugin that never loads, the script that nobody runs, the migration whose reverse was never executed, and the gate whose refusal is discarded are all the same defect wearing different registries, and the defect is that presence was mistaken for connection.

PART XIV FINAL LAW

Internal surfaces are under-protected because infrastructure is assumed to work, and the assumption is exactly inverted. A hook's tests can all pass while the hook is inert, and the passing tests are what make the inertness invisible — the only admissible evidence of liveness is an observed firing, and for a refusing gate, an observed refusal. A build failure is a violated toolchain contract, not a product defect. A rollback described in a document is a hypothesis with the word rollback written on it. For external surfaces ask who can reach them; for internal ones, ask what proves they are reached at all.


PART XV — BINDING EACH SURFACE TO ITS TESTS, MONITORS, CONTRACTS, AND EVIDENCE

15.1 The binding is the operation that converts an inventory into an instrument, and it is the operation that no ordinary tooling performs. A test count is a property of a suite. A coverage figure is a property of a file. Neither is a property of a surface, and a surface is the only object whose protection anybody actually cares about. The binding asks, for each entry point in the map, which tests exercise it, which monitors would observe its failure in production, which contract states what it promises, and which evidence licenses any claim that it is protected. Four edges per surface, each of which may legitimately be empty, none of which may be silently empty.

15.2 The test edge is built by reachability rather than by naming, and the distinction is the difference between a real binding and a decorative one. A test is bound to a surface if invoking the test invokes the surface — not if the test's filename resembles the surface's name, and not if the test lives in a directory whose name matches the module. Naming conventions are a hint and are frequently wrong: a test named for a component may exercise a helper the component no longer uses, and a component may be exercised thoroughly by a test that names something else entirely. The binding is therefore established by execution with instrumentation, or, where that is unavailable, by a symbol search whose limitations are recorded as part of the edge.

15.3 The symbol-search fallback is worth describing precisely because it is what the founding audit used and what it found. A search across every test file in a plugin tree for any reference to the component governing the movement of value between actors returned nothing. That is a weaker instrument than instrumented reachability — a test could conceivably reach the component through an intermediary without naming it — and it is decisive in the negative direction, because a component that is not named anywhere in any test, and that is not reachable through any named intermediary, is not being exercised by anything. A zero from a symbol search is not proof of absence in general; it is proof of absence when the component is the entry point of its own subsystem, and it was.

15.4 The monitor edge is the one that estates omit universally, and its omission is what determines how long a production defect lives. A test tells you whether a surface was correct at a commit. A monitor tells you whether it is correct now. For every surface whose effect class is above local, the binding asks what signal would reveal a failure in production and what is watching that signal, and the honest answer for most surfaces in most repositories is nothing — which is a finding, and which is the production-behaviour-without-a-monitor class of the sealed taxonomy. A surface with a strong test edge and an empty monitor edge is protected against the defects that existed at the commit and is blind to every defect introduced by the environment it runs in.

15.5 The contract edge names what the surface promises, and its absence is the quietest of the four failures because nothing breaks when it is missing. A surface without a stated contract can still be tested — the tests assert whatever the current implementation does — and the resulting suite is a photograph of the behaviour rather than an assertion about it. Such a suite cannot distinguish a defect from a design, it will fail on every legitimate change, and it will pass on any wrong behaviour that was wrong at the moment the test was written. The contract is what makes a test's failure meaningful, because it is what the failure is a failure against.

15.6 The evidence edge is what makes the binding auditable, and it is where this dataset composes rather than rebuilds. Every protection claim on a surface carries the evidence that licenses it, and evidence is classified by the admissibility tiers the constitution defines: a complete runner output that reached this surface is strong, a directed test is medium, and a reading of the test file that concludes it probably exercises the surface is weak. The strength of the protection claim is capped by the weakest evidence supporting it, and the cap is not negotiable by volume — ten readings of a test file do not sum to an execution. The epistemic level attached to the claim is the estate's existing ladder, not a scale invented here.

15.7 The binding's most valuable output is the empty set, and this inverts how the artifact is normally read. A surface whose test edge is empty is a finding. A surface whose monitor edge is empty and whose effect class is destructive is a finding of higher severity. A surface whose contract edge is empty is a finding about the tests that do exist. The instrument is therefore consulted for its holes rather than for its fills, and the holes are exactly what a coverage report cannot show, because a coverage report is computed over the lines that exist and has no representation for a surface that nothing reached.

15.8 The binding is also what makes the deny-test gap mechanically discoverable, and this is the join that justifies the entire six-field inventory. Take every surface whose ingress admits an unprivileged or anonymous caller, whose effect class is above read-only, and whose test edge contains no assertion of refusal. That is a query over three fields. Its result is the estate's authorization exposure, it requires no judgement to produce, and in the founding audit it would have surfaced the value-moving component in the plugin tree without anyone needing to have the insight, because the insight is a join rather than an idea.

15.9 The binding's records feed the estate's existing pipelines rather than a new one, and the routing must be stated so that this Part is not read as a licence to fork. A surface with an empty test edge and a high effect class is a candidate failure record, and the record is produced here and handed to the layer that transmutes an insight into its destination form — a structural rule, a process rule, a trap, a corpus entry, a benchmark case, or a considered discard. This dataset produces records. It does not route them, it does not score them against a done-gate, and it does not maintain a registry of laws, because each of those organs exists, works, and is fired by something else in this estate.

15.10 The binding expires with its commit, and its expiry is more aggressive than the map's. A map changes when a surface is added. A binding changes when any test, monitor, contract, or piece of evidence changes, which is on nearly every commit. A binding cited without its commit is therefore not merely stale in the ordinary way; it is a claim about a relationship that may have been severed by a change to either endpoint, and the severing is silent — deleting a test does not announce that a surface has become unprotected. That silence is the exact reason the binding must be recomputed rather than remembered.

PART XV FINAL LAW

Test counts are properties of suites and coverage is a property of files; neither is a property of a surface, and the surface is the only object whose protection anyone cares about. Four edges per surface — tests, monitors, contract, evidence — each of which may be legitimately empty and none of which may be silently empty. The instrument is read for its holes, not its fills, because a coverage report has no representation for a surface that nothing reached. The authorization exposure of an entire estate is a three-field join, not an insight.


PART XVI — FINDING PRODUCTIVE LOGIC WITH NO PROTECTION DESPITE A LARGE TOTAL TEST COUNT

16.1 This Part addresses the single most dangerous configuration in software verification, and it is dangerous precisely because every visible indicator is favourable. The suite is large. The suite is green. The count is impressive and is quoted with justified pride. And somewhere inside the tree, the component that a hostile actor would attack first, whose defects are irreversible, and whose failure harms every participant, has no assertion of any kind bound to it. The aggregate conceals the distribution, the distribution is where protection actually lives, and no metric in ordinary use has any representation for the difference.

16.2 The founding instance is exact and must be stated without embellishment because its power comes from its ordinariness. A plugin tree in this estate produced, on a corrected toolchain, a very large body of passing assertions — a number well into four figures — across hundreds of test files. A search across every test file in that entire tree for any reference to the component that governs the movement of value between actors returned zero. The two tests in that component's own module guard a route guard and a legacy colour-parsing routine, and neither touches the value logic. The most abusable surface in the system runs in production with nothing asserting its behaviour, and the repository's headline number is excellent.

16.3 The mechanism that produces this configuration is not negligence, and diagnosing it as negligence guarantees it will recur. Tests accumulate where testing is cheap. A component with pure functions, no host dependency, and a stable interface is pleasant to test, and it acquires tests. A component that requires an in-memory host, a populated world, a participant with holdings, and a concurrency assumption is expensive to test, and it does not. The distribution of tests across a tree is therefore a map of testing convenience, not a map of risk, and the two are frequently anticorrelated because the components that are hard to test are hard to test for the same reasons that make them dangerous.

16.4 The second mechanism is the aggregate itself, which actively suppresses the signal. A tree with one component holding the overwhelming majority of its test files and dozens of components holding almost none reports, at the root, a healthy figure. Every incentive that operates on that figure — a badge, a threshold, a report to a stakeholder — is satisfied. Nobody is lying, nobody is careless, and the number is arithmetically correct. What the number cannot do is fall when the distribution worsens, because adding a thousand tests to the well-tested component raises it exactly as much as adding one test to the unprotected one, and the former is easier.

16.5 The detection procedure follows directly and is mechanical rather than intuitive. Take the surface map. Filter to surfaces whose effect class is shared, persisted, external, or destructive. Join to the binding's test edge. Sort by effect class descending and by test-edge cardinality ascending. The top of that list is the answer, and it is an answer that requires no insight, no domain expertise, and no suspicion — it is a query over an inventory the scanner already produced. In the founding audit, this ordering would have placed the value-moving component at the top of a repository whose headline number was among the best in the estate.

16.6 The procedure's counterintuitive property is that it is entirely insensitive to test count, and the insensitivity is the point. It does not ask how many tests exist. It does not ask what the coverage is. It asks, for each surface that can cause harm, whether anything asserts against it, and the answer is binary and is usually available without executing anything. A repository with ten tests can pass this check if the ten guard the ten surfaces that matter. A repository with four thousand can fail it, and one in this estate does, and that inversion is the entire reason the check exists.

16.7 The countermetric that must accompany any test count follows from this Part and is stated in the ontology as risk-weighted coverage. A count that is not weighted by the effect class of what it protects is a claim about volume, and volume can be increased indefinitely without protecting anything additional. The pairing is not a refinement; it is a defence, because a metric shipped alone becomes a target and a target that can be satisfied by writing more tests where tests already are will be satisfied that way, sincerely, by people who believe they are improving the system.

16.8 The same procedure surfaces the mechanism gap, which is a distinct finding and must not be collapsed into the coverage gap. In the estate's plugin tree, the mechanism that makes headless testing possible — an in-memory implementation of the host — was found wired into exactly one build descriptor out of many. The mechanism works. It is proven in the component that uses it. It has not propagated. That gap will not be closed by writing tests, because the components that lack tests also lack the ability to run them, and an instruction to increase coverage in those components is an instruction to do something that is currently impossible. Diagnosing the gap as a coverage failure produces a remediation that cannot succeed.

16.9 A large test count also conceals a second class of hollow protection that this procedure does not catch and that must be named so the procedure is not oversold. A surface may have a test edge that is non-empty and whose assertions are too weak to fail: the test checks that a result was returned rather than that the correct result was returned, or checks a type rather than a value. The binding reports that surface as protected, because a test reaches it, and the report is wrong. The only instrument that distinguishes a real assertion from a decorative one is a mutation check — break the code deliberately and demand that the suite notice — and that is why mutation adequacy is the mandatory countermetric to coverage and why an empty-set analysis is a floor rather than a ceiling.

16.10 The instruction that closes this Part is the one that should be executed first in any new repository, before any test is written and before any coverage tool is installed. List the surfaces that can cause irreversible harm. For each, ask what asserts that it refuses to do so. The list is short, the question is binary, the answer is available in an afternoon, and it reorders every remediation queue in the estate — which, in the founding audit, it did: the top of the queue was not the repository with the fewest tests but the environments that could not build, the configuration line that orphaned hundreds of passing tests, the harness that polluted its own host, and the unprotected component inside the repository with the largest green number.

PART XVI FINAL LAW

Every visible indicator can be favourable while the component that a hostile actor would attack first has no assertion bound to it: a large green count, a concentrated distribution, and a value-moving surface with zero test references anywhere in the tree. Tests accumulate where testing is cheap, and cheapness is anticorrelated with danger, so the distribution of tests is a map of convenience rather than of risk. The detection is a query, not an insight: filter surfaces by effect class, join to the test edge, sort ascending. A count that is not weighted by what it protects is a claim about volume.


PART XVII — UNKNOWN-STATE PRESERVATION: THE HONEST STATES AND WHY THEY MUST NEVER ROUND TO PASS

17.1 An unknown is a value, not an absence, and the entire integrity of this dataset rests on that sentence. Every stage described so far produces holes: a manifest that could not be parsed, a language whose runner could not be identified, a suite that was counted and not executed, an environment that would not build, a surface whose ingress could not be determined from any registry. Each of those is a genuine result — the instrument looked and did not find — and the temptation at every one of them is to fill the hole with the most plausible value, because a report with holes looks like an incomplete report and a report without them looks finished. The filling is the defect. A hole that is honestly recorded directs attention; a hole that is filled with a plausible default is a fabrication with an inventory's authority behind it.

17.2 The rounding pressure is not a character flaw and cannot be legislated away by exhorting people to be careful. It is structural, and it operates in one direction only. An unknown that is rounded down to a failure produces a false alarm, and false alarms are noticed, resented, and corrected. An unknown that is rounded up to a pass produces silence, and silence is never corrected, because nothing in the pipeline is positioned to notice that a green field was never actually measured. The asymmetry means that any process which permits rounding at all will, over time, round exclusively upward, not through anyone's decision but through the differential survival of the two errors.

17.3 The honest states are the ones the constitution enumerates, and this dataset's contribution is to specify which of them each of its own holes maps to, so that the mapping is mechanical rather than discretionary. An environment that will not build is blocked. A suite that exists and was not executed is unverified. A repository with no executable surface returns not-applicable for coverage. A failure attributable to a contaminated harness is environment-invalid. A surface whose protection could not be determined is unverified, not unprotected — the distinction between those two matters, because unprotected is a finding about the repository and unverified is a finding about the instrument.

17.4 The blocked state is the one this estate exercised most heavily and the one whose correct use paid the highest return. Two service repositories could not be compiled: the toolchain refused because their dependencies were unresolved and unlocked. Between them they carry well over a hundred test files. An undisciplined audit records those repositories as failing, and the record dispatches an engineer to repair product code that was never broken, and the engineer, being competent, will change the product until the harness stops complaining. The honest record is that they are unverifiable in this environment, that the fault is environmental, and that restoring their build converts two entire repositories from unknown to knowable, which made it the highest-return action in the estate.

17.5 The environment-invalid state is the blocked state's sharper sibling, and its instance is the cleanest attribution lesson in the audit. A data-pipeline repository produced a suite that was almost entirely green with a single failure, and the failure traced to a test that wrote outside its allocated temporary space to a location at the host's drive root, and the pollution defeated a sibling suite. The product logic was correct throughout. The red was real, it was reproducible, and it belonged to the harness. Attributing it to the product would have destroyed correct behaviour; attributing it to the environment identified the actual remedy, which is to point the offending test at its temporary directory.

17.6 The counted-but-not-executed case is where the unverified state earns its place, and this estate's handling of it is a model worth copying. Front-end specification files were counted across three repositories and were deliberately not run, because running them required installing dependencies with side effects that the audit's own scope forbade. The audit did not estimate their result. It did not assume they would pass because they were maintained. It did not omit them, which would have made its silence about them indistinguishable from an assertion that they did not exist. It recorded the count, the reason for non-execution, and the verdict that the evidence licensed, which was unverified — neither passing nor failing.

17.7 The scope boundary is therefore an artifact rather than an embarrassment, and stating it explicitly is what converts a limitation into a finding with an owner. The audit's boundaries were enumerated: it did not resolve the Elixir dependency sets, it did not provision the database those suites require, it did not stand up the device toolchain the hardware-coupled repository needs, it did not run the largest suite to completion, and it did not delete the polluted host path because doing so would have violated a standing destructive-operation rule. Each of those is a sentence, each names work that someone can now do, and each closes a hole that would otherwise have been filled by an assumption.

17.8 The not-applicable state is the one that a naive pipeline will never emit and that a correct one must, and its instance is the founding correction of this entire dataset. A content and knowledge repository has no executable surface, and its coverage is not zero. Zero is a measurement; not-applicable is a statement that the measurement does not exist. The difference is not pedantic: zero places the repository at the top of every remediation queue that sorts by coverage, attracts attention proportional to a gap that does not exist, and generates a self-reinforcing stream of findings, each of which reconfirms the misclassification because each is produced by the same tooling that made it.

17.9 The preservation requirement extends downstream and is the part most often lost. An unknown that is recorded honestly at the point of measurement and then absorbed into an average, a percentage, or a summary sentence has been rounded after all, and the rounding is now invisible because it happened in a transformation rather than in a judgement. A percentage cannot represent an unknown; it can only exclude it or count it as zero, and both destroy it. The rule is therefore that any aggregate over a set containing unknowns must carry the unknown count alongside it, and that an aggregate which cannot do so must not be computed. A number that cannot be produced honestly is better absent than present.

17.10 The closing instruction is a question asked at every field the scanner writes, and it takes a second. Did I observe this, or did I infer it. If observed, record it with its basis. If inferred, record the unknown and the inference separately, so that a downstream reader can accept the inference or reject it with full knowledge of what it rests on. There is no third option, and the absence of a third option is what separates an inventory from a description — which is the distinction this dataset was founded to defend and the one that every pressure in the pipeline is working to erase.

PART XVII FINAL LAW

An unknown is a value, not an absence, and the pressure to fill it operates in one direction only: an unknown rounded down produces a false alarm that gets corrected, and an unknown rounded up produces a silence that never does. Blocked, unverified, environment-invalid, and not-applicable are results, not failures of the instrument. A content repository's coverage is not zero — zero is a measurement, and not-applicable is the statement that the measurement does not exist. Any aggregate over a set containing unknowns must carry the unknown count, and an aggregate that cannot must not be computed.


PART XVIII — FAILURE MODES AND ADVERSARIAL CASES OF THE SCANNER ITSELF

18.1 The scanner is an instrument, and every instrument in this corpus is subject to the doctrine it enforces, which means the scanner's own claims are hypotheses about the disk until something checks them. This Part enumerates the ways it is wrong, because a scanner whose failure modes are undocumented is worse than one that is merely imperfect: its consumers cannot distinguish the regions where it is reliable from the regions where it is guessing, and they will therefore trust it uniformly, which is exactly the trust the rest of this dataset spends its length withholding from every other description of a repository.

18.2 The first failure mode is the vendored tree. A repository that carries a copy of a dependency's source, or a generated artifact directory, or an example project, presents the scanner with files whose extensions, manifests, and test conventions belong to something other than the project under inspection. A naive census counts them, and the resulting classification, language ratio, and test count are all inflated by code the project does not own and is not responsible for. The countermeasure is that the manifest, not the tree, defines the unit, and that any directory the manifest does not claim is excluded and its exclusion is recorded — because a directory excluded silently is a hole, and this dataset does not permit silent holes.

18.3 The second failure mode is the generated artifact, and it is the vendored tree's more insidious relative because it looks like first-party code. Build outputs, compiled classes, transpiled sources, and dependency caches all live inside the tree and all resemble the project. A test-file census that counts a compiled test class alongside its source counts the same test twice; a language census that counts a transpiled output counts a language nobody wrote. The countermeasure is the same as the first — claimed by a manifest or excluded — and the residual risk is that a project's manifest does not always list its own build output directory, which means the exclusion list carries a heuristic component whose basis must travel with it.

18.4 The third failure mode is the false test file, and it produces the most confidently wrong number the scanner emits. A file whose name matches the runner's discovery convention is not necessarily a test: it may be a fixture generator, a shared helper, a data builder, or a module that the convention captures by accident. Counting it inflates the authored surface, which inflates the gap between authored and executed, which produces a finding about orphaned tests that do not exist. The countermeasure is that the authored-surface count is qualified by what the runner itself would collect from the file, and where that qualification cannot be made cheaply, the count is recorded as an upper bound rather than as a figure.

18.5 The fourth failure mode is the dynamic surface, and it is the one the surface mapper cannot fix. An entry point registered at runtime — by reflection, by a plugin loader, by a configuration-driven dispatcher, by a decorator evaluated at import — does not appear in any static registry. The map will not contain it, and the map's completeness claim, if it makes one, is false. The countermeasure is not a better parser; it is the scope field. The map states what it enumerated and what class of registration it cannot see, and a consumer who needs certainty about a dynamically registered surface must obtain it by executing the system and observing the registry at runtime, which is a different instrument in a different phase.

18.6 The fifth failure mode is the stale scan, and it is the one that turns the scanner into precisely the thing it was built to replace. A scan is a statement about a commit. A scan cached, reused, and cited without its commit is a description of a repository, produced once, trusted thereafter — which is the exact epistemic object that this dataset's first Part disqualifies. The countermeasure is structural rather than procedural: every scan output carries its commit, every claim citing a scan inherits it, and a claim whose scan commit differs from the current one is not stale in the ordinary sense of being slightly out of date; it is unverified, and it must be re-derived rather than adjusted.

18.7 The adversarial cases are distinct from the failure modes because they involve an agent that benefits from the scanner being wrong, and the relevant agent is usually the one whose work is being scanned. The scanner can be defeated by adding a manifest that declares a runner nobody uses, which makes the automation map show an edge that does not exist in practice. It can be defeated by a test file that imports nothing and asserts nothing, which raises the authored count. It can be defeated by a hook that is registered and returns success unconditionally, which passes a liveness check by firing without ever refusing. Each of these passes every static check the scanner performs.

18.8 The defence against all three is identical and is inherited rather than invented: liveness is proven by observed refusal, never by observed presence. A gate that has fired is not thereby a gate. A gate that has refused the condition it exists to catch is a gate. The check is to construct the condition, present it, and watch, and it is the same five-minute test that the estate's own history demands — because this estate has shipped gates that were never wired, gates whose refusals were discarded by the aggregation layer, and gates whose committed source diverged from the live copy that ran, and all three produced a control that everyone believed was enforcing and that enforced nothing.

18.9 The scanner's most dangerous property is that it is trusted more than the artifacts it replaces, and that trust is earned and is therefore hard to withhold. It is grounded in observation, its outputs carry provenance, and it has been right before. None of that licenses a claim today. The constitution's rule that a system's track record confers no evidentiary weight on its current output applies to this instrument with full force: a scan is exactly as strong as the reads beneath it, and its history of being correct is used to decide how much scrutiny to apply, never to decide whether a specific field is true.

18.10 The scanner is therefore subject to the same treatment as any other surface in this dataset, and the symmetry is the point rather than a flourish. It has entry points. It has an effect class, which is advisory rather than destructive and is therefore lower than most, but not nil, because an inventory that is wrong misdirects every action taken on it. It has a contract, which is that every field is an observation with its basis or an explicit unknown. And it has a protection set, which is a suite of adversarial fixtures — a vendored tree, a generated directory, a false test file, a dynamic registration, a hook that fires without refusing — each of which the scanner must be observed to handle correctly. A scanner that has never been run against a repository designed to fool it is an unverified instrument, and its outputs inherit that state.

PART XVIII FINAL LAW

The scanner is subject to the doctrine it enforces: its claims are hypotheses until something checks them, and its history of being right confers no weight on today's field. Vendored trees, generated artifacts, false test files, dynamically registered surfaces, and stale caches are its documented failure modes, and each is countered by the manifest-defines-the-unit rule plus an explicit scope field. Its adversarial defeat is a gate that fires without ever refusing, and the only defence is the same five-minute test the estate's own history demands: liveness is proven by observed refusal, never by observed presence.


PART XIX — METRICS, ANTI-GOODHART COUNTERMETRICS, AND THE MATURITY LADDER

19.1 Every metric in this dataset ships with a countermetric, and the pairing is not a refinement of the metric but a defence against it. A metric that ships alone becomes a target, a target is optimized, and the optimization of a proxy is indistinguishable from the improvement of the thing it proxies right up until the moment somebody looks. The countermetric is chosen so that it moves adversarially when the first is gamed: the cheapest way to inflate the metric must be the way that degrades its partner. A metric defined without its countermetric is not partially specified. It is an attack surface, and it will be attacked by sincere people who believe they are improving the system.

19.2 The first metric is surface coverage — the fraction of enumerated surfaces whose test edge is non-empty — and its countermetric is effect-weighted surface coverage, in which each surface is weighted by its effect class. The pairing exists because surface coverage alone is gamed by binding a trivial assertion to every read-only entry point in the repository, which is cheap, produces a beautiful figure, and protects nothing. The weighted figure does not move when a read-only surface acquires a test and moves sharply when a destructive one does, which makes the cheap path visible in the same instrument that would otherwise have rewarded it.

19.3 The second metric is the authored-versus-executed reconciliation ratio, which compares the test files the scanner counted against the tests the canonical invocation actually collected, and its countermetric is the executed-surface reach. The ratio alone is gamed by deleting the orphaned tests, which raises it to unity instantly and destroys protection that was one configuration line away from being real. The reach — how many mapped surfaces the canonical invocation actually touched — falls when tests are deleted and rises only when they are connected, and the pairing therefore distinguishes the repository that fixed its configuration from the repository that deleted its inconvenient evidence.

19.4 The third metric is classification stability, the rate at which a repository's derived domain changes between scans, and its countermetric is classification challenge rate, the frequency with which a scan's classification is contradicted by the disk on re-derivation. Stability alone is gamed by caching, which produces perfect stability and reintroduces exactly the description-over-disk failure this dataset was founded to kill. The challenge rate can only be measured by re-deriving, which means a repository with perfect stability and a zero challenge rate has been re-derived and confirmed, while one with perfect stability and no challenge rate at all has simply not been looked at.

19.5 The fourth metric is surface-map completeness, expressed as the fraction of enumerated entry points whose six fields are all populated with observations rather than unknowns, and its countermetric is the unknown-preservation rate — the fraction of fields the scanner could not determine and honestly recorded as unknown. The pairing is the sharpest in the set, because completeness alone is gamed by exactly the behaviour this dataset most fears: filling unknowns with plausible defaults. A rising completeness figure with a falling unknown rate is either a genuinely better instrument or a scanner that has started guessing, and only the pair, read together, can tell the difference.

19.6 Two supreme measures sit above all four, and they are inherited rather than invented here. The fraction of observed failures that became permanent capability, and the fraction of quality claims that admissible evidence actually supports. Neither can be improved by making the estate look better. Neither is a target: an estate that sets a numeric objective for institutionalization will institutionalize trivia, because trivia is cheap and the number does not distinguish. They are read to detect drift, and the moment either appears with a target beside it, it has become the thing it was built to detect.

19.7 The maturity ladder for this dataset's own capability is observed, never declared, and its levels are specific to the reality-and-classification axis rather than borrowed wholesale from the family ladder. At the first level a repository has been scanned once and its inventory exists. At the second, the inventory is re-derived on a cadence and its classification has been challenged at least once by the disk. At the third, the surface map is complete against its stated scope and every surface carries an effect class. At the fourth, the binding is computed and the empty-set analysis has been run. At the fifth, the reconciliation between authored and executed protection is automated and its regression is a gate. At the sixth, the estate's surface maps are indexed together and a cross-repository query for an unprotected effect class returns in seconds. At the seventh, the scanner's own adversarial fixtures are part of the estate's suite and the instrument is verified as rigorously as the repositories it inspects.

19.8 The ladder's levels are not achievements to be claimed but states to be observed, and the observation is mechanical. A repository is at the level whose evidence exists, not the level its team believes it occupies, and the evidence for each level is an artifact: a scan output with a commit, a challenge record, a surface map with a scope field, a binding with an empty set, a gate that has been observed to refuse a reconciliation regression, an index that answers a query, an adversarial fixture that the scanner passed. A level claimed without its artifact is a description, and descriptions rank below disks.

19.9 The ladder also encodes the estate's actual position honestly, which is the only way a ladder is useful. This estate has performed the scan, has been challenged by the disk and corrected, has built the taxonomy, and has enumerated the surfaces of its worst offender by hand. It has not automated the reconciliation, its cross-repository index does not yet answer the unprotected-effect-class query in seconds, and its scanner's adversarial fixtures do not yet exist. Stating that plainly is not an admission of weakness; it is the difference between a ladder that measures and a ladder that flatters, and a ladder that flatters is a metric shipped without its countermetric.

19.10 The final anti-Goodhart provision applies to this Part itself. Every measure defined here can be computed by the instrument that this dataset describes, which means the instrument is grading its own output, which is the exact structure the constitution forbids in every other context. The provision is therefore that no metric in this Part may be cited by the scanner as evidence of the scanner's own correctness. The metrics describe repositories. The scanner's own correctness is established by the adversarial fixtures of the previous Part, produced by a different actor, and by nothing else.

19.11 The cadence at which these measures are read is itself load-bearing, and it must be fixed rather than triggered by concern, because drift is invisible from inside itself and is trivially visible only against a fixed reference. A measure consulted whenever someone happens to feel uneasy is consulted in the weeks when nothing is wrong and quietly skipped in the months when the answer would have mattered, since unease is not correlated with drift and is frequently anticorrelated with it. The scan is therefore re-derived on a schedule, the four pairs are recomputed on that schedule, and the delta between the current reading and the previous one is reported as a finding in its own right — because a classification that has silently changed, a reconciliation ratio that has silently fallen, or an unknown rate that has silently collapsed toward zero are each a signal that something in the estate or in the instrument has moved, and none of the three announces itself.

PART XIX FINAL LAW

A metric shipped without its countermetric is an attack surface, and it will be attacked by sincere people who believe they are improving the system. Surface coverage is paired with effect-weighted coverage, because binding a trivial assertion to every read-only entry point is cheap and protects nothing. The reconciliation ratio is paired with executed reach, because deleting the orphaned tests raises the ratio to unity instantly and destroys protection that was one configuration line from being real. Map completeness is paired with unknown preservation, because the cheapest way to complete a map is to start guessing. The ladder's level is the level whose artifact exists.


PART XX — INTERFACES TO SIBLING DATASETS, THE COMPOUND-INTELLIGENCE PIPELINE, AND DONE CRITERIA

20.1 This dataset occupies the first position in the verification axis and produces exactly three artifacts that every sibling consumes: the repository inventory, the domain classification, and the executable-surface map with its bindings. Everything downstream is a function of those three, which means that an error here is not a local error — it propagates into every judgement the axis subsequently makes, and it propagates while acquiring the appearance of rigour from the machinery applied on top of it. The interfaces below are stated as contracts rather than as suggestions, because a consumer that cannot rely on the shape of what it receives will reconstruct it, and a reconstructed inventory is a second inventory, and two inventories over one estate is divergence.

20.2 The environment-qualification sibling consumes the inventory's lock state, toolchain constraint, and required-services fields, and it produces the qualification verdict that gates the interpretation of every result. The contract is directional and must not be inverted: this dataset predicts that an environment will not build, from a read of two files, and never asserts that it did. The assertion is the sibling's, it is made from an execution, and it is the only admissible basis for a blocked verdict. What this dataset supplies is the prediction and the reason, which is what converts an unexplained compilation error into a lock-state finding before anyone spends a build second on it.

20.3 The reconciliation sibling consumes the authored-surface census and the runner's collection configuration and computes the gap between what exists and what executes. The contract here is that this dataset supplies both numbers independently and never the difference, because the difference is a judgement and the numbers are observations. The distinction matters: an instrument that emits only the gap has destroyed the information required to diagnose it, since a gap can be produced by an orphaning scope line, by a false test-file count, or by a runner that failed to import a module, and those three demand entirely different remedies.

20.4 The risk-mapping sibling consumes the surface map's effect classes and ingress fields and produces the risk tier that determines what evidence a change must carry. The contract is that this dataset classifies effects and never computes blast radius or reversibility, because both are owned by the estate's decision layer and are already computed there. Where a risk tier is needed, it is requested and the answer is recorded. A second blast-radius computation standing beside the first is not redundancy; it is a guarantee that two components will one day report different tiers for the same change, and that neither will be wrong.

20.5 The epistemic interface is the strictest and admits no local dialect. Every claim this dataset emits carries a level drawn from the estate's existing evidence ladder, and this dataset defines no confidence scale of its own. A classification derived from a manifest read is at the level a direct observation licenses. A classification derived from an extension census in the absence of a manifest is weaker and is recorded as weaker. A surface inferred from a naming convention rather than from a registry is weaker still. The ceiling is set by the weakest read beneath the claim, and volume of weak reads does not raise it, because repetition is not strength.

20.6 The compound-intelligence pipeline is where this dataset's contribution to the estate's compounding is defined, and the definition must be concrete rather than gestural. A single failure discovered through this dataset's instruments must yield at least four permanent assets, and the four are named. The causal record: the failure captured with its repository, commit, component, environment, and demonstrated cause, and — specific to this axis — the inventory field that was wrong or missing, because a reality-layer failure always has one. The regression record: an assertion that fails against the unfixed state and passes against the fixed one, observed in both directions, because a test never seen to fail may be asserting a tautology.

20.7 The third and fourth assets are what make the pipeline compound rather than merely record. The training example: the negative case, in which the description was trusted and the fabricated finding was produced, paired with the positive case, in which the disk was consulted and the premise collapsed — and the pairing is what makes it a training example rather than an anecdote, because a model or an agent that sees only the correction learns the correction and not the discrimination. The benchmark case: a scenario carrying a seeded defect that a correct pass must detect, together with the false-positive trap that a naive detector would wrongly flag, derived from the real failure rather than invented, because a benchmark with no real origin is an exercise and not a guard.

20.8 This dataset's own benchmark family is drawn directly from the founding audit and is stated so that it can be built. A repository whose name implies an executable domain and whose disk contains none — a correct pass classifies it as content and returns not-applicable for coverage, and a naive pass emits a critical finding about untested logic that does not exist. A repository whose default runner collects a subtree while functional tests sit in siblings — a correct pass reports the gap between authored and executed, and a naive pass reports a green suite. A repository whose value-moving component has no test reference while its aggregate count is large — a correct pass surfaces the unprotected surface, and a naive pass reports excellent coverage. A repository whose harness pollutes its host — a correct pass attributes the failure to the environment, and a naive pass repairs the product.

20.9 The routing of every one of those records is not this dataset's job, and stating the boundary here is what keeps the family coherent. The transformation of an insight into its destination form — a structural rule, a process rule, a trap, a corpus entry, a benchmark case, a prompt fragment, or a considered discard — is owned by the frontier layer's router, which exists, is wired, and fires. The graph that holds the surfaces, contracts, failures, and invariants as nodes and edges is owned by the navigation layer. The compilation of a bug into a permanent structural rule is owned by the hard-rule extractor. The scoring of a deliverable against a done-gate is owned by the output-contract layer. This dataset produces records and hands them over, and any Part of it that appears to be routing, scoring, graphing, or promoting has drifted and must be corrected.

20.10 The done criteria for any application of this dataset are five, and all five are artifacts rather than assertions. First, a scan output exists for every unit in the estate, carrying its commit and its explicit unknowns. Second, every unit's classification has been derived from the disk in this pass and not inherited from a prior one. Third, a surface map exists per unit with a stated scope field, and every enumerated surface carries an effect class. Fourth, the binding has been computed and the empty-set analysis — high effect class, empty test edge — has been produced and read. Fifth, every unknown has survived into the final artifact without being rounded, and every aggregate carries its unknown count. A pass that produces four of the five has not produced eighty percent of the value; it has produced an artifact whose missing field is invisible, and an invisible missing field is the failure this dataset exists to prevent.

PART XX FINAL LAW

This dataset produces three artifacts — inventory, classification, surface map with bindings — and every downstream judgement is a function of them, so an error here propagates while acquiring the appearance of rigour. One failure must yield four permanent assets: the causal record naming the inventory field that was wrong, the regression observed to fail before it passed, the training example pairing the trusted description with the consulted disk, and the benchmark carrying both a seeded defect and the false-positive trap a naive detector would flag. Routing, graphing, scoring, and promotion are owned elsewhere; this axis produces records and hands them over.
