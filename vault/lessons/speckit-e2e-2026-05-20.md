## B. Live E2E — Owner-driven session required


Synthetic feature chosen so the structural test exercises every template section: P1 (the basic list), P2 (remote flag), P3 (relative date formatting). ## C. Worked synthetic spec (proof of template fill)

This is what `/speckit-spec "<branch-lister description>"` should produce — filled deterministically from `vault/templates/speckit/spec.md.template` with no Owner clarifications because every template section maps to a unique aspect of the description:

```markdown
# Feature Specification: branch-lister CLI

## User Scenarios

### P1 — list local branches sorted by recency  (must-have, MVP)
Independent test: `branch-lister` from a fresh clone returns local refs
ordered by most-recent commit date, one per line.

### P2 — include remote-tracking refs  (should-have)
Independent test: `branch-lister --remote` includes `origin/*` refs in
the same chronological ordering.

### P3 — relative date formatting  (nice-to-have)
Independent test: output includes "N {day|week|month} ago" suffix
matching git's `--relative` semantics.

## Functional Requirements
- FR-001: The CLI MUST enumerate local branches via the git plumbing
  (`git for-each-ref refs/heads/`), not by parsing porcelain output.
- FR-002: The CLI MUST sort by `committerdate:iso8601` descending.
- FR-003: The CLI MUST accept a `--remote` flag that additionally
  enumerates `refs/remotes/origin/`.
- FR-004: The CLI MUST format each line as `<branch> <relative-date>`
  separated by a single tab.

## Key Entities
- BranchRef: name, committer_date (UTC), is_remote (bool)

## Success Criteria
- SC-001: p95 < 500 ms on a 200-branch repo (warm cache).
- SC-002: Output is byte-identical across two consecutive runs when the
  repo has not changed.

## Edge Cases
- Detached HEAD: omit from output (no branch ref).
- Empty repo (no commits): exit 0 with empty stdout.
- Packed refs: handled transparently by git plumbing.

## Assumptions
- `git` is on PATH and is >= 2.0 (for `git for-each-ref --format`).
- Caller's shell handles tab-separated output.
```

Note: zero `[NEEDS CLARIFICATION: ...]` markers. The Owner's natural-language description was already specific enough that every template slot fills deterministically. If the description had been "list branches", FR-002/003/004 and SC-001 would all carry markers and `/speckit-clarify` would resolve them in one batch.
